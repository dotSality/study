## Часть 1. Ортографическая проекция

### § 9.1.1. Идея: параллельное проецирование

*Ортографическая* (orthographic) проекция отбрасывает глубину «в лоб»: все лучи проецирования параллельны. Удалённые объекты **не** уменьшаются — рельсы не сходятся на горизонте. Так рисуют чертежи, изометрические игры, интерфейсы и карты теней [S6, ch. 5; S8, ch. 4; S9].

### § 9.1.2. Матрица: коробка → куб

Задача матрицы: взять «рабочую коробку» view-пространства — x ∈ [l, r] (left/right), y ∈ [b, t] (bottom/top), глубины от n до f (near/far, расстояния **перед** камерой, то есть z_view от −n до −f) — и линейно перегнать её в стандартный куб **NDC** (normalized device coordinates) [−1, 1]³, с которым дальше работает железо.

Для каждой оси это одно и то же аффинное действие «сдвинь и растяни» (гл. 7), например по x: центр (l+r)/2 → 0, длина r−l → 2. Собрав три оси:

```
              ⎡ 2/(r−l)     0         0        −(r+l)/(r−l) ⎤
P_ortho   =   ⎢    0     2/(t−b)      0        −(t+b)/(t−b) ⎥
              ⎢    0        0      −2/(f−n)    −(f+n)/(f−n) ⎥
              ⎣    0        0         0              1      ⎦
```

[S9, «Coordinate Systems»; S8, ch. 4; S10].

Знак «−» в z-строке — знакомая тонкость из главы 1: камера смотрит вдоль **−z**, а NDC-ось z растёт **в глубь экрана** (левая тройка!), матрица разворачивает ось.

Обрати внимание: последняя строка (0, 0, 0, 1) — матрица **аффинная**, w не трогает. Ортопроекция — обычное «сдвинь-растяни» из главы 7; вся неаффинная магия достанется части 2.

**Пример 1.** Камера UI: l = 0, r = 800, b = 0, t = 600, n = −1, f = 1 (глубина почти не нужна). Матрица: x′ = 2x/800 − 1, y′ = 2y/600 − 1, z′ = −z.
Точка «центр экрана» (400, 300, 0): → (0, 0, 0) — центр NDC. Угол (800, 600, 0) → (1, 1, 0). Это в точности формулы пересчёта пикселей из Г1.Ч1.З10 — ортопроекция и есть та самая смена координат.

**Пример 2.** Карта теней от солнца: солнце «смотрит» ортографической камерой с коробкой шириной в сцену. Параллельные лучи проекции = параллельные лучи солнца — потому для направленного света берут именно орто.

**Типичные ошибки.**

1. Задать n и f так, что объекты не попали в коробку, — «у меня чёрный экран» (всё отсечено).
2. Растянуть коробку не по аспекту окна — картинка сплющена (соотношение (r−l)/(t−b) должно совпадать с соотношением сторон вьюпорта).
3. Ждать от орто перспективного уменьшения: его не будет по построению.

**Источники части**: [S6] Dunn & Parberry, ch. 5; [S8] Real-Time Rendering, ch. 4; [S9] learnopengl.com, «Coordinate Systems»; [S10] Scratchapixel.

## Часть 2. Перспективная проекция

### § 9.2.1. Идея: дальше — меньше

Глаз и камера-обскура работают иначе: лучи сходятся в одной точке (центре проекции). Предмет вдвое дальше выглядит вдвое меньше. Геометрия подобных треугольников даёт для экрана на расстоянии d от глаза:

```
x_экрана = d · x / (−z)          (камера смотрит вдоль −z, z < 0 перед камерой)
```

Деление на глубину — **нелинейная** операция: никакая матрица 3×3/4×4 сама по себе так не умеет (матрица делает только линейные комбинации координат). Хитрость индустрии: матрица **готовит** деление, записывая глубину в w, а само деление выполняется отдельным шагом [S6, ch. 5; S10; S8, ch. 4].

### § 9.2.2. Perspective divide: зачем нужен w

Вспомни § 7.2.4: (x, y, z, w) читается как точка (x/w, y/w, z/w). План перспективной проекции:

1. матрица кладёт в w_clip значение **−z_view** (положительную глубину точки перед камерой);
2. железо делит: (x, y, z, w) → (x/w, y/w, z/w) — *perspective divide*;
3. результат — NDC в кубе [−1, 1]³.

Пространство между шагами 1 и 2 называется **clip space** — именно в нём живёт `gl_Position` и выполняется отсечение (clipping): видимое ограничено неравенствами −w ≤ x, y, z ≤ w (после деления это и даст куб ±1) [S11; S8, ch. 4].

### § 9.2.3. Матрица перспективы

Стандартная матрица (в стиле gluPerspective; параметры: вертикальный угол обзора fovy, аспект a = ширина/высота, ближняя и дальняя плоскости n, f > 0) [S9, «Coordinate Systems»; S8, ch. 4]:

```
c = 1 / tan(fovy / 2)

              ⎡ c/a    0        0             0       ⎤
P_persp   =   ⎢  0     c        0             0       ⎥
              ⎢  0     0   −(f+n)/(f−n)  −2fn/(f−n)   ⎥
              ⎣  0     0       −1             0       ⎦
```

Читаем по строкам:

- **x, y**: масштаб по углу обзора (чем шире fov, тем сильнее сжатие — больше мира влезает в кадр); деление на аспект уравнивает пиксели по осям;
- **w**: строка (0, 0, −1, 0) кладёт в w_clip ровно −z_view — глубину. Вот оно, «деление на расстояние»;
- **z**: коэффициенты подобраны так, чтобы **после деления** глубина z = −n отобразилась в −1, а z = −f — в +1 (проверим на числах ниже). Последняя строка НЕ (0,0,0,1) — матрица не аффинная, единственная такая в пайплайне.

**Пример 3 (проверка near/far).** Возьмём n = 1, f = 5: z-строка = (0, 0, −6/4, −10/4) = (0, 0, −1.5, −2.5).

Точка на ближней плоскости, view-координаты (0, 0, −1, 1):
z_clip = −1.5·(−1) − 2.5 = −1; w_clip = −(−1) = 1. После деления: z_ndc = −1/1 = **−1**. ✓

Точка на дальней, (0, 0, −5, 1):
z_clip = −1.5·(−5) − 2.5 = 5; w_clip = 5; z_ndc = 5/5 = **+1**. ✓

Середина по глубине, (0, 0, −3, 1):
z_clip = 4.5 − 2.5 = 2; w_clip = 3; z_ndc = 2/3 ≈ **0.67** — не 0! Глубина в NDC **нелинейна** (гиперболична по z_view) — важнейший факт части 3.

**Пример 4 («дальше — меньше» на числах).** fovy = 90° (c = 1), a = 1. Две точки одного «роста»: (1, 1, −2, 1) и (1, 1, −10, 1).
Первая: clip (1, 1, …, 2) → NDC x = y = **0.5**.
Вторая: clip (1, 1, …, 10) → NDC x = y = **0.1**.
Один и тот же метровый предмет в пяти шагах дальше стал в пять раз меньше на экране. Перспектива работает — и это просто деление на w.

**Типичные ошибки.**

1. Читать `gl_Position` как «пиксели» или «NDC»: это clip space, **до** деления. Руками делить не надо — железо само; но интерполировать в нём что-то своё — нужно с пониманием.
2. n ≤ 0 в перспективе: формулы содержат деление на глубину; near обязан быть > 0 (и это не каприз API — при z = 0 деление на ноль).
3. Ставить fov «на глаз» в градусах туда, где функция ждёт радианы (классика: сцена «в рыбьем глазу» или «в телескопе»).
4. Забыть аспект при ресайзе окна — круги становятся яйцами.

**Источники части**: [S6] Dunn & Parberry, ch. 5; [S8] Real-Time Rendering, ch. 4; [S9] learnopengl.com, «Coordinate Systems»; [S10] Scratchapixel (perspective projection); [S11] спецификация GLSL (clip space).

## Часть 3. Глубина, NDC и вьюпорт

### § 9.3.1. NDC: последняя общая система

После деления все видимые точки лежат в **NDC-кубе**: x, y, z ∈ [−1, 1] (соглашение OpenGL; в других API куб может отличаться по z — например, [0, 1]). Напоминание из гл. 1: NDC OpenGL — **левая** система (x вправо, y вверх, z в экран). Здесь заканчивается «математика сцены» и начинается растеризация [S9; S10; S8].

### § 9.3.2. Буфер глубины и нелинейность z

z_ndc используется для **теста глубины**: кто ближе — тот и виден. Из примера 3 мы знаем: z_ndc нелинеен по z_view. Распишем общий вид (из z-строки после деления):

```
z_ndc = (f + n)/(f − n) + 2fn / ((f − n)·z_view)
```

— гипербола по z_view. Следствие: **точность глубины распределена неравномерно** — львиная доля диапазона z_ndc расходуется на зону возле near-плоскости, а дальняя половина сцены ютится в хвостике значений.

**Пример 5.** n = 0.1, f = 100. Точка на z_view = −1 (в метре): z_ndc = (100.1/99.9) + 20/(99.9·(−1)) ≈ 1.002 − 0.2002 ≈ **0.802**. Точка на z_view = −50 (в 50 м): z_ndc ≈ 1.002 − 0.004 ≈ **0.998**. Первые «метр» сцены занял 90% диапазона [−1, 1] (от −1 до 0.8), последние 50 метров — доли процента. Два дальних объекта с разницей глубин в полметра имеют z_ndc, различающиеся в шестом знаке, — привет, **z-fighting** (мерцание поверхностей, спорящих за пиксель).

Практические правила, следующие из формулы: **отодвигай near так далеко, как можешь терпеть** (n в числителе 2fn — главный рычаг точности), и не проси f «про запас».

### § 9.3.3. Viewport: NDC → пиксели

Финальный шаг — аффинное растяжение NDC в прямоугольник окна (viewport с началом (x₀, y₀), размером W×H):

```
x_win = x₀ + (x_ndc + 1)/2 · W
y_win = y₀ + (y_ndc + 1)/2 · H          (начало — левый нижний угол, OpenGL)
z_win = (z_ndc + 1)/2                   (глубина в [0, 1] по умолчанию)
```

Круг замкнулся: мы вернулись к экранным координатам главы 1 (§ 1.1.5) — включая «y вверх у OpenGL» и пиксельные центры на полуцелых значениях.

**Связь с практикой (сквозной пример через ВЕСЬ пайплайн).** Соберём главы 7–9 воедино. Сцена: вершина p_local = (1, 1, 0, 1); M = T((0, 0, −2)); камера из примера 3 гл. 8 (eye = (0, 0, 5), V сдвигает на −5 по z); перспектива примера 4 (c = 1, a = 1, n = 1, f = 5); окно 800×600.

1. world: (1, 1, −2, 1) (гл. 8, пример 4);
2. view: (1, 1, −7, 1) — стоп, f = 5 < 7: точка **за дальней плоскостью**. Подвинем дальнюю: n = 1, f = 10 ⟹ z-строка (0, 0, −11/9, −20/9);
3. clip: x = 1, y = 1, z = −11/9·(−7) − 20/9 = 57/9 ≈ 6.33, w = 7;
4. NDC: (1/7, 1/7, 0.905) ≈ (0.143, 0.143, 0.905) — внутри куба ✓;
5. viewport 800×600: x_win = (0.143+1)/2·800 ≈ **457**, y_win = (0.143+1)/2·600 ≈ **343**, глубина z_win ≈ 0.952.

Пиксель (457, 343), чуть правее-выше центра (400, 300), — разумно: точка была правее и выше оптической оси. Семь глав математики → один пиксель.

**Типичные ошибки.**

1. near = 0.001 «чтобы ничего не резалось» — и весь дальний план тонет в z-fighting (пример 5).
2. Сравнивать z_ndc как «расстояние в метрах»: он нелинеен; для честной метрической глубины восстанавливай z_view из формулы § 9.3.2.
3. Забыть, что viewport-y растёт вверх (OpenGL), а мышиные координаты — вниз (гл. 1): пикинг попадает «в зеркальную» точку по вертикали.

## Резюме главы

- Ортопроекция — аффинное «коробка → куб [−1,1]³»: без уменьшения с расстоянием; UI, изометрия, карты теней. Аспект коробки = аспект окна.
- Перспектива = подобные треугольники = деление на глубину. Матрица неаффинная: кладёт −z_view в w (строка (0,0,−1,0)), z-строка калибрует near → −1, far → +1; деление на w (perspective divide) делает железо. gl_Position — это clip space, отсечение: −w ≤ x, y, z ≤ w.
- z_ndc нелинеен (гипербола): точность толпится у near; маленький near убивает дальнюю глубину (z-fighting). Правило: отодвигай near.
- Viewport: (ndc + 1)/2 · размер — возвращение к экранным координатам главы 1. Полный маршрут: local −M→ world −V→ view −P→ clip −÷w→ NDC −viewport→ пиксель.
