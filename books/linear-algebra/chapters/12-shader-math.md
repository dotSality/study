# Глава 12. Математика шейдеров (капстоун)

> Самоучитель «Линейная алгебра для 3D-разработки». Программа — в `book/outline.md`, источники — в `book/sources.md`. Решения заданий — в `book/solutions/12-shader-math.md` (создаётся на этапе 5).

## Введение

Финал. Одиннадцать глав математики собираются в работающий код. Шейдер — маленькая программа, выполняющаяся на GPU для каждой вершины (вертексный) или каждого пикселя (фрагментный), — и почти всё, что шейдер делает, это линейная алгебра этого курса: dot-ы, cross-ы, нормализации и умножения матриц. Здесь мы пройдёмся по языку GLSL глазами математика, разберём канонический вертексный шейдер построчно, соберём освещение из dot-произведений и завершим курс проектом: собственная мини-библиотека векторной математики + осмысленная шейдерная сцена.

Практическая среда: подойдёт любой WebGL/OpenGL-каркас — в этом репозитории уже есть песочница `shaders/beginner`, годятся и онлайн-редакторы (Shadertoy для фрагментных экспериментов, The Book of Shaders editor [S12]).

После главы ты будешь уметь:

- читать и писать GLSL-типы и функции с пониманием их точной семантики (по спецификации, а не по фольклору);
- собирать вертексный шейдер: атрибуты, uniform-ы, MVP, нормали;
- строить освещение Ламберта и Блинна–Фонга и объяснять каждый dot в нём;
- (проект) реализовать vec3/mat4/quat с нуля и проверить их против выкладок всего курса.

Требуется: весь курс, особенно главы 2 (dot/cross), 7 (TRS, нормали), 8 (пайплайн), 9 (проекции), 10 (кватернионы).

## Часть 1. Векторы и матрицы в GLSL

### § 12.1.1. Типы

GLSL — язык со встроенной линейной алгеброй [S11, § 4.1]:

- скаляры: `float`, `int`, `uint`, `bool`;
- векторы: `vec2`, `vec3`, `vec4` (float); есть `ivec*`, `uvec*`, `bvec*`;
- матрицы: `mat2`, `mat3`, `mat4` (квадратные, float; есть и прямоугольные `matCxR`).

Конструкторы — рабочие лошадки семантики (гл. 7!):

```glsl
vec4 p = vec4(pos, 1.0);       // точка: w = 1
vec4 d = vec4(dir, 0.0);       // направление: w = 0
mat3 linearPart = mat3(model); // верхний левый 3×3 блок mat4 — линейная часть!
```

`mat3(mat4)` вырезает ровно тот блок, который в гл. 7 назывался «линейной частью аффинной матрицы» [S11, § 5.4]. Напоминание из гл. 4: матрицы в GLSL хранятся и конструируются **по столбцам**, `m[i]` — i-й столбец, `m[i][j]` — строка j столбца i.

### § 12.1.2. Свиззлинг

Доступ к компонентам — тремя эквивалентными наборами имён: `.xyzw` (геометрия), `.rgba` (цвет), `.stpq` (текстуры) — смешивать наборы в одном обращении нельзя [S11, § 5.5]:

```glsl
vec3 v = vec3(1.0, 2.0, 3.0);
v.zyx        // vec3(3.0, 2.0, 1.0) — перестановка
v.xx         // vec2(1.0, 1.0) — дублирование
color.rgb    // первые три компоненты цвета
uv.st        // текстурные координаты
```

Свиззлинг — не синтаксический сахар, а линейная алгебра: `v.zyx` — это умножение на матрицу перестановки (гл. 4), просто бесплатное.

### § 12.1.3. Операторы: где покомпонентно, а где по-настоящему

Критичная таблица [S11, § 5.9]:

| Выражение | Что делает |
|---|---|
| `u + v`, `u - v` | покомпонентно (гл. 2) |
| `u * v` (vec × vec) | **покомпонентно!** НЕ dot и не cross |
| `f * v` | скаляр × вектор (гл. 2) |
| `M * v` (mat × vec) | матрица × вектор-столбец (гл. 4) |
| `M * N` (mat × mat) | матричное произведение (гл. 4) |
| `v * M` | vᵀ·M — вектор-строка слева (транспонированное соглашение!) |
| `matrixCompMult(M, N)` | покомпонентное произведение матриц (редко нужно) |

`u * v` для векторов — покомпонентное (произведение Адамара): идиоматично для цветов (`baseColor * lightColor`), но геометрического смысла произведений главы 2 не имеет — для них есть `dot(u, v)` и `cross(u, v)`.

### § 12.1.4. Встроенные функции — словарь курса

Геометрические функции GLSL [S11, § 8.5] — прямые цитаты из глав 2 и 7:

```glsl
length(v)            // ‖v‖                           (гл. 2, § 2.2.1)
distance(a, b)       // ‖b − a‖                       (гл. 1, § 1.1.4)
dot(u, v)            // скалярное произведение         (гл. 2, ч. 3)
cross(u, v)          // векторное, только vec3         (гл. 2, ч. 4)
normalize(v)         // v/‖v‖; v = 0 → не определено!  (гл. 2, § 2.2.2)
reflect(I, N)        // I − 2·dot(N, I)·N              (отражение; N единичный!)
refract(I, N, eta)   // преломление (Снеллиус)
faceforward(N, I, Nref) // разворот нормали к наблюдателю
inverse(M), transpose(M) // гл. 4; в шейдере — дорого, готовь на CPU
mix(a, b, t)         // a(1−t) + tb — линейная комбинация (гл. 2)
clamp(x, lo, hi), step(edge, x), smoothstep(e0, e1, x), min, max, pow
```

Разберём `reflect(I, N) = I − 2·dot(N, I)·N` как упражнение курса: dot(N, I) — проекция падающего вектора на нормаль (гл. 2, § 2.3.4); вычесть её дважды — значит развернуть нормальную составляющую, сохранив касательную. Зеркало за одну строку. Важно из спецификации: **I направлен К поверхности**, N должен быть единичным — иначе формула тихо врёт.

**Типичные ошибки.**

1. `u * v` там, где нужен `dot(u, v)`, — компилируется, считает чушь.
2. `normalize()` нулевого вектора и `reflect()` с ненормированным N — «неопределённое поведение» по спецификации, NaN на практике (гл. 2).
3. `v * M` вместо `M * v` — транспонированное соглашение, «почти работает» на симметричных матрицах (гл. 4, ч. 2).
4. `inverse()` в фрагментном шейдере на каждый пиксель — тысячекратная переплата за то, что вычисляется один раз на CPU (гл. 7, ч. 4).

**Источники части**: [S11] спецификация GLSL 4.60, § 4–5, 8.5; [S12] The Book of Shaders (гл. о векторах); [S9] learnopengl.com.

## Часть 2. Вертексный шейдер и трансформации

### § 12.2.1. Канонический вертексный шейдер, построчно

```glsl
#version 300 es
// Атрибуты: свои у КАЖДОЙ вершины (из вершинного буфера)
in vec3 aPos;        // позиция в ЛОКАЛЬНОМ пространстве (гл. 8, ч. 1)
in vec3 aNormal;     // нормаль в локальном пространстве

// Uniform-ы: одинаковы для всех вершин в рамках draw call
uniform mat4 u_model;        // local → world       (гл. 7–8)
uniform mat4 u_view;         // world → view        (гл. 8, ч. 2)
uniform mat4 u_projection;   // view → clip         (гл. 9)
uniform mat3 u_normalMatrix; // (M⁻¹)ᵀ, 3×3          (гл. 7, ч. 4)

// Выходы: интерполируются по треугольнику для фрагментного шейдера
out vec3 vWorldPos;
out vec3 vNormal;

void main() {
    vec4 worldPos = u_model * vec4(aPos, 1.0);        // ТОЧКА: w = 1
    vWorldPos = worldPos.xyz;
    vNormal   = u_normalMatrix * aNormal;             // вектор, 3×3, inverse-transpose
    gl_Position = u_projection * u_view * worldPos;   // → clip space (гл. 9)
}
```

Каждая строка — глава курса. `gl_Position` — clip space: дальше железо само выполнит отсечение (−w ≤ x,y,z ≤ w), perspective divide и viewport (гл. 9). Ничего из этого руками делать не нужно — но теперь ты знаешь, что происходит.

### § 12.2.2. Что происходит между шейдерами: интерполяция

Выходы вертексного шейдера (`out`) приходят во фрагментный (`in`) **интерполированными** по треугольнику: для пикселя внутри треугольника железо смешивает значения трёх вершин с весами, пропорциональными близости (барицентрические координаты — та самая «разложи точку по вершинам» система уравнений из гл. 3, § 3.1.4). По умолчанию интерполяция перспективно-корректная — с учётом w (гл. 9), чтобы текстуры не «плыли» на наклонных поверхностях [S11; S8, ch. 2; S10].

Практическое следствие для нормалей: интерполяция трёх **единичных** векторов даёт, вообще говоря, **неединичный** (хорда короче дуги — тот же эффект, что у lerp кватернионов, гл. 10, § 10.4.2). Отсюда железное правило: `normalize(vNormal)` **во фрагментном шейдере**, даже если нормализовал в вертексном.

### § 12.2.3. Анимация трансформаций

Всё управление движением — на CPU (или в вертексном шейдере из uniform-времени): пересобери M = T·R·S с новым углом — и объект вращается. Порядок глав 7–8 обязателен: масштаб → поворот → перенос, иерархии — умножением родительских матриц. Кватернионы (гл. 10) живут на CPU-стороне: `model = translate(pos) * mat4(R(q)) * scale(s)`.

**Типичные ошибки.**

1. `vec4(aPos, 0.0)` — вершина стала «направлением», объект не двигается с места (гл. 7, ч. 2). Симметрично: `vec4(aNormal, 1.0)` — нормали «едут» при переносе (Г7.Ч2.З7).
2. Нормали через `mat3(u_model)` при неравномерном масштабе — сплющенное освещение (гл. 7, ч. 4).
3. Забытый `normalize` во фрагментном шейдере после интерполяции (§ 12.2.2).
4. Пересчёт `u_projection * u_view` на каждую вершину вместо склейки на CPU (гл. 8, § 8.3.2) — работает, но оплачивается миллионами лишних умножений.

**Источники части**: [S9] learnopengl.com («Coordinate Systems», «Basic Lighting»); [S11] спецификация GLSL; [S8] Real-Time Rendering, ch. 2–4; [S10] Scratchapixel.

## Часть 3. Освещение как линейная алгебра

### § 12.3.1. Три вектора освещения

Для каждого освещаемого фрагмента (все — единичные, все — в одном пространстве, гл. 8, § 8.3.3!):

```
N — нормаль поверхности
L — направление НА источник света (от фрагмента)
V — направление НА камеру (от фрагмента)
```

Для точечного света L = normalize(lightPos − fragPos) — задача Г2.Ч2.З9, для направленного (солнце) L = −lightDir — константа.

### § 12.3.2. Диффузное освещение (Ламберт)

Матовая поверхность рассеивает свет пропорционально косинусу угла падения (закон Ламберта):

```glsl
float diff = max(dot(N, L), 0.0);
vec3 diffuse = diff * lightColor * baseColor;
```

— dot единичных векторов = косинус (гл. 2, § 2.3.2); `max(…, 0)` срезает «отрицательный свет» с обратной стороны (гл. 2, таблица знаков) [S9, «Basic Lighting»; S8, ch. 5].

**Пример 1.** Пол с N = (0, 1, 0). Свет из-под 45°: L = normalize((1, 1, 0)) ⟹ diff = dot = 1/√2 ≈ 0.71 — 71% яркости. Свет в зените: L = (0, 1, 0), diff = 1. Свет из-под пола: dot < 0 → 0. (Это Г2.Ч3.З10, ставшая кодом.)

### § 12.3.3. Блики: Фонг и Блинн–Фонг

Глянец — свет, отражённый «почти в глаз». Классический Фонг сравнивает отражённый луч с направлением на камеру:

```glsl
vec3 R = reflect(-L, N);                          // -L: падающий луч К поверхности!
float spec = pow(max(dot(R, V), 0.0), shininess);
```

Блинн–Фонг — стандартная оптимизация: вместо R сравнивает нормаль с «половинным вектором» (биссектрисой между L и V — их нормированной суммой, гл. 2):

```glsl
vec3 H = normalize(L + V);
float spec = pow(max(dot(N, H), 0.0), shininess);
```

Геометрия: N совпала с H ⟺ поверхность стоит ровно так, чтобы зеркально отбросить свет в глаз. `pow` сужает блик: shininess 8 — воск, 128 — полированный металл. Блинн–Фонг дешевле (нет reflect) и лучше ведёт себя при скользящих углах — потому он «дефолт индустрии» [S9, «Advanced Lighting»; S8, ch. 5].

Итоговая классика (ambient + diffuse + specular — модель Фонга по компонентам):

```glsl
vec3 color = ambient * baseColor
           + diff * lightColor * baseColor
           + spec * lightColor;
```

### § 12.3.4. Затухание и прожектор — снова dot и length

- **Затухание точечного света**: физическое ~ 1/d² (d = `distance(lightPos, fragPos)` — гл. 1!); на практике 1/(k_c + k_l·d + k_q·d²).
- **Прожектор (spotlight)**: фрагмент внутри конуса ⟺ `dot(-L, spotDir) > cos(cutoff)` — ровно проверка поля зрения охранника из Г2.Ч3.З11.

Весь свет — четыре инструмента главы 2: dot, normalize, length, max.

**Типичные ошибки.**

1. Векторы в разных пространствах (N — мир, L — view): свет «ездит» с камерой (гл. 8, § 8.3.3).
2. `reflect(L, N)` вместо `reflect(-L, N)`: по спецификации I смотрит К поверхности; L у нас ОТ поверхности — знак обязателен.
3. Ненормализованные N (после интерполяции!), L, H — блики тускнеют и плавают.
4. Забытый `max(0, …)`: отрицательный diffuse затемняет ambient, отрицательный specular в `pow` с нецелой степенью — NaN.

**Источники части**: [S9] learnopengl.com («Basic Lighting», «Advanced Lighting»); [S8] Real-Time Rendering, ch. 5; [S11] спецификация GLSL, § 8.5; [S12] The Book of Shaders.

## Резюме главы

- GLSL — язык со встроенной линейной алгеброй: vec/mat, свиззлинг, конструкторы с семантикой (w = 1/0; mat3(mat4) = линейная часть). `*` векторов — покомпонентное; геометрия — в dot/cross/normalize/reflect (у которых есть строгие предусловия из спецификации).
- Вертексный шейдер = главы 7–9 в пять строк: атрибуты (на вершину), uniform-ы (на объект), varying-и (интерполяция по треугольнику — барицентрические веса, перспективно-корректная); интерполированные нормали нормализуй заново.
- Освещение = dot-произведения: Ламберт max(dot(N, L), 0); Блинн–Фонг pow(max(dot(N, H), 0), sh); прожекторы и затухание — те же dot и length. Все векторы — единичные и в одном пространстве.
- Проект: vec3/mat4/quat своими руками, сверенные с числовыми примерами курса, + сцена с кватернионной анимацией и Блинном–Фонгом. Сдал — курс пройден.
