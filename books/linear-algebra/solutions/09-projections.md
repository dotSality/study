## Часть 1. Ортографическая проекция

### Г9.Ч1.З1

**Условие.** Выпиши P_ortho для коробки l = −10, r = 10, b = −5, t = 5, n = 0.1, f = 100.

**Решение.** Подставляем в формулу § 9.1.2: 2/(r−l) = 2/20 = 0.1; 2/(t−b) = 0.2; −(r+l)/(r−l) = 0; −(t+b)/(t−b) = 0; −2/(f−n) = −2/99.9 ≈ −0.02002; −(f+n)/(f−n) = −100.1/99.9 ≈ −1.002:

```
            ⎡0.1   0      0         0   ⎤
P_ortho  =  ⎢ 0   0.2     0         0   ⎥
            ⎢ 0    0   −0.02002  −1.002 ⎥
            ⎣ 0    0      0         1   ⎦
```

### Г9.Ч1.З2

**Условие.** Примени матрицу из З1 к view-точкам (0, 0, −0.1, 1), (0, 0, −100, 1), (10, 5, −50, 1). Куда они попали в NDC? Какие на границах куба?

**Решение (код подтвердил).**
- (0, 0, −0.1, 1): z′ = −0.02002·(−0.1) − 1.002 = **−1** — ближняя плоскость, граница z = −1;
- (0, 0, −100, 1): z′ = 2.002 − 1.002 = **+1** — дальняя плоскость, граница z = +1;
- (10, 5, −50, 1): x′ = 1, y′ = 1, z′ ≈ −0.001 ≈ 0 — **угол куба по x и y** (правый верх), середина по глубине.

**Ответ:** (0,0,−1); (0,0,1); (1,1,≈0) — первые две на границах глубины, третья на границах x, y.

### Г9.Ч1.З3

**Условие.** Выведи x-строку P_ortho: найди a и c в x′ = a·x + c из условий l → −1, r → +1.

**Решение.** Система из двух уравнений (гл. 3):

```
a·l + c = −1
a·r + c = +1
```

Вычитаем: a·(r − l) = 2 ⟹ **a = 2/(r − l)**. Подставляем: c = −1 − a·l = −1 − 2l/(r − l) = −(r − l + 2l)/(r − l) = **−(r + l)/(r − l)**. ∎ Совпало с матрицей § 9.1.2. Смысл: a — растяжение отрезка длины (r − l) в отрезок длины 2, c — сдвиг центра (l + r)/2 в ноль.

### Г9.Ч1.З4

**Условие.** Окно 1600×900. Подбери l, r, b, t для орто-камеры «высотой 10 мировых единиц» с правильным аспектом и центром в нуле.

**Решение.** Высота 10 ⟹ b = −5, t = 5. Аспект окна 1600/900 = 16/9 ⟹ ширина коробки = 10·16/9 = 160/9 ≈ 17.78 ⟹ l = −80/9 ≈ −8.89, r = 80/9 ≈ 8.89.
Проверка: (r − l)/(t − b) = 17.78/10 = 16/9 ✓ — круги останутся кругами.

**Ответ:** l ≈ −8.89, r ≈ 8.89, b = −5, t = 5.

### Г9.Ч1.З5

**Условие.** Точка имеет NDC-координаты (0.5, −0.5). Окно 800×600, viewport на всё окно. В какой пиксель она попадёт?

**Решение.** Формула § 9.3.3: x_win = (x_ndc + 1)/2·W = 1.5/2·800 = **600**; y_win = (−0.5 + 1)/2·600 = 0.5/2·600 = **150** (код подтвердил).
Напоминание: начало оконных координат OpenGL — левый **нижний** угол, так что пиксель (600, 150) — правее центра и в нижней части экрана (y_ndc = −0.5 — ниже середины ✓).

**Ответ:** (600, 150).

### Г9.Ч1.З6

**Условие.** Докажи, что P_ortho переводит параллельные прямые в параллельные. Какое свойство какого класса преобразований это гарантирует?

**Решение.** P_ortho — аффинная матрица: последняя строка (0, 0, 0, 1), общий вид «линейная часть + сдвиг» (диагональное масштабирование + перенос центра коробки). А аффинные преобразования сохраняют параллельность (гл. 7, § 7.1.2): прямая p + t·d переходит в (Ap + c) + t·(Ad) — направление образа Ad зависит только от направления d, так что прямые с равными d остаются параллельными. ∎
Это и отличает орто от перспективы: та делит на w и параллельность разрушает (рельсы сходятся).

### Г9.Ч1.З7

**Условие.** Изометрическая игра: художник жалуется, что «дальние домики должны быть меньше, а они такие же». Программист меняет near/far — не помогает. Объясни, почему параметры орто-матрицы тут не виноваты, и что на самом деле выбирают между орто и перспективой.

**Решение.** «Дальше — меньше» — эффект **деления на глубину**, которого в орто нет по построению: её z-строка (как и x, y) — обычное аффинное «сдвинь-растяни», глубина не попадает в знаменатель ни при каких l, r, b, t, n, f. Параметры меняют лишь, *какая часть сцены* влезает в куб, но не закон проецирования — параллельные лучи остаются параллельными (З6).
Что выбирают на самом деле: **проекционную матрицу целиком**. Нужен перспективный масштаб от расстояния — бери P_persp (часть 2), возможно с узким fov для «почти изометрии». (Либо оставаться в орто и масштабировать спрайты вручную — художественный трюк классических изометрий.)

**Ответ:** уменьшение с расстоянием — свойство перспективного деления, у орто его нет в принципе; выбирается сам тип проекции.

## Часть 2. Перспективная проекция

### Г9.Ч2.З1

**Условие.** Вычисли c для fovy = 60° (tan 30° ≈ 0.577) и выпиши P_persp для a = 16/9, n = 0.5, f = 100.

**Решение.** c = 1/tan(30°) = 1/0.577 ≈ **1.732** (= √3, код подтвердил).
Элементы: c/a = 1.732/(16/9) ≈ 0.974; −(f+n)/(f−n) = −100.5/99.5 ≈ −1.010; −2fn/(f−n) = −100/99.5 ≈ −1.005:

```
            ⎡0.974    0      0       0   ⎤
P_persp  =  ⎢  0    1.732    0       0   ⎥
            ⎢  0      0   −1.010  −1.005 ⎥
            ⎣  0      0     −1       0   ⎦
```

### Г9.Ч2.З2

**Условие.** Прогони через матрицу примера 3 (c = 1, a = 1, n = 1, f = 5) точку (2, 0, −4, 1): clip, divide, NDC. Внутри ли куба ±1?

**Решение (код подтвердил).** z-строка: (0, 0, −1.5, −2.5).
clip: x = 2, y = 0, z = −1.5·(−4) − 2.5 = 3.5, w = −(−4) = 4 ⟹ **(2, 0, 3.5, 4)**.
divide: NDC = (2/4, 0/4, 3.5/4) = **(0.5, 0, 0.875)**.
Все компоненты в [−1, 1] — точка видима ✓ (по глубине уже близко к дальней плоскости: 0.875 при f, дающем 1).

**Ответ:** clip (2, 0, 3.5, 4); NDC (0.5, 0, 0.875); внутри.

### Г9.Ч2.З3

**Условие.** Повтори вывод «подобных треугольников»: глаз в начале, экран на расстоянии d, точка (x, z) — получи x′ = d·x/(−z). Какую роль играет c в первой строке матрицы?

**Решение.**

```
        x |            * (x, z)      Треугольники О-экран-x′ и О-точка-x
          |           /              подобны: x′/d = x/(−z)
       x′ +          /               (−z — положительное расстояние
          |  экран  /                 до точки перед камерой)
        О +---+----+------ −z        ⟹ x′ = d·x/(−z)
              d
```

Луч из глаза О к точке пересекает экран; вертикальные катеты относятся как горизонтальные расстояния — отсюда формула. ∎
Роль c: сравни с матрицей — после деления на w = −z получаем x_ndc = c·x/(−z): **c и есть «расстояние до экрана»** d для экрана полувысоты 1. c = 1/tan(fovy/2): узкий fov ⟹ экран «дальше» ⟹ объекты крупнее (телеобъектив); широкий ⟹ ближе ⟹ мельче (ширик).

### Г9.Ч2.З4

**Условие.** Для установки примера 4 найди NDC-координаты точек (1, 1, −1, 1) и (1, 1, −100, 1). К чему стремится проекция при z → −∞? Как это связано с «точкой схода»?

**Решение (код подтвердил).** c = a = 1: clip x = y = 1, w = −z.
(1, 1, −1, 1): NDC (1/1, 1/1) = **(1, 1)** — угол кадра.
(1, 1, −100, 1): NDC = **(0.01, 0.01)** — почти центр.
При z → −∞: x_ndc = x/(−z) → 0 — проекция стремится к **(0, 0)**, центру экрана. Это и есть точка схода: все точки с фиксированными x, y и растущей глубиной (например, точки рельса, параллельного взгляду) стягиваются в одну точку на горизонте. Перспектива «рисует» сходящиеся рельсы автоматически — простым делением.

**Ответ:** (1, 1) и (0.01, 0.01); предел (0, 0) — точка схода для направлений, параллельных оси взгляда.

### Г9.Ч2.З5

**Условие.** Для произвольных n, f покажи алгеброй, что z_view = −n даёт z_ndc = −1 и z_view = −f даёт z_ndc = +1.

**Решение.** z-строка: z_clip = −((f+n)/(f−n))·z_view − 2fn/(f−n); w_clip = −z_view.
*При z_view = −n:* z_clip = (f+n)·n/(f−n) − 2fn/(f−n) = (fn + n² − 2fn)/(f−n) = (n² − fn)/(f−n) = −n(f − n)/(f−n) = **−n**; w = n; z_ndc = −n/n = **−1** ✓.
*При z_view = −f:* z_clip = (f+n)·f/(f−n) − 2fn/(f−n) = (f² − fn)/(f−n) = f(f−n)/(f−n) = **f**; w = f; z_ndc = **+1** ✓. ∎
(Код прогнал обе проверки для двух пар (n, f).) Коэффициенты z-строки — это в точности решение системы «−n ↦ −1, −f ↦ +1» для дроби вида (αz + β)/(−z).

### Г9.Ч2.З6

**Условие.** Точка стоит ровно в плоскости камеры: z_view = 0. Что с ней делает матрица (чему равно w_clip)? Почему отсечение обязано выкинуть её ДО деления?

**Решение.** w_clip = −z_view = **0**: perspective divide превратился бы в деление на ноль — NaN/Inf на выходе. Геометрически честно: точка сбоку от глаза проецируется «в бесконечность» (луч из глаза к ней параллелен экрану).
Поэтому отсечение работает в **clip space, до деления**: условие −w ≤ z ≤ w при w = 0 не выполняется (кроме вырожденного z = 0, отсекаемого соседними неравенствами по x, y... и в целом точка вне пирамиды видимости) — примитив режется или отбрасывается раньше, чем возникнет деление на ноль. Порядок «clip → divide» — не прихоть, а защита арифметики.

**Ответ:** w_clip = 0, деление невозможно; клиппинг до divide гарантирует, что до деления доживают только точки с w > 0.

### Г9.Ч2.З7

**Условие.** «Более киношная картинка — сплющить перспективу»: что происходит при уменьшении fovy (90° → 30°) с одновременным отъездом камеры, сохраняющим размер главного объекта? Опиши эффект на фоне (dolly zoom).

**Решение.** Два процесса компенсируются на главном объекте: c = 1/tan(fovy/2) растёт (всё укрупняется), отъезд уменьшает — объект в кадре тот же. Но фон живёт по другой арифметике: видимый размер ∝ c/расстояние, а **отношение** расстояний до объекта и до фона при отъезде стремится к 1 (с 10 м и 20 м → 100 м и 110 м). Следствие: фон «подтягивается» к объекту и укрупняется, перспективные сокращения исчезают, сцена сплющивается — при fov → 0 перспектива вырождается в орто (часть 1!). Обратный ход (наезд + расширение fov) «растягивает» пространство — эффект головокружения из «Челюстей» и «Головокружения» Хичкока.

**Ответ:** объект стабилен, фон наплывает и сплющивается; в пределе — ортопроекция.

### Г9.Ч2.З8

**Условие.** `projection = perspective(fov, width/height, 0.0, 1000.0);` — сцена не рисуется или глубина сходит с ума. Какой параметр невозможен и почему?

**Решение.** **near = 0.0.** Подставим n = 0 в матрицу: слагаемое −2fn/(f−n) = 0 и −(f+n)/(f−n) = −1 ⟹ z_clip = z_view·(−1)·... = −z_view = w_clip ⟹ z_ndc = z_clip/w_clip = **1 для всех точек**: вся сцена ложится на дальнюю плоскость, буфер глубины теряет смысл (кто впереди — лотерея). Вдобавок объекты при z_view → 0 дают w → 0 — деления на ноль у самой камеры.
Перспективная проекция требует **n > 0** — и не микроскопического (пример 5 главы: крошечный near съедает точность дали). Лечение: near = 0.1…1.0 по масштабу сцены.

**Ответ:** near = 0 запрещён: глубина вырождается (z_ndc ≡ 1), у камеры — деление на ноль.

### Г9.Ч2.З9

**Условие.** Проверь −w ≤ z ≤ w для трёх точек примера 3 (near, mid, far) и для точки за камерой (0, 0, +2, 1) при n = 1, f = 5. Какая отсеклась и по какому неравенству?

**Решение (clip-координаты перепроверены кодом).**

| точка (view) | z_clip | w_clip | −w ≤ z | z ≤ w | вердикт |
|---|---|---|---|---|---|
| (0, 0, −1) near | −1 | 1 | −1 ≤ −1 ✓ | −1 ≤ 1 ✓ | видима (граница near) |
| (0, 0, −3) mid | 2 | 3 | ✓ | ✓ | видима |
| (0, 0, −5) far | 5 | 5 | ✓ | 5 ≤ 5 ✓ | видима (граница far) |
| (0, 0, +2) за камерой | −5.5 | **−2** | −(−2) = 2 ≤ −5.5 ✗ | | **отсечена** по −w ≤ z |

Заметь диагностический признак: у точки за камерой w_clip < 0 — такие точки отсекаются всегда (обычно уже проверкой w > 0), и именно поэтому «отражения сцены за спиной» не прорываются в кадр.

**Ответ:** отсеклась (0, 0, +2) — нарушено −w ≤ z (и вообще w < 0).

## Часть 3. Глубина, NDC и вьюпорт

### Г9.Ч3.З1

**Условие.** Для n = 1, f = 5 вычисли z_ndc точек с z_view = −1, −2, −3, −4, −5 и построй табличку. Равные ли шаги по z_ndc? Где гуще?

**Решение.** Формула § 9.3.2: z_ndc = 1.5 + 2.5/z_view (код подтвердил все значения):

| z_view | −1 | −2 | −3 | −4 | −5 |
|---|---|---|---|---|---|
| z_ndc | −1 | 0.25 | 0.667 | 0.875 | 1 |
| шаг | | 1.25 | 0.417 | 0.208 | 0.125 |

Равным шагам глубины (по 1 метру) отвечают **экспоненциально убывающие** шаги z_ndc: первый метр занял 62% диапазона, последний — 6%. Плотность значений (точность) — у ближней плоскости; гипербола, не прямая.

**Ответ:** шаги 1.25 → 0.417 → 0.208 → 0.125; сгущение у near.

### Г9.Ч3.З2

**Условие.** Для n = 0.1 и n = 1 (f = 100) сравни z_ndc точки z_view = −50. Где у дальних объектов больше «запас различимости»? Сформулируй правило про near.

**Решение (код подтвердил).**
n = 0.1: z_ndc = 100.1/99.9 + 20/(99.9·(−50)) ≈ 1.002 − 0.004 = **0.998** — до дальней границы 0.002;
n = 1: z_ndc = 101/99 + 200/(99·(−50)) ≈ 1.0202 − 0.0404 = **0.980** — до границы 0.020, в **10 раз** больше запаса.
Правило: точность дальней глубины пропорциональна n (слагаемое 2fn растёт с n) — **отодвигай near настолько, насколько терпит сцена**; десятикратное увеличение near даёт примерно десятикратный выигрыш точности вдали.

**Ответ:** 0.998 против 0.980; больший near — больше различимости дали.

### Г9.Ч3.З3

**Условие.** Выведи формулу § 9.3.2 из z-строки матрицы P_persp.

**Решение.** z_clip = −((f+n)/(f−n))·z_view − 2fn/(f−n); w_clip = −z_view. Делим:

```
z_ndc = z_clip / w_clip = [−((f+n)/(f−n))·z_view − 2fn/(f−n)] / (−z_view)
      = (f+n)/(f−n) + 2fn / ((f−n)·z_view)      ∎
```

(первое слагаемое: −(…)·z_view/(−z_view) = +(…); второе: −(…)/(−z_view) = +(…)/z_view). Константа плюс гипербола по z_view — вся нелинейность глубины в одном слагаемом 2fn/((f−n)·z_view).

### Г9.Ч3.З6

**Условие.** Прогони сквозной пример § 9.3.3 целиком для вершины p_local = (−1, 0, 0, 1), та же сцена (n = 1, f = 10). Все пять шагов, с числами.

1. **world**: M = T((0, 0, −2)): (−1, 0, −2, 1);
2. **view**: V (камера в (0, 0, 5), смотрит на начало): (−1, 0, −7, 1);
3. **clip**: P (c = 1, a = 1, n = 1, f = 10; z-строка (0, 0, −11/9, −20/9)): x = −1, y = 0, z = 77/9 − 20/9 = 57/9 ≈ 6.333, w = 7 ⟹ (−1, 0, 6.333, 7);
4. **NDC** (÷w): (−1/7, 0, 57/63) ≈ (−0.143, 0, 0.905) — внутри куба ✓;
5. **viewport** 800×600: x_win = (1 − 0.143)/2·800 ≈ **343**, y_win = (0 + 1)/2·600 = **300**; глубина z_win = (0.905 + 1)/2 ≈ 0.952.

**Ответ:** пиксель ≈ (343, 300), глубина ≈ 0.952 — зеркально по x к вершине (1, 1, 0) из главы (та дала ≈(457, 343)): вершина левее оси взгляда легла левее центра ✓.

### Г9.Ч3.З7

**Условие.** Флора на горизонте мерцает; near = 0.01, far = 5000. Объясни механизм и предложи два независимых исправления.

**Решение.** Механизм (пример 5 + З2): при n = 0.01 слагаемое 2fn крошечное, и почти весь диапазон z_ndc расходуется на первые сантиметры; на дистанции «горизонта» соседние поверхности (листья, ветки, земля) отличаются по z_ndc в единицах последних битов буфера глубины. Тест глубины превращается в лотерею, победитель меняется от кадра к кадру — **z-fighting**, мерцание.
Исправления:
1. **Отодвинуть near** (0.5–1.0): точность дали вырастает пропорционально (в 50–100 раз) — самое дешёвое лекарство; заодно проверить, не завышен ли far «про запас».
2. **Улучшить сам буфер**: 32-битный float depth и/или **reversed-z** — идея: отобразить near → 1, far → 0; плотность значений float максимальна около нуля, и она достаётся дальнему плану — вместе с гиперболой распределение точности выравнивается почти идеально.

**Ответ:** микроскопический near съел точность дали; лечится увеличением near и reversed-z/float-буфером.

### Г9.Ч3.З8

**Условие.** Два API: OpenGL (z_ndc ∈ [−1, 1]) и другой с z_ndc ∈ [0, 1]. Одинаковой ли матрицей перспективы можно пользоваться? Какую строку менять и почему остальные не трогаются?

**Решение.** Матрица почти одинакова: x-, y- и w-строки не зависят от целевого диапазона глубины (они отвечают за кадр и за деление). Различается только **z-строка** — она калибруется под «near ↦ левый конец, far ↦ правый»:
- для [−1, 1]: z-строка (0, 0, −(f+n)/(f−n), −2fn/(f−n)) (§ 9.2.3);
- для [0, 1]: решаем ту же пару условий −n ↦ 0, −f ↦ 1 и получаем (0, 0, −f/(f−n), −fn/(f−n)).
Одной матрицей на оба API пользоваться нельзя (глубина уедет — половина диапазона потеряется или сцена обрежется), но конверсия — замена двух коэффициентов одной строки. Библиотеки (glm и др.) для этого держат парные варианты функций перспективы.

**Ответ:** нет — z-строка своя для каждого диапазона: (−(f+n)/(f−n), −2fn/(f−n)) против (−f/(f−n), −fn/(f−n)); остальные строки к глубине отношения не имеют.
