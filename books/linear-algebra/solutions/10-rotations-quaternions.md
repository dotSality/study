# Решения. Глава 10: Повороты в 3D — углы Эйлера и кватернионы

> Сборник решений самоучителя «Линейная алгебра для 3D-разработки». Задания — в `book/chapters/10-rotations-quaternions.md`. Все выкладки перепроверены кодом (матрицы, Родриг, кватернионная алгебра, интерполяции). Соглашение композиции углов: R = R_y(yaw)·R_x(pitch)·R_z(roll), вектор-столбец.

## Часть 1. Углы Эйлера и их пределы

### Г10.Ч1.З1

**Условие.** Выпиши, вокруг какой оси крутят yaw, pitch, roll самолёта. Какая тройка углов у самолёта, летящего носом строго вверх?

**Решение.** yaw (рыскание) — вокруг **вертикальной** оси (y): «нос влево-вправо»; pitch (тангаж) — вокруг **поперечной** оси (x, вдоль крыльев): «нос вверх-вниз»; roll (крен) — вокруг **продольной** оси (z, нос–хвост): «покачивание крыльями».
Нос строго вверх: pitch = ±90° (у нашего соглашения — +90°). Значения yaw и roll при этом **не определены однозначно**: это поза шарнирного замка, годится любая пара с одинаковой разностью (см. З4), например (0°, 90°, 0°) или (30°, 90°, 30°).

**Ответ:** вертикальная / поперечная / продольная оси; тройка (α, 90°, β) — целое семейство с фиксированным α − β.

### Г10.Ч1.З2

**Условие.** Вычисли R_y(90°)·R_x(90°)·(1, 0, 0) и R_x(90°)·R_y(90°)·(1, 0, 0). Сверь с экспериментом «с телефоном».

**Решение (код подтвердил).**
*Первый порядок (сначала pitch, потом yaw — читаем справа налево):* R_x(90°)·(1, 0, 0) = (1, 0, 0) — вектор на оси вращения, не сдвинулся; R_y(90°)·(1, 0, 0) = (0, 0, −1). Итог: **(0, 0, −1)**.
*Второй:* R_y(90°)·(1, 0, 0) = (0, 0, −1); R_x(90°)·(0, 0, −1) = (0, 1, 0). Итог: **(0, 1, 0)**.
Разные результаты — повороты вокруг разных осей не коммутируют; телефон в руках показывает то же самое.

**Ответ:** (0, 0, −1) и (0, 1, 0).

### Г10.Ч1.З3

**Условие.** Вычисли R_x(90°)·(0, 0, 1) и R_x(90°)·(0, 1, 0) и опиши, куда легли локальные оси после pitch = 90°.

**Решение (код подтвердил).** R_x(90°)·(0, 0, 1) = **(0, −1, 0)**; R_x(90°)·(0, 1, 0) = **(0, 0, 1)**.
После pitch = 90° локальная ось z (ось roll) легла на **−y** — на ту же прямую, что мировая ось yaw (y). Две «ручки» из трёх крутят теперь вокруг одной прямой — вот анатомия замка из § 10.1.2.

**Вариант 2 (не умножать, а прочитать матрицу по столбцам).** Оба произведения — умножения матрицы на **базисные** векторы, а такие произведения вычислять не нужно: по теореме § 5.1.2 столбцы матрицы и есть образы базисных векторов. Выпишем R_x(90°) (гл. 7, § 7.3.1) с cos 90° = 0, sin 90° = 1 и просто прочитаем её:

```
          ⎡1  0   0⎤     столбец 1 = e₁′ = ( 1,  0,  0)   ← ось поворота на месте
R_x(90°) = ⎢0  0  −1⎥     столбец 2 = e₂′ = ( 0,  0,  1)   ← y легла на z
          ⎣0  1   0⎦     столбец 3 = e₃′ = ( 0, −1,  0)   ← z легла на −y
```

Оба ответа получены без единого умножения, и сразу все три — вместе с картиной «куда поехал весь репер» (код сверил оба пути).
*Сравнение.* Умножение — универсальный приём: оно работает для любого вектора, а не только для орта. Чтение столбцов применимо лишь к базисным векторам, зато мгновенно и даёт полную картину поворота — это тот самый навык «читать матрицу глазами» (§ 10.2.3, п. 5). Практическая польза именно здесь: увидев в отладчике третьим столбцом (0, −1, 0), сразу понимаешь — локальная z смотрит вдоль мировой −y, то есть объект в позе pitch = 90°, на пороге замка; проверять это прогоном вектора не нужно. Обратная сторона медали: для произвольного v чтение столбцов ничего не даёт — придётся раскладывать v по ортам, а это и есть умножение на матрицу.

### Г10.Ч1.З4

**Условие.** Перемножь символьно R_y(α)·R_x(90°)·R_z(β) и покажи, что результат зависит только от α ± β. Сколько различных ориентаций достижимо из этой позы двумя «ручками»?

**Решение.** Шаг 1: R_x(90°) = [[1,0,0],[0,0,−1],[0,1,0]]. Умножаем на R_z(β) (строки R_x комбинируют строки R_z):

```
R_x(90°)·R_z(β) = ⎡cos β  −sin β   0⎤
                  ⎢  0       0    −1⎥
                  ⎣sin β   cos β   0⎦
```

Шаг 2: слева R_y(α) (строки (cos α, 0, sin α), (0,1,0), (−sin α, 0, cos α)):

```
                     ⎡cos α·cos β + sin α·sin β    −cos α·sin β + sin α·cos β    0⎤   ⎡ cos(α−β)   sin(α−β)   0⎤
R_y(α)R_x(90°)R_z(β) = ⎢          0                            0                −1⎥ = ⎢    0          0      −1⎥
                     ⎣−sin α·cos β + cos α·sin β    sin α·sin β + cos α·cos β    0⎦   ⎣−sin(α−β)   cos(α−β)   0⎦
```

(свёрнуто по формулам косинуса/синуса разности). Матрица зависит **только от α − β** ∎ (код проверил: тройки (10°, 90°, 0°) и (30°, 90°, 20°) дали одну матрицу поэлементно).
Достижимо: однопараметрическое семейство ориентаций (окружность) — вместо двумерного множества, которое дали бы две независимые ручки. Одна степень свободы потеряна — это и есть замок в алгебре.

**Ответ:** зависимость только от α − β; достижима лишь «окружность» ориентаций.

**Вариант 2 (сопряжение: поворот вокруг повёрнутой оси — без единого перемножения).** Символьная выкладка *доказывает* факт, но не *объясняет* его: свёртка в cos(α − β) выглядит совпадением. Объяснение даёт лемма, которую стоит знать отдельно от этой задачи.

**Лемма (поворот вокруг повёрнутой оси).** Пусть Q — какой-нибудь поворот, R_n(θ) — поворот вокруг единичной оси n на угол θ. Тогда

```
Q · R_n(θ) · Q⁻¹ = R_{Qn}(θ)
```

— то же самое вращение, но вокруг **повёрнутой** оси. Доказательство в три шага, без координат:

1. *Это поворот.* Произведение ортогональных матриц ортогонально, а det = det Q · 1 · det Q⁻¹ = 1 (§ 4.3.3, п. 2; § 10.2.3, п. 1–2, 4).
2. *Ось — это Qn.* Подставим: (Q·R_n·Q⁻¹)(Qn) = Q·R_n·n = Q·n, поскольку R_n не двигает свою ось. Вектор Qn неподвижен ⟹ он и есть ось.
3. *Угол — тот же θ.* Возьмём u ⟂ n. Поворот Q сохраняет скалярные произведения (Г6.Ч3.З5), поэтому Qu ⟂ Qn, а угол между Qu и Q(R_n·u) равен углу между u и R_n·u, то есть θ. Направление обхода тоже сохранено: det Q = +1 (отражение с det = −1 перевернуло бы знак угла — гл. 4). ∎

Это ровно pivot-паттерн § 7.3.4 («смени систему, сделай дело, вернись»), только в роли «смены системы» здесь не перенос, а поворот.

Применяем. Из **З3** уже известно: R_x(90°) переводит ось z в −y. Значит, по лемме с Q = R_x(90°) и n = e₃:

```
R_x(90°) · R_z(β) · R_x(90°)⁻¹ = R_{−y}(β) = R_y(−β)
```

(поворот на β вокруг −y — это поворот на −β вокруг +y). Переносим R_x(90°) направо и подставляем в исходную композицию:

```
R_x(90°)·R_z(β) = R_y(−β)·R_x(90°)
  ⟹  R_y(α)·R_x(90°)·R_z(β) = R_y(α)·R_y(−β)·R_x(90°) = R_y(α−β)·R_x(90°)
```

Последний шаг — сложение углов поворотов вокруг **одной** оси: R_y(a)·R_y(b) = R_y(a + b) (правило Г5.Ч3.З2; поворот вокруг y — это поворот плоскости xz, вложенный в 3D). Зависимость только от α − β получена без единого перемножения матриц ∎ (код сверил обе записи на шести парах (α, β): расхождение до 1e−12).
Ответ на второй вопрос читается тут же: достижимо семейство {R_y(γ)·R_x(90°)}, где γ = α − β, — одна окружность ориентаций, ровно как в варианте 1.

*Сравнение.* Вариант 1 — честный труд: он ничего не требует знать заранее и одинаково сработает для любого порядка осей и любого фиксированного pitch. Вариант 2 объясняет **механизм** замка: средний поворот укладывает ось третьего поворота на ось первого, после чего третья ручка перестаёт быть независимой и просто добавляется к первой со знаком. Отсюда мгновенно получаются варианты условия: при pitch = **−**90° ось z ложится на **+**y, лемма даёт R_y(+β), и матрица зависит от α **+** β (код подтвердил) — вот откуда оговорка «или α + β, по знакам» в примере 2 главы. Тот же приём отвечает и на вопрос «а при других порядках осей замок есть?»: он наступает у **любой** тройки — при том критическом угле среднего поворота, который совмещает ось третьего с осью первого (для порядка Y·X·Z это pitch = ±90°). Наконец, лемма живёт далеко за пределами этой задачи: сопряжением объясняется и структура кватернионного сэндвича q·v·q⁻¹ (§ 10.3.2), и приём «повернуть кость вокруг мировой оси» M·R·M⁻¹ в иерархиях (гл. 8).

### Г10.Ч1.З5

**Условие.** FPS-камера: yaw без ограничений, pitch клампится в [−89°, 89°]. Объясни каждую цифру.

**Решение.** *Yaw без ограничений:* вращение вокруг вертикали не приближает к замку (замок — свойство pitch) и физически осмысленно на любые углы — игрок оборачивается сколько угодно.
*Pitch ±89°, а не ±90°:* на ±90° (взгляд строго вверх/вниз) наступает шарнирный замок (З3–З4) и, что практичнее, вырождается look-at: forward ∥ up, cross(f, up) ≈ 0 — NaN (Г2.Ч4.З10, гл. 8, § 8.2.3). Градус запаса держит систему на безопасном расстоянии и от алгебраической беды, и от численной. *Почему не ±89.9°:* можно и так; 89° — традиционный запас с учётом float-шума и интерполяций.

### Г10.Ч1.З6

**Условие.** «`euler.y += input * dt; rotation = fromEuler(euler)` — безопасно; `rotation = fromEuler(rotationToEuler(rotation) + delta)` — иногда дёргается». В чём разница?

**Решение.** *Первый паттерн:* углы — **единственный источник истины**; конверсия euler → rotation односторонняя, выполняется заново от полного значения угла. Ошибки не накапливаются, неоднозначность обратной конверсии не участвует.
*Второй:* цикл rotation → euler → rotation. Обратная конверсия rotationToEuler **неоднозначна**: у полюса (pitch ≈ ±90°) одной ориентации соответствует целое семейство троек (З4), и библиотека выбирает представителя по внутренним правилам — от кадра к кадру выбор может скакать (yaw 0° → 179°, roll компенсирует). Прибавка delta к «скакнувшей» тройке даёт видимый рывок. Плюс накопление float-ошибок двойной конверсии.
Правило (из «типичных ошибок» § 10.1.2): хранить ориентацию в кватернионе/матрице, углы — только как входной интерфейс, конверсию «в углы» не зацикливать.

### Г10.Ч1.З7

**Условие.** В редакторе объект показывает углы (0°, 90°, 0°), художник вводит (10°, 90°, 10°) — объект не шевелится визуально. Объясни через пример 2, почему две разные тройки дают одну ориентацию, и почему у полюса инспектор «сходит с ума».

**Решение.** При pitch = 90° матрица зависит только от разности yaw − roll (З4). У обеих троек она одинакова: 0° − 0° = 10° − 10° = 0 ⟹ **одна и та же ориентация** (код подтвердил поэлементным сравнением матриц). Прибавка +10° к yaw в точности скомпенсирована +10° к roll: обе ручки у полюса крутят вокруг одной прямой в противоположные (с учётом знаков сцепки) стороны.
«Сходит с ума» инспектор потому, что обратная задача «матрица → тройка углов» у полюса имеет бесконечно много ответов: редактор вынужден выбирать произвольного представителя семейства, и после любой операции (гизмо, анимация) может показать другую тройку той же ориентации — числа прыгают, объект стоит.

**Ответ:** у полюса ориентация определяется только yaw − roll; тройки с равной разностью неотличимы, а разложение в углы неоднозначно.

## Часть 2. Ось–угол и матрицы поворота

### Г10.Ч2.З1

**Условие.** По формуле Родрига поверни v = (0, 0, 1) вокруг n = (0, 1, 0) на 90° и на 180°. Сверь первый ответ с R_y из гл. 7.

**Решение (код подтвердил).** Слагаемые: n·v = 0 (третье слагаемое нулевое); n × v = (0,1,0) × (0,0,1) = (1, 0, 0).
*90°:* v·cos 90° + (n×v)·sin 90° = 0 + (1, 0, 0) = **(1, 0, 0)**. Сверка: R_y(90°)·(0, 0, 1) = (sin 90°, 0, cos 90°) = (1, 0, 0) ✓.
*180°:* v·(−1) + (n×v)·0 = **(0, 0, −1)** — развернулся, как и положено полуобороту.

**Ответ:** (1, 0, 0); (0, 0, −1).

### Г10.Ч2.З2

**Условие.** Поверни v = (1, 1, 0) вокруг n = (0, 0, 1) на 90° по Родригу. Затем то же — матрицей R_z(90°). Совпало?

**Решение (код подтвердил).** n·v = 0; n × v = (0,0,1) × (1,1,0) = (−1, 1, 0).
Родриг: v·0 + (−1, 1, 0)·1 + 0 = **(−1, 1, 0)**.
Матрицей: R_z(90°)·(1, 1, 0) = (1·0 − 1·1, 1·1 + 1·0, 0) = (−1, 1, 0) ✓ — два формализма, один ответ.

**Ответ:** (−1, 1, 0), совпало.

### Г10.Ч2.З3

**Условие.** Поверни v = (1, 0, 0) вокруг диагональной оси n = (1, 1, 1)/√3 на 120°.

**Решение (код подтвердил).** cos 120° = −1/2; sin 120° = √3/2; 1 − cos = 3/2.
- Слагаемое 1: v·cos = (−1/2, 0, 0).
- Слагаемое 2: n × v = (1,1,1)/√3 × (1,0,0) = (0, 1, −1)/√3; умножить на √3/2: (0, 1/2, −1/2).
- Слагаемое 3: n·v = 1/√3; n·(1/√3)·(3/2) = (1,1,1)·(1/2)/1 = (1/2, 1/2, 1/2).
Сумма: (−1/2 + 1/2, 1/2 + 1/2, −1/2 + 1/2) = **(0, 1, 0)**.
Красота, как и обещано: поворот на 120° вокруг диагонали куба циклически переставляет оси — x перешёл в y (y перейдёт в z, z в x).

**Ответ:** (0, 1, 0).

**Вариант 2 (третий формализм: кватернионный сэндвич).** *Метод части 3 — формальное забегание вперёд; вариант рассчитан на повторное чтение после § 10.3.2. Глава сама обещает эту сверку: «три формализма — один поворот» (пример 4).*
Кватернион поворота (§ 10.3.2): q = (cos(θ/2), n·sin(θ/2)) с θ/2 = 60°. Здесь происходит маленькое чудо: sin 60° = √3/2 сокращается с 1/√3 из нормированной оси, и все иррациональности исчезают —

```
q = (1/2, (1,1,1)/√3 · √3/2) = (1/2, (1/2, 1/2, 1/2)) = (0.5, 0.5, 0.5, 0.5)
```

Вектор как «чистый» кватернион: p = (0, (1, 0, 0)). Считаем сэндвич v′ = q·p·q* по формуле § 10.3.1.
*Шаг 1* — q·p (w₂ = 0, поэтому слагаемое w₂v₁ выпадает):
скаляр = 1/2·0 − (1/2,1/2,1/2)·(1,0,0) = **−1/2**;
вектор = 1/2·(1,0,0) + (1/2,1/2,1/2)×(1,0,0) = (1/2, 0, 0) + (0, 1/2, −1/2) = **(1/2, 1/2, −1/2)**.
*Шаг 2* — умножаем на q* = (1/2, (−1/2, −1/2, −1/2)):
скаляр = (−1/2)(1/2) − (1/2, 1/2, −1/2)·(−1/2, −1/2, −1/2) = −1/4 − (−1/4) = **0** ✓ (скалярная часть обязана занулиться — иначе где-то ошибка: сэндвич единичного кватерниона возвращает чистый кватернион);
вектор = (−1/2)·(−1/2,−1/2,−1/2) + (1/2)·(1/2,1/2,−1/2) + (1/2,1/2,−1/2)×(−1/2,−1/2,−1/2)
    = (1/4, 1/4, 1/4) + (1/4, 1/4, −1/4) + (−1/2, 1/2, 0) = **(0, 1, 0)** ✓
Тот же ответ, что у Родрига (код сверил оба пути и оба промежуточных шага).
*Сравнение.* Родриг короче на один шаг и требует только двух тригонометрических значений — для разового поворота одного вектора он и дешевле (а в оптимизированной записи Г10.Ч3.З9 — ещё дешевле). Кватернион в этой задаче интересен другим: половинный угол превратил «некрасивые» 120° в арифметику половинок, и полученные 4 числа — не разовый расчёт, а **хранимый объект**: его можно скомпоновать с другими поворотами (Г10.Ч3.З6 показывает этот же q как произведение двух четвертьоборотов вокруг y и x), проинтерполировать (часть 4) или превратить в матрицу (§ 10.3.3) — ту самую перестановочную M из Г10.Ч2.З5. Родригом так не поработаешь: ось-угол не складывается (§ 10.2.1).

### Г10.Ч2.З4

**Условие.** Проверь «паспортные данные» матрицы R_x(90°): RᵀR = I и det = 1 — явным вычислением.

**Решение.** R = [[1, 0, 0], [0, 0, −1], [0, 1, 0]]; Rᵀ = [[1, 0, 0], [0, 0, 1], [0, −1, 0]].
RᵀR: диагональ — dot-ы столбцов R с собой: 1, 0+1, 1+0 = 1, 1, 1; внедиагональ — dot-ы разных столбцов: все нули (проверь: столбцы (1,0,0), (0,0,1), (0,−1,0) попарно перпендикулярны). RᵀR = I ✓.
det (разложение по первой строке): 1·(0·0 − (−1)·1) = 1 ✓ — поворот, не зеркало.

### Г10.Ч2.З5

**Условие.** M = [[0, 0, 1], [1, 0, 0], [0, 1, 0]]. Докажи, что это поворот (ортогональность + det). Найди его ось (Mv = v) и угол.

**Решение.** Столбцы (0, 1, 0), (0, 0, 1), (1, 0, 0) — единичные, попарно перпендикулярные ⟹ ортогональна. det: перестановочная матрица цикла x→y→z→x — чётная перестановка, det = +1 (или разложением: 0 − 0 + 1·(1·1 − 0·0) = 1) ⟹ **поворот**.
*Ось:* решаем (M − I)v = 0: M·(x, y, z) = (z, x, y) = (x, y, z) ⟹ z = x, x = y, y = z ⟹ x = y = z. Ось — **(1, 1, 1)/√3** (код подтвердил M·(1,1,1) = (1,1,1)).
*Угол:* M переводит e₁ → e₂ (x в y) — а из З3 мы знаем: ровно это делает поворот на **120°** вокруг диагонали. (Контроль: след M = 0 = 1 + 2cos θ ⟹ cos θ = −1/2 ⟹ θ = 120° ✓.)

**Ответ:** поворот вокруг (1, 1, 1)/√3 на 120°.

**Вариант 2 (ось из антисимметричной части — алгоритм движков).** Гаусс ищет ось как решение однородной системы. Но у матрицы поворота ось и угол **читаются формулой**, без всякой системы, — выведем её из Родрига.
Ключ: транспонирование поворота даёт обратный поворот, Rᵀ = R⁻¹ = R_n(−θ) (§ 10.2.3, п. 1). Распишем действие обеих матриц по формуле Родрига (§ 10.2.2), пользуясь чётностью косинуса и нечётностью синуса:

```
R·v  = v cos θ + (n × v) sin θ + n(n·v)(1 − cos θ)
Rᵀ·v = v cos θ − (n × v) sin θ + n(n·v)(1 − cos θ)
```

Одинаковые (симметричные) слагаемые при вычитании гибнут, остаётся чистая геометрия:

```
(R − Rᵀ)·v = 2 sin θ · (n × v)      для любого v
```

Подставим v = e₁. Слева стоит первый столбец матрицы R − Rᵀ, то есть (0, r₂₁ − r₁₂, r₃₁ − r₁₃); справа 2 sin θ·(n × e₁) = 2 sin θ·(0, n_z, −n_y). Сравнивая покомпонентно: r₂₁ − r₁₂ = 2 sin θ·n_z и r₁₃ − r₃₁ = 2 sin θ·n_y; подстановка v = e₂ добавляет r₃₂ − r₂₃ = 2 sin θ·n_x. Собираем три равенства в один вектор:

```
w = (r₃₂ − r₂₃,  r₁₃ − r₃₁,  r₂₁ − r₁₂) = 2 sin θ · n
```

Для нашей M = [[0, 0, 1], [1, 0, 0], [0, 1, 0]]: w = (1 − 0, 1 − 0, 1 − 0) = (1, 1, 1) ⟹ **n = (1, 1, 1)/√3** ✓ — та же ось, что дал Гаусс, но за три вычитания.
Угол берётся из той же пары чисел: ‖w‖ = 2 sin θ = √3 и след M = 1 + 2 cos θ = 0 дают sin θ = √3/2, cos θ = −1/2, то есть θ = atan2(‖w‖, tr M − 1) = atan2(√3, −1) = **120°** ✓ (код проверил формулу и на общем случае: матрица поворота вокруг (2, −1, 3)/‖·‖ на 53° вернула ровно эту ось и этот угол).
Почему atan2 от двух чисел, а не acos от следа: acos даёт только |θ| — знака он не различает; пара (‖w‖, tr − 1) задаёт угол однозначно, и его знак автоматически согласован с направлением оси n = w/‖w‖.
**Где формула ломается.** При θ = 180° синус обнуляется, а с ним и вся антисимметричная часть: R становится симметричной, w = (0, 0, 0) (код подтвердил). Это не численная неприятность, а принципиальная: у полуоборота нет предпочтительного направления оси, R_n(180°) = R_{−n}(180°). Промышленный обход — отдельная ветка: диагональ Родрига r_ii = cos θ + n_i²(1 − cos θ) при θ = 180° превращается в r_ii = 2n_i² − 1, откуда |n_i| = √((r_ii + 1)/2) (код сверил на оси (2, −1, 3)/‖·‖), а знаки восстанавливают по внедиагональным элементам. Ровно эта развилка «общий случай / полуоборот» сидит внутри любой библиотечной конверсии mat3 → quat.
*Сравнение.* Гаусс — метод «от определения»: он ничего не знает о структуре R, честно показывает, что неподвижное направление существует и единственно (ядро M − I одномерно — гл. 3, гл. 5), и одинаково работает для матрицы любого происхождения. Skew-формула — метод «от структуры»: никаких систем, десяток арифметических операций, ось и угол за один проход, — поэтому именно ею в рантайме конвертируют матрицы в ось-угол и кватернионы (импорт анимаций, скиннинг, физика). Плата двойная: ветка при 180° и требование, чтобы матрица **действительно была поворотом** — для матрицы со скосом формула молча вернёт бессмысленный вектор, тогда как Гаусс просто не нашёл бы неподвижного направления. Отсюда практическое правило: сперва «паспорт» (З4: RᵀR = I, det = +1), потом формула. Мост вперёд: ось поворота — это собственный вектор с λ = 1 (гл. 11, Г11.Ч1.З7), и skew-формула находит его, ни разу не написав характеристическое уравнение.

### Г10.Ч2.З6

**Условие.** Выведи из формулы Родрига матрицу поворота вокруг n = (0, 1, 0) (подставь образы e₁, e₂, e₃ как столбцы) и убедись, что получилась R_y(θ).

**Решение.** Подставляем в v′ = v cos θ + (n×v) sin θ + n(n·v)(1 − cos θ):
- e₁ = (1,0,0): n×e₁ = (0,0,−1); n·e₁ = 0 ⟹ e₁′ = (cos θ, 0, −sin θ);
- e₂ = (0,1,0): n×e₂ = 0; n·e₂ = 1 ⟹ e₂′ = e₂ cos θ + n(1 − cos θ) = (0, 1, 0) — ось на месте;
- e₃ = (0,0,1): n×e₃ = (1,0,0); n·e₃ = 0 ⟹ e₃′ = (sin θ, 0, cos θ).
Столбцами:

```
⎡ cos θ   0   sin θ⎤
⎢   0     1     0  ⎥  =  R_y(θ)  ✓   (гл. 7, § 7.3.1 — включая «перевёрнутые» синусы)
⎣−sin θ   0   cos θ⎦
```

(код сверил Родрига и R_y на θ = 37° — совпадение поэлементное). Родриг — «родитель» всех трёх осевых матриц гл. 7.

### Г10.Ч2.З7

**Условие.** Физдвижок хранит ω = (0, 2, 0) рад/с. За кадр dt = 0.016 с тело надо довернуть. Запиши приращение в форме ось–угол и объясни, почему «повернуть ориентацию» — это R_приращения · R_старая, а не наоборот.

**Решение.** Ось — направление ω: n = (0, 1, 0); угол — «скорость × время»: θ = ‖ω‖·dt = 2·0.016 = **0.032 рад ≈ 1.83°**. Приращение: (n, θ) = ((0, 1, 0), 0.032 рад).
Порядок: ω задана в **мировых** осях (так принято в физдвижках), а матрица, применяемая к вектору последней (левая в цепочке, гл. 5), действует в мировом пространстве: R_new = R_Δ · R_old — «к уже повёрнутому телу применить мировой доворот». Если бы ω была в осях самого тела (локальная угловая скорость), приращение вставало бы справа: R_old · R_Δ. Тот же выбор «мир слева, локаль справа», что у иерархий гл. 8.

**Ответ:** ось (0, 1, 0), угол 0.032 рад; мировое приращение умножается слева.

**Вариант 2 (то же приращение кватернионом — как это устроено в физдвижке).** *Метод части 3; читать после § 10.3.2.* Ось-угол хорош, чтобы приращение **записать**, но хранить в нём ориентацию нельзя: два ось-угла не складываются (§ 10.2.1). Переведём ответ на язык, на котором физика и считает.
Приращение из найденных (n, θ) по рецепту § 10.3.2:

```
q_Δ = (cos(θ/2), n·sin(θ/2)) = (cos 0.016, (0, sin 0.016, 0)) ≈ (0.999872, 0, 0.015999, 0)
```

Обновление: **q_new = normalize(q_Δ · q_old)** — мировое приращение слева, по тому же правилу, что и с матрицами (композиция «сначала q_old, потом q_Δ» записывается как q_Δ·q_old — § 10.3.2). Порядок сомножителей при смене формализма не меняется, меняется только цена.
В коде движка обычно не считают синус с косинусом на каждое тело: при малом θ работает тот же предел, что в Г10.Ч4.З4 (cos x ≈ 1, sin x ≈ x). Подставим θ = ‖ω‖·dt:

```
q_Δ = (cos(‖ω‖dt/2), n·sin(‖ω‖dt/2)) ≈ (1, ω·dt/2)
  ⟹  q_new ≈ normalize( q_old + ½·(0, ω)·q_old·dt )
```

(в последнем переходе (1, ω dt/2) = единица + ½·(0, ω)·dt, и умножение на q_old раскрыто по дистрибутивности). Это и есть стандартный шаг интегрирования ориентации. Нормализация тут не косметика: она чинит и срез ряда, и float-дрейф (Г10.Ч4.З7).
Насколько груб первый порядок — считаем: для ω = (0, 2, 0), dt = 0.016 и произвольной стартовой ориентации приближение разошлось с точным q_Δ·q_old на 1.3·10⁻⁶ по компонентам, то есть на **1.6·10⁻⁴ градуса за кадр** (код). При 60 fps это сотая доля градуса в секунду — и та не копится вслепую, потому что каждый кадр ω берётся заново из состояния тела.
*Сравнение.* Вариант 1 прямее объясняет **порядок**: «мир слева, локаль справа» видно на самих матрицах, и это рассуждение кватернионы не отменяют — оно переносится дословно. Зато цена другая: приращение — 4 числа против 9, композиция — 16 умножений против 27 (таблица § 10.3.3), а главное — лечение накопленного дрейфа: одна нормализация против ре-ортогонализации Грам–Шмидтом (Г10.Ч3.З7, Г6.Ч3.З9). Поэтому в физдвижках ориентация тела **хранится и интегрируется** кватернионом, а в матрицу конвертируется один раз за кадр — уже для рендера.

### Г10.Ч2.З8

**Условие.** Найди ошибку: `rotateAroundAxis(v, axis, θ)` с axis = (0, 2, 0) поворачивает правильно по направлению, но вектор удлиняется. Какое предусловие нарушено и в каком слагаемом Родрига вылезает лишний множитель?

**Решение.** Нарушено предусловие **‖n‖ = 1**. С n = (0, 2, 0) (длина 2):
- слагаемое (n × v)·sin θ получает лишний множитель 2 (cross линеен по аргументу);
- слагаемое n·(n·v)·(1 − cos θ) — лишний множитель **4** (n входит дважды: и в dot, и множителем).
Слагаемые складываются в неправильных пропорциях: направление при оси вдоль координатной оси может выглядеть терпимо, но длина плывёт (и для наклонных осей направление тоже врёт). Починка: нормализовать axis на входе (или assert).

**Ответ:** ненормированная ось; линейный множитель во втором слагаемом и квадратичный — в третьем.

### Г10.Ч3.З1

**Условие.** Составь кватернионы поворотов: 180° вокруг y; 90° вокруг x; 120° вокруг (1, 1, 1)/√3. Проверь единичность.

**Решение (код подтвердил единичность всех трёх).** Рецепт q = (cos(θ/2), n·sin(θ/2)):
- 180° вокруг y: (cos 90°, (0, 1, 0)·sin 90°) = **(0, 0, 1, 0)** — чисто «мнимый» кватернион: полуобороты имеют w = 0;

### Г10.Ч3.З2

**Условие.** Перемножь: (0, (1, 0, 0))·(0, (0, 1, 0)) и в обратном порядке. Что дало dot-слагаемое, что cross?

**Решение (код подтвердил).** Формула: (w₁w₂ − v₁·v₂, w₁v₂ + w₂v₁ + v₁×v₂). Здесь w₁ = w₂ = 0:
- скаляр: −(1,0,0)·(0,1,0) = 0 (векторы перпендикулярны — dot промолчал);
- вектор: 0 + 0 + (1,0,0)×(0,1,0) = (0, 0, 1). Итого **(0, 0, 0, 1)**.
Обратный порядок: cross сменит знак: **(0, 0, 0, −1)**.
Это классические i·j = k и j·i = −k Гамильтона: некоммутативность кватернионов — в точности антикоммутативность cross (гл. 2).

### Г10.Ч3.З3

**Условие.** Прогони расчёт примера 4 для v = (0, 0, 1) (та же q — 90° вокруг y). Сверь с R_y(90°)·(0, 0, 1).

**Решение.** q = (√2/2, (0, √2/2, 0)); v как кватернион: (0, (0, 0, 1)).
*Шаг 1:* q·(0, v): скаляр = −(0, √2/2, 0)·(0, 0, 1) = 0; вектор = √2/2·(0,0,1) + (0,√2/2,0)×(0,0,1) = (0, 0, √2/2) + (√2/2, 0, 0) = (√2/2, 0, √2/2). Итог: (0, (√2/2, 0, √2/2)).

Сверка: R_y(90°)·(0, 0, 1) = (1, 0, 0) ✓ (код подтвердил оба пути).

**Ответ:** (1, 0, 0).

### Г10.Ч3.З4

**Условие.** Проверь для q из примера 4: q·q* = (1, (0, 0, 0)). Почему для единичных кватернионов сопряжение = обращение?

**Решение.** Общий расчёт: q·q* = (w, v)·(w, −v) = (w² − v·(−v), w(−v) + wv + v×(−v)) = (w² + ‖v‖², 0) = (‖q‖², 0).
Для q из примера 4: ‖q‖² = 1/2 + 1/2 = 1 ⟹ q·q* = **(1, 0, 0, 0)** — единица кватернионной алгебры (умножение на неё ничего не меняет; код подтвердил).
Отсюда: если ‖q‖ = 1, то q·q* = 1 ⟹ q* и есть обратный q⁻¹ (аналогично q*·q = 1). Полная аналогия с R⁻¹ = Rᵀ у ортогональных матриц: у «правильных» представлений поворотов обращение бесплатно.

### Г10.Ч3.З5

**Условие.** Подставь q = (√2/2, 0, √2/2, 0) в матричную формулу § 10.3.3 и получи R_y(90°). Затем подставь −q и убедись, что матрица та же.

### Г10.Ч3.З6

**Условие.** q_а — 90° вокруг y, q_б — 90° вокруг x. Вычисли q_б·q_а и примени к (1, 0, 0). Сверь с Г10.Ч1.З2.

### Г10.Ч3.З7

**Условие.** Докажи: норма произведения = произведение норм. Почему произведение единичных кватернионов единично — и почему это спасает от «раздувания» объектов, в отличие от матриц?

**Решение.** Ключевая лемма: (pq)* = q*p* (сопряжение разворачивает порядок — проверяется на формуле умножения). Тогда:

```
‖pq‖² = (pq)(pq)* = p·q·q*·p* = p·‖q‖²·p* = ‖q‖²·(p·p*) = ‖q‖²·‖p‖²  ∎
```

(‖q‖² — скаляр, выносится). Следствие: единичный × единичный = единичный **точно**, в идеальной арифметике композиция никогда не выводит из «множества поворотов».
Практическая разница с матрицами: float-шум и там и там, но у кватерниона «дефект» — одно число (норма), лечится одной нормализацией (4 умножения + корень); у матрицы дефект — потеря ортогональности всех трёх столбцов сразу, лечение — Грам–Шмидт (гл. 6, Г6.Ч3.З9), в разы дороже и грубее. Поэтому длинные цепочки поворотов (анимация, физика) копят кватернионами.

**Вариант 2 (без леммы: раскрыть обе части покомпонентно).** Условие разрешает «принять лемму», но её можно и обойти — прямым раскрытием. Оно длиннее на десять строк и попутно показывает, откуда норма кватерниона знает про dot и cross.
Пусть p = (w₁, v₁), q = (w₂, v₂). По формуле § 10.3.1:

```
‖pq‖² = (w₁w₂ − v₁·v₂)² + ‖w₁v₂ + w₂v₁ + v₁×v₂‖²
```

Раскрываем оба квадрата (второй — как скалярный квадрат суммы трёх векторов, гл. 2):

- из скалярной части: w₁²w₂² − 2w₁w₂(v₁·v₂) + (v₁·v₂)²;
- из векторной: w₁²‖v₂‖² + w₂²‖v₁‖² + ‖v₁×v₂‖² + 2w₁w₂(v₁·v₂) + 2w₁·(v₂·(v₁×v₂)) + 2w₂·(v₁·(v₁×v₂)).

Два последних слагаемых — нули: v₁×v₂ перпендикулярно и v₁, и v₂ (гл. 2, § 2.4.1). А члены ±2w₁w₂(v₁·v₂) взаимно уничтожаются — вот та точка, где «dot-половина» и «cross-половина» кватернионного умножения договариваются между собой; будь в формуле § 10.3.1 другой знак, ничего бы не сошлось. Остаётся:

```
‖pq‖² = w₁²w₂² + w₁²‖v₂‖² + w₂²‖v₁‖² + [ (v₁·v₂)² + ‖v₁×v₂‖² ]
```

Квадратная скобка — **тождество Лагранжа** (Г2.Ч4.З7): (v₁·v₂)² + ‖v₁×v₂‖² = ‖v₁‖²·‖v₂‖². Подставляем и группируем:

```
‖pq‖² = w₁²w₂² + w₁²‖v₂‖² + w₂²‖v₁‖² + ‖v₁‖²‖v₂‖² = (w₁² + ‖v₁‖²)(w₂² + ‖v₂‖²) = ‖p‖²·‖q‖²  ∎
```

(код прогнал на паре произвольных кватернионов и итог, и каждое промежуточное зануление.) В компонентах это равенство известно как тождество Эйлера о четырёх квадратах: произведение двух сумм четырёх квадратов снова есть сумма четырёх квадратов.
*Сравнение.* Вариант 1 короче и концептуальнее: он работает на уровне алгебры («сопряжение разворачивает порядок»), не заглядывая в компоненты, и потому переносится на любые произведения — сразу даёт ‖q₁q₂…q_k‖ = ∏‖q_i‖ для цепочки любой длины. Его цена — лемма (pq)* = q*p*, которую всё равно надо доказать раскрытием. Вариант 2 самодостаточен: кроме школьной алгебры, ему нужны только два факта главы 2 — перпендикулярность cross и тождество Лагранжа. И он показывает **источник** мультипликативности: за ней стоит sin²θ + cos²θ = 1, спрятанный в тождестве Лагранжа (Г2.Ч4.З7). Отсюда же и практический вывод, ради которого задание существует: после умножения кватернионов испортиться может ровно одно число — норма, — потому что все «векторные» перекосы сократились точно.

### Г10.Ч3.З8

**Условие.** Найди ошибку: «кватернион (0.5, 0.5, 0.5, 0.5) я построил как поворот на 90° вокруг (1, 1, 1)». Проверь единичность, затем соответствие рецепту (какой угол и какая ось на самом деле?).

**Решение.** Единичность: 4·0.25 = 1 ✓ — кватернион валидный, но не тот.
Обратная задача: w = cos(θ/2) = 0.5 ⟹ θ/2 = 60° ⟹ **θ = 120°** (код: 2·acos(0.5) = 120°). Векторная часть: (0.5, 0.5, 0.5) = n·sin 60° = n·(√3/2) ⟹ n = (1, 1, 1)/√3 ✓ ось верная.
Ошибка — классика половинного угла: для 90° нужно w = cos 45° = √2/2 ≈ 0.707, а не 0.5. Автор, видимо, взял cos(θ)/2 или «просто поровну». Правильный кватернион 90° вокруг (1,1,1)/√3: (0.707, 0.408, 0.408, 0.408).

**Ответ:** это поворот на 120° (не 90°) вокруг (1, 1, 1)/√3; забыт половинный угол.

**Вариант 2 (не разбирать кватернион, а посмотреть, что он делает).** Обратный рецепт — арифметика над компонентами; но опровергнуть автора можно и экспериментом, вообще не решая уравнений: применим q сэндвичем к базисным векторам (§ 10.3.2).

```
q·e₁·q* = e₂,   q·e₂·q* = e₃,   q·e₃·q* = e₁      (код; ручной расчёт первого — Г10.Ч2.З3, вариант 2)
```

Оси переставлены по циклу x→y→z→x — а это, как уже установлено дважды (Г10.Ч2.З3 и Г10.Ч2.З5), поворот на **120°** вокруг диагонали куба. Ещё более быстрый тест — порядок: q³ = (−1, 0, 0, 0), то есть тождественный поворот (double cover), значит трёхкратное применение возвращает всё на место и θ = 360°/3 = 120°; будь угол 90°, тождество давало бы только q⁴.
Для контраста — что делает **настоящий** кватернион 90° вокруг той же оси, (0.7071, 0.4082, 0.4082, 0.4082): он переводит e₁ в (0.3333, 0.9107, −0.2440) — вектор единичной длины (проверено), но ни на какую ось не похожий. Никакой перестановки осей у него нет — и это видно с первого прогона.
*Сравнение.* Вариант 1 отвечает точно и в общем виде: рецепт θ = 2·acos(w), n = v/sin(θ/2) годится для любого кватерниона и прямо ложится в код (так пишут quatToAxisAngle). Вариант 2 формулы не даёт, зато не требует доверия к арифметике: он **показывает** поворот в действии. Такой тест ловит и ошибки, до которых разбор компонент не добирается, — например, перепутанное соглашение сэндвича (q·v·q* против q*·v·q) или левую систему координат: там рецепт молча вернёт «правильные» 120°, а объект будет крутиться в другую сторону. Приём общий — «прогони преобразование по базису и посмотри на образы» (гл. 5, § 5.1.2); в применении к кватернионам он ещё и объясняет, почему у этого q такие подозрительно ровные компоненты: поворот, переставляющий оси куба, обязан быть симметричным по всем трём.

### Г10.Ч3.З9

**Условие.** Реализуй `quatMul(q1, q2)`, `quatConj(q)`, `quatRotate(q, v)` и воспроизведи пример 4 кодом. Тест: композиция 4 поворотов по 90° вокруг y возвращает вектор на место.

**Решение (код выполнен, оба теста зелёные).**

```js
// q = [w, x, y, z]
function quatMul(q1, q2) {                    // (w₁w₂ − v₁·v₂, w₁v₂ + w₂v₁ + v₁×v₂)
  const [w1, ...v1] = q1, [w2, ...v2] = q2;
  return [w1*w2 - dot(v1, v2),
          ...add(add(scale(v2, w1), scale(v1, w2)), cross(v1, v2))];
}
const quatConj = q => [q[0], -q[1], -q[2], -q[3]];
const quatRotate = (q, v) =>                  // v′ = q·(0,v)·q*
  quatMul(quatMul(q, [0, ...v]), quatConj(q)).slice(1);
```

Тесты: `quatRotate(qFromAxisAngle([0,1,0], 90°), [1,0,0])` → (0, 0, −1) — совпало с примером 4 ✓; композиция q⁴ (четыре умножения) применена к произвольному вектору (0.3, 0.7, −0.2) — вернулся без изменений (360° = ничего) ✓. Заметь: сам q⁴ равен (−1, 0, 0, 0), а не (1, 0, 0, 0), — но это тот же поворот-тождество (double cover в действии).

**Вариант 2 (оптимизированная формула вместо двух умножений).** Сэндвич буквально следует определению, но делает лишнюю работу: он честно перемножает два кватерниона целиком, включая скалярную часть, о которой заранее известно, что она ноль. В библиотеках (glm, DirectXMath, движковые математики) `quatRotate` пишут иначе:

```js
function quatRotateFast(q, v) {          // q = [w, x, y, z], ‖q‖ = 1
  const [w, ...u] = q;                   // u — векторная часть
  const t = scale(cross(u, v), 2);       // t = 2·(u × v)
  return add(add(v, scale(t, w)), cross(u, t));   // v′ = v + w·t + u × t
}
```

Формула не взята с потолка — это **Родриг, переписанный в половинных углах**. Пусть q кодирует поворот на θ вокруг единичной n, то есть w = cos(θ/2), u = n·sin(θ/2). Тогда:

- t = 2(u × v) = 2 sin(θ/2)·(n × v), поэтому w·t = 2 sin(θ/2)cos(θ/2)·(n × v) = **sin θ·(n × v)** — в точности второе слагаемое Родрига;
- u × t = 2 sin²(θ/2)·n×(n×v) = **(1 − cos θ)·n×(n×v)**, а двойное векторное произведение здесь берётся тем же разложением, что в «Интуиции вывода» § 10.2.2: n×(n×v) = n×(n×v⊥) = −v⊥ = **n(n·v) − v** (cross с единичной n поворачивает v⊥ на 90° в его плоскости, два таких поворота дают −v⊥).

Складываем и получаем ровно формулу § 10.2.2:

```
v + w·t + u×t = v + sin θ·(n×v) + (1 − cos θ)·(n(n·v) − v)
              = v·cos θ + (n×v)·sin θ + n(n·v)(1 − cos θ)   ∎
```

(код сверил и каждое слагаемое по отдельности, и итог — на четырёх парах ось/угол, включая полуоборот; ассерты «быстрая форма = сэндвич» добавлены в `tests/verify_ch10_12.js`.)
Цена: **18 умножений и 12 сложений** против 32 и 24 у пары `quatMul` — вдвое дешевле, и по-прежнему ни одного вызова тригонометрии (все sin и cos уже зашиты в компоненты q).
*Сравнение.* Сэндвич — определение: его пишут первым, по нему проверяют всё остальное, и он незаменим, когда «внутри» стоит не вектор, а настоящий кватернион (например, перенос приращения из локальных осей в мировые: q_мир = q_ор·q_лок·q_ор*). Быстрая форма — реализация: она применима только к чистому вектору и только к **единичному** q. Причём нарушение этого предусловия не смягчается, как у сэндвича: для q = (2, 0, 2, 0) (‖q‖² = 8) сэндвич даёт (0, 0, −8) — правильный поворот с честным масштабом ‖q‖², его хотя бы можно поделить; быстрая форма возвращает (−7, 0, −8) — не поворот и не подобие вообще (код). Отсюда рабочее правило, общее для всех таких оптимизаций: определение — эталон для тестов, оптимизированная запись — для рантайма, и первое обязано подтверждать второе на каждом прогоне (что и делает добавленный ассерт).

### Г10.Ч4.З3

**Условие.** q₁ = (1, 0, 0, 0), q₂ = (−√2/2, 0, −√2/2, 0). Каков dot? Какой поворот кодирует q₂? Что сделает наивный slerp без поправки знака?

**Решение.** dot(q₁, q₂) = −√2/2 ≈ −0.707 < 0 — сигнал double cover.
q₂ = −(√2/2, 0, √2/2, 0): по знаку это «дальняя» запись того же поворота — **90° вокруг y** (код подтвердил: qToMat4(q₂) = R_y(90°) поэлементно). Формально из рецепта: cos(θ/2) = −√2/2 ⟹ θ = 270° вокруг −y — что геометрически тот же поворот.
Наивный slerp: cos Ω = −0.707 ⟹ Ω = 135° — интерполяция пойдёт по дуге **270°-ного** пути (через «всю сферу»): персонаж сделает эффектный лишний оборот. Поправка: заметив dot < 0, заменить q₂ на −q₂ (dot станет +0.707, Ω = 45°) — короткая дуга.

**Ответ:** dot ≈ −0.707; тот же поворот 90° вокруг y в «дальней» записи; без поправки — путь в 270° вместо 90°.

### Г10.Ч4.З4

**Условие.** Выведи из формулы slerp значения при t = 0 и t = 1 и покажи, что при Ω → 0 slerp → lerp.

**Решение.** t = 0: sin(Ω)/sin(Ω)·q₁ + sin(0)/sin(Ω)·q₂ = q₁ ✓. t = 1: симметрично q₂ ✓ — граничные условия интерполяции выполнены.
Ω → 0: sin x ≈ x для малых x ⟹ sin((1−t)Ω)/sin Ω ≈ (1−t)Ω/Ω = 1−t и sin(tΩ)/sin Ω ≈ t:

```
slerp → (1−t)·q₁ + t·q₂ = lerp  ∎
```

Отсюда легальность библиотечного трюка «при cos Ω ≈ 1 подменяй slerp на nlerp»: это не приближение, а предел самой формулы (плюс уход от деления на sin Ω ≈ 0 — численная гигиена, гл. 2).

### Г10.Ч4.З5

**Условие.** Камера должна повернуться из A в B (между ними 120°) за 2 секунды с постоянной скоростью. Напиши формулу ориентации в момент t и укажи, почему nlerp здесь даст видимую неравномерность, а на 60-fps-интерполяции между соседними кадрами анимации — нет.

**Решение.** Формула (с предварительной поправкой знака: если dot(A, B) < 0, взять −B):

```
q(t) = slerp(A, B, t / 2),  t ∈ [0, 2]
```

slerp гарантирует постоянную **угловую** скорость (120°/2 с = 60°/с) — камера крутится равномерно.
Почему не nlerp: на дуге 120° (Ω = 60° по кватерниону) nlerp заметно ускоряется в середине (нормализованная хорда проходит середину дуги быстрее краёв) — глаз ловит «рывок разгона-торможения» у плавного двухсекундного панорамирования.
Почему nlerp хорош между кадрами анимации: там интерполируются соседние ключи с дугой порядка 60°/с ÷ 60 кадров ≈ 1–2° — на таких углах nlerp и slerp совпадают до долей процента (З4: предел Ω → 0), а nlerp в разы дешевле — и его гоняют на тысячах костей.

**Ответ:** slerp(A, B, t/2) с поправкой знака; неравномерность nlerp пропорциональна дуге — 120° видно, 2° нет.

### Г10.Ч4.З6

**Условие.** Реализуй `nlerp(q1, q2, t)` c поправкой знака и прогони: identity → поворот 170° вокруг y, t = 0.5. Проверь, что результат — поворот на ~85°, а не на ~95° «в другую сторону» (пересчитай угол из θ = 2·acos(w)).

**Решение (код выполнен).**

```js
function nlerp(q1, q2, t) {
  if (dot4(q1, q2) < 0) q2 = q2.map(x => -x);   // поправка double cover
  return qNormalize(q1.map((x, i) => x * (1 - t) + q2[i] * t));
}
```

Прогон: q₂ = (cos 85°, 0, sin 85°, 0); dot(q₁, q₂) = cos 85° ≈ 0.087 > 0 — поправка не потребовалась (дуга и так короткая). Результат nlerp(0.5): θ = 2·acos(w) = **85.000°** (код подтвердил; в середине nlerp = slerp — З2, потому ровно половина от 170°). Ось — y, направление то же.
Для полноты: поправка сработала бы, если бы q₂ пришёл в «дальней» записи (−cos 85°, 0, −sin 85°, 0) — без неё получилось бы ~95° в противоположную сторону (интерполяция к −q₂ по дуге 190°).

**Ответ:** 85° вокруг y; поправка знака превращает «длинный путь» в короткий.

**Вариант 2 (тот же прогон честным slerp).** Ответ «85°, а не 95°» — утверждение о **пути** интерполяции, и надёжнее всего проверить его эталоном: slerp (§ 10.4.3), у которого равномерность угловой скорости заложена в построение.

```js
function slerp(q1, q2, t) {
  let d = dot4(q1, q2);
  if (d < 0) { q2 = q2.map(x => -x); d = -d; }      // та же поправка знака
  if (d > 0.9995) return nlerp(q1, q2, t);          // защита Ω ≈ 0 (§ 10.4.3)
  const O = Math.acos(d), s = Math.sin(O);
  return q1.map((x, i) => (Math.sin((1 - t) * O) * x + Math.sin(t * O) * q2[i]) / s);
}
```

Прогон того же случая (identity → 170° вокруг y): dot = cos 85° ⟹ Ω = 85°, slerp(0.5) даёт θ = 2·acos(w) = **85.000°**, причём покомпонентно тот же кватернион, что и nlerp (в середине они совпадают — Г10.Ч4.З2). Вывод варианта 1 подтверждён независимым методом.
Интереснее посмотреть, где методы **расходятся**, — для этого достаточно сойти с середины. Угол поворота в момент t (код):

| Дуга | t | slerp | nlerp | разница |
|---|---|---|---|---|
| 170° | 0.25 | 42.500° | 35.769° | −6.731° |
| 120° | 0.25 | 30.000° | 27.796° | −2.204° |
| 60° | 0.25 | 15.000° | 14.739° | −0.261° |
| 10° | 0.25 | 2.500° | 2.499° | −0.001° |
| 2° | 0.25 | 0.500° | 0.500° | −0.00001° |

Таблица читается сразу. Slerp строго линеен по углу: t = 0.25 даёт ровно четверть дуги, чему бы та ни равнялась. Nlerp в первой половине пути отстаёт (а во второй навёрстывает — при t = 0.75 знак разницы меняется) — это и есть «неравномерная скорость» § 10.4.2, теперь в числах. И видно, как стремительно разница тает с укорочением дуги: 6.7° на 170°, но уже 0.001° на 10° — обоснование Г10.Ч4.З5 «120° видно, 2° нет» перестало быть качественным.
Защита Ω ≈ 0 — тоже не паранойя: со снятой веткой `d > 0.9995` дуга в 10⁻⁷ градуса даёт деление вида 0/0 и NaN во всех четырёх компонентах (код), тогда как с защитой на той же паре возвращается корректный кватернион.
*Сравнение.* Nlerp дешевле (сложение, умножение, нормализация — ни одного acos или sin) и не имеет вырожденных случаев вовсе. Slerp дороже, требует ветки при Ω ≈ 0, но даёт постоянную угловую скорость на любой дуге. Главный практический вывод — из таблицы: выбор между ними определяется **длиной дуги**, а не вкусом. На соседних кадрах анимации (доли градуса) разница лежит ниже точности float, и nlerp безальтернативен по цене; на длинных доворотах камеры (десятки и сотни градусов) она вырастает до единиц градусов, то есть видна глазом как рывок разгона-торможения. Ровно поэтому библиотеки держат обе функции — и внутри slerp честно вызывают nlerp, когда дуга схлопывается: это не хак, а предел самой формулы (Г10.Ч4.З4).

### Г10.Ч4.З7

**Условие.** «`rot = slerp(rot, target, 0.1)` каждый кадр, БЕЗ нормализации — slerp же возвращает единичный кватернион». Формально верно, но после тысяч кадров масштаб персонажа дрожит. Откуда грязь и какая однострочная профилактика?

**Решение.** slerp возвращает единичный кватернион **в идеальной арифметике**. В float каждая операция (sin, acos, деления, сложения) вносит ~10⁻⁷–10⁻⁸ относительной ошибки; итеративная схема `rot = f(rot, …)` **накапливает** её в самом хранимом значении — за десятки тысяч кадров норма уползает на заметные величины. Неединичный q, пропущенный через матричную формулу § 10.3.3, даёт матрицу с масштабом ‖q‖² — персонаж «дышит» размером.
Профилактика:

```js
rot = qNormalize(slerp(rot, target, 0.1));   // 4 умножения + sqrt — копейки
```

(та же логика, что периодическая ре-ортогонализация матриц в Г6.Ч3.З9 — только дешевле на порядок; это и есть преимущество из таблицы § 10.3.3).

### Г10.Ч4.З8

**Условие.** Скелет из 3 костей, 30 ключевых кадров в секунду, рендер — 144 fps. Опиши конвейер интерполяции: что в ключах, что вычисляется на кадр рендера, где nlerp/slerp и где конверсия в матрицы.

**Решение.** Ключи (30/с) хранят **позы**: на кость — кватернион ориентации + вектор позиции (+ масштаб при необходимости). Кадры рендера (144/с) почти никогда не совпадают с ключами: на интервал между ключами приходится 144/30 = 4.8 кадра рендера.
Конвейер на каждый кадр рендера:
1. По времени анимации найти соседние ключи k, k+1 и параметр t ∈ [0, 1] внутри интервала.
2. Для каждой из 3 костей: ориентация = **nlerp**(q_k, q_{k+1}, t) с поправкой знака (дуги между соседними ключами крошечные — nlerp неотличим от slerp и дешевле; slerp приберечь для камеры/корня, если дуги велики); позиция = lerp.
3. Из (q, pos) каждой кости собрать локальную матрицу: R(q) из § 10.3.3 + перенос (гл. 7).
4. Пройти иерархию: M_кости_в_мире = M_родителя · M_локальная (гл. 8) — три матрицы палитры скиннинга.
5. Отдать палитру в вершинный шейдер (GPU понимает матрицы — гл. 12).
Кватернионы живут в пунктах 1–2 (хранение и смешивание), матрицы — в 3–5 (иерархия и рендер): «храним и анимируем кватернионами, рендерим матрицами» (§ 10.3.3).
