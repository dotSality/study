# Решения. Глава 10: Повороты в 3D — углы Эйлера и кватернионы

> Сборник решений самоучителя «Линейная алгебра для 3D-разработки». Задания — в `book/chapters/10-rotations-quaternions.md`. Все выкладки перепроверены кодом (матрицы, Родриг, кватернионная алгебра, интерполяции). Соглашение композиции углов: R = R_y(yaw)·R_x(pitch)·R_z(roll), вектор-столбец.

## Часть 1. Углы Эйлера и их пределы

### Г10.Ч1.З1

**Условие.** Выпиши, вокруг какой оси крутят yaw, pitch, roll самолёта. Какая тройка углов у самолёта, летящего носом строго вверх?

**Решение.** yaw (рыскание) — вокруг **вертикальной** оси (y): «нос влево-вправо»; pitch (тангаж) — вокруг **поперечной** оси (x, вдоль крыльев): «нос вверх-вниз»; roll (крен) — вокруг **продольной** оси (z, нос–хвост): «покачивание крыльями».
Нос строго вверх: pitch = ±90° (у нашего соглашения — +90°). Значения yaw и roll при этом **не определены однозначно**: это поза шарнирного замка, годится любая пара с одинаковой разностью (см. З4), например (0°, 90°, 0°) или (30°, 90°, 30°).

**Ответ:** вертикальная / поперечная / продольная оси; тройка (α, 90°, β) — целое семейство с фиксированным α − β.

### Г10.Ч1.З2

**Условие.** Вычисли R_y(90°)·R_x(90°)·(1, 0, 0) и R_x(90°)·R_y(90°)·(1, 0, 0). Сверь с экспериментом «с телефоном».

**Решение (код подтвердил).**
*Первый порядок (сначала pitch, потом yaw — читаем справа налево):* R_x(90°)·(1, 0, 0) = (1, 0, 0) — вектор на оси вращения, не сдвинулся; R_y(90°)·(1, 0, 0) = (0, 0, −1). Итог: **(0, 0, −1)**.
*Второй:* R_y(90°)·(1, 0, 0) = (0, 0, −1); R_x(90°)·(0, 0, −1) = (0, 1, 0). Итог: **(0, 1, 0)**.
Разные результаты — повороты вокруг разных осей не коммутируют; телефон в руках показывает то же самое.

**Ответ:** (0, 0, −1) и (0, 1, 0).

### Г10.Ч1.З3

**Условие.** Вычисли R_x(90°)·(0, 0, 1) и R_x(90°)·(0, 1, 0) и опиши, куда легли локальные оси после pitch = 90°.

**Решение (код подтвердил).** R_x(90°)·(0, 0, 1) = **(0, −1, 0)**; R_x(90°)·(0, 1, 0) = **(0, 0, 1)**.
После pitch = 90° локальная ось z (ось roll) легла на **−y** — на ту же прямую, что мировая ось yaw (y). Две «ручки» из трёх крутят теперь вокруг одной прямой — вот анатомия замка из § 10.1.2.

### Г10.Ч1.З4

**Условие.** Перемножь символьно R_y(α)·R_x(90°)·R_z(β) и покажи, что результат зависит только от α ± β. Сколько различных ориентаций достижимо из этой позы двумя «ручками»?

**Решение.** Шаг 1: R_x(90°) = [[1,0,0],[0,0,−1],[0,1,0]]. Умножаем на R_z(β) (строки R_x комбинируют строки R_z):

```
R_x(90°)·R_z(β) = ⎡cos β  −sin β   0⎤
                  ⎢  0       0    −1⎥
                  ⎣sin β   cos β   0⎦
```

Шаг 2: слева R_y(α) (строки (cos α, 0, sin α), (0,1,0), (−sin α, 0, cos α)):

```
                     ⎡cos α·cos β + sin α·sin β    −cos α·sin β + sin α·cos β    0⎤   ⎡ cos(α−β)   sin(α−β)   0⎤
R_y(α)R_x(90°)R_z(β) = ⎢          0                            0                −1⎥ = ⎢    0          0      −1⎥
                     ⎣−sin α·cos β + cos α·sin β    sin α·sin β + cos α·cos β    0⎦   ⎣−sin(α−β)   cos(α−β)   0⎦
```

(свёрнуто по формулам косинуса/синуса разности). Матрица зависит **только от α − β** ∎ (код проверил: тройки (10°, 90°, 0°) и (30°, 90°, 20°) дали одну матрицу поэлементно).
Достижимо: однопараметрическое семейство ориентаций (окружность) — вместо двумерного множества, которое дали бы две независимые ручки. Одна степень свободы потеряна — это и есть замок в алгебре.

**Ответ:** зависимость только от α − β; достижима лишь «окружность» ориентаций.

### Г10.Ч1.З5

**Условие.** FPS-камера: yaw без ограничений, pitch клампится в [−89°, 89°]. Объясни каждую цифру.

**Решение.** *Yaw без ограничений:* вращение вокруг вертикали не приближает к замку (замок — свойство pitch) и физически осмысленно на любые углы — игрок оборачивается сколько угодно.
*Pitch ±89°, а не ±90°:* на ±90° (взгляд строго вверх/вниз) наступает шарнирный замок (З3–З4) и, что практичнее, вырождается look-at: forward ∥ up, cross(f, up) ≈ 0 — NaN (Г2.Ч4.З10, гл. 8, § 8.2.3). Градус запаса держит систему на безопасном расстоянии и от алгебраической беды, и от численной. *Почему не ±89.9°:* можно и так; 89° — традиционный запас с учётом float-шума и интерполяций.

### Г10.Ч1.З6

**Условие.** «`euler.y += input * dt; rotation = fromEuler(euler)` — безопасно; `rotation = fromEuler(rotationToEuler(rotation) + delta)` — иногда дёргается». В чём разница?

**Решение.** *Первый паттерн:* углы — **единственный источник истины**; конверсия euler → rotation односторонняя, выполняется заново от полного значения угла. Ошибки не накапливаются, неоднозначность обратной конверсии не участвует.
*Второй:* цикл rotation → euler → rotation. Обратная конверсия rotationToEuler **неоднозначна**: у полюса (pitch ≈ ±90°) одной ориентации соответствует целое семейство троек (З4), и библиотека выбирает представителя по внутренним правилам — от кадра к кадру выбор может скакать (yaw 0° → 179°, roll компенсирует). Прибавка delta к «скакнувшей» тройке даёт видимый рывок. Плюс накопление float-ошибок двойной конверсии.
Правило (из «типичных ошибок» § 10.1.2): хранить ориентацию в кватернионе/матрице, углы — только как входной интерфейс, конверсию «в углы» не зацикливать.

### Г10.Ч1.З7

**Условие.** В редакторе объект показывает углы (0°, 90°, 0°), художник вводит (10°, 90°, 10°) — объект не шевелится визуально. Объясни через пример 2, почему две разные тройки дают одну ориентацию, и почему у полюса инспектор «сходит с ума».

**Решение.** При pitch = 90° матрица зависит только от разности yaw − roll (З4). У обеих троек она одинакова: 0° − 0° = 10° − 10° = 0 ⟹ **одна и та же ориентация** (код подтвердил поэлементным сравнением матриц). Прибавка +10° к yaw в точности скомпенсирована +10° к roll: обе ручки у полюса крутят вокруг одной прямой в противоположные (с учётом знаков сцепки) стороны.
«Сходит с ума» инспектор потому, что обратная задача «матрица → тройка углов» у полюса имеет бесконечно много ответов: редактор вынужден выбирать произвольного представителя семейства, и после любой операции (гизмо, анимация) может показать другую тройку той же ориентации — числа прыгают, объект стоит.

**Ответ:** у полюса ориентация определяется только yaw − roll; тройки с равной разностью неотличимы, а разложение в углы неоднозначно.

## Часть 2. Ось–угол и матрицы поворота

### Г10.Ч2.З1

**Условие.** По формуле Родрига поверни v = (0, 0, 1) вокруг n = (0, 1, 0) на 90° и на 180°. Сверь первый ответ с R_y из гл. 7.

**Решение (код подтвердил).** Слагаемые: n·v = 0 (третье слагаемое нулевое); n × v = (0,1,0) × (0,0,1) = (1, 0, 0).
*90°:* v·cos 90° + (n×v)·sin 90° = 0 + (1, 0, 0) = **(1, 0, 0)**. Сверка: R_y(90°)·(0, 0, 1) = (sin 90°, 0, cos 90°) = (1, 0, 0) ✓.
*180°:* v·(−1) + (n×v)·0 = **(0, 0, −1)** — развернулся, как и положено полуобороту.

**Ответ:** (1, 0, 0); (0, 0, −1).

### Г10.Ч2.З2

**Условие.** Поверни v = (1, 1, 0) вокруг n = (0, 0, 1) на 90° по Родригу. Затем то же — матрицей R_z(90°). Совпало?

**Решение (код подтвердил).** n·v = 0; n × v = (0,0,1) × (1,1,0) = (−1, 1, 0).
Родриг: v·0 + (−1, 1, 0)·1 + 0 = **(−1, 1, 0)**.
Матрицей: R_z(90°)·(1, 1, 0) = (1·0 − 1·1, 1·1 + 1·0, 0) = (−1, 1, 0) ✓ — два формализма, один ответ.

**Ответ:** (−1, 1, 0), совпало.

### Г10.Ч2.З3

**Условие.** Поверни v = (1, 0, 0) вокруг диагональной оси n = (1, 1, 1)/√3 на 120°.

**Решение (код подтвердил).** cos 120° = −1/2; sin 120° = √3/2; 1 − cos = 3/2.
- Слагаемое 1: v·cos = (−1/2, 0, 0).
- Слагаемое 2: n × v = (1,1,1)/√3 × (1,0,0) = (0, 1, −1)/√3; умножить на √3/2: (0, 1/2, −1/2).
- Слагаемое 3: n·v = 1/√3; n·(1/√3)·(3/2) = (1,1,1)·(1/2)/1 = (1/2, 1/2, 1/2).
Сумма: (−1/2 + 1/2, 1/2 + 1/2, −1/2 + 1/2) = **(0, 1, 0)**.
Красота, как и обещано: поворот на 120° вокруг диагонали куба циклически переставляет оси — x перешёл в y (y перейдёт в z, z в x).

**Ответ:** (0, 1, 0).

### Г10.Ч2.З4

**Условие.** Проверь «паспортные данные» матрицы R_x(90°): RᵀR = I и det = 1 — явным вычислением.

**Решение.** R = [[1, 0, 0], [0, 0, −1], [0, 1, 0]]; Rᵀ = [[1, 0, 0], [0, 0, 1], [0, −1, 0]].
RᵀR: диагональ — dot-ы столбцов R с собой: 1, 0+1, 1+0 = 1, 1, 1; внедиагональ — dot-ы разных столбцов: все нули (проверь: столбцы (1,0,0), (0,0,1), (0,−1,0) попарно перпендикулярны). RᵀR = I ✓.
det (разложение по первой строке): 1·(0·0 − (−1)·1) = 1 ✓ — поворот, не зеркало.

### Г10.Ч2.З5

**Условие.** M = [[0, 0, 1], [1, 0, 0], [0, 1, 0]]. Докажи, что это поворот (ортогональность + det). Найди его ось (Mv = v) и угол.

**Решение.** Столбцы (0, 1, 0), (0, 0, 1), (1, 0, 0) — единичные, попарно перпендикулярные ⟹ ортогональна. det: перестановочная матрица цикла x→y→z→x — чётная перестановка, det = +1 (или разложением: 0 − 0 + 1·(1·1 − 0·0) = 1) ⟹ **поворот**.
*Ось:* решаем (M − I)v = 0: M·(x, y, z) = (z, x, y) = (x, y, z) ⟹ z = x, x = y, y = z ⟹ x = y = z. Ось — **(1, 1, 1)/√3** (код подтвердил M·(1,1,1) = (1,1,1)).
*Угол:* M переводит e₁ → e₂ (x в y) — а из З3 мы знаем: ровно это делает поворот на **120°** вокруг диагонали. (Контроль: след M = 0 = 1 + 2cos θ ⟹ cos θ = −1/2 ⟹ θ = 120° ✓.)

**Ответ:** поворот вокруг (1, 1, 1)/√3 на 120°.

### Г10.Ч2.З6

**Условие.** Выведи из формулы Родрига матрицу поворота вокруг n = (0, 1, 0) (подставь образы e₁, e₂, e₃ как столбцы) и убедись, что получилась R_y(θ).

**Решение.** Подставляем в v′ = v cos θ + (n×v) sin θ + n(n·v)(1 − cos θ):
- e₁ = (1,0,0): n×e₁ = (0,0,−1); n·e₁ = 0 ⟹ e₁′ = (cos θ, 0, −sin θ);
- e₂ = (0,1,0): n×e₂ = 0; n·e₂ = 1 ⟹ e₂′ = e₂ cos θ + n(1 − cos θ) = (0, 1, 0) — ось на месте;
- e₃ = (0,0,1): n×e₃ = (1,0,0); n·e₃ = 0 ⟹ e₃′ = (sin θ, 0, cos θ).
Столбцами:

```
⎡ cos θ   0   sin θ⎤
⎢   0     1     0  ⎥  =  R_y(θ)  ✓   (гл. 7, § 7.3.1 — включая «перевёрнутые» синусы)
⎣−sin θ   0   cos θ⎦
```

(код сверил Родрига и R_y на θ = 37° — совпадение поэлементное). Родриг — «родитель» всех трёх осевых матриц гл. 7.

### Г10.Ч2.З7

**Условие.** Физдвижок хранит ω = (0, 2, 0) рад/с. За кадр dt = 0.016 с тело надо довернуть. Запиши приращение в форме ось–угол и объясни, почему «повернуть ориентацию» — это R_приращения · R_старая, а не наоборот.

**Решение.** Ось — направление ω: n = (0, 1, 0); угол — «скорость × время»: θ = ‖ω‖·dt = 2·0.016 = **0.032 рад ≈ 1.83°**. Приращение: (n, θ) = ((0, 1, 0), 0.032 рад).
Порядок: ω задана в **мировых** осях (так принято в физдвижках), а матрица, применяемая к вектору последней (левая в цепочке, гл. 5), действует в мировом пространстве: R_new = R_Δ · R_old — «к уже повёрнутому телу применить мировой доворот». Если бы ω была в осях самого тела (локальная угловая скорость), приращение вставало бы справа: R_old · R_Δ. Тот же выбор «мир слева, локаль справа», что у иерархий гл. 8.

**Ответ:** ось (0, 1, 0), угол 0.032 рад; мировое приращение умножается слева.

### Г10.Ч2.З8

**Условие.** Найди ошибку: `rotateAroundAxis(v, axis, θ)` с axis = (0, 2, 0) поворачивает правильно по направлению, но вектор удлиняется. Какое предусловие нарушено и в каком слагаемом Родрига вылезает лишний множитель?

**Решение.** Нарушено предусловие **‖n‖ = 1**. С n = (0, 2, 0) (длина 2):
- слагаемое (n × v)·sin θ получает лишний множитель 2 (cross линеен по аргументу);
- слагаемое n·(n·v)·(1 − cos θ) — лишний множитель **4** (n входит дважды: и в dot, и множителем).
Слагаемые складываются в неправильных пропорциях: направление при оси вдоль координатной оси может выглядеть терпимо, но длина плывёт (и для наклонных осей направление тоже врёт). Починка: нормализовать axis на входе (или assert).

**Ответ:** ненормированная ось; линейный множитель во втором слагаемом и квадратичный — в третьем.

## Часть 3. Кватернионы

### Г10.Ч3.З1

**Условие.** Составь кватернионы поворотов: 180° вокруг y; 90° вокруг x; 120° вокруг (1, 1, 1)/√3. Проверь единичность.

**Решение (код подтвердил единичность всех трёх).** Рецепт q = (cos(θ/2), n·sin(θ/2)):
- 180° вокруг y: (cos 90°, (0, 1, 0)·sin 90°) = **(0, 0, 1, 0)** — чисто «мнимый» кватернион: полуобороты имеют w = 0;
- 90° вокруг x: (cos 45°, sin 45°, 0, 0) = **(√2/2, √2/2, 0, 0)**;
- 120° вокруг (1,1,1)/√3: (cos 60°, (1,1,1)/√3 · sin 60°) = (1/2, (1,1,1)/√3 · √3/2) = **(1/2, 1/2, 1/2, 1/2)** — знаменитый «кватернион перестановки осей» (ср. З5 части 2).

### Г10.Ч3.З2

**Условие.** Перемножь: (0, (1, 0, 0))·(0, (0, 1, 0)) и в обратном порядке. Что дало dot-слагаемое, что cross?

**Решение (код подтвердил).** Формула: (w₁w₂ − v₁·v₂, w₁v₂ + w₂v₁ + v₁×v₂). Здесь w₁ = w₂ = 0:
- скаляр: −(1,0,0)·(0,1,0) = 0 (векторы перпендикулярны — dot промолчал);
- вектор: 0 + 0 + (1,0,0)×(0,1,0) = (0, 0, 1). Итого **(0, 0, 0, 1)**.
Обратный порядок: cross сменит знак: **(0, 0, 0, −1)**.
Это классические i·j = k и j·i = −k Гамильтона: некоммутативность кватернионов — в точности антикоммутативность cross (гл. 2).

### Г10.Ч3.З3

**Условие.** Прогони расчёт примера 4 для v = (0, 0, 1) (та же q — 90° вокруг y). Сверь с R_y(90°)·(0, 0, 1).

**Решение.** q = (√2/2, (0, √2/2, 0)); v как кватернион: (0, (0, 0, 1)).
*Шаг 1:* q·(0, v): скаляр = −(0, √2/2, 0)·(0, 0, 1) = 0; вектор = √2/2·(0,0,1) + (0,√2/2,0)×(0,0,1) = (0, 0, √2/2) + (√2/2, 0, 0) = (√2/2, 0, √2/2). Итог: (0, (√2/2, 0, √2/2)).
*Шаг 2:* умножаем на q* = (√2/2, (0, −√2/2, 0)): скаляр = 0 − (√2/2, 0, √2/2)·(0, −√2/2, 0) = 0; вектор = √2/2·(√2/2, 0, √2/2) + (√2/2, 0, √2/2)×(0, −√2/2, 0) = (1/2, 0, 1/2) + (1/2, 0, −1/2) = **(1, 0, 0)**.
Сверка: R_y(90°)·(0, 0, 1) = (1, 0, 0) ✓ (код подтвердил оба пути).

**Ответ:** (1, 0, 0).

### Г10.Ч3.З4

**Условие.** Проверь для q из примера 4: q·q* = (1, (0, 0, 0)). Почему для единичных кватернионов сопряжение = обращение?

**Решение.** Общий расчёт: q·q* = (w, v)·(w, −v) = (w² − v·(−v), w(−v) + wv + v×(−v)) = (w² + ‖v‖², 0) = (‖q‖², 0).
Для q из примера 4: ‖q‖² = 1/2 + 1/2 = 1 ⟹ q·q* = **(1, 0, 0, 0)** — единица кватернионной алгебры (умножение на неё ничего не меняет; код подтвердил).
Отсюда: если ‖q‖ = 1, то q·q* = 1 ⟹ q* и есть обратный q⁻¹ (аналогично q*·q = 1). Полная аналогия с R⁻¹ = Rᵀ у ортогональных матриц: у «правильных» представлений поворотов обращение бесплатно.

### Г10.Ч3.З5

**Условие.** Подставь q = (√2/2, 0, √2/2, 0) в матричную формулу § 10.3.3 и получи R_y(90°). Затем подставь −q и убедись, что матрица та же.

**Решение.** w = y = √2/2, x = z = 0 ⟹ w² = y² = 1/2, wy = 1/2, остальные произведения — нули:

```
⎡1−2(1/2)      0      2(1/2)⎤   ⎡ 0  0  1⎤
⎢    0     1−2(1/2)      0  ⎥ = ⎢ 0  1  0⎥ = R_y(90°) ✓
⎣−2(1/2)       0     1−2(1/2)⎦  ⎣−1  0  0⎦
```

Для −q: в формуле участвуют только **произведения пар** компонент (w², xy, wz, …) — у обеих компонент пары сменился знак, произведение не изменилось ⟹ матрица та же (код подтвердил поэлементно). Это и есть double cover в матричном зеркале: q и −q — один поворот.

### Г10.Ч3.З6

**Условие.** q_а — 90° вокруг y, q_б — 90° вокруг x. Вычисли q_б·q_а и примени к (1, 0, 0). Сверь с Г10.Ч1.З2.

**Решение (код подтвердил).** q_а = (√2/2, 0, √2/2, 0), q_б = (√2/2, √2/2, 0, 0).
q_б·q_а: скаляр = 1/2 − (√2/2,0,0)·(0,√2/2,0) = 1/2 − 0 = 1/2; вектор = √2/2·(0,√2/2,0) + √2/2·(√2/2,0,0) + (√2/2,0,0)×(0,√2/2,0) = (0,1/2,0) + (1/2,0,0) + (0,0,1/2) = (1/2, 1/2, 1/2).
Итого q = **(1/2, 1/2, 1/2, 1/2)** — поворот на 120° вокруг (1,1,1)/√3 (З1)! Композиция двух четверть-оборотов вокруг разных осей — треть-оборот вокруг диагонали: неожиданно и типично для 3D.
Применение к (1, 0, 0): перестановка осей x→y (З5 части 2) ⟹ **(0, 1, 0)**. Сверка с Г10.Ч1.З2 (матричный путь R_x(90°)·R_y(90°)·(1,0,0)): (0, 1, 0) ✓.

**Ответ:** (1/2, 1/2, 1/2, 1/2); вектор → (0, 1, 0).

### Г10.Ч3.З7

**Условие.** Докажи: норма произведения = произведение норм. Почему произведение единичных кватернионов единично — и почему это спасает от «раздувания» объектов, в отличие от матриц?

**Решение.** Ключевая лемма: (pq)* = q*p* (сопряжение разворачивает порядок — проверяется на формуле умножения). Тогда:

```
‖pq‖² = (pq)(pq)* = p·q·q*·p* = p·‖q‖²·p* = ‖q‖²·(p·p*) = ‖q‖²·‖p‖²  ∎
```

(‖q‖² — скаляр, выносится). Следствие: единичный × единичный = единичный **точно**, в идеальной арифметике композиция никогда не выводит из «множества поворотов».
Практическая разница с матрицами: float-шум и там и там, но у кватерниона «дефект» — одно число (норма), лечится одной нормализацией (4 умножения + корень); у матрицы дефект — потеря ортогональности всех трёх столбцов сразу, лечение — Грам–Шмидт (гл. 6, Г6.Ч3.З9), в разы дороже и грубее. Поэтому длинные цепочки поворотов (анимация, физика) копят кватернионами.

### Г10.Ч3.З8

**Условие.** Найди ошибку: «кватернион (0.5, 0.5, 0.5, 0.5) я построил как поворот на 90° вокруг (1, 1, 1)». Проверь единичность, затем соответствие рецепту (какой угол и какая ось на самом деле?).

**Решение.** Единичность: 4·0.25 = 1 ✓ — кватернион валидный, но не тот.
Обратная задача: w = cos(θ/2) = 0.5 ⟹ θ/2 = 60° ⟹ **θ = 120°** (код: 2·acos(0.5) = 120°). Векторная часть: (0.5, 0.5, 0.5) = n·sin 60° = n·(√3/2) ⟹ n = (1, 1, 1)/√3 ✓ ось верная.
Ошибка — классика половинного угла: для 90° нужно w = cos 45° = √2/2 ≈ 0.707, а не 0.5. Автор, видимо, взял cos(θ)/2 или «просто поровну». Правильный кватернион 90° вокруг (1,1,1)/√3: (0.707, 0.408, 0.408, 0.408).

**Ответ:** это поворот на 120° (не 90°) вокруг (1, 1, 1)/√3; забыт половинный угол.

### Г10.Ч3.З9

**Условие.** Реализуй `quatMul(q1, q2)`, `quatConj(q)`, `quatRotate(q, v)` и воспроизведи пример 4 кодом. Тест: композиция 4 поворотов по 90° вокруг y возвращает вектор на место.

**Решение (код выполнен, оба теста зелёные).**

```js
// q = [w, x, y, z]
function quatMul(q1, q2) {                    // (w₁w₂ − v₁·v₂, w₁v₂ + w₂v₁ + v₁×v₂)
  const [w1, ...v1] = q1, [w2, ...v2] = q2;
  return [w1*w2 - dot(v1, v2),
          ...add(add(scale(v2, w1), scale(v1, w2)), cross(v1, v2))];
}
const quatConj = q => [q[0], -q[1], -q[2], -q[3]];
const quatRotate = (q, v) =>                  // v′ = q·(0,v)·q*
  quatMul(quatMul(q, [0, ...v]), quatConj(q)).slice(1);
```

Тесты: `quatRotate(qFromAxisAngle([0,1,0], 90°), [1,0,0])` → (0, 0, −1) — совпало с примером 4 ✓; композиция q⁴ (четыре умножения) применена к произвольному вектору (0.3, 0.7, −0.2) — вернулся без изменений (360° = ничего) ✓. Заметь: сам q⁴ равен (−1, 0, 0, 0), а не (1, 0, 0, 0), — но это тот же поворот-тождество (double cover в действии).

## Часть 4. Интерполяция поворотов

### Г10.Ч4.З1

**Условие.** Вычисли lerp((1,0,0,0), (0,0,1,0), 0.5) и его норму. Насколько «внутрь сферы» ушла середина? Нормализуй — получи nlerp.

**Решение (код подтвердил).** lerp = (0.5, 0, 0.5, 0); норма = √(0.25 + 0.25) = **√0.5 ≈ 0.707** — хорда просела внутрь сферы на ~29% (интерполируются identity и полуоборот вокруг y — дуга большая, 90° по кватерниону).
nlerp = (0.5, 0, 0.5, 0)/0.707 = **(√2/2, 0, √2/2, 0)** — а это ровно поворот на 90° вокруг y: честная середина между «ничего» (0°) и полуоборотом (180°). Нормализация вернула точку на сферу — и попала в геометрически правильную середину дуги.

**Ответ:** (0.5, 0, 0.5, 0), норма ≈ 0.707; nlerp = (√2/2, 0, √2/2, 0) = поворот 90°.

### Г10.Ч4.З2

**Условие.** Для пары из примера 5 вычисли nlerp при t = 0.5 и сравни с ответом slerp. Насколько близки при Ω = 45°?

**Решение (код подтвердил).** q₁ = (1,0,0,0), q₂ = (√2/2, 0, √2/2, 0). lerp(0.5) = ((1+√2/2)/2, 0, √2/4·2⁻¹… считаем числами: (0.8536, 0, 0.3536, 0); норма 0.9239; nlerp = **(0.9239, 0, 0.3827, 0)** = (cos 22.5°, 0, sin 22.5°, 0).
Это **в точности** ответ slerp из примера 5 — не «близко», а совпадение до машинного эпсилона.
Причина: в середине (t = 0.5) хорда и дуга симметричны — середина хорды нормализуется ровно в середину дуги. Расхождения nlerp/slerp живут на **несерединных** t (nlerp в середине «быстрее»); и растут с Ω.

**Ответ:** (0.9239, 0, 0.3827, 0); при t = 0.5 nlerp = slerp точно (различия — на других t).

### Г10.Ч4.З3

**Условие.** q₁ = (1, 0, 0, 0), q₂ = (−√2/2, 0, −√2/2, 0). Каков dot? Какой поворот кодирует q₂? Что сделает наивный slerp без поправки знака?

**Решение.** dot(q₁, q₂) = −√2/2 ≈ −0.707 < 0 — сигнал double cover.
q₂ = −(√2/2, 0, √2/2, 0): по знаку это «дальняя» запись того же поворота — **90° вокруг y** (код подтвердил: qToMat4(q₂) = R_y(90°) поэлементно). Формально из рецепта: cos(θ/2) = −√2/2 ⟹ θ = 270° вокруг −y — что геометрически тот же поворот.
Наивный slerp: cos Ω = −0.707 ⟹ Ω = 135° — интерполяция пойдёт по дуге **270°-ного** пути (через «всю сферу»): персонаж сделает эффектный лишний оборот. Поправка: заметив dot < 0, заменить q₂ на −q₂ (dot станет +0.707, Ω = 45°) — короткая дуга.

**Ответ:** dot ≈ −0.707; тот же поворот 90° вокруг y в «дальней» записи; без поправки — путь в 270° вместо 90°.

### Г10.Ч4.З4

**Условие.** Выведи из формулы slerp значения при t = 0 и t = 1 и покажи, что при Ω → 0 slerp → lerp.

**Решение.** t = 0: sin(Ω)/sin(Ω)·q₁ + sin(0)/sin(Ω)·q₂ = q₁ ✓. t = 1: симметрично q₂ ✓ — граничные условия интерполяции выполнены.
Ω → 0: sin x ≈ x для малых x ⟹ sin((1−t)Ω)/sin Ω ≈ (1−t)Ω/Ω = 1−t и sin(tΩ)/sin Ω ≈ t:

```
slerp → (1−t)·q₁ + t·q₂ = lerp  ∎
```

Отсюда легальность библиотечного трюка «при cos Ω ≈ 1 подменяй slerp на nlerp»: это не приближение, а предел самой формулы (плюс уход от деления на sin Ω ≈ 0 — численная гигиена, гл. 2).

### Г10.Ч4.З5

**Условие.** Камера должна повернуться из A в B (между ними 120°) за 2 секунды с постоянной скоростью. Напиши формулу ориентации в момент t и укажи, почему nlerp здесь даст видимую неравномерность, а на 60-fps-интерполяции между соседними кадрами анимации — нет.

**Решение.** Формула (с предварительной поправкой знака: если dot(A, B) < 0, взять −B):

```
q(t) = slerp(A, B, t / 2),  t ∈ [0, 2]
```

slerp гарантирует постоянную **угловую** скорость (120°/2 с = 60°/с) — камера крутится равномерно.
Почему не nlerp: на дуге 120° (Ω = 60° по кватерниону) nlerp заметно ускоряется в середине (нормализованная хорда проходит середину дуги быстрее краёв) — глаз ловит «рывок разгона-торможения» у плавного двухсекундного панорамирования.
Почему nlerp хорош между кадрами анимации: там интерполируются соседние ключи с дугой порядка 60°/с ÷ 60 кадров ≈ 1–2° — на таких углах nlerp и slerp совпадают до долей процента (З4: предел Ω → 0), а nlerp в разы дешевле — и его гоняют на тысячах костей.

**Ответ:** slerp(A, B, t/2) с поправкой знака; неравномерность nlerp пропорциональна дуге — 120° видно, 2° нет.

### Г10.Ч4.З6

**Условие.** Реализуй `nlerp(q1, q2, t)` c поправкой знака и прогони: identity → поворот 170° вокруг y, t = 0.5. Проверь, что результат — поворот на ~85°, а не на ~95° «в другую сторону» (пересчитай угол из θ = 2·acos(w)).

**Решение (код выполнен).**

```js
function nlerp(q1, q2, t) {
  if (dot4(q1, q2) < 0) q2 = q2.map(x => -x);   // поправка double cover
  return qNormalize(q1.map((x, i) => x * (1 - t) + q2[i] * t));
}
```

Прогон: q₂ = (cos 85°, 0, sin 85°, 0); dot(q₁, q₂) = cos 85° ≈ 0.087 > 0 — поправка не потребовалась (дуга и так короткая). Результат nlerp(0.5): θ = 2·acos(w) = **85.000°** (код подтвердил; в середине nlerp = slerp — З2, потому ровно половина от 170°). Ось — y, направление то же.
Для полноты: поправка сработала бы, если бы q₂ пришёл в «дальней» записи (−cos 85°, 0, −sin 85°, 0) — без неё получилось бы ~95° в противоположную сторону (интерполяция к −q₂ по дуге 190°).

**Ответ:** 85° вокруг y; поправка знака превращает «длинный путь» в короткий.

### Г10.Ч4.З7

**Условие.** «`rot = slerp(rot, target, 0.1)` каждый кадр, БЕЗ нормализации — slerp же возвращает единичный кватернион». Формально верно, но после тысяч кадров масштаб персонажа дрожит. Откуда грязь и какая однострочная профилактика?

**Решение.** slerp возвращает единичный кватернион **в идеальной арифметике**. В float каждая операция (sin, acos, деления, сложения) вносит ~10⁻⁷–10⁻⁸ относительной ошибки; итеративная схема `rot = f(rot, …)` **накапливает** её в самом хранимом значении — за десятки тысяч кадров норма уползает на заметные величины. Неединичный q, пропущенный через матричную формулу § 10.3.3, даёт матрицу с масштабом ‖q‖² — персонаж «дышит» размером.
Профилактика:

```js
rot = qNormalize(slerp(rot, target, 0.1));   // 4 умножения + sqrt — копейки
```

(та же логика, что периодическая ре-ортогонализация матриц в Г6.Ч3.З9 — только дешевле на порядок; это и есть преимущество из таблицы § 10.3.3).

### Г10.Ч4.З8

**Условие.** Скелет из 3 костей, 30 ключевых кадров в секунду, рендер — 144 fps. Опиши конвейер интерполяции: что в ключах, что вычисляется на кадр рендера, где nlerp/slerp и где конверсия в матрицы.

**Решение.** Ключи (30/с) хранят **позы**: на кость — кватернион ориентации + вектор позиции (+ масштаб при необходимости). Кадры рендера (144/с) почти никогда не совпадают с ключами: на интервал между ключами приходится 144/30 = 4.8 кадра рендера.
Конвейер на каждый кадр рендера:
1. По времени анимации найти соседние ключи k, k+1 и параметр t ∈ [0, 1] внутри интервала.
2. Для каждой из 3 костей: ориентация = **nlerp**(q_k, q_{k+1}, t) с поправкой знака (дуги между соседними ключами крошечные — nlerp неотличим от slerp и дешевле; slerp приберечь для камеры/корня, если дуги велики); позиция = lerp.
3. Из (q, pos) каждой кости собрать локальную матрицу: R(q) из § 10.3.3 + перенос (гл. 7).
4. Пройти иерархию: M_кости_в_мире = M_родителя · M_локальная (гл. 8) — три матрицы палитры скиннинга.
5. Отдать палитру в вершинный шейдер (GPU понимает матрицы — гл. 12).
Кватернионы живут в пунктах 1–2 (хранение и смешивание), матрицы — в 3–5 (иерархия и рендер): «храним и анимируем кватернионами, рендерим матрицами» (§ 10.3.3).
