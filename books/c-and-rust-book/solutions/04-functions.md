## Часть 4.1. Определение и вызов функций

### Г4.Ч1.З1

**Условие.** Напишите функцию `int cube(int x)`, возвращающую `x`³. В `main` напечатайте `cube(2)`, `cube(3)`, `cube(5)`. Проверьте, что получилось `8`, `27`, `125`.

**Решение.** Куб — это `x * x * x`; функция вычисляет его и отдаёт через `return` (§4.1.1). Один раз описав `cube`, вызываем её трижды с разными аргументами прямо внутри `printf` — значение вызова подставляется в строку формата как обычное выражение.

```c
#include <stdio.h>

int cube(int x)
{
    return x * x * x;
}

int main(void)
{
    printf("cube(2) = %d\n", cube(2));
    printf("cube(3) = %d\n", cube(3));
    printf("cube(5) = %d\n", cube(5));
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra cube.c -o cube
./cube
```

```
cube(2) = 8
cube(3) = 27
cube(5) = 125
```

Все три значения совпали с ожидаемыми. Аргумент подставляется на место параметра `x`, тело вычисляет произведение, `return` возвращает его в точку вызова.

**Ответ:** `cube(2) = 8`, `cube(3) = 27`, `cube(5) = 125`. Функция `int cube(int x)` возвращает `x * x * x`; переиспользование достигается тремя вызовами одного описания.

### Г4.Ч1.З2

**Условие.** Напишите функцию `double average(double a, double b)`, возвращающую среднее арифметическое двух чисел. Вызовите её для `average(3.0, 4.0)` и `average(2.5, 7.5)`, напечатайте результаты с `%.2f`.

**Решение.** Среднее двух чисел — это их сумма, делённая пополам. Делим на `2.0` (а не `2`), чтобы деление было вещественным (§2.2.5): при `double`-операндах это не обязательно, но так намерение «делим как дробное» видно явно. Результат печатаем с `%.2f` — два знака после точки.

```c
#include <stdio.h>

double average(double a, double b)
{
    return (a + b) / 2.0;
}

int main(void)
{
    printf("average(3.0, 4.0) = %.2f\n", average(3.0, 4.0));
    printf("average(2.5, 7.5) = %.2f\n", average(2.5, 7.5));
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra average.c -o average
./average
```

```
average(3.0, 4.0) = 3.50
average(2.5, 7.5) = 5.00
```

`(3.0 + 4.0) / 2.0 = 3.50`, `(2.5 + 7.5) / 2.0 = 5.00`. Тип параметров и результата — `double`, поэтому дробная часть не теряется.

**Ответ:** `average(3.0, 4.0) = 3.50`, `average(2.5, 7.5) = 5.00`. Функция возвращает `(a + b) / 2.0`; печать с `%.2f` даёт два знака после запятой.

### Г4.Ч1.З3

**Условие.** Напишите функцию `void print_sep(void)`, печатающую строку из 20 символов `=`. Используя её, напечатайте заголовок вида «разделитель / строка `MENU` / разделитель». Убедитесь, что тело строки-разделителя описано в программе один раз.

**Решение.** Разделитель — действие, а не вычисление, поэтому функция имеет тип `void` и пустой список параметров `(void)` (§4.1.1). Описав строку из 20 знаков `=` внутри `print_sep` **один раз**, мы вызываем её дважды — до и после заголовка; сама строка в тексте программы не дублируется.

```c
#include <stdio.h>

void print_sep(void)
{
    printf("====================\n");   // 20 знаков '='
}

int main(void)
{
    print_sep();
    printf("MENU\n");
    print_sep();
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra menu.c -o menu
./menu
```

```
====================
MENU
====================
```

`print_sep()` вызывается как самостоятельная инструкция (возвращать нечего). Одно имя избавило от повторного набора двадцати символов: захотим сменить разделитель — правим в одном месте.

**Ответ:** печатаются разделитель, `MENU` и снова разделитель. Функция `void print_sep(void)` содержит строку `=`-ов единожды, а вызывается дважды — это и есть переиспользование.

**Вариант 2 (разделитель циклом).** Чтобы не отсчитывать двадцать знаков `=` вручную (легко ошибиться на один), ширину можно задать **числом** и напечатать символы циклом (§3.2.2):

```c
#include <stdio.h>

void print_sep(void)
{
    for (int i = 0; i < 20; i++)   // ширина задана одним числом
        printf("=");
    printf("\n");
}

int main(void)
{
    print_sep();
    printf("MENU\n");
    print_sep();
    return 0;
}
```

```
====================
MENU
====================
```

*Сравнение.* Вывод идентичен. Литерал `"===================="` короче и нагляднее, но длину приходится выверять посимвольно, а изменить её (скажем, на 30) — значит переписывать строку и пересчитывать. Вариант с циклом делает ширину явным параметром `20`: одно число легко поменять, и ошибиться в длине невозможно. Для фиксированной небольшой строки годятся оба; как только ширина становится настраиваемой, цикл предпочтительнее.

### Г4.Ч1.З4

**Условие.** Напишите функцию-предикат `int is_positive(int n)`, возвращающую `1`, если `n > 0`, и `0` иначе. В `main` прочитайте число `scanf("%d", &n)` и через `if (is_positive(n))` напечатайте `positive` или `not positive`. Проверьте на `5`, `0`, `-3`.

**Решение.** Предикат возвращает «да/нет» в виде `1`/`0`. Само сравнение `n > 0` уже даёт `1` или `0` (§3.1.1), поэтому его достаточно вернуть напрямую — это идиома C для функций-предикатов. В `main` результат читается прямо в условии `if`.

```c
#include <stdio.h>

int is_positive(int n)
{
    return n > 0;                    // сравнение уже даёт 1 или 0
}

int main(void)
{
    int n;
    if (scanf("%d", &n) != 1) {
        printf("bad input\n");
        return 1;
    }
    if (is_positive(n))
        printf("positive\n");
    else
        printf("not positive\n");
    return 0;
}
```

```
printf '5\n'  | ./positive
printf '0\n'  | ./positive
printf '-3\n' | ./positive
```

```
positive
not positive
not positive
```

`is_positive(5)` вернула `1` → ветвь `positive`; `is_positive(0)` и `is_positive(-3)` вернули `0` → `not positive` (ноль по условию не положителен).

**Ответ:** `5` → `positive`, `0` → `not positive`, `-3` → `not positive`. Предикат `return n > 0;` возвращает `1`/`0`, и вызов `is_positive(n)` читается в `if` как логическое условие.

**Вариант 2 (явный `if`/`else` в функции).** Ту же логику можно записать разветвлённо — это нагляднее показывает контракт «1 при положительном, иначе 0»:

```c
int is_positive(int n)
{
    if (n > 0)
        return 1;
    else
        return 0;
}
```

Вывод программы с такой функцией не меняется (`positive` / `not positive` / `not positive`).

*Сравнение.* Результат одинаков. Форма `return n > 0;` короче и опирается на то, что сравнение в C — это уже значение `1`/`0`; так пишут предикаты опытные программисты. Явный `if`/`else` многословнее, зато не требует помнить, что сравнение даёт число, — он полезен как учебный промежуточный шаг и в случаях, когда в ветвях есть что-то ещё, кроме возврата флага.

### Г4.Ч1.З5

**Условие.** *(понимание)* Не запуская, предскажите вывод и объясните каждую строку через передачу по значению (§4.1.3):

```c
#include <stdio.h>

void f(int n) { n = n * 10; printf("f: %d\n", n); }
int main(void) {
    int k = 3;
    f(k);
    printf("main: %d\n", k);
    return 0;
}
```

**Решение.** При вызове `f(k)` значение `k` (`3`) **копируется** в параметр `n` (§4.1.3): `n` — отдельная локальная переменная функции `f`. Внутри `f` строка `n = n * 10` меняет **копию**: `n` становится `30`, что и печатает `f: 30`. Переменная `k` в `main` при этом не затронута — она осталась `3`, поэтому `main: 3`.

```
f: 30
main: 3
```

Прогон подтверждает предсказание. Ключевой момент: `f` не может изменить `k`, потому что работает не с самим `k`, а с его копией в параметре `n`; изменения копии исчезают вместе с кадром `f` при возврате.

**Ответ:** печатается `f: 30`, затем `main: 3`. Внутри `f` меняется копия (`n` → `30`), а оригинал `k` в `main` остаётся `3` — следствие передачи по значению.

### Г4.Ч1.З6

**Условие.** *(понимание)* Программа вызывает функцию `quadruple` до её определения и без прототипа:

```c
#include <stdio.h>
int main(void) { printf("%d\n", quadruple(5)); return 0; }
int quadruple(int x) { return x * 4; }
```

Скомпилируется ли она под `gcc -std=c17`? Если нет — какое сообщение выдаст компилятор и почему? Приведите **два** разных исправления (перестановкой и прототипом).

**Решение.** Компилятор читает файл сверху вниз и к моменту вызова `quadruple(5)` в `main` ещё **не знает** этой функции: её определение — ниже, прототипа нет (§4.1.2). В современном C (C99 и новее) вызов необъявленной функции — **ошибка**, а не предупреждение.

```
gcc -std=c17 -Wall -Wextra decl.c -o decl
```

```
decl.c: In function 'main':
decl.c:2:33: error: implicit declaration of function 'quadruple' [-Wimplicit-function-declaration]
    2 | int main(void) { printf("%d\n", quadruple(5)); return 0; }
      |                                 ^~~~~~~~~
```

Сборка не проходит — компилятору неоткуда взять контракт функции (сколько параметров, каких типов, что возвращает).

**Исправление 1 — перестановка** (определение выше вызова само служит объявлением):

```c
#include <stdio.h>
int quadruple(int x) { return x * 4; }   // определение выше вызова
int main(void) { printf("%d\n", quadruple(5)); return 0; }
```

**Исправление 2 — прототип** (контракт перед `main`, определение остаётся внизу):

```c
#include <stdio.h>
int quadruple(int x);                    // прототип перед вызовом
int main(void) { printf("%d\n", quadruple(5)); return 0; }
int quadruple(int x) { return x * 4; }
```

Обе версии компилируются и печатают одно и то же:

```
20
```

**Ответ:** не скомпилируется — `error: implicit declaration of function 'quadruple'`, потому что вызов встречается раньше, чем компилятор узнал функцию. Лечится либо перестановкой определения выше `main`, либо прототипом `int quadruple(int x);` перед `main`; оба варианта дают `20`.

### Г4.Ч1.З7

**Условие.** *(понимание)* Дана функция `void swap(int a, int b)` из примера 3 §4.1.3. Объясните, почему после `swap(x, y)` переменные `x` и `y` в `main` не меняются местами. Что печатается «внутри» и что «снаружи»? Какой механизм языка (из ещё не пройденной главы) понадобится, чтобы обмен сработал?

**Решение.** При вызове `swap(x, y)` значения `x` и `y` **копируются** в параметры `a` и `b` (§4.1.3). Функция честно меняет местами `a` и `b` — но это её локальные копии, а не сами `x` и `y`. Поэтому «внутри» копии обменялись, а «снаружи» оригиналы не тронуты.

```c
#include <stdio.h>

void swap(int a, int b)              // НЕ меняет переменные вызывающего
{
    int t = a;
    a = b;
    b = t;
    printf("inside:  a=%d b=%d\n", a, b);
}

int main(void)
{
    int x = 1, y = 2;
    swap(x, y);
    printf("outside: x=%d y=%d\n", x, y);
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra swap.c -o swap
./swap
```

```
inside:  a=2 b=1
outside: x=1 y=2
```

Внутри `a` и `b` действительно поменялись (`a=2 b=1`), но `x` и `y` остались `1` и `2`. Чтобы функция могла изменить **переменные вызывающего**, ей нужно передать их **адреса** — то есть **указатели** (глава 6). Обычный параметр-копия на это неспособен по определению.

**Ответ:** `x` и `y` не меняются, потому что `swap` работает с копиями `a`, `b`, а не с оригиналами. «Внутри» печатается `a=2 b=1`, «снаружи» — `x=1 y=2`. Настоящий обмен требует передать адреса `x` и `y` — механизм указателей из главы 6.

### Г4.Ч1.З8

**Условие.** *(понимание)* В чём разница между этими двумя функциями и когда какая уместна? Приведите для каждой пример корректного использования в `main`.

```c
int  doubled(int x) { return x * 2; }
void print_doubled(int x) { printf("%d\n", x * 2); }
```

**Решение.** `doubled` **возвращает** удвоенное значение — его можно использовать дальше: присвоить, сложить, сравнить, передать в другую функцию. `print_doubled` **печатает** удвоенное значение и ничего не возвращает — результат сразу «ушёл на экран», в программе им больше не воспользоваться. Это разница между «вернуть» и «напечатать» (§4.1.1, типичные ошибки): первое даёт данные для дальнейших вычислений, второе — только вывод.

```c
#include <stdio.h>

int  doubled(int x) { return x * 2; }
void print_doubled(int x) { printf("%d\n", x * 2); }

int main(void)
{
    int r = doubled(21);             // результат можно использовать дальше
    printf("doubled(21) + 1 = %d\n", r + 1);
    print_doubled(21);               // просто печатает — возвращать нечего
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra doubled.c -o doubled
./doubled
```

```
doubled(21) + 1 = 43
42
```

`doubled(21)` вернула `42`, и мы прибавили к нему `1` → `43`: результат участвовал в вычислении. `print_doubled(21)` напечатала `42` и всё — прибавить к её выводу ничего нельзя.

**Ответ:** `doubled` возвращает результат (тип `int`) — годится, когда значение нужно программе для дальнейших действий (`r + 1` → `43`); `print_doubled` (тип `void`) лишь печатает (`42`) и уместна, когда результат нужен только на экране. Если значение понадобится дальше — возвращайте через `return`, а не печатайте внутри.

### Г4.Ч1.З9

**Условие.** *(понимание)* Функция должна вернуть максимум из двух чисел, но автор забыл `return` в одной ветви:

```c
int max2(int a, int b) {
    if (a > b) return a;
    // если a <= b — ничего не возвращаем
}
```

Что не так? Какое предупреждение даст `gcc -Wall`? Почему возвращаемое в этом случае значение непредсказуемо? Исправьте функцию (двумя способами: через `if/else` и через `?:`).

**Решение.** Если `a > b` ложно, выполнение доходит до `}` функции, **не встретив** `return` со значением. Функция объявлена возвращающей `int`, но фактически ничего не возвращает — вызывающий получит мусор. Это неопределённое поведение (глава 13); эталон при `-Wall` предупреждает:

```
gcc -std=c17 -Wall -Wextra maxbug.c -o maxbug
```

```
maxbug.c: In function 'max2':
maxbug.c:6:1: warning: control reaches end of non-void function [-Wreturn-type]
    6 | }
      | ^
```

Запуск на эталонной машине показывает суть проблемы — «максимумом» `3` и `9` объявляется `3`:

```
max2(3, 9) = 3
```

Значение непредсказуемо потому, что при `a <= b` функция возвращает то, что случайно оказалось в регистре/ячейке результата (здесь совпало со значением `a`); на другой машине или при других соседних вызовах это был бы другой мусор. Полагаться на него нельзя.

**Исправление через `if/else`:**

```c
int max2(int a, int b) {
    if (a > b)
        return a;
    else
        return b;
}
```

**Исправление через `?:`:**

```c
int max2(int a, int b) {
    return (a > b) ? a : b;
}
```

Обе версии возвращают значение на любом пути. Проверка:

```
max2(3, 9) = 9
max2(9, 3) = 9
```

**Ответ:** при `a <= b` функция не выполняет `return`, и результат — мусор (UB); `gcc -Wall` выдаёт `warning: control reaches end of non-void function [-Wreturn-type]` (на эталоне сломанная `max2(3, 9)` вернула `3`). Исправления: `if (a > b) return a; else return b;` или `return (a > b) ? a : b;` — обе дают `max2(3, 9) = max2(9, 3) = 9`.

### Г4.Ч1.З10

**Условие.** *(прикладное)* Оформите проверку простоты числа из §3.2.5 как функцию `int is_prime(int n)` (возвращает `1`/`0`, для `n < 2` — `0`). В `main` циклом `while (scanf("%d", &n) == 1)` читайте числа до конца ввода и для каждого печатайте `n: prime` или `n: not prime`. Проверьте на `1 2 17 91 97`.

**Решение.** Логику простоты (§3.2.5) выносим в функцию-предикат `is_prime`: числа меньше 2 сразу не простые (`return 0`), иначе ищем делитель в диапазоне `2..n-1`; нашли — число составное (`return 0`), не нашли — простое (`return 1`). Ранний `return` из цикла заменяет флаг: как только делитель найден, продолжать незачем. В `main` цикл `while (scanf(...) == 1)` читает поток чисел до конца ввода.

```c
#include <stdio.h>

int is_prime(int n)
{
    if (n < 2)
        return 0;
    for (int d = 2; d < n; d++)
        if (n % d == 0)
            return 0;
    return 1;
}

int main(void)
{
    int n;
    while (scanf("%d", &n) == 1) {
        if (is_prime(n))
            printf("%d: prime\n", n);
        else
            printf("%d: not prime\n", n);
    }
    return 0;
}
```

```
printf '1 2 17 91 97\n' | ./primes
```

```
1: not prime
2: prime
17: prime
91: not prime
97: prime
```

`1 < 2` → сразу `not prime`. Для `2` цикл `d = 2; d < 2` не выполняется — делителей нет, `prime`. `17` и `97` простые. `91 = 7·13` — на `d = 7` находится делитель, `not prime`.

**Ответ:** `1: not prime`, `2: prime`, `17: prime`, `91: not prime`, `97: prime`. Предикат `is_prime` инкапсулирует проверку из §3.2.5; ранний `return 0` при первом делителе заменяет флаг из главы 3.

**Вариант 2 (делители до корня).** Перебирать делители до `n - 1` избыточно: у составного `n` есть делитель, не превосходящий `√n`. Условие цикла `d * d <= n` заметно сокращает работу для больших чисел (тот же приём, что в Г3.Ч2.З11):

```c
int is_prime(int n)
{
    if (n < 2)
        return 0;
    for (int d = 2; d * d <= n; d++)   // делители ищем только до корня из n
        if (n % d == 0)
            return 0;
    return 1;
}
```

```
1: not prime
2: prime
17: prime
91: not prime
97: prime
```

*Сравнение.* Результаты для всех пяти чисел совпадают. Вариант `d < n` буквально повторяет определение «нет делителей, кроме 1 и себя», но для простого `97` впустую перебирает все `d` до 96; вариант `d * d <= n` для `97` проверяет лишь `d` до 9. На учебных числах разница незаметна, но именно так проверку на простоту пишут на практике. (Оговорка та же, что в главе 3: при очень больших `n` произведение `d * d` близко к границе `int` — тогда берут условие `d <= n / d`.)

**Решение.** Приём «`% 10` даёт последнюю цифру, `/= 10` её отбрасывает» (§3.2.5) обходит число справа налево. Заводим накопитель `sum = 0` и в цикле, пока `n > 0`, прибавляем очередную цифру. Для `n = 0` цикл не выполняется ни разу, и сумма остаётся `0` — что верно (у нуля «сумма цифр» равна нулю).

```c
#include <stdio.h>

int digit_sum(int n)
{
    int sum = 0;
    while (n > 0) {
        sum += n % 10;
        n /= 10;
    }
    return sum;
}

int main(void)
{
    printf("digit_sum(1234) = %d\n", digit_sum(1234));
    printf("digit_sum(507)  = %d\n", digit_sum(507));
    printf("digit_sum(0)    = %d\n", digit_sum(0));
    return 0;
}
```

```

./digitsum
```

```
digit_sum(0)    = 0
```

**Ответ:** `digit_sum(1234) = 10`, `digit_sum(507) = 12`, `digit_sum(0) = 0`. Функция накапливает `n % 10`, отбрасывая разряд `n /= 10`, пока `n > 0`; для `0` сразу возвращает `0`.

### Г4.Ч2.З1

**Условие.** Напишите функцию `int ticket(void)` с `static`-счётчиком, которая при каждом вызове возвращает следующий номер: `1`, `2`, `3`, … В `main` вызовите её пять раз и напечатайте выданные номера.

**Решение.** «Помнить» между вызовами позволяет `static`-локальная переменная (§4.2.2): её время жизни статическое, инициализатор `= 0` выполняется один раз, а значение переживает выход из функции. Каждый вызов увеличивает счётчик и возвращает новое значение. Пять вызовов в цикле дают `1..5`.

```c
#include <stdio.h>

int ticket(void)
{
    static int counter = 0;      // инициализируется один раз
    counter++;                   // переживает выход из функции
    return counter;
}

int main(void)
{
    for (int i = 0; i < 5; i++)
        printf("%d ", ticket());
    printf("\n");
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra ticket.c -o ticket
./ticket
```

```
1 2 3 4 5 
```

`counter` не обнуляется на каждом вызове — `static` даёт ему статическое время жизни, и он накапливает: `1, 2, 3, 4, 5`. Здесь вызовы идут по одному за итерацию (слева направо), поэтому порядок номеров — прямой.

**Ответ:** печатается `1 2 3 4 5`. `static int counter` сохраняет значение между вызовами `ticket`, поэтому каждый вызов выдаёт следующий номер.

**Вариант 2 (глобальный счётчик).** «Помнить» состояние можно и через глобальную переменную — она тоже имеет статическое время жизни:

```c
#include <stdio.h>

int counter = 0;                 // глобальная: то же состояние, но общее

int ticket(void)
{
    counter++;
    return counter;
}

int main(void)
{
    for (int i = 0; i < 5; i++)
        printf("%d ", ticket());
    printf("\n");
    return 0;
}
```

```
1 2 3 4 5 
```

*Сравнение.* Вывод тот же. Разница — в **инкапсуляции**. `static`-переменная видна только внутри `ticket`: никакая другая функция не может её случайно изменить, счётчик «спрятан» в своей функции. Глобальный `counter` виден **всем** функциям ниже — его может незаметно поменять любой код, и такие ошибки трудно локализовать (§4.2.2, подводные камни). Когда состояние нужно одной функции, `static`-локальная предпочтительнее глобальной именно потому, что не засоряет общее пространство имён и защищает данные от посторонних.

### Г4.Ч2.З3

**Условие.** *(понимание)* Определите область видимости каждой переменной и предскажите, скомпилируется ли код; если нет — где ошибка:

```c
#include <stdio.h>

int main(void) {
    int a = 1;
    { int b = 2; printf("%d %d\n", a, b); }
    printf("%d\n", a);
    printf("%d\n", b);   // ?

}
```

**Решение.** Области видимости (§4.2.1): `a` объявлена в теле `main` — видна от объявления до конца `main`. `b` объявлена **внутри вложенного блока** `{ … }` — её область видимости заканчивается на закрывающей `}` этого блока. Значит, внутри блока доступны и `a`, и `b` (первый `printf` корректен), а вот последний `printf("%d\n", b)` стоит **вне** блока `b` — там имя `b` уже не существует. Код **не компилируется**.

```
gcc -std=c17 -Wall -Wextra scope.c -o scope
```

```
scope.c: In function 'main':
scope.c:7:20: error: 'b' undeclared (first use in this function)
    7 |     printf("%d\n", b);   // ?
      |                    ^
scope.c:7:20: note: each undeclared identifier is reported only once for each function it appears in
```

Ошибка именно на строке 7 — обращение к `b` за пределами её блока. Строки внутри блока и `printf("%d\n", a)` были бы корректны; проблема одна — «утёкшая» из области видимости `b`.

**Ответ:** не скомпилируется. `a` видна во всём теле `main`; `b` — только внутри вложенного блока `{ }`. Ошибка на `printf("%d\n", b);` вне блока: `error: 'b' undeclared` — переменная за пределами своей области видимости не существует.

### Г4.Ч2.З4

**Условие.** *(понимание)* Предскажите вывод и объясните разницу через время жизни (§4.2.1–4.2.2):

```c
#include <stdio.h>

int f(void) { int n = 0;        n++; return n; }
int g(void) { static int n = 0; n++; return n; }
int main(void) {
    printf("%d %d %d\n", f(), f(), f());
    printf("%d %d %d\n", g(), g(), g());
    return 0;
}
```

(Учтите, что аргументы `printf` на вашей платформе могут вычисляться справа налево — сверьтесь с фактическим выводом и объясните порядок.)

**Решение.** В `f` переменная `n` — **обычная локальная**: она создаётся заново на каждом вызове (автоматическое время жизни, §4.2.1), `n++` делает её `1`, и функция всегда возвращает `1`. Поэтому три вызова `f()` дают три единицы — независимо от порядка вычисления. В `g` переменная `n` — **`static`**: инициализируется один раз, переживает вызовы (§4.2.2), поэтому последовательные вызовы возвращают `1`, `2`, `3` в **порядке вызова**.

```
1 1 1
3 2 1
```

Первая строка — `1 1 1`: все три `f()` вернули `1`, порядок вычисления не важен, значения одинаковы. Вторая строка — `3 2 1`: аргументы `printf` на эталонной платформе вычисляются **справа налево**, поэтому первым вызывается самый правый `g()` (вернул `1`), затем средний (`2`), затем левый (`3`); печатаются же они слева направо — отсюда `3 2 1`. Порядок вычисления аргументов в C не задан стандартом (глава 13), но на суть — «`static` накапливает, обычная локальная нет» — это не влияет.

**Ответ:** печатается `1 1 1`, затем `3 2 1`. `f` каждый раз заводит свежую `n` и возвращает `1` (порядок неважен); `g` со `static int n` накапливает `1, 2, 3` по вызовам, а видимый порядок `3 2 1` — из-за вычисления аргументов `printf` справа налево на этой платформе.

### Г4.Ч2.З6

**Условие.** *(понимание)* Предскажите вывод и объясните, что такое затенение (shadowing):

```c
#include <stdio.h>

void show(void) { printf("global x = %d\n", x); }
int main(void) {
    int x = 20;             // локальная затеняет глобальную

    return 0;
}
```

**Решение.** Есть две разные переменные с именем `x`: глобальная (`= 10`) и локальная в `main` (`= 20`). Внутри `main` локальная `x` **затеняет** (shadowing) глобальную: имя `x` в теле `main` относится к локальной, поэтому `printf` в `main` печатает `20`. Функция `show` локальной `x` из `main` не видит (§4.2.1 — локальные не выходят наружу) и обращается к **глобальной** `x`, печатая `10`.

```
gcc -std=c17 -Wall -Wextra shadow.c -o shadow
./shadow
```

```
local x = 20
global x = 10
```

Затенение — это когда во внутренней области видимости объявлено имя, совпадающее с именем из внешней; внутри «побеждает» ближайшее (локальное), внешнее временно недоступно, но **не изменено**. Глобальная `x` осталась `10`, что и подтверждает `show`.

**Ответ:** печатается `local x = 20`, затем `global x = 10`. Локальная `x` в `main` затеняет глобальную (внутри `main` имя `x` — это локальная `20`), а `show`, не видя локальной, читает глобальную `10`. Затенение = совпадение имён, где ближайшая переменная перекрывает внешнюю, не меняя её.

### Г4.Ч2.З7

**Условие.** *(понимание)* Программа передаёт данные между функциями через глобальную переменную:

```c
#include <stdio.h>

int result;
void compute(int a, int b) { result = a + b; }
int main(void) { compute(3, 4); printf("%d\n", result); return 0; }
```

Объясните, чем такой стиль плох. Перепишите так, чтобы `compute` **возвращала** результат (`int compute(int a, int b)`), без глобальной переменной.

**Решение.** Здесь `compute` не возвращает результат, а «прячет» его в глобальную `result`, откуда `main` его потом достаёт. Плохо это тем, что связь между функциями становится **неявной**: чтобы понять, как `main` получает ответ, надо знать о существовании глобальной `result` и о том, что `compute` её тихо меняет. Функция перестаёт быть «чёрным ящиком» с чётким контрактом (§4.2.2, подводные камни): её нельзя вызвать, не затронув общее состояние; две такие функции, пишущие в один `result`, легко переплетутся, а ошибки трудно локализовать. Штатный способ отдать результат — `return` (§4.1.3).

```c
#include <stdio.h>

int compute(int a, int b)        // результат возвращается, а не пишется в глобальную
{
    return a + b;
}

int main(void)
{
    int result = compute(3, 4);
    printf("%d\n", result);
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra compute.c -o compute
./compute
```

```
7
```

Теперь `result` — **локальная** переменная `main`, а `compute` честно возвращает сумму. Контракт функции виден по сигнатуре `int compute(int, int)`: «принимает два `int`, возвращает `int», — и никакого скрытого общего состояния.

**Ответ:** стиль плох тем, что данные передаются через скрытое глобальное состояние — связь неявная, функция не самодостаточна, ошибки труднее локализовать. Исправление: `int compute(int a, int b) { return a + b; }`, а в `main` — `int result = compute(3, 4);`. Вывод `7` не меняется, но зависимость стала явной.

**Условие.** *(прикладное)* Напишите функцию `void trace(int depth)`, которая печатает свой параметр `depth` и, если `depth > 0`, вызывает `trace(depth - 1)` (простейшая цепочка вложенных вызовов). Вызвав `trace(3)`, объясните по выводу, в каком порядке кладутся и снимаются кадры стека (§4.2.3). Сколько кадров `trace` одновременно лежит на стеке в момент печати `0`?

**Решение.** Функция печатает `depth`, затем (пока `depth > 0`) вызывает себя с `depth - 1` — это цепочка вложенных вызовов. Каждый вызов кладёт на стек **свой кадр** с собственным параметром `depth` (§4.2.3); печать идёт при спуске, до вложенного вызова, поэтому значения печатаются по убыванию.

```c
#include <stdio.h>

void trace(int depth)
{
    printf("%d ", depth);        // печатаем свой параметр
    if (depth > 0)
        trace(depth - 1);        // вложенный вызов
}

int main(void)
{
    trace(3);
    printf("\n");
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra trace.c -o trace
./trace
```

```
3 2 1 0 
```

Порядок кадров: `main` вызывает `trace(3)` — кладётся кадр `trace(3)` (печатает `3`), он вызывает `trace(2)` — новый кадр поверх (печатает `2`), тот `trace(1)` (печатает `1`), тот `trace(0)` (печатает `0`; `depth > 0` ложно — глубже не идём). Затем кадры **снимаются** в обратном порядке: `trace(0)` возвращается → `trace(1)` → `trace(2)` → `trace(3)` → `main`. То есть кладутся сверху вниз по `depth` (3→2→1→0), снимаются снизу вверх (0→1→2→3) — стек работает по принципу LIFO.

В момент печати `0` на стеке лежат **четыре** кадра `trace` одновременно: `trace(3)`, `trace(2)`, `trace(1)` и `trace(0)` — ни один ещё не вернулся, они «висят», ожидая завершения самого глубокого.

**Ответ:** печатается `3 2 1 0`. Кадры `trace` кладутся на стек в порядке `trace(3) → trace(2) → trace(1) → trace(0)` (каждый печатает своё `depth` при спуске) и снимаются в обратном порядке при возвратах (LIFO). В момент печати `0` на стеке лежат **4** кадра `trace` одновременно.

## Часть 4.3. Рекурсия

### Г4.Ч3.З1

**Условие.** Напишите рекурсивную функцию `long long power(int base, int exp)` (для `exp >= 0`), возвращающую `base` в степени `exp`. Базовый случай — `exp == 0` (результат `1`), шаг — `base * power(base, exp - 1)`. Проверьте: `power(2, 10)` = `1024`, `power(5, 3)` = `125`, `power(7, 0)` = `1`.

**Решение.** Определение степени рекурсивно: `base⁰ = 1` (базовый случай, останавливает рекурсию), а `baseⁿ = base · baseⁿ⁻¹` (шаг, уменьшает `exp` на 1 — приближает к базовому). Тип результата `long long`, поэтому произведение `base * power(...)` вычисляется в `long long` (второй множитель уже `long long`), и промежуточные степени не переполняют `int`.

```c
#include <stdio.h>

long long power(int base, int exp)
{
    if (exp == 0)                    // базовый случай
        return 1;
    return base * power(base, exp - 1);   // рекурсивный шаг
}

int main(void)
{
    printf("power(2, 10) = %lld\n", power(2, 10));
    printf("power(5, 3)  = %lld\n", power(5, 3));
    printf("power(7, 0)  = %lld\n", power(7, 0));
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra power.c -o power
./power
```

```
power(2, 10) = 1024
power(5, 3)  = 125
power(7, 0)  = 1
```

`power(2, 10)` разворачивается в `2·2·…·2` (десять двоек) = `1024`; `power(5, 3) = 5·5·5 = 125`; `power(7, 0)` сразу срабатывает базовый случай → `1`. Каждый шаг уменьшает `exp`, гарантируя достижение `exp == 0`.

**Ответ:** `power(2, 10) = 1024`, `power(5, 3) = 125`, `power(7, 0) = 1`. Базовый случай `exp == 0` возвращает `1`, шаг `base * power(base, exp - 1)` сводит задачу к меньшему показателю.

**Вариант 2 (быстрое возведение в степень).** Наивная версия делает `exp` умножений. Если заметить, что `baseⁿ = (baseⁿ⁄²)²` для чётного `n`, число шагов падает до ~`log₂(exp)`: считаем степень половинного показателя один раз и возводим в квадрат (для нечётного `exp` домножаем ещё на `base`).

```c
long long power(int base, int exp)   // быстрое возведение в степень
{
    if (exp == 0)
        return 1;
    long long half = power(base, exp / 2);   // один вызов вместо exp
    if (exp % 2 == 0)
        return half * half;
    else
        return half * half * base;
}
```

```
power(2, 10) = 1024
power(5, 3)  = 125
power(7, 0)  = 1
```

*Сравнение.* Результаты совпадают. Наивная `power(base, exp - 1)` наглядна и буквально повторяет определение, но для `power(2, 10)` делает 10 вложенных вызовов; быстрая версия для того же показателя обходится ~4 уровнями (`10 → 5 → 2 → 1 → 0`), так как на каждом шаге делит `exp` пополам, а не вычитает 1. На маленьких показателях выигрыш незаметен, но приём «разделяй пополам» — основа быстрого возведения в степень (и мотива «разделяй и властвуй», §4.3.2). Обратите внимание: `half` вычисляется **один раз** и переиспользуется — если написать `power(base, exp/2) * power(base, exp/2)`, преимущество теряется (два одинаковых вызова, как в наивном Фибоначчи).

**Условие.** Напишите рекурсивную функцию `void print_1_to_n(int n)`, печатающую числа от `1` до `n` **по возрастанию** через пробел. Подсказка: чтобы получить возрастающий порядок, рекурсивный вызов `print_1_to_n(n - 1)` должен идти **до** печати `n`. Проверьте на `n = 5` (ожидается `1 2 3 4 5`).

**Решение.** Хитрость — в порядке действий. Если сначала **рекурсивно** обработать `n - 1` (напечатать `1..n-1`), а **потом** напечатать `n`, то числа выйдут по возрастанию: самый глубокий вызов (с наименьшим `n`) напечатает раньше всех, а печать текущего `n` отложена до возврата из рекурсии. Базовый случай — `n <= 0`: печатать нечего, просто возвращаемся (он же защищает от бесконечной рекурсии при `n <= 0`).

```c
#include <stdio.h>

void print_1_to_n(int n)

        return;
    print_1_to_n(n - 1);             // сперва меньшие — потом текущее
    printf("%d ", n);
}

int main(void)
{
    print_1_to_n(5);
    printf("\n");
    return 0;
}
```

```
./updown
```


Спуск идёт `5 → 4 → 3 → 2 → 1 → 0` (пока ничего не печатая), на `0` разворот; затем при подъёме печатаются `1`, `2`, `3`, `4`, `5` — печать `n` выполняется **после** возврата из `print_1_to_n(n - 1)`. Поменяй строки местами (сначала `printf`, потом вызов) — порядок стал бы убывающим (`5 4 3 2 1`).

**Ответ:** для `n = 5` печатается `1 2 3 4 5`. Рекурсивный вызов `print_1_to_n(n - 1)` **до** `printf` откладывает печать текущего числа до возврата из более глубоких вызовов, поэтому меньшие числа печатаются первыми.

### Г4.Ч3.З3

**Условие.** Напишите рекурсивную функцию `int digit_count(int n)` (для `n >= 0`), возвращающую количество десятичных цифр в числе. Базовый случай: числа `0`…`9` имеют одну цифру; шаг: `1 + digit_count(n / 10)`. Проверьте: `digit_count(0)` = `1`, `digit_count(7)` = `1`, `digit_count(1234)` = `4`.

**Решение.** Однозначные числа (`n < 10`, включая `0`) имеют одну цифру — это базовый случай. Для больших чисел отбрасываем младшую цифру (`n / 10`) и считаем: цифр на одну больше, чем в укороченном числе (`1 + digit_count(n / 10)`). Деление `/ 10` монотонно уменьшает `n`, гарантируя достижение базового случая.

```c
#include <stdio.h>

int digit_count(int n)
{
    if (n < 10)                      // базовый случай: одна цифра
        return 1;
    return 1 + digit_count(n / 10);  // шаг: отбрасываем младшую цифру
}

int main(void)
{
    printf("digit_count(0)    = %d\n", digit_count(0));
    printf("digit_count(7)    = %d\n", digit_count(7));
    printf("digit_count(1234) = %d\n", digit_count(1234));
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra count.c -o count
./count
```

```
digit_count(0)    = 1
digit_count(7)    = 1
digit_count(1234) = 4
```

`0` и `7` меньше 10 → сразу `1`. `1234`: `1 + digit_count(123)` = `1 + (1 + digit_count(12))` = `1 + 1 + (1 + digit_count(1))` = `1 + 1 + 1 + 1` = `4`. Базовый случай `n < 10` корректно ловит и `0` (у нуля одна цифра).

**Ответ:** `digit_count(0) = 1`, `digit_count(7) = 1`, `digit_count(1234) = 4`. Базовый случай `n < 10` даёт `1`, шаг `1 + digit_count(n / 10)` добавляет по цифре, отбрасывая младший разряд.

**Вариант 2 (итеративный подсчёт).** Ту же задачу решает цикл: начинаем с `count = 1` и, пока `n >= 10`, отбрасываем разряд и увеличиваем счётчик.

```c
int digit_count(int n)               // итеративный вариант
{
    int count = 1;
    while (n >= 10) {
        n /= 10;
        count++;
    }
    return count;
}
```

```

digit_count(7)    = 1
digit_count(1234) = 4
```

*Сравнение.* Выводы одинаковы. Рекурсивная версия ближе к словесному определению («на цифру больше, чем в числе без младшего разряда») и красиво разворачивается, но каждый разряд стоит кадра на стеке. Итеративная держит счётчик в одной переменной и не наращивает стек (§4.3.2). Для подсчёта цифр глубина рекурсии крошечная (число разрядов), так что оба варианта безопасны; но там, где глубина могла бы быть большой, цикл предпочтительнее — он экономнее по памяти и не рискует переполнить стек.

**Условие.** *(понимание)* Для функции `factorial` из §4.3.1 составьте таблицу трассировки вызова `factorial(4)` со столбцами «этап (спуск/подъём)», «активный вызов», «что делает», «возвращает» (как в примере 1 §4.3.2).     printf("<- factorial(%d) returns %lld\n", n, result);   // подъём: кадр снят

```c
int main(void)
```

Все четыре `->` идут подряд до разворота на `factorial(1)`, затем четыре `<-` — это и есть спуск и подъём из таблицы.

### Г4.Ч3.З5

**Условие.** *(понимание)* Почему эта функция приведёт к переполнению стека и что при этом произойдёт с программой? Найдите причину и исправьте, добавив корректный базовый случай:

```c
int down(int n) {
    return n + down(n - 1);   // где остановка?
}
```

**Решение.** У функции **нет базового случая** — она вызывает себя всегда, ни при каком `n` не останавливаясь. Аргумент убывает (`n - 1`), но не к какому-то порогу, а бесконечно: `... 2 → 1 → 0 → -1 → -2 → ...`. Каждый вызов кладёт на стек новый кадр (§4.2.3), кадры копятся без конца, и рано или поздно стек **исчерпывается** — происходит переполнение стека, и программа **аварийно завершается** (крах), не напечатав ничего.

Эталон при `-Wall` статически распознаёт проблему ещё при компиляции:

```
gcc -std=c17 -Wall -Wextra down.c -o down
```

```
down.c: In function 'down':
down.c:3:5: warning: infinite recursion detected [-Winfinite-recursion]
    3 | int down(int n) {
      |     ^~~~
down.c:4:16: note: recursive call
    4 |     return n + down(n - 1);   // где остановка?
      |                ^~~~~~~~~~~
```

А запуск такой программы завершается крахом (переполнение стека) без вывода. Лечится добавлением базового случая — например, `n == 0` возвращает `0` (тогда `down` считает сумму `n + (n-1) + … + 1 + 0`):

```c
#include <stdio.h>

int down(int n) {
    if (n == 0)               // добавленный базовый случай
        return 0;
    return n + down(n - 1);
}

int main(void)
{
    printf("down(5) = %d\n", down(5));
    return 0;
}
```

```
down(5) = 15
```

Теперь спуск останавливается на `down(0) = 0`, и сумма `5 + 4 + 3 + 2 + 1 + 0 = 15` корректно «всплывает» обратно. (Замечание: если вызвать исправленную `down` с отрицательным `n`, базовый случай `n == 0` будет обойдён — для полной надёжности берут `n <= 0`.)

**Ответ:** нет базового случая — рекурсия бесконечна, кадры копятся, стек переполняется, и программа аварийно завершается (не напечатав результата); `gcc -Wall` даже предупреждает `warning: infinite recursion detected`. Исправление — добавить базовый случай `if (n == 0) return 0;`; тогда `down(5) = 15`.

### Г4.Ч3.З6

**Условие.** *(понимание)* Даны рекурсивная (`sum_to` из §4.3.1) и требуемая итеративная версии суммы `1..n`. Напишите итеративную `int sum_to_iter(int n)` циклом. Какая из двух версий использует больше памяти стека при большом `n` и почему? Что произойдёт с рекурсивной версией при `n = 1000000`?

**Решение.** Итеративная сумма — классический накопитель (§3.2.5): один цикл `for` от `1` до `n`, прибавляющий `i` к `sum`. Никаких вложенных вызовов — работает в **одном** кадре.

```c
#include <stdio.h>

int sum_to_iter(int n)
{
    int sum = 0;
    for (int i = 1; i <= n; i++)
        sum += i;
    return sum;
}

int main(void)
{
    printf("sum_to_iter(5)   = %d\n", sum_to_iter(5));
    printf("sum_to_iter(100) = %d\n", sum_to_iter(100));
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra sumiter.c -o sumiter
./sumiter
```

```
sum_to_iter(5)   = 15
sum_to_iter(100) = 5050
```

**Память стека.** Рекурсивная `sum_to` использует стека **больше**: чтобы досчитать сумму, она уходит вглубь до `sum_to(0)`, и все промежуточные вызовы `sum_to(n), sum_to(n-1), …, sum_to(1)` одновременно висят на стеке — глубина ≈ `n` кадров (§4.3.2). Итеративная версия держит всё в двух переменных (`sum`, `i`) внутри одного кадра, и глубина стека от `n` не зависит.

**При `n = 1000000`.** Рекурсивная `sum_to(1000000)` попытается уложить на стек около миллиона кадров и **переполнит стек** — программа аварийно завершится (краш), не вернув результата. Прямой прогон рекурсивной версии это подтверждает: запуск `sum_to(1000000)` завершается крахом без вывода. Итеративная же `sum_to_iter(1000000)` по стеку отработала бы спокойно (хотя её `int`-результат при таком `n` — отдельная история: истинная сумма `500000500000` не помещается в `int`, это уже переполнение **типа**, а не стека, §2.1.2).

**Ответ:** итеративную версию см. выше (`sum_to_iter(5) = 15`, `sum_to_iter(100) = 5050`). Больше памяти стека использует **рекурсивная** версия: её глубина ≈ `n` кадров (все вложенные вызовы висят одновременно), тогда как итеративная работает в одном кадре. При `n = 1000000` рекурсивная `sum_to` переполняет стек и аварийно завершается; итеративная по стеку справляется (переполнение `int`-результата — отдельный вопрос).

### Г4.Ч3.З7

**Условие.** *(понимание)* Для наивной рекурсивной `fib` из §4.3.2 подсчитайте, сколько всего вызовов `fib` происходит при вычислении `fib(5)` (постройте дерево вызовов). Объясните, почему `fib(3)` и `fib(2)` вычисляются повторно, и как это связано с ростом времени работы. Чем принципиально лучше итеративная версия из §3.2.5?

**Решение.** Наивная `fib(n)` при `n >= 2` делает **два** вызова: `fib(n-1)` и `fib(n-2)`. Построим дерево вызовов для `fib(5)` (в скобках — что каждый узел вызывает):

```
fib(5)
├─ fib(4)
│  ├─ fib(3)
│  │  ├─ fib(2)
│  │  │  ├─ fib(1)
│  │  │  └─ fib(0)
│  │  └─ fib(1)
│  └─ fib(2)
│     ├─ fib(1)
│     └─ fib(0)
└─ fib(3)
   ├─ fib(2)
   │  ├─ fib(1)
   │  └─ fib(0)
   └─ fib(1)
```

Считаем узлы: `fib(5)`—1, `fib(4)`—1, `fib(3)`—2, `fib(2)`—3, `fib(1)`—5, `fib(0)`—3. Итого **15** вызовов. Проверим счётчиком (`static`-переменная считает каждый вход в `fib`):

```c
#include <stdio.h>

static int call_count = 0;       // считаем каждый вход в fib

long long fib(int n)
{
    call_count++;
    if (n < 2)
        return n;
    return fib(n - 1) + fib(n - 2);
}

int main(void)
{
    long long result = fib(5);
    printf("fib(5) = %lld, calls = %d\n", result, call_count);
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra fibcount.c -o fibcount
./fibcount
```

```
fib(5) = 5, calls = 15
```

Счётчик подтверждает: 15 вызовов. **Почему повторы.** `fib(3)` считается дважды: один раз как левое поддерево `fib(5)` (через `fib(4)`), другой — как правое поддерево `fib(5)` напрямую. `fib(2)` считается трижды. Причина — функция **не запоминает** уже вычисленные значения: каждое поддерево считается заново, с нуля. Чем больше `n`, тем катастрофичнее дублирование: число вызовов растёт примерно как сами числа Фибоначчи (экспоненциально), поэтому уже `fib(45)` наивно считается ощутимо долго.

**Чем лучше итеративная версия (§3.2.5).** Она идёт по ряду **один раз**, храня две последние величины (`prev`, `curr`) и сдвигая их: `n`-е число получается за `n` шагов, каждое значение вычисляется **ровно один раз**, без дерева и повторов. Время растёт линейно (`~n`), память постоянна (две переменные), стек не задействован. Разница — экспонента против линии.

**Ответ:** при `fib(5)` происходит **15** вызовов `fib` (дерево выше, подтверждено счётчиком). `fib(3)` вычисляется 2 раза, `fib(2)` — 3 раза, потому что наивная рекурсия не запоминает результаты и пересчитывает каждое поддерево заново; отсюда экспоненциальный рост числа вызовов и времени. Итеративная версия из §3.2.5 считает каждое число один раз за линейное время с постоянной памятью — принципиально эффективнее.

### Г4.Ч3.З8

**Условие.** *(прикладное)* Напишите рекурсивную функцию `int gcd(int a, int b)`, вычисляющую наибольший общий делитель по алгоритму Евклида: базовый случай — `b == 0` (результат `a`), шаг — `gcd(b, a % b)`. В `main` напечатайте `gcd(48, 36)`, `gcd(17, 5)`, `gcd(100, 0)`. Ожидается `12`, `1`, `100`.

**Решение.** Алгоритм Евклида в рекурсивной форме почти дословно повторяет задание: если `b == 0`, наибольший общий делитель равен `a` (базовый случай); иначе `gcd(a, b) = gcd(b, a % b)` — заменяем пару на «(меньшее, остаток)». Остаток `a % b` строго меньше `b`, поэтому второй аргумент убывает и рано или поздно станет `0` — рекурсия конечна.

```c
#include <stdio.h>

int gcd(int a, int b)
{
    if (b == 0)                  // базовый случай
        return a;
    return gcd(b, a % b);        // шаг алгоритма Евклида
}

int main(void)
{
    printf("gcd(48, 36) = %d\n", gcd(48, 36));
    printf("gcd(17, 5)  = %d\n", gcd(17, 5));
    printf("gcd(100, 0) = %d\n", gcd(100, 0));
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra gcd.c -o gcd
./gcd
```

```
gcd(48, 36) = 12
gcd(17, 5)  = 1
gcd(100, 0) = 100
```

Разбор `gcd(48, 36)`: `gcd(48,36) → gcd(36, 48%36=12) → gcd(12, 36%12=0) → 12`. `gcd(17,5) → gcd(5,2) → gcd(2,1) → gcd(1,0) → 1` (числа взаимно просты). `gcd(100, 0)` — сразу базовый случай, `b == 0` → `100`.

**Ответ:** `gcd(48, 36) = 12`, `gcd(17, 5) = 1`, `gcd(100, 0) = 100`. Базовый случай `b == 0` возвращает `a`, шаг `gcd(b, a % b)` уменьшает второй аргумент до нуля.

**Вариант 2 (итеративный Евклид).** Тот же алгоритм разворачивается в цикл: пока `b != 0`, заменяем пару `(a, b)` на `(b, a % b)`.

```c
int gcd(int a, int b)            // итеративный Евклид
{
    while (b != 0) {
        int t = a % b;
        a = b;
        b = t;
    }
    return a;
}
```

```
gcd(48, 36) = 12
gcd(17, 5)  = 1
gcd(100, 0) = 100
```

*Сравнение.* Результаты совпадают. Рекурсивная форма читается прямо как математическое определение Евклида и очень компактна; итеративная держит пару в двух переменных и не расходует кадры стека. Глубина рекурсии здесь мала (Евклид сходится за число шагов порядка логарифма от чисел), поэтому рекурсия безопасна; итеративная версия чуть экономнее (нет накладных расходов на вызовы) и надёжна при любых входах. Это ровно тот случай, когда обе формы одинаково хороши — выбор диктует стиль.

```c
#include <stdio.h>

int digit_sum_rec(int n)
```

### Г4.Ч3.З10

**Условие.** *(прикладное)* Напишите рекурсивную функцию `void print_binary(int n)` (для `n >= 1`), печатающую двоичную запись числа. Идея: чтобы старшие разряды шли первыми, сначала рекурсивно обработайте `n / 2` (если `n > 1`), затем напечатайте `n % 2`. Проверьте: `print_binary(13)` → `1101`, `print_binary(10)` → `1010`, `print_binary(1)` → `1`.

**Решение.** Биты числа `% 2`/`/ 2` получаются справа налево (младший разряд — первым), а печатать надо слева направо (старший — первым). Порядок исправляет рекурсия: сначала **рекурсивно** печатаем старшую часть `print_binary(n / 2)` (если есть, т. е. `n > 1`), и лишь **потом** — младший бит текущего числа `n % 2`. Базовый случай — `n == 1` (или в общем `n <= 1`): рекурсивный вызов не делается, печатается единственный бит.

```c
#include <stdio.h>

void print_binary(int n)
{
    if (n > 1)                   // сперва старшие разряды
        print_binary(n / 2);
    printf("%d", n % 2);         // затем младший бит текущего n
}

int main(void)
{
    printf("print_binary(13): "); print_binary(13); printf("\n");
    printf("print_binary(10): "); print_binary(10); printf("\n");
    printf("print_binary(1):  "); print_binary(1);  printf("\n");
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra binary.c -o binary
./binary
```

```
print_binary(13): 1101
print_binary(10): 1010
print_binary(1):  1
```

`13`: спуск `13 → 6 → 3 → 1`, затем печать при подъёме `1`, `1`, `0`, `1` → `1101` (проверка: `1·8 + 1·4 + 0·2 + 1·1 = 13`). `10`: `10 → 5 → 2 → 1`, печать `1`, `0`, `1`, `0` → `1010` (`8 + 2 = 10`). `1`: `n > 1` ложно, сразу печатается `1`. Печать `n % 2` **после** рекурсивного вызова и даёт порядок «старший бит первым».

**Ответ:** `print_binary(13)` → `1101`, `print_binary(10)` → `1010`, `print_binary(1)` → `1`. Рекурсивный вызов `print_binary(n / 2)` **до** печати `n % 2` выводит старшие разряды раньше младших, давая правильный порядок битов.
