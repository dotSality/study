## Часть 1.1. Как устроены компьютер и программа

### Г1.Ч1.З1

**Условие.** Переведите в десятичную систему: а) 101₂; б) 1111₂; в) 100000₂; г) 10101010₂.

**Решение.** Выписываем под цифрами веса разрядов справа налево (1, 2, 4, 8, …) и складываем те веса, под которыми стоит единица (§1.1.1, пример 2).

а) `101₂`:

```
1  0  1
4  2  1   →  4 + 1 = 5
```

б) `1111₂`: 8 + 4 + 2 + 1 = **15**. Обратите внимание на закономерность: четыре единицы подряд дают 2⁴ − 1 = 15, восемь единиц — 2⁸ − 1 = 255 (пример 3 §1.1.1).

в) `100000₂`: единственная единица стоит в разряде с весом 2⁵ = **32**. Общее правило: единица и *n* нулей после неё — это ровно 2ⁿ.

г) `10101010₂`:

```
1    0   1   0   1  0  1  0
128  64  32  16  8  4  2  1   →  128 + 32 + 8 + 2 = 170
```

**Ответ:** а) 5; б) 15; в) 32; г) 170.

**Вариант 2 (схема Горнера — удвоение слева направо).** Идём по цифрам слева направо, начиная с нуля, и на каждом шаге удваиваем накопленное и прибавляем очередную цифру: `r ← 2·r + цифра`.

- `101₂`: 0 → 2·0+1 = 1 → 2·1+0 = 2 → 2·2+1 = **5**;
- `1111₂`: 1 → 3 → 7 → **15**;
- `100000₂`: 1 → 2 → 4 → 8 → 16 → **32**;
- `10101010₂`: 1 → 2 → 5 → 10 → 21 → 42 → 85 → **170**.

*Сравнение.* Метод весов нагляднее и позволяет мгновенно ответить на вопрос об *отдельном* разряде («какой вклад даёт пятый бит?»), но требует держать в голове или выписывать таблицу степеней двойки — для длинных чисел это утомительно и легко ошибиться на один разряд. Схема Горнера не требует таблицы вовсе и одинаково работает при любой длине числа, потому что на каждом шаге выполняется одна и та же пара действий; именно так перевод делает и программа (удвоение — это сдвиг влево). Для ручной работы с байтами удобнее веса, для длинных цепочек битов — Горнер.

**Типичная неверная попытка.** Считать разряды слева направо («первая цифра — единицы»). Вес зависит от позиции *с конца*: в `100000₂` единица стоит в разряде 32, а не в разряде 1.

### Г1.Ч1.З2

**Условие.** Переведите в двоичную систему: а) 5; б) 12; в) 64; г) 200. Для каждого числа покажите разложение по весам разрядов (как в примере 1 §1.1.1).

**Решение.** Идём от старшего подходящего веса к младшему, каждый раз забирая из числа максимально возможную степень двойки (§1.1.1, пример 1).

а) **5** = 4 + 1 → разряды 4 и 1 включены, разряд 2 выключен → `101₂`.

б) **12** = 8 + 4 → `1100₂` (разряды 2 и 1 выключены).

в) **64** — сама степень двойки, 64 = 2⁶ → единица в седьмом разряде и шесть нулей → `1000000₂`.

г) **200**: 200 ≥ 128 → берём 128, остаток 72; 72 ≥ 64 → берём 64, остаток 8; 32 и 16 не помещаются (8 < 16) → нули; 8 ≥ 8 → берём 8, остаток 0; дальше нули.

```
128 + 64 + 8 = 200
1    1    0    0    1  0  0  0
128  64   32   16   8  4  2  1   →  11001000₂
```

**Ответ:** а) 5 = 101₂; б) 12 = 1100₂; в) 64 = 1000000₂; г) 200 = 11001000₂.

**Вариант 2 (деление на 2 с остатками).** Делим число на 2, записывая остатки, пока частное не станет нулём; затем читаем остатки **снизу вверх**. Для 200:

```
200 : 2 = 100, остаток 0
100 : 2 =  50, остаток 0
 50 : 2 =  25, остаток 0
 25 : 2 =  12, остаток 1
 12 : 2 =   6, остаток 0
  6 : 2 =   3, остаток 0
  3 : 2 =   1, остаток 1
  1 : 2 =   0, остаток 1
```

Читаем остатки снизу вверх: `11001000₂` — совпало с ответом.

*Сравнение.* Разложение по весам требует «увидеть» старшую подходящую степень двойки — это быстро для небольших чисел и хорошо показывает смысл записи (какие веса включены), но при числах в тысячи приходится угадывать старший разряд. Деление с остатками полностью механично: никакого выбора на каждом шаге нет, ошибиться можно только в порядке чтения остатков. Оно же обобщается на любую систему счисления (делить на 8, на 16) и именно в таком виде записывается в коде. Разумная практика — считать делением, а проверять сложением весов, как сделано выше.

### Г1.Ч1.З3

**Условие.** Ответьте с обоснованием: а) сколько различных значений можно закодировать 4 битами? б) 16 битами? в) каково наибольшее неотрицательное число, помещающееся в 2 байта?

**Решение.**

а) Каждый бит независимо принимает 2 значения, поэтому при добавлении бита число комбинаций удваивается: 1 бит — 2 комбинации, 2 бита — 4, 3 бита — 8, 4 бита — 2⁴ = **16** (от `0000` до `1111`).

б) Тем же рассуждением 16 битов дают 2¹⁶ = **65 536** комбинаций.

в) 2 байта — это 2 × 8 = 16 битов, то есть 65 536 различных комбинаций (п. б). Если трактовать их как неотрицательные числа, они нумеруются подряд начиная с нуля: 0, 1, …, значит наибольшее — на единицу меньше числа комбинаций: 65 536 − 1 = **65 535**. Это та же логика, что и для одного байта: «256 комбинаций, максимум 255» (§1.1.1, пример 3).

**Ответ:** а) 16; б) 65 536; в) 65 535 (= 2¹⁶ − 1).

**Вариант 2 (через сумму весов — для пункта в).** Наибольшее число получается, когда включены все разряды, поэтому это сумма всех весов двух байтов: 32768 + 16384 + 8192 + 4096 + 2048 + 1024 + 512 + 256 + 128 + 64 + 32 + 16 + 8 + 4 + 2 + 1 = **65 535**.

*Сравнение.* Рассуждение «комбинаций 2ⁿ, максимум на единицу меньше» короче и не зависит от количества битов — им пользуются, когда нужно быстро назвать границу типа. Суммирование весов длиннее, зато ничего не требует принимать на веру и заодно объясняет, *почему* максимум равен 2ⁿ − 1: сумма всех степеней двойки от 1 до 2ⁿ⁻¹ ровно на единицу меньше следующей степени. Полезно один раз проделать вручную, дальше пользоваться первым способом.

### Г1.Ч1.З4

**Условие.** Коды ASCII букв идут подряд: `A` = 65, `B` = 66 и так далее. Закодируйте слово `BEE` последовательностью байтов (в десятичной записи) и расшифруйте последовательность 67, 65, 66.

**Решение.** Раз коды идут подряд, код буквы = 65 + (номер буквы в алфавите − 1):

- `B` — вторая буква: 65 + 1 = 66;
- `E` — пятая буква: 65 + 4 = 69.

Значит `BEE` — это три байта: 66, 69, 69. В двоичной записи (для наглядности, §1.1.1): `01000010 01000101 01000101`.

Обратная задача — вычитаем 65 и отсчитываем букву: 67 − 65 = 2 → третья буква `C`; 65 − 65 = 0 → первая буква `A`; 66 − 65 = 1 → вторая буква `B`. Получается слово `CAB`.

**Ответ:** `BEE` → 66, 69, 69; последовательность 67, 65, 66 → `CAB`.

**Типичная неверная попытка.** Считать, что `A` = 65 значит «`B` = 65 + 2». Смещение берётся от первой буквы: сдвиг для `A` равен 0, для `B` — 1, для `E` — 4.

### Г1.Ч1.З5

**Условие.** Объясните своими словами роли памяти и процессора и разницу между адресом и значением по адресу. Нарисуйте (на бумаге или в текстовом файле) схему участка памяти с адресами 2000–2004, в котором начиная с адреса 2001 лежит слово `OK` (найдите коды букв, зная, что `A` = 65).

**Решение.**

*Роли.* Память — это хранилище: длинная последовательность пронумерованных байтов, в которой лежат и данные программы, и её машинный код (§1.1.2). Сама память ничего не вычисляет — она только отдаёт байт по номеру и записывает байт по номеру. Процессор — исполнитель: он читает из памяти машинные инструкции и выполняет их, обращаясь к памяти за операндами и складывая туда результаты. Кратко: память отвечает на вопрос «что где лежит», процессор — «что с этим сделать».

*Адрес против значения.* Адрес — номер байта, ответ на вопрос **где**; значение — содержимое этого байта, ответ на вопрос **что**. Оба записываются числами, и в этом источник путаницы: фраза «по адресу 2001 лежит 79» содержит два разных числа, 2001 — номер дома, 79 — то, что в доме лежит. Аналогия §1.1.2: улица с пронумерованными домами.

*Коды букв.* `O` — пятнадцатая буква алфавита: 65 + 14 = 79. `K` — одиннадцатая: 65 + 10 = 75.

*Схема.* Слово `OK` занимает два байта начиная с адреса 2001, то есть байты 2001 и 2002; байты 2000, 2003, 2004 к слову не относятся:

```
адрес:    2000    2001    2002    2003    2004
        ┌───────┬───────┬───────┬───────┬───────┐
память: │  ???  │  79   │  75   │  ???  │  ???  │
        └───────┴───────┴───────┴───────┴───────┘
                  'O'     'K'
        (не наши)                (не наши)
```

**Ответ:** память хранит пронумерованные байты, процессор выполняет над ними инструкции; адрес — номер байта (где), значение — его содержимое (что). На схеме: по адресу 2001 — байт 79 (`O`), по адресу 2002 — байт 75 (`K`); «слово лежит по адресу 2001» означает адрес его **первого** байта.

### Г1.Ч1.З6

**Условие.** Объясните разницу между компиляцией и интерпретацией. Назовите по одному практическому преимуществу каждого подхода.

**Решение.** Различие — в том, *когда* происходит перевод программы в действия машины (§1.1.3).

- **Компиляция**: компилятор один раз, заранее и целиком переводит исходный текст в машинный код и сохраняет результат в исполняемый файл. Дальше этот файл запускается сам: процессор выполняет готовые инструкции, компилятор больше не нужен.
- **Интерпретация**: интерпретатор разбирает исходный текст во время работы программы и сам выполняет предписанные действия. Исполняемого файла не появляется, поэтому при каждом запуске нужны и исходный текст, и установленный интерпретатор.

Практические преимущества (достаточно назвать по одному):

- у компиляции — **скорость выполнения**: процессор работает с готовым машинным кодом без посредника (а также, как отмечено в §1.1.3, ранняя диагностика ошибок и автономность собранного файла — его можно отдать человеку, у которого компилятор не установлен);
- у интерпретации — **отсутствие этапа сборки**: тот же самый исходный файл запускается на любой машине, где есть интерпретатор, без перекомпиляции под её платформу (сравните с примером 3 §1.1.3: `hello.exe`, собранный для Windows, на Linux не запустится).

**Ответ:** компилятор переводит программу заранее и целиком, создавая самостоятельный исполняемый файл; интерпретатор выполняет исходный текст на лету, и при каждом запуске нужен сам. Преимущество компиляции — скорость выполнения готового машинного кода (плюс ранняя диагностика и автономность файла); преимущество интерпретации — не нужен этап сборки, один и тот же исходник работает везде, где есть интерпретатор.

### Г1.Ч1.З7

**Условие.** Найдите и исправьте ошибки в утверждении: «Когда я запускаю скомпилированную программу, компилятор читает мой исходный код строка за строкой и выполняет его».

**Решение.** В одном предложении три ошибки (§1.1.3, «типичные ошибки»):

1. **«компилятор … выполняет»** — компилятор ничего не выполняет, он переводчик. Выполняет инструкции процессор.
2. **«при запуске компилятор читает…»** — к моменту запуска компилятор не участвует вообще; он отработал раньше, при сборке, и на машине пользователя может быть не установлен.
3. **«читает мой исходный код строка за строкой»** — запускается не исходный текст, а исполняемый файл с машинным кодом; исходный файл при запуске не нужен и не читается. Описание «строка за строкой во время работы» относится к интерпретатору, а не к компилируемому языку.

**Ответ (исправленная формулировка):** «Когда я запускаю скомпилированную программу, процессор выполняет машинные инструкции из исполняемого файла, который компилятор создал заранее — при сборке; ни компилятор, ни исходный код в момент запуска не нужны».

### Г1.Ч1.З8

**Условие.** В ячейке памяти лежит байт 01000001₂. Один человек говорит, что там число 65, другой — что буква `A`. Кто прав и почему? Что этот пример говорит о природе данных в памяти?

**Решение.** Сначала посчитаем: 01000001₂ = 64 + 1 = 65.

В памяти лежит именно набор битов, и его числовое значение — 65; это факт, не зависящий ни от чьей трактовки. «Буква `A`» — это **интерпретация** того же байта по таблице ASCII, а не что-то, дополнительно записанное в память. Поэтому:

- первый прав в том смысле, что содержимое байта как числа — 65;
- второй прав в том смысле, что если программа использует этот байт как символ, он отобразится как `A`;
- оба неправы, если утверждают, что байт «является» только числом или только буквой: в памяти нет пометки о том, чем он «на самом деле» является.

Тот же байт с равным успехом может быть частью машинной инструкции, одним из байтов большего числа или пикселем картинки — всё решает то, как его использует программа (§1.1.1, третий подводный камень).

**Ответ:** правы оба, но каждый наполовину. Байт хранит 65; `A` — прочтение этого числа как ASCII-кода. Природа данных в памяти такова, что смысл байтов задаётся не памятью, а программой, которая их использует; из самого байта его назначение не следует.

### Г1.Ч1.З9

**Условие.** *(практическое исследование)* Выясните для своего компьютера: а) объём оперативной памяти; б) разрядность системы (32 или 64 бита). Переведите объём памяти из гигабайтов в байты (примите 1 ГБ = 2³⁰ байт) и запишите результат.

**Решение.** Ответ у каждого свой; ниже — ход работы и результат на машине, где готовилась книга (Windows 10, 64-разрядная).

*Как узнать (Windows).* «Параметры» → «Система» → «О системе»: там сразу и «Оперативная память», и «Тип системы». То же из PowerShell:

```
Get-CimInstance Win32_ComputerSystem | Select-Object TotalPhysicalMemory
(Get-CimInstance Win32_OperatingSystem).OSArchitecture
```

На эталонной машине: `TotalPhysicalMemory = 68597473280`, `OSArchitecture = 64-bit`. На Linux те же сведения дают `free -b` (или `/proc/meminfo`) и `uname -m`, на macOS — «Об этом Mac».

*Перевод.* Установлено 64 ГБ, по условию 1 ГБ = 2³⁰ = 1 073 741 824 байта:

```
64 × 2³⁰ = 64 × 1 073 741 824 = 68 719 476 736 байт  (= 2³⁶)
```

*Отчего расхождение с показаниями системы.* ОС сообщила 68 597 473 280 байт — на 122 003 456 байт (≈116 МиБ) меньше расчётных: часть физической памяти зарезервирована оборудованием и не отдаётся операционной системе. Это нормальная ситуация, а не ошибка счёта; при сверке смотрите на порядок величины.

*Разрядность.* 64 бита — это ширина адреса (§1.1.2: адрес есть число). 64-битный адрес нумерует 2⁶⁴ байт — величина, недостижимая для нынешней памяти; 32-битный адресует лишь 2³² = 4 294 967 296 байт (4 ГиБ), поэтому 64 ГБ на 32-разрядной системе были бы попросту не адресуемы.

**Ответ (для эталонной машины):** а) 64 ГБ (система сообщает 68 597 473 280 байт); б) 64-разрядная; перевод: 64 × 2³⁰ = **68 719 476 736 байт**. Свои числа запишите тем же способом: значение из системы, расчёт, комментарий о расхождении.

**Условие.** *(практическое исследование)* Найдите на диске любой текстовый файл (`.txt`, `.md`) и любую исполняемую программу (например, `C:\Windows\System32\notepad.exe` на Windows). Сравните их размеры, попробуйте открыть оба файла текстовым редактором. Объясните увиденное, опираясь на §1.1.3: чем содержимое исполняемого файла принципиально отличается от текста?


*Что видно при открытии текстовым редактором.* `.md` читается как обычный текст: каждый байт файла — код символа, и редактор ровно так его и показывает. `notepad.exe` выглядит набором нечитаемых знаков, среди которых изредка попадаются осмысленные фрагменты. Первые байты (шестнадцатеричный дамп начала файла) объясняют, что именно попадается:

```

0000100 016 037 272 016  \0 264  \t 315   ! 270 001   L 315   !   T   h
0000120   i   s       p   r   o   g   r   a   m       c   a   n   n   o
0000140   t       b   e       r   u   n       i   n       D   O   S
0000160   m   o   d   e   .  \r  \r  \n   $  \0  \0  \0  \0  \0  \0  \0
```

Файл начинается с двух букв `MZ` — метки формата исполняемых файлов Windows, дальше идут служебные числа (`\0`, `377`, `220` — это байты, не буквы), а затем встречается настоящая текстовая строка `This program cannot be run in DOS mode.`. Она читается именно потому, что *является* текстом внутри файла; всё окружающее — машинный код и служебные структуры, и редактор всё равно пытается показать эти байты как символы — отсюда «мусор».

*Объяснение (§1.1.3).* Текстовый файл — исходный материал для человека и компилятора: последовательность кодов символов, и ничего кроме. Исполняемый файл — результат компиляции: последовательность машинных инструкций плюс служебные заголовки для операционной системы (где начинается код, какие библиотеки нужны). Байты в нём не являются кодами символов, и трактовать их как текст бессмысленно — это разные *назначения* одинаковых по природе байтов (ср. З8).

*Про размер.* Исполняемый файл почти всегда крупнее исходника, и не потому, что «программа сложная»: к вашему коду добавляются служебные структуры и присоединённый компоновщиком библиотечный код (§1.2.3, этап 4). Наглядно: программа-визитка из З10 части 1.2 — это 10 строк исходного текста, а собранный `visit.exe` занимает **240 461 байт**.

**Ответ:** текстовый файл целиком состоит из кодов символов и потому читается редактором; исполняемый файл содержит машинные инструкции и служебные заголовки, поэтому в редакторе выглядит нечитаемым (осмысленно читаются лишь редкие текстовые строки внутри него, вроде `This program cannot be run in DOS mode.`). Размер исполняемого файла определяется не только объёмом вашего кода, но и служебными структурами и присоединённым кодом библиотек.

**Осторожно.** Открыв `.exe` в редакторе, ничего не сохраняйте: сохранение как текста портит файл. Надёжнее смотреть копию или пользоваться просмотрщиком (`od -c файл | head` в Git Bash, `Format-Hex` в PowerShell).

### Г1.Ч2.З2

**Условие.** Наберите `hello.c` из §1.2.2 (вручную, не копированием — руки должны привыкать), соберите и запустите. Убедитесь, что вывод совпадает с приведённым в параграфе.

**Решение.** Файл `hello.c` (набирается вручную; комментарии добавлены здесь для разбора и в вашем файле не обязательны):

```c
#include <stdio.h>          // подставляет объявления ввода-вывода, в том числе printf

int main(void)              // точка входа: выполнение программы начинается здесь
{
    printf("Hello, world!\n");   // вывод строки; \n — перевод строки
    return 0;                    // статус 0 — успешное завершение
}
```

Сборка и запуск в папке с файлом:

```
gcc -std=c17 -Wall -Wextra hello.c -o hello
./hello
```

Вывод:

```
Hello, world!
```

Совпадение с параграфом означает сразу три вещи: компилятор установлен и работает, файл сохранён там, где вы его собираете, и цикл «редактор → сборка → запуск» у вас замкнулся. Компилятор при сборке молчит — и это правильный результат: с `-Wall -Wextra` любое замечание было бы напечатано.

**Ответ:** программа собирается без замечаний и печатает `Hello, world!`.

**Частые заминки при первом запуске.** «No such file or directory» — терминал находится не в той папке, где лежит `hello.c` (перейдите командой `cd`); «`hello` не является командой» — забыт префикс `./` (в PowerShell — `.\hello`); `error: stray '\342' in program` — в файл попали типографские кавычки (`“…”`) из скопированного текста; наберите строку заново прямыми кавычками `"…"` (§1.2.2).

### Г1.Ч2.З3

**Условие.** Измените программу так, чтобы она выводила приветствие с вашим именем (латиницей), пересоберите и запустите. Что произойдёт, если запустить программу, изменив код, но забыв пересобрать? Проверьте.

**Решение.** Исходная версия `greet.c`:

```c
#include <stdio.h>

int main(void)
{
    printf("Hello, Max!\n");   // подставьте своё имя латиницей
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra greet.c -o greet
./greet
```

```
Hello, Max!
```

Теперь меняем строку на `printf("Hi, Max!\n");`, **сохраняем файл** и, не пересобирая, запускаем снова:

```
./greet
```

```
Hello, Max!
```

Печатается прежний текст. После пересборки:

```
gcc -std=c17 -Wall -Wextra greet.c -o greet
./greet
```

```
Hi, Max!
```

*Почему так.* Команда `./greet` запускает **исполняемый файл** — машинный код, созданный при прошлой сборке (§1.1.3). Этот файл никак не связан с текстом `greet.c`: правка исходника меняет только текст, а перевод текста в машинный код выполняет компилятор — и только когда его позвали. Пока команда `gcc` не выполнена заново, на диске лежит старая программа, и она честно печатает то, что в неё скомпилировали.

**Ответ:** запуск без пересборки выполняет **старую** версию программы — вывод не меняется, как будто правка не подействовала. Порядок всегда один: сохранить файл → собрать (`gcc …`) → запустить. Это одна из самых частых потерь времени у начинающих: «мой код не работает» при том, что запускается вообще не он.

### Г1.Ч2.З4

**Условие.** Напишите программу, выводящую три строки: название вашего города, страну и год — сначала тремя вызовами `printf`, затем одним. Убедитесь, что вывод обеих версий идентичен.

**Решение.** Версия с тремя вызовами (`card3.c`):

```c
#include <stdio.h>

int main(void)
{
    printf("Moscow\n");   // каждая строка выводится отдельным вызовом
    printf("Russia\n");
    printf("2026\n");
    return 0;
}
```

Версия с одним вызовом (`card1.c`) — тот же текст одним строковым литералом, переводы строк внутри него:

```c
#include <stdio.h>

int main(void)
{
    printf("Moscow\nRussia\n2026\n");   // три \n дают три строки
    return 0;
}
```

Сборка, запуск и посимвольное сравнение выводов:

```
gcc -std=c17 -Wall -Wextra card3.c -o card3
gcc -std=c17 -Wall -Wextra card1.c -o card1
./card3 > o3.txt
./card1 > o1.txt
diff o3.txt o1.txt && echo IDENTICAL
```

```
Moscow
Russia
2026
IDENTICAL
```

(`diff` не напечатал ни одной строки различий, поэтому выполнилось `echo`. Глазами выводы сравнивать тоже можно, но `diff` замечает и невидимые различия — лишний пробел или пропущенный перевод строки.)

*Вывод из опыта.* Разбиение на строки задаётся исключительно символами `\n` внутри выводимого текста. Отдельный вызов `printf` сам по себе новой строки не начинает — он просто дописывает свой текст туда, где закончился предыдущий вывод.

**Ответ:** обе версии печатают три строки `Moscow`, `Russia`, `2026`, и их вывод побайтово одинаков; число вызовов `printf` на результат не влияет — влияет только расстановка `\n`.

### Г1.Ч2.З5

**Условие.** Уберите все `\n` из предыдущей программы, соберите и запустите. Опишите и объясните результат: что именно решает, где закончится строка вывода?

**Решение.** Программа `nonl.c`:

```c
#include <stdio.h>

int main(void)
{
    printf("Moscow");   // ни одного \n во всей программе
    printf("Russia");
    printf("2026");
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra nonl.c -o nonl
./nonl
```

Вывод (приглашение терминала оказывается в той же строке — здесь оно обозначено комментарием):

```
MoscowRussia2026[здесь же оказывается приглашение терминала]
```

Смотрим на фактические байты вывода (`od -c` показывает каждый байт):

```
$ ./nonl | od -c
0000000   M   o   s   c   o   w   R   u   s   s   i   a   2   0   2   6
0000020
```

Шестнадцать байтов, ни одного управляющего символа: перевода строки в выводе нет вообще — ни между «строками», ни в конце. Слова слиплись именно поэтому, а не из-за «слишком быстрых» вызовов `printf`.

**Ответ:** всё выводится в одну строку — `MoscowRussia2026`, — и следующее приглашение терминала печатается сразу за текстом. Где закончится строка вывода, решает **только** символ `\n` в выводимом тексте: ни завершение вызова `printf`, ни завершение самой программы новой строки не создают. Отсюда практическое правило: последняя выводимая строка программы должна заканчиваться `\n`, иначе вывод «сольётся» с приглашением терминала.

### Г1.Ч2.З6

**Условие.** Намеренно удалите точку с запятой после вызова `printf` и соберите программу. Выпишите сообщение компилятора и разберите его по схеме из примера 2 §1.2.2: файл, строка, позиция, суть, подсказка. На какую строку указывает компилятор — с ошибкой или следующую?

**Решение.** Файл `broken.c` (сломан намеренно):

```c
#include <stdio.h>

int main(void)
{
    printf("Moscow\n")     // ⛔ пропущена точка с запятой
    printf("Russia\n");
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra broken.c -o broken
```

```
broken.c: In function 'main':
broken.c:5:23: error: expected ';' before 'printf'
    5 |     printf("Moscow\n")
      |                       ^
      |                       ;
    6 |     printf("Russia\n");
      |     ~~~~~~
```

Разбор по схеме §1.2.2:

| Элемент | Значение | Что означает |
|---|---|---|
| файл | `broken.c` | где искать |
| строка | `5` | строка с вызовом `printf("Moscow\n")` |
| позиция | `23` | символ сразу за закрывающей `)` — там ожидалась `;` |
| суть | `error: expected ';' before 'printf'` | ожидалась точка с запятой, а встретилось `printf` |
| подсказка | `^` и подставленная `;` под позицией 23 | место и сам недостающий символ |
| контекст | строка 6 с подчёркиванием `~~~~~~` | конструкция, на которой компилятор «споткнулся» |

*Ответ на вопрос задания.* Позиция ошибки (`5:23`) указывает на **строку с ошибкой** — точнее, на её конец, где точка с запятой и должна была стоять. При этом в тексте сообщения назван токен из **следующей** строки (`before 'printf'`) и она же показана в контексте: компилятор обнаруживает пропуск лишь тогда, когда доходит до следующей конструкции. Из-за этого сообщение и выглядит «указывающим на следующую строку» (§1.2.2, подводные камни) — читать его надо так: «недостающий символ — в конце строки 5, а заметил я это, дойдя до `printf` в строке 6».

Проверка того же на другом месте пропуска: если убрать `;` после `return 0`, следующей конструкцией окажется закрывающая скобка блока:

```
broken2.c:6:13: error: expected ';' before '}' token
    6 |     return 0
      |             ^
      |             ;
    7 | }
```

Схема та же: позиция — конец строки 6, «виновник» в тексте — токен `}` из строки 7.

**Ответ:** сообщение — `broken.c:5:23: error: expected ';' before 'printf'`; файл `broken.c`, строка 5, позиция 23, суть — не хватает `;`, подсказка — стрелка и подставленный символ `;`. Компилятор указывает на строку с ошибкой (её конец), но называет конструкцию из следующей строки, потому что именно на ней обнаружил пропуск. Исполняемый файл не создан: `error` останавливает сборку.

### Г1.Ч2.З7

**Условие.** Верните точку с запятой и удалите строку `#include <stdio.h>`. Соберите, выпишите диагностику. Какое решение подсказывает сам компилятор в строке `note:`? Объясните, почему без этой строки `printf` неизвестен компилятору.

**Решение.** Файл `noinc.c`:

```c
int main(void)          // ⛔ нет строки #include <stdio.h>
{
    printf("Moscow\n");
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra noinc.c -o noinc
```

```
noinc.c: In function 'main':
noinc.c:3:5: error: implicit declaration of function 'printf' [-Wimplicit-function-declaration]
    3 |     printf("Moscow\n");
      |     ^~~~~~
noinc.c:1:1: note: include '<stdio.h>' or provide a declaration of 'printf'
  +++ |+#include <stdio.h>
    1 | int main(void)
noinc.c:3:5: warning: incompatible implicit declaration of built-in function 'printf' [-Wbuiltin-declaration-mismatch]
    3 |     printf("Moscow\n");
      |     ^~~~~~
noinc.c:3:5: note: include '<stdio.h>' or provide a declaration of 'printf'
```

*Что подсказывает компилятор.* Строка `note:` предлагает ровно то, что нужно: `include '<stdio.h>' or provide a declaration of 'printf'` — подключить заголовочный файл или объявить `printf` самостоятельно. Более того, GCC показывает готовую правку: строка `+++ |+#include <stdio.h>` означает «вставьте сюда, перед строкой 1, вот эту строку».

*Почему `printf` неизвестен.* `#include <stdio.h>` — указание препроцессору подставить в ваш файл содержимое заголовочного файла, где находятся **объявления** библиотечных функций ввода-вывода (§1.2.2, §1.2.3). Объявление сообщает компилятору имя функции и то, с какими аргументами и результатом она используется. Без него в строке 3 компилятору встречается имя `printf`, о котором он не знает ничего, — и он сообщает об ошибке `implicit declaration of function`. Второе сообщение (`warning: incompatible implicit declaration of built-in function`) — следствие первого: `printf` компилятору известен как встроенная функция, и «додуманное» по месту вызова описание с этим знанием не согласуется.

Важно не путать два разных механизма (§1.2.3, подводные камни): заголовочный файл даёт только *объявление* — обещание, что такая функция существует; сам код `printf` присоединяет компоновщик на четвёртом этапе сборки из стандартной библиотеки.

**Ответ:** диагностика — `error: implicit declaration of function 'printf'` плюс `note: include '<stdio.h>' or provide a declaration of 'printf'` (с готовой строкой правки) и предупреждение `incompatible implicit declaration of built-in function 'printf'`. Компилятор предлагает подключить `<stdio.h>` либо объявить функцию самому. Без этой строки компилятор не имеет объявления `printf` — имени, аргументов и типа результата, — а придумывать их сам он не вправе, поэтому сборка останавливается.

### Г1.Ч2.З8

**Условие.** Назовите четыре этапа сборки по порядку. Для каждого укажите: что поступает на вход, что получается на выходе и каким флагом `gcc` можно остановить конвейер на этом этапе.

**Решение.** По §1.2.3, на примере `hello.c`:

| № | Этап | Вход | Выход | Флаг остановки |
|---|---|---|---|---|
| 1 | Препроцессирование | `hello.c` — ваш текст с директивами `#include`, `#define` | `hello.i` — текст на C, где `#include` заменён содержимым заголовочных файлов | `-E` (результат направляют в файл: `gcc -E hello.c > hello.i`) |
| 2 | Компиляция | `hello.i` — подготовленный C-код | `hello.s` — текст на ассемблере, инструкции конкретного процессора | `-S` |
| 3 | Ассемблирование | `hello.s` — ассемблерный текст | `hello.o` — двоичный объектный файл: машинный код, но ещё не программа | `-c` |
| 4 | Компоновка | `hello.o` + код стандартной библиотеки (реализация `printf`) | `hello.exe` — исполняемый файл | останавливать нечего: это последний этап; отдельно вызывается как `gcc hello.o -o hello` |

Ключевые переходы, которые стоит удерживать в голове: после этапа 1 — всё ещё текст на C; после этапа 2 — всё ещё текст, но уже на языке процессора; после этапа 3 — уже двоичный код, но ещё не запускаемый; после этапа 4 — программа.

**Ответ:** препроцессирование (`.c` → `.i`, флаг `-E`), компиляция (`.i` → `.s`, флаг `-S`), ассемблирование (`.s` → `.o`, флаг `-c`), компоновка (`.o` + библиотеки → исполняемый файл; отдельный вызов `gcc файл.o -o программа`).

### Г1.Ч2.З9

**Условие.** Пройдите конвейер вручную для своей программы из З4: получите `.i`, `.s`, `.o` и соберите исполняемый файл из объектного. Посчитайте число строк в `.i`-файле (командой `wc -l файл.i` в Git Bash/Linux или `(Get-Content файл.i).Count` в PowerShell) и объясните, откуда взялись «лишние» строки по сравнению с исходником.

**Решение.** Берём `card3.c` из З4 (9 строк вместе с пустыми) и проходим четыре этапа по очереди.

```
gcc -E card3.c > card3.i      # 1. препроцессор
wc -l card3.i card3.c
```

```
 1060 card3.i
    9 card3.c
```

```
gcc -S card3.c                # 2. компиляция → card3.s (ассемблер)
head -6 card3.s
```

```
	.file	"card3.c"
	.text
	.def	printf;	.scl	3;	.type	32;	.endef
	.seh_proc	printf
printf:
	pushq	%rbp
```

```
gcc -c card3.c                # 3. ассемблирование → card3.o (1171 байт)
./card3.o                     # попытка запустить объектный файл
```

```
bash: ./card3.o: cannot execute binary file: Exec format error
```

```
gcc card3.o -o card3b         # 4. компоновка
./card3b
```

```
Moscow
Russia
2026
```

*Откуда 1060 строк вместо 9.* Строка `#include <stdio.h>` заменена полным содержимым заголовочного файла `stdio.h`, а он, в свою очередь, подключает другие заголовки — так набираются объявления всех функций ввода-вывода и связанных с ними типов, независимо от того, пользуетесь вы ими или нет (мы используем один `printf`). Плюс к этому препроцессор расставляет строки-маркеры вида `# 1642 "…/stdio.h" 2 3`, которые сообщают компилятору, из какого файла и какой строки взят следующий фрагмент, — благодаря им в сообщениях об ошибках указывается ваш файл и ваша строка, а не номер строки в разбухшем `.i`.

Ваш собственный код при этом не изменён ни на символ — в конце `card3.i` он лежит как есть:

```
# 2 "card3.c" 2


# 3 "card3.c"
int main(void)
{
    printf("Moscow\n");
    printf("Russia\n");
    printf("2026\n");
    return 0;
}
```

Ещё два наблюдения из прогона: объектный файл запустить нельзя (`Exec format error`) — это «полуфабрикат» без присоединённого кода `printf` (§1.2.3); а собранный из него `card3b` работает точно так же, как программа из З4, — конвейер, пройденный вручную, эквивалентен одной команде `gcc card3.c -o card3`.

**Ответ:** `card3.c` — 9 строк, `card3.i` — **1060** строк; разница целиком приходится на подставленное содержимое `stdio.h` (и подключаемых им заголовков) плюс служебные строки-маркеры `# номер "файл"`. Объектный файл не запускается, исполняемый получается только после компоновки.

**Вариант 2 (`-save-temps` — одной командой).** Все промежуточные файлы можно получить, не разбивая сборку на шаги: флаг `-save-temps` (документирован в C17 — GCC Manual) велит компилятору сохранить `.i` и `.s` по пути к готовой программе.

```
gcc -std=c17 -Wall -Wextra -save-temps card3.c -o card3
ls
wc -l card3.i
```

```
card3.c  card3.exe  card3.i  card3.o  card3.s
    1060 card3.i
```

Число строк то же — 1060.

*Сравнение.* Пошаговый способ (`-E`, `-S`, `-c`) *показывает* конвейер: каждая команда останавливает сборку на своём этапе, и видно, что дальше без следующего шага дело не идёт (объектный файл не запускается). Он же позволяет вмешаться между этапами — например, скомпоновать вручную заранее полученный `.o`. Флаг `-save-temps` ничего не останавливает: он выполняет полную сборку и попутно оставляет промежуточные файлы; это короче и удобнее в реальной работе, когда нужно просто заглянуть в `.i` или `.s` (например, проверить, во что раскрылся макрос — глава 11), но учебной картины «что во что превращается» он не даёт. Для этого задания правильный порядок — сначала руками по этапам, потом, зная устройство, пользоваться `-save-temps`.

### Г1.Ч2.З10

**Условие.** *(прикладное)* Напишите программу-«визитку»: рамка из символов `*`, внутри — имя и девиз (латиницей). Пример вывода:

```
****************
* Max          *
* Learning C   *
****************
```

**Решение.** Ширину рамки удобно выбрать заранее (здесь 16 символов) и подогнать под неё каждую строку: `*`, пробел, текст, пробелы-заполнители, `*`. Тогда закрывающие звёздочки выстроятся в столбец.

```c
#include <stdio.h>

int main(void)
{
    printf("****************\n");   // верхняя граница, 16 звёздочек
    printf("* Max          *\n");   // "* " + текст + пробелы до ширины + "*"
    printf("* Learning C   *\n");
    printf("****************\n");   // нижняя граница — точная копия верхней
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra visit.c -o visit
./visit
```

```
****************
* Max          *
* Learning C   *
****************
```

**Ответ:** программа из четырёх вызовов `printf` (см. код выше) печатает визитку в рамке; вывод совпадает с образцом задания.

**Вариант 2 (один вызов `printf`).** Вся визитка — один строковый литерал, границы строк заданы четырьмя `\n` (тот же приём, что во второй версии З4):

```c
#include <stdio.h>

int main(void)
{
    printf("****************\n* Max          *\n* Learning C   *\n****************\n");
    return 0;
}
```

Прогон даёт тот же вывод, что и основной вариант (проверено `diff` — файлы вывода совпали побайтово).

*Сравнение.* Вариант с четырьмя вызовами прямолинеен: одна строка кода — одна строка вывода, легко добавить или убрать строку визитки, при поиске опечатки видно, какой вызов за что отвечает, и рамка в исходнике выглядит так же, как на экране. Вариант с одним вызовом короче и явно показывает, что переводы строк создаёт `\n`, а не сам вызов (ср. З4 и З5). Его недостаток — ровно в потере наглядности: длинный литерал в одну строку неудобно править, легко пропустить `\n` на стыке, и тогда две строки визитки сольются, а рамка «поедет». Для рисунка, который вы будете подгонять руками, надёжнее первый вариант; второй уместен, когда текст короткий и фиксированный.

**Типичная неверная попытка.** Набрать внутренние строки «на глаз», не считая символы: если в `* Max          *` заполнитель короче или длиннее нужного, правая стенка рамки перестанет быть вертикальной. Считайте: длина внутренней строки должна равняться длине строки-границы (здесь 16 символов).

### Г1.Ч2.З11

**Условие.** *(прикладное)* Соберите визитку без флага `-o`. Найдите получившийся файл, запустите его, затем пересоберите с осмысленным именем. Объясните, откуда берётся имя по умолчанию и почему полагаться на него в работе неудобно.

**Решение.**

```
gcc -std=c17 -Wall -Wextra visit.c
ls -l a.exe
./a.exe
```

```
-rwxr-xr-x 1 Max 197609 240461 Jul 22 22:17 a.exe
****************
* Max          *
* Learning C   *
****************
```

Файл называется `a.exe` (на Linux/macOS — `a.out`) и лежит в текущей папке. Программа при этом собрана полноценно и работает. Пересборка с осмысленным именем:

```
gcc -std=c17 -Wall -Wextra visit.c -o visit
./visit
```

*Откуда берётся имя.* Флаг `-o` задаёт имя результата сборки; если его не указать, компилятор подставляет собственное имя по умолчанию — `a.exe` на Windows-целях и `a.out` на Unix-подобных системах (§1.2.3). Оно не зависит от имени исходного файла: и `visit.c`, и `hello.c` дадут `a.exe`.

*Почему на него неудобно полагаться.* Во-первых, имя одинаково для всех программ: собрали вторую — она молча затёрла первую, и запускается уже не то, что вы думаете (родственник ошибки из З3). Во-вторых, по имени `a.exe` невозможно понять, что это за программа, — а в учебной папке их накопятся десятки. В-третьих, любые автоматические шаги (запуск из скрипта, `make` в главе 11, тесты) требуют предсказуемого имени, привязанного к программе, а не общего для всех.

**Ответ:** без `-o` получается файл `a.exe` (на Linux/macOS `a.out`) — имя по умолчанию, подставляемое компилятором и не зависящее от имени исходника. Полагаться на него неудобно: одинаковое имя для всех программ приводит к затиранию предыдущей сборки, ничего не говорит о содержимом и мешает автоматизации. Правильная привычка — всегда указывать `-o` с осмысленным именем.

### Г1.Ч2.З12

**Условие.** *(прикладное исследование)* Добавьте в любую свою программу переменную `int x = 5;`, нигде её не используя. Соберите дважды: с флагами `-Wall -Wextra` и без них. Сравните вывод компилятора, объясните разницу между `warning` и `error` и сформулируйте (одним предложением) правило, почему предупреждения стоит держать включёнными всегда.

**Решение.** Файл `unused.c`:

```c
#include <stdio.h>

int main(void)
{
    int x = 5;                    // переменная объявлена и нигде не используется
    printf("Hello, world!\n");
    return 0;
}
```

Сборка **с** флагами:

```
gcc -std=c17 -Wall -Wextra unused.c -o unused
```

```
unused.c: In function 'main':
unused.c:5:9: warning: unused variable 'x' [-Wunused-variable]
    5 |     int x = 5;
      |         ^
```

Сборка **без** флагов:

```
gcc -std=c17 unused.c -o unused0
```

Компилятор не печатает ничего. При этом в обоих случаях сборка успешна (код возврата 0), исполняемый файл создан и работает одинаково:

```
./unused0
```

```
Hello, world!
```

*Разница между `warning` и `error`.*

| | `error` (ошибка) | `warning` (предупреждение) |
|---|---|---|
| Что означает | компилятор не может перевести код | код перевести можно, но он подозрителен |
| Сборка | прерывается | продолжается |
| Исполняемый файл | не создаётся | создаётся и работает |
| Примеры из главы | пропущенная `;` (З6), неизвестный `printf` (З7) | неиспользуемая переменная (это задание) |

Предупреждение — это не «мелкая ошибка», а сообщение другого рода: язык позволяет так писать, но компилятор заметил признак недосмотра. Неиспользуемая переменная почти всегда означает одно из двух: код остался от прежней версии либо переменную забыли применить там, где собирались (например, вычислили значение и напечатали не его). Отсюда и практическая ценность: сама по себе `int x = 5;` безвредна, но она указывает на место, где мысль автора разошлась с кодом.

Обратите внимание и на источник разницы: без `-Wall -Wextra` компилятор молчит не потому, что проблемы нет, — он просто не обязан о ней сообщать, пока его об этом не попросили.

**Ответ:** с `-Wall -Wextra` компилятор выдаёт `warning: unused variable 'x' [-Wunused-variable]`, без флагов — молчит; в обоих случаях программа собирается и работает. `error` останавливает сборку (файла не будет), `warning` её не останавливает, но указывает на подозрительное место. Правило одним предложением: **предупреждения стоит держать включёнными всегда, потому что они бесплатно указывают на ошибки замысла ещё до запуска программы — молчание компилятора без `-Wall -Wextra` означает не отсутствие проблем, а лишь то, что о них не попросили сообщить.**
