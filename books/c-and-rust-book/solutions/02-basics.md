## Часть 2.1. Переменные и базовые типы

### Г2.Ч1.З1

**Условие.** Объявите переменную `int year` и инициализируйте её текущим годом, затем присвойте ей год вашего рождения и в обоих случаях напечатайте значение через `printf("%d\n", …)`.

**Решение.** В задании три разных действия из §2.1.1: объявление вводит имя и тип, инициализация задаёт значение в момент рождения переменной, присваивание записывает новое значение в уже существующую.

```c
#include <stdio.h>

int main(void)
{
    int year = 2026;      // объявление с инициализацией: текущий год
    printf("%d\n", year);
    year = 1998;          // присваивание: год рождения
    printf("%d\n", year);
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra year.c -o year
./year
```

```
2026
1998
```

(Год рождения взят произвольно — важно не само число, а то, что вторая печать показывает уже другое значение той же переменной.)

Переменная одна, а значений у неё за время работы программы было два. `int year = 2026;` выполняется один раз — это объявление *с инициализацией*; `year = 1998;` — присваивание, их может быть сколько угодно. Прежнее значение при этом нигде не сохраняется: 2026 просто затёрто.

**Ответ:** программа печатает две строки — `2026`, затем `1998`. Тип (`int`) указывается только при объявлении; повторная запись значения делается присваиванием без имени типа.

**Типичная неверная попытка.** Написать второй раз `int year = 1998;`. Это не присваивание, а повторное объявление той же переменной в той же области видимости (§2.1.1, подводные камни) — компилятор откажется:

```
redecl.c: In function 'main':
redecl.c:7:9: error: redefinition of 'year'
    7 |     int year = 1998;   /* ⛔ повторное объявление вместо присваивания */
      |         ^~~~
redecl.c:5:9: note: previous definition of 'year' with type 'int'
    5 |     int year = 2026;
      |         ^~~~
```

### Г2.Ч1.З2

**Условие.** Напишите программу, печатающую размеры (`sizeof`, спецификатор `%zu`) всех пяти целых типов (`char`, `short`, `int`, `long`, `long long`) и обоих вещественных (`float`, `double`). Запишите, что получилось на вашей машине.

**Решение.** `sizeof` применяется к имени типа в скобках, результат имеет тип `size_t` и печатается спецификатором `%zu` (§2.1.4).

```c
#include <stdio.h>

int main(void)
{
    printf("char       %zu\n", sizeof(char));
    printf("short      %zu\n", sizeof(short));
    printf("int        %zu\n", sizeof(int));
    printf("long       %zu\n", sizeof(long));
    printf("long long  %zu\n", sizeof(long long));
    printf("float      %zu\n", sizeof(float));
    printf("double     %zu\n", sizeof(double));
    return 0;
}
```

Вывод на эталонной машине (Windows, GCC 15.2.0):

```
char       1
short      2
int        4
long       4
long long  8
float      4
double     8
```

Что здесь стоит зафиксировать. Во-первых, `sizeof(char)` равен 1 по определению — байт в C и есть «размер `char`». Во-вторых, порядок `char ≤ short ≤ int ≤ long ≤ long long` соблюдён, как и требует стандарт (§2.1.2). В-третьих, на этой платформе `long` **не шире** `int` — оба по 4 байта; на Linux и macOS та же программа напечатала бы `long 8`. Именно поэтому «`long` — это большой `int`» — ложное правило (§2.1.2, подводные камни).

**Ответ:** на эталонной машине — `char` 1, `short` 2, `int` 4, `long` 4, `long long` 8, `float` 4, `double` 8 байт. Стандарт фиксирует не эти числа, а лишь минимальные ширины и неубывающий порядок, поэтому на другой платформе (например, Linux) `long` окажется 8-байтовым.

**Вариант 2 (`sizeof` от переменных, а не от типов).** `sizeof` принимает и выражение — тогда скобки не нужны, а тип берётся у самого выражения:

```c
#include <stdio.h>

int main(void)
{
    char c;  short s;  int i;  long l;  long long ll;
    float f;  double d;

    // sizeof применён к переменным: скобки не нужны, тип берётся у выражения
    printf("char       %zu\n", sizeof c);
    printf("short      %zu\n", sizeof s);
    printf("int        %zu\n", sizeof i);
    printf("long       %zu\n", sizeof l);
    printf("long long  %zu\n", sizeof ll);
    printf("float      %zu\n", sizeof f);
    printf("double     %zu\n", sizeof d);
    return 0;
}
```

```
char       1
short      2
int        4
long       4
long long  8
float      4
double     8
```

Обратите внимание: переменные не инициализированы, и это **не** ошибка — `sizeof` не вычисляет операнд, а лишь измеряет тип его результата (§2.1.4), поэтому неинициализированные переменные здесь не читаются.

*Сравнение.* Форма `sizeof(тип)` короче и не требует заводить переменные — ей пользуются, когда нужен размер сам по себе (например, `n * sizeof(int)` при выделении памяти в главе 7). Форма `sizeof переменная` не повторяет имя типа, поэтому остаётся правильной, даже если тип переменной потом поменяют — при рефакторинге это ценнее краткости. Плата за неё — необходимость иметь саму переменную и лишний повод задуматься, вычисляется ли операнд (не вычисляется).

### Г2.Ч1.З3

**Условие.** Объявите `char c = 'M';` и напечатайте её двумя способами: как символ (`%c`) и как число (`%d`). Затем напечатайте символ со следующим кодом.

**Решение.** `char` — это целый тип шириной один байт, хранящий код символа (§2.1.3). Один и тот же байт печатается как буква (`%c`) или как число (`%d`) — разница только в спецификаторе.

```c
#include <stdio.h>

int main(void)
{
    char c = 'M';
    printf("as char: %c\n", c);              // символ
    printf("as int:  %d\n", c);              // его код
    printf("next:    %c\n", (char)(c + 1));  // код на единицу больше
    return 0;
}
```

```
as char: M
as int:  77
next:    N
```

Код 77 получается так: `A` = 65 (§1.1.1), `M` — тринадцатая буква алфавита, значит 65 + 12 = 77. Символ со следующим кодом — 78, это `N`: раз коды латинских букв идут подряд, «следующий символ» — это «код плюс один».

Приведение `(char)` в третьей строке не обязательно (`%c` печатает и `int`), но оно делает намерение явным: мы просим напечатать именно символ с кодом 78, а не число.

**Ответ:** `M` (как символ), `77` (как код), `N` (символ со следующим кодом). Один и тот же байт трактуется по-разному в зависимости от спецификатора — это прямая иллюстрация мысли §1.1.1 о том, что смысл байтов задаётся их использованием.

### Г2.Ч1.З4

**Условие.** Запишите число 60 тремя литералами: десятичным, шестнадцатеричным и восьмеричным. Напечатайте все три одним `printf` и убедитесь, что вывод одинаков.

**Решение.** Переводим 60 в нужные системы счисления (§1.1.1, §2.1.4):

- шестнадцатеричная: 60 = 3·16 + 12, где 12 — это цифра `C`, значит `0x3C`;
- восьмеричная: 60 = 7·8 + 4, значит `074` (ведущий ноль — признак восьмеричной записи).

```c
#include <stdio.h>

int main(void)
{
    printf("%d %d %d\n", 60, 0x3C, 074);  // десятичный, шестнадцатеричный, восьмеричный
    return 0;
}
```

```
60 60 60
```

Три одинаковых числа в выводе — не совпадение, а суть дела: `60`, `0x3C` и `074` — это три *записи* одного и того же значения. Форма записи существует только в исходном тексте; после компиляции от неё не остаётся и следа, в программу попадает одно число. Поэтому спецификатор `%d` печатает все три одинаково.

**Ответ:** `60`, `0x3C`, `074`; программа печатает `60 60 60`. Система счисления — свойство записи в исходном коде, а не свойство значения.

**Типичная неверная попытка.** Записать восьмеричное как `60` с ведущим нулём «для красоты» — `060`. Это допустимая запись, но она означает 6·8 = 48, а не 60. Ведущий ноль всегда меняет смысл литерала (§2.1.4).

### Г2.Ч1.З5

**Условие.** *(понимание)* Не запуская программу, предскажите вывод, затем проверьте:

```c
int a = 5;
int b = a;
a = 10;
printf("%d %d\n", a, b);
```

Объясните, почему `b` не изменилось вслед за `a`.

**Решение.** Разбираем по шагам, помня, что переменная — это участок памяти, а присваивание записывает в него *значение* (§2.1.1):

1. `int a = 5;` — заведена коробка `a`, в неё положено 5.
2. `int b = a;` — заведена **отдельная** коробка `b`; выражение справа вычисляется прямо сейчас и даёт 5; это значение копируется в `b`. Никакой связи между коробками не создаётся — копируется содержимое, а не «право смотреть на `a`».
3. `a = 10;` — в коробку `a` записано новое значение. Коробки `b` это не касается: в ней по-прежнему лежит копия, сделанная на шаге 2.

```
шаг 2:  a[ 5]  b[ 5]      шаг 3:  a[10]  b[ 5]
        └── копия ─┘               меняется только a
```

Проверка:

```c
#include <stdio.h>

int main(void)
{
    int a = 5;
    int b = a;   // b получает копию текущего значения a
    a = 10;      // меняется только a
    printf("%d %d\n", a, b);
    return 0;
}
```

```
10 5
```

**Ответ:** печатается `10 5`. `b` не изменилось, потому что `int b = a;` копирует *значение* переменной `a` в момент выполнения строки, а не связывает две переменные. У `a` и `b` разные участки памяти, и запись в один из них никак не затрагивает другой. (Связь «смотреть на чужую переменную» в C тоже бывает — она делается указателями, глава 6, — но требует явной записи и выглядит совсем иначе.)

### Г2.Ч1.З6

**Условие.** *(понимание)* Что напечатает `printf("%d\n", 010);` и почему? Что произойдёт при попытке скомпилировать `int x = 09;`? Проверьте и объясните, сославшись на §2.1.4.

**Решение.** По §2.1.4 ведущий ноль в целом литерале означает **восьмеричную** запись. Поэтому `010` — это не «десять», а 1·8 + 0 = 8:

```c
#include <stdio.h>

int main(void)
{
    printf("%d\n", 010);   // ведущий ноль -> восьмеричная запись
    return 0;
}
```

```
8
```

Из того же правила следует и судьба `09`: раз запись восьмеричная, её цифры могут быть только `0`–`7`. Цифра `9` в восьмеричной системе не существует, и это ошибка не времени выполнения, а времени компиляции — программа просто не соберётся:

```c
int x = 09;   /* 9 не является восьмеричной цифрой */
```

```
p1z6b.c: In function 'main':
p1z6b.c:5:13: error: invalid digit '9' in octal constant
    5 |     int x = 09;   /* 9 не является восьмеричной цифрой */
      |             ^~
```

Сообщение прямо называет причину: «недопустимая цифра `9` в восьмеричной константе».

**Ответ:** `printf("%d\n", 010);` печатает `8`, потому что ведущий ноль делает литерал восьмеричным (1·8 + 0). `int x = 09;` **не компилируется**: `error: invalid digit '9' in octal constant` — в восьмеричной записи цифр `8` и `9` нет. Практический вывод: незначащие нули перед десятичными числами писать нельзя — выравнивать столбцы чисел в коде ими особенно опасно.

### Г2.Ч1.З7

**Условие.** *(понимание)* В программе объявлена, но не инициализирована переменная `int n;`, после чего её печатают. Объясните, что не так, какое значение будет напечатано и почему полагаться на него нельзя. Как исправить одной правкой?

**Решение.** Объявление `int n;` только отводит место в памяти — оно ничего в это место не кладёт (§2.1.1). В отведённых байтах остаётся то, что там лежало раньше: остатки данных предыдущих вычислений. Чтение такой переменной по стандарту — **неопределённое поведение** (глава 13), то есть у программы вообще нет гарантированного результата.

Компилятор с `-Wall -Wextra` предупреждает:

```c
#include <stdio.h>

int main(void)
{
    int n;                 /* объявлена, но не инициализирована */
    printf("%d\n", n);
    return 0;
}
```

```
p1z7.c: In function 'main':
p1z7.c:6:5: warning: 'n' is used uninitialized [-Wuninitialized]
    6 |     printf("%d\n", n);
      |     ^~~~~~~~~~~~~~~~~
p1z7.c:5:9: note: 'n' was declared here
    5 |     int n;                 /* объявлена, но не инициализирована */
      |         ^
```

Запуск на эталонной машине напечатал:

```
604
```

Это число ничего не значит: в отведённых байтах оказалось то, что осталось от предыдущей работы программы и операционной системы. Насколько мало на него можно полагаться, видно из простого опыта: при подготовке сборника **тот же исходник**, собранный тем же компилятором с теми же флагами, но в другом каталоге, напечатал уже другое:

```
444
```

Ни одно из этих чисел не является «правильным ответом» — правильного ответа тут нет вовсе. Значение может измениться от сборки к сборке, от запуска к запуску, от машины к машине; иногда оно оказывается нулём — и это самый коварный случай, потому что программа выглядит работающей.

Исправление — одна правка, инициализация при объявлении:

```c
int n = 0;      /* теперь значение задано явно */
```

**Ответ:** переменная не инициализирована, поэтому печатается мусор — то, что случайно оказалось в её байтах (на эталонной машине `604`, а при повторной сборке того же файла — `444`); полагаться на это значение нельзя, так как чтение неинициализированной переменной — неопределённое поведение, а не «чтение нуля». Исправление: `int n = 0;`. Компилятор ловит такой случай предупреждением `-Wuninitialized` — предупреждения на `-Wall -Wextra` нужно читать, а не пролистывать.

### Г2.Ч1.З8

**Условие.** *(понимание)* Даны имена: `_total`, `2sum`, `my-value`, `Sum`, `sum`, `return`. Определите, какие из них — допустимые имена переменных, а какие нет и почему. Для допустимых укажите, являются ли `Sum` и `sum` одной переменной.

**Решение.** Правило §2.1.1: идентификатор состоит из латинских букв, цифр и `_`, не начинается с цифры и не совпадает с ключевым словом; регистр значим.

| Имя | Допустимо | Почему |
|---|---|---|
| `_total` | да | начинается с подчёркивания — это допустимый первый символ |
| `2sum` | **нет** | начинается с цифры |
| `my-value` | **нет** | дефис не входит в идентификатор; компилятор видит вычитание `my - value` |
| `Sum` | да | обычный идентификатор |
| `sum` | да | обычный идентификатор |
| `return` | **нет** | ключевое слово языка |

Три недопустимых имени дают три разные диагностики — по ним хорошо видно, как компилятор их читает.

`int 2sum = 0;`:

```
p1z8a.c:5:9: error: invalid suffix 'sum' on integer constant
    5 |     int 2sum = 0;   /* имя начинается с цифры */
      |         ^~~~
p1z8a.c:5:9: error: expected identifier or '(' before numeric constant
```

Компилятор начал читать `2` как число и принял `sum` за суффикс литерала (вроде `U` или `L`, §2.1.4) — отсюда странная формулировка.

`int my-value = 0;`:

```
p1z8b.c:5:11: error: expected '=', ',', ';', 'asm' or '__attribute__' before '-' token
    5 |     int my-value = 0;   /* дефис - это оператор вычитания */
      |           ^
p1z8b.c:5:12: error: 'value' undeclared (first use in this function)
```

Имя оборвалось на `my`, а дальше компилятор увидел операцию вычитания и незнакомое имя `value`.

`int return = 0;`:

```
p1z8c.c:5:9: error: expected identifier or '(' before 'return'
    5 |     int return = 0;   /* return - ключевое слово */
      |         ^~~~~~
```

На месте имени встретилось ключевое слово — компилятор сообщает, что ждал идентификатор.

Про регистр: `Sum` и `sum` — **разные** переменные, и это легко показать, заведя обе сразу:

```c
#include <stdio.h>

int main(void)
{
    int _total = 1;
    int Sum = 2;
    int sum = 3;      // sum и Sum - разные переменные
    printf("_total=%d Sum=%d sum=%d\n", _total, Sum, sum);
    return 0;
}
```

```
_total=1 Sum=2 sum=3
```

Если бы это было одно имя, компилятор сообщил бы о повторном объявлении (см. Г2.Ч1.З1); программа собралась и напечатала три разных значения.

**Ответ:** допустимы `_total`, `Sum`, `sum`; недопустимы `2sum` (начинается с цифры), `my-value` (дефис — не часть идентификатора, а оператор вычитания), `return` (ключевое слово). `Sum` и `sum` — две **разные** переменные: регистр букв в именах C значим.

### Г2.Ч1.З9

**Условие.** *(понимание)* Программа печатает `printf("%.17f\n", 0.1 + 0.2);` и выводит `0.30000000000000004`. Объясните, почему результат не равен ровно `0.3`, опираясь на §2.1.3. Как в таком случае корректно проверять «равенство» двух вещественных?

**Решение.** Причина не в `printf` и не в сложении, а в самом хранении чисел. `double` хранит значение как двоичную научную запись — мантисса, умноженная на степень двойки, с ограниченным числом разрядов мантиссы (§2.1.3). Точно представимы только дроби вида *m*/2ᵏ. Числа 0,1 и 0,2 к ним не относятся: в двоичной системе это бесконечные периодические дроби, ровно как 1/3 в десятичной. Поэтому уже до всякого сложения в памяти лежат не 0,1 и 0,2, а ближайшие представимые к ним числа. Сумма этих приближений даёт число, чуть большее, чем ближайшее представимое к 0,3, — и `%.17f` это показывает.

Показательно распечатать рядом сумму и сам литерал `0.3`:

```c
#include <stdio.h>

int main(void)
{
    double a = 0.1 + 0.2;
    double b = 0.3;
    printf("a       = %.17f\n", a);
    printf("b       = %.17f\n", b);
    printf("a == b  = %d\n", a == b);   // сравнение даёт 0 или 1 (§2.2.2)
    printf("a - b   = %.17f\n", a - b);
    return 0;
}
```

```
a       = 0.30000000000000004
b       = 0.29999999999999999
a == b  = 0
a - b   = 0.00000000000000006
```

Видно главное: `0.3` тоже хранится неточно (`0.29999999999999999`), и это *другое* приближение, чем сумма. Разница — около 5,6·10⁻¹⁷, то есть примерно в одну единицу последнего разряда мантиссы. Оператор `==` сравнивает числа побитово точно и честно отвечает «не равны».

Корректный приём: считать числа равными, если модуль их разности меньше выбранного порога (§2.1.3):

```
|a − b| < ε,   например ε = 1e-9
```

В нашем случае разность 5,6·10⁻¹⁷ меньше 1e-9, значит по такому критерию числа «равны с точностью до погрешности» — что и требуется. (Записать это одним условием позволяет оператор `&&`: `a - b < eps && b - a < eps`; сам оператор разбирается в §2.2.2 этой же главы, поэтому здесь мы ограничиваемся печатью разности.)

**Ответ:** ни 0,1, ни 0,2, ни 0,3 не представимы в двоичной плавающей арифметике точно; программа складывает их приближения и получает число, отличающееся от приближения к 0,3 на ~5,6·10⁻¹⁷ (`0.30000000000000004` против `0.29999999999999999`). Сравнивать такие значения оператором `==` бессмысленно — проверяют, что модуль разности меньше заранее выбранного порога ε.

**Вариант 2 (относительный порог вместо абсолютного).** Абсолютный порог ε привязан к масштабу чисел: 1e-9 разумно для величин порядка единицы, но для миллиардов погрешность самого представления уже больше него. Сравним:

```c
#include <stdio.h>

int main(void)
{
    double a = 0.1 + 0.2;
    double b = 0.3;
    double diff = a - b;              // может быть и отрицательной

    printf("diff         = %.17f\n", diff);
    printf("diff < 1e-9  = %d\n", diff < 1e-9);
    printf("-diff < 1e-9 = %d\n", -diff < 1e-9);

    // относительный порог: погрешность соизмеряется с величиной чисел
    double big1 = 1e9 + 0.1 + 0.2;
    double big2 = 1e9 + 0.3;
    printf("big diff       = %.17f\n", big1 - big2);
    printf("big rel. diff  = %.17f\n", (big1 - big2) / big2);
    return 0;
}
```

```
diff         = 0.00000000000000006
diff < 1e-9  = 1
-diff < 1e-9 = 1
big diff       = 0.00000011920928955
big rel. diff  = 0.00000000000000012
```

Для чисел порядка единицы абсолютная разность — 5,6·10⁻¹⁷, порог 1e-9 срабатывает. Для тех же слагаемых, сдвинутых на 10⁹, абсолютная разность выросла до 1,2·10⁻⁷ — **больше** порога 1e-9, и абсолютный критерий ошибочно объявил бы числа разными. Относительная же разность осталась крошечной: 1,2·10⁻¹⁶.

*Сравнение.* Абсолютный порог проще, не требует делений и корректен, когда заранее известен масштаб величин (проценты, координаты в известных пределах) — а ещё он единственный работает, если одно из чисел равно нулю, где относительный порог теряет смысл из-за деления. Относительный порог не зависит от масштаба и потому годится для чисел неизвестного порядка, но требует делить на одно из значений и не применим при нуле. Практика: для величин известного диапазона брать абсолютный порог, для произвольных — относительный (часто комбинируют оба).

### Г2.Ч1.З10

**Условие.** *(прикладное)* Заведите переменные для товара: `unsigned int quantity` (количество), `double price` (цена за штуку). Инициализируйте их любыми значениями, вычислите и напечатайте общую стоимость `quantity * price` с двумя знаками после точки (`%.2f`). Подумайте, почему количество разумно сделать беззнаковым.

**Решение.**

```c
#include <stdio.h>

int main(void)
{
    unsigned int quantity = 7;      // количество не бывает отрицательным
    double price = 249.99;          // цена за штуку
    double total = quantity * price;
    printf("quantity = %u\n", quantity);
    printf("price    = %.2f\n", price);
    printf("total    = %.2f\n", total);
    return 0;
}
```

```
quantity = 7
price    = 249.99
total    = 1749.93
```

Два момента в этой программе стоят разбора.

*Смешение типов в умножении.* `quantity` — целое беззнаковое, `price` — `double`. По правилам обычных арифметических преобразований (§2.2.5) операнды приводятся к более широкому типу: `quantity` превращается в `double`, умножение выполняется в вещественной арифметике, результат — `double`. Это как раз тот случай, когда неявное преобразование работает нам на пользу; ловушка §2.2.5 (`7 / 2` в `double`) возникает в обратной ситуации — когда *оба* операнда целые.

*Спецификаторы.* `quantity` имеет тип `unsigned int`, поэтому печатается через `%u`, а не `%d` (§2.1.2); `total` — `double`, печатается `%.2f` — два знака после точки, ровно как принято для денег.

**Ответ:** программа печатает `total = 1749.93` (7 × 249.99). Количество разумно сделать беззнаковым по двум причинам: отрицательное количество не имеет смысла в предметной области, поэтому тип документирует намерение и делает бессмысленное состояние невыразимым; кроме того, при том же размере беззнаковый тип даёт вдвое больший положительный диапазон (§2.1.2). Плата за это — осторожность с вычитанием: `quantity - 10` при `quantity = 7` даст не −3, а огромное число (§2.2.5, а также Г2.Ч2.З6).

### Г2.Ч1.З11

**Условие.** *(прикладное)* Напишите программу, которая по коду ASCII печатает символ и наоборот: задайте `char ch = 'z';`, напечатайте его код, затем задайте `int code = 100;` и напечатайте соответствующий символ (`%c`). Проверьте, какая буква соответствует коду 100.

**Решение.** Обе задачи решаются одним и тем же фактом: `char` — это число, а `%c` и `%d` лишь по-разному его печатают (§2.1.3).

```c
#include <stdio.h>

int main(void)
{
    char ch = 'z';
    printf("char '%c' -> code %d\n", ch, ch);

    int code = 100;
    printf("code %d -> char '%c'\n", code, code);
    return 0;
}
```

```
char 'z' -> code 122
code 100 -> char 'd'
```

Заметьте: в каждой строке `printf` одна и та же переменная передана дважды — под `%c` и под `%d`. Никакого преобразования между «символом» и «числом» в программе нет и не нужно, меняется только способ печати.

Проверим код 100 вручную. Строчные латинские буквы тоже идут подряд, `a` = 97 (это на 32 больше, чем `A` = 65). Тогда 100 − 97 = 3, то есть четвёртая буква алфавита — `d`. Совпадает с выводом. Аналогично `z` — 26-я буква: 97 + 25 = 122.

**Ответ:** `z` имеет код 122; коду 100 соответствует символ `d`. Программа печатает `char 'z' -> code 122` и `code 100 -> char 'd'`. Преобразование «код ↔ символ» не требует никаких действий: это одно и то же значение, напечатанное разными спецификаторами.

**Вариант 2 (арифметика от начала алфавита вместо готовых кодов).** Если помнить только то, что буквы идут подряд, конкретные числа 97 и 122 вообще не нужны — их можно не выписывать, а вычислить от символьного литерала:

```c
#include <stdio.h>

int main(void)
{
    // код буквы вычисляется от начала алфавита, без обращения к таблице ASCII
    char ch = 'a' + 25;          // 26-я буква латинского алфавита
    printf("char '%c' -> code %d\n", ch, ch);

    int code = 'a' + 3;          // четвёртая буква
    printf("code %d -> char '%c'\n", code, code);
    return 0;
}
```

```
char 'z' -> code 122
code 100 -> char 'd'
```

Вывод совпал с основным решением дословно.

*Сравнение.* Основное решение записывает коды прямо (`'z'`, `100`) — это короче и точнее, когда нужен конкретный, заранее известный символ. Запись через `'a' + n` не требует помнить или искать числовые коды и делает намерение явным («n-я буква алфавита»), а заодно не сломается на платформе, где `a` имеет иной код: формула опирается не на конкретное число 97, а лишь на то, что буквы идут подряд. (Нормативно стандарт гарантирует непрерывность кодов только для цифр `0`–`9`; для латинских букв это верно в ASCII и производных от неё кодировках — именно ими мы и пользуемся, §1.1.1.) Расплата — формула читается медленнее числа, и легко ошибиться на единицу (26-я буква — это `'a' + 25`, а не `'a' + 26`).

### Г2.Ч1.З12

**Условие.** *(прикладное исследование)* Соберите и запустите программу из примера 3 §2.1.3 (`0.1 + 0.2` с `%.17f`) для `double` и её вариант для `float` (`float x = 0.1f + 0.2f; printf("%.17f\n", x);`). Сравните погрешности и объясните, почему у `float` она больше.

**Решение.**

```c
#include <stdio.h>

int main(void)
{
    double d = 0.1 + 0.2;     // double: ~15-16 значащих цифр
    float  f = 0.1f + 0.2f;   // float:  ~7 значащих цифр
    printf("double: %.17f\n", d);
    printf("float : %.17f\n", f);
    printf("sizeof(float)=%zu sizeof(double)=%zu\n", sizeof(float), sizeof(double));
    return 0;
}
```

```
double: 0.30000000000000004
float : 0.30000001192092896
sizeof(float)=4 sizeof(double)=8
```

Сравним отклонения от точного значения 0,3:

| Тип | Напечатано | Отклонение от 0,3 | Первая неверная цифра |
|---|---|---|---|
| `double` | `0.30000000000000004` | ≈ 4·10⁻¹⁷ | 17-я |
| `float` | `0.30000001192092896` | ≈ 1,2·10⁻⁸ | 8-я |

Отношение погрешностей — около 3·10⁸, то есть примерно в 300 миллионов раз. Причина — разрядность мантиссы: оба типа хранят число как мантиссу, умноженную на степень двойки, но `float` занимает 4 байта, а `double` — 8 (что и подтверждает `sizeof` в выводе). На мантиссу в `float` приходится 24 двоичных разряда, в `double` — 53. Каждый лишний разряд вдвое уменьшает шаг между соседними представимыми числами, поэтому 29 дополнительных разрядов делают этот шаг у `double` в 2²⁹ ≈ 5·10⁸ раз мельче. Наблюдаемое отношение погрешностей того же порядка; точного совпадения и не должно быть — конкретная ошибка зависит от того, куда именно попало округление каждого из слагаемых. В десятичном выражении это ≈7 надёжных значащих цифр против ≈15–16 (§2.1.3).

Отдельно стоит заметить: печатать `float` с `%.17f` в принципе можно (в `printf` он всё равно расширяется до `double`, §2.3.1), но все цифры после восьмой — уже «шум» пересчёта, а не информация.

**Ответ:** `double` даёт `0.30000000000000004` (ошибка ~4·10⁻¹⁷), `float` — `0.30000001192092896` (ошибка ~1,2·10⁻⁸). Погрешность у `float` больше, потому что под мантиссу в нём отведено вдвое меньше битов (4 байта против 8, ≈24 значащих двоичных разряда против ≈53), а значит шаг между соседними представимыми числами гораздо крупнее. Практический вывод: `double` — тип по умолчанию для дробных вычислений, `float` берут только там, где важна экономия памяти.

## Часть 2.2. Операторы и выражения

### Г2.Ч2.З1

**Условие.** Вычислите (сначала на бумаге, потом проверьте программой): `17 / 5`, `17 % 5`, `17.0 / 5`, `(-17) / 5`, `(-17) % 5`. Для каждого укажите тип результата.

**Решение.** На бумаге рассуждаем так (§2.2.1).

- `17 / 5`: **оба операнда целые**, значит деление целочисленное — «сколько целых пятёрок помещается в 17». Три пятёрки дают 15, четвёртая не влезает → **3**, дробная часть отброшена, а не округлена (3,4 → 3, а не 4).
- `17 % 5`: остаток от того же деления: 17 − 3·5 = **2**.
- `17.0 / 5`: один операнд вещественный (`17.0` имеет тип `double`), поэтому по обычным арифметическим преобразованиям (§2.2.5) второй тоже становится `double`, и деление вещественное → **3.4**.
- `(-17) / 5`: с C99 деление усекает **к нулю**, а не «вниз»: −3,4 → **−3** (не −4).
- `(-17) % 5`: знак остатка совпадает со знаком делимого → **−2**. Проверка тождеством из §2.2.1: (−3)·5 + (−2) = −15 − 2 = −17 ✓.

Типы: у первых двух — `int` (оба операнда `int`), у третьего — `double`, у последних двух — `int`. Тип результата можно не угадывать, а измерить `sizeof`: 4 байта — `int`, 8 — `double` (§2.1.4).

```c
#include <stdio.h>

int main(void)
{
    printf("17 / 5    = %d\n", 17 / 5);
    printf("17 %% 5    = %d\n", 17 % 5);
    printf("17.0 / 5  = %g\n", 17.0 / 5);
    printf("(-17) / 5 = %d\n", (-17) / 5);
    printf("(-17) %% 5 = %d\n", (-17) % 5);
    // sizeof показывает тип результата: 4 байта = int, 8 байт = double
    printf("sizeof(17 / 5)   = %zu\n", sizeof(17 / 5));
    printf("sizeof(17.0 / 5) = %zu\n", sizeof(17.0 / 5));
    // проверка тождества (a / b) * b + (a % b) == a
    printf("check: %d\n", ((-17) / 5) * 5 + ((-17) % 5));
    return 0;
}
```

```
17 / 5    = 3
17 % 5    = 2
17.0 / 5  = 3.4
(-17) / 5 = -3
(-17) % 5 = -2
sizeof(17 / 5)   = 4
sizeof(17.0 / 5) = 8
check: -17
```

**Ответ:**

| Выражение | Значение | Тип результата |
|---|---|---|
| `17 / 5` | 3 | `int` |
| `17 % 5` | 2 | `int` |
| `17.0 / 5` | 3.4 | `double` |
| `(-17) / 5` | −3 | `int` |
| `(-17) % 5` | −2 | `int` |

Последняя строка вывода (`check: -17`) подтверждает согласованность пары `/` и `%`: `(a / b) * b + (a % b)` восстанавливает `a` и для отрицательных чисел.

**Типичная неверная попытка.** Считать, что `(-17) / 5` равно −4 «потому что округление вниз». Округления здесь нет вообще: дробная часть просто отбрасывается, то есть результат сдвигается **к нулю**. Ожидание −4 приходит из математического деления с остатком (где остаток неотрицателен) и из языков вроде Python, где `//` действительно округляет вниз, а `-17 % 5` даёт 3.

```
a *= 2  -> 44
a /= 4  -> 11
```

### Г2.Ч2.З3

**Условие.** Не запуская, предскажите вывод, затем проверьте:

```c
int a = 5;
int post = a++;
int b = 5;
int pre = ++b;
printf("%d %d %d %d\n", a, post, b, pre);
```

**Решение.** Обе формы одинаково увеличивают переменную на 1; различаются они тем, какое значение даёт само выражение (§2.2.3):

- `post = a++` — постфикс: выражение `a++` возвращает **старое** значение `a` (5), затем `a` становится 6. Итог: `a` = 6, `post` = 5.
- `pre = ++b` — префикс: сначала `b` становится 6, затем выражение возвращает **новое** значение. Итог: `b` = 6, `pre` = 6.

Аргументы `printf` идут в порядке `a, post, b, pre`, значит предсказание: `6 5 6 6`.

```c
#include <stdio.h>

int main(void)
{
    int a = 5;
    int post = a++;   // post получает старое значение, a становится 6
    int b = 5;
    int pre = ++b;    // b становится 6, pre получает новое значение
    printf("%d %d %d %d\n", a, post, b, pre);
    return 0;
}
```

```
6 5 6 6
```

**Ответ:** `6 5 6 6`. Обе переменные после инкремента равны 6 — разница только в том, что успело попасть в `post` и `pre`: постфикс отдаёт значение *до* увеличения, префикс — *после*.

**Типичная неверная попытка.** Прочитать `post = a++` как «`post` станет равным 6, ведь `a` увеличивается». Увеличение действительно происходит, и `a` в момент печати равно 6 — но в `post` попало то значение, которое выражение вернуло *в момент присваивания*, то есть 5. Практическое правило: если результат инкремента используется, пишите его отдельной инструкцией (`a = a + 1;`) — читателю кода не придётся вспоминать эти тонкости.

### Г2.Ч2.З4

**Условие.** Расставьте скобки согласно приоритету и вычислите: а) `2 + 3 * 4 - 1`; б) `10 % 3 + 2`; в) `1 + 2 < 4`; г) `20 / 4 / 5`. Проверьте программой.

**Решение.** Пользуемся таблицей приоритетов §2.2.4: `*`, `/`, `%` связывают сильнее `+` и `-`; арифметика сильнее сравнений; операторы одного приоритета левоассоциативны.

а) `2 + 3 * 4 - 1` → умножение первым, затем `+` и `-` слева направо: `((2 + (3 * 4)) - 1)` = (2 + 12) − 1 = **13**.

б) `10 % 3 + 2` → `%` того же приоритета, что `*` и `/`, значит сильнее `+`: `((10 % 3) + 2)` = 1 + 2 = **3**.

в) `1 + 2 < 4` → сложение сильнее сравнения: `((1 + 2) < 4)` = 3 < 4 → **1** (истина обозначается единицей, §2.2.2).

г) `20 / 4 / 5` → два оператора одного приоритета, левая ассоциативность: `((20 / 4) / 5)` = 5 / 5 = **1**. (Если бы группировка была правой, вышло бы 20 / (4 / 5) = 20 / 0 — деление на ноль; то, что программа не падает, само по себе подтверждает левую ассоциативность.)

```c
#include <stdio.h>

int main(void)
{
    printf("a) 2 + 3 * 4 - 1 = %d\n", 2 + 3 * 4 - 1);
    printf("b) 10 %% 3 + 2    = %d\n", 10 % 3 + 2);
    printf("c) 1 + 2 < 4     = %d\n", 1 + 2 < 4);
    printf("d) 20 / 4 / 5    = %d\n", 20 / 4 / 5);
    // те же выражения с явными скобками - результат обязан совпасть
    printf("a') (2 + (3 * 4)) - 1 = %d\n", (2 + (3 * 4)) - 1);
    printf("b') (10 %% 3) + 2      = %d\n", (10 % 3) + 2);
    printf("c') (1 + 2) < 4       = %d\n", (1 + 2) < 4);
    printf("d') (20 / 4) / 5      = %d\n", (20 / 4) / 5);
    return 0;
}
```

```
a) 2 + 3 * 4 - 1 = 13
b) 10 % 3 + 2    = 3
c) 1 + 2 < 4     = 1
d) 20 / 4 / 5    = 1
a') (2 + (3 * 4)) - 1 = 13
b') (10 % 3) + 2      = 3
c') (1 + 2) < 4       = 1
d') (20 / 4) / 5      = 1
```

Каждая пара строк совпала — значит скобки расставлены так же, как их «мысленно расставляет» компилятор.

**Ответ:** а) `((2 + (3 * 4)) - 1)` = 13; б) `((10 % 3) + 2)` = 3; в) `((1 + 2) < 4)` = 1; г) `((20 / 4) / 5)` = 1. Приём проверки универсален: выписать выражение со скобками, прогнать обе версии и сравнить — если результаты разошлись, приоритет понят неверно.

### Г2.Ч2.З5

**Условие.** *(понимание)* Программа должна печатать среднее двух целых `a = 5`, `b = 8`. Автор написал `double avg = (a + b) / 2;` и получил `6`, а не `6.5`. Объясните причину и предложите две разных исправленных версии.

**Решение.** Разбираем строку по шагам, справа налево от знака `=` (§2.2.5):

1. `a + b` — оба операнда `int`, сумма 13 типа `int`.
2. `13 / 2` — **оба операнда целые**, значит деление целочисленное: 6 (дробная часть отброшена).
3. Только теперь готовый результат 6 присваивается переменной типа `double` и превращается в 6.0.

Ошибка в том, что тип *переменной слева* не влияет на то, как вычисляется выражение справа. Преобразование в `double` происходит слишком поздно — когда терять уже нечего.

Обе исправленные версии делают вещественным один из **операндов деления**:

```c
#include <stdio.h>

int main(void)
{
    int a = 5, b = 8;

    double avg_bad = (a + b) / 2;        // ошибка: деление целочисленное
    printf("bad          = %g\n", avg_bad);

    double avg_fix1 = (a + b) / 2.0;     // вариант 1: делитель вещественный
    printf("fix 1 (2.0)  = %g\n", avg_fix1);

    double avg_fix2 = (double)(a + b) / 2;  // вариант 2: приведение делимого
    printf("fix 2 (cast) = %g\n", avg_fix2);

    return 0;
}
```

```
bad          = 6
fix 1 (2.0)  = 6.5
fix 2 (cast) = 6.5
```

Чем версии отличаются на практике. Литерал `2.0` короче и читается как «делим на два, результат дробный» — годится, когда делитель записан константой прямо в коде. Приведение `(double)` работает и тогда, когда делитель — переменная (`(double)(a + b) / n`), где заменить его литералом невозможно; зато оно длиннее и требует помнить, что приводить нужно **до** деления: `(double)((a + b) / 2)` — это снова 6.0, приведение применено к уже испорченному результату.

**Ответ:** причина — целочисленное деление: `(a + b)` даёт `int` 13, `13 / 2` вычисляется целиком в целой арифметике и даёт 6, а присваивание в `double` лишь превращает готовую шестёрку в 6.0. Две исправленные версии: `double avg = (a + b) / 2.0;` и `double avg = (double)(a + b) / 2;` — обе печатают 6.5. Общее правило: чтобы получить дробный результат, вещественным должен стать **операнд** деления, а не переменная, в которую записывают итог.

### Г2.Ч2.З6

**Условие.** *(понимание)* Что напечатает и почему:

```c
unsigned int a = 3;
unsigned int b = 5;
printf("%u\n", a - b);
```

Как изменить типы, чтобы получить ожидаемое `-2`?

**Решение.** Оба операнда — `unsigned int`, поэтому и вычитание выполняется в беззнаковой арифметике. Отрицательных значений у такого типа нет вовсе, и стандарт задаёт результат однозначно: он берётся **по модулю 2ᴺ** (§2.2.5). При `N` = 32:

```
3 − 5 = −2  →  −2 + 2³² = 4294967296 − 2 = 4294967294
```

Это не «мусор» и не ошибка вычислений: беззнаковая арифметика работает как часовой циферблат на 2³² делений, и шаг назад от нуля приводит к максимуму (§2.2.5, интуиция).

```c
#include <stdio.h>

int main(void)
{
    unsigned int a = 3;
    unsigned int b = 5;
    printf("%u\n", a - b);                 // заворачивание по модулю 2^32

    int sa = 3;
    int sb = 5;
    printf("%d\n", sa - sb);               // знаковые: обычное -2

    printf("%d\n", (int)(a - b));          // то же число, но как знаковое
    printf("sizeof(a - b) = %zu\n", sizeof(a - b));
    return 0;
}
```

```
4294967294
-2
-2
sizeof(a - b) = 4
```

Строка `sizeof(a - b) = 4` подтверждает, что переполнения в смысле «не поместилось» не было: результат занимает те же 4 байта, просто эти 32 бита прочитаны как беззнаковое число.

**Ответ:** программа печатает `4294967294` = 2³² − 2. Вычитание беззнаковых даёт результат по модулю 2ᴺ, а `%u` печатает эти биты как неотрицательное число. Чтобы получить `-2`, нужно объявить переменные знаковыми и печатать `%d`:

```c
int a = 3;
int b = 5;
printf("%d\n", a - b);      /* -2 */
```

**Вариант 2 (оставить типы, привести результат).** Если по смыслу задачи переменные обязаны быть беззнаковыми (счётчики, размеры), но разность нужна со знаком, её приводят к знаковому типу в момент использования:

```c
printf("%d\n", (int)(a - b));   /* -2 */
```

Вывод — `-2`, третья строка прогона выше.

*Сравнение.* Смена типов на `int` — решение «в корне»: знак сохраняется на всех промежуточных шагах, и любые дальнейшие сравнения и вычитания ведут себя привычно; цена — теряется вдвое больший положительный диапазон и уходит документирующий смысл «эта величина неотрицательна». Приведение результата оставляет типы данных прежними и потому годится, когда переменные пришли извне (например, `sizeof` возвращает беззнаковый `size_t`), но оно опасно тем, что чинит одно конкретное место: стоит написать рядом ещё одно `a - b` без приведения — и заворачивание вернётся. Кроме того, приведение опирается на то, что число помещается в знаковый тип: `(int)` от разности двух больших беззнаковых значений — уже реализационно-определённое поведение. Практика: тип выбирают по смыслу величины, а приведение используют как локальную заплату там, где тип менять нельзя.

### Г2.Ч2.З7

**Условие.** *(понимание)* Объясните, почему `1 <= x <= 10` не проверяет принадлежность диапазону. Разберите вычисление по шагам для `x = 100` и запишите правильное условие.

**Решение.** В C нет «цепочек сравнений»: `<=` — обычный бинарный оператор, левоассоциативный (§2.2.4). Значит запись читается как `(1 <= x) <= 10`. Разбираем для `x = 100`:

1. `1 <= x` → `1 <= 100` → истина → значение **1** (сравнение даёт ровно 0 или 1, §2.2.2).
2. Полученная единица подставляется в оставшееся сравнение: `1 <= 10` → истина → **1**.

Результат — 1, то есть «x в диапазоне», хотя 100 в диапазон 1…10 не входит. Хуже того, результат будет 1 для **любого** `x`: первый шаг всегда даёт 0 или 1, и оба эти числа не превосходят 10. Условие бессмысленно — оно тождественно истинно.

Правильная запись — два самостоятельных сравнения, соединённых `&&` (§2.2.2): `(1 <= x) && (x <= 10)`.

```c
#include <stdio.h>

int main(void)
{
    int x = 100;
    printf("1 <= x            = %d\n", 1 <= x);
    printf("(1 <= x) <= 10    = %d\n", (1 <= x) <= 10);
    printf("1 <= x <= 10      = %d\n", 1 <= x <= 10);       // тот же результат
    printf("(1<=x) && (x<=10) = %d\n", (1 <= x) && (x <= 10));

    int y = 7;
    printf("y=7:  1 <= y <= 10      = %d\n", 1 <= y <= 10);
    printf("y=7:  (1<=y) && (y<=10) = %d\n", (1 <= y) && (y <= 10));

    int z = -5;
    printf("z=-5: 1 <= z <= 10      = %d\n", 1 <= z <= 10);
    printf("z=-5: (1<=z) && (z<=10) = %d\n", (1 <= z) && (z <= 10));
    return 0;
}
```

```
1 <= x            = 1
(1 <= x) <= 10    = 1
1 <= x <= 10      = 1
(1<=x) && (x<=10) = 0
y=7:  1 <= y <= 10      = 1
y=7:  (1<=y) && (y<=10) = 1
z=-5: 1 <= z <= 10      = 1
z=-5: (1<=z) && (z<=10) = 0
```

Три значения `x` = 100, 7, −5 дают у «цепочки» один и тот же ответ 1, а у правильного условия — 0, 1, 0. Совпадение с ожиданием только на `x = 7` — и это ровно та ситуация, когда ошибка не замечается при беглом тестировании.

Компилятор, кстати, предупреждает — и очень точно:

```
p2z7.c:8:47: warning: comparison of constant '10' with boolean expression is always true [-Wbool-compare]
    8 |     printf("1 <= x <= 10      = %d\n", 1 <= x <= 10);       // тот же результат
      |                                               ^~
p2z7.c:8:42: warning: comparisons like 'X<=Y<=Z' do not have their mathematical meaning [-Wparentheses]
    8 |     printf("1 <= x <= 10      = %d\n", 1 <= x <= 10);       // тот же результат
      |                                        ~~^~~~
```

Вторая формулировка — «сравнения вида `X<=Y<=Z` не имеют своего математического смысла» — прямо про эту ошибку.

**Ответ:** `1 <= x <= 10` вычисляется как `(1 <= x) <= 10`: первое сравнение даёт 0 или 1, и это число сравнивается с 10 — что истинно всегда. Для `x = 100`: `1 <= 100` → 1, затем `1 <= 10` → 1, итог 1, хотя 100 вне диапазона. Правильное условие: `(1 <= x) && (x <= 10)`. Компилятор с `-Wall` ловит такую запись предупреждением `-Wparentheses`.

### Г2.Ч2.З8

**Условие.** *(понимание)* Найдите ошибку и объясните её последствие: программист хотел прибавить 3 и написал `x =+ 3;`. Что реально сделает эта строка при `x = 10`? Как правильно?

**Решение.** Составной оператор пишется одним «слипшимся» знаком `+=`. Запись `=+` — это два отдельных оператора: присваивание `=` и **унарный плюс** `+` перед литералом 3 (§2.2.3). Компилятор читает строку как `x = (+3);`, то есть просто записывает в `x` тройку, стирая прежнее значение 10.

Коварство ошибки в том, что программа остаётся полностью корректной с точки зрения языка: типы согласованы, синтаксис верный, никакого предупреждения `-Wall -Wextra` не выдаёт. Ошибка чисто смысловая, и найти её можно только чтением кода или по неверному результату.

```c
#include <stdio.h>

int main(void)
{
    int x = 10;
    x =+ 3;              // читается как x = (+3)
    printf("after x =+ 3 : x = %d\n", x);

    int y = 10;
    y += 3;              // правильная запись
    printf("after y += 3 : y = %d\n", y);
    return 0;
}
```

```
after x =+ 3 : x = 3
after y += 3 : y = 13
```

**Ответ:** строка `x =+ 3;` присваивает `x` значение `+3`, то есть при `x = 10` переменная станет равна **3**, а не 13. Правильно — `x += 3;` (знаки без пробела между ними). Компилятор об ошибке не сообщает: обе записи допустимы, различается только смысл. Симметричная опечатка `x =- 3;` присвоит −3, что ещё больше похоже на «вычитание» и потому опаснее.

### Г2.Ч2.З9

**Условие.** *(понимание)* Дано `int x = 5;`. Определите значения выражений и итоговое значение `x` для каждого *по отдельности* (каждый пункт стартует с `x = 5`): а) `x++`; б) `++x`; в) `x-- + 1`; г) `!x`. Поясните разницу а) и б).

**Решение.** По §2.2.3 и §2.2.2:

а) `x++` — постфиксный инкремент: значение выражения — **старое** `x`, то есть 5; после выполнения `x` = 6.

б) `++x` — префиксный инкремент: сначала `x` становится 6, значение выражения — **новое**, то есть 6; после выполнения `x` = 6.

в) `x-- + 1` — постфиксный декремент даёт старое значение 5, к нему прибавляется 1 → значение выражения **6**; переменная при этом уменьшилась: `x` = 4. Забавное совпадение: выражение равно 6, и в пункте б) тоже 6, но `x` там 6, а здесь 4.

г) `!x` — логическое отрицание: `x` = 5 — ненулевое, то есть «истина», её отрицание — «ложь» → значение **0**. Оператор `!` переменную **не меняет**: `x` остаётся 5.

```c
#include <stdio.h>

int main(void)
{
    int x;
    int v;

    // каждый пункт стартует заново с x = 5;
    // значение выражения сохраняем в v отдельной инструкцией,
    // чтобы не менять и не читать x дважды в одном выражении
    x = 5;  v = x++;      printf("a) x++     : value = %d, x = %d\n", v, x);
    x = 5;  v = ++x;      printf("b) ++x     : value = %d, x = %d\n", v, x);
    x = 5;  v = x-- + 1;  printf("c) x-- + 1 : value = %d, x = %d\n", v, x);
    x = 5;  v = !x;       printf("d) !x      : value = %d, x = %d\n", v, x);
    return 0;
}
```

```
a) x++     : value = 5, x = 6
b) ++x     : value = 6, x = 6
c) x-- + 1 : value = 6, x = 4
d) !x      : value = 0, x = 5
```

Обратите внимание на устройство проверочной программы: значение выражения сохраняется в отдельную переменную `v` **до** печати. Написать `printf("%d %d\n", x++, x)` было бы ошибкой — в одном выражении переменная и изменяется, и читается, а это неопределённое поведение (§2.2.3, подводные камни). Правило «меняй переменную не более одного раза за выражение» распространяется и на диагностический код.

**Ответ:**

| Пункт | Значение выражения | `x` после |
|---|---|---|
| а) `x++` | 5 | 6 |
| б) `++x` | 6 | 6 |
| в) `x-- + 1` | 6 | 4 |
| г) `!x` | 0 | 5 |

Разница а) и б): переменную обе формы меняют одинаково (после каждой `x` = 6), но как *выражение* постфикс отдаёт значение **до** увеличения, а префикс — **после**. Если результат выражения нигде не используется (отдельная инструкция `x++;`), формы неразличимы.

### Г2.Ч2.З10

**Условие.** *(прикладное)* Напишите программу, которая переводит заданное число секунд (`int total = 3725;`) в часы, минуты и секунды, используя `/` и `%`, и печатает в формате `1 h 2 m 5 s`.

**Решение.** Пара «целочисленное деление и остаток» ровно для этого и предназначена (§2.2.1): деление отвечает «сколько целых единиц следующего разряда получилось», остаток — «сколько осталось».

- часы: `total / 3600` (в часе 3600 секунд) → 3725 / 3600 = 1;
- остаток секунд после часов: `total % 3600` = 125;
- минуты: этот остаток делим на 60 → 125 / 60 = 2;
- секунды: `total % 60` = 3725 − 62·60 = 5.

```c
#include <stdio.h>

int main(void)
{
    int total = 3725;

    int hours = total / 3600;           // сколько полных часов
    int minutes = total % 3600 / 60;    // из остатка - полные минуты
    int seconds = total % 60;           // остаток от целых минут

    printf("%d h %d m %d s\n", hours, minutes, seconds);
    return 0;
}
```

```
1 h 2 m 5 s
```

Выражение `total % 3600 / 60` обходится без скобок: `%` и `/` имеют одинаковый приоритет и левоассоциативны, значит оно читается как `(total % 3600) / 60` — именно то, что нужно (§2.2.4). Скобки здесь всё же уместны для читателя.

Проверка обратным счётом: 1·3600 + 2·60 + 5 = 3600 + 120 + 5 = 3725 ✓.

**Ответ:** программа печатает `1 h 2 m 5 s`. Схема универсальна: делением получаем старший разряд, остатком — то, что нужно разложить дальше.

**Вариант 2 (последовательное «сдирание» младших разрядов).** Вместо того чтобы каждый раз работать с исходным числом и подбирать константы 3600 и 60, можно двигаться снизу вверх, деля на 60 и сохраняя частное:

```c
#include <stdio.h>

int main(void)
{
    int total = 3725;

    int seconds = total % 60;    // младший разряд
    int rest = total / 60;       // всего полных минут
    int minutes = rest % 60;
    int hours = rest / 60;

    printf("%d h %d m %d s\n", hours, minutes, seconds);
    return 0;
}
```

```
1 h 2 m 5 s
```

*Сравнение.* Первый способ вычисляет каждую величину независимо, прямо из `total`, — его удобно читать «по определению» (часы это столько-то секунд), и любую строку можно взять отдельно. Плата — приходится вводить составную константу 3600 и правильно комбинировать `%` и `/` в одном выражении, а при добавлении суток появится ещё 86400. Второй способ использует одну-единственную константу 60 и повторяет одинаковый шаг «остаток — частное»; он лучше масштабируется (добавить сутки — это ещё одна пара строк с делением на 24) и меньше зависит от арифметики в уме, но каждая величина зависит от предыдущей, так что порядок строк менять нельзя. Для двух-трёх разрядов равноценно; для длинных разложений (сутки, недели) второй заметно аккуратнее.

### Г2.Ч2.З11

**Условие.** *(прикладное)* Для `double c = 36.6;` (температура в Цельсиях) вычислите Фаренгейты по формуле `F = C * 9 / 5 + 32` и напечатайте с одним знаком после точки. Затем намеренно напишите `C * (9 / 5) + 32` и объясните, почему результат изменился.

**Решение.**

```c
#include <stdio.h>

int main(void)
{
    double c = 36.6;

    double f = c * 9 / 5 + 32;        // 9 и 5 попадают в вещественное выражение
    printf("C = %.1f -> F = %.1f\n", c, f);

    double wrong = c * (9 / 5) + 32;  // 9 / 5 вычисляется как целое = 1
    printf("wrong: C * (9 / 5) + 32 = %.1f\n", wrong);

    printf("9 / 5 = %d\n", 9 / 5);
    return 0;
}
```

```
C = 36.6 -> F = 97.9
wrong: C * (9 / 5) + 32 = 68.6
9 / 5 = 1
```

Почему версии расходятся. В первой формуле операторы `*` и `/` одного приоритета и левоассоциативны, поэтому она читается как `((c * 9) / 5) + 32` (§2.2.4). Умножение выполняется первым: `c` — `double`, значит целое 9 преобразуется в `double`, и произведение 329,4 — вещественное. Дальше деление на 5 идёт уже в вещественной арифметике: 65,88. Плюс 32 → 97,88 → печать с одним знаком даёт 97.9.

Во второй формуле скобки заставили вычислить `9 / 5` **до** всякого участия `c`. Оба операнда — целые литералы, деление целочисленное, результат 1 (последняя строка вывода подтверждает). Дальше 36,6 · 1 + 32 = 68,6. Коэффициент 1,8 превратился в 1 — ошибка около 30 градусов.

Мораль: скобки не «просто уточняют» порядок — они меняют типы промежуточных значений. Целые литералы, изолированные скобками от вещественных переменных, считаются в целой арифметике.

**Ответ:** правильная формула даёт `F = 97.9`; версия `c * (9 / 5) + 32` печатает `68.6`, потому что скобки выделили целочисленное деление `9 / 5`, равное 1 (дробная часть отброшена), и множитель 1,8 подменился единицей. Надёжная запись — сделать литералы вещественными: `c * 9.0 / 5` или `c * 1.8`.

**Вариант 2 (свернуть дробь в вещественный коэффициент).** Если записать 9/5 сразу как 1.8, целочисленному делению просто негде возникнуть:

```c
#include <stdio.h>

int main(void)
{
    double c = 36.6;

    double f1 = c * 9 / 5 + 32;   // дробь как деление
    double f2 = c * 1.8 + 32;     // дробь свёрнута в вещественный коэффициент

    printf("c * 9 / 5 + 32 = %.1f\n", f1);
    printf("c * 1.8   + 32 = %.1f\n", f2);
    printf("difference     = %.17f\n", f1 - f2);
    return 0;
}
```

```
c * 9 / 5 + 32 = 97.9
c * 1.8   + 32 = 97.9
difference     = 0.00000000000000000
```

На этих данных оба выражения дали побитово одинаковый результат — разность в точности 0.

*Сравнение.* Форма `c * 9 / 5` дословно повторяет учебную формулу (девять пятых), поэтому её легко сверить с источником; зато в ней два деления смысла «дробь» и «операция» слиты, и достаточно неудачных скобок или замены `c` на целую переменную, чтобы всё выражение молча стало целочисленным. Форма `c * 1.8` устойчива к этому: коэффициент — вещественный литерал, и целочисленное деление в выражении отсутствует как класс. Её недостаток — числовой коэффициент придётся пересчитывать вручную, и для дробей вроде 1/3 точной десятичной записи попросту нет (0.333… придётся округлить, внеся собственную погрешность), тогда как деление 1.0/3 даёт наилучшее представимое значение. Практика: сворачивать в коэффициент имеет смысл, когда дробь точно представима (как 1,8), и оставлять деление — когда нет.

### Г2.Ч2.З12

**Условие.** *(прикладное исследование)* Экспериментально нащупайте границу `int`: напечатайте `INT_MAX` (из `<limits.h>`), затем результат `INT_MAX + 1` при знаковом `int`. Опишите, что вывелось. Почему на такой результат нельзя полагаться (сошлитесь на §2.2.5)? Повторите с `unsigned int` и `UINT_MAX + 1` — чем принципиально отличается этот случай?

**Решение.**

```c
#include <stdio.h>
#include <limits.h>

int main(void)
{
    int big = INT_MAX;
    printf("INT_MAX      = %d\n", big);
    big = big + 1;                 /* знаковое переполнение: неопределённое поведение */
    printf("INT_MAX + 1  = %d\n", big);

    unsigned int ubig = UINT_MAX;
    printf("UINT_MAX     = %u\n", ubig);
    ubig = ubig + 1;               /* определено: по модулю 2^32 */
    printf("UINT_MAX + 1 = %u\n", ubig);
    return 0;
}
```

Вывод на эталонной машине:

```
INT_MAX      = 2147483647
INT_MAX + 1  = -2147483648
UINT_MAX     = 4294967295
UINT_MAX + 1 = 0
```

Внешне обе строки выглядят одинаково — «дошли до максимума, следующий шаг привёл к минимуму». Но природа у этих двух результатов совершенно разная.

*Беззнаковый случай определён.* Для беззнаковых типов стандарт прямо предписывает арифметику по модулю 2ᴺ (§2.2.5). `UINT_MAX + 1` = 2³² по модулю 2³² = 0 — это гарантированный результат на любой платформе и при любой оптимизации. На такое поведение можно опираться (им пользуются, например, хеш-функции и счётчики с циклическим переполнением).

*Знаковый случай — неопределённое поведение.* Полученное `-2147483648` не является «правильным ответом»: стандарт не задаёт никакого ответа вовсе. Компилятор вправе исходить из того, что знакового переполнения в корректной программе не бывает, и оптимизировать код на этом допущении — вплоть до выбрасывания проверок вида «если стало отрицательным». Тот же исходник с другими флагами оптимизации, на другом компиляторе или в другом окружении может дать иной результат, а под `-fsanitize=undefined` — аварийно завершиться с диагностикой (глава 13). Наблюдённое «заворачивание в `INT_MIN`» — лишь то, что выдал конкретный прогон, а не свойство языка.

Что важно и в самом эксперименте: чтобы переполнение вообще случилось во время выполнения, здесь используется переменная (`big = big + 1`). Если написать переполнение прямо в константном выражении, компилятор посчитает его сам — ещё на этапе компиляции и с предупреждением:

```c
printf("%d\n", INT_MAX + 1);   /* переполнение прямо в константном выражении */
```

```
p2z12b.c: In function 'main':
p2z12b.c:6:28: warning: integer overflow in expression of type 'int' results in '-2147483648' [-Woverflow]
    6 |     printf("%d\n", INT_MAX + 1);   /* переполнение прямо в константном выражении */
      |                            ^
```

**Ответ:** на эталонной машине `INT_MAX` = 2147483647, а `INT_MAX + 1` напечаталось как −2147483648; `UINT_MAX` = 4294967295, `UINT_MAX + 1` = 0. Принципиальная разница: беззнаковое переполнение **определено** стандартом (результат по модулю 2ᴺ) и воспроизводимо везде, а знаковое — **неопределённое поведение** (§2.2.5), поэтому увиденное число ничего не гарантирует: оно может измениться от флагов оптимизации, версии компилятора или платформы, и строить на нём логику программы нельзя.

**Вариант 2 (поймать переполнение до запуска — константным выражением).** Вместо того чтобы наблюдать результат во время выполнения, можно заставить компилятор вычислить выражение самому: тогда о переполнении сообщается диагностикой `-Woverflow` (текст выше), и оно обнаруживается ещё до запуска.

*Сравнение.* Прогон с переменной показывает, что реально произошло в машинной арифметике, и потому нагляднее в учебном эксперименте; но он же и опаснее педагогически — увиденное число легко принять за правило. Константная форма ничего не показывает про выполнение, зато демонстрирует важное практическое обстоятельство: значительная часть переполнений в реальном коде — это константы, и компилятор ловит их бесплатно, если читать предупреждения. Полноценная же защита от знакового переполнения — не эксперимент, а проверка **до** операции (не «сложил и посмотрел», а «влезет ли») либо санитайзер `-fsanitize=undefined` в отладочной сборке; к этому мы вернёмся в главе 13.

## Часть 2.3. Консольный ввод-вывод

Программы этой части читают данные, поэтому запускались с подачей ввода из конвейера: `printf '7 4\n' | ./prog`. Так проверка воспроизводима и не зависит от того, что набрано с клавиатуры.

### Г2.Ч3.З1

**Условие.** Напишите программу, печатающую целое `1234` четырьмя способами: как есть (`%d`), в поле ширины 8 с выравниванием вправо, влево и с ведущими нулями. Прокомментируйте вывод каждой строки.

**Решение.** Между `%` и буквой спецификатора помещаются флаги и ширина поля (§2.3.1): `%8d` — поле шириной 8 с выравниванием вправо (по умолчанию), `%-8d` — влево, `%08d` — дополнение нулями. Квадратные скобки в строке формата поставлены нарочно: без них границы поля не видны.

```c
#include <stdio.h>

int main(void)
{
    int n = 1234;
    printf("[%d]\n", n);      // как есть
    printf("[%8d]\n", n);     // ширина 8, выравнивание вправо
    printf("[%-8d]\n", n);    // ширина 8, выравнивание влево
    printf("[%08d]\n", n);    // ширина 8, дополнение нулями
    return 0;
}
```

```
[1234]
[    1234]
[1234    ]
[00001234]
```

Построчно:

- `[1234]` — без указания ширины поле равно длине самого числа, четыре знака.
- `[    1234]` — восемь позиций, число прижато вправо, слева четыре пробела (8 − 4).
- `[1234    ]` — те же восемь позиций, но флаг `-` прижимает число влево, пробелы уходят вправо.
- `[00001234]` — ширина та же, а свободные позиции заполнены нулями, а не пробелами.

Важное свойство ширины: она задаёт **минимум**, а не максимум. Если бы число было длиннее восьми знаков, поле просто расширилось бы — `printf` никогда не обрезает целое ради ширины поля.

**Ответ:** выводятся четыре строки — `[1234]`, `[    1234]`, `[1234    ]`, `[00001234]`. Ширина поля выравнивает столбцы в таблицах (все числа занимают одинаковое место), флаг `-` меняет сторону выравнивания, флаг `0` заполняет пустоту нулями — так печатают, например, время `07:05` или номера с ведущими нулями.

### Г2.Ч3.З2

**Условие.** Напечатайте число `3.14159265` с точностью 0, 2 и 5 знаков после точки (`%.0f`, `%.2f`, `%.5f`) и в научной форме (`%e`). Запишите результаты.

**Решение.** Точность записывается после точки в спецификаторе и означает число знаков **после** десятичной точки (§2.3.1). Знак `%%` в строке формата печатает сам символ процента — он понадобился, чтобы показать в выводе имя спецификатора.

```c
#include <stdio.h>

int main(void)
{
    double x = 3.14159265;
    printf("%%.0f -> [%.0f]\n", x);
    printf("%%.2f -> [%.2f]\n", x);
    printf("%%.5f -> [%.5f]\n", x);
    printf("%%e   -> [%e]\n", x);
    return 0;
}
```

```
%.0f -> [3]
%.2f -> [3.14]
%.5f -> [3.14159]
%e   -> [3.141593e+00]
```

Что видно из результатов.

- `%.0f` печатает **округлённое** значение без дробной части и **без точки**: `3`. Это округление, а не усечение: для 3,7 та же запись дала бы `4` — в отличие от приведения `(int)3.7`, которое даёт 3 (§2.2.5). Печать и преобразование типа — разные операции.
- `%.2f` → `3.14`: третья цифра (1) меньше пяти, округление вниз.
- `%.5f` → `3.14159`: обрезано ровно после пяти знаков, шестая цифра — 2, округления вверх не произошло.
- `%e` печатает мантиссу с шестью знаками после точки по умолчанию и порядок из двух цифр со знаком: `3.141593e+00`, то есть 3,141593 · 10⁰. Мантисса здесь округлена до шести знаков (`…59265` → `…593`).

**Ответ:** `[3]`, `[3.14]`, `[3.14159]`, `[3.141593e+00]`. Точность управляет только *печатью*: само значение в переменной не меняется, и все четыре строки выведены из одного и того же `double`.

### Г2.Ч3.З3

**Условие.** Напишите программу, которая читает одно целое число (`scanf("%d", …)`) и печатает его квадрат. Проверьте, подав число через `echo`.

**Решение.** Три обязательных элемента ввода из §2.3.2: переменная инициализирована, имя передано с `&`, возврат `scanf` не выброшен, а показан.

```c
#include <stdio.h>

int main(void)
{
    int n = 0;                        // инициализация: при неудачном вводе не будет мусора
    printf("Enter an integer: ");
    int got = scanf("%d", &n);        // & - адрес переменной
    printf("read %d items\n", got);
    printf("%d * %d = %d\n", n, n, n * n);
    return 0;
}
```

```
gcc -std=c17 -Wall -Wextra square.c -o square
printf '12\n' | ./square
```

```
Enter an integer: read 1 items
12 * 12 = 144
```

`got` равен 1 — прочитано ровно одно значение, как и просили форматом `"%d"`. Приглашение `Enter an integer: ` напечатано без `\n` и потому слилось со следующей строкой вывода — это буферизация (§2.3.3), подробнее в Г2.Ч3.З7.

**Ответ:** при вводе `12` программа печатает `read 1 items` и `12 * 12 = 144`. Квадрат вычисляется обычным умножением `n * n`; отдельной операции возведения в степень в C нет (для вещественных есть функция `pow` из `<math.h>`, но для целого квадрата умножение и проще, и точнее).

### Г2.Ч3.З4

**Условие.** Прочитайте `double` (спецификатор `%lf`) и напечатайте его с двумя знаками после точки (`%f` с точностью). Убедитесь, что путаница `%f`/`%lf` между вводом и выводом не вкралась.

**Решение.** Ключевая асимметрия §2.3.2: в `scanf` спецификатор указывает, куда класть результат, поэтому `%f` означает `float`, а `%lf` — `double`; в `printf` аргумент типа `float` автоматически расширяется до `double`, поэтому для обоих типов служит `%f`.

```c
#include <stdio.h>

int main(void)
{
    double x = 0;
    printf("Enter a real number: ");
    int got = scanf("%lf", &x);      // ввод double - обязательно %lf
    printf("read %d items\n", got);
    printf("x = %.2f\n", x);         // вывод double - %f (с точностью)
    return 0;
}
```

```
printf '2.5\n' | ./readdouble
```

```
Enter a real number: read 1 items
x = 2.50
```

Проверка «путаница не вкралась» проста: на вводе стоит `%lf` (переменная — `double`), на выводе `%.2f`. Если бы на вводе оказался `%f`, компилятор с `-Wall` сообщил бы о несоответствии типов — см. Г2.Ч3.З5, где эта ошибка разобрана специально.

**Ответ:** при вводе `2.5` печатается `x = 2.50`. Правило запоминается одной фразой: **`%lf` — только на входе**; на выходе `%f` годится и для `float`, и для `double`, потому что `float` в аргументах `printf` расширяется до `double`.

### Г2.Ч3.З5

**Условие.** *(понимание)* В программе есть строки:

```c
double x;
scanf("%f", &x);
```

Найдите ошибку, объясните последствие и исправьте. Почему для `double` во вводе нужен `%lf`, а в `printf` достаточно `%f`?

**Решение.** Ошибка в спецификаторе: `x` имеет тип `double`, а `%f` в `scanf` означает «читаю вещественное и записываю его как **`float`**, то есть в 4 байта». Компилятор с `-Wall` формулирует это буквально:

```
      |                       |   double *
      |                       float *
      |                      %lf
```

Компилятор даже подсказывает правильный спецификатор — `%lf` в последней строке.

Последствие — не «немного меньшая точность», а бессмыслица. `scanf` кладёт по указанному адресу 4 байта в формате `float`; переменная же занимает 8 байт и будет прочитана как `double`. Половина байтов остаётся тем, чем была, и получившаяся комбинация не имеет отношения к введённому числу:

```c
#include <stdio.h>

int main(void)
{
    double x = 0;
    int got = scanf("%f", &x);    /* ⛔ %f ждёт float*, передан double* */
    printf("got=%d x=%g\n", got, x);
    return 0;
}
```

```
printf '2.5\n' | ./bad
```

```
got=1 x=5.31535e-315
```

Обратите внимание: `got` равен 1 — с точки зрения `scanf` чтение прошло *успешно*. Ошибку не показывает ни возврат функции, ни падение программы; наружу она выходит только диким значением `5.31535e-315` (число на эталонной машине; по стандарту это неопределённое поведение, и результат может быть любым, вплоть до аварийного завершения).

Исправление — один символ:

```c
double x = 0;
int got = scanf("%lf", &x);   /* правильный спецификатор для double */
```

```
got=1 x=2.5
```

Причина асимметрии `scanf`/`printf` — в том, как передаются аргументы. В `printf` значение передаётся **по значению**, и правила языка для таких вызовов автоматически расширяют `float` до `double` — до функции просто не может дойти `float`, поэтому отдельный спецификатор для него не нужен, и `%f` печатает оба типа. В `scanf` передаётся **адрес** переменной, и никакого расширения не происходит: адрес не несёт информации о типе, поэтому единственный способ сообщить функции, сколько байтов и в каком формате записывать, — спецификатор. Отсюда `%f` = «пиши 4-байтовый `float`», `%lf` = «пиши 8-байтовый `double`».

**Ответ:** для `double` во вводе нужен `%lf`; запись `scanf("%f", &x)` заставляет функцию записать по адресу `x` 4-байтовое значение `float`, что портит переменную (на эталонной машине получилось `5.31535e-315`), причём `scanf` возвращает 1, как при успехе. Исправление: `scanf("%lf", &x)`. В `printf` дополнительный спецификатор не нужен, потому что `float`-аргумент расширяется до `double` при вызове, а в `scanf` передаётся адрес, и формат — единственный источник сведений о размере записываемого значения.

### Г2.Ч3.З6

**Условие.** *(понимание)* Ниже — пять вызовов; для каждого определите, корректен он или нет, и почему:

```c
printf("%d\n", 3.5);
printf("%c\n", 65);
scanf("%d", age);
printf("100%\n");
scanf("%lf", &value);   // value имеет тип double
```

**Решение.** Разбираем по правилу согласования из §2.3.3: число, порядок и типы спецификаторов обязаны совпадать с аргументами.

**1. `printf("%d\n", 3.5);` — некорректно.** `%d` ждёт `int`, передан `double`. Несоответствие типа и спецификатора — неопределённое поведение; никакого преобразования «на лету» не происходит, потому что `printf` узнаёт тип аргумента только из формата. Правильно — `printf("%f\n", 3.5);` (или `%g`); если же нужна именно целая часть, её берут явным приведением: `printf("%d\n", (int)3.5);`.

**2. `printf("%c\n", 65);` — корректно.** `%c` ожидает целое, трактуемое как код символа (§2.3.1), а символьные значения в аргументах и без того имеют тип `int` (§2.1.3). Вызов напечатает `A`.

**3. `scanf("%d", age);` — некорректно.** Забыт `&`: передано *значение* переменной, а `scanf` ждёт адрес, куда записать результат (§2.3.2). Число, случайно оказавшееся в `age`, будет истолковано как адрес — типичный сбой с аварийным завершением. Правильно — `scanf("%d", &age);`.

**4. `printf("100%\n");` — некорректно.** Одиночный `%` начинает спецификатор, и следующим символом оказывается перевод строки — такого спецификатора нет. Правильная запись знака процента — `%%`: `printf("100%%\n");`.

**5. `scanf("%lf", &value);` при `double value` — корректно.** Тип и спецификатор согласованы, адрес передан (см. Г2.Ч3.З5).

### Г2.Ч3.З7

**Условие.** *(понимание)* Программа печатает `printf("Enter n: ");` (без `\n`), затем читает число. Пользователь жалуется, что приглашение появляется «в неправильном месте». Объясните причину через буферизацию (§2.3.3) и предложите простое изменение вывода, делающее приглашение нагляднее.

**Решение.** Вывод программы не попадает на экран немедленно — он накапливается в буфере и «сбрасывается» порциями: при переводе строки, при заполнении буфера или при завершении программы (§2.3.3). Приглашение `"Enter n: "` перевода строки не содержит, поэтому оседает в буфере, а на экран выходит позже — вместе с последующим выводом.

```c
#include <stdio.h>

int main(void)
{
    int n = 0;
    printf("Enter n: ");        // без \n - остаётся в буфере
    int got = scanf("%d", &n);
    printf("got=%d n=%d\n", got, n);
    return 0;
}
```

```
printf '5\n' | ./prompt
```

```
Enter n: got=1 n=5
```

Приглашение и результат оказались в одной строке. Пользователь при этом видит не «вопрос, затем ответ», а слипшуюся строку — отсюда и жалоба «приглашение не на своём месте».

Простое изменение — завершить приглашение переводом строки:

```c
printf("Enter n:\n");       // приглашение завершено \n - строка уходит на экран
```

```
Enter n:
got=1 n=5
```

Теперь приглашение — отдельная строка, и вывод читается как диалог. Ровно то же самое делает `\n` в конце **любой** выводимой строки: он не «переводит курсор» ради красоты, а является границей, на которой буфер отдаёт накопленное (§2.3.3).

**Ответ:** причина — построчная буферизация вывода: приглашение без `\n` остаётся в буфере и печатается позже, слипаясь со следующей строкой; это свойство буферизации, а не ошибка программы. Простейшее исправление — писать приглашение с переводом строки (`printf("Enter n:\n");`), тогда оно гарантированно уходит на экран до чтения. (Существует и способ форсировать сброс буфера, не добавляя `\n`, — он появится в главах о вводе-выводе; в пределах материала этой главы правильный инструмент — `\n`.)

### Г2.Ч3.З8

**Условие.** *(понимание)* Что напечатает программа при вводе `hello` (буквы вместо числа)?

```c
int n = -1;
int got = scanf("%d", &n);
printf("got=%d n=%d\n", got, n);
```

Объясните значения `got` и `n`, сославшись на возврат `scanf` и на то, записал ли он что-нибудь.

**Решение.** `scanf` с форматом `"%d"` пропускает ведущие пробелы, а затем пытается прочитать целое. Первый же символ `h` целым числом быть не может, поэтому преобразование не начинается: функция возвращает число **успешно прочитанных и записанных** элементов, то есть 0 (§2.3.2). Раз ни одного элемента не записано, переменная `n` не тронута и сохраняет своё прежнее значение −1.

```c
#include <stdio.h>

int main(void)
{
    int n = -1;
    int got = scanf("%d", &n);
    printf("got=%d n=%d\n", got, n);
    return 0;
}
```

```
printf 'hello\n' | ./readint
```

```
got=0 n=-1
```

Два вывода из этого эксперимента.

*Инициализация не «на всякий случай», а по делу.* Именно потому, что при неудаче `scanf` ничего не пишет, начальное значение переменной становится тем, с чем программа продолжит работать. Будь `n` неинициализированной, программа считала бы дальше по мусору (Г2.Ч1.З7).

*Возврат нужно проверять.* `got` — единственный признак того, что ввод не состоялся: сама переменная выглядит вполне «нормально». Осмысленно среагировать (попросить ввести заново, завершиться с сообщением) мы сможем, когда появятся ветвления в главе 3; пока возврат печатается.

Стоит знать и то, что непрочитанный текст `hello` остаётся во входном потоке: `scanf` не «съедает» символы, которые не смог преобразовать. Повторное чтение тем же форматом снова наткнулось бы на `h` и снова вернуло 0 — это причина «зависших» циклов ввода, к которым мы вернёмся в главе 3.

**Ответ:** программа печатает `got=0 n=-1`. `got` равен 0, потому что `scanf` возвращает количество успешно прочитанных элементов, а преобразовать `hello` в целое не удалось; `n` осталось равным −1, потому что при неудачном преобразовании функция в переменную ничего не записывает — печатается ровно то значение, которым переменная была инициализирована.

### Г2.Ч3.З9

**Условие.** *(прикладное)* «Калькулятор»: прочитайте два целых `a` и `b` одним `scanf("%d %d", …)` и напечатайте таблицей их сумму, разность, произведение, целое частное `a/b` и остаток `a%b`. (Считайте, что пользователь вводит корректные числа и `b ≠ 0`; проверку добавим в главе 3.)

**Решение.** Один вызов `scanf` с двумя спецификаторами читает оба числа; пробел в формате соответствует любому количеству пробелов и переводов строк между ними (§2.3.2), поэтому ввод `17 5` и ввод двумя строками равнозначны.

```c
#include <stdio.h>

int main(void)
{
    int a = 0, b = 0;
    printf("Enter a and b: ");
    int got = scanf("%d %d", &a, &b);
    printf("scanf read %d items\n", got);
    printf("a         = %d\n", a);
    printf("b         = %d\n", b);
    printf("a + b     = %d\n", a + b);
    printf("a - b     = %d\n", a - b);
    printf("a * b     = %d\n", a * b);
    printf("a / b     = %d\n", a / b);   // b != 0 по условию
    printf("a %% b     = %d\n", a % b);
    return 0;
}
```

```
printf '17 5\n' | ./calc
```

```
Enter a and b: scanf read 2 items
a         = 17
b         = 5
a + b     = 22
a - b     = 12
a * b     = 85
a / b     = 3
a % b     = 2
```

Ручная сверка: 17 + 5 = 22 ✓, 17 − 5 = 12 ✓, 17 · 5 = 85 ✓, 17 / 5 = 3 (целочисленное, §2.2.1) ✓, 17 % 5 = 2 ✓.

Две технические детали. Во-первых, `%%` в строке формата — единственный способ напечатать знак процента (§2.3.1); в исходном коде видно `a %% b`, в выводе — `a % b`. Во-вторых, `got` печатается не для красоты: это единственный признак того, что оба числа действительно прочитаны (Г2.Ч3.З8).

**Ответ:** при вводе `17 5` программа печатает сумму 22, разность 12, произведение 85, частное 3 и остаток 2. Оговорка `b ≠ 0` в условии существенна: `a / 0` и `a % 0` — неопределённое поведение (§2.2.1), и защититься от него можно будет только ветвлением из главы 3.

**Вариант 2 (два отдельных чтения и выровненная таблица).** Числа можно читать порознь, а результат оформить настоящей таблицей — ширина поля выравнивает столбец значений (§2.3.1):

```c
#include <stdio.h>

int main(void)
{
    int a = 0, b = 0;
    printf("a = ");
    int got_a = scanf("%d", &a);     // два отдельных чтения
    printf("b = ");
    int got_b = scanf("%d", &b);
    printf("read %d + %d items\n", got_a, got_b);

    printf("sum        %8d\n", a + b);   // ширина поля выравнивает столбец
    printf("diff       %8d\n", a - b);
    printf("product    %8d\n", a * b);
    printf("quotient   %8d\n", a / b);
    printf("remainder  %8d\n", a % b);
    return 0;
}
```

```
printf '17 5\n' | ./calc2
```

```
a = b = read 1 + 1 items
sum              22
diff             12
product          85
quotient          3
remainder         2
```

(Подписи столбцов здесь — обычный текст внутри строки формата, а выравнивание чисел делает ширина поля `%8d`: подписи дополнены пробелами вручную, чтобы обойтись материалом этой главы.)

*Сравнение.* Один `scanf` с двумя спецификаторами короче и даёт единый признак успеха: `got == 2` означает «оба числа на месте». Два отдельных вызова позволяют показать своё приглашение перед каждым числом и различить, какое именно чтение не удалось (`got_a` и `got_b` по отдельности) — это ценно в диалоговых программах, но требует вдвое больше кода и двух проверок вместо одной. Что касается вывода: печать «в столбик» с фиксированной шириной поля читается заметно легче, когда чисел много или они разной длины, — сравните выровненный столбец выше со строками основного решения, где значения «плавают» вслед за длиной подписи.

```
printf '\n' | ./readchar
```

```
got=1 char='
' code=10
```

Разбор кодов (§2.1.3, §1.1.1):

- `A` = **65** — начало латинского алфавита в ASCII, от него отсчитываются остальные буквы.
- `7` = **55** — это код *символа* «семь», а не число 7. Цифры в ASCII начинаются с `'0'` = 48, поэтому `'7'` = 48 + 7 = 55. Различие принципиально: `'7' - '0'` даёт число 7, а сам `'7'` — нет.
- пробел = **32** — пробел такой же полноправный символ, как буква, и занимает такой же байт.
- перевод строки (`'\n'`) = **10** — управляющий символ; в выводе он проявился тем, что строка разорвалась прямо внутри кавычек.

Последний прогон объясняет и главный подводный камень `%c` (§2.3.2): этот спецификатор, в отличие от `%d` и `%lf`, **не пропускает** пробельные символы. Если ввести `A` и нажать Enter, в потоке останется `'\n'`, и следующее чтение через `%c` прочитает именно его — код 10, а не то, что наберут дальше. Обойти это в пределах материала главы нечем; приёмы появятся с циклами (глава 3) и строками (глава 5).

**Ответ:** `A` → 65, `7` → 55, пробел → 32, перевод строки → 10; `got` во всех случаях равен 1. Код символа — это просто содержимое байта: `%c` печатает его как символ, `%d` — как число. Отдельно стоит запомнить, что код цифры не равен самой цифре (`'7'` = 55 = 48 + 7) и что `%c` читает любой символ, включая пробел и `\n`.

### Г2.Ч3.З12

**Условие.** *(прикладное исследование)* Расширьте сквозную программу примера 1 §2.3.3: помимо суммы, разности и двух средних выведите остаток `a % b` и произведение. Подайте на вход несколько пар чисел через `echo` (в том числе с отрицательными) и сверьте вывод с ручным расчётом. (Деление на ноль пока не подавайте — почему это опасно, вспомните из §2.2.1.)

**Решение.** К программе примера 1 добавлены две строки — произведение и остаток; остальное сохранено, включая печать возврата `scanf` и пару «целое среднее / вещественное среднее».

```c
#include <stdio.h>

int main(void)
{
    int a = 0, b = 0;
    printf("Enter two integers: ");
    int got = scanf("%d %d", &a, &b);
    printf("scanf read %d items\n", got);
    printf("a = %d, b = %d\n", a, b);
    printf("sum        = %d\n", a + b);
    printf("difference = %d\n", a - b);
    printf("product    = %d\n", a * b);
    printf("remainder  = %d\n", a % b);
    printf("int avg    = %d\n", (a + b) / 2);
    printf("real avg   = %.2f\n", (a + b) / 2.0);
    return 0;
}
```

Прогон 1, `7 4` — обе величины положительны:

```
Enter two integers: scanf read 2 items
a = 7, b = 4
sum        = 11
difference = 3
product    = 28
remainder  = 3
int avg    = 5
real avg   = 5.50
```

Ручная сверка: 7 + 4 = 11 ✓, 7 − 4 = 3 ✓, 7 · 4 = 28 ✓, 7 % 4 = 3 ✓, (7 + 4)/2 = 11/2 = 5 (целочисленное, дробная часть отброшена) ✓, 11/2,0 = 5,5 ✓.

Прогон 2, `-9 4` — отрицательное делимое:

```
Enter two integers: scanf read 2 items
a = -9, b = 4
sum        = -5
difference = -13
product    = -36
remainder  = -1
int avg    = -2
real avg   = -2.50
```

Сверка: −9 + 4 = −5 ✓, −9 − 4 = −13 ✓, −9 · 4 = −36 ✓. Остаток: −9 % 4 = **−1**, потому что знак остатка совпадает со знаком делимого (§2.2.1); проверка тождеством: (−9 / 4)·4 + (−9 % 4) = (−2)·4 + (−1) = −9 ✓. Целое среднее: −5 / 2 = **−2**, а не −3 — усечение идёт **к нулю**, поэтому целое среднее оказалось *больше* вещественного (−2 против −2,50). Это тонкое место и есть главный улов прогона с отрицательными числами.

Прогон 3, `5 -3` — отрицательный делитель:

```
Enter two integers: scanf read 2 items
a = 5, b = -3
sum        = 2
difference = 8
product    = -15
remainder  = 2
int avg    = 1
real avg   = 1.00
```

Сверка: 5 + (−3) = 2 ✓, 5 − (−3) = 8 ✓, 5 · (−3) = −15 ✓. Остаток 5 % (−3) = **2** — знак снова взят у делимого (положительного), хотя делитель отрицателен; проверка: (5 / −3)·(−3) + 2 = (−1)·(−3) + 2 = 5 ✓. Средние совпали (1 и 1.00), потому что сумма 2 делится на 2 нацело.

Про деление на ноль: подавать `b = 0` нельзя, потому что `a / 0` и `a % 0` — неопределённое поведение (§2.2.1). Программа не «напечатает бесконечность» и не сообщит об ошибке: на большинстве платформ целочисленное деление на ноль вызывает аппаратное прерывание и аварийное завершение процесса, а формально стандарт не обещает вообще ничего. Отсеять такой ввод можно будет ветвлением из главы 3.

**Ответ:** расширенная программа печатает восемь величин; на входах `7 4`, `-9 4` и `5 -3` все результаты совпали с ручным расчётом. Прогон с отрицательными числами выявил две особенности целой арифметики C, которые легко пропустить на положительных данных: знак остатка определяется **делимым** (−9 % 4 = −1, 5 % −3 = 2), а целочисленное деление усекает **к нулю**, из-за чего целое среднее −5/2 равно −2, тогда как вещественное — −2,50.
