## 2. Жёсткие правила (действуют в каждом заходе)

1. **Никаких догадок** (План, §4.4): каждое фактическое утверждение подтверждено источниками из `sources.md`; не удалось подтвердить — не пишем, фиксируем в «Открытых вопросах» `outline.md`.
2. **Минимум два независимых источника на часть** (План, §4.3); идентификаторы (C1…C12, R1…R11) — в блоке «Источники» в конце части. Арбитры при расхождениях: C5 (стандарт C) для тома I, R7 (Rust Reference) для тома II.
3. **Весь код проверяется запуском**: C — компиляция `gcc -std=c17 -Wall -Wextra` (для глав ≥7 дополнительно прогон под `-fsanitize=address` там, где уместно); Rust — `cargo build && cargo test` (или `rustc` для одиночных файлов). Некомпилирующийся фрагмент в текст не попадает, кроме специально помеченных примеров ошибок («⛔ не компилируется — и вот почему»). GUI-главы (12 и 26): агент проверяет компиляцию и smoke-запуск (окно открывается без ошибок); функциональную проверку интерфейса выполняет владелец на ревизии — это фиксируется в отчёте захода как открытый пункт.
4. **Только пройденный материал**: глава N ссылается лишь на главы 1…N−1. Проверка «забеганий вперёд» — обязательный пункт ревизии.
5. **Решений в главах нет** — только условия заданий. Решения — исключительно в `solutions/`.
6. **Прямое копирование текста источников запрещено** — только пересказ/синтез со ссылкой.
7. Новый источник — сначала в `sources.md` (с проверкой по критериям §4.2 Плана и датой), потом в текст.

## 3. Шаблон главы (`chapters/NN-<slug>.md`)

```markdown
# Глава N. <Название>

## Введение
<зачем тема, где применяется, что читатель будет уметь — План §2.1>

## Часть N.M. <Название части>

### §N.M.1. <Название параграфа>
<каждый параграф ОБЯЗАН содержать все 6 элементов Плана §2.3:
 1) определения/формулировки, 2) интуиция (ASCII-схемы памяти/стека — норма для этой дисциплины),
 3) 2–3 полностью разобранных примера с выкладками, 4) связь с практикой (рабочий код),
 5) типичные ошибки и подводные камни, 6) ссылки на источники>

### §N.M.k. ...

### Практика части N.M
<6–15 заданий, по возрастанию сложности:
 ~30% механика → ~40% понимание (объяснить/найти ошибку/сравнить) → ~30% прикладные.
 Формат каждого: **ГN.ЧM.Зk.** <условие>. Для кодовых — точная постановка: вход/выход/ограничения.>

**Источники части:** C2 (гл. …), C3 (…), …

## Часть N.M+1. ...

## Резюме главы
<краткое резюме + связка со следующей главой — План §2.1>
```

Специфика курса: в томе II каждая часть, где это осмысленно, содержит рубрику-сопоставление **«Как это было в C»** (задано в `outline.md`); в главах 25 и 26 (проекты) и части 12.Ч2 вместо классических параграфов допустимы проектные чекпоинты.

## 4. Шаблон файла решений (`solutions/NN-<slug>.md`)

Для каждого задания по его ID (План §6): 1) полное условие (продублировано); 2) подробное пошаговое решение с объяснением каждого шага; 3) явно выделенный итоговый ответ; 4) для кодовых — полный рабочий код с комментариями (код реально запущен, вывод приведён); 5) где уместно — альтернативные решения и разбор типичных неверных подходов.

### Протокол одного захода (этап 3, глава N)

1. `/sc:load`; прочитать резюме предыдущей главы (N−1) — для связки.
2. Свериться с `outline.md`: состав частей, источники, пометки сложности.
3. Написать теорию по частям (шаблон §3), сверяя каждую часть минимум с двумя источниками; всё неподтверждённое — в «Открытые вопросы».
4. Написать практику: 6–15 заданий на часть, ID сплошной нумерацией внутри части, распределение сложности 30/40/30.
5. Прогнать весь код (C: `gcc -std=c17 -Wall -Wextra`; Rust: `cargo build && cargo test`); в репозитории есть скрипт запуска C — использовать при удобстве.
6. Самопроверка по чеклисту §6 (мини-ревизия главы).
7. Обновить трекер §7 (клетка «Глава»), `/sc:save`, commit (§8). **Остановиться** — следующий заход начнётся заново с `/sc:load` (не терять качество на длинном контексте — План §5).

Протокол захода этапа 5 аналогичен: условия берутся из готовой главы, решения пишутся по шаблону §4, код запускается, трекер («Решения») и commit.

Ключевые подтверждённые факты: (1) выход за границу массива (`for i<=3` по `int a[3]`) компилируется **без единого предупреждения** даже с `-Wall -Wextra` — использовано в §5.1.2 как иллюстрация того, что C не проверяет границы (детерминированный вывод у UB-примера в текст не выносится); (2) `sizeof "hello"`-массива = 6, `strlen` = 5 (терминатор `'\0'` занимает ячейку); (3) `scanf("%9s", …)` усекает длинный ввод `Alexandragreat` → `Alexandra` (защита от переполнения); (4) `strncpy(dst, src, 3)` при `strlen(src)>3` `'\0'` не дописывает; (5) `strcmp` возвращает знак (на платформе −1/0/+1), в тексте оговорено, что гарантирован лишь знак.
