# Глава 2. Основы языка: типы, переменные, выражения

## Введение

В главе 1 мы разобрались, *из чего* состоит работающая программа: биты и байты в памяти, адреса, процессор, конвейер сборки. Теперь пора научиться этой памятью распоряжаться — но уже не на языке «байт по адресу 1000», а средствами C.

Эта глава даёт базовый «словарный запас» языка. К её концу вы сможете завести в программе именованные ячейки данных нужного размера (**переменные**), вычислять над ними арифметические и логические **выражения**, осознанно преобразовывать одни типы в другие — и, наконец, впервые сделать программу *интерактивной*: она станет печатать форматированный результат и читать данные от пользователя.

Всё, что здесь появится, — фундамент: любая последующая конструкция C записывается в этих терминах. Управляющие конструкции главы 3 будут ветвиться по значениям выражений; функции главы 4 — принимать и возвращать переменные этих типов; указатели главы 6 обопрутся на понятие адреса переменной, которое мы затронем уже здесь.

После изучения главы вы будете уметь:

- объявлять и инициализировать переменные, выбирая подходящий тип;
- называть целые, символьный и вещественные типы, их размеры и диапазоны и объяснять, откуда эти диапазоны берутся;
- вычислять выражения с учётом приоритета операторов и правил преобразования типов;
- распознавать переполнение и целочисленное деление как источники реальных ошибок;
- выводить данные функцией `printf` с форматированием и читать их функцией `scanf`, проверяя результат чтения.

> В примерах этой главы намеренно нет ветвлений и циклов (глава 3), своих функций (глава 4), массивов (глава 5) и указателей (глава 6): весь код — прямая последовательность инструкций внутри `main`. Единственное исключение — символ `&` в `scanf`; он опирается на понятие адреса переменной из §1.1.2, а полностью раскроется в главе 6.

## Часть 2.1. Переменные и базовые типы

### §2.1.1. Переменные: объявление и инициализация

**Определения.**

- **Переменная** (variable) — именованный участок памяти определённого **типа**, в котором хранится значение, доступное по имени. Вспомните §1.1.2: у любой переменной есть адрес (номер её первого байта) и содержимое; имя переменной избавляет нас от необходимости помнить адреса.
- **Тип** (type) переменной задаёт две вещи: сколько байтов она занимает и как эти байты интерпретируются (целое число, символ, вещественное число, …).
- **Объявление** (declaration) вводит имя и тип переменной: `int count;`. С этого момента компилятор знает, что `count` — целое, и отводит под него место.
- **Инициализация** (initialization) — задание начального значения *в момент объявления*: `int count = 10;`.
- **Присваивание** (assignment) — запись нового значения в уже существующую переменную: `count = 20;`. Инициализация происходит один раз при рождении переменной, присваиваний может быть сколько угодно.

**Правила имён.** Имя переменной (идентификатор) состоит из латинских букв, цифр и знака подчёркивания `_`, не может начинаться с цифры и не может совпадать с ключевым словом языка (`int`, `return`, `if`, …). Регистр важен: `count`, `Count` и `COUNT` — три разных имени.

**Интуиция.** Переменную удобно представлять подписанной коробкой фиксированного размера на «улице памяти» из §1.1.2. Объявление ставит коробку и вешает на неё табличку с именем; инициализация и присваивание кладут внутрь значение; имя в коде — это обращение к коробке по табличке, а не по номеру дома.

```
объявление int count;        инициализация int count = 10;
┌──────────────┐             ┌──────────────┐
│   count      │             │   count      │
│  ┌────────┐  │             │  ┌────────┐  │
│  │  ???    │  │  ← мусор    │  │   10    │  │
│  └────────┘  │             │  └────────┘  │
└──────────────┘             └──────────────┘
 место есть,                  место есть,
 значение не задано           значение задано
```

**Разобранные примеры.**

*Пример 1. Объявление, инициализация, присваивание.*

```c
#include <stdio.h>

int main(void)
{
    int count = 10;      // объявление с инициализацией
    printf("%d\n", count);
    count = 20;          // присваивание нового значения
    printf("%d\n", count);
    count = count + 5;   // справа вычисляется 20+5=25, результат кладётся в count
    printf("%d\n", count);
    return 0;
}
```

Вывод:

```
10
20
25
```

Строка `count = count + 5;` читается не как математическое равенство, а как команда: «вычисли выражение справа от `=` и запиши результат в переменную слева». Правая часть использует *старое* значение `count` (20), левая получает *новое* (25).

*Пример 2. Несколько переменных.* Объявлять можно по одной или списком одного типа через запятую:

```c
int a = 1;
int b = 2, c = 3;    // две переменные типа int в одной строке
int sum = a + b + c; // инициализация выражением: sum получает 6
```

*Пример 3. Комментарии.* В коде выше `//` начинает комментарий до конца строки (введён в §1.2.2). C понимает и «блочные» комментарии `/* … */`, которые могут занимать часть строки или несколько строк; компилятор игнорирует и те и другие.

```c
int width = 80;   /* ширина в символах */
// int old = 5;   ← эта строка отключена, компилятор её не видит
```

**Связь с практикой.** Переменные — это память, которой вы дали имена; всё остальное в программе так или иначе читает и меняет переменные. Привычка *инициализировать переменную при объявлении* — одна из самых дешёвых страховок в C: она избавляет от целого класса ошибок, к которому мы вернёмся в подводных камнях и который в полную силу проявится в главе 7.

**Типичные ошибки и подводные камни.**

- **Чтение неинициализированной переменной.** `int x; printf("%d\n", x);` напечатает «мусор» — то, что случайно оказалось в отведённых байтах. Более того, по стандарту это **неопределённое поведение** (undefined behavior, глава 13): полагаться на такое значение нельзя. Правило: не уверены в начальном значении — пишите `= 0`.
- **Имя, начинающееся с цифры или содержащее дефис.** `2nd`, `my-var` — не идентификаторы (дефис компилятор примет за вычитание). Используйте `second`, `my_var`.
- **Путаница `=` и `==`.** `=` — присваивание, `==` — сравнение (§2.2.2). Их смешение — классическая ошибка, к которой мы ещё вернёмся.
- **Объявление той же переменной дважды** в одной области видимости — ошибка компиляции («redeclaration»).

**Источники:** C2 (гл. 2, 4), C1 (гл. 1–2), C3 (уровень 0); нормативно об инициализации и неопределённом значении — C5.

### §2.1.2. Целочисленные типы и их диапазоны

**Определения.** Для целых чисел C предлагает несколько типов, отличающихся размером (а значит, диапазоном):

| Тип | Гарантированный минимум ширины | Обычная ширина* |
|---|---|---|
| `char` | 8 бит | 1 байт |
| `short` | 16 бит | 2 байта |
| `int` | 16 бит | 4 байта |
| `long` | 32 бита | 4 (Windows) / 8 (Linux, macOS) байт |
| `long long` | 64 бита | 8 байт |

*«Обычная ширина» — на распространённых 64-битных платформах; стандарт её *не* фиксирует, гарантируя лишь минимумы и порядок `sizeof(char) ≤ sizeof(short) ≤ sizeof(int) ≤ sizeof(long) ≤ sizeof(long long)`.

Каждый целый тип бывает **знаковым** (signed, умеет хранить отрицательные числа) и **беззнаковым** (unsigned, только неотрицательные, зато вдвое больший положительный диапазон). По умолчанию `int`, `short`, `long`, `long long` знаковые; беззнаковый вариант записывается словом `unsigned`: `unsigned int`, `unsigned long`. Тип `int` — «рабочая лошадка»: если нет причин выбрать другой, берут его.

**Интуиция.** Тип шириной *N* бит хранит 2ᴺ различных комбинаций (§1.1.1). Как их трактовать — вопрос знаковости:

```
8-битный байт, 256 комбинаций:

беззнаковый (unsigned char):   0 .......................... 255
знаковый  (signed char):    -128 ........ -1  0 ........... 127
                             └── половина ──┘└── половина ──┘
```

Беззнаковый тип «тратит» все комбинации на неотрицательные числа (0…2ᴺ−1); знаковый делит их примерно пополам между отрицательными и неотрицательными. Точные границы для всех типов объявлены в заголовке `<limits.h>` именованными константами (`INT_MAX`, `UINT_MAX`, `LONG_MIN`, …).

**Разобранные примеры.**

*Пример 1. Размеры типов на этой машине.* Оператор `sizeof` (подробно — §2.1.4) сообщает размер типа в байтах; печатаем его спецификатором `%zu`:

```c
#include <stdio.h>

int main(void)
{
    printf("char       %zu\n", sizeof(char));
    printf("short      %zu\n", sizeof(short));
    printf("int        %zu\n", sizeof(int));
    printf("long       %zu\n", sizeof(long));
    printf("long long  %zu\n", sizeof(long long));
    return 0;
}
```

Вывод на машине, где готовилась книга (Windows, GCC 15.2.0):

```
char       1
short      2
int        4
long       4
long long  8
```

*Пример 2. Границы диапазонов.* Подключив `<limits.h>`, распечатаем несколько границ:

```c
#include <stdio.h>
#include <limits.h>

int main(void)
{
    printf("INT_MIN   %d\n", INT_MIN);
    printf("INT_MAX   %d\n", INT_MAX);
    printf("UINT_MAX  %u\n", UINT_MAX);   // %u — беззнаковое десятичное
    printf("LONG_MAX  %ld\n", LONG_MAX);  // %ld — long
    return 0;
}
```

Вывод:

```
INT_MIN   -2147483648
INT_MAX   2147483647
UINT_MAX  4294967295
LONG_MAX  2147483647
```

Здесь `int` — 4 байта: 2³² = 4 294 967 296 комбинаций, знаковый диапазон примерно ±2,1 млрд, беззнаковый — до ≈4,3 млрд. Обратите внимание: `LONG_MAX` совпал с `INT_MAX` — на Windows `long` тоже 4 байта (см. пример 3).

*Пример 3. Непереносимость размеров.* Тот же код, собранный на Linux или macOS, напечатал бы `long 8` и `LONG_MAX 9223372036854775807`: там `long` — 8 байт. Это не ошибка ни одной из систем — обе соблюдают стандарт, который для `long` требует лишь «не менее 32 бит». Вывод: **нельзя полагать, что тип имеет тот же размер на другой платформе**. Когда важен точный размер, используют типы фиксированной ширины из `<stdint.h>` (`int32_t`, `uint64_t`, …) — мы вернёмся к ним при работе с файлами (глава 10).

**Связь с практикой.** Выбор типа — это выбор диапазона. Счётчик количества людей в очереди логично объявить `unsigned int` (отрицательных значений быть не может); число, которое заведомо может превысить 2 млрд, — `long long`. Ошибка в выборе диапазона молча портит данные: значение, не влезающее в тип, не «обрезается с предупреждением», а превращается в другое число (переполнение, §2.2.5).

**Типичные ошибки и подводные камни.**

- **«int всегда 4 байта».** На типичных настольных платформах — да, но стандарт этого не обещает; на микроконтроллере `int` вполне может быть 2 байта.
- **`long` как «большой int».** На Windows `long` не больше `int`. Нужен гарантированно 64-битный тип — берите `long long` или `int64_t`.
- **Печать не тем спецификатором.** `unsigned` печатается `%u`, `long` — `%ld`, `long long` — `%lld`; несоответствие спецификатора и типа — неопределённое поведение (подробно в §2.3.1).
- **Смешивание знаковых и беззнаковых** в одном выражении даёт неожиданные результаты (например, `unsigned` «съедает» знак) — разберём в §2.2.5.

**Источники:** C2 (гл. 7), C3 (уровень 0–1); нормативные минимумы диапазонов и `<limits.h>` — C5; константы предела — C6.

### §2.1.3. Символы и вещественные числа

**Определения.**

- **`char`** — целый тип шириной один байт, предназначенный для хранения **кода символа** (§1.1.1: в ASCII букве `A` соответствует число 65). По сути это маленькое целое: над `char` работает обычная целочисленная арифметика. Знаковость `char` определяется реализацией (на x86-платформах он обычно знаковый); когда байт хранит именно код или «сырые» данные, надёжнее `unsigned char`.
- **Символьный литерал** записывается в одинарных кавычках: `'A'`, `'7'`, `' '`. Это число — код символа. (Тонкость C: символьный литерал имеет тип `int`, а не `char`; на практике это почти никогда не мешает.)
- **Управляющие последовательности** — специальные символы, записываемые с обратной косой чертой: `'\n'` (перевод строки), `'\t'` (табуляция), `'\0'` (символ с кодом 0, понадобится для строк в главе 5), `'\\'` (сама обратная косая), `'\''` (одинарная кавычка).
- **`float`** и **`double`** — типы для **вещественных** (дробных) чисел. `double` («двойной точности») хранит больше значащих цифр, чем `float`, и является типом по умолчанию для дробных литералов. На распространённых платформах это форматы IEEE 754: `float` — 4 байта (около 7 значащих десятичных цифр), `double` — 8 байт (около 15–16 цифр).

**Интуиция.** `char` — это «число, которое мы договорились читать как букву». А вещественные числа хранятся **приближённо**: как научная запись в двоичной системе (мантисса × 2^порядок) с ограниченным числом разрядов мантиссы. Отсюда ключевое свойство: большинство дробей представлено не точно, а с крошечной погрешностью — как 1/3 нельзя точно записать конечной десятичной дробью, так 0,1 нельзя точно записать конечной двоичной.

**Разобранные примеры.**

*Пример 1. `char` — это число.*

```c
#include <stdio.h>

int main(void)
{
    char letter = 'A';
    printf("as char: %c\n", letter);   // %c печатает символ
    printf("as int:  %d\n", letter);   // %d печатает его код
    printf("next:    %c\n", (char)(letter + 1)); // код 66 -> 'B'
    return 0;
}
```

Вывод:

```
as char: A
as int:  65
next:    B
```

Один и тот же байт печатается то как буква (`%c`), то как число (`%d`) — ровно иллюстрация мысли из §1.1.1: интерпретация зависит от того, *как* мы используем данные.

*Пример 2. Управляющие последовательности.*

```c
printf("col1\tcol2\tcol3\n");   // \t — табуляция между словами
printf("line1\nline2\n");        // \n — перевод строки
printf("backslash: \\  quote: \" \n");
```

Вывод:

```
col1	col2	col3
line1
line2
backslash: \  quote: "
```

*Пример 3. Вещественные числа приблизительны.*

```c
#include <stdio.h>

int main(void)
{
    double x = 0.1 + 0.2;
    printf("%g\n", x);         // короткая запись: выглядит как 0.3
    printf("%.17f\n", x);      // 17 знаков после точки — видно погрешность
    printf("1/3 = %.17f\n", 1.0 / 3.0);
    return 0;
}
```

Вывод:

```
0.3
0.30000000000000004
1/3 = 0.33333333333333331
```

Формат `%g` округлил результат до «красивого» `0.3`, но на самом деле в `double` лежит `0.30000000000000004`. Поэтому сравнивать вещественные на точное равенство (`==`) почти всегда неправильно (см. подводные камни).

**Связь с практикой.** `char` — основа работы с текстом (глава 5): строка в C есть последовательность `char`. `double` — стандартный выбор для любых вычислений с дробями (координаты, проценты, физические величины); `float` берут, когда критична экономия памяти (например, большие массивы чисел). Понимание приблизительности вещественных чисел спасает от загадочных расхождений «в последнем знаке» и от неверных сравнений.

**Типичные ошибки и подводные камни.**

- **Одинарные и двойные кавычки — разное.** `'A'` — один символ (число 65). `"A"` — строка из двух байтов (`A` и завершающий `'\0'`; строки — глава 5). Их нельзя путать.
- **`'\0'` — это не `'0'`.** `'\0'` имеет код 0 (пустой символ-терминатор), `'0'` — код 48 (цифра ноль).
- **Сравнение вещественных через `==`.** `0.1 + 0.2 == 0.3` даёт *ложь*. Сравнивают, проверяя, что разность по модулю мала (например, меньше 1e-9), а не на точное равенство.
- **`float` там, где нужна точность.** Деньги в копейках и точные счётчики хранят в целых, а не в `float`/`double`: накопление погрешностей исказит суммы.
- **Знаковость `char`.** Записав в `char` код больше 127 на платформе со знаковым `char`, получите отрицательное число. Для «байтов как данных» берите `unsigned char`.

**Источники:** C2 (гл. 7), C1 (гл. 2); нормативно о `char`, символьных литералах и плавающих типах — C5; форматы и точность — C6.

### §2.1.4. Литералы и оператор `sizeof`

**Определения.**

- **Литерал** (literal, в стандарте — *константа*) — значение, записанное в коде напрямую. У каждого литерала есть тип:
  - **Целые:** `42` (десятичный, тип `int`), `0x2A` (шестнадцатеричный, префикс `0x`), `052` (восьмеричный, префикс `0`!). Суффиксы задают тип: `42U` — `unsigned`, `42L` — `long`, `42LL` — `long long`, их сочетания (`42UL`).
  - **Вещественные:** `3.14` и `1e3` (тип `double`), `3.14f` (`float`), `.5`, `2.0`. Запись `1e3` означает 1×10³ = 1000.
  - **Символьные:** `'A'` (§2.1.3).
  - **Строковые:** `"text"` (глава 5).
- **`sizeof`** — оператор, возвращающий размер типа или значения выражения **в байтах**. Применяется к типу в скобках (`sizeof(int)`) или к выражению (`sizeof x`). Результат имеет специальный беззнаковый тип `size_t`, для печати которого служит спецификатор `%zu`.

**Интуиция.** Литерал — это «значение без имени»: `count = 10;` кладёт в переменную безымянную константу `10`. `sizeof` же — «линейка»: он не вычисляет выражение, а лишь измеряет, сколько байтов заняло бы его значение, и делает это ещё на этапе компиляции.

**Разобранные примеры.**

*Пример 1. Системы счисления в литералах.* Один и тот же байт можно записать по-разному:

```c
#include <stdio.h>

int main(void)
{
    printf("%d %d %d\n", 42, 0x2A, 052);  // десятичный, шестн., восьм.
    printf("%d\n", 0xFF);                  // 255
    printf("%d\n", 010);                   // ВНИМАНИЕ: это восьмеричное 8, не 10!
    return 0;
}
```

Вывод:

```
42 42 42
255
8
```

`42`, `0x2A` и `052` — записи одного числа 42. А `010` из-за ведущего нуля прочитано как восьмеричное и равно 8 — частый источник ошибок (см. подводные камни).

*Пример 2. `sizeof` типа и выражения.*

```c
#include <stdio.h>

int main(void)
{
    printf("sizeof(int)    = %zu\n", sizeof(int));
    printf("sizeof(double) = %zu\n", sizeof(double));
    double d = 1.0;
    printf("sizeof d       = %zu\n", sizeof d);   // по значению переменной
    printf("sizeof(1 + 1)  = %zu\n", sizeof(1 + 1)); // тип выражения — int
    return 0;
}
```

Вывод:

```
sizeof(int)    = 4
sizeof(double) = 8
sizeof d       = 8
sizeof(1 + 1)  = 4
```

*Пример 3. Тип символьного литерала.* Проверим упомянутую в §2.1.3 тонкость:

```c
printf("sizeof('A')  = %zu\n", sizeof('A'));  // 4: символьный литерал имеет тип int
printf("sizeof(char) = %zu\n", sizeof(char)); // 1
```

Вывод:

```
sizeof('A')  = 4
sizeof(char) = 1
```

**Связь с практикой.** `sizeof` — не учебная игрушка: он понадобится всякий раз, когда программа должна знать размер данных, — при выделении памяти под массивы и структуры (`malloc(n * sizeof(int))`, глава 7), при работе с файлами и при копировании блоков памяти. Писать `sizeof(тип)` вместо «магического числа» надёжнее: код останется правильным и на платформе с другими размерами типов.

**Типичные ошибки и подводные камни.**

- **Ведущий ноль = восьмеричная запись.** `010` — это 8, а `018` вообще не скомпилируется (8 и 9 — не восьмеричные цифры). Не пишите незначащих нулей перед десятичными числами.
- **Печать `sizeof` через `%d`.** `sizeof` возвращает `size_t` (беззнаковый), правильный спецификатор — `%zu`. С `%d` — формальное несоответствие типов и неопределённое поведение.
- **`sizeof` не выполняет выражение.** В `sizeof(x + 1)` сложение не вычисляется — измеряется лишь *тип* результата. Побочные эффекты внутри `sizeof` (например, `sizeof(x++)`) не происходят.
- **`sizeof массива` против `sizeof указателя`** — важное различие, но оно ждёт нас в главах 5–6; пока `sizeof` применяем к типам, числам и простым переменным.

**Источники:** C2 (гл. 7), C3 (уровень 0); нормативно о литералах, `sizeof` и `size_t` — C5; `%zu` и представление констант — C6.

## Часть 2.2. Операторы и выражения

### §2.2.1. Арифметические операторы

**Определения.** **Выражение** (expression) — запись, вычисляющая значение: `2 + 3`, `a * b`, `count`. **Оператор** (operator) — знак операции; его **операнды** — то, над чем он работает. Арифметические операторы C:

| Оператор | Значение | Пример | Результат |
|---|---|---|---|
| `+` | сложение | `2 + 3` | `5` |
| `-` | вычитание (и унарный минус) | `5 - 2`, `-x` | `3` |
| `*` | умножение | `4 * 3` | `12` |
| `/` | деление | `7 / 2` | `3` (целые!) |
| `%` | остаток от деления | `7 % 2` | `1` |

Ключевая особенность: если **оба операнда `/` целые**, деление тоже целочисленное — дробная часть **отбрасывается** (не округляется). Чтобы получить дробный результат, хотя бы один операнд должен быть вещественным. Оператор `%` (остаток) определён только для целых.

**Интуиция.** Целочисленное деление `a / b` и остаток `a % b` — это ответ на вопрос «сколько целых `b` помещается в `a` и что остаётся»: `7 / 2 = 3` (три двойки), `7 % 2 = 1` (остаётся один). Пара всегда согласована: `(a / b) * b + (a % b)` восстанавливает `a`.

```
 7 / 2:   |██|██|██| .    → 3 полных двойки
 7 % 2:                 . → остаток 1
```

**Разобранные примеры.**

*Пример 1. Целое против вещественного деления.*

```c
#include <stdio.h>

int main(void)
{
    printf("7 / 2   = %d\n", 7 / 2);      // целочисленное
    printf("7 %% 2   = %d\n", 7 % 2);     // остаток; %% печатает знак %
    printf("1 / 2   = %d\n", 1 / 2);      // 0 — частая ловушка
    printf("7.0 / 2 = %g\n", 7.0 / 2);    // вещественное: один операнд double
    return 0;
}
```

Вывод:

```
7 / 2   = 3
7 % 2   = 1
1 / 2   = 0
7.0 / 2 = 3.5
```

*Пример 2. Остаток и деление для отрицательных.* В современном C (C99 и новее) деление усекает *к нулю*, а знак остатка совпадает со знаком делимого:

```c
printf("-7 / 2 = %d\n", -7 / 2);   // -3 (к нулю, не к -4)
printf("-7 %% 2 = %d\n", -7 % 2);  // -1 (знак делимого)
printf("7 / -2 = %d\n", 7 / -2);   // -3
printf("7 %% -2 = %d\n", 7 % -2);  // 1
```

Вывод:

```
-7 / 2 = -3
-7 % 2 = -1
7 / -2 = -3
7 % -2 = 1
```

*Пример 3. Классическое применение остатка — чётность и «оборачивание».* `n % 2` равно 0 для чётных и 1 (или −1) для нечётных; `n % 12` «зацикливает» значение в диапазоне 0…11 (часы):

```c
printf("14 %% 2  = %d\n", 14 % 2);    // 0 — чётное
printf("15 %% 2  = %d\n", 15 % 2);    // 1 — нечётное
printf("(9 + 5) %% 12 = %d\n", (9 + 5) % 12);  // 2: 14 часов = 2 часа
```

Вывод:

```
14 % 2  = 0
15 % 2  = 1
(9 + 5) % 12 = 2
```

**Связь с практикой.** Целочисленное деление и остаток — рабочие инструменты системного программиста: `%` проверяет делимость и «заворачивает» индексы по кругу, пара `/` и `%` разбивает число на разряды (например, секунды на минуты и остаток секунд). Но именно целочисленное деление — источник тихих ошибок в расчётах средних и процентов (см. ниже).

**Типичные ошибки и подводные камни.**

- **Целочисленное деление там, где ждали дробь.** `1 / 2` — это `0`, а не `0.5`. Среднее двух чисел `(a + b) / 2` при целых `a`, `b` теряет дробную часть; пишите `(a + b) / 2.0`.
- **`%` с вещественными.** `7.5 % 2` не компилируется: остаток определён только для целых (для вещественных есть функция `fmod` из `<math.h>` — понадобится позже).
- **Деление и остаток на ноль.** `x / 0` и `x % 0` — **неопределённое поведение** (глава 13), программа может аварийно завершиться. Делитель, приходящий извне, всегда под подозрением.
- **Порядок вычисления в цепочке** (`a / b * c`) определяется приоритетом и ассоциативностью — об этом §2.2.4.

**Источники:** C2 (гл. 4), C1 (гл. 2); нормативно о делении, остатке и делении на ноль — C5; C3.

### §2.2.2. Сравнения и логические операторы

**Определения.**

- **Операторы сравнения** (относительные и равенства): `<`, `<=`, `>`, `>=`, `==` (равно), `!=` (не равно). Результат — целое: **1**, если отношение истинно, **0**, если ложно.
- **Логические операторы:** `&&` (И), `||` (ИЛИ), `!` (НЕ). Операнды трактуются как «истина/ложь»: **ноль — ложь, любое ненулевое значение — истина**. Результат — снова 0 или 1.
- **Укороченное вычисление** (short-circuit): `&&` вычисляет правый операнд, только если левый истинен; `||` — только если левый ложен. Как только результат ясен по левому операнду, правый не вычисляется вовсе.

> Как эти нули и единицы управляют ветвлением и циклами — тема главы 3. Здесь нам важно лишь, что сравнения и логические операции суть выражения, дающие 0 или 1.

**Интуиция.** `&&` — как последовательные ворота: не прошёл первые (левый ложен) — до вторых дело не дойдёт. `||` — как запасные выходы: нашёл первый открытый (левый истинен) — остальные не проверяешь. Это не только про скорость: правый операнд может быть намеренно «опасным» и выполняться лишь при безопасном левом (пригодится с указателями в главе 6).

**Разобранные примеры.**

*Пример 1. Сравнения возвращают 0 или 1.*

```c
#include <stdio.h>

int main(void)
{
    printf("3 < 5  = %d\n", 3 < 5);    // 1
    printf("3 > 5  = %d\n", 3 > 5);    // 0
    printf("4 == 4 = %d\n", 4 == 4);   // 1
    printf("4 != 4 = %d\n", 4 != 4);   // 0
    return 0;
}
```

Вывод:

```
3 < 5  = 1
3 > 5  = 0
4 == 4 = 1
4 != 4 = 0
```

*Пример 2. Логические операторы и `!`.*

```c
printf("(3<5) && (2>1) = %d\n", (3 < 5) && (2 > 1));  // 1 && 1 = 1
printf("(3>5) || (2>1) = %d\n", (3 > 5) || (2 > 1));  // 0 || 1 = 1
printf("!0 = %d, !1 = %d, !42 = %d\n", !0, !1, !42);   // 1, 0, 0
```

Вывод:

```
(3<5) && (2>1) = 1
(3>5) || (2>1) = 1
!0 = 1, !1 = 0, !42 = 0
```

`!42` равно 0, потому что 42 — «истина», а её отрицание — «ложь». Обратите внимание: `!` любого ненулевого числа даёт 0, а `!0` — 1.

*Пример 3. Диапазон через `&&`.* Проверка «`x` лежит в 1…10» — это два условия, соединённых `&&`:

```c
int x = 7;
printf("1<=x && x<=10 : %d\n", (1 <= x) && (x <= 10));  // 1
int y = 15;
printf("1<=y && y<=10 : %d\n", (1 <= y) && (y <= 10));  // 0
```

Вывод:

```
1<=x && x<=10 : 1
1<=y && y<=10 : 0
```

**Связь с практикой.** Сравнения и логика — язык, на котором формулируются условия: «ввод корректен», «файл открылся», «индекс в границах». Все ветвления и циклы главы 3 управляются именно такими выражениями, поэтому уверенность в их значении (ровно 0 или 1) — база.

**Типичные ошибки и подводные камни.**

- **`=` вместо `==`.** `x = 5` — это присваивание (значение выражения — 5, «истина»), а `x == 5` — сравнение. Перепутать легко, последствия коварны; в главе 3 это станет частой причиной «всегда истинного» условия.
- **«Цепочка» сравнений.** `1 <= x <= 10` в C **не** означает «x между 1 и 10». Вычисляется слева направо: `(1 <= x)` даёт 0 или 1, затем `(0или1) <= 10` — всегда истина. Правильно: `(1 <= x) && (x <= 10)`.
- **Сравнение вещественных на равенство.** `a == b` для `double` ненадёжно (§2.1.3).
- **Путаница «истинности» и печатаемого значения.** Любое ненулевое — истина, но `!x` и сравнения дают именно 0/1; не рассчитывайте, что «истина» — это всегда 1 (она может прийти как 42).

**Источники:** C2 (гл. 5), C1 (гл. 2); нормативно о результате 0/1 и укороченном вычислении — C5; C3.

### §2.2.3. Присваивание, составные операторы и инкременты

**Определения.**

- **Присваивание** `=` записывает значение правого выражения в левый операнд (переменную). При этом `=` само является выражением и «возвращает» присвоенное значение — поэтому `a = b = 0` работает: сначала `b = 0` (значение 0), затем `a = 0`.
- **Составные операторы присваивания:** `+=`, `-=`, `*=`, `/=`, `%=`. Запись `x += 3` — краткая форма `x = x + 3` (и аналогично для остальных).
- **Инкремент** `++` и **декремент** `--` увеличивают/уменьшают переменную на 1. У них две формы: **префиксная** (`++x`) и **постфиксная** (`x++`). Обе меняют `x` одинаково, но как *выражение* дают разное значение: `++x` — новое значение (после увеличения), `x++` — старое (до увеличения).

**Интуиция.** `x++` говорит: «дай мне текущее значение, а потом увеличь переменную»; `++x`: «сначала увеличь, потом дай». Если результат никак не используется (отдельная инструкция `x++;`), разница исчезает — обе просто прибавляют 1.

```
int a = 5;  b = a++;   →  b получает 5 (старое),  a стало 6
int a = 5;  b = ++a;   →  a стало 6,  b получает 6 (новое)
```

**Разобранные примеры.**

*Пример 1. Составные операторы.*

```c
#include <stdio.h>

int main(void)
{
    int a = 5;
    a += 3;  printf("a += 3 -> %d\n", a);   // 8
    a *= 2;  printf("a *= 2 -> %d\n", a);    // 16
    a %= 5;  printf("a %%= 5 -> %d\n", a);   // 1  (16 % 5)
    return 0;
}
```

Вывод:

```
a += 3 -> 8
a *= 2 -> 16
a %= 5 -> 1
```

*Пример 2. Префикс против постфикса.*

```c
int a = 5;
int post = a++;    // post = 5, затем a = 6
printf("post: a=%d post=%d\n", a, post);

int b = 5;
int pre = ++b;     // b = 6, затем pre = 6
printf("pre:  b=%d pre=%d\n", b, pre);
```

Вывод:

```
post: a=6 post=5
pre:  b=6 pre=6
```

*Пример 3. Инкремент как отдельная инструкция.* Когда значение выражения не используется, `x++` и `++x` неотличимы:

```c
int c = 10;
c++;    // то же, что ++c; то же, что c = c + 1;
c--;    // c = c - 1;
printf("c = %d\n", c);   // 10
```

Вывод:

```
c = 10
```

**Связь с практикой.** Составные операторы делают код короче и яснее (`total += price` вместо `total = total + price`), а инкремент — основной способ продвигать счётчик; в главе 3 `i++` станет каноничным шагом цикла. Пока важно твёрдо усвоить разницу префикса и постфикса *как значения выражения* — она ещё аукнется при работе с указателями и массивами.

**Типичные ошибки и подводные камни.**

- **Несколько изменений одной переменной в одном выражении.** `i++ + i++`, `x = x++`, `a[i] = i++` — **неопределённое поведение** (глава 13): стандарт не задаёт, в каком порядке применяются изменения. Не пишите так; меняйте переменную не более одного раза за выражение.
- **Ожидать от `x++` немедленного эффекта в том же выражении.** В `y = x++ + 1` берётся *старое* `x`. Если нужно новое — используйте `++x` или отдельную инструкцию.
- **`=` вместо `==` в условии** (см. §2.2.2) — обратная сторона того, что присваивание есть выражение.
- **Путать `+=` и `=+`.** `x =+ 3` компилятор прочитает как `x = (+3)` (присваивание унарного плюса), а не как прибавление. Составной оператор пишется строго `+=`.

**Источники:** C2 (гл. 4), C1 (гл. 2); нормативно об инкрементах и порядке изменений — C5; C3.

### §2.2.4. Приоритет, ассоциативность и порядок вычислений

**Определения.**

- **Приоритет** (precedence) определяет, какой оператор «связывает» сильнее, то есть выполняется раньше при отсутствии скобок: `*` и `/` — раньше `+` и `-`, как в обычной математике.
- **Ассоциативность** (associativity) решает, в каком порядке выполняются операторы *одного* приоритета: большинство бинарных операторов **левоассоциативны** (`10 - 3 - 2` = `(10 - 3) - 2`), а присваивание и унарные — **правоассоциативны** (`a = b = 0`).
- **Скобки** `( )` переопределяют любой приоритет: то, что в скобках, вычисляется как единое целое.

Упрощённая таблица приоритетов (от высшего к низшему), в пределах этой главы:

| Приоритет | Операторы | Ассоциативность |
|---|---|---|
| высший | постфиксные `x++` `x--`, скобки `()` | слева направо |
| | унарные `!` `-` `+`, префиксные `++x` `--x`, приведение `(тип)` | справа налево |
| | `*` `/` `%` | слева направо |
| | `+` `-` | слева направо |
| | `<` `<=` `>` `>=` | слева направо |
| | `==` `!=` | слева направо |
| | `&&` | слева направо |
| | `\|\|` | слева направо |
| низший | `=` `+=` `-=` … | справа налево |

**Интуиция.** Приоритет и ассоциативность отвечают на вопрос «*что с чем* связано» (как расставить скобки). Отдельный, часто путаемый вопрос — «*в каком порядке* вычисляются операнды». Для `+`, `*` и большинства операторов порядок вычисления операндов стандартом **не задан**: в `f() + g()` неизвестно, что выполнится раньше. Пока у выражений нет побочных эффектов, это неважно; но полагаться на порядок нельзя (исключения — `&&`, `||` и запятая, у которых порядок строго слева направо).

**Разобранные примеры.**

*Пример 1. Приоритет `*` над `+`.*

```c
#include <stdio.h>

int main(void)
{
    printf("2 + 3 * 4   = %d\n", 2 + 3 * 4);     // 14: сначала 3*4
    printf("(2 + 3) * 4 = %d\n", (2 + 3) * 4);   // 20: скобки меняют порядок
    return 0;
}
```

Вывод:

```
2 + 3 * 4   = 14
(2 + 3) * 4 = 20
```

*Пример 2. Левая ассоциативность.*

```c
printf("10 - 3 - 2 = %d\n", 10 - 3 - 2);   // (10-3)-2 = 5, не 10-(3-2)=9
printf("16 / 4 / 2 = %d\n", 16 / 4 / 2);   // (16/4)/2 = 2
```

Вывод:

```
10 - 3 - 2 = 5
16 / 4 / 2 = 2
```

*Пример 3. Сравнение слабее арифметики, но сильнее присваивания.*

```c
int r = 2 + 3 > 4;    // (2+3) > 4  →  5 > 4  →  1; затем r = 1
printf("r = %d\n", r);
int t = 5 == 2 + 3;   // 5 == (2+3) → 5 == 5 → 1
printf("t = %d\n", t);
```

Вывод:

```
r = 1
t = 1
```

Арифметика (`+`) выполнилась раньше сравнения (`>`), а сравнение — раньше присваивания (`=`), поэтому дополнительные скобки не нужны, хотя для читаемости их часто ставят.

**Связь с практикой.** Знание приоритета избавляет от лишних скобок в простых случаях и от ошибок в сложных. Практическое правило зрелого кода: **если сомневаешься в приоритете — ставь скобки**. Они ничего не стоят во время выполнения и делают намерение явным для читателя (в том числе для будущего себя).

**Типичные ошибки и подводные камни.**

- **Смешение «приоритета» и «порядка вычисления операндов».** Приоритет говорит, как группировать; он *не* обещает, что левый операнд вычислится раньше правого. В выражениях с побочными эффектами (`i++`, вызовы функций) не закладывайтесь на порядок — кроме `&&`, `||`, `?:` и запятой.
- **Недооценка низкого приоритета присваивания.** `int c = a > b`; присвоит `c` результат сравнения (0/1), а не «`a`, если больше». Это часто и есть намерение, но об этом стоит помнить.
- **Слишком мало скобок в условиях с логикой.** Смешивая сравнения и `&&`/`||`, ставьте скобки: `(a < b) && (b < c)` читается однозначно.

**Источники:** C2 (гл. 4, 5), C1 (гл. 2); нормативные таблицы приоритета и правила порядка вычислений — C5; C3.

### §2.2.5. Преобразования типов и переполнение

**Определения.**

- **Неявное преобразование** (implicit conversion) выполняет компилятор автоматически. Два главных случая:
  - *В арифметическом выражении* операнды разных типов приводятся к «общему», более широкому: при смешивании целого и `double` целое превращается в `double` (обычные арифметические преобразования, usual arithmetic conversions).
  - *При присваивании* правое значение приводится к типу левой переменной; если тип левого «уже», значение может измениться (усечение дробной части, отбрасывание старших битов).
- **Явное преобразование** (приведение, cast) записывается `(тип)выражение`: `(int)3.9`, `(double)count`. Программист берёт ответственность за преобразование на себя.
- **Переполнение** (overflow) — попытка сохранить значение вне диапазона типа. Для **беззнаковых** типов результат определён: значение «заворачивается» по модулю 2ᴺ. Для **знаковых** целых переполнение — **неопределённое поведение** (глава 13): полагаться на «как получится» нельзя.

**Интуиция.** Преобразование к «более широкому» типу безопасно (значение помещается), к «более узкому» — со срезом. Беззнаковую арифметику удобно представлять как часы: после максимума стрелка идёт снова с нуля (2ᴺ−1 → 0). Знаковое переполнение так представлять *нельзя* — стандарт вообще не гарантирует результата.

```
unsigned char (0..255) как часовой циферблат на 256 делений:

   255 → +1 → 0        (перескок через «полночь»)
     0 → -1 → 255
```

**Разобранные примеры.**

*Пример 1. Целое деление до присваивания в `double` — классическая ловушка.*

```c
#include <stdio.h>

int main(void)
{
    double avg = 7 / 2;        // сначала 7/2 = 3 (int!), потом 3 -> 3.0
    printf("7 / 2   -> %g\n", avg);     // 3
    double avg2 = 7.0 / 2;     // вещественное деление
    printf("7.0 / 2 -> %g\n", avg2);    // 3.5
    printf("(double)7 / 2 -> %g\n", (double)7 / 2);  // 3.5
    return 0;
}
```

Вывод:

```
7 / 2   -> 3
7.0 / 2 -> 3.5
(double)7 / 2 -> 3.5
```

Присваивание переменной типа `double` **не спасает**: деление `7 / 2` целочисленное, потому что оба операнда целые; преобразование в `double` происходит уже над готовым результатом 3. Чтобы получить 3.5, нужно сделать вещественным *операнд* — литералом `7.0` или приведением `(double)7`.

*Пример 2. Явное приведение усекает дробную часть.*

```c
printf("(int)3.9  = %d\n", (int)3.9);    // 3 — к нулю, не округление
printf("(int)-3.9 = %d\n", (int)-3.9);   // -3
```

Вывод:

```
(int)3.9  = 3
(int)-3.9 = -3
```

Приведение `double`→`int` отбрасывает дробную часть (усечение к нулю), а не округляет: `(int)3.9` — это 3, не 4.

*Пример 3. Переполнение: беззнаковое определено, усечение при присваивании.*

```c
#include <stdio.h>
#include <limits.h>

int main(void)
{
    unsigned char uc = 255;
    uc = uc + 1;                  // определено: 256 по модулю 256 = 0
    printf("255 + 1 (u8) = %d\n", uc);

    unsigned int u = 0;
    u = u - 1;                    // определено: 0 - 1 = UINT_MAX
    printf("0 - 1 (u32)  = %u\n", u);

    int big = 300;
    unsigned char small = big;    // 300 не влезает в байт: 300-256
    printf("(u8)300       = %d\n", small);   // 44
    return 0;
}
```

Вывод:

```
255 + 1 (u8) = 0
0 - 1 (u32)  = 4294967295
(u8)300       = 44
```

Все три результата — не «случайность», а определённое стандартом поведение беззнаковой арифметики (по модулю 2ᴺ). А вот знаковый аналог (`int`, доведённый до `INT_MAX + 1`) — уже неопределённое поведение, и мы намеренно не строим на нём примеров.

**Связь с практикой.** Преобразования и переполнение — источник самых коварных, «тихих» ошибок C: программа не падает и не предупреждает, а просто считает неверно. Отсюда две привычки: (1) делать хотя бы один операнд вещественным, когда нужен дробный результат; (2) следить за диапазоном типа, особенно при вычитании беззнаковых (`a - b` при `a < b` даёт огромное число, а не отрицательное) и при сужающих присваиваниях. В главе 13 мы соберём эти случаи в общий каталог неопределённого поведения — и увидим, как Rust закрывает многие из них ещё на этапе компиляции.

**Типичные ошибки и подводные камни.**

- **`double x = a / b;` при целых `a`, `b`.** Деление уже целочисленное; приведите операнд: `(double)a / b`.
- **Вычитание беззнаковых.** `unsigned a = 3, b = 5; a - b` — не −2, а очень большое число. Сравнивая или вычитая беззнаковые, убедитесь, что результат неотрицателен.
- **Знаковое переполнение.** `INT_MAX + 1` — неопределённое поведение; не рассчитывайте, что оно «завернётся» в `INT_MIN`, даже если однажды так вышло.
- **Сужающее присваивание.** Запись `int` в `char`, `double` в `int` теряет данные без ошибки компиляции (иногда — с предупреждением при `-Wconversion`). Убедитесь, что значение помещается.
- **Смешивание знаковых и беззнаковых в сравнении.** `-1 < 0u` может оказаться *ложным*: знаковое −1 преобразуется в огромное беззнаковое. Избегайте сравнений разной знаковости.

**Источники:** C2 (гл. 4, 7), C1 (гл. 2); нормативно об арифметических преобразованиях, беззнаковом «заворачивании» и знаковом переполнении (UB) — C5; C10 (правила о переполнении); C3.

## Часть 2.3. Консольный ввод-вывод

### §2.3.1. Форматированный вывод: `printf`

**Определения.** `printf` (от *print formatted*) выводит текст в терминал по **строке формата** — первому аргументу-строке, в которой обычные символы печатаются как есть, а **спецификаторы преобразования**, начинающиеся с `%`, заменяются значениями следующих аргументов по порядку.

Основные спецификаторы:

| Спецификатор | Тип аргумента | Печатает |
|---|---|---|
| `%d` | `int` | целое десятичное |
| `%u` | `unsigned int` | беззнаковое десятичное |
| `%ld`, `%lld` | `long`, `long long` | длинные целые |
| `%c` | `int`/`char` | одиночный символ |
| `%f` | `double` | вещественное (6 знаков по умолчанию) |
| `%g` | `double` | вещественное «покороче» |
| `%e` | `double` | в научной форме (`1.5e+02`) |
| `%x`, `%o` | `int` | шестнадцатерично / восьмерично |
| `%zu` | `size_t` | результат `sizeof` |
| `%%` | — | литеральный знак `%` |

Между `%` и буквой можно указать **флаги, ширину и точность**: `%5d` — ширина поля 5 (выравнивание вправо), `%-5d` — влево, `%05d` — дополнение нулями, `%.2f` — 2 знака после точки, `%10.2f` — ширина 10 и 2 знака.

> Тонкость: в вызове `printf` аргумент типа `float` автоматически превращается в `double`, поэтому для обоих вещественных типов используется `%f` (отдельного спецификатора для `float` в `printf` нет).

**Интуиция.** Строку формата удобно представлять как бланк с пропусками: обычный текст — напечатанные слова, каждый `%…` — пропуск, в который по очереди вписываются значения аргументов. Число пропусков и их типы должны точно совпадать с числом и типами аргументов.

```
printf("age %d, height %.2f\n", 25, 1.83);
             └─┬┘          └┬─┘   │    │
               └─── 25 ────┘      │    │
                    └─── 1.83 ────┘    │
                                    (перевод строки)
→  age 25, height 1.83
```

**Разобранные примеры.**

*Пример 1. Ширина и выравнивание.*

```c
#include <stdio.h>

int main(void)
{
    int n = 42;
    printf("[%d]\n", n);      // [42]
    printf("[%5d]\n", n);     // [   42]  ширина 5, вправо
    printf("[%-5d]\n", n);    // [42   ]  влево
    printf("[%05d]\n", n);    // [00042]  нулями
    return 0;
}
```

Вывод:

```
[42]
[   42]
[42   ]
[00042]
```

*Пример 2. Точность вещественных.*

```c
double pi = 3.14159265358979;
printf("[%f]\n", pi);       // [3.141593]  — 6 знаков по умолчанию
printf("[%.2f]\n", pi);     // [3.14]
printf("[%10.2f]\n", pi);   // [      3.14]  ширина 10
printf("[%e]\n", pi);       // [3.141593e+00]
```

Вывод:

```
[3.141593]
[3.14]
[      3.14]
[3.141593e+00]
```

*Пример 3. Символ, шестнадцатеричное, процент.*

```c
int code = 81;
printf("char %c has code %d (hex %x)\n", code, code, code);
printf("progress: %d%%\n", 50);   // %% — печатает один знак %
```

Вывод:

```
char Q has code 81 (hex 51)
progress: 50%
```

**Связь с практикой.** `printf` — ваш основной инструмент вывода и отладки на протяжении всего тома I: аккуратное форматирование (выравнивание столбцов, фиксированное число знаков) делает вывод программ читаемым, а вставка «печати переменной» — простейший и самый ходовой способ понять, что программа делает на самом деле.

**Типичные ошибки и подводные камни.**

- **Спецификатор не совпадает с типом аргумента.** `printf("%d\n", 3.5)` (передан `double`, ждут `int`) — **неопределённое поведение**: вывод бессмыслен. Всегда сверяйте спецификатор и тип. Компилятор с `-Wall` часто это ловит — не игнорируйте предупреждение.
- **`%f` для целого или `%d` для вещественного.** Частный, но очень частый случай предыдущего пункта.
- **Забытый `%%`.** Чтобы напечатать знак процента, нужен `%%`; одиночный `%` компилятор сочтёт началом спецификатора.
- **Меньше аргументов, чем спецификаторов.** `printf("%d %d\n", 5)` — второму `%d` нечего печатать, поведение не определено.
- **Русские буквы в выводе на Windows** могут отображаться «кракозябрами» (конфликт кодировок консоли, §1.2.2). Пока выводимый текст пишем латиницей.

**Источники:** C2 (гл. 3, 22), C1 (гл. 1, 7); нормативно о `printf` и спецификаторах — C5; полный список спецификаторов — C6.

### §2.3.2. Форматированный ввод: `scanf`

**Определения.** `scanf` (*scan formatted*) читает данные из терминала по строке формата, похожей на формат `printf`, и **записывает** прочитанные значения в переменные. Ключевое отличие: чтобы функция могла *изменить* вашу переменную, ей нужен **адрес** этой переменной, а не её значение. Адрес берётся оператором `&`: `scanf("%d", &age)`.

> Вспомните §1.1.2: у каждой переменной есть адрес. `&age` — это и есть «адрес переменной `age`». Полный механизм адресов и то, почему без `&` функция не смогла бы записать результат, разбирается в главе 6 (указатели). Пока достаточно правила: **имя переменной в `scanf` пишется с `&` впереди**.

Спецификаторы ввода в основном те же, но есть важные различия для вещественных:

| Спецификатор | Тип переменной | Читает |
|---|---|---|
| `%d` | `int` | целое |
| `%u` | `unsigned int` | беззнаковое целое |
| `%f` | `float` | вещественное |
| `%lf` | `double` | вещественное (обязательно `l`!) |
| `%c` | `char` | один символ |

`scanf` **возвращает** число успешно прочитанных и записанных элементов — по нему проверяют, что ввод корректен. При чтении чисел (`%d`, `%f`, `%lf`) `scanf` пропускает ведущие пробелы и переводы строк.

**Интуиция.** `printf` отдаёт значения наружу — ему достаточно их копий. `scanf` кладёт значения внутрь ваших переменных — ему нужно знать, *куда* класть, то есть адрес. `&` перед именем и есть ответ на вопрос «куда»:

```
scanf("%d", &age);
              │
              └── адрес переменной age: "запиши прочитанное число сюда"
```

**Разобранные примеры.**

*Пример 1. Чтение двух чисел.*

```c
#include <stdio.h>

int main(void)
{
    int age = 0;
    double height = 0;
    printf("Enter age and height: ");
    int got = scanf("%d %lf", &age, &height);   // &, и %lf для double
    printf("read %d items: age=%d height=%g\n", got, age, height);
    return 0;
}
```

Если на вход подать `25 1.83`, вывод:

```
Enter age and height: read 2 items: age=25 height=1.83
```

`got` равно 2 — прочитаны оба значения. Пробел в формате `"%d %lf"` соответствует любому количеству пробелов/переводов строк между числами, поэтому тот же результат получится, если ввести `25` и `1.83` на разных строках.

*Пример 2. Возврат `scanf` сигналит об ошибке.* Если на тот же запрос ввести `abc`, вывод (переменные остались с начальными значениями):

```
Enter age and height: read 0 items: age=0 height=0
```

`got` равно 0: `scanf` не смог прочитать целое из `abc` и ничего не записал. Именно поэтому переменные важно инициализировать (`= 0`), а возврат `scanf` — проверять (проверка через ветвление — глава 3; пока мы лишь печатаем `got`).

*Пример 3. Правильные спецификаторы для `double`.* В `scanf` для `double` нужен именно `%lf`; `%f` в `scanf` предназначен для `float`. Это отличается от `printf`, где для обоих типов используется `%f`:

```c
float  f = 0;
double d = 0;
scanf("%f", &f);    // float  — %f
scanf("%lf", &d);   // double — %lf (в scanf!)
```

**Связь с практикой.** `scanf` превращает программу из «вычислителя одного набора чисел» в инструмент, реагирующий на пользователя. В связке с ветвлениями и циклами (глава 3) он станет основой интерактивных утилит. Дисциплина «инициализируй переменную, читай через `&`, проверяй возврат» закладывается уже сейчас.

**Типичные ошибки и подводные камни.**

- **Забытый `&`.** `scanf("%d", age)` вместо `&age` передаёт значение вместо адреса — типичный сбой (часто аварийное завершение). Компилятор с `-Wall` предупреждает — читайте предупреждения.
- **`%f` вместо `%lf` для `double`.** В `scanf` это разные вещи: `%f` ждёт `float`, `%lf` — `double`. Несоответствие ведёт к неверному чтению (неопределённое поведение).
- **Игнорирование возврата.** Не проверив, сколько элементов прочитано, легко работать с мусором при неверном вводе.
- **Смешение `%c` и чисел.** В отличие от `%d`/`%lf`, спецификатор `%c` *не пропускает* пробелы и переводы строк и прочитает, например, оставшийся от предыдущего ввода символ конца строки. Аккуратная работа с `%c` требует приёмов, к которым мы вернёмся, когда появятся циклы (глава 3) и строки (глава 5).

**Источники:** C2 (гл. 3, 22), C1 (гл. 7); нормативно о `scanf` и спецификаторах ввода — C5; различие `%f`/`%lf` и возвращаемое значение — C6.

### §2.3.3. Ошибки формата и буферизация

**Определения.**

- **Буфер** (buffer) — промежуточная область памяти, где ввод-вывод накапливается перед фактической передачей. Стандартный вывод обычно **буферизуется построчно** при работе с терминалом: символы копятся в буфере и «сбрасываются» на экран при переводе строки `\n`, при завершении программы или при явном сбросе. Ввод с клавиатуры тоже проходит через буфер: система обычно отдаёт введённое программе целой строкой, после нажатия Enter.
- **Согласование формата и аргументов** — общее правило и `printf`, и `scanf`: число, порядок и типы спецификаторов должны точно соответствовать числу, порядку и типам аргументов. Нарушение — неопределённое поведение.

**Интуиция.** Буфер — как накопительный лоток: программа «пишет» в лоток, а на экран содержимое высыпается порциями (по строкам). Отсюда практическое следствие: приглашение без завершающего `\n` (`printf("Enter x: ")`) может остаться в буфере и появиться на экране одновременно с последующим выводом, а не «до ввода», как ожидалось. Это не ошибка программы, а свойство буферизации.

```
printf("Enter x: ");   // без \n — оседает в буфере
scanf("%d", &x);       // при чтении/завершении буфер сбрасывается
```

**Разобранные примеры.**

*Пример 1. Сквозная программа: ввод, вычисления, форматированный вывод.* Соберём воедино материал главы — прочитаем два целых и напечатаем несколько величин (без ветвлений: возврат `scanf` просто печатаем):

```c
#include <stdio.h>

int main(void)
{
    int a = 0, b = 0;
    printf("Enter two integers: ");
    int got = scanf("%d %d", &a, &b);
    printf("scanf read %d items\n", got);
    printf("a = %d, b = %d\n", a, b);
    printf("sum        = %d\n", a + b);
    printf("difference = %d\n", a - b);
    printf("int avg    = %d\n", (a + b) / 2);     // целочисленное
    printf("real avg   = %.2f\n", (a + b) / 2.0); // вещественное
    return 0;
}
```

При вводе `7 4`:

```
Enter two integers: scanf read 2 items
a = 7, b = 4
sum        = 11
difference = 3
int avg    = 5
real avg   = 5.50
```

Здесь видно всё сразу: приглашение без `\n` слилось со строкой вывода (буферизация), целочисленное и вещественное деление дали `5` и `5.50`, `scanf` прочитал 2 элемента.

*Пример 2. Тот же код на неверном вводе.* При вводе `x` (не число):

```
Enter two integers: scanf read 0 items
a = 0, b = 0
sum        = 0
difference = 0
int avg    = 0
real avg   = 0.00
```

`got` равен 0, и поскольку переменные инициализированы нулями, программа не работает с мусором — она считает по `a = b = 0`. Осмысленно среагировать на такой случай (например, сообщить об ошибке) мы сможем в главе 3.

*Пример 3. Каталог несоответствий формата.* Каждая строка ниже — ошибка согласования (не пишите так); приведены как «⛔ так делать нельзя»:

```c
printf("%d\n", 3.14);   // ⛔ %d ждёт int, передан double
printf("%f\n", 42);     // ⛔ %f ждёт double, передан int
printf("%d %d\n", 5);   // ⛔ аргументов меньше, чем спецификаторов
scanf("%d", age);       // ⛔ забыт &  (нужно &age)
scanf("%f", &d);        // ⛔ d — double, нужен %lf
```

Первые три — ошибки `printf`, последние две — `scanf`. Все они дают неопределённое поведение и/или неверный результат; большинство ловится флагом `-Wall`, поэтому предупреждения компилятора здесь особенно ценны.

**Связь с практикой.** Понимание буферизации снимает первое недоумение новичка («почему приглашение печатается после ввода?») и готовит к более тонким эффектам при вводе в цикле (глава 3). А железная дисциплина согласования формата и аргументов — то, что отделяет работающую программу ввода-вывода от источника трудноуловимых сбоев.

**Типичные ошибки и подводные камни.**

- **Ожидание, что приглашение без `\n` появится строго до ввода.** Из-за буферизации оно может «прилипнуть» к последующему выводу; это не ошибка. (Форсированный сброс буфера — тема будущих глав.)
- **Несоответствие формата и аргументов** (см. пример 3) — самая частая причина «мистического» вывода; ловится `-Wall`.
- **Работа с мусором при неудачном вводе.** Инициализируйте переменные и помните про возврат `scanf`; полноценно реагировать научимся в главе 3.
- **Уверенность, что ввод приходит посимвольно.** Обычно строка отдаётся программе целиком после Enter — это скажется на работе с `%c` и при построчном чтении.

**Источники:** C2 (гл. 3, 22), C1 (гл. 1, 7); нормативно о потоках, буферизации и согласовании формата — C5; поведение спецификаторов — C6.

## Резюме главы

- **Переменная** — именованный участок памяти известного типа; объявление отводит место, инициализация задаёт начальное значение (инициализировать стоит всегда). Тип определяет размер (`sizeof`, спецификатор `%zu`) и трактовку байтов.
- **Типы данных:** целые (`char`, `short`, `int`, `long`, `long long`, знаковые и `unsigned`) с диапазонами из `<limits.h>`; их точные размеры платформозависимы (гарантированы лишь минимумы и порядок). Символьный `char` — это код символа (маленькое целое). Вещественные `float`/`double` хранят числа *приближённо* — сравнивать их через `==` нельзя.
- **Выражения и операторы:** арифметика (целочисленное `/` отбрасывает дробь; `%` — остаток; деление на ноль — неопределённое поведение), сравнения и логика (результат 0/1, укороченное вычисление), присваивание и составные операторы, инкремент/декремент (префикс даёт новое значение, постфикс — старое). Порядок задают приоритет и ассоциативность; сомневаешься — ставь скобки.
- **Преобразования типов:** неявные в выражениях (к более широкому типу) и при присваивании (возможное усечение); явное приведение `(тип)`. **Переполнение:** беззнаковое определено (по модулю 2ᴺ), знаковое — неопределённое поведение. Классические ловушки: `int`-деление до присваивания `double`, вычитание беззнаковых.
- **Ввод-вывод:** `printf` форматирует вывод по спецификаторам (`%d`, `%f`, `%c`, ширина/точность); `scanf` читает в переменные *по адресу* (`&x`), возвращает число прочитанных элементов, для `double` требует `%lf`. Согласование формата и аргументов обязательно; вывод буферизуется.

**Дальше.** Теперь программа умеет хранить и вычислять данные и обмениваться ими с пользователем — но выполняется строго сверху вниз, одинаково при любых входных данных. Глава 3 снимает это ограничение: **ветвления** (`if`/`else`, `switch`) заставят программу выбирать путь в зависимости от условий (тех самых выражений с результатом 0/1 из §2.2.2), а **циклы** (`while`, `for`, `do-while`) — повторять действия, и мы наконец сможем осмысленно реагировать на неверный ввод и обрабатывать много значений подряд.
