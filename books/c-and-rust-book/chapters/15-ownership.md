# Глава 15. Владение и заимствование

## Введение

Это центральная глава тома II — и, пожалуй, всего самоучителя. Всё, ради чего Rust затевался, сосредоточено здесь: **владение** (ownership), **заимствование** (borrowing) и **срезы** (slices).

Хорошая новость: главную идею вы уже знаете. Глава 7 научила вас дисциплине динамической памяти — «у каждого блока ровно один владелец; кто выделил — тот освобождает; освободил — не трогай». Глава 13 показала, что бывает, когда дисциплина держится на комментариях: use-after-free, двойное освобождение, висячие указатели — строки каталога UB, которые ловились только санитайзерами, и то на тех прогонах, где ошибка соизволила проявиться. Глава 14 дала синтаксис Rust, но мы нарочно ходили по краю: ни одна программа не передавала владение и не брала ссылок.

Теперь — сам закон. В этой главе дисциплина главы 7 перестаёт быть памяткой на полях и становится **правилами языка**, которые компилятор проверяет на каждой сборке: без запуска, без санитайзеров, для всех путей выполнения сразу.

После изучения главы вы будете уметь:

- объяснять три правила владения и говорить, *где именно* освобождается каждое значение (`drop`);
- предсказывать, что произойдёт при присваивании и передаче в функцию: перемещение (move) или копия (`Copy`), — и выбирать между move, `clone` и ссылкой;
- писать функции, которые *заимствуют* значения через `&T` и `&mut T`, и объяснять правило «сколько угодно читателей либо один писатель»;
- читать ошибки заимствования — E0382, E0499, E0502, E0106, E0597 — как рассказ о недопущенной ошибке памяти;
- пользоваться срезами `&[T]` и `&str` — безопасной заменой пары «указатель + длина» из C.

Весь код главы прогнан на эталонном окружении курса: rustc/cargo **1.97.0 stable** (Windows, `x86_64-pc-windows-gnu`, издание 2024); выводы программ и тексты ошибок приведены дословно. Примеры одиночных файлов собирались `rustc файл.rs` — как в §14.1.1; всё то же самое работает и в cargo-проекте.

## Часть 15.1. Владение

### §15.1.1. Владелец, область видимости и drop

**Определения.** Три правила владения — фундамент всей модели памяти Rust (R1, гл. 4; впервые мы видели их в §13.2.1):

1. У каждого значения есть **владелец** (owner) — переменная.
2. Владелец **ровно один** в каждый момент времени.
3. Когда владелец покидает **область видимости** (scope), значение **освобождается**: Rust вызывает функцию `drop`.

Область видимости вам знакома по C: блок в фигурных скобках (§4.2.1 тома I работал так же — локальные переменные умирали на закрывающей скобке). Разница в том, *что именно* происходит на скобке. В C на ней исчезали только автоматические переменные — стековые; всё, что вы взяли у кучи через `malloc`, продолжало жить, пока вы вручную не позовёте `free` (§7.1.2). В Rust закрывающая скобка освобождает **всё, чем владели локальные переменные**, — включая память в куче. Вызов `drop` вставляет компилятор, автоматически и детерминированно: в точности там, где владелец уходит.

Нормативный порядок освобождения задаёт справочник языка (R7, Destructors): при выходе из области видимости переменные уничтожаются **в порядке, обратном объявлению** (последняя объявленная — первой). Наблюдать это мы сможем в главе 22, когда научимся вешать на уничтожение собственный код; пока достаточно знать правило.

**Интуиция.** Представьте, что к каждой закрывающей скобке компилятор дописывает невидимые `free` — ровно те, которые вы в главе 7 писали руками и по одному разу забывали:

```
C (глава 7): вручную                     Rust: вставит компилятор
──────────────────────────               ──────────────────────────
{                                        {
    char *s = str_dup("hello");             let s = String::from("hello");
    /* ... работа ... */                     // ... работа ...
    free(s);      /* не забыть! */       }   // ← здесь drop(s): наш free,
}                                            //   привязанный к скобке
```

`String` — первый тип Rust с данными в куче, который мы разбираем всерьёз: строка переменной длины, как `char *` из `str_dup` (§7.1.2). Устройство `String` подробно ждёт нас в главе 17; в этой главе нам хватит того, что он **владеет буфером в куче** — и потому на нём видна вся механика владения.

**Разобранные примеры.**

*Пример 1 — скобка освобождает, присваивание тоже.* Файл `scope1.rs`:

```rust
fn main() {
    {
        let s = String::from("hello");
        println!("inside: {s}");
    }                              // ← drop(s): буфер "hello" освобождён
    let mut s = String::from("hello");
    println!("before: {s}");
    s = String::from("ahoy");      // ← старое "hello" освобождено ПРЯМО ЗДЕСЬ
    println!("{s}, world!");
}                                  // ← drop(s): освобождён буфер "ahoy"
```

Вывод (проверено):

```
inside: hello
before: hello
ahoy, world!
```

Обратите внимание на присваивание `s = String::from("ahoy")`: старое значение `"hello"` осталось без владельца — и Rust освобождает его немедленно, в момент присваивания, не дожидаясь конца блока (R1, гл. 4; R7: присваивание запускает деструктор своего левого операнда). В C аналогичная строка без предварительного `free(s)` была бы утечкой — первая строка нашего каталога ошибок §7.2.1.

*Пример 2 — освободить раньше скобки: функция `drop`.* Иногда значение нужно отпустить до конца блока — в C вы просто звали `free` посередине функции. В Rust для этого есть готовая функция стандартной библиотеки `drop(значение)` (R9). Но попробуйте воспользоваться значением после неё — файл `dropfn.rs`:

```rust
fn main() {
    let s = String::from("hello");
    drop(s);
    println!("{s}");
}
```

⛔ Не компилируется (проверено):

```
error[E0382]: borrow of moved value: `s`
 --> dropfn.rs:4:16
  |
2 |     let s = String::from("hello");
  |         - move occurs because `s` has type `String`, which does not implement the `Copy` trait
3 |     drop(s);
  |          - value moved here
4 |     println!("{s}");
  |                ^ value borrowed here after move
  |
help: consider cloning the value if the performance cost is acceptable
```

Сравните с §7.2.1: `puts(s)` после `free(s)` — это use-after-free, который компилятор C пропускал молча (ловил только ASan, и только в запуске). Здесь тот же сюжет — отказ **компиляции**. Устроена `drop` неожиданно просто: её тело **пустое** — она лишь принимает значение *по владению*, и значение уничтожается на выходе из неё, по обычному правилу 3 (R9: «поскольку значение перемещено в функцию, оно автоматически уничтожается до возврата»). Слово `move` в сообщении об ошибке — как раз про эту передачу владения; ему посвящён следующий параграф.

*Пример 3 — правило 2: владелец ровно один.* Правило «ровно один владелец» — то самое, что в главе 7 мы формулировали как «за блок отвечает ровно один указатель, остальные — гости». Что происходит, когда владельца пытаются «раздвоить» присваиванием, — тоже §15.1.2; здесь зафиксируем сами слова: **не бывает двух переменных, владеющих одним значением**. Именно поэтому у скобки никогда нет вопроса «а кто освобождает?» — и двойное освобождение (строка 4 каталога §13.1.2) не выразимо в безопасном Rust в принципе.

**Связь с практикой.** Приём «освобождение привязано к времени жизни владельца» существовал и до Rust: в C++ его зовут RAII (Resource Acquisition Is Initialization), и книга Rust прямо отсылает к нему (R1, гл. 4). Практическое следствие для вас: **в Rust-коде нет пар «выделил/освободил»**, которые нужно выверять взглядом, — нет и наших выверенных комментариями правил из §7.1.2 («кто выделил — тот освобождает»). Освобождение — не действие программиста, а свойство структуры программы: посмотрел, где кончается блок владельца, — знаешь, где умирает значение.

**Типичные ошибки и подводные камни.**

- **«Это сборщик мусора».** Нет. Сборщик мусора — отдельный механизм *времени выполнения*, который ищет недостижимые объекты «когда-нибудь». `drop` вставляется *компилятором* в фиксированное место программы и стоит ровно столько же, сколько ваш ручной `free`; никакой фоновой работы нет (R1, гл. 4).
- **Ждать, что переприсваивание копит мусор до конца блока.** Пример 1: старое значение освобождается в момент присваивания, немедленно.
- **Забыть, что порядок drop — обратный.** В одном блоке последняя объявленная переменная умирает первой (R7). Пока это не влияет ни на один наш пример, но станет важно, когда значения начнут зависеть друг от друга (глава 22).
- **Пытаться «позвать free дважды», обходя drop.** Второго `drop(s)` компилятор не пропустит по той же причине, что и пример 2: после первого вызова значение уже перемещено.

**Источники:** R1 (гл. 4: правила владения, scope, drop, RAII-примечание, немедленное освобождение при переприсваивании); R7 (Destructors: момент вызова, обратный порядок, присваивание запускает деструктор); R9 (`std::mem::drop`: пустое тело, уничтожение через передачу владения). Сторона C: §4.1, §7.1.2, §7.2.1, §13.1.2, §13.2.1 курса.

### §15.1.2. Перемещение: владение переезжает

**Определения.** Значение `String` состоит из двух частей (R1, гл. 4; подробное устройство — глава 17):

- **на стеке** — маленькая тройка: указатель на буфер (`ptr`), длина (`len`), ёмкость (`cap`);
- **в куче** — сам буфер с байтами строки.

Присваивание `let s2 = s1;` копирует **только тройку со стека** — указатель, длину, ёмкость. Буфер в куче не копируется. Ровно так же вело бы себя присваивание `char *q = p;` в C — два указателя на один блок. Но дальше — развилка: C оставил бы обе переменные «живыми» (и обе стали бы кандидатами на `free` — здравствуй, двойное освобождение), а Rust **объявляет старую переменную недействительной**. Эта операция — копия тройки плюс аннулирование источника — называется **перемещением** (move). Книга Rust подчёркивает: это не «поверхностная копия» (shallow copy) из других языков именно потому, что источник аннулируется (R1, гл. 4).

Перемещение происходит везде, где значение «уходит в новое место» (R1, R6):

- присваивание: `let s2 = s1;`
- передача аргумента: `f(s)` — значение переезжает в параметр функции;
- возврат из функции: `return s` / хвостовое выражение — значение переезжает к вызывающему.

**Интуиция.** Состояние памяти после `let s2 = s1;`:

```
        стек                                куча
  s1  ┌─────┬────┐
 (❌) │ ptr │ ●──┼────────────┐
      │ len │ 5  │            ▼
      │ cap │ 5  │      ┌───┬───┬───┬───┬───┐
      └─────┴────┘      │ h │ e │ l │ l │ o │
  s2  ┌─────┬────┐      └───┴───┴───┴───┴───┘
      │ ptr │ ●──┼────────────▲
      │ len │ 5  │
      │ cap │ 5  │
      └─────┴────┘
```

Буфер один, тройки две — но имя `s1` перечёркнуто: компилятор помнит, что владение уехало, и любое обращение к `s1` отклонит. У скобки блока освобождение выполнит только `s2`. Сравните с «двумя указателями на один блок» из §7.2.1 — там обе переменные были равноправны, и защищала вас только дисциплина.

**Разобранные примеры.**

*Пример 1 — использование после перемещения.* Витрина §13.2.1, теперь на нашем эталоне 1.97 — файл `move1.rs`:

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1;
    println!("{s1}, world! {s2}");
}
```

⛔ Не компилируется (проверено):

```
error[E0382]: borrow of moved value: `s1`
 --> move1.rs:4:16
  |
2 |     let s1 = String::from("hello");
  |         -- move occurs because `s1` has type `String`, which does not implement the `Copy` trait
3 |     let s2 = s1;
  |              -- value moved here
4 |     println!("{s1}, world! {s2}");
  |                ^^ value borrowed here after move
  |
help: consider cloning the value if the performance cost is acceptable
  |
3 |     let s2 = s1.clone();
  |                ++++++++
```

Прочитайте сообщение как протокол расследования — все четыре пометки на своих местах: *почему* случился move (тип `String` не умеет копироваться неявно — про `Copy` расскажет §15.1.3); *где* значение уехало (строка 3); *где* к нему обратились после переезда (строка 4); и *легальный выход* (`help`: клонируйте, если цена приемлема — с 1.97 компилятор даже показывает готовую правку). Если бы Rust этого не запретил, у скобки `main` оба имени попытались бы освободить один буфер — двойное освобождение, «одна из ошибок безопасности памяти» (R1, гл. 4).

*Пример 2 — move в функцию.* Файл `movefn.rs`:

```rust
fn takes_ownership(s: String) {
    println!("got: {s}");
}                                  // ← здесь drop(s): буфер освобождён

fn main() {
    let s = String::from("hello");
    takes_ownership(s);
    println!("still here? {s}");
}
```

⛔ Не компилируется (проверено):

```
error[E0382]: borrow of moved value: `s`
 --> movefn.rs:8:28
  |
6 |     let s = String::from("hello");
  |         - move occurs because `s` has type `String`, which does not implement the `Copy` trait
7 |     takes_ownership(s);
  |                     - value moved here
8 |     println!("still here? {s}");
  |                            ^ value borrowed here after move
  |
note: consider changing this parameter type in function `takes_ownership` to borrow instead
      if owning the value isn't necessary
 --> movefn.rs:1:23
  |
1 | fn takes_ownership(s: String) {
  |    ---------------    ^^^^^^ this parameter takes ownership of the value
  |    |
  |    in this function
help: consider cloning the value if the performance cost is acceptable
```

Передача аргумента — то же перемещение, что и присваивание (R1: «передача переменной в функцию перемещает или копирует, как присваивание»; R6). Значение переехало в параметр `s` функции — и умерло на её закрывающей скобке. Присмотритесь к `note:`: компилятор советует «поменять тип параметра на заимствование, если владение не нужно» — это он отправляет вас в часть 15.2, к ссылкам.

*Пример 3 — владение туда и обратно.* Возврат значения — тоже move. Файлы `ret1.rs` и `measure.rs` (оба проверены):

```rust
fn gives_ownership() -> String {
    let s = String::from("yours");
    s                                  // владение уезжает к вызывающему
}

fn takes_and_gives_back(s: String) -> String {
    s                                  // принял — и вернул обратно
}

fn main() {
    let s1 = gives_ownership();
    let s2 = String::from("hello");
    let s3 = takes_and_gives_back(s2); // s2 уехала и вернулась под именем s3
    println!("s1 = {s1}, s3 = {s3}");
}
```

```
s1 = yours, s3 = hello
```

А если функция должна и *использовать* значение, и вернуть вызывающему *результат*? Приходится возвращать кортеж (§14.2.2) — «и строку обратно, и ответ»:

```rust
fn measure(s: String) -> (String, usize) {
    let len = s.len();
    (s, len)                           // вернуть и строку, и её длину
}

fn main() {
    let s = String::from("hello");
    let (s, len) = measure(s);         // затенение: новое s — вернувшаяся строка
    println!("len of '{s}' is {len}");
}
```

```
len of 'hello' is 5
```

Работает — но согласитесь, церемонии много: функция всего лишь *посмотрела* на строку, а оформлять пришлось полный переезд туда-обратно. Книга Rust называет это ровно тем словом, которое напрашивается, — «утомительно» (R1, гл. 4). Часть 15.2 избавит нас от этой церемонии: там функция сможет **одолжить** значение, не забирая владения.

**Как это было в C.** Сопоставление механики присваивания:

| Ситуация | C (том I) | Rust |
|---|---|---|
| `q = p;` для `char *` | два указателя на один блок; оба «живые», владелец — по договорённости (§7.1.2) | move: тройка скопирована, источник аннулирован — владелец снова один |
| Передача в функцию | копия указателя; владение «по документации» (§7.1.2) | move в параметр; функция — новый владелец, освободит на своей скобке |
| Возврат из функции | вернуть указатель; вызывающий *должен догадаться*, что теперь освобождать ему | move к вызывающему; компилятор *знает*, что владелец теперь он |
| Забыли про переезд | UAF/double free в рантайме — если повезёт, поймает ASan (§7.2.2) | E0382 на компиляции, с готовой подсказкой |

**Связь с практикой.** Перемещение — это **дёшево**, и это осознанный выбор языка: копируются три машинных слова, что бы ни лежало в куче — 5 байт или 5 гигабайт. Comprehensive Rust отмечает, что это умолчание — противоположность C++, где присваивание по умолчанию копирует данные (R6). Правило чтения кода Rust: увидели передачу `String` по значению — читайте «владение переехало», а не «строку скопировали». Дорогая операция в Rust всегда видна глазами — ей посвящён следующий параграф.

**Типичные ошибки и подводные камни.**

- **Читать E0382 как «Rust запрещает присваивание».** Присваивание состоялось; запрещено пользоваться *старым именем* после переезда. Мы предупреждали об этом ещё в §13.2.1 — теперь у вас есть полный разбор.
- **Писать функции, забирающие `String`, «на всякий случай».** Компилятор сам намекает (`note:` в примере 2): если владение не нужно — берите ссылку (часть 15.2). Забирать владение стоит, когда функция действительно *оставляет значение себе* или *потребляет* его.
- **Думать, что после move переменная «пустая».** Ни нулей, ни «пустой строки» там нет — имя просто перестаёт существовать для проверок компилятора. Это не рантайм-состояние, а статический факт.
- **Возвращать владение кортежем там, где просится ссылка.** `measure` — учебная демонстрация неудобства, а не образец для подражания; идиоматичное решение появится в §15.2.1.

**Источники:** R1 (гл. 4: устройство String — ptr/len/cap, семантика move, инвалидация источника, double free, move в функции и из функций, «утомительно»); R6 (Memory Management / Move Semantics: присваивание передаёт владение, сравнение с умолчанием C++); R4 (гл. об Ownership/Moves — механика перемещения, согласуется с R1/R6). Сторона C: §7.1.2, §7.2.1–7.2.2, §13.2.1 курса.

### §15.1.3. Clone и Copy: когда копия — настоящая

**Определения.** Из E0382 мы уже дважды видели формулу «does not implement the `Copy` trait». Разложим её на два механизма (R1, гл. 4; R6):

- **`clone()`** — *явная глубокая копия*: метод, который копирует и тройку на стеке, **и буфер в куче**. После `let s2 = s1.clone();` существуют две независимые строки; обе переменные действительны, у каждой свой буфер и своё освобождение.
- **`Copy`** — *свойство типа*, разрешающее неявное побитовое копирование. Если тип «умеет `Copy`», присваивание и передача в функцию не перемещают, а **копируют**: источник остаётся действительным. Слово «трейт» (trait) здесь — термин Rust для «свойства/умения типа»; систематически трейты нас ждут в главе 19, пока достаточно интуиции «пометка на типе».

Какие типы `Copy` (R1, гл. 4; массивы — R9):

- все целые (`i32`, `u8`, …), все вещественные (`f64`, …), `bool`, `char`;
- кортежи, **если все их элементы `Copy`**: `(i32, i32)` — да, `(i32, String)` — нет;
- массивы `[T; N]`, если `T` — `Copy` (R9, документация типа массива).

Общее правило от авторов языка: `Copy` — для «групп простых скалярных значений»; всё, что требует выделения памяти или является ресурсом, `Copy` быть не может (R1). Более того, компилятор **запрещает** объявлять `Copy` тип, которому нужны особые действия при уничтожении (реализующий `Drop`): «побитово скопировать и освободить дважды» — несовместимые желания (R1, гл. 4).

**Интуиция.** Решающий вопрос — «есть ли за значением что-то, кроме его собственных байтов?»:

```
                     присваивание let b = a;
                                │
              ┌─────────────────┴─────────────────┐
              ▼                                   ▼
   только байты на стеке               за значением буфер/ресурс
   (i32, bool, char, f64,              (String, — а позже Vec, File...)
    (i32, i32), [u8; 4]...)                       │
              │                                   ▼
              ▼                              MOVE: тройка скопирована,
   COPY: побитовая копия,                    источник аннулирован;
   обе переменные живут дальше               настоящая копия — только clone()
```

C-шная интуиция работает и здесь: `int b = a;` в C всегда был честной копией — потому что в `int` нет ничего, кроме четырёх байт. Проблемы начинались с указателей — «скопировал указатель, а думал, что скопировал данные» (§6.1, §7.2.1). Rust развёл эти случаи по разным механизмам на уровне типов.

**Разобранные примеры.**

*Пример 1 — clone: две независимые строки.* Файл `clone1.rs`:

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1.clone();
    println!("s1 = {s1}, s2 = {s2}");
}
```

```
s1 = hello, s2 = hello
```

Обе переменные действительны — сравните с ⛔-примером 1 из §15.1.2, где без `clone` та же печать не собиралась. Это аналог нашего `str_dup` из §7.1.2 — «новый блок + копия содержимого», только освобождение обеих строк снова на компиляторе.

*Пример 2 — Copy-типы: присваивание не аннулирует.* Файл `copy1.rs`:

```rust
fn main() {
    let x = 5;
    let y = x;
    println!("x = {x}, y = {y}");

    let p = (3, 7);
    let q = p;
    let (px, py) = p;
    let (qx, qy) = q;
    println!("p = ({px}, {py}), q = ({qx}, {qy})");

    let a = [1, 2, 3];
    let b = a;
    println!("a[0] = {}, b[0] = {}, a.len() = {}", a[0], b[0], a.len());
}
```

```
x = 5, y = 5
p = (3, 7), q = (3, 7)
a[0] = 1, b[0] = 1, a.len() = 3
```

`i32`, кортеж `(i32, i32)`, массив `[i32; 3]` — всё `Copy`: источники (`x`, `p`, `a`) живы после присваивания. То же и с функциями — вспомните `makes_copy(x)` в §15.1.2: `x` остался действителен, потому что `i32` копируется, а не переезжает.

*Пример 3 — один не-Copy элемент «заражает» весь кортеж.* Файл `copy_string_tuple.rs`:

```rust
fn main() {
    let t = (5, String::from("hi"));
    let u = t;
    println!("{} {}", t.0, u.1);
}
```

⛔ Не компилируется (проверено):

```
error[E0382]: borrow of moved value: `t`
 --> copy_string_tuple.rs:4:23
  |
2 |     let t = (5, String::from("hi"));
  |         - move occurs because `t` has type `(i32, String)`, which does not implement the `Copy` trait
3 |     let u = t;
  |             - value moved here
4 |     println!("{} {}", t.0, u.1);
  |                       ^^^ value borrowed here after move
```

Кортеж `(i32, String)` не `Copy` — внутри него ресурс. Обратите внимание: ошибка указана даже для обращения к `t.0` — числу, которое «могло бы скопироваться»; переехал весь кортеж целиком.

**Связь с практикой.** Дизайн языка здесь сознательно асимметричен: **дешёвое — неявно, дорогое — явно**. Копия `i32` — четыре байта, пусть происходит сама. Копия `String` может стоить сколько угодно — поэтому она всегда выглядит как видимый вызов `.clone()`: книга Rust прямо называет его «визуальным индикатором, что происходит нечто иное» (R1, гл. 4), а Comprehensive Rust уточняет разницу механизмов: `Copy` — побитовая копия без пользовательской логики, `Clone` — обычный код, который может делать что угодно (R6). На код-ревью каждый `clone` — повод спросить «а нельзя ли одолжить?» (часть 15.2). И ещё следствие, знакомое по §15.1.1: `drop(x)` для Copy-типа — пустая операция: в функцию уезжает *копия*, оригинал живёт дальше (R9).

**Типичные ошибки и подводные камни.**

- **Бояться `clone` как огня.** `clone` — легальный и часто правильный инструмент (нужны *две независимые* строки — клонируйте). Ошибка — клонировать *рефлекторно*, лишь бы замолчал E0382: сперва спросите себя, не нужна ли здесь ссылка.
- **Путать `Copy` и `Clone`.** `Copy` — неявный, побитовый, только для «плоских» типов; `Clone` — явный вызов, есть почти у всех стандартных типов (и у `String`, и у `i32`). Кстати, `x.clone()` для `i32` тоже работает — просто он эквивалентен копии.
- **Ожидать `Copy` у кортежа/массива «с ресурсом внутри»**. Пример 3: одно поле `String` — и весь агрегат перемещается. Правило композиционное: `Copy` есть только там, где `Copy` у всех частей.
- **Пытаться сделать «свой тип с Copy и Drop сразу».** Запрещено компилятором (R1); осмысленный выбор для собственных типов появится в главах 16 и 19 (`derive`).

**Источники:** R1 (гл. 4: clone как глубокая копия и «визуальный индикатор», список Copy-типов, кортежи, несовместимость Copy и Drop, общее правило «скаляры — да, ресурсы — нет»); R6 (Memory Management / Copy types: Copy — побитово и неявно, Clone — явно и с произвольной логикой); R9 (массивы `[T; N]` реализуют Copy при `T: Copy`; `drop` для Copy-типов — «ничего не делает»). Сторона C: §6.1, §7.1.2, §7.2.1 курса.

## Часть 15.2. Ссылки и заимствование

### §15.2.1. Ссылка: доступ без владения

**Определения.** **Ссылка** (reference) `&T` — это адрес, по которому можно обратиться к значению, *принадлежащему другой переменной*; в отличие от указателя C, ссылка **гарантированно указывает на действительное значение своего типа всё время своей жизни** (R1, гл. 4). Создание ссылки называется **заимствованием** (borrowing): вы одалживаете значение, не становясь владельцем, — а значит, когда ссылка исчезает, **drop не вызывается**: освобождать чужое — не ваша забота.

Синтаксис знаком по C (§6.1) до степени смешения:

- `&s` — взять ссылку на `s` (как `&x` в C — взять адрес);
- `&T` в сигнатуре — параметр-ссылка (аналог `const T *`);
- `*r` — разыменование (как `*p` в C).

Официальное имя `&T` — **разделяемая ссылка** (shared reference): пока она существует, значение нельзя менять, зато таких ссылок может быть сколько угодно (R7, Pointer types). По умолчанию ссылки, как и переменные, **только для чтения**; изменяющим ссылкам `&mut` посвящён §15.2.2.

**Интуиция.** Владение — это «ответственность за освобождение». Ссылка — это «пропуск посетителя»: смотреть можно, выносить и перестраивать — нет, и на выходе за вами не запирают здание. Схема памяти для `let r = &s;`:

```
      стек                 стек                куча
  r ┌─────┐            s ┌─────┬────┐
    │  ●──┼───────────▶  │ ptr │ ●──┼──▶ ┌───┬───┬───┬───┬───┐
    └─────┘              │ len │ 5  │    │ h │ e │ l │ l │ o │
   ссылка НА s           │ cap │ 5  │    └───┴───┴───┴───┴───┘
   (не владеет)          └─────┴────┘
                         владелец буфера
```

**Разобранные примеры.**

*Пример 1 — функция одалживает, а не забирает.* Переписываем неуклюжий `measure` из §15.1.2 — файл `ref1.rs`:

```rust
fn calc_len(s: &String) -> usize {
    s.len()
}                                  // s (ссылка) исчезает; drop НЕ вызывается —
                                   // функция не владела строкой

fn main() {
    let s = String::from("hello");
    let len = calc_len(&s);        // одолжили...
    println!("len of '{s}' is {len}");  // ...и s по-прежнему наша

    let x = 5;
    let r = &x;
    println!("x + 1 = {}", *r + 1);     // разыменование — как *p в C
}
```

```
len of 'hello' is 5
x + 1 = 6
```

Кортежная церемония исчезла: `calc_len` берёт `&String`, вызывающий передаёт `&s` — владение никуда не переезжает, `s` действительна и после вызова. Заметьте симметрию синтаксиса: в сигнатуре `&String`, в вызове `&s` — по паре `&` сразу видно, что функция *одалживает* (сравните с C, где по `f(p)` не понять, заберёт ли функция блок себе — §7.1.2 требовал это документировать словами).

*Пример 2 — попытка изменить одолженное.* Файл `refmut_bad.rs`:

```rust
fn change(s: &String) {
    s.push_str(", world");
}

fn main() {
    let s = String::from("hello");
    change(&s);
    println!("{s}");
}
```

⛔ Не компилируется (проверено):

```
error[E0596]: cannot borrow `*s` as mutable, as it is behind a `&` reference
 --> refmut_bad.rs:2:5
  |
2 |     s.push_str(", world");
  |     ^ `s` is a `&` reference, so it cannot be borrowed as mutable
  |
help: consider changing this to be a mutable reference
  |
1 | fn change(s: &mut String) {
  |               +++
```

Как переменные неизменяемы по умолчанию (E0384, §14.2.1), так и ссылки: через `&T` менять нельзя (R1, гл. 4). Обратите внимание, что `push_str` требует права на изменение — формально это метод, принимающий изменяющую ссылку на строку; вот почему ошибка звучит как «cannot borrow as mutable». Компилятор снова подсказывает следующий шаг — `&mut`.

*Пример 3 — ссылки на разные значения — без ограничений.* Разделяемых ссылок может быть сколько угодно даже на одно значение — а на разные и подавно; они свободно копируются (`&T` — `Copy`-тип: скопировать «пропуск» дёшево и безопасно, R7). Фрагмент из `ref1.rs` выше это уже показал: `&s` существовала одновременно в `main` и в параметре `calc_len`.

**Как это было в C.** Сопоставление с указателями главы 6:

| C (глава 6) | Rust | Комментарий |
|---|---|---|
| `int *p = &x;` | `let r = &x;` | взять адрес — тот же `&` |
| `*p` | `*r` | разыменование — та же `*` |
| `p` может быть `NULL` (§6.1) | `NULL` не существует | «может отсутствовать» — это `Option`, гл. 16 |
| `p` может пережить `x` (§7.2.1) | отклонено компилятором | §15.2.3, E0106/E0597 |
| `p++`, `p + n` (§6.2) | арифметики ссылок нет | «диапазон элементов» — это срез, часть 15.3 |
| `const int *p` (§6.3.1) | `&T` — умолчание | право на запись — отдельный тип `&mut T` |
| выходной параметр `int *out` (§6.1) | `&mut T` | §15.2.2 |

Направление мысли то же, что в паре «`const` по умолчанию»: C делает изменяемым всё и просит помечать неизменяемое; Rust — наоборот.

**Связь с практикой.** Правило выбора сигнатуры, которое пригодится в каждой программе тома II: **функции по умолчанию берут `&T`**; владение (`T`) — только если функция потребляет значение или оставляет его себе; `&mut T` — если должна изменить (§15.2.2). Это не стилистика, а информация в типе: сигнатура `fn calc_len(s: &String) -> usize` *доказывает*, что после вызова строка цела. В C такую гарантию давал лишь комментарий.

**Типичные ошибки и подводные камни.**

- **Прочитать `&` как «адрес в смысле C» — и ждать `NULL` и арифметики.** Ссылка — не «число-адрес», а *право доступа с гарантиями*: не бывает нулевой, не бывает висячей, не умеет `+ 1`. Всё «опасное» от указателей C в Rust живёт в других механизмах: отсутствие — `Option` (гл. 16), диапазоны — срезы (часть 15.3), сырые указатели — `unsafe` (гл. 24).
- **Вернуть из функции ссылку на локальное значение.** Компилятор отклонит — это «висячая ссылка», разбор в §15.2.3.
- **Забыть, что через `&T` нельзя менять.** E0596 из примера 2; права на запись просите явно — `&mut`.
- **Смешивать «ссылка исчезла» и «значение освобождено».** Уход ссылки — это *ничего*: ни drop, ни free. Освобождает только владелец.

**Источники:** R1 (гл. 4 «References and Borrowing»: определение ссылки и гарантия действительности, термин borrowing, неизменяемость ссылок по умолчанию, E0596); R7 (Pointer types: shared reference запрещает прямое изменение, ссылки-`Copy`; арбитр терминологии); R6 (Borrowing a value: функция одалживает вместо передачи владения). Сторона C: §6.1–6.3, §7.1.2 курса.

### §15.2.2. `&mut` и правила заимствования: читатели XOR писатель

**Определения.** **Изменяющая ссылка** (mutable reference) `&mut T` даёт право менять одолженное значение. Права выдаются по строгим правилам (R1, гл. 4):

1. **Если существует `&mut` на значение — никаких других ссылок на него нет.** Ни второй `&mut`, ни даже читающей `&` (R1: «у изменяющей ссылки одно большое ограничение: если она есть, других ссылок на это значение быть не может»).
2. Разделяемых `&` без писателя — сколько угодно.

Коротко: **сколько угодно читателей ЛИБО ровно один писатель** (Comprehensive Rust формулирует дословно так: «one or more shared references, *or* exactly one exclusive reference», R6; XOR — «исключающее или»: либо то, либо другое, но не оба сразу). Формула уже встречалась вам в §13.2.1 — теперь она официально ваша.

Важное смягчение: правила действуют на **пересекающиеся времена жизни** ссылок, а время жизни ссылки тянется от создания до **последнего использования** — не до конца блока (R1, гл. 4). Это правило компилятора называют NLL (non-lexical lifetimes — «нелексические времена жизни»); благодаря ему код «попользовался читателями, потом завёл писателя» компилируется.

**Интуиция.**

```
            владелец: let mut s = String::from("hello");
                              │ выдаёт права
        ┌─────────────────────┴──────────────────────┐
        ▼                                            ▼
   ЧИТАТЕЛИ  &s, &s, &s, ...                 ПИСАТЕЛЬ  &mut s
   сколько угодно одновременно;              ровно один; пока он жив —
   пока жив хоть один — запись               ни читателей, ни второго
   запрещена ВСЕМ (и владельцу)              писателя (и владелец молчит)
```

Почему так строго? Вы уже знаете оба ответа из тома I. Первый — **гонки данных**: «два указателя на одни данные, хотя бы один пишет, синхронизации нет» — три условия из §13.1.4, дословно совпадающие с определением в книге Rust (R1, гл. 4; мы сверяли их с N3220 в §13.2.2). Правило «читатели XOR писатель» делает третье условие невыполнимым *даже в одном потоке* — а в главе 23 оно же станет фундаментом «бесстрашной многопоточности». Второй ответ — **протухшие указатели**: помните §7.1.3 и §9.1.1 — `realloc` переносит буфер, и все старые указатели молча становятся ядовитыми? «Читатель» и «писатель, способный переместить буфер» не должны существовать одновременно — Rust это правило вшил в язык.

**Разобранные примеры.**

*Пример 1 — исправляем E0596.* Файл `refmut1.rs`:

```rust
fn change(s: &mut String) {
    s.push_str(", world");
}

fn main() {
    let mut s = String::from("hello");
    change(&mut s);
    println!("{s}");
}
```

```
hello, world
```

Три правки по сравнению с ⛔-примером §15.2.1 — и все три *видны в коде*: `mut` у переменной (право менять есть только у изменяемой переменной), `&mut String` в сигнатуре (функция просит право на запись), `&mut s` в вызове (вызывающий явно его выдаёт). Сравните с C: `change(&s)` и `change_mut(&s)` выглядели там одинаково — узнать, будет ли функция писать, можно было только по `const` (если автор не поленился, §6.3.1).

*Пример 2 — два писателя.* Файл `twomut.rs`:

```rust
fn main() {
    let mut s = String::from("hello");
    let r1 = &mut s;
    let r2 = &mut s;
    println!("{r1}, {r2}");
}
```

⛔ Не компилируется (проверено):

```
error[E0499]: cannot borrow `s` as mutable more than once at a time
 --> twomut.rs:4:14
  |
3 |     let r1 = &mut s;
  |              ------ first mutable borrow occurs here
4 |     let r2 = &mut s;
  |              ^^^^^^ second mutable borrow occurs here
5 |     println!("{r1}, {r2}");
  |                -- first borrow later used here
```

Прочитайте третью пометку: «first borrow **later used** here». Именно использование `r1` *после* создания `r2` заставляет их времена жизни пересечься. Уберите `r1` из печати — и первый писатель «закончится» на строке своего создания: программа скомпилируется, лишь предупредив о неиспользуемой `r1` (проверено). Последовательные `&mut` — каждый до рождения следующего — вообще легальны: две записи по очереди через две изменяющие ссылки компилируются и работают (проверено; этим займётся задание З4).

*Пример 3 — читатели против писателя.* Файл `mixref.rs`:

```rust
fn main() {
    let mut s = String::from("hello");
    let r1 = &s;
    let r2 = &s;
    let r3 = &mut s;
    println!("{r1}, {r2}, and {r3}");
}
```

⛔ Не компилируется (проверено):

```
error[E0502]: cannot borrow `s` as mutable because it is also borrowed as immutable
 --> mixref.rs:5:14
  |
3 |     let r1 = &s;
  |              -- immutable borrow occurs here
4 |     let r2 = &s;
5 |     let r3 = &mut s;
  |              ^^^^^^ mutable borrow occurs here
6 |     println!("{r1}, {r2}, and {r3}");
  |                -- immutable borrow later used here
```

Логика читателей (R1): взяв `&s`, вы рассчитываете, что значение **не изменится у вас под ногами**. Писатель, влезающий между созданием читателя и его использованием, ломает этот контракт — отказ.

*Пример 4 — NLL: та же комбинация, другой порядок.* Файл `nll.rs`:

```rust
fn main() {
    let mut s = String::from("hello");
    let r1 = &s;
    let r2 = &s;
    println!("{r1} and {r2}");   // последнее использование читателей
    let r3 = &mut s;             // ...их время жизни уже закончилось
    r3.push_str(", world");
    println!("{r3}");
}
```

```
hello and hello
hello, world
```

Тот же набор ссылок, что в примере 3, — но читатели *отработали своё* до появления писателя: времена жизни не пересекаются, конфликт исчез (R1: «область действия ссылки — от введения до последнего использования»).

**Связь с практикой.** Скрытые писатели — главное, что нужно научиться видеть. `s.push_str(...)` выглядит безобидно, но это запись: метод берёт `&mut` на строку *неявно*. Поэтому фрагмент

```rust
let r = &s;          // читатель
s.push_str("b");     // скрытый писатель — E0502!
println!("{r}");
```

не компилируется (проверено; полный разбор — в задании З5). И это не педантизм: `push_str` может **переместить буфер** (рост строки — тот же `realloc`, гл. 17), после чего читатель `r` указывал бы в освобождённую память. E0502 — это use-after-free, пойманный до рождения. ASan из §7.2.2 ловил такое *после* смерти — в запуске, на конкретных данных; borrow checker — до, на всех путях сразу.

**Типичные ошибки и подводные камни.**

- **Читать E0499/E0502 как «в Rust нельзя две ссылки».** Можно — сколько угодно читателей (пример 4), и писателей сколько угодно *по очереди*. Нельзя только пересекать писателя с кем-либо ещё.
- **Не замечать скрытого писателя-метода.** Любой метод, меняющий значение (`push_str`, `clear`, — а позже `push` у `Vec` и т.д.), — это `&mut`. Если рядом жив читатель — отказ.
- **Бороться с NLL переупорядочиванием вслепую.** Правило простое: ссылка «жива» до последнего использования. Читайте пометки `first/immutable borrow later used here` — компилятор всегда показывает, *какое использование* продлевает жизнь ссылки.
- **`mut` в трёх местах — три разных смысла не путать:** `let mut s` — переменную можно менять; `&mut s` — выдаю право на запись; `s: &mut String` — параметр требует право на запись. Забыли любое из трёх — получите внятную ошибку (E0596 подскажет, какое).

**Источники:** R1 (гл. 4: «большое ограничение» изменяющих ссылок, комбинирование `&`/`&mut`, три условия гонки данных, NLL — «до последнего использования», логика «значение не изменится под читателем»); R6 (Borrow checking: «one or more shared references… or exactly one exclusive reference», конфликт по факту существования пересекающихся ссылок); R7 (Pointer types: изменяющая ссылка — единственный путь доступа к значению; арбитр). Сторона C: §6.3.1, §7.1.3, §9.1.1, §13.1.4, §13.2.2 курса.

### §15.2.3. Висячие ссылки: то, что C прощал

**Определения.** **Висячая ссылка** (dangling reference) — ссылка на память, из которой значение уже ушло. В C это целый букет из §7.2.1 и §13.1.2: возврат адреса локальной переменной, указатель на освобождённый блок, указатель в буфер после `realloc`. Rust даёт гарантию (R1, гл. 4): **компилятор не допустит существования ссылки дольше, чем существует значение** — «если у вас есть ссылка на данные, компилятор обеспечит, что данные не выйдут из области видимости раньше ссылки». Второе правило ссылок из резюме R1 звучит коротко: **ссылки всегда действительны** (references must always be valid).

**Интуиция.** У каждого значения есть отрезок жизни — от создания до drop. У каждой ссылки — свой отрезок: от создания до последнего использования. Проверка компилятора геометрическая: **отрезок ссылки обязан лежать внутри отрезка значения**. Всё «висячее» — это ссылка, чей отрезок торчит за край:

```
значение s:   ──[создано ..................... drop]──────────────
ссылка &s:          ──[создана ..... посл. исп.]──          ✓ внутри
ссылка-беглец:                     ──[создана ......... исп.]──  ⛔ пережила drop
```

**Разобранные примеры.**

*Пример 1 — вернуть ссылку на локальное.* Витрина §13.2.1, полный разбор на 1.97 — файл `dangle.rs`:

```rust
fn dangle() -> &String {
    let s = String::from("hello");
    &s
}                                  // здесь s умирает — на что указывала бы ссылка?

fn main() {
    let r = dangle();
    println!("{r}");
}
```

⛔ Не компилируется (проверено):

```
error[E0106]: missing lifetime specifier
 --> dangle.rs:1:16
  |
1 | fn dangle() -> &String {
  |                ^ expected named lifetime parameter
  |
  = help: this function's return type contains a borrowed value, but there is
    no value for it to be borrowed from
help: consider using the `'static` lifetime, but this is uncommon unless
      you're returning a borrowed value from a `const` or a `static`
help: instead, you are more likely to want to return an owned value
  |
1 - fn dangle() -> &String {
1 + fn dangle() -> String {
```

Дословный смысл `help`-строки: «возвращаемый тип содержит одолженное значение, но одалживать его не у кого» — всё, чем функция владеет, умрёт на её скобке. Правильный совет даёт второй `help`: **верните владение** (`String`, без `&`) — файл `nodangle.rs` с таким исправлением собирается и печатает `hello` (проверено). Слова «lifetime specifier» из заголовка ошибки — тема «времён жизни» (lifetimes); их аннотации мы разберём в главе 19, но заметьте: для *отклонения* висячей ссылки они не понадобились.

*Пример 2 — ссылка переживает владельца в блоке.* Не только функции — та же геометрия в любом блоке. Файл `t_outlive.rs`:

```rust
fn main() {
    let r;
    {
        let x = 5;
        r = &x;
    }                              // x умирает, r — снаружи
    println!("{r}");
}
```

⛔ Не компилируется (проверено):

```
error[E0597]: `x` does not live long enough
 --> t_outlive.rs:5:13
  |
4 |         let x = 5;
  |             - binding `x` declared here
5 |         r = &x;
  |             ^^ borrowed value does not live long enough
6 |     }
  |     - `x` dropped here while still borrowed
7 |     println!("{r}");
  |                - borrow later used here
```

Три пометки рисуют ровно нашу схему-отрезки: где родился владелец, где он умер «ещё одолженным» (`dropped here while still borrowed`), и какое использование ссылки торчит за край (`borrow later used here`).

*Пример 3 — а что C?* Тот же сюжет на C (файл `dangle_c.c`, компилятор тома I — MinGW GCC 15.2, `-std=c17 -Wall -Wextra`):

```c
#include <stdio.h>

int *dangle(void) {
    int x = 42;
    return &x;
}

int main(void) {
    int *p = dangle();
    printf("%d\n", *p);
    return 0;
}
```

Проверено:

```
dangle_c.c: In function 'dangle':
dangle_c.c:5:12: warning: function returns address of local variable [-Wreturn-local-addr]
    5 |     return &x;
      |            ^~
```

GCC молодец — **предупредил**… и собрал исполняемый файл (запуск — UB, §13.1.2, поэтому вывод мы не приводим). Причём предупреждение срабатывает лишь на простых случаях: пропустите адрес через переменную-посредника или верните его через выходной параметр — и `-Wreturn-local-addr` замолчит (мы видели это в §7.2.1). Разница с Rust принципиальная: там отказ — не «предупреждение лучших усилий», а следствие правил типов, и обойти его переусложнением кода нельзя.

**Связь с практикой.** Как *писать*, чтобы не воевать с E0106/E0597: у функции два честных варианта ответа — **владение** (`String`, `Vec`, … — «вот твоё новое значение, освобождать тебе») или **ссылка, произведённая из параметра-ссылки** (например, `first_word(s: &str) -> &str` в части 15.3 — вернуть кусок того, что одолжили, легально: владелец остался у вызывающего). Ссылка «в никуда» — на локальное, на временное — не вариант никогда. Если поймали себя на желании «вернуть и значение, и ссылку внутрь него» — вернитесь к этому месту после главы 19.

**Типичные ошибки и подводные камни.**

- **Лечить E0106 по первому `help` (`'static`).** Компилятор сам предупреждает: «это редкий случай». Для локально созданных значений правильный ответ — второй `help`: вернуть владение.
- **Считать E0597 «бюрократией»: «я же знаю, что успею воспользоваться».** Пример 2 — буквально use-after-scope; в C такой код (адрес из вложенного блока) — строка 3 каталога §13.1.2, и там он *иногда работал* — худший вид ошибки.
- **Путать «вернуть ссылку на локальное» (нельзя никогда) и «вернуть ссылку, полученную из параметра» (можно и нужно — часть 15.3).**
- **Ждать таких же гарантий от C-компилятора, «раз он тоже предупреждает».** `-Wreturn-local-addr` — эвристика на простейший паттерн; не сигнал «остальное безопасно».

**Источники:** R1 (гл. 4: гарантия отсутствия висячих ссылок, разбор dangle/E0106, «верните владение», резюме двух правил ссылок); R6 (Borrow checking: правило «ссылка не может пережить заимствованное значение»); R7 (арбитр по семантике времён жизни ссылок); C17 (GCC Manual — семейство предупреждений `-W...`, `-Wreturn-local-addr`). Сторона C: §7.2.1, §13.1.2 курса.

## Часть 15.3. Срезы: указатель + длина, но по правилам

### §15.3.1. Срезы массивов: `&[T]`

**Определения.** **Срез** (slice) — это «окно» в непрерывную последовательность элементов: ссылка на её участок. Срез — **разновидность ссылки**: он не владеет данными, а заимствует их у владельца (R1, гл. 4; R7: разделяемый срез `&[T]` «не владеет данными, на которые указывает, — он их одалживает»). На срезы распространяются все правила части 15.2 — «читатели XOR писатель», запрет переживать владельца.

Синтаксис — взятие ссылки с диапазоном (диапазоны — §14.2.4):

```rust
let a = [10, 20, 30, 40, 50];
let s = &a[1..3];        // элементы с индекса 1 до 3 (не включая) — [20, 30]
```

Сокращения: `&a[..2]` — от начала; `&a[3..]` — до конца; `&a[..]` — весь массив (R1). Тип среза из массива `[i32; 5]` — **`&[i32]`**: длины в типе больше нет. Внутри срез устроен как пара **(указатель на первый элемент, число элементов)** (R1, гл. 4; R7). Есть и изменяющая версия — `&mut [T]`: одалживает участок с правом записи (R7).

Формально `[T]` — «тип с размером, неизвестным при компиляции» (dynamically sized type), поэтому в программах он всегда ходит за ссылкой: `&[T]`, `&mut [T]` (R7; другие формы — гл. 22). И принципиальное свойство: **каждый доступ к срезу проверяется по границам** — в безопасном Rust у среза «выйти за край» можно только в панику, не в чужую память (R7).

**Интуиция.** Пара «указатель + длина» — вы носили её в руках весь том I, просто по отдельности:

```
C, главы 5–6: два параметра,        Rust: одно значение,
связанные только дисциплиной        связанное типом

int sum(const int *a, size_t n);    fn sum(xs: &[i32]) -> i32
        ▲            ▲
        │            └─ соврать можно: передать n=10 при 5 элементах —
        │               и «сумма» уйдёт в чужую память (§5.1.2, §13.1.2)
        └─ а можно передать NULL

      стек                    массив a (у владельца)
  s ┌─────┬───┐             ┌────┬────┬────┬────┬────┐
    │ ptr │ ●─┼──────┐      │ 10 │ 20 │ 30 │ 40 │ 50 │
    │ len │ 2 │      │      └────┴────┴────┴────┴────┘
    └─────┴───┘      └───────────▶ ▲
   &a[1..3]: указатель на a[1], длина 2
```

Срез — та же пара, но: собрана в одно значение; длина всегда честная (её вычислил компилятор из диапазона); указатель не бывает нулевым; и весь механизм заимствован — владелец жив, пока жив срез.

**Разобранные примеры.**

*Пример 1 — вырезаем окна.* Файл `slice1.rs`:

```rust
fn main() {
    let a = [10, 20, 30, 40, 50];
    let s = &a[1..3];
    println!("len = {}", s.len());
    println!("first = {}, last = {}", s[0], s[s.len() - 1]);

    let head = &a[..2];
    let tail = &a[3..];
    let all = &a[..];
    println!("head: {} {}", head[0], head[1]);
    println!("tail: {} {}", tail[0], tail[1]);
    println!("all: len = {}", all.len());
}
```

```
len = 2
first = 20, last = 30
head: 10 20
tail: 40 50
all: len = 5
```

Индексация внутри среза — от нуля *по срезу*: `s[0]` — это `a[1]`. У среза есть `.len()` — длина больше не живёт в отдельной переменной, которую можно забыть обновить.

*Пример 2 — одна функция для всех длин.* Файл `slicefn.rs`:

```rust
fn sum(xs: &[i32]) -> i32 {
    let mut total = 0;
    for i in 0..xs.len() {
        total += xs[i];
    }
    total
}

fn main() {
    let a = [1, 2, 3, 4, 5];
    let b = [10, 20];
    println!("sum a   = {}", sum(&a[..]));
    println!("sum b   = {}", sum(&b[..]));
    println!("sum mid = {}", sum(&a[1..4]));
    println!("sum &a  = {}", sum(&a));
}
```

```
sum a   = 15
sum b   = 30
sum mid = 9
sum &a  = 15
```

Одна сигнатура обслуживает массив из 5 элементов, из 2, и середину массива — в C это и была главная работа пары `(a, n)`. Обратите внимание на последний вызов: `sum(&a)` — компилятор сам превращает ссылку на массив в срез (проверено); писать `&a[..]` каждый раз не обязательно. А вот жёсткая сигнатура `&[i32; 5]` принимала бы **только** массивы ровно из 5 элементов — совсем не то, что «массивный» параметр C, где размер в `int a[5]` был фикцией (§6.2.1: gcc предупреждал о `sizeof` такого параметра). В Rust размер в типе — настоящий контракт; гибкость же даёт срез.

*Пример 3 — записать через срез.* Файл `mutslice.rs`:

```rust
fn fill(xs: &mut [i32], val: i32) {
    for i in 0..xs.len() {
        xs[i] = val;
    }
}

fn main() {
    let mut a = [1, 2, 3, 4, 5];
    fill(&mut a[1..4], 0);
    println!("{} {} {} {} {}", a[0], a[1], a[2], a[3], a[4]);
}
```

```
1 0 0 0 5
```

`&mut [i32]` — «окно с правом записи»: функция обнулила середину, не зная (и не имея возможности узнать) ничего про края. Пока жив такой срез, других ссылок на массив нет — правило §15.2.2 в полный рост.

*Пример 4 — граница проверяется всегда.* Файл `sliceoob.rs`:

```rust
fn main() {
    let a = [10, 20, 30, 40, 50];
    let n = std::env::args().count() + 6;   // 7 при запуске без аргументов
    let s = &a[2..n];
    println!("{}", s.len());
}
```

Компилируется, а при выполнении (проверено):

```
thread 'main' (10680) panicked at sliceoob.rs:4:15:
range end index 7 out of range for slice of length 5
```

Паника с точным диагнозом — как у индексации массива в §13.2.2 (номер в скобках после `thread 'main'` — служебный, §14.2.2). Важная деталь, отличающая срезы от индексации из §14.2.2: там константный `a[7]` **не компилировался** (`deny(unconditional_panic)`), а вот срез с константными границами `&a[2..7]` компилируется молча и паникует в рантайме (проверено) — статический анализ диапазонов слабее. Урок: срез снял с вас арифметику границ, но не отменил вопрос «а есть ли там столько элементов?».

**Как это было в C.** Идиома «функция получает кусок массива»:

| | C (главы 5–6) | Rust |
|---|---|---|
| Сигнатура | `int sum(const int *a, size_t n)` | `fn sum(xs: &[i32]) -> i32` |
| Право записи | `int *a` без `const` (легко забыть) | отдельный тип `&mut [i32]` |
| «Середина массива» | `sum(a + 1, 3)` — арифметика руками (§6.2) | `sum(&a[1..4])` — диапазон проверяется |
| Соврать про длину | легко, компилятор молчит (§5.1.2) | длину вычисляет компилятор |
| Выход за край | UB: мусор/порча памяти (§13.1.2, стр. 1) | паника с диагнозом — всегда |
| Пустой кусок | `n == 0`, указатель «куда-нибудь» | `&a[2..2]` — честный срез длины 0 |

**Связь с практикой.** Правило сигнатур из §15.2.1 дополняется: если функции нужен «ряд элементов только посмотреть» — берите `&[T]`; менять — `&mut [T]`; сам массив фиксированного размера (`&[T; N]`) — почти никогда (разве что размер — часть контракта, как RGB-тройка). Это буквально C-шный хороший тон «передавай `(ptr, len)` и `const`-корректность», ставший системой типов. В главе 17 те же срезы станут окнами в `Vec<T>` — код, написанный против `&[T]`, примет и массив, и вектор.

**Типичные ошибки и подводные камни.**

- **Ждать, что `..` включает правую границу.** `&a[1..3]` — два элемента (индексы 1, 2). Включительно — `..=` (§14.2.4). Off-by-one не умер — он просто паникует вместо UB.
- **Считать индекс среза индексом массива.** `s = &a[1..3]` ⇒ `s[0]` — это `a[1]`. Смещение уже «вшито» в указатель среза.
- **Думать, что константные границы среза проверит компилятор.** Пример 4: не проверит (в отличие от константной индексации) — эта проверка живёт в рантайме.
- **Пытаться «нарастить» срез или сохранить его «на потом».** Срез — заимствование: он не может пережить массив и держит владельца под правилами части 15.2 (пока жив `&mut`-срез — к массиву никому нельзя).

**Источники:** R1 (гл. 4 «The Slice Type»: определение, синтаксис диапазонов, устройство «указатель + длина», `&mut`-срезы — в примерах гл. 8, базово); R7 (Slice types: `[T]` — DST, `&[T]` одалживает, `&mut [T]`, всякий доступ проверяется по границам; арбитр); R6 (Slices: «view into a larger collection», срез заимствует у исходного типа). Сторона C: §5.1.2, §6.2, §13.1.2 курса.

### §15.3.2. Строковые срезы: `&str`

**Определения.** **Строковый срез** `&str` — срез по байтам строки: пара «указатель + длина в байтах», указывающая в буфер, которым владеет кто-то другой (R1, гл. 4). Три факта, замыкающие знакомые концы:

- **Строковый литерал — это `&str`** (R1): `let s = "hello";` — срез, указывающий на байты, вшитые прямо в исполняемый файл. Отсюда и его неизменяемость — `&str` есть разделяемая ссылка. (Помните загадку `spaces.len()` из §14.2.1? Тип `&str` там уже мелькал в E0308.)
- Срез берётся диапазоном по `String`: `&s[..5]`, — с одной оговоркой: границы обязаны попадать на границы символов UTF-8, иначе паника; для ASCII-текста «байт = символ» и вопрос не встаёт, подробности — глава 17 (R1).
- Метод `s.as_bytes()` даёт `&[u8]` — те же байты строкой байтового среза (байтовые литералы вида `b' '` — §14.2.2, З2); это наш инструмент посимвольного обхода до главы 17.

**Интуиция.** Классическая задача (R1, гл. 4): «вернуть первое слово строки». Том I решал бы её как §5.2 — вернуть *индекс* конца слова:

```
String s: "hello world"     функция вернула: 5
                             ▲
              а дальше — молитва, что s не изменится,
              пока индекс 5 у вас на руках (§5.2: индекс и
              строка связаны только у вас в голове)
```

Индекс — «полссылки»: число, осмысленное только вместе со строкой, о чём компилятор не знает. Срез — ссылка целиком: он *держит* строку правилами заимствования, пока вы им пользуетесь.

**Разобранные примеры.**

*Пример 1 — версия с индексом: компилируется и врёт.* Файл `fw_index.rs`:

```rust
fn first_word_end(s: &String) -> usize {
    let bytes = s.as_bytes();
    for i in 0..bytes.len() {
        if bytes[i] == b' ' {
            return i;
        }
    }
    s.len()
}

fn main() {
    let mut s = String::from("hello world");
    let word = first_word_end(&s);
    s.clear();                     // строка опустела...
    println!("word = {word}, s = '{s}', s.len() = {}", s.len());
}
```

Компилируется и выводит (проверено):

```
word = 5, s = '', s.len() = 0
```

`word == 5` при строке длины 0 — «первое слово кончается на индексе 5» пустой строки. Ошибки памяти нет, но данные рассинхронизированы: любое использование `word` для среза/индексации теперь паника или чепуха. Книга Rust называет такое состояние «index out of sync with data» (R1) — это логическая бомба, знакомая по C.

*Пример 2 — версия со срезом: рассинхрон не компилируется.* Файл `fw_slice_bad.rs`:

```rust
fn first_word(s: &str) -> &str {
    let bytes = s.as_bytes();
    for i in 0..bytes.len() {
        if bytes[i] == b' ' {
            return &s[..i];
        }
    }
    &s[..]
}

fn main() {
    let mut s = String::from("hello world");
    let word = first_word(&s);
    s.clear();
    println!("first word: {word}");
}
```

⛔ Не компилируется (проверено):

```
error[E0502]: cannot borrow `s` as mutable because it is also borrowed as immutable
  --> fw_slice_bad.rs:14:5
   |
13 |     let word = first_word(&s);
   |                           -- immutable borrow occurs here
14 |     s.clear();
   |     ^^^^^^^^^ mutable borrow occurs here
15 |     println!("first word: {word}");
   |                            ---- immutable borrow later used here
```

Всё сошлось воедино: `word` — срез, то есть *читатель* строки; `clear()` — скрытый *писатель* (§15.2.2); печать `word` продлевает жизнь читателя за писателя — E0502. Заметьте: функция возвращает ссылку — и это законно, в отличие от `dangle` из §15.2.3: срез **произведён из параметра-ссылки**, владелец (строка в `main`) жив. Книга Rust подводит итог именно здесь: «Rust не просто сделал API удобнее — он устранил целый класс ошибок на этапе компиляции» (R1, гл. 4).

Рабочая версия (`fw_ok.rs`, без `clear`) печатает `first word: hello` (проверено); на строке без пробелов вернётся вся строка, на пустой — пустой срез (проверено: `'hello'`, `''`).

*Пример 3 — `&str` как тип параметра: одна функция для всего строкового.* Обратите внимание на сигнатуру: мы написали `first_word(s: &str)`, а вызывали `first_word(&s)`, где `s: String`. Работает и так (проверено, файл `fw_ok.rs`):

```rust
fn main() {
    let s = String::from("hello world");
    println!("first word: {}", first_word(&s));        // из &String

    let lit = "fast brown fox";
    println!("first word: {}", first_word(lit));       // литерал — уже &str
    println!("first word of slice: {}", first_word(&lit[5..])); // срез среза
}
```

```
first word: hello
first word: fast
first word of slice: brown
```

`&String` при вызове сам превращается в `&str` — это механика **deref coercion**, устройство которой мы разберём в главе 22 (R1 отсылает туда же); пока запомните правило хорошего тона (R1): **строковые параметры функций объявляйте как `&str`** — такая функция примет и `String` (по ссылке), и литерал, и чужой срез, без потерь и копий. Сигнатура `&String` не даёт ничего, кроме сужения круга друзей.

**Связь с практикой.** Итоговая картина строковых типов на сегодня: `String` — владеющая строка в куче (наш `char *` + честный `free`); `&str` — невладеющее окно в чью-то строку (наши `const char *` + длина, §5.2 — только теперь их двое не разлучить). Пишете функцию — почти всегда хотите `&str` в параметрах и либо `String` (новое значение), либо `&str`-из-параметра (кусок входа) в результате. Этой пары типов хватит до главы 17, где к ним присоединятся методы и UTF-8 во всех подробностях.

**Типичные ошибки и подводные камни.**

- **`&String` в сигнатурах.** Компилируется, но беднее `&str` (пример 3). Привыкайте сразу к идиоме.
- **Пытаться менять через `&str`.** У `&str` нет `push_str` — «no method named `push_str` found for reference `&str`» (E0599, проверено). Наращивают владеющую `String`; `&str` — окно, а не буфер.
- **Хранить индексы вместо срезов** «по C-привычке» (пример 1). Индекс — договорённость, срез — контракт. Если нужен именно индекс (например, чтобы отдать наружу число) — помните, что компилятор больше не страхует его связь со строкой.
- **Резать не-ASCII строку по произвольному байту.** `&s[..5]` по русской строке может попасть в середину символа — паника. До главы 17 держите примеры на ASCII (наши — такие).

**Источники:** R1 (гл. 4 «The Slice Type»: задача first_word, рассинхрон индекса, строковые срезы, литералы — `&str`, `&str`-параметры и deref coercion (отсылка), «устранил класс ошибок»); R7 (Slice types: `str` — срезовый тип; арбитр); R9 (`String::as_bytes`, `String::clear` — сигнатуры, справочно). ## Резюме главы

- **Владение**: у каждого значения ровно один владелец; на закрывающей скобке владельца компилятор вставляет `drop` — ваш `free` из главы 7, который нельзя ни забыть, ни вызвать дважды (порядок — обратный объявлению, R7). Переприсваивание освобождает старое значение немедленно; `drop(x)` — досрочное освобождение через передачу владения.
- **Перемещение**: присваивание/передача/возврат значений с ресурсом (пока это `String`) копируют тройку `ptr/len/cap` и **аннулируют источник** — использование старого имени ловит E0382 с полным протоколом: где уехало, где обратились, как починить. Двойное освобождение невыразимо.
- **Clone и Copy**: настоящая копия буфера — только явный `.clone()` (дорогое — видно глазами); «плоские» типы (числа, `bool`, `char`, их кортежи и массивы) — `Copy`: присваивание копирует, источник жив. Copy и Drop несовместимы; один не-Copy элемент лишает Copy весь агрегат.
- **Заимствование**: `&T` — одолжить почитать (владение на месте, drop не ваш), `&mut T` — одолжить с правом записи. Закон: **сколько угодно читателей XOR один писатель** (пересечение времён жизни; жизнь ссылки — до последнего использования, NLL). E0596 — писали через `&`; E0499 — два писателя; E0502 — писатель при живых читателях (главный подозреваемый — метод-«скрытый писатель» вроде `push_str`/`clear`).
- **Висячих ссылок нет**: ссылка не может пережить владельца — ни из функции (E0106: «одалживать не у кого» — верните владение), ни в блоке (E0597, схема-отрезки). GCC на тот же сюжет *предупреждал и собирал* — Rust отказывает, и это разница между эвристикой и правилом типов.
- **Срезы**: `&[T]`/`&mut [T]` и `&str` — пара «указатель + длина» из C (главы 5–6), собранная в одно значение под правилами заимствования: длина честная, границы проверяются всегда (константные границы среза — тоже в рантайме!), рассинхрон «индекс отдельно, данные отдельно» не компилируется. Литерал — `&str`; строковые параметры объявляйте `&str` — примут и `String`, и литерал (механика — гл. 22).
- **Правило сигнатур тома II**: по умолчанию — `&T`/`&str`/`&[T]`; `&mut` — если функция меняет; владение — если потребляет или отдаёт новое. Сигнатура стала документацией, которую проверяет компилятор.

**Дальше.** Владение и заимствование — это грамматика Rust; теперь нужны *слова*. Глава 16 даёт моделирование данных: структуры и перечисления с данными, методы, `Option<T>` вместо `NULL` и `match`, который не даст забыть ветку `None`, — всё то, что в C собиралось из `struct` + функций + дисциплины (глава 8), а здесь станет системой типов. Правила этой главы заработают на ваших собственных типах.
