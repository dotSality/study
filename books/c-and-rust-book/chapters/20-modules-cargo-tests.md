## Часть 20.1. Модули, crates и Cargo

### §20.1.1. Пакет и crate: из чего сделан проект

**Определения.** До сих пор слово «crate» мелькало в ошибках компилятора и в названиях (`crate root`), а строгих определений мы не давали. Даём (R1, гл. 7.1; R7):

- **Crate** — наименьшая единица кода, которую компилятор Rust рассматривает за один раз: дерево модулей, компилируемое вместе. Даже одиночный файл, который мы отдавали `rustc` все шесть глав, — это crate.
- **Crate бывает двух видов.** **Бинарный** (binary crate) компилируется в исполняемый файл и обязан иметь функцию `main` — все наши программы до сих пор. **Библиотечный** (library crate) не имеет `main` и ни во что исполняемое не компилируется: это функциональность, предназначенная для использования другими crate — как `libds.a` из §11.2.3, только с проверками языка.
- **Корень crate** (crate root) — исходный файл, с которого компилятор начинает сборку: он образует корневой модуль дерева. По конвенции Cargo: `src/main.rs` — корень бинарного crate, `src/lib.rs` — корень библиотечного.
- **Пакет** (package) — один или несколько crate плюс файл `Cargo.toml`, описывающий, как их собирать. Правила состава (R1, гл. 7.1): в пакете **не больше одной библиотеки**, бинарных crate — сколько угодно (дополнительные кладут в `src/bin/`, по файлу на бинарник), и хотя бы один crate быть обязан.

**Интуиция.** Три уровня — как матрёшка:

```
   пакет textstat  ←  Cargo.toml (имя, версия, зависимости)
   │
   ├── библиотечный crate  ←  корень src/lib.rs
   │        crate           дерево модулей —
   │         ├── stats      «оглавление» библиотеки:
   │         └── clean      crate::stats::word_count
   │
   └── бинарный crate  ←  корень src/main.rs
            маленький main, который зовёт библиотеку
            по её имени: textstat::...
```

В C «проект» был соглашением: набор `.c`-файлов, которые вы не забыли перечислить в Makefile (§11.2.3). В Rust проект — это объект, о котором знает инструмент: у него есть имя, версия и явный состав.

**Разобранные примеры.**

*Пример 1 — бинарный пакет.* Команду `cargo new` мы выполняли ещё в §14.1.2; теперь прочитаем результат глазами этой главы:

```
$ cargo new hello_pkg
    Creating binary (application) `hello_pkg` package
note: see more `Cargo.toml` keys and their definitions at https://doc.rust-lang.org/cargo/reference/manifest.html
```

Cargo сам говорит, что создал: **пакет** с бинарным crate. Внутри:

```
hello_pkg
├── .gitignore        ← содержит одну строку: /target
├── Cargo.toml        ← паспорт пакета
└── src
    └── main.rs       ← корень бинарного crate
```

```toml
[package]
name = "hello_pkg"
version = "0.1.0"
edition = "2024"

[dependencies]
```

Помимо файлов, `cargo new` инициализирует git-репозиторий (если пакет не лежит внутри существующего) и записывает в `.gitignore` каталог `target/` — собранное в систему контроля версий не попадает, ровно как наша договорённость «бинарники в репозиторий не коммитим» из главы 11. Секция `[dependencies]` пока пуста — она понадобится в §20.1.4.

*Пример 2 — библиотечный пакет.* Флаг `--lib` меняет вид создаваемого crate:

```
$ cargo new --lib textstat
    Creating library `textstat` package
```

Вместо `src/main.rs` появляется `src/lib.rs`, и не пустой:

```rust
pub fn add(left: u64, right: u64) -> u64 {
    left + right
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_works() {
        let result = add(2, 2);
        assert_eq!(result, 4);
    }
}
```

Cargo дарит заготовку сразу с **тестом** — намёк, что библиотека без тестов считается недоделанной. В шаблоне видна почти вся первая часть главы: ключевое слово `mod`, незнакомое `pub`, путь `super::*` — и вся вторая: атрибуты `#[cfg(test)]` и `#[test]`, макрос `assert_eq!`. К концу главы вы будете читать этот файл как открытую книгу; тест из шаблона мы запустим в §20.2.1.

*Пример 3 — пакет с двумя crate.* Если в `src/` лежат **и** `main.rs`, **и** `lib.rs`, пакет содержит два crate с одним именем: библиотеку и бинарник (R1, гл. 7.1). Это стандартная форма утилиты: вся логика — в библиотеке (её можно тестировать и переиспользовать), `main.rs` — тонкая обёртка. Именно так мы соберём проект `textstat` в §20.1.4, и именно этого потребует §20.2.2: интеграционные тесты умеют проверять только библиотеку.

**Связь с практикой.** Всё, что вы знали о Cargo с главы 14, работает на уровне пакета: `cargo build` собирает все его crate, `cargo run` запускает бинарный, `cargo check` быстро проверяет без генерации кода, `cargo clean` сносит `target/`. Появление библиотеки ничего в этих командах не меняет.

**Типичные ошибки и подводные камни.**

- Путаница «crate или пакет». Разговорно «crate» часто говорят про пакет целиком (и сайт называется crates.io). Технически: пакет — единица *распространения и сборки*, crate — единица *компиляции*. Ошибки компилятора всегда говорят о crate.
- Попытка положить в пакет вторую библиотеку. Нельзя: не больше одной (R1, гл. 7.1). Несколько библиотек — это несколько пакетов; как их объединить, покажет workspace (§20.1.4).
- Ожидание, что `main.rs` «видит» внутренности `lib.rs`. Не видит: это **разные crate**. Бинарник пользуется библиотекой по имени пакета (`textstat::...`) и только через публичный интерфейс — как совершенно посторонний код. Почему это удача, а не помеха — §20.2.2.

**Источники:** R1 (гл. 7.0–7.1), R10 (guide: project layout), R7 (crates and source files).

### §20.1.2. Модули и видимость: приватность по умолчанию

**Определения.** **Модуль** (module) — контейнер для элементов: функций, структур, перечислений, констант, других модулей (R7 — арбитр: items/modules). Модуль вводится ключевым словом `mod` и образует узел в **дереве модулей** crate, растущем из корневого модуля (корня crate):

```rust
mod stats {            // модуль stats — потомок корня
    pub mod words {    // модуль words — потомок stats
        pub fn count(text: &str) -> usize { /* ... */ 0 }
    }
}
```

Главный закон дерева — **видимость** (R7 — арбитр: visibility and privacy):

1. **По умолчанию всё приватно.** Приватный элемент виден в модуле, где объявлен, и во всех его *потомках* — и больше нигде. Исключений два, оба знакомые: варианты `pub enum` публичны сами (§16.2.1 — мы писали `Shape::Circle` снаружи без всяких `pub` на вариантах), и элементы `pub trait` публичны сами (методы `Summary` из §19.2.1 не требовали отдельного `pub`).
2. **`pub` открывает элемент наружу** — но доступ по пути требует «проходимости» **каждого звена**: элемент доступен из чужого модуля, только если доступны все его модули-предки. `pub fn` в приватном модуле снаружи не видна.
3. Между «только моё» и «всем» есть промежуточные ступени; самая ходовая — **`pub(crate)`**: элемент виден всему текущему crate, но не внешним пользователям (R7; R6). Остальные (`pub(super)`, `pub(in путь)`) оставим справочнику.

**Интуиция.** Видимость смотрит *вверх и внутрь*, но не *вниз*:

```
crate (корень)
 └── mod stats                     ← сам приватный: но корень — его родитель,
      ├── pub fn answer()             родителю потомки видны? НЕТ — наоборот:
      └── mod internal                потомок видит всё у предков,
           └── pub fn magic()         предок у потомка — только pub

 доступ из main (корень) к:
   stats::answer()            ok: stats виден в своём родителе, answer — pub
   stats::internal::magic()   ⛔ E0603: звено internal приватно,
                                 и pub на magic не спасает
```

Сравните с C: там инструмент сокрытия один — `static` на уровне файла (§11.2.2), «спрятать функцию внутри половины файла» нельзя. Здесь границы видимости расставляются деревом модулей с любой вложенностью, и проверяет их компилятор, а не дисциплина команды.

**Разобранные примеры.**

*Пример 1 — приватный помощник.* Модуль наводит порядок: наружу — опрятная `normalize`, внутри — подробность реализации `is_junk`. Файл `mods_private_fn.rs` (⛔ вторая строка `main`):

```rust
mod clean {
    // Приватный помощник: виден только внутри clean (и его потомков).
    fn is_junk(c: char) -> bool {
        c == '.' || c == ',' || c == '!' || c == '?'
    }

    pub fn normalize(text: &str) -> String {
        let mut out = String::new();
        for c in text.chars() {
            if !is_junk(c) {
                out.push(c);
            }
        }
        out
    }
}

fn main() {
    println!("{}", clean::normalize("тише, мыши!")); // ок: pub-функция
    println!("{}", clean::is_junk('!')); // ⛔ приватная функция
}
```

```
error[E0603]: function `is_junk` is private
  --> mods_private_fn.rs:20:27
   |
20 |     println!("{}", clean::is_junk('!')); // ⛔ приватная функция
   |                           ^^^^^^^ private function
   |
note: the function `is_junk` is defined here
```

Уберите вторую строку `main` — программа печатает `тише мыши`. Внутри `clean` вызов `is_junk(c)` законен (правило 1: свой модуль); снаружи — E0603. Заметьте: `normalize` зовёт помощника без всяких путей — элементы своего модуля доступны по короткому имени.

*Пример 2 — приватный модуль.* Теперь приватно само звено пути. Файл `mods_private_mod.rs` (⛔ вторая строка `main`):

```rust
mod stats {
    mod internal {
        pub fn magic() -> u32 {
            42
        }
    }

    pub fn answer() -> u32 {
        internal::magic() // внутри stats модуль internal виден
    }
}

fn main() {
    println!("{}", stats::answer()); // ок
    println!("{}", stats::internal::magic()); // ⛔ модуль internal приватный
}
```

```
error[E0603]: module `internal` is private
  --> mods_private_mod.rs:15:27
   |
15 |     println!("{}", stats::internal::magic()); // ⛔ модуль internal приватный
   |                           ^^^^^^^^  ----- function `magic` is not publicly re-exported
   |                           |
   |                           private module
```

`pub fn magic` не помог: правило 2 требует `pub` на каждом звене. Компилятор вдобавок роняет подсказку «not publicly re-exported» — что такое реэкспорт, увидим в §20.1.3. Без второй строки `main` программа печатает `42`: `answer` — «официальное окно» в приватную кухню.

*Пример 3 — поля структур против вариантов enum.* У `pub struct` публичность полей настраивается **по одному**; у `pub enum` варианты публичны все сразу (исключение из правила 1). Файл `fields_private.rs` (⛔ третья строка `main`):

```rust
mod report {
    pub struct Summary {
        pub words: usize, // публичное поле
        longest: usize,   // приватное поле (по умолчанию)
    }

    pub fn build(words: usize, longest: usize) -> Summary {
        Summary { words, longest }
    }
}

fn main() {
    let s = report::build(4, 7);
    println!("слов: {}", s.words); // ок: поле pub
    println!("длиннейшее: {}", s.longest); // ⛔ поле приватное
}
```

```
error[E0616]: field `longest` of struct `Summary` is private
  --> fields_private.rs:15:34
   |
15 |     println!("длиннейшее: {}", s.longest); // ⛔ поле приватное
   |                                  ^^^^^^^ private field
```

Ошибка на этот раз называется E0616 — компилятор различает «приватный элемент модуля» и «приватное поле». Обратите внимание: раз поля приватны, снаружи модуля литерал `Summary { ... }` не написать — потому в `report` и лежит функция-конструктор `build` (ассоциированные функции-конструкторы вроде `Rect::square` из §16.1.2 — это оно же). А с перечислением такой заботы не нужно, файл `enum_pub.rs`:

```rust
mod report {
    // У pub enum все варианты публичны — pub на каждом не нужен.
    pub enum Verdict {
        Short,
        Long,
    }

    pub fn judge(words: usize) -> Verdict {
        if words < 100 { Verdict::Short } else { Verdict::Long }
    }
}

fn main() {
    match report::judge(4) {
        report::Verdict::Short => println!("короткий текст"),
        report::Verdict::Long => println!("длинный текст"),
    }
}
```

```
короткий текст
```

Логика исключения по R1 (гл. 7.3): enum без доступных вариантов бесполезен — по нему нельзя сделать `match`; а структура с приватными полями как раз полезна: это инкапсуляция.

**Связь с практикой.** Приватные поля — не формальность, а механизм **инвариантов**. Вспомните `Vec` (§17.1.1): `len` и ёмкость связаны обещанием «len ≤ cap, первые len ячеек инициализированы». Если бы `len` было публичным полем, любой код мог бы написать `v.len = 100` — и чтение `v[50]` полезло бы в неинициализированную память. Приватность полей плюс публичные методы = все изменения проходят через код, который поддерживает инвариант. В C мы могли только договориться «не трогайте поля `Vec` напрямую» (§9.1.1) — компилятор договор не проверял.

**Типичные ошибки и подводные камни.**

- `pub mod` не делает публичным содержимое: он открывает только само звено пути. Публичность каждого элемента объявляется отдельно (R1, гл. 7.3).
- Обратная ошибка: «раз потомки видят предков, то и предок видит потомков». Нет: у потомка предок видит только `pub` (пример 2).
- `pub` «на всякий случай». Каждый `pub` — обещание пользователям: убрать его потом — сломать их код. Правило простое: начинайте со всего приватного и открывайте по требованию компилятора — он честно скажет, где не хватает (E0603).
- Забытая разница E0603/E0616: первый — про элементы и звенья пути, второй — про поля. Чинятся по-разному: E0616 чаще лечится не `pub`-полем, а методом доступа.

**Источники:** R1 (гл. 7.2–7.3), R7 (items/modules; visibility and privacy — арбитр), R6 (modules: visibility, encapsulation).

### §20.1.3. Пути и use: как называть и как сокращать

**Определения.** **Путь** (path) — имя элемента в дереве модулей; сегменты разделяются `::` (R1, гл. 7.3). Путь бывает:

- **абсолютным** — от корня crate: `crate::stats::words::count` (для элементов чужих crate — от имени crate: `std::collections::HashMap`);
- **относительным** — от текущего модуля: `stats::words::count`, либо явно от себя (`self::...`), либо от родителя — **`super::`**, аналог `..` в путях файловой системы.

**`use`** создаёт **ярлык** (shortcut) на путь, действующий **только в той области видимости, где объявлен** (R1, гл. 7.4). `use crate::stats::words;` — и дальше в этой области пишется просто `words::count(...)`. Дополнения: `as` даёт ярлыку другое имя; `pub use` делает ярлык публичным — **реэкспорт** (re-export): элемент становится доступен снаружи по новому, короткому пути; вложенные списки `use std::{cmp, io};` сокращают пачку ярлыков; глоб `use путь::*` тащит всё публичное (R1 предостерегает: становится неясно, откуда имя, — но узаконивает глоб в тестах, см. §20.2.1).

**Интуиция.** Дерево модулей — файловая система в миниатюре:

```
   путь              аналог в файловой системе
   crate::a::b::f    /a/b/f        (абсолютный, от корня)
   a::b::f           a/b/f         (относительный, от текущей папки)
   super::g          ../g          (от родителя)
   use crate::a::b;  «ярлык на папку b на рабочем столе»
```

Абсолютный путь переживает перенос *вызывающего кода* в другой модуль; относительный — совместный перенос вызывающего и вызываемого. R1 (гл. 7.3) советует по умолчанию абсолютные: код чаще ездит по дереву порознь.

**Разобранные примеры.**

*Пример 1 — обе формы пути.* Файл `mods_tree.rs`:

```rust
// Дерево модулей в одном файле: crate (корень) → stats → words.
mod stats {
    pub mod words {
        // Слово — максимальная серия непробельных символов.
        pub fn count(text: &str) -> usize {
            let mut n = 0;
            let mut in_word = false;
            for c in text.chars() {
                if c.is_whitespace() {
                    in_word = false;
                } else if !in_word {
                    in_word = true;
                    n += 1;
                }
            }
            n
        }
    }
}

fn main() {
    let text = "во поле берёза стояла";
    let abs = crate::stats::words::count(text); // абсолютный путь от корня
    let rel = stats::words::count(text); // относительный путь
    println!("слов (абсолютный путь): {abs}");
    println!("слов (относительный путь): {rel}");
}
```

```
слов (абсолютный путь): 4
слов (относительный путь): 4
```

Счётчик слов — «конечный автомат в миниатюре» из двух состояний (`in_word`), как проверка скобок в §9.2.1; методы `chars` и `is_whitespace` знакомы по главам 17–18. Обе формы пути ведут к одной функции.

*Пример 2 — `super::`.* Потомок дотягивается до элемента родителя. Файл `super_demo.rs`:

```rust
// Элемент корня crate — «родитель» для модуля stats.
fn separator() -> char {
    ' '
}

mod stats {
    pub fn describe() {
        // super:: — путь от родительского модуля (как .. в путях файлов)
        let sep = super::separator();
        println!("разделитель слов: {sep:?}");
    }
}

fn main() {
    stats::describe();
}
```

```
разделитель слов: ' '
```

`separator` приватна — но §20.1.2 (правило 1) разрешает: потомок видит приватное у предков. `super::` здесь нельзя заменить на `crate::` бездумно: сработает так же, но перестанет переживать перенос `stats` вглубь дерева.

*Пример 3 — идиомы `use`.* Как выбирать, до какого места сокращать путь (R1, гл. 7.4). Файл `use_demo.rs` (тело `count` — как в примере 1):

```rust
mod stats {
    pub mod words {
        pub fn count(text: &str) -> usize {
            let mut n = 0;
            let mut in_word = false;
            for c in text.chars() {
                if c.is_whitespace() {
                    in_word = false;
                } else if !in_word {
                    in_word = true;
                    n += 1;
                }
            }
            n
        }
    }
}

use crate::stats::words; // функции — через родительский модуль
use std::collections::HashMap; // типы — полным путём (так мы делали в главе 17)

fn main() {
    let text = "во поле берёза стояла";
    println!("слов: {}", words::count(text));

    let mut freq: HashMap<char, usize> = HashMap::new();
    for c in text.chars() {
        *freq.entry(c).or_insert(0) += 1;
    }
    println!("буква 'о' встретилась: {:?}", freq.get(&'о'));
}
```

```
слов: 4
буква 'о' встретилась: Some(3)
```

Идиома двухэтажная. **Функции** приводят через родителя: вызов `words::count(...)` честно показывает, что функция не местная, — голое `count(...)` читалось бы как своя. **Типы** приводят полным путём: `HashMap::new()` — все и так знают, что `HashMap` не ваш. Наш `use std::collections::HashMap;` из §17.2.1 был ровно этой идиомой. Исключение — тёзки: два `Result` из разных модулей в одной области требуют либо привода родителей (`fmt::Result` / `io::Result`), либо `as`:

```rust
mod plain {
    pub fn header() -> String {
        String::from("ОТЧЁТ")
    }
}

mod fancy {
    pub fn header() -> String {
        String::from("*** ОТЧЁТ ***")
    }
}

use plain::header; // ярлык прямо на функцию (для демонстрации конфликта)
use fancy::header as fancy_header; // as: переименование при конфликте имён

fn main() {
    println!("{}", header());
    println!("{}", fancy_header());
}
```

```
ОТЧЁТ
*** ОТЧЁТ ***
```

*Пример 4 — ⛔ ярлык не наследуется вложенной областью.* Самая частая ошибка новичка с `use`. Файл `use_scope.rs` (модуль `stats` — как в примере 3):

```rust
use crate::stats::words; // ярлык действует только в этой области — корне crate

mod report {
    pub fn print_stats(text: &str) {
        // ⛔ внутри модуля report ярлык из корня не действует
        println!("слов: {}", words::count(text));
    }
}
```

```
error[E0433]: cannot find module or crate `words` in this scope
  --> use_scope.rs:24:30
   |
24 |         println!("слов: {}", words::count(text));
   |                              ^^^^^ use of unresolved module or unlinked crate `words`
   |
   = help: you might be missing a crate named `words`

warning: unused import: `crate::stats::words`
  --> use_scope.rs:19:5
   |
19 | use crate::stats::words; // ярлык действует только в этой области — корне crate
   |     ^^^^^^^^^^^^^^^^^^^
```

Две диагностики — одна картина: модуль `report` — *другая область видимости*, ярлык корня туда не проникает (E0433), и потому сам ярлык остался неиспользованным (предупреждение). Починки три: перенести `use` внутрь `report`, написать `super::stats::words::count(...)` или полный путь `crate::...`.

**Связь с практикой: прелюдия.** Теперь можно закрыть вопрос, висевший с главы 17: почему `HashMap` потребовал `use`, а `String`, `Vec`, `Option`, `Result`, `Clone` — никогда? Потому что существует **прелюдия** (prelude) — небольшой список имён, который Rust автоматически приводит в область видимости каждого модуля (R9 — std::prelude). В нём типы `String`, `Vec`, `Option` с вариантами `Some`/`None`, `Result` с `Ok`/`Err`, трейты `Clone`, `Copy`, `PartialEq`, `PartialOrd`, `From`, функция `drop`, стандартные макросы (`println!` и родня) — а также имена, знакомство с которыми впереди: `Box` (глава 22), `Iterator` (глава 21). Список сознательно держат коротким — «то, что нужно почти каждой программе»; `HashMap` в него не прошёл. Всё прочее из `std` доступно без записей в Cargo.toml, но со своим `use` — стандартная библиотека поставляется с компилятором (R1, гл. 7.4).

**Типичные ошибки и подводные камни.**

- Ярлык — не «импорт файла». `use` ничего не подключает и не компилирует (в отличие от `mod`, §20.1.4) — только сокращает имя уже существующего в дереве элемента.
- Забытая область: `use` в корне не действует в модулях (пример 4). Симптом-подсказка — пара «E0433 + предупреждение unused import» в одной сборке.
- Глоб `use m::*` вне тестов: имена появляются «из ниоткуда», а при обновлении зависимости — ещё и конфликтуют. Идиоматическое применение одно, увидим его в §20.2.1.
- `pub use` — не просто ярлык: это обещание. Реэкспортированный путь — часть вашего публичного API (пример — в §20.1.4).

**Источники:** R1 (гл. 7.3–7.4), R7 (paths; use declarations), R9 (std::prelude), R6 (use, super, self).

### §20.1.4. Модули по файлам, зависимости и workspace

**Определения.** Пока все модули жили в одном файле. Разъезд по файлам — механический (R1, гл. 7.5): в родителе остаётся **объявление** `mod имя;` (с точкой с запятой, без тела), а тело переезжает в файл, который компилятор ищет по фиксированным правилам. Для модуля `clean`, объявленного в корне crate:

1. inline-тело `mod clean { ... }` — прямо на месте;
2. файл `src/clean.rs` — современный стиль;
3. файл `src/clean/mod.rs` — старый стиль (поддерживается; в одном проекте стили лучше не смешивать).

Подмодули файла-модуля ищутся в одноимённой папке: `mod trim;` внутри `src/clean.rs` — это файл `src/clean/trim.rs`. Ключевое отличие от C: **`mod` — не include**. Файл подключается к дереву **ровно одним** объявлением `mod`; все остальные места обращаются к нему по путям и `use` (R1, гл. 7.5). Текстовой вставки, ради обуздания которой C городил стражи включения (§11.1.4), здесь нет вовсе.

**Интуиция.** Дерево модулей ↔ дерево файлов:

```
   дерево модулей            дерево файлов (современный стиль)
   crate ─┬─ stats           src/lib.rs      ← mod stats;  mod clean;
          └─ clean           src/stats.rs    ← тело stats
                             src/clean.rs    ← тело clean

   объявление mod X; — «дырка» в дереве, файл X.rs — «заплатка»;
   компилятор сшивает их по именам. Никакого препроцессора:
   один файл = один узел дерева, сколько бы раз его ни использовали.
```

**Разобранные примеры.**

*Пример 1 — проект `textstat` целиком.* Собираем библиотеку статистики текста — тот самый пакет с двумя crate из §20.1.1. Четыре файла:

`src/lib.rs` — корень библиотеки, «оглавление»:

```rust
pub mod stats;

mod clean; // внутренний модуль: снаружи crate не виден

pub use stats::word_count; // реэкспорт: короткий путь к главной функции
```

`src/clean.rs` — приватная «кухня»:

```rust
// Внутренняя «кухня» crate: подготовка текста.
// Модуль объявлен в lib.rs без pub — снаружи crate его не видно.

fn is_junk(c: char) -> bool {
    c == '.' || c == ',' || c == '!' || c == '?' || c == ':' || c == ';'
}

// pub здесь означает «виден там, где виден сам модуль clean» —
// то есть внутри нашего crate, но не за его пределами.
pub fn normalize(text: &str) -> String {
    let mut out = String::new();
    for c in text.chars() {
        if !is_junk(c) {
            out.push(c);
        }
    }
    out
}
```

`src/stats.rs` — публичный интерфейс:

```rust
use std::collections::HashMap;

// Считает слова: пунктуация предварительно убирается, словом
// считается максимальная серия непробельных символов.
pub fn word_count(text: &str) -> usize {
    let cleaned = crate::clean::normalize(text);
    let mut n = 0;
    let mut in_word = false;
    for c in cleaned.chars() {
        if c.is_whitespace() {
            in_word = false;
        } else if !in_word {
            in_word = true;
            n += 1;
        }
    }
    n
}

// Считает, сколько раз встречается каждый символ исходного текста.
pub fn char_freq(text: &str) -> HashMap<char, usize> {
    let mut freq = HashMap::new();
    for c in text.chars() {
        *freq.entry(c).or_insert(0) += 1;
    }
    freq
}

// Средняя длина слова в символах (целочисленно, с округлением вниз).
// Паникует, если в тексте нет ни одного слова.
pub fn avg_word_len(text: &str) -> usize {
    let words = word_count(text);
    if words == 0 {
        panic!("пустой текст: слов нет");
    }
    let mut letters = 0;
    for c in crate::clean::normalize(text).chars() {
        if !c.is_whitespace() {
            letters += 1;
        }
    }
    letters / words
}
```

`src/main.rs` — бинарный crate пакета:

```rust
// Бинарный crate того же пакета: пользуется библиотекой по её имени,
// как любой внешний код, — видит только pub-элементы.
use textstat::stats;

fn main() {
    let text = "тише, мыши! кот на крыше";
    println!("слов: {}", textstat::word_count(text)); // через реэкспорт
    let freq = stats::char_freq(text);
    println!("буква 'и' встретилась: {:?}", freq.get(&'и'));
}
```

```
$ cargo run

    Finished `dev` profile [unoptimized + debuginfo] target(s) in 1.07s
     Running `target\debug\textstat.exe`
слов: 5
буква 'и' встретилась: Some(2)
```

Разберите каждую границу видимости: `stats` — `pub mod`, его функции — `pub`: это API. `clean` — `mod` без `pub`: модуль виден только внутри crate, и его `pub fn normalize` — «pub внутри забора» (§20.1.2, правило 2); `stats` зовёт её абсолютным путём `crate::clean::normalize`. Наконец, `pub use stats::word_count` в корне даёт внешнему миру короткое имя `textstat::word_count` — не заставляя помнить, в каком модуле функция живёт на самом деле; помните подсказку «not publicly re-exported» из §20.1.2 — это она о таком ярлыке. Реэкспорт — инструмент независимости *внутренней структуры* от *публичного фасада* (R1, гл. 14.2).

*Пример 5 — workspace, обзорно.* Когда пакетов становится несколько (библиотека + утилита, семейство библиотек), их объединяет **workspace** — набор пакетов с общими `Cargo.lock` и каталогом сборки (R1, гл. 14.3). Минимальный работающий пример — корневой `Cargo.toml` **без** секции `[package]`:

```toml
[workspace]
resolver = "3"
members = ["textstat_core", "wc"]
```

и два обычных пакета в подпапках, где `wc` зависит от соседа по пути:

```toml
[dependencies]
textstat_core = { path = "../textstat_core" }
```

```
$ cargo build          ← из корня: собирает всех членов
   Compiling textstat_core v0.1.0 (...\textstat_core)
   Compiling wc v0.1.0 (...\wc)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.82s
$ cargo run -p wc      ← флаг -p выбирает пакет
слов: 4
```

Проверено: `target/` и `Cargo.lock` существуют **только в корне** workspace — члены делят одну сборку (не пересобирать общие зависимости на каждого) и один набор залоченных версий (все члены видят одинаковые зависимости — R1, гл. 14.3). Зависимость между членами объявляется явно (`path = ...`): соседство по папке само по себе ничего не значит. Этого обзора хватит до конца курса; подробности — в R10.

**Связь с практикой.** Итоговая карта многофайлового проекта Rust (сравните с макетом из §11.2.1):

```
   textstat/
   ├── Cargo.toml        описание пакета и зависимостей
   ├── Cargo.lock        точные версии всего графа (генерируется, коммитится)
   ├── src/
   │   ├── lib.rs        корень библиотеки: mod-объявления, pub use
   │   ├── main.rs       бинарник-обёртка
   │   ├── stats.rs      модуль
   │   └── clean.rs      модуль
   ├── tests/            интеграционные тесты      → §20.2.2
   └── target/           всё собранное (в .gitignore)
```

**Типичные ошибки и подводные камни.**

- `mod` там, где нужен `use`. Написать `mod stats;` в двух файлах — значит попытаться подключить файл к дереву дважды: в одном модуле это E0428, в разных — два независимых узла-двойника с несовместимыми типами. Правило: `mod` — один раз в родителе, везде дальше — пути и `use`.
- Файл лежит в `src/`, но не объявлен `mod`-ом нигде — он просто не компилируется. В C забытый в Makefile файл ловился на линковке (`undefined reference`, §11.2.2); тут — «функция не найдена» уже на этапе разрешения имён.
- Правка `Cargo.toml` руками с опечаткой в версии или имени пакета: Cargo честно скажет, что такого пакета/версии нет, — но лучше привыкать к `cargo add`.
- Ожидание, что workspace даст пакетам доступ к приватному коду друг друга. Нет: члены workspace — обычные пакеты, видимость — по общим правилам (`pub` и только `pub`).

### Как это было в C

Глава 11 руками строила то, что здесь выдают из коробки. Сведём противопоставление (стороны C — §11.1–11.2):

- **Интерфейс и реализация.** C: пара `vec.h`/`vec.c`, заголовок — обещание, реализация — где-то в другой единице трансляции; согласованность — на совести автора (и линковщика, если повезёт). Rust: один модуль, интерфейс — это `pub`-подмножество, компилятор видит всё целиком.
- **Сокрытие.** C: `static` прячет имя в пределах файла — одна ступень, один уровень. Rust: приватность по умолчанию на каждом уровне дерева, `pub`/`pub(crate)` открывают ровно столько, сколько решено.
- **Повторное подключение.** C: `#include` — текстовая вставка, двойная вставка — ошибка переопределения, лекарство — стражи `#ifndef` в каждом заголовке (§11.1.4). Rust: `mod` подключает файл единожды; повторение — явная ошибка E0428 с указанием обоих мест.
- **Сборка.** C: Makefile перечисляет файлы, флаги и порядок линковки; забыл `-lm` в конце — `undefined reference` (§11.2.3). Rust: состав проекта следует из дерева `mod` и конвенций путей; зависимости — декларация в Cargo.toml, версии — в Cargo.lock.
- Чего в C **не было совсем**: реестра библиотек с версионированием (crates.io), автоматического графа транзитивных зависимостей, воспроизводимой фиксации версий. «Скачай, разложи, пропиши пути, молись при обновлении» — против одной строки `cargo add`.

## Часть 20.2. Тесты и документация

### §20.2.1. Юнит-тесты: #[test], assert и паника

**Определения.** **Тест** в Rust — обычная функция, помеченная атрибутом `#[test]` (R1, гл. 11.1). Команда `cargo test` собирает crate в специальном режиме — с **программой-прогонщиком** (test runner), которая вызывает все помеченные функции и составляет отчёт. Механика вердикта проста и целиком опирается на главу 18: **тест провален, если его функция запаниковала; иначе — пройден**. Утверждения — макросы, которые паникуют при лжи:

- `assert!(условие)` — паника, если условие ложно; вторым и дальше аргументами можно дать своё сообщение с `format!`-подстановками;
- `assert_eq!(лево, право)` и `assert_ne!(лево, право)` — сравнивают `==`/`!=` и при провале печатают **оба значения**. Отсюда требования к типам аргументов: `PartialEq` (чтобы сравнить) и `Debug` (чтобы напечатать) — вот зачем §19.2.3 приучал вешать на свои типы `#[derive(PartialEq, Debug)]`;
- `#[should_panic]` рядом с `#[test]` выворачивает вердикт: тест пройден, если паника **случилась**. Параметр `expected = "подстрока"` сужает: паника обязана содержать подстроку в сообщении;
- тест может возвращать `Result<(), E>`: `Err` — провал, `Ok(())` — успех (R1, гл. 11.1). Такой тест нельзя помечать `#[should_panic]`.

Тесты модуля по конвенции живут в подмодуле `tests` с атрибутом **`#[cfg(test)]`** — «компилировать только в режиме `cargo test`»: в обычную сборку `cargo build` тестовый код не попадает вовсе (R1, гл. 11.3).

**Интуиция.** Каждый тест — микропрограмма из трёх шагов (R1, гл. 11.1):

```
   настроить  →  выполнить  →  сверить
   (данные)      (наш код)     (assert-семейство)

   сверка лжёт  →  паника  →  прогонщик ловит панику в потоке теста
                              и пишет FAILED, остальные тесты живут дальше
```

А весь механизм — автоматизация того, что мы делали в томе I руками: «запусти и сверь вывод глазами» (так мы проверяли вектор в §9.1.1 и модуль в §11.2.1). Теперь сверяет машина, помнит все случаи сразу и делает это при каждом изменении.

**Разобранные примеры.**

*Пример 1 — шаблон как первый тест.* Заготовка `cargo new --lib` из §20.1.1 теперь читается целиком:

```rust
pub fn add(left: u64, right: u64) -> u64 {

}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_works() {
        let result = add(2, 2);
        assert_eq!(result, 4);
    }
}
```

`mod tests` — обычный подмодуль (§20.1.2); `use super::*` — глоб-ярлык «всё из родителя» (§20.1.3): тесты — *потомок* модуля с кодом, поэтому им видно всё, включая приватное. Это то самое узаконенное применение глоба. Запуск — в свежесозданном пакете:

```rust
        assert_eq!(freq.get(&'м'), Some(&2));
```

Читается сверху вниз: имя упавшего; перехваченный вывод теста — внутри знакомая по главе 18 паника (поток называется именем теста!) с расшифровкой `left`/`right`; итог `FAILED` и код выхода 101 (наблюдение из §18.1.1 про панику — работает и здесь). `10 filtered out` — остальные тесты не запускались: `broken` в команде — фильтр по имени (§20.2.2).

*Пример 4 — точность `expected`.* Параметр `expected` сверяет **подстроку** сообщения паники. Поставим заведомо чужую: `#[should_panic(expected = "деление на ноль")]` на тот же `avg_len_panics_on_empty` — тест падает, и прогонщик показывает обе строки:

```
note: panic did not contain expected string
      panic message: "пустой текст: слов нет"
 expected substring: "деление на ноль"
```

Зачем это нужно: `#[should_panic]` без `expected` пройдёт от **любой** паники — например, от случайного `v[10]` в настроечном коде теста, — и замаскирует настоящую ошибку. С `expected` тест проверяет, что паника — именно та (R1, гл. 11.1).

**Связь с практикой.** Куда класть тесты: подмодуль `tests` заводится в каждом файле-модуле, рядом с проверяемым кодом, — у нас в `stats.rs` и `clean.rs`. Правило курса «предупреждений ноль» распространяется и на тестовый код. И главная дисциплина: новый случай — сначала падающий тест, потом правка (увидите в Г20.Ч2.З9).

**Типичные ошибки и подводные камни.**

- Забытый `#[cfg(test)]` на `mod tests`. Проверено сборкой: `cargo build` проходит, но модуль компилируется и в обычную сборку — время и, с ростом тестов, зависимости; первый симптом — предупреждение `unused import: super::*` (сами `#[test]`-функции в обычную сборку не попадают, и ярлык остаётся без работы).
- `assert_eq!` на типе без `derive(PartialEq, Debug)` — E0277 дважды, как в §19.2.3. Лечится derive-ом в **основном** коде, не в тестовом.
- `#[should_panic]` без `expected` — тест-всеядина: любой сбой сойдёт за успех (пример 4).
- Result-тест плюс `#[should_panic]` — запрещённое сочетание (R1, гл. 11.1); паника в Result-тесте — всё равно провал, но уже «неструктурированный».

**Источники:** R1 (гл. 11.1, 11.3), R6 (testing: unit tests), R7 (attributes — testing; поведение паники: гл. 18), R9 (макросы assert/assert_eq/assert_ne).

### §20.2.2. Запуск тестов и интеграционные тесты

**Определения.** Прогонщик запускает тесты **параллельно, в отдельных потоках** (threads; сами потоки — материал главы 23, пока достаточно «несколько тестов исполняются одновременно») и **перехватывает** их стандартный вывод (R1, гл. 11.2). Управление — аргументами; всё после разделителя `--` уходит не Cargo, а тестовому бинарнику:

| Команда | Эффект |
|---|---|
| `cargo test имя` | только тесты, чьё полное имя **содержит** подстроку |
| `cargo test -- --test-threads=1` | последовательный запуск (общий внешний ресурс, отладка) |
| `cargo test -- --show-output` | показать вывод и у прошедших тестов |
| `cargo test -- --ignored` | только тесты с атрибутом `#[ignore]` |
| `cargo test -- --include-ignored` | вообще все, включая `#[ignore]` |
| `cargo test --lib` / `--test файл` | одна секция: юнит-тесты / конкретный файл из `tests/` |

По месту жительства тесты делятся на два вида (R1, гл. 11.3). **Юнит-тесты** живут в модулях кода (`#[cfg(test)] mod tests`), смотрят на модуль **изнутри** — видят приватное. **Интеграционные тесты** живут в каталоге `tests/` рядом с `src/`; каждый файл там — **отдельный crate**, который подключает вашу библиотеку по имени, как обычный пользователь, — и видит только `pub`. Атрибут `#[cfg(test)]` им не нужен: каталог `tests/` и так собирается только по `cargo test`.

**Интуиция.**

```
   ┌─ crate textstat ─────────────────────┐
   │  stats ─ tests ← юнит: внутри забора │   tests/public_api.rs
   │  clean ─ tests    видят приватное    │   ← интеграционный: СНАРУЖИ,
   │                                      │     отдельный crate, только
   └──────────── pub-фасад ───────────────┘     use textstat::...
                     ↑
   юнит отвечает: «верна ли деталь?»
   интеграционный: «работает ли изделие у пользователя?»
```

**Разобранные примеры.**

*Пример 1 — интеграционные тесты `textstat`.* Файл `tests/public_api.rs`:

Если общего ресурса не избежать — `cargo test -- --test-threads=1` (R1, гл. 11.2).

**Связь с практикой.** Каталог `tests/` — то, ради чего §20.1.1 советовал форму «библиотека + тонкий `main.rs`»: интеграционные тесты умеют подключать **только библиотечный** crate. Пакету из одного `src/main.rs` каталог `tests/` недоступен — функции бинарника не экспортируются (R1, гл. 11.3); лекарство — вынести логику в `lib.rs`, оставив в `main` вызовы. Наш `textstat` так и устроен, поэтому его `main.rs` можно не покрывать тестами: он тривиален, а вся логика проверена с двух сторон.

**Типичные ошибки и подводные камни.**

- Тесты, зависящие друг от друга или от порядка запуска: работают на одной машине, падают на другой. Симптом — «чинится» флагом `--test-threads=1`; это не починка, а диагноз.
- Помощник в `tests/common.rs` вместо `tests/common/mod.rs`: лишний пустой прогон в отчёте (`running 0 tests`) и путаница.
- Попытка `use textstat::...` на приватное — E0603 (пример 2): не открывайте внутренности ради теста, проверяйте их юнит-тестами изнутри.
- Фильтр — подстрока, не точное имя: `cargo test word` запустит и `counts_simple_words`, и `single_word_text`, и `full_pipeline_word_count`. Точнее — длиннее подстрока (или `--test` для выбора файла).

**Источники:** R1 (гл. 11.2 — параллельность, вывод, фильтры, ignore; гл. 11.3 — организация, tests/, common/mod.rs, бинарные crate), R10 (guide: project layout — каталог tests), R6 (testing: unit tests, other types of tests).

### §20.2.3. Документация, форматирование, линты

**Определения.** Последний слой инженерной гигиены — три инструмента, поставляемые вместе с Rust:

- **Doc-комментарии.** `///` документирует **следующий** элемент, `//!` — **объемлющий** (файл-модуль или корень crate); внутри — разметка Markdown (R1, гл. 14.2). Команда **`cargo doc`** прогоняет инструмент `rustdoc` и собирает из них HTML-справочник в `target/doc` (`--open` — сразу в браузер); документация зависимостей строится заодно. Конвенция заголовков: `# Examples`, а для функций с особенностями — `# Panics`, `# Errors`, `# Safety` (R1, гл. 14.2 — чек-лист того, что важно пользователю).
- **Doc-тесты.** Блок кода в doc-комментарии — не мёртвый текст: `cargo test` **компилирует и запускает** его как тест (секция `Doc-tests` из §20.2.2). Пример, разошедшийся с кодом, роняет сборку тестов — документация не умеет устаревать молча.
- **`cargo fmt`** (инструмент rustfmt) переформатирует весь код пакета под единый стандартный стиль, меняя только запись, не семантику; **`cargo clippy`** — линтер с сотнями проверок «компилятору формально не за что зацепиться, но так не пишут» (R1, приложение D; R6). Оба идут в стандартной поставке.

**Интуиция.** Комментарии из главы 11 и doc-комментарии — разные жанры:

```
   // комментарий            /// doc-комментарий
   для читателя ЭТОГО        для ПОЛЬЗОВАТЕЛЯ, который видит
   файла: почему так         только сигнатуру: что делает,
   написано                  что вернёт, когда паникует
        │                            │
        └── никто не проверяет       ├── пример = запускаемый тест
                                     └── cargo doc = сайт-справочник
   документация std, которой вы пользуетесь с главы 14, —
   это те же /// в исходниках стандартной библиотеки
```

```rust
    if s.is_empty() {
        println!("пустая строка");
    }
}
```

```
площадь круга: 12.566370614359172
```

`cargo clippy` теперь молчит. Многие подсказки компилятора и clippy умеет вносить сам `cargo fix` (R1, приложение D) — но читать их полезнее, чем автоприменять: почти каждая — микроурок идиоматики.

**Связь с практикой.** Предкоммитный ритуал проекта на Rust стандартен: `cargo fmt` → `cargo clippy` → `cargo test`. Все три — из коробки, без настройки; в командах разработчиков этот ритуал делают автоматическим условием принятия изменений. С этой главы задания курса будут считать решение готовым по этому же ритуалу.

**Типичные ошибки и подводные камни.**

- `///` перед приватным элементом: в справочник не попадёт (документируется публичный фасад), doc-тест не запустится. Для внутренних заметок — обычные `//`.
- Doc-тест без «сцены»: пример пишется от лица внешнего пользователя, поэтому пути — полные, от имени crate (`textstat::word_count(...)`), а не короткие внутренние.
- Ожидание, что `cargo fmt` исправит смысл: он только про запись. «Лишний `clone`» или `len() == 0` — работа clippy.
- Спор с форматером в отдельных местах. Возможен (`#[rustfmt::skip]`), но каждый такой островок — минус к главной ценности: единообразию.

**Источники:** R1 (гл. 14.2 — doc-комментарии, `//!`, doc-тесты, реэкспорты в документации; приложение D — rustfmt, cargo fix, clippy), R6 (testing: compiler lints and clippy — уровни линтов), R10 (команды cargo doc/test).

### Как это было в C

- **Проверка программ.** Том I проверял код так: собрать, запустить, вглядеться в вывод (§9.1.1, §11.2.1 — «тест-драйв» в `main`). Проверки жили в голове и в истории терминала; после правки — всё заново, руками. `cargo test` делает прогон машинным: все случаи, каждый раз, за доли секунды — и с вердиктом, который нельзя «не заметить».
- **Приватность и тестируемость.** В C `static`-функцию из другого файла не проверить в принципе — тест в неё физически не позовёт (§11.2.2). В Rust юнит-тест — потомок модуля: приватное тестируемо без вскрытия API.
- **Документация.** Комментарий в `vec.h` — обещание без залога: сигнатура уехала, комментарий остался. Doc-тест — обещание под присягой: разошёлся с кодом — упал конвейер.
- **Стиль и линты.** В C единый стиль держался на ревью и вкусах старшего; предупреждения компилятора мы включали флагами `-Wall -Wextra` и уговаривали себя их читать (§11.2). Здесь стиль пишет rustfmt, а clippy — это «-Wall для идиоматики», причём часть проверок запрещающие.

## Резюме главы

- **Проект имеет форму**: пакет (Cargo.toml) содержит crate — не больше одной библиотеки и сколько угодно бинарников; каждый crate — дерево модулей, растущее из корня (`src/lib.rs` / `src/main.rs`). Единицы трансляции и заголовки C схлопнулись в одну сущность: crate компилируется целиком, сигнатуры видны без дублирования.
- **Видимость — закон, а не соглашение**: всё приватно по умолчанию (виден элемент своему модулю и потомкам), `pub` открывает по звену пути, поля структур приватны поштучно, варианты `pub enum` публичны скопом. E0603/E0616 — компилятор на страже инкапсуляции; `static` из C дорос до системы границ на каждом уровне дерева.
- **Пути и `use`**: абсолютные от `crate::`, относительные и `super::`; `use` — ярлык, живущий только в своей области (E0433 — забыли об этом); идиомы — функции через родителя, типы целиком; `as` против тёзок; `pub use` — реэкспорт, отделяющий фасад от планировки; прелюдия объясняет, почему `String` и `Vec` «просто работали».
- **`mod` — не `#include`**: файл подключается к дереву ровно одним объявлением (E0428, E0761 — попытки иначе), стражи включения остались в томе I. Зависимости — строка в `[dependencies]` (или `cargo add`), точные версии всего графа — в Cargo.lock, воспроизводимо у всех; workspace объединяет пакеты общей сборкой.
- **Тесты — часть языка**: `#[test]`-функция падает паникой (`assert!`, `assert_eq!` с `left`/`right`, `#[should_panic(expected)]`, Result-тесты). Юнит-тесты в `#[cfg(test)] mod tests` видят приватное; интеграционные в `tests/` — отдельные crate, видят только `pub`; прогонщик параллелит, перехватывает вывод, фильтрует по имени и уважает `#[ignore]`.
- **Документация не врёт**: `///`/`//!` + Markdown → `cargo doc`; примеры в doc-комментариях — исполняемые тесты (секция Doc-tests). `cargo fmt` закрывает вопрос стиля, `cargo clippy` учит идиоматике (и запрещает худшее). Ритуал зрелого проекта: fmt → clippy → test.

**Дальше.** Инструментарий инженера собран, и им пора пользоваться на полную: глава 21 — **замыкания и итераторы**, функциональное сердце Rust. Наши ручные циклы со счётчиками — `word_count` с его флагом `in_word`, подсчёт букв, суммирование частот — превратятся в цепочки `chars().filter(...).count()`: короче, без изменяемых переменных и, вопреки интуиции из тома I, без потери скорости. Заодно станет видно, что `for` по вектору, `entry`/`or_insert` и даже `?` всю дорогу опирались на трейты — `Iterator` и его семью, — а тесты из этой главы будут проверять каждую переписанную цепочку.
