# Глава 18. Обработка ошибок

## Введение

Том I приучил вас к дисциплине: `malloc` вернул `NULL` — проверь (§7.1.2); `fopen` вернул `NULL` — проверь и позови `perror` (§10.1.1); `fscanf` вернул не столько преобразований, сколько ждал, — проверь (§10.1.2). Дисциплина работала, но держалась на честном слове: C **не заставляет** проверять возвращаемые значения. Программа, игнорирующая каждую из этих проверок, компилируется без единого предупреждения — а дальше §13.1.2: использование «полузначения» как настоящего — неопределённое поведение.

Глава 16 закрыла половину проблемы: `Option<T>` сделал «значения может не быть» типом, который нельзя прочитать невнимательно. Но `Option` молчит о **причине**: `None` не расскажет, файла не было или не хватило прав. Эта глава — про вторую половину: систему ошибок Rust целиком.

Два сюжета:

- **Два рода отказов.** Rust разводит их на уровне языка (R1, гл. 9): **невосстановимые** — баг в программе, жить с которым дальше нельзя, — это паника (`panic!`), знакомая вам по `v[99]` из главы 14; **восстановимые** — файл не открылся, ввод не разобрался — это `Result<T, E>`: обычное значение, которое несёт либо результат, либо причину неудачи. Плюс оператор `?` — одна закорючка вместо лесенок проверок из главы 10.
- **Проектирование ошибок.** Свой тип ошибки — это `enum` из главы 16 с данными в вариантах; `From` научит `?` автоматически переводить чужие ошибки в ваш тип; сквозной пример — программа, читающая числа из файла, — соберёт всё вместе: `panic!`, `Result`, `?`, собственные ошибки и границу, где отказ превращается в сообщение для человека.

После изучения главы вы будете уметь:

- вызывать и читать панику: сообщение, файл:строка:колонка, код выхода, трассировка `RUST_BACKTRACE=1` — и объяснять, что происходит со стеком при размотке;
- разбирать `Result<T, E>` match-ем, различать `unwrap`/`expect`/`unwrap_or` и говорить, когда каждый из них честен;
- пробрасывать ошибки оператором `?` — в том числе из `main`, — и расшифровывать E0277, когда `?` применён не там;
- проектировать собственные типы ошибок: `enum` с данными в вариантах вместо строк и «магических» кодов;
- писать `impl From<...>` (по рецепту — что такое трейт, узнаете в главе 19), чтобы `?` сшивал разные источники отказов в один тип;
- проводить границу: где `panic!`, где `Result`, и в каком месте программы ошибка должна стать человеческим сообщением.

Весь код главы прогнан на эталонном окружении курса: rustc/cargo **1.97.0 stable** (Windows, `x86_64-pc-windows-gnu`, издание 2024); выводы программ и тексты ошибок приведены дословно (служебные пути стандартной библиотеки сокращены — как в §16). Примеры — одиночные файлы, но с одним новым флагом: **`rustc --edition 2024 файл.rs`**. Голый `rustc` без флага собирает в старом издании языка 2015 года, где `panic!` ещё не умеет подставлять переменные в сообщение; cargo-проекты (§14.1.2) записывают `edition = "2024"` в `Cargo.toml` сами, а одиночному файлу издание надо назвать явно. Проверено на нашей машине: тот же файл без флага печатает `перегрев: {t}°` вместо `перегрев: 95°`.

## Часть 18.1. panic! и Result: два рода отказов

### §18.1.1. panic!: невосстановимые ошибки

**Определения.** **Паника (panic)** — механизм досрочного завершения программы при ошибке, с которой продолжать бессмысленно (R1, гл. 9.1). Панику вызывают два пути:

- **сам язык** — при нарушении гарантий безопасности: вы уже собрали целый каталог таких паник в главах 14–17 (индекс за границей, срез не по границе, переполнение в debug-профиле, `unwrap` на `None`);
- **вы явно** — макросом **`panic!("сообщение")`**. Сообщение форматируется как в `println!`: `panic!("перегрев: {t}°")` подставит значение `t`.

Что происходит дальше — по умолчанию **размотка (unwinding)** стека (R1, гл. 9.1; R2): выполнение прерывается, и стек вызовов сворачивается кадр за кадром — для каждого живого значения в каждом кадре вызывается `drop`, как будто функции вернулись обычным путём (R6). `String`, `Vec`, открытые файлы — всё освобождается по правилам главы 15. Затем процесс завершается с ненулевым кодом выхода — на нашей машине это код 101 (наблюдение прогона; стандартом курса номер не обещан).

Альтернатива размотке — **аварийное прекращение (abort)**: строка `panic = 'abort'` в секции профиля `Cargo.toml` (R1, гл. 9.1) велит завершаться немедленно, без очистки — память вернёт операционная система, как при `abort()` в C. Так делают ради размера и скорости исполняемого файла; курс везде использует размотку по умолчанию.

Важная граница с томом I: паника — **не** неопределённое поведение. `a[99]` в C — UB: молчаливый мусор, порча памяти, «повезло/не повезло» (§13.1.1). `v[99]` в Rust — определённая, документированная реакция: диагностика с точным местом и гарантированная остановка. Мост главы 13 работает: класс ошибок не исчез, но из невидимого стал громким.

**Интуиция.** Стек при обычном возврате и при размотке:

```
      обычный возврат                      паника в level_two
   ┌─────────────┐                      ┌─────────────┐
   │ main        │ ◀─ return           │ main        │ ◀─ 3) drop локальных; процесс
   ├─────────────┤                      ├─────────────┤       завершается, код 101
   │ level_one   │ ◀─ return           │ level_one   │ ◀─ 2) drop локальных
   ├─────────────┤                      ├─────────────┤
   │ level_two   │                      │ level_two   │ ── 1) panic! → drop локальных
   └─────────────┘                      └─────────────┘
```

Тот же маршрут, что у `return` из каждой функции по очереди, — только без возвращаемых значений и без права остановиться на полпути. Программа не «падает где попало», как C-процесс после UB, — она организованно сворачивается.

**Разобранные примеры.**

*Пример 1 — явная паника.* Файл `overheat.rs`:

```rust
fn check_temperature(t: i32) {
    if t > 90 {
        panic!("перегрев: {t}°");
    }
    println!("температура {t}° — норма");
}

fn main() {
    check_temperature(40);
    check_temperature(95);
    check_temperature(20);
}
```

```
температура 40° — норма

thread 'main' (26876) panicked at overheat.rs:3:9:
перегрев: 95°
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace
```

Разбор построчно. Первый вызов отработал. Второй дошёл до `panic!` — и программа завершилась: третий вызов **не выполнился**. В сообщении: имя потока (`main`; число в скобках — служебный номер потока, у вас будет другим — §14.2.2), **место** `файл:строка:колонка` (`overheat.rs:3:9` — именно строка макроса `panic!`), наше отформатированное сообщение и подсказка про трассировку. Код выхода — не ноль: `echo $?` в Git Bash печатает `101` (в PowerShell — `$LASTEXITCODE`); скрипты и системы сборки увидят, что программа не справилась, — как ненулевой `return` из `main` в C (§4.1).

*Пример 2 — каталог знакомых паник.* Всё это вы уже видели — теперь у него есть общее имя:

| Паника | Где встречали | Сообщение (дословно из прогонов) |
|---|---|---|
| индекс за границей | §14.2.2, §17.1.1 | `index out of bounds: the len is 3 but the index is 99` |
| срез вне диапазона | §15.3.1 | `range end index 7 out of range for slice of length 5` |
| срез не по границе символа | §17.1.4 | `end byte index 1 is not a char boundary; it is inside 'п' …` |
| `unwrap` на `None` | §16.2.3 | `called `Option::unwrap()` on a `None` value` |
| переполнение (debug-профиль) | §14.2.2 | `attempt to add with overflow` |

Контрольный прогон первого пункта — файл `v_oob.rs`:

```rust
fn main() {
    let v = vec![10, 20, 30];
    println!("{}", v[99]);
}
```

```
thread 'main' (9020) panicked at v_oob.rs:3:21:
index out of bounds: the len is 3 but the index is 99
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace
```

*Пример 3 — трассировка стека.* Паника глубоко в вызовах — где искать виноватого? Файл `unwind.rs`:

```rust
fn level_two(v: &[i32]) -> i32 {
    v[99]
}

fn level_one(v: &[i32]) -> i32 {
    level_two(v)
}

fn main() {
    let v = vec![10, 20, 30];
    println!("{}", level_one(&v));
}
```

Запуск с переменной окружения `RUST_BACKTRACE=1` (в Git Bash: `RUST_BACKTRACE=1 ./unwind.exe`; вывод сокращён — служебные кадры стандартной библиотеки укорочены):

```
thread 'main' (27824) panicked at unwind.rs:2:5:
index out of bounds: the len is 3 but the index is 99
stack backtrace:
   0: __rustc::rust_begin_unwind
   1: core::panicking::panic_fmt
   2: core::panicking::panic_bounds_check
   3: unwind::level_two
   4: unwind::level_one
   5: unwind::main
   6: <fn() as core::ops::function::FnOnce<()>>::call_once
note: Some details are omitted, run with `RUST_BACKTRACE=full` for a verbose backtrace.
```

Читается сверху вниз до **первого своего** кадра (R1, гл. 9.1): кадры 0–2 — внутренности механизма паники, кадр 3 — `unwind::level_two` — вот место бага; ниже — кто его вызвал (`level_one` ← `main`). Это тот самый стек вызовов, который вы рисовали для C в §4.2 и ловили отладчиком и санитайзерами в §7.2, — только выдаваемый языком из коробки при каждой панике.

**Связь с практикой.** Паника — это **сигнал о баге программисту**, а не способ поговорить с пользователем. Уместна там, где продолжение исполнения опаснее остановки: нарушен инвариант, «этого не может быть». В тестах (глава 20) паника — штатный механизм: упавший тест — это паника внутри него. Существует и способ поймать панику (`std::panic::catch_unwind` — R6), но он для особых случаев вроде серверов, переживающих отказ одного запроса, — строить на нём обработку ошибок «как исключения в других языках» не надо: для ожидаемых отказов у Rust есть `Result`, и это тема следующего параграфа.

**Типичные ошибки и подводные камни.**

- **Паника на ожидаемый отказ.** Файл не найден, пользователь опечатался — это не баг программы, а нормальная жизнь; паниковать здесь — как в C звать `abort()` вместо `if`. Правильный инструмент — `Result` (§18.1.2); полный свод правил «когда что» — §18.2.3.
- **Сообщение без контекста.** `panic!("ошибка")` оставит вас наедине со строкой-колонкой. В сообщение — то, что нельзя восстановить по коду: значения переменных (`panic!("перегрев: {t}°")`).
- **Ожидание «поймается выше».** Привычка из языков с исключениями. По умолчанию паника — это конец процесса; код после паникующего вызова не выполнится никогда.
- **Расчёт на размотку.** С `panic = 'abort'` очистки не будет. Логику программы нельзя строить на «drop точно вызовется при панике» — это деталь конфигурации.

**Источники:** R1 (гл. 9.1 — panic!, размотка против abort, чтение backtrace, buffer overread); R2 (error/panic — паника разматывает стек и освобождает ресурсы потока); R6 (error-handling/panics — паники для невосстановимых ошибок; catch_unwind и его границы; drop значений при размотке). Формат сообщения, код 101 и вид трассировки — прогоны на эталоне курса. Сторона C: §4.1–4.2, §7.2, §13.1 курса.

### §18.1.2. Result<T, E>: ошибка как значение

**Определения.** **`Result<T, E>`** — перечисление стандартной библиотеки для операций, которые могут не получиться (R1, гл. 9.2; R9 — арбитр по API):

```rust
enum Result<T, E> {
    Ok(T),
    Err(E),
}
```

Это `enum` с данными из §16.2.1 и близнец `Option<T>` из §16.2.3, только параметра типа два: `T` — тип результата при успехе (внутри `Ok`), `E` — тип ошибки при неудаче (внутри `Err`). Как и `Option`, `Result` с вариантами входит в прелюдию: `Ok(...)` и `Err(...)` пишутся без префиксов и без `use` (R9).

Рабочий пример на всю главу — открытие файла. **`File::open("имя")`** из `std::fs` возвращает `Result<File, io::Error>` (R1, гл. 9.2):

- **`File`** — дескриптор открытого файла, владелец ресурса. Это тот самый тип, который мы анонсировали в §16.1.3 как «две копии — ошибка по смыслу»: `File` не `Copy` и не `Clone`, только перемещается; при `drop` файл **закрывается сам** (R9) — наша дисциплина `fopen`/`fclose` из §10.1.1 стала свойством типа. (Одна честная оговорка R9: ошибки самого закрытия `drop` игнорирует — вспомните, что и возвращаемое значение `fclose` в томе I мало кто проверял.)
- **`io::Error`** — ошибка операционной системы: почему не открылось. У неё есть метод **`kind()`**, возвращающий категорию — перечисление **`io::ErrorKind`**: `NotFound`, `PermissionDenied` и десятки других (R9). `ErrorKind` помечен как пополняемый (`non_exhaustive`): в будущих версиях языка вариантов может стать больше, поэтому `match` по нему **обязан** иметь рукав `_` — исчерпываемость из §16.2.2 иначе недостижима (R9).

И главное отличие от C: `Result` помечен атрибутом **`#[must_use]`** — проигнорированный результат вызова даёт предупреждение компилятора (R9). Возвращаемое значение, которое «легко забыть проверить», в Rust забыть **шумно**.

Достать `T` из `Result` можно только разобрав его — как `Option`:

- **`match`** — полный разбор обоих русел (примеры ниже);
- **`unwrap()`** — «вскрыть или паниковать»: `Ok(v)` → `v`, `Err(e)` → паника со стандартным сообщением (R1; R9);
- **`expect("почему я уверен")`** — тот же `unwrap`, но паника с вашим сообщением (R1; R9);
- **`unwrap_or(умолчание)`** — значение или умолчание, без паники (R9; близнец `unwrap_or` у `Option`, §16.2.3).

**Интуиция.** Сравните каналы доставки ошибки:

```
   C (§10.1.1):                          Rust:
   FILE *f = fopen("x", "r");            let r = File::open("x");
        │                                     │
   ┌────┴─────────────┐                  ┌────┴──────────────┐
   │ адрес            │ NULL             │ Ok(File)          │ Err(io::Error)
   │                  │ причина — errno, │   результат       │   причина — внутри,
   │ можно молча      │ сбоку, в глобали │                   │   привязана к отказу
   │ использовать     │ можно молча      │ использовать «как File» нельзя:
   │                  │ разыменовать: UB │ сначала match — компилятор проследит
   └──────────────────┘                  └───────────────────┘
```

В C успех и неудача возвращаются **в одном канале** («указатель или NULL», «число или −1», «число или EOF» — §13.1.2), а причина едет отдельно, в глобальной `errno`. В Rust оба русла — **один тип-сумма**, причина упакована в само значение ошибки, и система типов не даст перепутать русла: `Result<File, io::Error>` — это не `File`, пока вы не доказали match-ем, что там `Ok`.

**Разобранные примеры.**

*Пример 1 — полный разбор match-ем.* Файл `open_match.rs`:

```rust
use std::fs::File;

fn main() {
    let result = File::open("notes.txt");
    let file = match result {
        Ok(f) => f,
        Err(e) => {
            eprintln!("не удалось открыть notes.txt: {e}");
            return;
        }
    };
    println!("файл открыт: {file:?}");
}
```

Новое здесь `eprintln!` — тот же `println!`, но в **stderr**: прямой аналог `fprintf(stderr, ...)` из §10.1.1 — сообщения об ошибках не смешиваются с полезным выводом и переживают перенаправление stdout (проверено прогоном: `./open_match.exe > /dev/null` — сообщение видно). `use std::fs::File` — второй `use` курса, читается как «дай короткое имя», по образцу `use std::collections::HashMap` из §17.2.1; система модулей — глава 20.

Запуск без файла и с файлом:

```
не удалось открыть notes.txt: The system cannot find the file specified. (os error 2)
```

```
файл открыт: File { handle: 0xcc, path: "\\\\?\\C:\\...\\notes.txt" }
```

Два наблюдения. Текст ошибки пришёл **от операционной системы** (Windows; на Linux будет `No such file or directory`), с кодом `os error 2` — это старый знакомый `errno == ENOENT`, только приехавший внутри значения. А отладочная печать `File` показывает дескриптор и путь с префиксом `\\?\` — расширенный синтаксис путей Win32, привет из главы 12 (путь сокращён, числа дескриптора у вас будут другими).

⛔ Приём для незнакомых функций: не помните тип результата — спросите компилятор заведомо неверной аннотацией. Файл `open_wrong_type.rs`. Не компилируется (проверено):

```rust
use std::fs::File;

fn main() {
    let f: u32 = File::open("notes.txt");
    println!("{f}");
}
```

```
error[E0308]: mismatched types
 --> open_wrong_type.rs:4:18
  |
4 |     let f: u32 = File::open("notes.txt");
  |            ---   ^^^^^^^^^^^^^^^^^^^^^^^ expected `u32`, found `Result<File, Error>`
  |            |
  |            expected due to this
  |
  = note: expected type `u32`
             found enum `Result<File, std::io::Error>`
```

Знакомый E0308 (§14.2.2) в роли справочной службы: `found enum Result<File, std::io::Error>` — вот и сигнатура (приём из R1, гл. 9.2).

*Пример 2 — разные причины: ErrorKind.* «Нет файла» и «нет прав» — разные ситуации: первую можно исправить, создав файл. Файл `open_or_create.rs`:

```rust
use std::fs::File;
use std::io::ErrorKind;

fn main() {
    let file = match File::open("journal.txt") {
        Ok(f) => f,
        Err(e) => match e.kind() {
            ErrorKind::NotFound => match File::create("journal.txt") {
                Ok(f) => {
                    println!("файла не было — создали пустой");
                    f
                }
                Err(e2) => panic!("не удалось создать journal.txt: {e2}"),
            },
            _ => panic!("не удалось открыть journal.txt: {e}"),
        },
    };
    println!("работаем с файлом: {file:?}");
}

Правило выбора между ними: **`Option`** — когда отсутствие значения нормально и причин не бывает («в векторе нет десятого элемента»); **`Result`** — когда отказ случился по причине, которую вызывающему может понадобиться знать или показать («файл не открылся, потому что…») (R2; R6). И правило для паникующих методов: в готовом коде каждый `unwrap`/`expect` — утверждение «`Err` здесь невозможен, и вот почему» — формулируйте это «почему» прямо в `expect` (R1, гл. 9.3).

**Типичные ошибки и подводные камни.**

- **Рефлекторный `unwrap`.** То же, что рефлекторный `unwrap` у `Option` (§16.2.3), но хуже: вы выбрасываете ещё и причину. В прототипе — честно; в готовой программе — мина.
- **`let _ =` как привычка.** Глушитель предупреждения не делает ошибку обработанной — он делает её невидимой. Это C-поведение, купленное за явный синтаксис.
- **`{e:?}` пользователю.** `Os { code: 2, kind: NotFound, ... }` — для программиста (Debug, §16.1.3); человеку — `{e}` (Display): `The system cannot find the file specified.` Красивый Display для *своих* типов — глава 19, пока их печатаем через `{:?}`.
- **Ставка на текст ошибки.** Сообщение ОС зависит от платформы и локали. Ветвить логику — по `e.kind()`, а не по подстроке сообщения.

### §18.1.3. Оператор ?: проброс наверх

**Определения.** Функция, которая не знает, как поступить с ошибкой, должна не решать за всех, а **пробросить** (propagate) ошибку вызывающему (R1, гл. 9.2): вернуть `Result` — пусть решает тот, у кого есть контекст. В томе I вы делали это руками: каждая функция проверяла каждый вызов и возвращала свой `-1`/`NULL` наверх.

Оператор **`?`** («вопросительный знак»; в справочнике R7 — *try propagation expression*, выражение проброса) сворачивает этот ритуал в один символ. Постфиксно, как метод: `выражение?`. Семантика по R7:

- `Ok(v)` → значение всего выражения — `v`, выполнение продолжается;
- `Err(e)` → **немедленный выход из текущей функции**: `return Err(From::from(e))`.

`From::from` — конвертация ошибки «на вылете»; пока тип ошибки совпадает с типом в сигнатуре, она тождественна, и о ней можно не думать — её звёздный час в §18.2.2. Работает `?` и с `Option<T>`: `Some(v)` → `v`, `None` → досрочный `return None` (R1, гл. 9.2; R7).

Отсюда два ограничения (оба проверяет компилятор, ошибка E0277):

1. `?` разрешён только в функции, возвращающей `Result` или `Option` — иначе «выпадать» некуда;
2. русла должны совпадать: `?` на `Result` — в функции с `Result` (и типы ошибок обязаны сходиться — через `From`, §18.2.2), `?` на `Option` — в функции с `Option`.

И особый случай: **`main` тоже может возвращать `Result`**. Сигнатура `fn main() -> Result<(), E>` легальна для любого типа ошибки `E`, который умеет печататься через `{:?}` (R1, гл. 9.2; R9 — механизм `Termination`, за кулисами превращающий возврат `main` в код выхода процесса): `Ok(())` → код 0, `Err(e)` → отладочная печать ошибки и ненулевой код.

**Интуиция.** Лесенка из главы 10 — и что от неё осталось:

```
   C, §10.1: каждый уровень проверяет сам          Rust: тот же контроль — один символ
   ─────────────────────────────────────           ──────────────────────────────────
   FILE *f = fopen("app.log", "r");                let f = File::open("app.log")?;
   if (f == NULL) { perror(...); return -1; }      let m = f.metadata()?;
   long n = get_size(f);                           Ok(m.len())
   if (n < 0)     { fclose(f);  return -1; }
   ...                                             каждая ? — «жёлоб»: Err выпадает
                                                   из функции сразу, с причиной внутри;
                                                   Ok едет по конвейеру дальше
```

Проверки не исчезли — они происходят **в каждой** `?`, просто написаны за вас. И в отличие от C, «жёлоб» невозможно забыть подключить: без `?` (или match) значение `Result` просто нельзя использовать.

**Разобранные примеры.**

*Пример 1 — вручную и через ?.* Размер файла журнала: два отказных вызова подряд — `File::open` и `metadata()` (метаданные файла; `.len()` у них — размер в байтах, R9). Файл `log_size.rs`:

```rust
use std::fs::File;
use std::io;

fn log_size_manual() -> Result<u64, io::Error> {
    let f = match File::open("app.log") {
        Ok(f) => f,
        Err(e) => return Err(e),
    };
    let meta = match f.metadata() {
        Ok(m) => m,
        Err(e) => return Err(e),
    };
    Ok(meta.len())
}

fn log_size() -> Result<u64, io::Error> {
    let f = File::open("app.log")?;
    let meta = f.metadata()?;
    Ok(meta.len())
}

fn main() {
    println!("вручную: {:?}", log_size_manual());
    println!("через ?: {:?}", log_size());
}
```

Запуск без `app.log` и с файлом в 12 байт:

```
вручную: Err(Os { code: 2, kind: NotFound, message: "The system cannot find the file specified." })
через ?: Err(Os { code: 2, kind: NotFound, message: "The system cannot find the file specified." })
```

```
вручную: Ok(12)
через ?: Ok(12)
```

Функции эквивалентны — по обоим руслам. `log_size_manual` — это и есть смысл `?`, выписанный целиком: образец «`match` + ранний `return Err`» вы уже писали в томе I десятки раз, просто на C. Обратите внимание на последнюю строку обеих функций: `Ok(meta.len())` — упаковать результат в `Ok` должны вы сами; `?` распаковывает, но не запаковывает.

*Пример 2 — ? в main.* Файл `main_result.rs`:

```rust
use std::fs::File;
use std::io;

fn main() -> Result<(), io::Error> {
    let f = File::open("data.txt")?;
    println!("файл открыт: {f:?}");
    Ok(())
}
```

Запуск без файла (код выхода — `echo $?` → `1`) и с файлом (код 0):

```

```

```
файл открыт: File { handle: 0xc0, path: "\\\\?\\C:\\...\\data.txt" }
```

Строку `Error: ...` печатает не наш код — это рантайм Rust отчитывается о `Err`, долетевшем до самого верха, отладочной печатью (потому и `Os { ... }`, а не человеческий текст — Debug, §16.1.3). Для утилит «сделай или скажи почему нет» этого достаточно; программа для людей разберёт ошибку match-ем сама — §18.2.3.

⛔ А вот `?` в обычном `main() -> ()` не скомпилируется. Файл `q_in_plain_main.rs` (проверено; хвост подсказки сокращён):

```rust

    let f = File::open("data.txt")?;
    println!("{f:?}");

```

```
error[E0277]: the `?` operator can only be used in a function that returns `Result` or `Option` (or another type that implements `FromResidual`)
 --> q_in_plain_main.rs:4:35
  |
3 | fn main() {
  | --------- this function should return `Result` or `Option` to accept `?`
4 |     let f = File::open("data.txt")?;
  |                                   ^ cannot use the `?` operator in a function that returns `()`
```

Подсказка компилятора (в сокращённом хвосте) предлагает `Result<(), Box<dyn std::error::Error>>` — «коробку с любой ошибкой»; `Box` — умный указатель из главы 22, нам пока хватает конкретных типов. Упомянутый в скобках `FromResidual` — служебный механизм `?` (нестабильный, R7), читайте просто «Result или Option».

*Пример 3 — ? на Option.* Всё то же, без ошибок — с отсутствием. Файл `delivery.rs`:

```rust
struct Delivery {
    weight_kg: Option<u32>,
    price_per_kg: Option<u32>,
}

fn cost(d: &Delivery) -> Option<u32> {
    let w = d.weight_kg?;
    let p = d.price_per_kg?;
    Some(w * p)
}

fn main() {
    let full = Delivery { weight_kg: Some(3), price_per_kg: Some(120) };
    let partial = Delivery { weight_kg: Some(3), price_per_kg: None };
    println!("{:?}", cost(&full));
    println!("{:?}", cost(&partial));
}
```

```
Some(360)
None
```

Заполнена ли анкета целиком? Вместо лесенки из двух `match` — два `?`: первое же `None` досрочно возвращает `None` из `cost`. Сравните с «протаскиванием −1 через формулы» из тома I — здесь недозаполненность не участвует в умножении, она выходит из функции раньше.

**Как это было в C.** Проброс ошибки на N уровней вверх — N ручных проверок: `main` проверяет `load_config`, тот проверяет `read_line`, тот — `fgetc` (§10.1–10.2). Каждый уровень обязан помнить три вещи: проверить, освободить своё (файлы, память — §7.1), вернуть правильный код. Оператор `?` автоматизирует первое и третье, а второе в Rust делает `drop` при раннем `return` — размотка «в миниатюре», как в §18.1.1, только без паники: локальные владельцы освобождаются при любом выходе из функции (§15.1.1). Трёхэтажная дисциплина тома I схлопнулась в символ.

**Связь с практикой.** Правило по умолчанию: **внутри логики — `?`, на границе — `match`**. Функции середины программы не знают, что делать с «файл не найден», — им пробрасывать; знает граница: `main`, обработчик команды, цикл интерфейса — там ошибка становится сообщением, повтором или значением по умолчанию. Признак хорошо спроектированной функции — сигнатура честно говорит `-> Result<T, E>`, а тело читается как счастливый путь с `?` после каждого отказного шага.

**Типичные ошибки и подводные камни.**

- **`?` там, где надо обработать.** Если у *этой* функции есть план Б (создать файл, взять умолчание) — нужен `match`/`unwrap_or`, а не проброс: `?` отдаёт решение наверх, хотя оно было вашим.
- **Забытый `Ok(())`.** Функция `-> Result<(), E>` обязана явно вернуть успех; без последней строки `Ok(())` — E0308 «expected Result, found ()».
- **Смешение русел.** `?` на `Option` внутри функции, возвращающей `Result` (и наоборот), — E0277: у русел разные типы. Преобразование между ними существует (`ok_or` и др. — R9), но в этом курсе появится позже.
- **`unwrap` рядом с `?`.** Если функция уже возвращает `Result`, паникующий `unwrap` в её теле почти всегда проигрыш: тот же символ `?` дал бы вызывающему шанс.

**Источники:** R7 (expressions/operator-expr, «The try propagation expression» — арбитр: семантика Ok/Err и Some/None, `Err(From::from(e))`, связь с Try/FromResidual); R1 (гл. 9.2 — проброс вручную и через ?, ? на Option, ограничения и текст E0277, main -> Result и коды выхода); R9 (std::process::Termination — механизм возврата из main; std::fs::File::metadata); R2 (error/result/enter_question_mark — «unwrap или return Err»); R6 (error-handling/try-operator). Сторона C: §7.1, §10.1–10.2 курса.

## Часть 18.2. Проектирование ошибок

### §18.2.1. Собственный тип ошибки

**Определения.** `io::Error` описывает отказы операционной системы — а чем описать отказы **вашей** логики? Тем же инструментом, которым стандартная библиотека сделала `Result` и `ErrorKind`: перечислением из главы 16. **Тип ошибки** — это обычный `enum`, где каждый вариант — один способ отказа, а данные варианта — подробности этого способа.

Каким должен быть хороший тип ошибки (критерии по R2):

1. **один тип на функцию** — все способы отказа `parse_u32` живут в одном `ParseError`, вызывающему не нужно гадать, чего ждать;
2. **конкретные варианты вместо строк** — `Err(ParseError::Empty)` можно разобрать match-ем и сравнить; `Err(String::from("пустая строка"))` — только напечатать (и опечатка в тексте станет логикой программы);
3. **данные внутри варианта** — `BadChar('x')` говорит, *какой* символ помешал; голое `BadChar` заставило бы пользователя искать самому;
4. **сочетаемость** — тип должен уметь встраиваться в чужие ошибки (этим займётся §18.2.2).

Технически всё уже знакомо: `#[derive(Debug)]` для отладочной печати (§16.1.3) — он же делает тип пригодным для `expect`/`unwrap` и для `main -> Result` (там ошибка печатается через `{:?}`, §18.1.3); данные в вариантах и `match` с вложенными образцами — §16.2.1–16.2.2.

**Интуиция.** Ошибка — это **данные о неудаче**, и проектируется как данные. В томе I «данными о неудаче» служил `int`-код, а расшифровка жила в комментарии или README:

```
   C (том I):                              Rust:
   #define ERR_EMPTY   -1                  enum ParseError {
   #define ERR_BADCHAR -2   ← а какой          Empty,
   #define ERR_TOOBIG  -3     символ? —        BadChar(char),   ← подробность едет
   int parse_u32(const char*, ...);          внутри значения
                                                 TooBig,
   расшифровка кодов — в голове           }
   компилятор их не различает:            компилятор различает варианты,
   вернуть -5 — пожалуйста                match требует разобрать все
```

**Разобранные примеры.**

*Пример 1 — parse_u32.* Разбор строки в число — руками, как мы делали в C посимвольные проходы (гл. 5), но с честной сигнатурой. Файл `parse_u32.rs` (первая половина листинга; продолжение — пример 2):

```rust
#[derive(Debug)]
enum ParseError {
    Empty,
    BadChar(char),
    TooBig,
}

fn parse_u32(s: &str) -> Result<u32, ParseError> {
    if s.is_empty() {
        return Err(ParseError::Empty);
    }
    let mut acc: u32 = 0;
    for c in s.chars() {
        let d = match c.to_digit(10) {
            Some(d) => d,
            None => return Err(ParseError::BadChar(c)),
        };
        if acc > (u32::MAX - d) / 10 {
            return Err(ParseError::TooBig);
        }
        acc = acc * 10 + d;
    }
    Ok(acc)
}
```

Разбор построчно. Сигнатура — контракт: «дай строковый срез, получишь либо число, либо одну из трёх причин». Три новых метода стандартной библиотеки (все — R9): `s.is_empty()` — то же, что `s.len() == 0`; `c.to_digit(10)` — «символ как цифра указанного основания»: `Some(7)` для `'7'`, `None` для `'x'` (и для кириллической `'а'` — проверено прогоном); `c.is_whitespace()` встретится в §18.2.3. Ранние `return Err(...)` — стиль тома I, официально одобренный: каждый отказ покидает функцию в момент обнаружения. Страж переполнения — школа главы 14: `acc * 10 + d` обязан влезть в `u32`, иначе — паника в debug-профиле (§14.2.2); проверка `acc > (u32::MAX - d) / 10` отсекает ровно те `acc`, на которых умножение выйдет за `u32::MAX` (граница проверена прогоном: `"4294967295"` — `Ok`, `"4294967296"` — `TooBig`).

*Пример 2 — разбор ошибки match-ем.* Вторая половина `parse_u32.rs`:

```rust
fn report(input: &str) {
    match parse_u32(input) {
        Ok(n) => println!("{input:?} -> число {n}"),
        Err(ParseError::Empty) => println!("{input:?} -> ошибка: пустая строка"),
        Err(ParseError::BadChar(c)) => println!("{input:?} -> ошибка: недопустимый символ {c:?}"),
        Err(ParseError::TooBig) => println!("{input:?} -> ошибка: не помещается в u32"),
    }
}

fn main() {
    println!("{:?}", parse_u32("2048"));
    println!("{:?}", parse_u32("12x7"));
    report("2048");
    report("");
    report("12x7");
    report("99999999999");
    report("4294967295");
    report("4294967296");
    report("007");
}
```

```

Err(BadChar('x'))

"" -> ошибка: пустая строка
"12x7" -> ошибка: недопустимый символ 'x'
"99999999999" -> ошибка: не помещается в u32

Образцы вида `Err(ParseError::BadChar(c))` — **вложенные**: снаружи вариант `Result`, внутри вариант `ParseError`, ещё глубже — связывание `c` (композиция образцов из §16.2.2). Исчерпываемость работает на обоих уровнях: забудете `TooBig` — E0004. Первые две строки `main` показывают «сырой» вид через `{:?}` — так ошибки выглядят в отладке и в панике `unwrap`; `report` — «человеческая» граница, о которой шла речь в §18.1.3.

**Как это было в C.** Приёмы тома I под этой лампой:

| Задача | C (том I) | Rust |
|---|---|---|
| перечислить способы отказа | `enum`/`#define`-коды (§8.2.1) | варианты `enum`-ошибки |
| вернуть и код, и результат | код возврата + out-параметр `int *out` (§6.1.3) | `Result<u32, ParseError>` — оба русла в типе |
| подробность отказа («какой символ?») | ещё один out-параметр или глобаль | данные варианта: `BadChar(char)` |
| расшифровать код в сообщение | `switch` + таблица строк | `match` — с проверкой полноты |
| выдумать новый код «на месте» | `return -5;` — компилятору всё равно | вариант надо объявить; опечатка — ошибка компиляции |

Глобальная `errno` (§10.1.1) — отдельный урок: один общий канал причин на всю программу, который каждый следующий вызов перезаписывает. Значение-ошибка Rust привязано к конкретному отказу конкретного вызова — гонок за глобаль нет по построению.

**Связь с практикой.** Правило слоёв (R1, гл. 9.3): функции-«библиотеки» (наша `parse_u32`) на ожидаемо плохой вход отвечают `Err`, а не паникой и не печатью, — что делать с отказом, решает вызывающий; печатают — границы (наш `report`). Проектируйте варианты от вопросов вызывающего: «что я смогу сделать по-разному?» — `Empty` можно молча пропустить, на `BadChar` показать символ, на `TooBig` предложить другой тип. Если по двум вариантам вызывающий не может поступить по-разному — объедините их в один.

**Типичные ошибки и подводные камни.**

- **`Result<T, String>`.** Работает, но проваливает критерии 2–4: не разобрать match-ем без сравнения строк, не пронести данные, не встроить в чужой тип. Строка — для человека, тип — для программы.
- **Паника в глубине.** `panic!("bad char")` внутри `parse_u32` отнимает у вызывающего выбор — а «плохой символ» для парсера не баг, а вторник (R1, гл. 9.3).
- **Вариант без подробностей.** `BadChar` без `char` — вызывающему придётся парсить заново, чтобы узнать, что не так. Кладите в вариант то, что уже знаете в точке отказа.
- **Один вариант `Error` на всё.** Тип есть, а критерия 1 нет: match всё равно не различит причин. Это `-1` в костюме.

**Источники:** R2 (error/multiple_error_types/define_error_type — критерии хорошего типа ошибки: один тип, сравнимость, данные, композиция); R1 (гл. 9.3 — Result для ожидаемых отказов, паника — для багов); R9 (std::result; str::is_empty, char::to_digit, u32::MAX); R6 (error-handling — Result со своим типом ошибки как стандартный образец). Сторона C: §5.2, §6.1.3, §8.2.1, §10.1.1 курса.

### §18.2.2. From: научить ? переводить

**Определения.** Реальная функция отказывает по-разному: чтение файла — `io::Error`, разбор содержимого — `ParseError`. А тип ошибки в сигнатуре один. Решение Rust — **объединяющий enum** плюс **конвертация**:

```rust
#[derive(Debug)]
enum AppError {
    Io(io::Error),      // вариант-обёртка: исходная ошибка внутри
    Parse(ParseError),
}
```

Осталось научить `?` заворачивать `io::Error` в `AppError::Io` автоматически. Вспомните семантику `?` из §18.1.3 (R7): `Err(e)` → `return Err(From::from(e))`. **`From`** — стандартный механизм преобразования «значение → значение» (R9): реализация `From<X> for Y` — это функция `from`, строящая `Y` из `X`. Оператор `?` вызывает её на каждой пробрасываемой ошибке: если тип ошибки выражения не совпадает с типом ошибки функции, но объявлено «как из первого сделать второй», — `?` переведёт сам (R1, гл. 9.2; R9).

Пишется это блоком `impl`, знакомым по §16.1.2, — с одной новой деталью в заголовке:

```rust
impl From<io::Error> for AppError {
    fn from(e: io::Error) -> AppError {
        AppError::Io(e)
    }
}
```

Читается: «реализую для `AppError` преобразование из `io::Error`: вот функция `from`». Слово `From` здесь — **трейт** (typeclass-механизм Rust, тема главы 19). Пока — пользуемся как рецептом: `impl From<Чужая> for Моя { fn from(...) -> Моя { ...обернуть... } }`; глава 19 объяснит, почему заголовок устроен именно так и что ещё так можно.

**Интуиция.** `?` с `From` — таможня на выходе из функции:

```
        fn sum_of_file(...) -> Result<u32, AppError>
   ┌──────────────────────────────────────────────────┐
   │  fs::read_to_string(path)?      parse_numbers(…)? │
   │        │ Err(io::Error)              │ Err(ParseError)
   │        ▼                             ▼            │
   │  From<io::Error>              From<ParseError>    │  ← «таможня» ?: каждый Err
   │        └────────► AppError ◄─────────┘            │    переодевается в форму
   └──────────────────────────────────────────────────┘    функции — и вылетает
```

Каждая ошибка пересекает границу функции только в «форме» этой функции. Вызывающий видит один тип — `AppError` — и разбирает его одним match-ем.

**Разобранные примеры.**

*Пример 1 — без From не собирается.* ⛔ Файл `no_from.rs` (проверено):

`read_config_manual` — это `?` без сахара: `match`, ранний `return`, `From::from`. Какой именно `From` вызовется, решает **сигнатура функции**: компилятор знает, что вернуть надо `AppError`, а на руках `io::Error`, — и ищет ровно `From<io::Error> for AppError`. Одна и та же строка `fs::read_to_string(...)?` в функции с другим типом ошибки вызвала бы другой `From` (или E0277, если его нет).

**Связь с практикой.** Обычный рисунок программы: один `AppError` на приложение (или на крупный слой), по варианту-обёртке на каждый «чужой» источник ошибок (`Io(io::Error)`, `Parse(ParseError)`, …) и по `impl From` на обёртку. После этого весь код слоя пишет `?` не задумываясь — таможня одета. Правило обёртки: **сохраняйте исходную ошибку внутри варианта**, а не пересказ строкой: `Io(io::Error)` позволит вызывающему спросить `e.kind()`, а `Io(String)` — нет.

**Типичные ошибки и подводные камни.**

- **Потерянная причина.** Вариант `Io` без данных (или с одной строкой) обнуляет ценность обёртки: «был какой-то ввод-вывод». Заворачивайте оригинал.
- **Ожидать от `?` чудес.** `?` вызывает только `From`, объявленный вами (или стандартной библиотекой). Нет `impl` — E0277, и это правильно: неявных «как-нибудь приведи» в Rust нет.
- **Путать направление.** `From<io::Error> for AppError` — «из io::Error в AppError». Написали наоборот — получите E0277 с тем же текстом; сверяйтесь с формулой «из Чужой для Моя».
- **`from` с логикой.** Конвертация должна быть тривиальной упаковкой. Решения («а может, повторить запрос?») — не её уровень: `From` обязан быть безотказным (R9 — «конвертация не должна проваливаться»).

**Источники:** R9 (std::convert::From — определение, значение для обработки ошибок и ?, «не должна проваливаться»; std::fs::read_to_string — контракт и ошибки); R7 (try propagation expression — Err(From::from(e)), арбитр); R1 (гл. 9.2 — «ошибки, проходящие через ?, идут через from из трейта From»); R6 (error-handling/try-conversions — развёртка ? с From::from, обёртка io::Error в свой enum, From против map_err). Сторона C: §10.1.1 курса.

### §18.2.3. Сквозной пример: сумма из файла

**Постановка.** Файл `scores.txt` содержит неотрицательные числа, разделённые пробелами и переводами строк. Программа печатает их сумму — или внятно объясняет, что не так. Это ровно та программа, которую мы писали в C в §10.1.2 (`fscanf`-цикл со счётчиком успешных преобразований) — теперь со всей машинерией главы.

**Полный листинг** — файл `sum_scores.rs` (собран из готовых деталей: `parse_u32` — §18.2.1, `AppError` и оба `From` — §18.2.2):

```rust
use std::fs;
use std::io;

#[derive(Debug)]
enum ParseError {
    Empty,
    BadChar(char),
    TooBig,
}

#[derive(Debug)]
enum AppError {
    Io(io::Error),
    Parse(ParseError),
}

impl From<io::Error> for AppError {
    fn from(e: io::Error) -> AppError {
        AppError::Io(e)
    }
}

impl From<ParseError> for AppError {
    fn from(e: ParseError) -> AppError {
        AppError::Parse(e)
    }
}

fn parse_u32(s: &str) -> Result<u32, ParseError> {
    if s.is_empty() {
        return Err(ParseError::Empty);
    }
    let mut acc: u32 = 0;
    for c in s.chars() {
        let d = match c.to_digit(10) {
            Some(d) => d,
            None => return Err(ParseError::BadChar(c)),
        };
        if acc > (u32::MAX - d) / 10 {
            return Err(ParseError::TooBig);
        }
        acc = acc * 10 + d;
    }
    Ok(acc)
}

fn parse_numbers(text: &str) -> Result<Vec<u32>, ParseError> {
    let mut nums = Vec::new();
    let mut token = String::new();
    for c in text.chars() {
        if c.is_whitespace() {
            if !token.is_empty() {
                nums.push(parse_u32(&token)?);
                token.clear();
            }
        } else {
            token.push(c);
        }
    }
    if !token.is_empty() {
        nums.push(parse_u32(&token)?);
    }
    Ok(nums)
}

fn sum_of_file(path: &str) -> Result<u32, AppError> {
    let text = fs::read_to_string(path)?;
    let nums = parse_numbers(&text)?;
    let mut total: u32 = 0;
    for n in &nums {
        total += *n;
    }
    Ok(total)
}

fn main() {
    match sum_of_file("scores.txt") {
        Ok(total) => println!("сумма: {total}"),
        Err(AppError::Io(e)) => eprintln!("не удалось прочитать scores.txt: {e}"),
        Err(AppError::Parse(ParseError::BadChar(c))) => {
            eprintln!("в файле не число: символ {c:?}");
        }
        Err(AppError::Parse(pe)) => eprintln!("файл повреждён: {pe:?}"),
    }
}
```

**Разбор по этажам.**

- **`parse_numbers`** — посимвольный сборщик слов, C-школа глав 5 и 9 на новых рельсах: накапливаем не-пробельные символы в `token` (метод `token.clear()` опустошает `String`, не отдавая буфер — R9), на каждом разделителе разбираем накопленное. Оператор `?` здесь пробрасывает `ParseError` **без конвертации** — тип ошибки функции тот же (`From` не нужен, как в §18.1.3). Пустой вход честно даёт `Ok(пустой вектор)`.
- **`sum_of_file`** — два `?` с **разными** типами ошибок, и оба вылетают как `AppError`: таможня §18.2.2 в деле. Сумма — цикл по ссылкам с разыменованием (§17.1.2); страж переполнения суммы для краткости опущен — числа в упражнении малы, а в debug-профиле переполнение всё равно не пройдёт молча (§14.2.2).
- **`main`** — граница: единственное место, где ошибки превращаются в сообщения для человека, в stderr (§18.1.2). Match-ем — по **вложенным** образцам: частый случай `BadChar` получил персональное сообщение с символом, остальной `Parse` — общее (образец `pe` связывает весь `ParseError`), `Io` печатается Display-ем ОС.

**Прогоны** (все проверены; файл создаётся в Git Bash `printf '10 25 7\n' > scores.txt` и т.п.):

| `scores.txt` | Вывод |
|---|---|
| `10 25 7` | `сумма: 42` |
| *(файла нет)* | `не удалось прочитать scores.txt: The system cannot find the file specified. (os error 2)` |
| `10 abc 5` | `в файле не число: символ 'a'` |
| `99999999999` | `файл повреждён: TooBig` |
| *(пустой файл)* | `сумма: 0` |

Обратите внимание на третий прогон: отказ произошёл на **третьем этаже** (`parse_u32` увидела `'a'`), поднялся через `parse_numbers` (`?`, без конвертации), через `sum_of_file` (`?`, конвертация `From<ParseError>`), и только `main` превратил его в текст — с сохранённой подробностью `'a'`. В C-версии из §10.1.2 `fscanf` вернул бы «преобразований: 1» — и ни слова о том, *что* встретилось.

А что с «файл прочитан, но это не числа, а картинка»? Второе русло `read_to_string` — не-UTF-8 — тоже накрыто: это `Err(io::Error)` с категорией `InvalidData` (R9), наш вариант `Io`.

**Когда panic, когда Result — свод правил** (R1, гл. 9.3):

| Ситуация | Инструмент | Почему |
|---|---|---|
| пример в книге, прототип, черновик | `unwrap`/`expect` | краткость важнее; `expect` — ещё и маркер «доделать» |
| тест (гл. 20) | паника | упавший тест и есть паника |
| «я знаю больше компилятора» | `expect("почему невозможно")` | инвариант доказан кодом рядом — задокументируйте его |
| нарушен инвариант/контракт вашего кода | `panic!` | это баг программиста; продолжать опасно |
| ожидаемый отказ внешнего мира: файл, ввод, разбор | `Result` | не баг; решает вызывающий |
| плохой вход пользователя | `Result` | «опечатка — не авария»: программа обязана пережить |

Лакмус: **чей это провал?** Программиста — паника (пусть будет громко и в тестах). Внешнего мира — `Result` (пусть решает граница).

**Как это было в C.** Сквозной пример против своего C-предка (§10.1.2):

| Точка | C (§10.1.2) | Rust (этот параграф) |
|---|---|---|
| открыть и прочитать | `fopen` + проверка NULL + `fgets`-цикл + `fclose` | `fs::read_to_string(path)?` |
| разобрать число | `fscanf("%d")`: «сколько преобразовано», без подробностей | `parse_u32` → `Err(BadChar('a'))` |
| донести отказ до main | код `-1` через каждый уровень руками | `?` + `From` — типы сходятся сами |
| закрыть файл при ошибке | `fclose` в каждой ветке (легко забыть) | `drop` при любом выходе (§15.1.1) |
| сообщить пользователю | `perror` где придётся | один `match` на границе, в stderr |
| забыть проверку | молча | `must_use` + типы не дадут |

**Типичные ошибки и подводные камни.**

- **`unwrap` в глубине.** Один `unwrap` внутри `parse_numbers` — и опечатка в файле валит программу паникой вместо сообщения. Глубина пробрасывает, граница решает.
- **Проглатывание.** `Err(_) => {}` — обработка, после которой ошибки не существует. Хуже C: там хотя бы `errno` оставалась. Если ветка пуста осознанно — напишите почему в комментарии.
- **Debug наружу.** `Error: Os { code: 2, ... }` из `main -> Result` — для утилит и отладки; людям — `match` с `{e}`. Выбирайте границу по аудитории.
- **Кто знает имя файла?** Заметьте: `io::Error` не хранит имя файла — «file not found», а какой файл? В C это решал аргумент `perror("scores.txt")` (§10.1.1); в Rust — тот, кто знает путь (наш `main` пишет `scores.txt` в сообщении сам). Пробрасывая ошибку далеко от места, где путь известен, — заверните путь в свой вариант ошибки.

**Источники:** R1 (гл. 9.2 — чтение файла как сквозной пример проброса; гл. 9.3 — полный свод «panic или Result»: прототипы/тесты, «больше информации, чем у компилятора», плохое состояние против ожидаемого отказа); R9 (std::fs::read_to_string — ошибки, включая не-UTF-8; String::clear; char::is_whitespace); R2 (error — общий рисунок: panic для невосстановимого, Result для остального); R6 (error-handling — упражнение той же формы: чтение + разбор с одним enum-ом ошибок). Сторона C: §5.2, §7.1, §10.1, §13.1.2 курса.

## Резюме главы

- **Два рода отказов.** Невосстановимые — баги: **паника** (`panic!`, индекс за границей, `unwrap` на пустом) с сообщением `файл:строка:колонка`, размоткой стека (`drop` всем живым значениям, как при обычном возврате), трассировкой по `RUST_BACKTRACE=1` и ненулевым кодом выхода. Восстановимые — жизнь: **`Result<T, E>`** — близнец `Option` с двумя руслами, `Ok(T)` и `Err(E)`, оба в одном типе. Паника ≠ UB: это громкий, определённый отказ — финальный ответ Rust главе 13.
- **Ошибку нельзя забыть.** `Result` помечен `#[must_use]` — проигнорированный результат шумит предупреждением (GCC молчал на забытый `fopen` даже с `-Wall -Wextra`); использовать `Result<File, _>` «как файл» не даст система типов — сначала `match`. `unwrap`/`expect` — осознанное превращение `Err` в панику; `expect` формулирует, почему автор считал отказ невозможным.
- **`?` — проброс в один символ** (по R7: `Ok(v)` → `v`, `Err(e)` → `return Err(From::from(e))`; для `Option` — `Some`/`None`). Лесенка «проверь — освободи — верни код» из главы 10 стала счастливым путём с закорючками; освобождение при раннем выходе делает `drop`. Работает только в функциях с `Result`/`Option` (E0277), включая `main -> Result<(), E>`: `Err` наверху — отладочная печать и код 1.
- **Свой тип ошибки — это `enum`** с данными в вариантах: конкретно (`BadChar('x')`, не строка), сравнимо match-ем, с подробностями внутри. Критерии R2: один тип на функцию, варианты вместо строк, данные в вариантах, сочетаемость. `errno`-коды и out-параметры тома I складываются в одну честную сигнатуру `-> Result<u32, ParseError>`.
- **`From` сшивает источники ошибок**: `enum AppError { Io(io::Error), Parse(ParseError) }` плюс `impl From<...> for AppError` на каждый источник — и `?` конвертирует на вылете сам (это его «таможня»; написано в тексте E0277 прямым текстом). Рецепт `impl From<Чужая> for Моя` — первый трейт курса; что такое трейты вообще — глава 19. Правило: заворачивайте исходную ошибку целиком, не пересказ.
- **Граница решает.** Внутри логики — `?`; на границе (main, интерфейс) — один `match`, где ошибка становится сообщением для человека в stderr (`eprintln!`), кодом выхода или планом Б. Свод «когда panic, когда Result»: прототипы и тесты — паникующие методы честны; нарушенный инвариант — `panic!` (баг!); ожидаемый отказ мира и плохой ввод — только `Result`. Лакмус: чей провал — программиста или внешнего мира?

**Дальше.** В этой главе вы дважды упёрлись в слово «трейт»: `From` писали по рецепту, а красивый вывод своих ошибок (`{e}` вместо `{pe:?}`) отложили. Глава 19 выдаёт документацию на сам механизм: **трейты** — что общего у `From`, `Debug`, `Clone` и `Copy` и как объявлять свои; **обобщённые типы** — почему `Result<T, E>`, `Vec<T>` и `Option<T>` пишутся один раз для всех типов (и что мы теряли с `void *` в §6.3); **времена жизни** — аннотации, которыми вы объясните компилятору ссылки в структурах. После неё «рецепты» этой главы станут прозрачными — включая то, почему `parse_u32` в стандартной библиотеке на самом деле не нужен: там есть обобщённый `parse`, и в главе 19 вы поймёте его сигнатуру.
