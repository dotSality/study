# Глава 19. Обобщённые типы, трейты, времена жизни

## Введение

Всю дорогу по тому II вы пользовались механизмами, устройство которых мы откладывали «на главу 19». Пора платить по векселям — их накопилось три пачки:

- **Записи с угловыми скобками.** `Vec<T>`, `Option<T>`, `Result<T, E>`, `HashMap<K, V>` — мы читали их как «шаблон, куда подставили тип» (§16.2.3, §17.1.1) и жили с этой интуицией три главы. Что это за «типы с дыркой под тип», кто и когда заполняет дырку и сколько это стоит в рантайме — первая часть главы. Спойлер из тома I: это честный ответ на боль главы 9, где наш вектор умел хранить только `int`, а обобщать его пришлось бы через `void *` с потерей всякой проверки типов (§6.3.2).
- **Слово «трейт».** Оно уже трижды прорывалось в текст: «свойство типа» у `Copy` (§15.1.3), «декларация умения» у `derive` (§16.1.3), «пользуемся как рецептом» у `From` (§18.2.2). Вторая часть выдаёт документацию на сам механизм: трейт — это контракт поведения, который тип обязуется выполнять; вы научитесь объявлять свои трейты, реализовывать чужие и требовать их от параметров типа. Здесь же закроются старые долги: ручной `Display` (красивое `{e}` для ваших ошибок — §18.1.2), сравнение enum-ов оператором `==` (§16.2.1), требования `HashMap` к ключам (§17.2.1) и сигнатура встроенного `parse`, после которой наш рукописный `parse_u32` из §18.2.1 можно отправлять на пенсию.
- **«Не хватает параметра времени жизни».** E0106 дважды останавливал нас без объяснений: на функции, возвращающей ссылку (§15.2.3), и на `&str`-поле структуры (§16.1.1). Третья часть — про **времена жизни** (lifetimes): язык, на котором вы объясняете компилятору, *чью* память одалживает ссылка и как долго. Это прямое продолжение borrow checker-а из главы 15 и последний ответ висячим указателям из §13.1 — часть помечена в программе курса как «повышенная сложность».

Сквозная идея главы: **абстракция без потери контроля**. В C обобщение всегда что-то отнимало: макрос — проверку типов и предсказуемость вычислений (§11.1.3), `void *` — проверку типов и скорость (§6.3.2), копипаста — единственность истины. Rust строит обобщения так, что проверки происходят *раньше* (при компиляции определения), а машинный код остаётся таким же, как у написанного руками под конкретный тип.

После изучения главы вы будете уметь:

- писать обобщённые функции, структуры и перечисления с параметрами типа и читать ошибки инстанцирования (E0369, E0308, E0599, E0277, E0282);
- объяснять мономорфизацию: во что компилятор разворачивает `largest(&numbers)` и `largest(&letters)` и почему это не медленнее ручного кода;
- объявлять трейты (в том числе с методами по умолчанию), реализовывать их для своих типов и требовать ограничениями `T: Трейт`, `impl Трейт` и `where`;
- читать `#[derive(...)]` как «impl, написанный компилятором», а `impl From<X> for Y` из главы 18 — как обычную реализацию обобщённого трейта;
- реализовывать `Display` и `FromStr` руками: печатать свои типы через `{}` и разбирать их из строк через `.parse()`;
- расставлять аннотации времён жизни в функциях и структурах, применять три правила умолчаний (elision) и объяснять, почему `'static` — не «чит-код».

## Часть 19.1. Обобщённые типы

### §19.1.1. От дублирования к параметру типа

**Определения.** **Обобщённая функция** (generic function) — функция, у которой помимо обычных параметров-значений есть **параметры типа** (type parameters): имена-заглушки, вместо которых при каждом использовании подставляется конкретный тип (R1, гл. 10.1). Параметры типа объявляются в угловых скобках после имени функции и по соглашению называются одной заглавной буквой, начиная с `T` (type):

```rust
fn largest<T>(list: &[T]) -> &T
```

Читается: «для любого типа `T`: беру срез элементов `T`, возвращаю ссылку на `T`». Обычный параметр обобщает *значения* — одна функция вместо `print_1()`, `print_2()`, `print_3()`; параметр типа обобщает *типы* — одна функция вместо `largest_i32`, `largest_char`, `largest_str`. Это тот же приём вынесения повторов, что и в главе 4, только этажом выше (R1, гл. 10).

Сразу главное правило, отличающее Rust и от макросов C, и от шаблонов многих других языков: **тело обобщённой функции проверяется в точке определения, а не в точке использования**. Внутри тела с значением типа `T` можно делать только то, что разрешено *любому* типу, — переместить, взять ссылку, вернуть. Всё остальное — сравнивать, складывать, печатать — надо явно разрешить **ограничением** (trait bound): записью `T: Умение` в угловых скобках. Слово после двоеточия — имя трейта; что это такое, разберёт часть 19.2, а пока нам хватит одного: **`PartialOrd`** — «значения типа можно сравнивать операторами `<`, `<=`, `>`, `>=`» (R1, гл. 10.1; R9).

**Интуиция.** Параметр типа — «дырка» в сигнатуре и теле:

```
   fn largest<T>(list: &[T]) -> &T          вызовы заполняют дырку:
                 │
        ┌────────┴────────┐                 largest(&numbers)  → T = i32
        │  дырка под тип  │                 largest(&letters)  → T = char
        └─────────────────┘                 largest(&words)    → T = &str
   компилятор проверяет тело ОДИН РАЗ,
   до всяких вызовов: «а любой ли T        подстановку типа в дырку компилятор
   такое умеет?» — и требует ограничений    выводит из аргументов сам (как вывод
   на всё сверх перемещения и ссылок        типа у let, §14.2.1)
```

**Разобранные примеры.**

*Пример 1 — боль, знакомая по тому I.* Наибольший элемент — в векторе чисел и в векторе символов. Без обобщений — две функции, различающиеся **только типами** в сигнатуре. Файл `largest_dup.rs`:

```rust
fn largest_i32(list: &[i32]) -> &i32 {
    let mut largest = &list[0];
    for item in list {
        if item > largest {
            largest = item;
        }
    }
    largest
}

fn largest_char(list: &[char]) -> &char {
    let mut largest = &list[0];
    for item in list {
        if item > largest {
            largest = item;
        }
    }
    largest
}

fn main() {
    let numbers = vec![34, 50, 25, 100, 65];
    println!("наибольшее число: {}", largest_i32(&numbers));

    let letters = vec!['y', 'm', 'a', 'q'];
    println!("наибольшая буква: {}", largest_char(&letters));
}
```

```
наибольшее число: 100
наибольшая буква: y
```

Механика тела — глава 15 в миниатюре: `largest` — разделяемая ссылка на текущий максимум (владение не трогаем), `for item in list` выдаёт ссылки на элементы (обход среза по ссылкам — §17.1.2), сравнение `item > largest` сравнивает значения за ссылками. В томе I мы бы скопировали такую функцию под каждый тип — и при правке алгоритма чинили бы все копии (наш вектор из §9.1.1 так и остался «только для int»).

*Пример 2 — первая попытка обобщить.* ⛔ Механическая замена `i32` на `T` не собирается. Файл `largest_nobound.rs` (проверено):

```rust

    let mut largest = &list[0];

        }
    }

fn main() {

}
```

```
 --> largest_nobound.rs:4:17
  |
4 |         if item > largest {
  |            ---- ^ ------- &T
  |            |

help: consider restricting type parameter `T` with trait `PartialOrd`
  |
```

Компилятор рассуждает за нас: «`T` — любой тип; а *любой* ли тип умеет `>`? Нет — значит, запрещаю». Заметьте: вызов с `i32` (который сравнивать умеет) не спасает — ошибка относится к **определению**, а не к вызову. Мы проверили и жёстче: файл `largest_nobound_nocall.rs`, где `largest` вообще ни разу не вызвана, даёт **ту же ошибку** — тело обязано быть корректным для любого допустимого `T` ещё до первого использования. Запомните контраст с макросами §11.1.3: там `SQUARE_BAD` лежал в коде молча и взрывался только в точке подстановки.

*Пример 3 — ограничение.* Применяем `help`-подсказку (короткая форма `T: PartialOrd` легальна: имя из прелюдии, как `Ok`/`Err` в §18.1.2). Файл `largest_generic.rs`:

```rust
fn largest<T: PartialOrd>(list: &[T]) -> &T {
    let mut largest = &list[0];
    for item in list {
        if item > largest {
            largest = item;
        }
    }
    largest
}

fn main() {
    let numbers = vec![34, 50, 25, 100, 65];
    println!("наибольшее число: {}", largest(&numbers));

    let letters = vec!['y', 'm', 'a', 'q'];
    println!("наибольшая буква: {}", largest(&letters));

    let words = vec!["pear", "apple", "watermelon"];
    println!("наибольшая строка: {}", largest(&words));
}
```

```
наибольшее число: 100
наибольшая буква: y
наибольшая строка: watermelon
```

Одно тело — три типа. Ограничение `T: PartialOrd` — это ещё и **документация**: сигнатура честно говорит вызывающему «мне подойдёт любой тип, который умеет сравниваться». Для `char` порядок задан скалярными значениями Unicode (§17.1.4), у `&str` `PartialOrd` тоже есть (R9) — на нашем прогоне порядок этих трёх слов совпал с алфавитным; тонкости сравнения строк (регистр, не-ASCII алфавиты) оставим до практики сортировок.

**Как это было в C.** Три способа «обобщить» из тома I и их цена:

| Способ | Пример из курса | Проверка типов | Цена |
|---|---|---|---|
| копипаста функции | `largest_i32` / `largest_char` | есть, в каждой копии | N копий истины: правки размножаются |
| макрос препроцессора | `SQUARE`, `MAX` (§11.1.3) | нет — подстановка текста; ошибка вылезет в точке использования | двойное вычисление аргументов, ловушки скобок |
| `void *` + казты | `qsort` с компаратором (§6.3.2) | нет — компилятор верит казту | вызов через указатель в рантайме; ошибка типа = UB |
| **Rust: параметр типа** | `largest<T: PartialOrd>` | **в точке определения** | ноль в рантайме (§19.1.3) |

**Связь с практикой.** Правило вынесения то же, что для функций в главе 4: **два одинаковых тела, различающихся только типами, — сигнал сделать параметр типа**. Стандартная библиотека так и устроена: один `Vec<T>` вместо тысячи векторов, один `Result<T, E>` на все пары «успех/ошибка». Но не обобщайте впрок: пока тип один, конкретная функция проще читается и лучше подсказывает ошибки; параметр типа заводят, когда второй тип реально появился (как у нас: сначала жили `largest_i32`, потом понадобился `char`).

**Типичные ошибки и подводные камни.**

- **Обобщить и забыть разрешение.** Любая операция над `T` сверх перемещения и ссылок требует ограничения — E0369/E0277 с подсказкой, какой трейт добавить. Читайте `help`: компилятор почти всегда называет нужное имя.
- **Чинить вызов, а не определение.** E0369 указывает в тело функции — бесполезно менять аргументы вызова: ограничение ставится в угловые скобки *определения*.
- **`T` в теле, но не в скобках.** Использовали `T` в сигнатуре, не объявив `<T>`, — E0425 «cannot find type `T` in this scope» с подсказкой «you might be missing a type parameter» (проверено): угловые скобки после имени функции — это *объявление* параметра, без него `T` — просто неизвестное имя.
- **Обобщение впрок.** Параметр типа с одним-единственным инстанцированием — шум в сигнатуре. Дублирование должно случиться, прежде чем вы его вынесете.

**Источники:** R1 (гл. 10 — вынесение дублирования, гл. 10.1 — синтаксис параметров типа, E0369 и ограничение PartialOrd); R2 (generics — параметризация типов); R6 (generics/generic-functions); R7 (items/generics — объявление обобщённых параметров; арбитр синтаксиса). Сторона C: §6.3.2, §9.1.1, §11.1.3 курса.

### §19.1.2. Обобщённые структуры, перечисления, методы

**Определения.** Параметры типа доступны не только функциям (R1, гл. 10.1):

- **обобщённая структура**: `struct Point<T> { x: T, y: T }` — оба поля *одного* типа `T`;
- **несколько параметров**: `struct Point<T, U> { x: T, y: U }` — поля могут различаться;
- **обобщённое перечисление**: вы знаете их с главы 16 — `Option<T>` (один параметр) и `Result<T, E>` (два) объявлены в стандартной библиотеке ровно этим синтаксисом;
- **методы обобщённого типа**: блок пишется `impl<T> Point<T> { ... }` — первое `<T>` объявляет параметр для всего блока; а блок `impl Point<f64> { ... }` реализует методы **только** для конкретного инстанцирования.

Конкретный тип, полученный подстановкой (`Point<i32>`, `Vec<String>`, `Result<u32, ParseError>`), — полноценное имя типа: годится в аннотации, в поля других структур, в сигнатуры.

Когда вывод типов не справляется сам, тип называют явно — двумя способами, знакомыми по курсу порознь: аннотацией `let v: Vec<i32> = Vec::new();` или **turbofish** («турборыба» — жаргонное имя записи `::<>`): `Vec::<i32>::new()`. Вторую форму вы видели в §14.2.2 у `std::mem::size_of::<T>()` с обещанием «привыкнем в главе 19» — вексель погашен: угловые скобки после `::` передают *тип-аргумент* так же, как круглые передают значения.

**Интуиция.**

```
   struct Point<T> { x: T, y: T }      ← форма для литья с дыркой под тип

   Point<i32>   ┌─ x: i32 ─┬─ y: i32 ─┐   ← подставили i32: конкретный тип
   Point<f64>   ┌─ x: f64 ─┬─ y: f64 ─┐   ← подставили f64: ДРУГОЙ конкретный тип
                                            (несовместимый с первым — §16.1.1,
                                             newtype-урок работает и здесь)
   знакомые формы той же природы:
   Vec<T>, Option<T>, Result<T, E>, HashMap<K, V>
```

**Разобранные примеры.**

*Пример 1 — структура и два вида impl.* Файл `point_generic.rs`:

```rust
struct Point<T> {
    x: T,
    y: T,
}

impl<T> Point<T> {
    fn x(&self) -> &T {
        &self.x
    }
}

impl Point<f64> {
    fn distance_from_origin(&self) -> f64 {
        (self.x.powi(2) + self.y.powi(2)).sqrt()
    }
}

fn main() {
    let integer = Point { x: 5, y: 10 };
    let float = Point { x: 1.5, y: 2.0 };

    println!("integer.x = {}", integer.x());
    println!("float.x = {}", float.x());
    println!("расстояние до начала координат: {}", float.distance_from_origin());
}
```

```
integer.x = 5
float.x = 1.5
расстояние до начала координат: 2.5
```

Разбор. Тип из литералов компилятор вывел сам: `integer` — `Point<i32>`, `float` — `Point<f64>` (вывод типов из §14.2.1 работает и для параметров типа). Метод `x()` объявлен в `impl<T>` — он есть у *всех* `Point<T>`. А `distance_from_origin` живёт в `impl Point<f64>` — методы `powi` (целая степень) и `sqrt` (корень) есть только у чисел с плавающей точкой (R9), поэтому и метод существует только для `f64`-точек. ⛔ Вызов его на `Point<i32>` — ошибка, и заметьте, насколько она точна (файл `point_f64_method_bad.rs`, проверено):

```
  --> point_f64_method_bad.rs:14:28
```

Знакомый E0599 (§15.3.2) с новой деталью в `note`: метод *существует*, но у другого инстанцирования.

По полю `x` компилятор решил «`T` = целое», и поле `y` обязано подчиниться: параметр один — тип один. Нужны разные типы полей — нужен второй параметр. Файл `point_two_params.rs`:

```rust
struct Point<T, U> {
    x: T,
    y: U,
}

fn main() {
    let both_int = Point { x: 5, y: 10 };
    let mixed = Point { x: 5, y: 4.0 };
    println!("both_int: ({}, {})", both_int.x, both_int.y);
    println!("mixed: ({}, {})", mixed.x, mixed.y);
}
```

```
both_int: (5, 10)
mixed: (5, 4)
```

`Point<T, U>` не запрещает типам совпасть: `both_int` — это `Point<i32, i32>`. Больше двух параметров — та же механика (`HashMap<K, V>` вы уже видели), но каждый параметр должен быть *нужен*: лишний параметр — лишняя сложность у каждого пользователя типа.


**Как это было в C.** Контейнер «под любой тип» в томе I имел два пути, оба плохие: скопировать структуру и все функции под новый тип (наш `Vec` из §9.1.1 — «только int»; для строк пришлось бы писать второй) или хранить `void *` — теряя проверку типов при каждом казте и добавляя косвенность (§6.3.2). Наша хеш-таблица §9.3.1 выбрала первый путь: типы ключа и значения зашиты жёстко, под другие — переписывать. `Point<T>` и `Vec<T>` — та же экономия текста, что копипаста, но истина одна, и проверка типов не выключается ни на строчку.

**Связь с практикой.** Читайте стандартную библиотеку как коллекцию образцов: `Option<T>` — «одна дырка», `Result<T, E>` — «две дырки с разными ролями», `HashMap<K, V>` — «дырки с разными обязанностями» (ключ обязан уметь больше — §19.2.3). Свои обобщённые структуры у вас появятся быстро: пара «значение + метка», результат замера «значение + единица измерения», узел «данные + что-то ещё» — везде, где форма данных не зависит от типа начинки.

**Типичные ошибки и подводные камни.**

- **Ждать от `Point<T>` разных типов полей.** Один параметр — один тип на все его вхождения; E0308 в примере 2. Разные поля — разные параметры.
- **`Point<i32>` и `Point<f64>` — «почти одно и то же».** Нет: это два несовместимых типа, как `Color` и `Point3` в §16.1.1. Функция от `Point<i32>` не примет `Point<f64>` — и это защита, а не неудобство.
- **Забыть `<T>` после `impl`.** `impl Point<T>` без объявления `impl<T>` — E0425 «cannot find type `T`» с подсказкой `impl<T> Point<T>` (проверено): в позиции после `impl` компилятор ждёт объявленные имена. Мнемоника: первое `<T>` — «объявляю переменную типа», второе — «использую».
- **Городить параметры впрок.** `Config<T, U, V>` там, где конфиг всегда один, — налог на каждого читателя. Правило то же, что у функций: обобщайте по факту второго типа.

### §19.1.3. Мономорфизация и ограничения

**Определения.** **Мономорфизация** (monomorphization) — превращение обобщённого кода в конкретный на этапе компиляции: для каждого набора типов, которым реально пользуется программа, компилятор изготавливает отдельный экземпляр функции или типа с подставленными типами (R1, гл. 10.1). Обобщённость **полностью исчезает до рантайма** — выполняется тот же машинный код, как если бы вы написали `largest_i32` и `largest_char` руками; R1 формулирует прямо: «за использование обобщённых типов вы не платите ничем во время выполнения».

Вторая половина параграфа — полный синтаксис **ограничений**:

- одно умение: `fn f<T: PartialOrd>(...)`;
- несколько: `fn f<T: Debug + PartialOrd>(...)` — плюс складывает *требования* (тип обязан уметь всё перечисленное);
- у разных параметров — свои списки: `fn f<T: Debug, U: Clone>(...)`;
- длинные списки выносятся в **`where`** — то же самое, но после сигнатуры, чтобы она читалась (R1, гл. 10.2):

```rust
fn stats<T>(items: &[T]) -> (&T, &T)
where
    T: Debug + PartialOrd,
```

**Интуиция.** Мономорфизация — «копипаста, которую делает компилятор»:

```
   исходник: ОДНО определение            бинарник: по экземпляру НА ТИП
   fn largest<T: PartialOrd>(...)   ┌──► largest::<i32>  (машинный код для i32)
                                    ├──► largest::<char> (машинный код для char)
   вызовы:                          └──► largest::<&str> (машинный код для &str)
   largest(&numbers)   T=i32
   largest(&letters)   T=char            вызов — обычный прямой вызов функции:
   largest(&words)     T=&str            никакого «выяснения типа» в рантайме
```

Сравните с `qsort` из §6.3.2: там одна универсальная функция жила в бинарнике в одном экземпляре, но платила за это вызовом компаратора через указатель на каждое сравнение — решение «какой код выполнить» принималось миллион раз в рантайме. Здесь решение принято один раз при компиляции. Честная цена мономорфизации тоже видна на схеме: экземпляров в бинарнике — по числу использованных типов.

**Разобранные примеры.**

*Пример 1 — во что разворачивается Option.* Программа с двумя инстанцированиями (R1, гл. 10.1 — разбираем её судьбу):

```rust
let integer = Some(5);
let float = Some(5.0);
```

Компилятор изготовит два конкретных перечисления — условно `Option_i32` и `Option_f64` — и подменит использования (имена условные, R1 приводит их как мысленную модель; настоящие внутренние имена видны разве что в отладчике). Обе «копии» — обычные enum-ы главы 16 со своими размерами и разметкой памяти: измеренные в §16.2.3 8 байт у `Option<i32>` — это размер именно экземпляра-для-i32.

*Пример 2 — несколько ограничений и where.* Обобщённая печать и обобщённый поиск краёв в одном файле `show_all.rs`:

```rust
use std::fmt::Debug;

fn show_all<T: Debug>(label: &str, items: &[T]) {
    for item in items {
        println!("{label}: {item:?}");
    }
}

fn min_max<T>(items: &[T]) -> (&T, &T)
where
    T: PartialOrd,
{
    let mut min = &items[0];
    let mut max = &items[0];
    for item in items {
        if item < min {
            min = item;
        }
        if item > max {
            max = item;
        }
    }
    (min, max)
}

fn main() {
    let temps = vec![19, 24, 21];
    let names = vec!["Bob", "Alice"];
    show_all("temps", &temps);
    show_all("names", &names);
    let (lo, hi) = min_max(&temps);
    println!("temps: от {lo} до {hi}");
    let (first, last) = min_max(&names);
    println!("names: от {first} до {last}");
}
```

```
temps: 19
temps: 24
temps: 21
names: "Bob"
names: "Alice"
temps: от 19 до 24
names: от Alice до Bob
```

`show_all` требует `Debug` — то самое умение печататься через `{:?}`, которое мы до сих пор *выдавали* типам derive-ом (§16.1.3), а теперь впервые *требуем* от чужого типа. `use std::fmt::Debug` — привычный «дай короткое имя» (§18.1.2; `PartialOrd` из прелюдии в use не нуждается). `min_max` возвращает кортеж ссылок: отдать `(T, T)` по значению мы не можем — для этого понадобилось бы право копировать (`T: Copy`) или клонировать, а наши ограничения такого не обещают. Ограничения работают в обе стороны: они и разрешают операции в теле, и *исчерпывают* их список.

**Как это было в C.** Итоговая таблица обобщений — теперь с ценой в рантайме:

| | Макрос (§11.1.3) | `void *` (§6.3.2) | Параметр типа |
|---|---|---|---|
| когда проверка типов | никогда (текст); ошибки — в точке подстановки | никогда (казты глушат) | при компиляции: тело — в определении, требования — в вызове |
| цена в рантайме | нет (но двойное вычисление аргументов) | косвенный вызов, потеря оптимизаций | **нет**: мономорфизация даёт прямой код |
| цена в бинарнике | копия текста в каждой точке | один экземпляр | экземпляр на каждый использованный тип |
| сообщение об ошибке | лавина в чужих строках | нет — молчаливое UB | E0277/E0369 с именем недостающего умения |

**Связь с практикой.** Ограничения — язык контрактов между автором обобщённого кода и его пользователем: автор перечисляет *минимум* умений, пользователь подключает типы, которые ими обладают. Правило минимальности: требуйте ровно то, чем пользуетесь, — `min_max` не просит `Debug`, потому что ничего не печатает; завтра им сможет пользоваться тип без Debug. И наоборот: если тело выросло и понадобилось новое умение — добавляйте его в ограничения, а не обходите (обходов, к счастью, и нет).

**Типичные ошибки и подводные камни.**

- **«Обобщённое = медленное».** Интуиция из мира `void *` и динамических языков. В Rust после мономорфизации код неотличим от рукописного под тип; цена — размер бинарника и время компиляции, не скорость.
- **Дописать операцию и забыть требование.** Добавили в тело `println!("{item:?}")` — получите E0277 у *определения*: список ограничений — исчерпывающий, компилятор не «догадается».
- **Чинить E0277 вызова правкой функции.** Ошибка «trait not implemented for `Reading`» чинится у типа `Reading` (derive или ручная реализация — §19.2.3), а не ослаблением ограничений: они и так минимальны, если вы следовали правилу выше.
- **Дерайвить только вершину.** `derive(PartialOrd)` без `PartialEq` не собирается — читайте ноту про супертрейт и добавляйте базу.

## Часть 19.2. Трейты

### §19.2.1. Трейт: контракт поведения

**Определения.** **Трейт** (trait) — описание абстрактного интерфейса, который типы могут реализовывать (R7 — арбитр; R1, гл. 10.2). Объявление трейта перечисляет **сигнатуры методов** — что тип обязан уметь; **реализация** (`impl Трейт for Тип`) — блок, где конкретный тип выполняет контракт, предоставляя тела этих методов:

```rust
trait Summary {
    fn summarize(&self) -> String;   // сигнатура с `;` — тела нет, это требование
}
```

Правила (R1, гл. 10.2; R7):

- реализация обязана определить **все** методы трейта, не имеющие тел по умолчанию, — иначе E0046;
- метод трейта может получить **тело по умолчанию** (default implementation) прямо в объявлении трейта — реализация вправе принять его молча или **переопределить**; тело по умолчанию может вызывать другие методы того же трейта, даже ещё не реализованные;
- в объявлении трейта доступен неявный тип **`Self`** — «тот тип, который реализует этот интерфейс» (R7); `&self` в сигнатурах — тот же сокращённый приёмник, что в обычных `impl` (§16.1.2);
- **правило сироты** (orphan rule, часть свойства когерентности — coherence): реализация `impl Трейт for Тип` разрешена, только если трейт **или** тип объявлен в вашем crate (R1, гл. 10.2; R7 — items/implementations). Реализовать свой трейт для `Vec<i32>` — можно; чужой `Display` для своего типа — можно; чужой `Display` для чужого `Vec<i32>` — нельзя (E0117): иначе два crate могли бы дать одному типу две разные реализации, и компилятор не знал бы, чью выбрать.

**Интуиция.** Трейт — колонка в таблице умений, которую компилятор сверяет на каждом вызове:

```
                 │ Summary │ Debug │ Clone │ PartialOrd │
   ──────────────┼─────────┼───────┼───────┼────────────┤
   Article       │   ✓     │       │       │            │   ✓ = есть impl
   Comment       │   ✓     │       │       │            │   (свой или derive)
   i32           │         │   ✓   │   ✓   │     ✓      │
   String        │         │   ✓   │   ✓   │     ✓      │
   ──────────────┴─────────┴───────┴───────┴────────────┘
   item.summarize() → компилятор смотрит в строку типа item:
   галочка в колонке Summary есть? нет — ошибка ДО запуска
```

В C ближайший родственник — таблица указателей на функции, но её никто не проверял: `qsort` верил, что компаратор согласован с типом массива (§6.3.2). Здесь таблицу ведёт компилятор.

**Разобранные примеры.**

*Пример 1 — контракт и две реализации.* Лента новостного сайта: статьи и комментарии показываются в одном списке, каждому нужна краткая сводка. Файл `summary.rs`:

```rust
trait Summary {
    fn summarize(&self) -> String;
}

struct Article {
    headline: String,
    author: String,
}

struct Comment {
    username: String,
    text: String,
}

impl Summary for Article {
    fn summarize(&self) -> String {
        format!("{} — {}", self.headline, self.author)
    }
}

impl Summary for Comment {
    fn summarize(&self) -> String {
        format!("@{}: {}", self.username, self.text)
    }
}

struct Divider;

impl Summary for Divider {
    fn summarize(&self) -> String {
        String::from("----------")
    }
}

fn main() {
    let article = Article {
        headline: String::from("Rust 1.97 вышел"),
        author: String::from("редакция"),
    };
    let comment = Comment {
        username: String::from("max"),
        text: String::from("наконец-то!"),
    };
    println!("{}", article.summarize());
    println!("{}", Divider.summarize());
    println!("{}", comment.summarize());
}
```

```
Rust 1.97 вышел — редакция
----------
@max: наконец-то!
```

Синтаксис `impl Summary for Article` читается «выполняю контракт Summary для типа Article» — и это ровно тот заголовок, который вы писали в §18.2.2 по рецепту: `impl From<io::Error> for AppError` — «выполняю контракт From (с типом-аргументом io::Error) для типа AppError». Рецепт стал прозрачным: `From` — обычный трейт, просто обобщённый (к нему вернёмся в §19.2.3). Вызов `article.summarize()` внешне неотличим от собственных методов главы 16 — методы из `impl Article` и из `impl Summary for Article` живут рядом и зовутся одинаково. И обратите внимание на `Divider`: **единичная структура** из §16.1.1 нашла обещанное там применение — данных у типа нет, а поведение есть; выражение `Divider` без скобок — её единственное значение (§16.1.1).

⛔ А что, если реализовать контракт наполовину? Файл `summary_missing_bad.rs` — пустой `impl Summary for Article {}` (проверено):

```
error[E0046]: not all trait items implemented, missing: `summarize`
 --> summary_missing_bad.rs:9:1
  |
2 |     fn summarize(&self) -> String;
  |     ------------------------------ `summarize` from trait
...
9 | impl Summary for Article {}
  | ^^^^^^^^^^^^^^^^^^^^^^^^ missing `summarize` in implementation
```

Контракт — весь или никакого: половинных реализаций не бывает.

*Пример 2 — тело по умолчанию.* Дадим трейту запасной вариант сводки, а обязательным оставим только «кто источник». Файл `summary_default.rs`:

```rust
trait Summary {
    fn source_name(&self) -> String;

    fn summarize(&self) -> String {
        format!("(материал от {})", self.source_name())
    }
}
```

Реализации (`Article` определяет только `source_name`; `Comment` — и `source_name`, и свой `summarize`) — и прогон:

```
(материал от редакция)
@max: наконец-то!
(заголовок: Rust 1.97 вышел)
```

`Article` получил `summarize` бесплатно — тело по умолчанию вызвало его же `source_name`: контракт «умолчание зовёт обязательное» работает, потому что к моменту вызова *какая-то* реализация `source_name` гарантированно есть (R1, гл. 10.2). `Comment` умолчание перекрыл. Это тот же принцип, что параметры по умолчанию в библиотеках C, — но без «магических NULL» и на уровне типа.

*Пример 3 — правило сироты.* ⛔ Попробуем научить чужой `Vec<i32>` чужому `Display`. Файл `orphan_bad.rs` (проверено):

```rust
use std::fmt;

impl fmt::Display for Vec<i32> {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "вектор из {} элементов", self.len())
    }
}
```

```
error[E0117]: only traits defined in the current crate can be implemented for types defined outside of the crate
 --> orphan_bad.rs:3:1
  |
3 | impl fmt::Display for Vec<i32> {
  | ^^^^^^^^^^^^^^^^^^^^^^--------
  |                       |
  |                       `Vec` is not defined in the current crate
  |
  = note: impl doesn't have any local type before any uncovered type parameters
  = note: for more information see https://doc.rust-lang.org/reference/items/implementations.html#orphan-rules
  = note: define and implement a trait or new type instead
```

И трейт, и тип — из стандартной библиотеки; «сирота» (реализация без местного родителя) отклонена. Обратите внимание на последнюю ноту: компилятор сам подсказывает выход — **newtype**: заверните `Vec<i32>` в свою кортежную структуру (`struct Ints(Vec<i32>);` — приём §16.1.2) и реализуйте `Display` для неё: обёртка — ваш тип, правило соблюдено.

**Как это было в C.** Передача поведения в §6.3.2 — `qsort` и компаратор:

| | Указатель на функцию (§6.3.2) | Трейт |
|---|---|---|
| что передаётся | адрес одной функции, в рантайме | «галочка умения» у типа, при компиляции |
| привязка к типу | никакой: компаратор для `int` можно отдать массиву строк — компилятор промолчит (казты в `void *`) | `impl` привязан к типу; вызов метода у типа без impl — ошибка компиляции |
| состав контракта | одна функция | набор методов, часть — с телами по умолчанию |
| цена вызова | косвенный вызов по указателю | после мономорфизации — прямой вызов (§19.1.3) |
| кто следит за полнотой | никто (забыли случай — UB или мусор) | E0046: контракт весь или никакого |

**Связь с практикой.** Трейт проектируют как минимальный контракт: то, без чего обобщённый код не сможет работать, — и ничего сверх. Хороший ориентир — стандартная библиотека: `Summary`-подобные «сводки» (Display, Debug), «умения-пометки» (Copy), «преобразования» (From). Отдельный выигрыш — методы по умолчанию: добавив в трейт новый метод с телом, вы не ломаете существующие реализации. А правило сироты — это то, что делает чтение чужого кода предсказуемым: реализация `Display` для вашего типа может лежать только в вашем crate — искать её больше негде.

**Типичные ошибки и подводные камни.**

- **Реализовать не всё.** E0046 перечисляет недостающее поимённо — читайте список, а не гадайте.
- **Чужой трейт для чужого типа.** E0117; выход — newtype-обёртка (подсказан самим компилятором). Это не бюрократия: без правила две библиотеки могли бы молча раздать одному типу разные реализации.
- **Трейт-«свалка».** Десять методов в одном контракте — значит, каждая реализация обязана уметь все десять. Дробите: лучше три трейта по делу, чем один «универсальный».
- **Дублировать умолчание.** Если реализация переписывает тело по умолчанию без изменений — удалите её: пусть работает умолчание, меньше копий истины.

**Источники:** R7 (items/traits — определение трейта, ассоциированные элементы, Self; items/implementations — trait implementation, правило сироты — арбитр); R1 (гл. 10.2 — синтаксис trait/impl, тела по умолчанию и вызов обязательных методов из умолчаний, формулировка orphan rule и её мотивация); R2 (trait — реализация трейтов для структур); R6 (generics/generic-traits). Сторона C: §6.3.2 курса.

### §19.2.2. Трейты как требования: bounds и impl Trait

**Определения.** Ограничение `T: Трейт` из §19.1 — основной способ *потребовать* умение. У него есть сокращённая форма для параметров: **`impl Трейт`** в позиции типа параметра — «любой тип, реализующий трейт» (R1, гл. 10.2):

```rust
fn notify(item: &impl Summary)              // сахар
fn notify_generic<T: Summary>(item: &T)     // полная форма — то же самое
```

Форма с `impl` короче; полная форма выразительнее: два параметра `item1: &impl Summary, item2: &impl Summary` могут быть *разными* типами, а `<T: Summary>(item1: &T, item2: &T)` требует один и тот же (R1).

`impl Трейт` работает и в **позиции результата**: `fn make() -> impl Summary` — «верну какой-то тип, умеющий Summary, а какой — не скажу». Ограничение: за таким результатом должен стоять **один** конкретный тип — вернуть «то `Article`, то `Comment`» нельзя (R1, гл. 10.2).

Наконец, ограничения работают и в `impl`-блоках — **условная реализация**: `impl<T: Display + PartialOrd> Pair<T>` определяет методы только для тех `Pair<T>`, чей `T` проходит по требованиям (R1, гл. 10.2). Частный случай, на котором стоит стандартная библиотека, — **общая реализация** (blanket implementation): `impl<T: Display> ToString for T` — «каждый тип с Display автоматически получает ToString» (R1); отсюда у чисел есть `.to_string()`, хотя никто не писал `impl ToString for i32`.

**Интуиция.**

```
   fn notify(item: &impl Summary)
                       │
        ┌──────────────┴───────────────┐
        │ фейс-контроль на входе:      │   Article  ──✓──►  пропущен
        │ «предъявите impl Summary»    │   Comment  ──✓──►  пропущен
        └──────────────────────────────┘   i32      ──✗──►  E0277 на вызове
   внутри функции доступно РОВНО то,
   что обещает Summary: summarize().       (сравните: qsort пропускал всех —
   Полей Article там не видно:              расплата приходила в рантайме)
   контракт, а не конкретный тип
```

**Разобранные примеры.**

*Пример 1 — обе формы параметра.* Файл `notify.rs` (типы и реализации — как в §19.2.1):

```rust
fn notify(item: &impl Summary) {
    println!("новое в ленте: {}", item.summarize());
}

fn notify_generic<T: Summary>(item: &T) {
    println!("новое в ленте: {}", item.summarize());
}

fn main() {
    let article = Article { headline: String::from("Rust 1.97 вышел") };
    let comment = Comment { username: String::from("max") };
    notify(&article);
    notify(&comment);
    notify_generic(&article);
    notify_generic(&comment);
}
```

```
новое в ленте: статья «Rust 1.97 вышел»
новое в ленте: комментарий @max
новое в ленте: статья «Rust 1.97 вышел»
новое в ленте: комментарий @max
```

Внутри `notify` тип конкретного `item` неизвестен и не нужен: доступен ровно контракт — `summarize()`. Попытка обратиться к `item.headline` не соберётся — E0609 «no field `headline` on type `&impl Summary`» (проверено): поле — не часть контракта. Это «программирование к интерфейсу» с проверкой компилятора.

*Пример 2 — несколько требований и возврат impl Trait.* Файл `clone_debug_bound.rs`:

```rust
use std::fmt::Debug;

fn backup<T: Clone + Debug>(value: &T) -> T {
    let copy = value.clone();
    println!("сохранил копию: {copy:?}");
    copy
}
```

```
сохранил копию: 42
оригинал 42, копия 42
сохранил копию: Config { retries: 3, verbose: true }
копия жива: retries = 3, verbose = true
```

`Clone + Debug`: тело зовёт и `clone()`, и `{:?}` — оба умения перечислены. Требования удовлетворяет и `i32`, и наш `Config` c `#[derive(Debug, Clone)]` — derive-пометки главы 16 предстали тем, чем были всегда: реализациями трейтов (§19.2.3). А возврат `impl Trait` — файл `return_impl.rs`:

```rust
fn sample_item() -> impl Summary {
    Comment {
        username: String::from("bot"),
        text: String::from("это автоматический комментарий"),
    }
}
```

```
@bot: это автоматический комментарий
```

⛔ Но «то одно, то другое» в таком результате не живёт — файл `return_impl_two_bad.rs` (проверено; вывод сокращён — вторая половина `help` предлагает обернуть обе ветки в `Box::new`):

```rust

    } else {
        Comment { username: String::from("max") }
    }
}
```

`impl Summary` — маска для *одного* типа, известного компилятору (мономорфизация должна знать, что штамповать). Подсказанный `Box<dyn Summary>` — «коробка с любым Summary, выбираемым в рантайме» — другой механизм с другой ценой; он дождётся главы 22, как и `Box<dyn Error>` из §18.1.3.

*Пример 3 — условная реализация.* Файл `pair_cmp_display.rs`:

```rust
use std::fmt::Display;

struct Pair<T> {
    first: T,
    second: T,
}

impl<T> Pair<T> {
    fn new(first: T, second: T) -> Pair<T> {
        Pair { first, second }
    }
}

impl<T: Display + PartialOrd> Pair<T> {
    fn show_larger(&self) {
        if self.first >= self.second {
            println!("больший элемент — первый: {}", self.first);
        } else {
            println!("больший элемент — второй: {}", self.second);
        }
    }
}
```

```
больший элемент — второй: 12
больший элемент — первый: pear
пара без порядка создана: NoOrder { _tag: 1 } NoOrder { _tag: 2 }
```

`Pair::new` доступен всем; `show_larger` — только парам из сравнимых и печатаемых элементов. Третья строка прогона — контрольная: `Pair<NoOrder>` из типа без `PartialOrd` спокойно *создался*; не существует у него только метода `show_larger` (вызов дал бы E0599 с перечнем несоблюдённых требований — как у `HashMap` в §19.2.3). Тип платит только за те умения, которыми пользуется, — как `distance_from_origin` у `Point<f64>` в §19.1.2, но условие теперь не «T равен f64», а «T умеет то-то».

**Связь с практикой.** Выбор формы — вопрос читаемости: один простой параметр — `&impl Трейт`; связанные параметры, повторное использование `T` в сигнатуре, длинные списки — `<T: ...>` и `where`. Возврат `impl Трейт` пригодится по-настоящему в главе 21: итераторные цепочки имеют «непроизносимые» типы, и `-> impl Iterator` — единственный вменяемый способ их вернуть (анонс). Условные реализации — способ строить «этажи» API: базовые методы для всех, дополнительные — для тех, кто умеет больше.

**Типичные ошибки и подводные камни.**

- **`&impl A` дважды ≠ `<T: A>` дважды.** Первое — два независимых типа, второе — один общий. Компилятор поймает не всегда в вашу пользу: выбирайте форму по смыслу, а не по краткости.
- **Возврат «разных» типов под `impl Trait`.** E0308 в ветках `if`/`else`. Один результат — один тип; «или-или» — это `enum` (гл. 16) или `Box<dyn>` (гл. 22).
- **Требовать в bounds лишнее.** `T: Clone + Debug + PartialOrd` «на всякий случай» отсекает пользователей без нужды. Требуйте по факту использования — правило минимальности §19.1.3.
- **Звать поля через контракт.** Внутри `fn f(item: &impl Summary)` есть только методы Summary. Нужны поля — вам нужен конкретный тип, а не обобщение.

**Источники:** R1 (гл. 10.2 — impl Trait в параметрах и результате, ограничение «один тип», `+`, where, условные и общие реализации, ToString); R7 (items/traits, items/generics — арбитр форм ограничений); R2 (trait — использование в функциях); R6 (generics/impl-trait, generics/trait-bounds). Сторона C: §6.3.2 курса.

### §19.2.3. Стандартные трейты: derive, Display, From, FromStr

**Определения.** Всё, чем вы пользовались «по рецепту», — реализации стандартных трейтов. Разложим по полочкам.

**`derive` — это impl, написанный компилятором.** Атрибут `#[derive(Debug)]` на типе — просьба сгенерировать `impl Debug for ВашТип` со стандартным телом (печать по полям). Так же генерируются:

- **`Clone`** — явное копирование `.clone()` (§15.1.3);
- **`Copy`** — пометка «копируй неявно» (§15.1.3; требует Clone);
- **`PartialEq`** — операторы `==`/`!=`: у структур — поле к полю, у enum — совпадение вариантов и их данных (R9);
- **`Eq`** — пометка поверх `PartialEq`: «равенство полное, без исключений». Методов у неё нет (R9); зачем она, видно от противного: `f64` умеет `PartialEq`, но не `Eq`, потому что `NaN != NaN` — даже сам себе не равен (R9). Контейнерам вроде `HashMap` нужна именно полная версия;
- **`Hash`** — хеширование значения, согласованное с равенством (R9);
- **`PartialOrd`** (и его полный близнец **`Ord`**) — операторы порядка: у структур derive сравнивает поля **в порядке объявления** (лексикографически), у enum более ранние варианты меньше поздних (R9).

**`Display` — печать для людей, только руками.** Формат `{}` требует трейт `std::fmt::Display`; derive для него **не существует** — представление «для пользователя» автор обязан выбрать сам (R9). Контракт — один метод:

```rust
impl fmt::Display for ВашТип {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        write!(f, "...как печатать...")
    }
}
```

Новый макрос **`write!`** — тот же язык форматов, что у `println!`/`format!` (§16.1.3), но пишет в переданный приёмник `f` и возвращает `Result` (R9) — его мы просто пробрасываем как результат `fmt`. Бонус общей реализации из §19.2.2: вместе с `Display` тип автоматически получает `.to_string()` (R9: blanket impl `ToString` для всех `T: Display`).

**`From` — тот самый рецепт из §18.2.2.** Его определение в стандартной библиотеке — три строки (R9):

```rust
pub trait From<T>: Sized {
    fn from(value: T) -> Self;
}
```

Читаем с высоты этой главы: обобщённый трейт с параметром типа `T` (тип-источник) и одним методом, который из значения `T` делает `Self` — реализующий тип. `impl From<io::Error> for AppError` — реализация этого контракта с подстановкой `T = io::Error`, `Self = AppError`. Приписка `: Sized` — техническое ограничение на `Self` («тип известного размера»; подробности — R9, нам не мешает). Требование «`From` не должен отказывать» (R9) тоже стало читаемым: сигнатура `fn from(T) -> Self` не оставляет места для `Result`.

**`FromStr` — разбор из строки, парный к Display.** Определение (R9):

```rust
pub trait FromStr: Sized {
    type Err;
    fn from_str(s: &str) -> Result<Self, Self::Err>;
}
```

Новинка — строка `type Err;`: трейт может требовать не только функции, но и **ассоциированный тип** (associated type) — «слот под тип», который заполняет реализация (R7: состав трейта — функции, типы, константы). Здесь слот отвечает на вопрос «а каким типом вы сообщаете об ошибке разбора?». Пользуются `FromStr` почти всегда не напрямую, а через метод **`parse`** строк: `"42".parse::<u32>()` — «разбери меня в тип из turbofish» — вернёт `Result<u32, ...>` с ошибкой того самого ассоциированного типа (R9).

**Интуиция.** Одна картинка на весь параграф:

```
                ваш тип
                   │
      ┌────────────┼──────────────────────────────┐
      ▼            ▼                              ▼
   derive       руками                      реализовано за вас
   Debug {:?}   Display {} ──► .to_string()  ToString (blanket)
   Clone        From<X>    ──► работает `?`  Into (зеркало From — R9; за рамками главы)
   Copy         FromStr    ──► .parse::<Тип>()
   PartialEq ==
   Eq, Hash  ──► годится в ключи HashMap
   PartialOrd <> ──► годится в largest, min_max, сортировки
```

Слева — умения со стандартным «телом», которые достаточно попросить; в середине — те, где содержание может задать только автор типа; справа — то, что механизмы главы достраивают сами.

*Пример 2 — Display для своих ошибок.* Главный должок главы 18: `ParseError` печатался только «программистским» `{:?}`. Файл `display_parse_error.rs`:

```rust
use std::fmt;

#[derive(Debug)]
enum ParseError {
    Empty,
    BadChar(char),
    TooBig,
}

impl fmt::Display for ParseError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            ParseError::Empty => write!(f, "пустая строка"),
            ParseError::BadChar(c) => write!(f, "недопустимый символ {c:?}"),
            ParseError::TooBig => write!(f, "число не помещается в u32"),
        }
    }
}

fn main() {
    let errors = vec![
        ParseError::Empty,
        ParseError::BadChar('x'),
        ParseError::TooBig,
    ];
    for e in &errors {
        println!("Display: {e}");
        println!("Debug:   {e:?}");
    }

    let s = ParseError::BadChar('#').to_string();
    println!("to_string: {s}");
    println!("а число: {}", 255.to_string());
}
```

```
Display: пустая строка
Debug:   Empty
Display: недопустимый символ 'x'
Debug:   BadChar('x')
Display: число не помещается в u32
Debug:   TooBig
to_string: недопустимый символ '#'
а число: 255
```

Тело `fmt` — обычный `match` по вариантам (§16.2.2): для каждого — своя человеческая формулировка, у `BadChar` в дело идёт полезная нагрузка. Теперь граница из §18.2.3 может печатать `{e}` вместо `{pe:?}` — сообщение для пользователя, а не для отладчика. Последние две строки — blanket-реализация в действии: `.to_string()` появился и у нашей ошибки, и у числа, хотя `ToString` никто не реализовывал.

*Пример 3 — From у своих типов: newtype с гарантией.* В §8.2.3 мы горевали, что `typedef int Meters` не мешает сложить метры с секундами; §16.1.1 дал newtype-защиту; `From` добавляет к ней честные конвертации. Файл `from_newtype.rs`:

```rust
#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
struct Minutes(u32);

#[derive(Debug, Clone, Copy, PartialEq, PartialOrd)]
struct Seconds(u32);

impl From<Minutes> for Seconds {
    fn from(m: Minutes) -> Seconds {
        Seconds(m.0 * 60)
    }
}

fn main() {
    let brew = Minutes(5);
    let s = Seconds::from(brew);
    println!("{brew:?} = {s:?}");

    let limit = Seconds(200);
    println!("дольше лимита? {}", s > limit);
}
```

```
Minutes(5) = Seconds(300)
дольше лимита? true
```

Сложить `Minutes` с `Seconds` по-прежнему нельзя — E0369 «cannot add `Seconds` to `Minutes`» (проверено): оператора `+` у newtype-обёрток нет, пока вы его не дадите, и неявных приведений между ними нет тоже. А вот **осознанно** перевести одно в другое — теперь одна строка с понятной семантикой. Именно так `?` в §18.2.2 переводил ошибки: `Seconds::from(brew)` и `AppError::from(e)` — один механизм.

Узнаёте таблицу отказов? `InvalidDigit` / `Empty` / `PosOverflow` — это наши `BadChar` / `Empty` / `TooBig` из §18.2.1: стандартная библиотека спроектировала ошибку разбора по тем же критериям R2, что и мы (тип вместо строки, категория внутри), и вдобавок дала ей `Display` (последняя строка — `{e}` печатает человеческое сообщение). Тип называйте turbofish-ем или аннотацией слева (`let n: i64 = ...`) — как с `Vec` в §19.1.2. Обещание из резюме главы 18 выполнено: сигнатура `parse` — обобщённая функция, возвращающая `Result<F, F::Err>` для любого `F: FromStr`, — читается целиком.

А научить `parse` **своему** типу — значит реализовать `FromStr`. Файл `fromstr_light.rs` (тип `Light` — из примера 1):

```rust
use std::str::FromStr;

#[derive(Debug)]
struct BadLight(String);

impl FromStr for Light {
    type Err = BadLight;

    fn from_str(s: &str) -> Result<Light, BadLight> {
        if s == "red" {
            Ok(Light::Red)
        } else if s == "yellow" {
            Ok(Light::Yellow)
        } else if s == "green" {
            Ok(Light::Green)
        } else {
            Err(BadLight(String::from(s)))
        }
    }
}
```

```
Ok(Red)
Ok(Green)
ошибка: не название сигнала: "blue"
```

Строка `type Err = BadLight;` заполняет ассоциированный слот: «мои ошибки разбора — вот такого типа» (сам `BadLight` — newtype над `String` с ручным `Display`; полный листинг — в файле). После этого `"red".parse::<Light>()` работает как встроенный — ваш тип встал в один ряд с `u32`.

*Пример 5 — ключи HashMap: долг §17.2.1.* «Ключ должен уметь `Eq` и `Hash`» — теперь это читается. Файл `hashmap_key.rs`:

```rust
use std::collections::HashMap;

#[derive(Debug, PartialEq, Eq, Hash)]
struct RoomId {
    floor: u32,
    number: u32,
}

fn main() {
    let mut occupancy = HashMap::new();
    occupancy.insert(RoomId { floor: 2, number: 201 }, "занята");
    occupancy.insert(RoomId { floor: 3, number: 305 }, "свободна");

    let key = RoomId { floor: 2, number: 201 };
    println!("комната {key:?}: {:?}", occupancy.get(&key));

    let mut prices = HashMap::new();
    prices.insert(String::from("apple"), 40);
    println!("цена яблока: {:?}", prices.get("apple"));
}
```

```
комната RoomId { floor: 2, number: 201 }: Some("занята")
цена яблока: Some(40)
```

Тройка `derive(PartialEq, Eq, Hash)` — стандартный пропуск в ключи (R9: требования — `Eq` и `Hash`; равные ключи обязаны давать равный хеш — derive согласованность гарантирует). ⛔ Без пропуска не пустят: файл `hashmap_key_bad.rs` даёт **три** ошибки разом (E0277 «`RoomId: Eq` is not satisfied», E0277 про `Hash` — обе с derive-подсказками — и E0599: метод `get` «существует, но требования не соблюдены» — как у `show_larger` в §19.2.2). И последняя загадка §17.2.1 — почему у таблицы с ключами-`String` можно спрашивать `get("apple")` со срезом: сигнатура `get` обобщена по типу запроса — подойдёт любой тип, которым ключ умеет «одалживаться» (`String` одалживается как `&str` — та же пара, что в §15.3.2); за точной механикой (трейт `Borrow`) отправляем в R9 — в курсе она не понадобится.

**Как это было в C.** Сводка закрытых долгов:

| Задача | C (том I) | Rust (эта глава) |
|---|---|---|
| напечатать свой тип | пишешь `print_point(...)`; `printf` не расширяем | `impl Display` — и тип работает в `{}`, `format!`, `to_string` |
| сравнить свои значения | `==` для структур не компилируется (§8.1.1); пишешь `points_equal` | `derive(PartialEq)` — оператор появляется у типа |
| разобрать из строки | `atoi` (без ошибок), `sscanf` (счётчик успехов), наш ручной парсер (§10, §18) | `parse::<T>()` + `FromStr` для своих типов; ошибки — типом |
| единицы измерения | `typedef` — синоним без защиты (§8.2.3) | newtype + `From`: смешать нельзя, перевести — явно |
| «этот тип годится в ключи» | никак не выразить; таблица §9.3.1 написана под один тип ключа | ограничения `K: Eq + Hash` проверяет компилятор |

**Связь с практикой.** Рабочий минимум для *каждого* нового типа в ваших программах: `#[derive(Debug)]` — всегда (отладка); `PartialEq` — как только тип появился в тестах или сравнениях; `Clone`/`Copy` — по семантике владения (§15.1.3); `Display` — только у типов с «лицом для пользователя»: ошибки, единицы, идентификаторы. Для ошибок сложился канон, соберите его сами из глав 18–19: `enum` с данными + `Debug` (derive) + `Display` (руками) + `From` для источников — такой тип и в `?` работает, и в `main -> Result` печатается, и человеку показывается.

**Типичные ошибки и подводные камни.**

- **Ждать derive у Display.** Его нет намеренно (R9): «как показать тип человеку» — решение автора, а не механики. Ошибка E0277 «`Тип` doesn't implement `std::fmt::Display`» у `{}` чинится ручным impl — или заменой на `{:?}`, если аудитория — вы сами.
- **`write!` с `println!`-привычками.** Внутри `fmt` пишите в `f`, а не на экран: `println!` там — баг (вывод пойдёт мимо `format!`/`to_string`). И не глотайте его `Result` — пробрасывайте.
- **Рассогласовать равенство и хеш.** Реализуя `PartialEq` руками, а `Hash` derive-ом (или наоборот), легко нарушить закон «равные ключи → равные хеши» (R9) — `HashMap` начнёт «терять» значения. Пока правило простое: для ключей — только полный derive-набор.
- **`Eq` там, где его нет.** Тип с `f64`-полем не получит `derive(Eq)` (NaN ломает полноту равенства — R9) — значит, и в ключи `HashMap` он не годится. Это не придирка компилятора, а честность о плавающей точке.
- **From с сюрпризом.** Конвертация с потерей или отказом — не `From` (R9: «must not fail»). Переполнение, усечение, «может не получиться» — для таких есть `TryFrom` (за рамками курса) или честная функция с `Result`.

**Источники:** R9 (арбитр по API: std::fmt::Display — «cannot be derived», пример impl с write!, blanket ToString; std::cmp::PartialEq/Eq — NaN и полнота равенства, derive; std::cmp::PartialOrd — derive лексикографический, супертрейт; std::convert::From — определение, «must not fail», роль в ?; std::str::FromStr — type Err, связь с parse, turbofish; std::collections::HashMap — требования Eq + Hash, закон хеша, сигнатура get); R1 (гл. 10.2 — трейты и derive, blanket implementations); R7 (items/traits — ассоциированные элементы: функции, типы, константы); R2 (trait/derive). Сторона C: §8.1.1, §8.2.3, §9.3.1, §10.1 курса.

## Часть 19.3. Времена жизни

> Часть помечена в программе курса как **повышенная сложность**. Хорошая новость: с самим явлением вы работаете с главы 15 — новых запретов здесь нет, есть новый язык для уже действующих правил.

### §19.3.1. Времена жизни в функциях

**Определения.** **Время жизни** (lifetime) — участок программы, на протяжении которого ссылка гарантированно указывает на живое значение. Оно есть у *каждой* ссылки и всегда было: E0597 «does not live long enough» из §15.2.3 — это borrow checker, сравнивающий времена жизни ссылки и её владельца. Главная задача механизма — та же, что у всей главы 15: **не дать существовать висячим ссылкам** (R1, гл. 10.3).

Новое — **аннотации времён жизни** (lifetime annotations): обобщённые параметры, описывающие *связи* между временами жизни ссылок в сигнатуре. Синтаксис: имя с апострофом, по соглашению короткое строчное — `'a` («тик-эй»); объявляется в тех же угловых скобках, что параметры типа (времена жизни — разновидность обобщённых параметров, R1, гл. 10.3), и приписывается к ссылкам после `&`:

```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str
```

Читается: «существует время жизни `'a`; оба входа живут как минимум `'a`, и результат действителен как минимум `'a`». На практике это значит: **результат годен, пока живы оба аргумента** — компилятор возьмёт за `'a` *меньшее* из времён жизни входов и проверит все использования результата против него (R1, гл. 10.3).

Два принципиальных «не»:

- аннотация **не продлевает** ничью жизнь и не меняет поведение программы — она только описывает связи, чтобы компилятор мог проверить вызовы (R1, гл. 10.3);
- аннотации нужны **не всегда**: у большинства сигнатур связь однозначна, и компилятор восстанавливает её сам по правилам умолчаний (§19.3.2). `longest` — честный случай неоднозначности: результат может оказаться и `x`, и `y`.

**Интуиция.** Тот же рисунок областей, которым мы объясняли E0597 в главе 15, — теперь с именами:

```
   fn main() {
       let outer = String::from("снаружи");          ──┐ время жизни outer
       let result;                                     │
       {                                               │
           let inner = String::from("внутри");   ──┐   │ время жизни inner
           result = longest(&outer, &inner);       │   │
       }                                         ──┘   │  ← inner умерла
       println!("{result}");   ⛔ result обещал        │
   }                              жить не дольше     ──┘
                                  МЕНЬШЕГО из двух — inner
```

Сигнатура `longest<'a>` — контракт: «моему результату нельзя пережить ни один из входов». Компилятор не выясняет, *какую* ветку вернул `if` в конкретном запуске (это знание рантайма!), — он страхует худший случай. Ровно так же вы рассуждали в C, только держали рассуждение в голове (§7.1.2: «указатель в буфер годен, пока буфер не освобождён»).

**Разобранные примеры.**

**Как это было в C.** Связь «этот указатель годен, пока живо то-то» в томе I существовала только как дисциплина: комментарий к функции, абзац в man-странице, устная договорённость («`strtok` портит исходную строку», «указатель из `get_field` не переживает буфера»). Компилятор её не видел: `dangle_c.c` из §15.2.3 — возврат адреса локальной — GCC собрал с предупреждением, а нарушения сложнее (вернуть указатель в буфер, который вызывающий освободит раньше времени) не ловит даже предупреждением — только ASan постфактум (§7.2.2). Аннотация `'a` — это тот самый комментарий, **переведённый на язык, который проверяется**: контракт из man-страницы переехал в сигнатуру.

**Связь с практикой.** Когда вы *пишете* функцию со ссылками и компилятор просит лайфтаймы — сначала спросите себя, чью память возвращает результат (правило из §15.2.3): владение (`String`) — и аннотации не нужны; кусок одного конкретного входа — обычно спасут правила умолчаний (§19.3.2); «кусок одного из нескольких входов» — вот честная работа для `'a`. Аннотации — не украшение и не заклинание против ошибок: это способ *рассказать* компилятору то, что вы и так обязаны знать о своих ссылках.

**Типичные ошибки и подводные камни.**

- **«Аннотирую — и заживёт дольше».** Не заживёт: `'a` описывает, не управляет. Если E0597 говорит «не доживает» — надо менять код (двигать владельца выше, возвращать владение), а не сочинять другую аннотацию.
- **Читать `'a` как «оба входа живут одинаково».** Нет: `'a` — общий минимум, входы с разными временами жизни законны (пример 2).
- **Обижаться на «лишний» запрет.** Пример 3: запрещён и тот исход, который «в этот раз сработал бы». Проверка статическая — гарантии даются на все исходы сразу; это и есть отличие от «повезло/не повезло» главы 13.
- **Лайфтаймы «на всякий случай».** Если сигнатура компилируется без аннотаций — их и не пишут (правила умолчаний, §19.3.2). `fn f<'a>(s: &'a str) -> usize` — шум: результат не ссылка, связывать нечего.

### §19.3.2. Ссылки в структурах, правила умолчаний, 'static

**Определения.** **Структура может хранить ссылку** — но только объявив время жизни для неё: `struct Excerpt<'a> { part: &'a str }`. Аннотация означает: **экземпляр структуры не может пережить ссылку, которую хранит** (R1, гл. 10.3). Это ответ на E0106 из §16.1.1: `&str`-поле без `'a` отклонялось, потому что у «структуры с чужой памятью внутри» обязан быть контракт с этой памятью. Методы такой структуры пишутся в `impl<'a> Excerpt<'a>` — объявление параметра то же, что `impl<T>` в §19.1.2.

**Правила умолчаний** (lifetime elision rules) — три правила, по которым компилятор сам восстанавливает аннотации в частых случаях (R1, гл. 10.3; R7 — арбитр; входные времена жизни — у параметров, выходные — у результата):

1. каждый параметр-ссылка получает **своё** время жизни: `fn f(x: &str, y: &str)` → `fn f<'a, 'b>(x: &'a str, y: &'b str)`;
2. если входное время жизни ровно **одно** — оно присваивается всем выходным: `fn f(s: &str) -> &str` → `fn f<'a>(s: &'a str) -> &'a str`;
3. если входных несколько, но среди них есть **`&self`/`&mut self`** (метод), — выходным присваивается время жизни `self`.

Прошёлся по правилам — и выхода без времени жизни не осталось? Аннотации не нужны. Остался — компилятор требует их явно (E0106); именно это случилось с `longest`: правило 1 дало `'a` и `'b`, правила 2–3 не применимы — выход неопределён (R7: «элидировать невыводимое — ошибка»).

**`'static`** — особое время жизни: «может жить всю программу». Им обладают все строковые литералы — их байты лежат в самом исполняемом файле (R1, гл. 10.3): `let s: &'static str = "я живу всегда";`.

**Интуиция.**

```
   struct Excerpt<'a> { part: &'a str }

   let quote;                        │ время жизни quote ────────────┐
   {                                 │                               │
       let book = String::from(...); │──┐ время жизни book           │
       quote = Excerpt {             │  │                            │
           part: first_word(&book),  │  │ part смотрит в book ──►    │
       };                            │──┘ ⛔ book умирает,           │
   }                                 │      а quote (и part) ещё жив─┘
   println!("{}", quote.part);       │      — запрещено контрактом 'a
```

Правило то же, что у простых ссылок в §15.2: «ссылка не переживает владельца», — только теперь ссылка едет внутри структуры, и `'a` протаскивает контракт сквозь неё.

```rust
{
    println!("Объявление: {ann}");
    if x.len() > y.len() {
        x
    } else {
        y
    }
}

fn main() {
    let a = String::from("длинная строка");
    let b = String::from("короче");
    let winner = longest_with_announcement(&a, &b, "сравниваем два заголовка");
    println!("длиннее: {winner}");
    let winner2 = longest_with_announcement(&a, &b, 2026);
    println!("длиннее: {winner2}");
}
```

```
Объявление: сравниваем два заголовка
длиннее: длинная строка
Объявление: 2026
длиннее: длинная строка
```

Читаем сигнатуру слева направо, как теперь умеем: `<'a, T>` — два обобщённых параметра, время жизни и тип (они живут в одном списке, потому что суть одна — R1); входы-ссылки связаны `'a` с результатом; `T` обязан уметь `Display` (where — §19.1.3); вызовы инстанцируют `T` как `&str` и как `i32`. Вся глава — в одной строке.

**Как это было в C.** Строковые литералы C тоже живут всю программу (статическая длительность хранения — C5, §6.4.5; курс — §5.2.1), но тип об этом молчал: `char *` одинаков и у литерала, и у поля кучи, и у указателя в стек — язык не различал их судьбы, различать должны были вы. Отсюда — каталог главы 13: use-after-free, возврат локального адреса, «указатель пережил буфер». Rust разделил эти судьбы **в типах**: `&'static str`, `&'a str`, `String` — три разные истории с проверяемыми контрактами.

**Связь с практикой.** Частота инструментов в реальном коде обратна их месту в главе: чаще всего вы не пишете лайфтаймы вовсе (умолчания), иногда — `'a` в функции-«выборщике» вроде `longest`, редко — структуру со ссылкой (и только когда «вид без копии» действительно нужен: парсеры, отчёты по большому буферу; во всех остальных случаях владеющее поле `String` проще и самодостаточнее — совет §16.1.1 остаётся в силе). Увидели в чужом коде `<'a>` — читайте как связи: «кто кому обещает не пережить кого».

**Типичные ошибки и подводные камни.**

- **`'static` как затычка.** Совет из ошибки «consider using the `'static` lifetime» часто маскирует настоящую проблему — попытку создать висячую ссылку; лечится исправлением владения, а не громким обещанием «живу вечно» (R1, гл. 10.3). Честный `'static` — литералы и константы.
- **Структура-ссылка «для экономии» везде.** `Excerpt<'a>` заражает лайфтаймом каждого, кто её хранит. Правило §16.1.1 никуда не делось: по умолчанию — владеющие поля; ссылки — когда «вид» осмыслен и короткоживущ.
- **Дописывать лайфтаймы до «пока не соберётся».** Перебор аннотаций вместо понимания связей — дорога в тупик: аннотации описывают факты о коде; неверный факт не спасёт никакая запись. Начните с вопроса «чью память возвращаю/храню».
- **Паника при виде E0106 в новом месте.** Алгоритм всегда один: применить три правила умолчаний вручную; если выход остался без времени жизни — решить, к какому входу (входам) он привязан по смыслу, и записать это.

**Источники:** R1 (гл. 10.3 — структуры со ссылками и смысл аннотации, три правила умолчаний, 'static и предостережение о нём, синтез с generics/bounds); R7 (lifetime-elision — формулировки правил и «элидировать невыводимое — ошибка»: арбитр; items/generics — лайфтаймы как обобщённые параметры); R2 (scope/lifetime/struct, scope/lifetime/elision); R6 (lifetimes/lifetime-annotations). ## Резюме главы

- **Параметры типа** обобщают функции, структуры и перечисления: `largest<T: PartialOrd>`, `Point<T>`, `Option<T>` — «типы с дыркой», которую заполняет вывод типов, аннотация или turbofish `::<>`. Тело проверяется **в точке определения** (E0369 даже без единого вызова — не как макросы §11.1), требования к типу — **в точке вызова** (E0277 с derive-подсказками). Внутри тела доступно ровно то, что разрешают ограничения (`T: A + B`, `where`) — ни операцией больше.
- **Мономорфизация** штампует по экземпляру кода на каждый использованный тип: обобщённость полностью исчезает к рантайму, вызовы прямые, цена — размер бинарника, а не скорость. Это «копипаста компилятором» — против макросов (текст без проверки) и `void *` (казты плюс косвенный вызов) из тома I.
- **Трейт — контракт поведения**: объявление перечисляет сигнатуры (плюс тела по умолчанию), `impl Трейт for Тип` выполняет контракт целиком (E0046 — «весь или никакого»). Правило сироты: трейт или тип должен быть вашим (E0117; выход — newtype). Требования пишутся ограничениями (`<T: Summary>`, `&impl Summary`, условные `impl<T: ...>`), «возврат одного невидимого типа» — `impl Trait` (двух разных — E0308; «или-или» ждёт `Box<dyn>` в главе 22).
- **Стандартные трейты — разгаданные рецепты курса**: `derive` — это impl, сгенерированный компилятором (`Debug`, `Clone`, `Copy`, `PartialEq`/`==`, `Eq`, `Hash`, `PartialOrd`); `Display` пишется только руками (write! в Formatter) и дарит `to_string` blanket-реализацией; `From` — обобщённый трейт с одним безотказным методом (`impl From<X> for Y` из §18.2.2 теперь прозрачен); `FromStr` с ассоциированным `type Err` стоит за `"42".parse::<u32>()` — и за пенсией нашего `parse_u32` (его ошибки Empty/InvalidDigit/PosOverflow стандартная библиотека спроектировала по тем же правилам). Ключи `HashMap` — это `Eq + Hash` (derive-тройка; у `f64` нет `Eq` из-за NaN).
- **Времена жизни** — контракты ссылок: `'a` описывает (не меняет!) связь «результат не переживёт входов» (`longest<'a>`), `Excerpt<'a>` — «структура не переживёт ссылку в поле» (E0106 из глав 15–16 закрыт). Три правила умолчаний покрывают большинство сигнатур — мы писали аннотированный код с главы 15, не зная об этом; `'static` — честное «живу всю программу» (литералы), а не затычка от E0597. Контракты из комментариев C переехали в сигнатуры и стали проверяемыми.
- **Сквозной итог**: `<'a, T: Display>` в одной сигнатуре — обобщение по типам, требованиям и временам жизни одним механизмом; проверки — при компиляции, санкции — до запуска, рантайм — как у ручного кода.

Глава 20 переключает регистр на инженерную гигиену — как из «файла с функциями» сделать **проект**: модули и `use` (три главы мы писали его «по образцу» — пора разобраться), видимость `pub`, библиотечные crate, внешние зависимости через Cargo — и главное, **тесты**: `cargo test`, в котором пригодятся и паника (упавший тест — это паника, §18.1.1), и `PartialEq` (сравнение ожидаемого с полученным — макросом `assert_eq!`).
