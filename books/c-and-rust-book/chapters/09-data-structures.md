# Глава 9. Структуры данных на C

## Введение

Всё готово для главной стройки тома I. Глава 6 дала указатели, глава 7 — кучу и дисциплину владения, глава 8 — структуры и узел с полем «указатель на такого же». По отдельности это детали; **структуры данных** — то, что из них собирается: способы *организовать* множество значений так, чтобы нужные операции — добавить, найти, убрать — были удобными и дешёвыми.

В этой главе мы своими руками построим шесть классических структур:

- **динамический массив** (вектор) — массив, который растёт сам;
- **односвязный список** — цепочка узлов в куче;
- **стек** — «последним пришёл — первым ушёл» (LIFO);
- **очередь** — «первым пришёл — первым ушёл» (FIFO);
- **хеш-таблицу** — поиск по строковому ключу «почти мгновенно»;
- **двоичное дерево поиска** (BST) — хранение в отсортированном порядке.

Зачем строить руками то, что в любом языке «уже есть»? Во-первых, в C этого *нет*: стандартная библиотека не даёт ни списка, ни таблицы — умение собрать их из `malloc` и указателей здесь просто необходимо. Во-вторых — и это важнее, — только собрав структуру сам, понимаешь её **цену**: почему вставка в начало массива дорога, а в начало списка дешева; почему поиск в таблице быстр, но требует хорошего хеша; почему дерево может выродиться в список. В томе II эти же структуры читатель получит «из коробки» (`Vec`, `HashMap` — глава 17; рекурсивные типы через `Box` — глава 22) — и будет точно знать, что у них под капотом и за что он платит.

После изучения главы вы будете уметь:

- реализовывать динамический массив с ростом ёмкости (`len`/`cap`, удвоение через `realloc`) и оценивать стоимость его операций;
- строить односвязный список: вставка в начало и после узла, поиск, удаление, освобождение — и выбирать между списком и массивом по стоимости операций;
- реализовывать стек и очередь поверх массива и поверх списка, формулировать их инварианты и применять их (проверка скобок, обработка «по кругу»);
- строить хеш-таблицу с разрешением коллизий цепочками: хеш-функция, корзины, вставка/поиск/обновление, владение копиями ключей;
- строить двоичное дерево поиска: рекурсивные вставка и поиск, inorder-обход, освобождение, вырожденный случай;
- прогонять всё это под AddressSanitizer и убеждаться, что памятью структура распоряжается корректно.

> Мы опираемся только на пройденное (главы 1–8): рекурсия (§4.3), строки и `string.h` (гл. 5), указатели и `int **` (гл. 6), `malloc`/`realloc`/`free` и владение (гл. 7), структуры, `typedef`, самоссылающиеся типы (гл. 8). **Все программы главы прогнаны и под AddressSanitizer** (§7.2.2) — утечек и ошибок памяти нет; делайте так же со своими решениями. Для краткости листингов повторяющиеся вспомогательные функции заменены комментарием-ссылкой (например, `/* vec_init … — как в §9.1.1 */`); каждый пример компилировался и запускался как **полная** программа.

## Часть 9.1. Динамический массив и связный список

### §9.1.1. Динамический массив (вектор)

**Определения.**

- **Динамический массив** (dynamic array, вектор) — структура из трёх полей: указатель на блок в куче, **длина** `len` (сколько элементов занято) и **ёмкость** `cap` (на сколько элементов выделен блок):

  ```c
  typedef struct {
      int *data;   /* блок в куче */
      int len;     /* сколько занято */
      int cap;     /* сколько выделено */
  } Vec;
  ```

  Инвариант: `0 <= len <= cap`, и `data` указывает на блок из `cap` элементов (либо `len == cap == 0` и `data == NULL` после освобождения).
- **Добавление в конец** (`push`): если `len < cap`, элемент просто записывается в `data[len]` и `len` растёт — ни одного обращения к куче. Если `len == cap`, ёмкость **удваивается** через `realloc` (безопасная идиома с временным указателем — §7.1.3), и только потом элемент записывается.
- **Стратегия удвоения** делает добавление дешёвым **в среднем** (амортизированно): перевыделения случаются всё реже (на 2, 4, 8, 16-м… элементе), и на `n` добавлений приходится лишь порядка `n` копирований суммарно. Плата — запас памяти: занято может быть до половины ёмкости.
- Владение: блоком `data` владеет вектор; освобождение — одно (`vec_free`), сколько бы `push` ни было (§7.1.3: цепочка `realloc` — один финальный `free`).

**Интуиция.** Вектор — это записная книжка с запасом чистых страниц. `len` — сколько страниц исписано, `cap` — сколько всего в книжке. Пишете подряд — быстро, пока есть чистые страницы. Кончились — покупаете книжку **вдвое толще**, переписываете в неё всё старое (это и есть `realloc`) и продолжаете. Переписывать неприятно, но чем толще книжка, тем реже это случается — потому в среднем запись остаётся дешёвой.

```
   Vec v:    data ──▶ ┌────┬────┬────┬────┐   блок в куче на cap элементов
             len = 2            │ 10 │ 20 │ ?? │ ?? │
             cap = 4            └────┴────┴────┴────┘
                                  занято      запас

   push при len == cap:   realloc в 2×cap   (2 → 4 → 8 → …)
```

**Разобранные примеры.**

*Пример 1. Вектор целиком: init, push с удвоением, free.*

```c
#include <stdio.h>
#include <stdlib.h>

typedef struct {
    int *data;   /* блок в куче */
    int len;     /* сколько занято */
    int cap;     /* сколько выделено */
} Vec;

int vec_init(Vec *v)
{
    v->len = 0;
    v->cap = 2;
    v->data = malloc(v->cap * sizeof *v->data);
    return v->data == NULL ? -1 : 0;
}

int vec_push(Vec *v, int x)
{
    if (v->len == v->cap) {                    /* места нет — удваиваем */
        int *tmp = realloc(v->data, 2 * v->cap * sizeof *v->data);
        if (tmp == NULL) return -1;            /* v->data ещё жив (§7.1.3) */
        v->data = tmp;
        v->cap *= 2;
    }
    v->data[v->len++] = x;
    return 0;
}

void vec_free(Vec *v)
{
    free(v->data);
    v->data = NULL;
    v->len = v->cap = 0;
}

int main(void)
{
    Vec v;
    if (vec_init(&v) != 0) return 1;
    for (int i = 1; i <= 5; i++) {
        if (vec_push(&v, i * 10) != 0) { vec_free(&v); return 1; }
        printf("push %d: len=%d cap=%d\n", i * 10, v.len, v.cap);
    }
    for (int i = 0; i < v.len; i++)
        printf("%d ", v.data[i]);
    printf("\n");
    vec_free(&v);
    return 0;
}
```

Вывод:

```
push 10: len=1 cap=2
push 20: len=2 cap=2
push 30: len=3 cap=4
push 40: len=4 cap=4
push 50: len=5 cap=8
10 20 30 40 50 
```

Видна вся механика: первые два `push` заполняют начальную ёмкость (`cap=2`); третий упирается в `len == cap` и удваивает до 4; пятый — до 8. Каждая функция возвращает код результата, и все ошибочные пути освобождают уже выделенное (дисциплина главы 7). Это тот самый «растущий массив» из §7.1.3, но теперь `data`/`len`/`cap` собраны в структуру (гл. 8) и обслуживаются семейством функций с префиксом `vec_` — по сути, готовый **тип данных с операциями**.

*Пример 2. Вставка в середину: за удобство платим сдвигом.*

```c
/* Vec, vec_init, vec_push, vec_free — как в примере 1 */

int vec_insert(Vec *v, int idx, int x)     /* вставка со сдвигом хвоста */
{
    if (idx < 0 || idx > v->len) return -1;
    if (v->len == v->cap) {
        int *tmp = realloc(v->data, 2 * v->cap * sizeof *v->data);
        if (tmp == NULL) return -1;
        v->data = tmp;
        v->cap *= 2;
    }
    for (int i = v->len; i > idx; i--)     /* сдвиг хвоста: справа налево */
        v->data[i] = v->data[i - 1];
    v->data[idx] = x;
    v->len++;
    printf("  (shifted: %d)\n", v->len - 1 - idx);
    return 0;
}

int main(void)
{
    Vec v;
    if (vec_init(&v) != 0) return 1;
    for (int i = 1; i <= 3; i++)
        if (vec_push(&v, i * 10) != 0) { vec_free(&v); return 1; }

    printf("insert 99 at 0:\n");
    if (vec_insert(&v, 0, 99) != 0) { vec_free(&v); return 1; }
    printf("insert 55 at 2:\n");
    if (vec_insert(&v, 2, 55) != 0) { vec_free(&v); return 1; }

    for (int i = 0; i < v.len; i++)
        printf("%d ", v.data[i]);
    printf("\n");
    vec_free(&v);
    return 0;
}
```

Вывод:

```
insert 99 at 0:
  (shifted: 3)
insert 55 at 2:
  (shifted: 2)
99 10 55 20 30 
```

Чтобы вставить элемент в позицию `idx`, весь хвост приходится сдвинуть на одну ячейку вправо — цикл идёт **справа налево**, чтобы не затирать ещё не сдвинутое. Счётчик в выводе показывает цену: вставка в начало трёхэлементного вектора сдвинула 3 элемента, вставка в середину пятиэлементного — 2. В худшем случае (вставка в начало) сдвигается **весь** массив — на миллионе элементов это миллион копирований. Запомните этот факт: сейчас появится структура, у которой вставка в начало стоит два присваивания.

**Связь с практикой.** Вектор — самая используемая структура данных вообще: непрерывный блок памяти дружит с кешем процессора, доступ по индексу мгновенен, добавление в конец дёшево. Правило по умолчанию: **не знаете, что взять, — берите вектор**. Он лежит в основе строковых буферов, списков записей, стека (§9.2.1); читая файл неизвестной длины (глава 10), вы будете наполнять именно его. Его слабое место — вставки/удаления в начале и середине (сдвиг хвоста). В Rust `Vec<T>` (гл. 17) устроен ровно так же — `указатель + len + cap` с удвоением, — только рост, проверка границ и освобождение там автоматические.

**Типичные ошибки и подводные камни.**

- **Обращение к «запасу»**: `data[len]`…`data[cap-1]` — выделенная, но **не занятая** память; читать её — мусор, и логика структуры ломается. Граница всегда `len`, не `cap`.
- **`v->data = realloc(v->data, ...)` без временного указателя** — при неудаче теряется единственный указатель на блок (утечка, §7.1.3). Только через `tmp`.
- **Сдвиг слева направо при вставке.** Цикл `for (i = idx; i < len; i++) data[i+1] = data[i];` затирает хвост первым же шагом (все ячейки станут копией `data[idx]`). Сдвигают справа налево.
- **Забыть, что после `vec_push` старые указатели внутрь `data` могли умереть.** `realloc` мог перенести блок (§7.1.3): сохранённый где-то `int *p = &v.data[3]` после роста вектора — висячий. Храните **индексы**, а не адреса элементов.
- **Раздача внутренностей наружу.** `data` принадлежит вектору: вызывающий, получивший `v.data`, не должен ни `free`-ить его, ни запоминать надолго (см. предыдущий пункт). Владелец один — `vec_free`.

**Источники:** C2 (гл. 17 — dynamic storage, растущие массивы); C8 (лекции о структурах данных); C7 (нед. 5); C4 (память, владение); C6.

### §9.1.2. Односвязный список: устройство, обход, поиск

**Определения.**

- **Односвязный список** (singly linked list) — набор **узлов** в куче, где каждый узел хранит значение и указатель на следующий узел; последний хранит `NULL`. Тип узла — самоссылающаяся структура §8.1.3:

  ```c
  typedef struct node {
      int value;
      struct node *next;
  } Node;
  ```

- **Голова** (head) — указатель на первый узел; сам список в коде представлен именно головой. **Пустой список** — `head == NULL`: это законное состояние, с которого всё начинается.
- **Вставка в начало** (`push_front`): новый узел выделяется в куче, его `next` смотрит на старую голову, головой становится он. Функция принимает **`Node **head`** — адрес указателя-головы (§6.3.2): ей нужно *изменить сам указатель* вызывающего.
- **Обход** (traversal): «бегунок» `cur` стартует с головы и шагает по `next`, пока не упрётся в `NULL`, — канонический цикл `for (cur = head; cur != NULL; cur = cur->next)`.
- **Освобождение**: каждый узел — отдельный `malloc`, значит, каждому нужен свой `free`. При освобождении текущего узла адрес следующего **запоминают до** `free` — иначе `head->next` читал бы освобождённую память (use-after-free, §7.2.1).

**Интуиция.** Продолжим образ поезда из §8.1.3: список — состав из вагонов, разбросанных по всей куче и сцепленных указателями. У вектора места «прибиты» подряд — чтобы втиснуть кого-то в начало, всех пассажиров пересаживают на место вперёд. В списке достаточно **перецепить сцепки**: новый вагон цепляют к старому первому, и локомотив (голова) теперь тянет его. Ни один существующий узел не шелохнулся — потому вставка в начало стоит копейки, сколько бы ни был длинен состав.

```
   list ─▶ ┌───┬────┐    ┌───┬────┐    ┌───┬──────┐
           │ 1 │ ●──┼───▶│ 2 │ ●──┼───▶│ 3 │ NULL │
           └───┴────┘    └───┴────┘    └───┴──────┘
   каждый узел — отдельный malloc, живёт где-то в куче

   push_front(&list, 0):
       до:     list ─▶ [1]─▶[2]─▶[3]─▶NULL
       узел:   [0].next = старая голова [1]
       после:  list ─▶ [0]─▶[1]─▶[2]─▶[3]─▶NULL   (никто не сдвинулся)
```

**Разобранные примеры.**

*Пример 1. Список целиком: push_front, обход, освобождение.*

```c
#include <stdio.h>
#include <stdlib.h>

typedef struct node {
    int value;
    struct node *next;
} Node;

int push_front(Node **head, int x)     /* 0 — успех, -1 — нет памяти */
{
    Node *n = malloc(sizeof *n);
    if (n == NULL) return -1;
    n->value = x;
    n->next = *head;    /* новый узел смотрит на старую голову */
    *head = n;          /* головой становится новый узел */
    return 0;
}

void print_list(const Node *head)
{
    for (const Node *cur = head; cur != NULL; cur = cur->next)
        printf("%d -> ", cur->value);
    printf("NULL\n");
}

void list_free(Node *head)
{
    while (head != NULL) {
        Node *next = head->next;   /* запомнить ДО free */
        free(head);
        head = next;
    }
}

int main(void)
{
    Node *list = NULL;                 /* пустой список */
    for (int x = 3; x >= 1; x--)       /* кладём 3, 2, 1 */
        if (push_front(&list, x) != 0) { list_free(list); return 1; }
    print_list(list);
    list_free(list);
    return 0;
}
```

Вывод:

```
1 -> 2 -> 3 -> NULL
```

Три вещи, на которые стоит смотреть. Первая: `push_front(&list, ...)` получает **адрес головы** — внутри `*head = n` меняет указатель `list` в `main` (без `Node **` функция меняла бы свою копию головы — список остался бы прежним, как в §8.1.2 с копией структуры). Вторая: клали 3, 2, 1 — получили `1 -> 2 -> 3`: каждая вставка в начало «обгоняет» предыдущие; хотите сохранить порядок — вставляйте в конец (практика З9). Третья: `list_free` сначала уводит `next` в сторонку, потом освобождает узел — попытка `head = head->next` **после** `free(head)` была бы чтением освобождённой памяти.

*Пример 2. Длина и поиск: обход как универсальный приём.*

```c
/* Node, push_front, print_list, list_free — как в примере 1 */

int list_length(const Node *head)
{
    int n = 0;
    for (const Node *cur = head; cur != NULL; cur = cur->next)
        n++;
    return n;
}

Node *list_find(Node *head, int x)     /* узел со значением x или NULL */
{
    for (Node *cur = head; cur != NULL; cur = cur->next)
        if (cur->value == x)
            return cur;
    return NULL;
}

int main(void)
{
    Node *list = NULL;
    for (int x = 3; x >= 1; x--)
        if (push_front(&list, x) != 0) { list_free(list); return 1; }
    print_list(list);
    printf("length = %d\n", list_length(list));

    Node *hit = list_find(list, 2);
    if (hit != NULL)
        hit->value = 20;               /* правим список через указатель */
    print_list(list);
    printf("find(99): %s\n", list_find(list, 99) == NULL ? "NULL" : "found");
    list_free(list);
    return 0;
}
```

Вывод:

```
1 -> 2 -> 3 -> NULL
length = 3
1 -> 20 -> 3 -> NULL
find(99): NULL
```

Оба алгоритма — один и тот же цикл-обход, различается лишь тело. Важная деталь: `list_find` возвращает **указатель на узел**, а не копию значения — через него вызывающий правит список на месте (`hit->value = 20`), как `top()` из Г8.Ч1.З9. И обратите внимание на стоимость: у списка нет «взять `i`-й за один шаг» — даже за длиной приходится идти по всем узлам. Индексация — сильная сторона массива, не списка.

**Связь с практикой.** Список — вторая базовая форма хранения после массива, и выбор между ними — типовое проектировочное решение (§9.1.3 даст таблицу). Списки незаменимы там, где элементы постоянно вставляются и удаляются «по месту», без сдвига соседей: очереди задач, цепочки коллизий хеш-таблицы (§9.3.1), «карманы» в планировщиках. Идея «узлы + указатели» — родовая для всех связных структур: дерево (§9.3.2) — тот же узел, но с двумя «сцепками». В Rust односвязный список — знаменитая учебная «боль» (владение узлами циклично переплетается — понадобится `Box`, гл. 22); наш C-опыт станет там точкой отсчёта.

**Типичные ошибки и подводные камни.**

- **`push_front(Node *head, ...)` вместо `Node **`.** Функция изменит свою локальную копию головы; список вызывающего не изменится, а новый узел утечёт. Менять указатель вызывающего можно только по его адресу (§6.3.2).
- **`head = head->next` после `free(head)`.** Use-after-free (§7.2.1): поле `next` читается из уже освобождённого узла. Только «запомнить → освободить → шагнуть».
- **Потерянный хвост.** Присвоили `n->next = NULL` вместо старой головы (или перепутали порядок строк в `push_front`) — остаток списка отцепился и утёк (Valgrind: `definitely lost`). После каждой операции полезно мысленно (или на бумаге) прорисовать стрелки.
- **Обход после освобождения.** `list_free(list); print_list(list);` — классика: сама переменная `list` всё ещё хранит старый адрес. Привычка из §7.2.1: после освобождения — `list = NULL;`.
- **Забыть случай пустого списка.** `head == NULL` — законный вход для любой функции списка: `list_find` и `list_length` из примера 2 корректны и для него (цикл просто не выполнится) — проверяйте на этом случае и свои функции.

**Источники:** C2 (гл. 17 — linked lists); C8 (лекции о списках); C7 (нед. 5 — linked lists); C4 (владение и указатели); C6.

### §9.1.3. Вставка, удаление и цена операций

**Определения.**

- **Вставка после узла** `pos`: новый узел `n` сначала **подхватывает хвост** (`n->next = pos->next`), затем `pos` цепляется к нему (`pos->next = n`). Порядок двух присваиваний критичен: в обратном порядке хвост списка теряется.
- **Удаление узла со значением `x`**: обход с парой указателей — `cur` (кандидат) и `prev` (предыдущий). Найден `cur`: если `prev == NULL`, удаляется голова (нужно `Node **head`, чтобы её переставить), иначе `prev->next = cur->next` перецепляет список **мимо** удаляемого; затем `free(cur)`.
- **Цена операций** (в числе «шагов» — обращений к элементам; строгая асимптотика — за пределами курса, интуиции достаточно):

  | операция | вектор | список |
  |---|---|---|
  | доступ по индексу `i` | 1 шаг: `data[i]` — адресная арифметика §6.2 | ~`i` шагов: идти по `next` |
  | поиск по значению | до `n` шагов | до `n` шагов |
  | вставка/удаление в начале | ~`n` шагов (сдвиг всего) | **2 присваивания** |
  | вставка в конец | 1 шаг в среднем (изредка `realloc`) | ~`n` шагов (дойти до хвоста)¹ |
  | вставка/удаление после **найденного** узла/индекса | ~`n − i` шагов (сдвиг хвоста) | **2 присваивания** |

  ¹ либо 1 шаг, если хранить отдельный указатель на хвост — так сделает очередь в §9.2.2.

**Интуиция.** Всё решает вопрос «кого придётся потревожить». В массиве элементы прибиты к местам — вставка/удаление тревожит **весь хвост**. В списке элементы держатся друг за друга сцепками — операция тревожит **только соседа**: перецепить две стрелки, и готово. Зато массив мгновенно отвечает на «дай пятого» (место пятого *вычисляется*), а списку для этого надо прошагать пять вагонов. Ни одна структура не лучше другой всюду — они разменивают одни дешёвые операции на другие.

```
   insert_after(pos = [2], 25):
        [1]─▶[2]─────────▶[3]         шаг 1:  [25].next = [3]
              └─▶[25]──▶[3]           шаг 2:  [2].next  = [25]
        итог: [1]─▶[2]─▶[25]─▶[3]

   remove(2):   prev=[1], cur=[2]
        [1]─▶[2]─▶[3]
        [1]────────▶[3]               prev->next = cur->next; free(cur)
```

**Разобранные примеры.**

*Пример 1. Вставка после найденного узла.*

```c
/* Node, push_front, print_list, list_free, list_find — как в §9.1.2 */

int insert_after(Node *pos, int x)     /* вставить x после узла pos */
{
    Node *n = malloc(sizeof *n);
    if (n == NULL) return -1;
    n->value = x;
    n->next = pos->next;   /* шаг 1: новый подхватывает хвост */
    pos->next = n;         /* шаг 2: pos цепляется к новому */
    return 0;
}

int main(void)
{
    Node *list = NULL;
    for (int x = 3; x >= 1; x--)
        if (push_front(&list, x) != 0) { list_free(list); return 1; }
    print_list(list);

    Node *pos = list_find(list, 2);
    if (pos == NULL || insert_after(pos, 25) != 0) { list_free(list); return 1; }
    print_list(list);
    list_free(list);
    return 0;
}
```

Вывод:

```
1 -> 2 -> 3 -> NULL
1 -> 2 -> 25 -> 3 -> NULL
```

Узел `25` встал между `2` и `3` за два присваивания — сколько бы элементов ни было после. Сначала новый узел «подхватил» продолжение (`n->next = pos->next`, т. е. `[3]`), и только потом `pos` перецепился на новый. Сделай мы наоборот — `pos->next = n` первым, — выражение `pos->next` уже указывало бы на `n`, и `n->next = pos->next` замкнуло бы узел **сам на себя**: хвост `[3]` потерян (утечка), а обход зациклился бы. Практика З5 разбирает эту аварию подробно.

*Пример 2. Удаление: голова и середина — два случая.*

```c
/* Node, push_front, print_list, list_free — как в §9.1.2 */

int list_remove(Node **head, int x)    /* 1 — удалён, 0 — не найден */
{
    Node *prev = NULL;
    Node *cur = *head;
    while (cur != NULL) {
        if (cur->value == x) {
            if (prev == NULL)
                *head = cur->next;      /* удаляем голову */
            else
                prev->next = cur->next; /* перецепляем мимо cur */
            free(cur);
            return 1;
        }
        prev = cur;
        cur = cur->next;
    }
    return 0;
}

int main(void)
{
    Node *list = NULL;
    for (int x = 3; x >= 1; x--)
        if (push_front(&list, x) != 0) { list_free(list); return 1; }
    print_list(list);
    printf("remove(2) = %d\n", list_remove(&list, 2));
    print_list(list);
    printf("remove(1) = %d\n", list_remove(&list, 1));   /* случай головы */
    print_list(list);
    printf("remove(99) = %d\n", list_remove(&list, 99));
    print_list(list);
    list_free(list);
    return 0;
}
```

Вывод:

```
1 -> 2 -> 3 -> NULL
remove(2) = 1
1 -> 3 -> NULL
remove(1) = 1
3 -> NULL
remove(99) = 0
3 -> NULL
```

Пара `prev`/`cur` шагает по списку вместе; на удалении срабатывает один из двух случаев: голова (`prev == NULL` — переставить сам `*head`, для этого функции и нужен `Node **`) или середина/хвост (`prev->next` перепрыгивает через `cur`). После перецепки узел освобождается — он больше недостижим, не освободить его значило бы утечку. Функция честно сообщает результат: `remove(99)` вернул 0, список не тронут.

**Связь с практикой.** Таблица цен из определений — практический инструмент выбора: часты вставки/удаления в произвольных местах при редком доступе по номеру — список; частый доступ по индексу и добавления в конец — вектор (и по умолчанию — тоже вектор: непрерывная память быстрее на современных процессорах). Этот же анализ «что почём» станет привычкой на всех структурах: у стека и очереди все операции дешёвые, но набор их урезан (§9.2); хеш-таблица покупает быстрый поиск ценой памяти и хеш-функции (§9.3.1); дерево — ценой сбалансированности (§9.3.2). В Rust выбор тот же (`Vec` против `LinkedList`), и документация стандартной библиотеки прямо приводит такую же таблицу (гл. 17).

**Типичные ошибки и подводные камни.**

- **Обратный порядок присваиваний в `insert_after`** — потеря хвоста и самозацикленный узел (разбор в примере 1, практика З5). Сначала `n->next = pos->next`, потом `pos->next = n` — всегда.
- **Удаление без `prev`.** Дойдя до `cur`, уже нельзя узнать, кто на него указывает, — перецеплять нечем. Либо пара `prev`/`cur` (как здесь), либо приём «указатель на указатель» (за пределами главы).
- **Не обработать удаление головы.** `prev == NULL` — отдельная ветка: переставляется сам `*head`. Забыли — и удаление первого элемента оставит голову висячей (она укажет на освобождённый узел).
- **`free` до перецепки.** Освободили `cur`, а потом читаете `cur->next` для перецепки — use-after-free. Порядок жёсткий: перецепить → `free`.
- **Смешивать «шаги» и микросекунды.** Таблица цен считает обращения к элементам; реальное время зависит ещё и от кеша (непрерывный массив быстрее «прыгающего» по куче списка). Таблица — о росте стоимости с размером, а не о наносекундах; при сомнениях — измеряйте.

**Источники части 9.1:** C2 (гл. 17 — dynamic storage allocation, linked lists: вставка, удаление); C8 (лекции о списках и динамических массивах); C7 (нед. 5 — data structures); C4 (динамическая память, владение); C6.

## Часть 9.2. Стек и очередь

### §9.2.1. Стек: LIFO

**Определения.**

- **Стек** (stack) — структура с дисциплиной доступа **LIFO** (last in, first out — «последним пришёл, первым ушёл») и тремя операциями: **`push`** (положить сверху), **`pop`** (снять верхний), **`empty`** (пуст ли). Доступа к элементам «в глубине» нет — только к **верхушке** (top).
- Стек — это **ограничение интерфейса**, а не новая раскладка памяти: его строят и поверх массива (верхушка — конец массива, `data[len-1]`), и поверх списка (верхушка — голова). Версия на растущем массиве — это вектор §9.1.1, у которого `push` — тот же `vec_push`, а `pop` — «прочитать `data[len-1]` и уменьшить `len`».
- **Инвариант**: `pop` возвращает элементы строго в порядке, обратном `push`. `pop` пустого стека — ошибка, которую интерфейс должен явно сообщать (у нас — код возврата `-1`).
- Знакомство уже состоялось: **стек вызовов** (§4.2.2) — тот же принцип (кадр последней вызванной функции снимается первым); рекурсия (§4.3) — вычисление на неявном стеке.

**Интуиция.** Стопка тарелок: ставить и брать можно только сверху. Или обратный путь в лабиринте: заходя в коридор, кладёте камешек в стопку, возвращаясь — снимаете верхний; стопка всегда «помнит» дорогу назад в правильном — обратном — порядке. Всюду, где нужно «отложить и вернуться к последнему» — разбор вложенных скобок, отмена последнего действия (undo), обход вложенных папок — работает стек.

```
   push(10); push(20); push(30);        pop() -> 30
                                        pop() -> 20
        ┌────┐ ◀── верхушка (top)       pop() -> 10
        │ 30 │     data[len-1]
        ├────┤
        │ 20 │     на массиве: push/pop двигают len,
        ├────┤     элементы не сдвигаются никогда
        │ 10 │ ◀── дно
        └────┘
```

**Разобранные примеры.**

*Пример 1. Стек на растущем массиве.*

```c
#include <stdio.h>
#include <stdlib.h>

typedef struct {
    int *data;
    int len;
    int cap;
} Stack;

int stack_init(Stack *s)
{
    s->len = 0;
    s->cap = 2;
    s->data = malloc(s->cap * sizeof *s->data);
    return s->data == NULL ? -1 : 0;
}

int stack_push(Stack *s, int x)        /* положить сверху */
{
    if (s->len == s->cap) {
        int *tmp = realloc(s->data, 2 * s->cap * sizeof *s->data);
        if (tmp == NULL) return -1;
        s->data = tmp;
        s->cap *= 2;
    }
    s->data[s->len++] = x;
    return 0;
}

int stack_pop(Stack *s, int *out)      /* снять верхний; -1 — стек пуст */
{
    if (s->len == 0) return -1;
    *out = s->data[--s->len];
    return 0;
}

int stack_empty(const Stack *s)
{
    return s->len == 0;
}

void stack_free(Stack *s)
{
    free(s->data);
    s->data = NULL;
    s->len = s->cap = 0;
}

int main(void)
{
    Stack s;
    if (stack_init(&s) != 0) return 1;
    for (int i = 1; i <= 3; i++) {
        if (stack_push(&s, i * 10) != 0) { stack_free(&s); return 1; }
        printf("push %d\n", i * 10);
    }
    int x;
    while (stack_pop(&s, &x) == 0)
        printf("pop  %d\n", x);
    printf("empty: %d\n", stack_empty(&s));
    stack_free(&s);
    return 0;
}
```

Вывод:

```
push 10
push 20
push 30
pop  30
pop  20
pop  10
empty: 1
```

Это вектор §9.1.1 под другим именем и с урезанным интерфейсом: `push` — тот же рост с удвоением, `pop` — `data[--len]` (уменьшили длину и отдали то, что было верхушкой). Элементы никогда не сдвигаются — обе операции дешевле некуда. Результат `pop` отдаётся через выходной параметр, а код возврата отличает «снял значение» от «стек пуст» — так же, как `make_range` в §7.1.2 разделяла значение и успех.

*Пример 2. Классика стека: проверка скобок.* Согласованы ли `()`, `[]`, `{}` в строке:

```c
/* Stack, stack_init, stack_push, stack_pop, stack_empty, stack_free —
   как в примере 1 */

int match(int open, int close)         /* парная ли пара скобок */
{
    return (open == '(' && close == ')') ||
           (open == '[' && close == ']') ||
           (open == '{' && close == '}');
}

int balanced(const char *s)            /* 1 — скобки согласованы */
{
    Stack st;
    if (stack_init(&st) != 0) return 0;
    int ok = 1;
    for (int i = 0; ok && s[i] != '\0'; i++) {
        char c = s[i];
        if (c == '(' || c == '[' || c == '{') {
            if (stack_push(&st, c) != 0)
                ok = 0;                /* нет памяти */
        } else if (c == ')' || c == ']' || c == '}') {
            int open;
            if (stack_pop(&st, &open) != 0 || !match(open, c))
                ok = 0;                /* закрыли то, чего не открывали */
        }
    }
    if (!stack_empty(&st))
        ok = 0;                        /* остались незакрытые */
    stack_free(&st);
    return ok;
}

int main(void)
{
    printf("%s -> %d\n", "({[]})", balanced("({[]})"));
    printf("%s -> %d\n", "(a[b]{c})", balanced("(a[b]{c})"));
    printf("%s -> %d\n", "([)]", balanced("([)]"));
    printf("%s -> %d\n", "((", balanced("(("));
    printf("%s -> %d\n", ")", balanced(")"));
    return 0;
}
```

Вывод:

```
({[]}) -> 1
(a[b]{c}) -> 1
([)] -> 0
(( -> 0
) -> 0
```

Алгоритм: открывающая скобка — `push`; закрывающая — `pop` и сверка пары; прочие символы игнорируются. Три способа провалиться: `pop` из пустого стека (закрыли, не открыв — тест `")"`), пара не совпала (тест `"([)]"`: снимаем `[`, а закрывают `)`), стек не пуст в конце (незакрытые — тест `"(("`). Почему именно стек? Скобки закрываются **в порядке, обратном открытию** — «последняя открытая закрывается первой» — это дословно LIFO. Тот же принцип разбирает вложенные конструкции в компиляторах и проверяет корректность любых «открыть/закрыть» (файлы, теги, блоки).

**Связь с практикой.** Стек — самая частая «идейная» структура: отмена действий (undo/redo — два стека), история переходов браузера, обход вложенных структур без рекурсии, вычисление выражений (стек операндов), стек вызовов в самом языке (§4.2.2 — переполнение его глубокой рекурсией мы уже видели). Правило узнавания: если в задаче звучит «вернуться к последнему…», «отменить последнее…», «самая внутренняя/последняя открытая…» — это стек. В Rust отдельного типа нет и не нужно: `Vec` с его `push`/`pop` — готовый стек (гл. 17).

**Типичные ошибки и подводные камни.**

- **`pop` без проверки пустоты.** `data[--len]` при `len == 0` уводит индекс в `-1` — чтение вне блока (heap-buffer-underflow под ASan). Интерфейс обязан отличать «пусто» — у нас код `-1`.
- **Читать `data[len]` как верхушку.** Верхушка — `data[len-1]`; `data[len]` — незанятый запас (мусор). Ошибка на единицу в самом чувствительном месте.
- **Смешивать снятие и подглядывание.** `pop` *удаляет*, «посмотреть верхушку, не снимая» — отдельная операция (`peek`/`last`, — З8 части 9.1 делала её для вектора). Два `pop` подряд «чтобы посмотреть» теряют элемент.
- **Забыть про LIFO при выводе.** Стек *переворачивает* порядок: сложили 1, 2, 3 — сняли 3, 2, 1. Это его суть, а не побочный эффект; нужен исходный порядок — нужна очередь (§9.2.2).
- **Утечка при ошибочном пути.** В `balanced` любой исход проходит через `stack_free` (флаг `ok` вместо ранних `return`) — иначе выход при несовпадении пары утёк бы вместе со стеком. Приём «один выход после очистки» — стандарт для функций с ресурсами.

**Источники:** C2 (гл. 17 — динамические структуры; стек как связная структура); C8 (лекции о структурах данных — стек); C7 (нед. 5 — stacks); C4 (владение ресурсами); C6.

### §9.2.2. Очередь: FIFO

**Определения.**

- **Очередь** (queue) — дисциплина **FIFO** (first in, first out — «первым пришёл, первым ушёл»): **`enqueue`** добавляет в **хвост**, **`dequeue`** извлекает из **головы**. Порядок извлечения совпадает с порядком добавления.
- **Очередь на списке**: узлы те же, что в §9.1.2, но структура держит **два** указателя — `head` (откуда брать) и `tail` (куда класть). Благодаря `tail` вставка в конец — 2 присваивания, а не проход по всему списку (сноска ¹ таблицы §9.1.3). Инварианты: оба `NULL` — очередь пуста; иначе `tail` — последний узел, и `tail->next == NULL`. Особые случаи: `enqueue` в пустую (оба указателя ставятся на новый узел) и `dequeue` последнего (оба сбрасываются в `NULL`).
- **Кольцевой буфер** (ring buffer, circular buffer) — очередь **фиксированной ёмкости** на массиве **без единого `malloc`**: индексы головы и хвоста двигаются по массиву по кругу — арифметика **по модулю** `% CAP` (остаток от деления, гл. 2) «заворачивает» их с конца на начало. Состояние — `head` (индекс первого занятого) и `len`; хвост вычисляется: `(head + len) % CAP`. Переполнение (`len == CAP`) — отказ (наш вариант) или затирание старейшего (вариант для телеметрии).

**Интуиция.** Очередь — она и есть очередь: в магазине встают в хвост, обслуживают из головы, и кто раньше встал, тот раньше ушёл. Указатель `tail` — это «конец очереди, вставать сюда»: без него каждому новому пришлось бы пройти вдоль всей очереди, чтобы найти, за кем встать. Кольцевой буфер — карусель с `CAP` креслами: посадка и высадка идут по кругу; когда «последнее» кресло позади, следующее — снова нулевое (`% CAP`). Кресла привинчены (память не выделяется и не освобождается) — крутятся только два числа.

```
   очередь на списке:
      head ─▶ [10] ─▶ [20] ─▶ [30] ◀─ tail
   dequeue ↑ (из головы)          ↑ enqueue (в хвост, tail->next = new)

   кольцевой буфер CAP=4 после: put 1,2,3,4  get,get  put 5, put 6
      индекс:   0    1    2    3
              ┌────┬────┬────┬────┐
              │ 5  │ 6  │ 3  │ 4  │    head=2, len=4
              └────┴────┴────┴────┘
      логический порядок: 3, 4, 5, 6 — хвост «завернулся» на 0 и 1
```

**Разобранные примеры.**

*Пример 1. Очередь на списке с головой и хвостом.*

```c
#include <stdio.h>
#include <stdlib.h>

typedef struct node {
    int value;
    struct node *next;
} Node;

typedef struct {
    Node *head;    /* откуда берём (dequeue) */
    Node *tail;    /* куда кладём (enqueue) */
} Queue;

void queue_init(Queue *q)
{
    q->head = NULL;
    q->tail = NULL;
}

int enqueue(Queue *q, int x)           /* в конец очереди */
{
    Node *n = malloc(sizeof *n);
    if (n == NULL) return -1;
    n->value = x;
    n->next = NULL;
    if (q->tail == NULL)               /* очередь была пуста */
        q->head = q->tail = n;
    else {
        q->tail->next = n;             /* старый хвост цепляется к новому */
        q->tail = n;
    }
    return 0;
}

int dequeue(Queue *q, int *out)        /* из начала; -1 — пусто */
{
    if (q->head == NULL) return -1;
    Node *n = q->head;
    *out = n->value;
    q->head = n->next;
    if (q->head == NULL)               /* забрали последний узел */
        q->tail = NULL;
    free(n);
    return 0;
}

void queue_free(Queue *q)
{
    int x;
    while (dequeue(q, &x) == 0) { }    /* выбрать всё */
}

int main(void)
{
    Queue q;
    queue_init(&q);
    for (int i = 1; i <= 3; i++) {
        if (enqueue(&q, i * 10) != 0) { queue_free(&q); return 1; }
        printf("enqueue %d\n", i * 10);
    }
    int x;
    while (dequeue(&q, &x) == 0)
        printf("dequeue %d\n", x);
    return 0;
}
```

Вывод:

```
enqueue 10
enqueue 20
enqueue 30
dequeue 10
dequeue 20
dequeue 30
```

FIFO как есть: вошли 10, 20, 30 — вышли 10, 20, 30 (сравните со стеком из §9.2.1 на тех же данных). Вся тонкость — в двух особых случаях. `enqueue` в пустую очередь ставит **оба** указателя на новый узел; `dequeue` последнего замечает, что `head` стал `NULL`, и сбрасывает `tail` — иначе тот остался бы висячим указателем на освобождённый узел, и следующий `enqueue` записал бы в чужую память (практика З6 разбирает этот сценарий). `queue_free` элегантно переиспользует `dequeue`: «выбрать всё» — и узлы освобождены.

*Пример 2. Кольцевой буфер: очередь без `malloc`.*

```c
#include <stdio.h>

enum { CAP = 4 };

typedef struct {
    int data[CAP];
    int head;      /* индекс первого занятого */
    int len;       /* сколько занято */
} Ring;

int ring_put(Ring *r, int x)      /* 0 — успех, -1 — буфер полон */
{
    if (r->len == CAP) return -1;
    r->data[(r->head + r->len) % CAP] = x;   /* хвост с заворотом */
    r->len++;
    return 0;
}

int ring_get(Ring *r, int *out)   /* 0 — успех, -1 — пусто */
{
    if (r->len == 0) return -1;
    *out = r->data[r->head];
    r->head = (r->head + 1) % CAP;           /* голова с заворотом */
    r->len--;
    return 0;
}

int main(void)
{
    Ring r = {0};
    for (int i = 1; i <= 5; i++)
        printf("put %d -> %s\n", i, ring_put(&r, i) == 0 ? "ok" : "FULL");
    int x;
    if (ring_get(&r, &x) == 0) printf("get -> %d\n", x);
    if (ring_get(&r, &x) == 0) printf("get -> %d\n", x);
    printf("put 5 -> %s\n", ring_put(&r, 5) == 0 ? "ok" : "FULL");
    printf("put 6 -> %s\n", ring_put(&r, 6) == 0 ? "ok" : "FULL");
    while (ring_get(&r, &x) == 0)
        printf("get -> %d\n", x);
    return 0;
}
```

Вывод:

```
put 1 -> ok
put 2 -> ok
put 3 -> ok
put 4 -> ok
put 5 -> FULL
get -> 1
get -> 2
put 5 -> ok
put 6 -> ok
get -> 3
get -> 4
get -> 5
get -> 6
```

Пятый `put` в полный буфер честно отвергнут. После двух `get` голова ушла на индекс 2 — и новые `put 5`, `put 6` записались по индексам `(2+2)%4 = 0` и `(2+3)%4 = 1`: хвост **завернулся** на освободившееся начало массива (ровно картинка из интуиции). Финальная выборка выдаёт логический порядок `3 4 5 6` — FIFO соблюдён, хотя физически элементы лежат «с разрывом». Ни одного `malloc`: вся память — поле `data[CAP]` в структуре; поэтому кольцевые буферы живут там, где куча недоступна или дорога — драйверы, встроенные системы, буферы клавиатуры и звука.

**Связь с практикой.** Очередь — модель любой обработки «в порядке поступления»: задания печати, сообщения между потоками (в томе II — канал `mpsc`, гл. 23, это очередь и есть), события GUI (в главе 12 увидим: программа Windows построена вокруг цикла сообщений, обслуживающего очередь событий), запросы к серверу. Обход в ширину (BFS) — систематический осмотр графа «волнами» — устроен на очереди: посетил вершину — соседей в хвост (реализуем идею на дереве в замечании к §9.3.2 — обзорно). Правило узнавания зеркально стековому: «обработать в порядке поступления» — очередь; «вернуться к последнему» — стек.

**Типичные ошибки и подводные камни.**

- **Висячий `tail` после опустошения.** Забыли `q->tail = NULL` при выемке последнего — следующий `enqueue` сделает `q->tail->next = n` по **освобождённому** узлу (use-after-free, поймает ASan). Инвариант «пусто ⇔ оба NULL» обязателен.
- **`enqueue` в пустую очередь без особой ветки.** `q->tail->next` при `tail == NULL` — разыменование NULL (§6.1.2), падение. Оба особых случая (первый узел, последний узел) проверяйте в первую очередь.
- **Очередь на векторе с выемкой из начала.** «Просто удалю `data[0]` со сдвигом» — каждый `dequeue` стоит ~`n` шагов (§9.1.1, пример 2). Правильные варианты: список с хвостом или кольцевой буфер.
- **Ошибки модульной арифметики в кольце.** `head + len` без `% CAP` — выход за массив; `%` с отрицательными операндами в C коварен (гл. 2) — держите индексы неотрицательными, как здесь (только сложение и `%`).
- **Путать `len == CAP` и `head == хвост`.** В кольце «полон» и «пуст» дают одинаковое равенство индексов — различать их надо отдельным счётчиком `len` (наш способ) или жертвовать одной ячейкой. Хранение явного `len` — простейший корректный вариант.

**Источники части 9.2:** C2 (гл. 17 — связные структуры как основа стека и очереди); C8 (лекции о структурах данных — стеки, очереди); C7 (нед. 5 — stacks, queues); C4 (ресурсы, владение); C6.

## Часть 9.3. Хеш-таблица и двоичное дерево поиска

> Часть помечена в программе курса как **«повышенная сложность»**: здесь соединяется всё разом — строки, владение копиями ключей, массивы указателей, списки-цепочки, рекурсия. Не торопитесь; рисуйте схемы к каждому примеру.

### §9.3.1. Хеш-таблица: поиск по ключу за «почти один шаг»

**Определения.**

- Задача: хранить пары **ключ → значение** (у нас: строка → `int`) так, чтобы поиск по ключу не требовал прохода по всем записям (список/массив дают только до `n` шагов, §9.1.3).
- **Хеш-функция** превращает ключ в число — **номер корзины** (bucket) в массиве фиксированного размера. Мы берём классическую функцию из K&R (C1, гл. 6): накопление `h = 31*h + код_символа` по всем символам, затем **`% NBUCKETS`** — остаток от деления укладывает произвольное число в диапазон `0..NBUCKETS-1`:

  ```c
  unsigned hash(const char *s)
  {
      unsigned h = 0;
      for (int i = 0; s[i] != '\0'; i++)
          h = 31 * h + (unsigned char)s[i];
      return h % NBUCKETS;
  }
  ```

  Тип `unsigned` не даёт накоплению уйти в отрицательные значения (переполнение беззнаковых определено как «по модулю», гл. 2); приведение `(unsigned char)` страхует от платформ со знаковым `char`. Функция **детерминирована**: один ключ — всегда одна корзина.
- **Коллизия** — два разных ключа с одной корзиной (при `NBUCKETS` корзин и большем числе ключей коллизии неизбежны). Разрешение **цепочками** (chaining): корзина — не одна запись, а **связный список** записей (§9.1.2); поиск в корзине сверяет ключи `strcmp` (гл. 5).
- **Таблица** — массив голов цепочек: `Entry *bucket[NBUCKETS];`. Запись (`Entry`) хранит **собственную копию ключа** (`str_dup` — куча, гл. 7): таблица не зависит от времени жизни переданной ей строки. Владение: таблица владеет и записями, и копиями ключей — `table_free` освобождает и то и другое.
- Операции: `table_get` (найти адрес значения по ключу или `NULL`) и `table_put` (обновить существующее или добавить новую запись в начало цепочки — `push_front` §9.1.2). При равномерном хеше средняя цепочка коротка — поиск стоит «хеш + пара сверок», почти не зависящих от числа записей.

**Интуиция.** Гардероб с `NBUCKETS` крючками. Хеш-функция — гардеробщик, который по имени владельца **вычисляет** номер крючка — всегда один и тот же для одного имени. Придя за курткой, вы не осматриваете весь гардероб: гардеробщик пересчитал имя в номер, и искать нужно только среди курток **одного крючка** (коллизии — на одном крючке несколько курток, отличаем по биркам-`strcmp`). Чем больше крючков и чем равномернее гардеробщик их раздаёт, тем короче осмотр.

```
   ключ ──▶ hash(ключ) ──▶ номер корзины 0..NBUCKETS-1

   bucket[0] ─▶ NULL
   bucket[4] ─▶ [dog=2] ─▶ NULL
   bucket[6] ─▶ [tac=3] ─▶ [cat=1] ─▶ NULL     ← коллизия: цепочка §9.1.2
   bucket[7] ─▶ NULL

   поиск "cat": hash("cat")=6 → идём только по цепочке корзины 6,
                strcmp с каждым ключом: "tac"? нет. "cat"? да!
```

**Разобранные примеры.**

*Пример 1. Хеш-функция в действии.*

```c
#include <stdio.h>

enum { NBUCKETS = 8 };

unsigned hash(const char *s)           /* хеш K&R: h = 31*h + символ */
{
    unsigned h = 0;
    for (int i = 0; s[i] != '\0'; i++)
        h = 31 * h + (unsigned char)s[i];
    return h % NBUCKETS;               /* номер корзины 0..NBUCKETS-1 */
}

int main(void)
{
    printf("hash(\"cat\") = %u\n", hash("cat"));
    printf("hash(\"dog\") = %u\n", hash("dog"));
    printf("hash(\"tac\") = %u\n", hash("tac"));
    printf("hash(\"act\") = %u\n", hash("act"));
    printf("hash(\"\")    = %u\n", hash(""));
    return 0;
}
```

Вывод:

```
hash("cat") = 6
hash("dog") = 4
hash("tac") = 6
hash("act") = 2
hash("")    = 0
```

Проследим `"cat"` вручную (коды ASCII: `c`=99, `a`=97, `t`=116): `h = 99` → `h = 31·99 + 97 = 3166` → `h = 31·3166 + 116 = 98262`; `98262 % 8 = 6`. Значения зависят только от алгоритма — на любой машине с 32-битным `unsigned` выйдут те же числа. Заметьте пару `"cat"`/`"tac"`: разные строки, одна корзина 6 — **коллизия** (а анаграмма `"act"` попала в 2: множитель 31 учитывает *порядок* символов, не только их набор). Пустая строка законна: цикл не выполнится, `h = 0`.

*Пример 2. Таблица целиком: вставка, поиск, коллизия, обновление.*

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

enum { NBUCKETS = 8 };

typedef struct entry {
    char *key;              /* собственная копия строки-ключа */
    int value;
    struct entry *next;     /* цепочка коллизий */
} Entry;

typedef struct {
    Entry *bucket[NBUCKETS];
} Table;

/* hash — как в примере 1 */

char *str_dup(const char *s)           /* копия строки в куче (Г7.Ч1.З11) */
{
    char *copy = malloc(strlen(s) + 1);
    if (copy != NULL)
        strcpy(copy, s);
    return copy;
}

void table_init(Table *t)
{
    for (int i = 0; i < NBUCKETS; i++)
        t->bucket[i] = NULL;
}

int *table_get(Table *t, const char *key)     /* адрес значения или NULL */
{
    for (Entry *e = t->bucket[hash(key)]; e != NULL; e = e->next)
        if (strcmp(e->key, key) == 0)         /* хеш совпал — сверяем ключ */
            return &e->value;
    return NULL;
}

int table_put(Table *t, const char *key, int value)
{
    int *v = table_get(t, key);
    if (v != NULL) {                   /* ключ уже есть — обновляем */
        *v = value;
        return 0;
    }
    Entry *e = malloc(sizeof *e);      /* нового ключа нет — создаём запись */
    if (e == NULL) return -1;
    e->key = str_dup(key);
    if (e->key == NULL) { free(e); return -1; }
    e->value = value;
    unsigned h = hash(key);
    e->next = t->bucket[h];            /* push_front в цепочку корзины */
    t->bucket[h] = e;
    return 0;
}

void table_free(Table *t)
{
    for (int i = 0; i < NBUCKETS; i++) {
        Entry *e = t->bucket[i];
        while (e != NULL) {
            Entry *next = e->next;
            free(e->key);              /* сначала ключ */
            free(e);                   /* потом запись */
            e = next;
        }
        t->bucket[i] = NULL;
    }
}

void table_print(const Table *t)
{
    for (int i = 0; i < NBUCKETS; i++) {
        printf("[%d]:", i);
        for (const Entry *e = t->bucket[i]; e != NULL; e = e->next)
            printf(" %s=%d", e->key, e->value);
        printf("\n");
    }
}

int main(void)
{
    Table t;
    table_init(&t);
    if (table_put(&t, "cat", 1) != 0 ||
        table_put(&t, "dog", 2) != 0 ||
        table_put(&t, "tac", 3) != 0) { table_free(&t); return 1; }

    table_print(&t);

    int *v = table_get(&t, "cat");
    if (v != NULL) printf("cat -> %d\n", *v);
    printf("cow: %s\n", table_get(&t, "cow") == NULL ? "not found" : "found");

    if (table_put(&t, "cat", 10) != 0) { table_free(&t); return 1; }
    v = table_get(&t, "cat");
    if (v != NULL) printf("cat -> %d (updated)\n", *v);

    table_free(&t);
    return 0;
}
```

Вывод:

```
[0]:
[1]:
[2]:
[3]:
[4]: dog=2
[5]:
[6]: tac=3 cat=1
[7]:
cat -> 1
cow: not found
cat -> 10 (updated)
```

Всё, что мы строили, сошлось в одном типе: **массив** корзин, в каждой — **связный список** (§9.1.2) записей-**структур** (гл. 8), ключи — **копии строк в куче** (гл. 5, 7). В корзине 6 видна цепочка коллизии: `tac` и `cat` (новая запись встаёт в начало — `push_front`). `table_get` возвращает **адрес значения** (`int *`) — через него `table_put` обновляет запись без повторного поиска, а вызывающий отличает «нет ключа» (`NULL`) от любого значения. Поиск `"cow"` заглянул только в свою корзину — прочие цепочки не тронуты; в этом и есть выигрыш таблицы.

**Связь с практикой.** Хеш-таблица — «рабочая лошадка» поиска по ключу: словари и индексы, кеши, подсчёт частот (практика З8), таблицы символов в компиляторах (имя переменной → информация о ней), заголовки HTTP, переменные окружения. Инженерные ручки, которые вы теперь понимаете изнутри: число корзин (у нас 8 для наглядности; в реальных таблицах оно растёт вместе с данными — «rehash»), качество хеш-функции (равномерность = короткие цепочки), стоимость `strcmp` на длинных ключах. В Rust — это `HashMap` (гл. 17): та же идея, но с автоматическим ростом и защитой от злонамеренно подобранных ключей; наш `table_get`, возвращающий `int *`, там превратится в `Option<&mut V>` — «может быть, ссылка» вместо «может быть, NULL».

**Типичные ошибки и подводные камни.**

- **Сравнивать только хеши.** Равенство корзин не значит равенство ключей (`"cat"`/`"tac"`!) — после хеша обязательна сверка `strcmp`. Пропустите — и таблица начнёт «находить» чужие записи.
- **Хранить чужой указатель на ключ.** `e->key = key;` без копии — записи зависят от времени жизни строки вызывающего: тот перезаписал буфер — все ключи «поехали». Таблица делает **свою** копию (`str_dup`) и сама её освобождает.
- **Утечка ключей в `table_free`.** Освободить записи, забыв `free(e->key)`, — по утечке на каждую запись (LeakSanitizer перечислит все). Владеешь двумя блоками — освобождаешь оба, в порядке «содержимое, потом контейнер».
- **`int` вместо `unsigned` в хеше.** Накопление переполнит `int` — а это неопределённое поведение (гл. 2, 13); у `unsigned` переполнение определено («по модулю»), потому хеш-функции пишут на беззнаковых.
- **Индекс корзины из знакового остатка.** Если бы `h` был знаковым и отрицательным, `h % NBUCKETS` мог быть отрицательным индексом — выход за массив. Наша `hash` возвращает `unsigned` — диапазон гарантирован.

**Источники:** C1 (гл. 6 — table lookup: хеш-таблица с цепочками, хеш-функция с множителем 31); C7 (нед. 5 — hash tables); C8 (лекции о структурах данных); C2 (гл. 17 — связные структуры); C5 (беззнаковая арифметика — §6.2.5); C6.

### §9.3.2. Двоичное дерево поиска

**Определения.**

- **Двоичное дерево** — связная структура, где узел указывает не на одного «следующего», а на **двух потомков**:

  ```c
  typedef struct tnode {
      int value;
      struct tnode *left;    /* всё, что меньше value */
      struct tnode *right;   /* всё, что больше value */
  } TNode;
  ```

  Верхний узел — **корень** (root); узел без потомков (`left == right == NULL`) — **лист**; пустое дерево — `NULL`.
- **Двоичное дерево поиска** (binary search tree, BST) добавляет **инвариант порядка**: в левом поддереве любого узла — только значения **меньше** его, в правом — только **больше** (наш вариант дубликаты не хранит). Инвариант действует **на всех уровнях** сразу.
- **Поиск**: сравнить с узлом — меньше — влево, больше — вправо, равно — нашли, `NULL` — нет. Каждый шаг **отсекает целое поддерево**; в «кустистом» (сбалансированном) дереве путь короток (~log₂ n шагов: миллион значений — около 20 сравнений), в **вырожденном** (вставка в отсортированном порядке — каждый узел имеет лишь одного потомка) дерево превращается в список, и поиск стоит до `n` шагов.
- **Вставка**: тот же спуск до `NULL`-места, куда и подвешивается новый узел. Рекурсия по полю-указателю через `TNode **` позволяет и создать узел, и вернуть код результата (вставлен/дубликат/нет памяти).
- **Inorder-обход** («лево — корень — право», рекурсивно) посещает узлы BST **в возрастающем порядке** — прямое следствие инварианта. **Освобождение** — обход «потомки, потом узел» (post-order): нельзя `free`-ить узел раньше, чем пройдены его поддеревья.

**Интуиция.** BST — игра «больше-меньше». Загадано число; каждый узел говорит: «меньше меня — налево, больше — направо». Один вопрос отбрасывает **половину** оставшегося (если дерево кустистое) — потому 20 вопросов хватает на миллион. Дерево — это ещё и «список, у которого два хвоста»: тот же узел-структура с указателями, та же куча, только форма ветвится. А вырожденное дерево — поучительный анти-случай: подавали значения по возрастанию — правые «сцепки» вытянулись в цепочку, и вся мощь исчезла.

```
   вставка 5, 3, 8, 1, 4, 7, 9:              вырожденное: 1, 2, 3, 4, 5

              [5]                                [1]
            /     \                                 \
         [3]       [8]                              [2]
        /   \     /   \                                \
      [1]   [4] [7]   [9]                              [3]
                                                          \ ...
   поиск 4: 5? меньше → влево;
            3? больше → вправо;  [4] — нашли (3 шага)
   inorder: 1 3 4 5 7 8 9  — отсортировано!
```

**Разобранные примеры.**

*Пример 1. Вставка и поиск.*

```c
#include <stdio.h>
#include <stdlib.h>

typedef struct tnode {
    int value;
    struct tnode *left;    /* всё, что меньше value */
    struct tnode *right;   /* всё, что больше value */
} TNode;

int bst_insert(TNode **rootp, int x)   /* 1 — вставлен, 0 — дубликат, -1 — нет памяти */
{
    if (*rootp == NULL) {              /* свободное место найдено */
        TNode *n = malloc(sizeof *n);
        if (n == NULL) return -1;
        n->value = x;
        n->left = NULL;
        n->right = NULL;
        *rootp = n;
        return 1;
    }
    if (x < (*rootp)->value)
        return bst_insert(&(*rootp)->left, x);    /* меньшие — влево */
    if (x > (*rootp)->value)
        return bst_insert(&(*rootp)->right, x);   /* большие — вправо */
    return 0;                                     /* равный уже есть */
}

const TNode *bst_search(const TNode *root, int x)
{
    if (root == NULL) return NULL;         /* дошли до низа — значения нет */
    if (x < root->value) return bst_search(root->left, x);
    if (x > root->value) return bst_search(root->right, x);
    return root;                           /* нашли */
}

void bst_free(TNode *root)
{
    if (root == NULL) return;
    bst_free(root->left);      /* сначала оба поддерева */
    bst_free(root->right);
    free(root);                /* потом сам узел */
}

int main(void)
{
    TNode *root = NULL;
    int keys[] = {5, 3, 8, 1, 4};
    for (int i = 0; i < 5; i++)
        if (bst_insert(&root, keys[i]) < 0) { bst_free(root); return 1; }

    printf("search(4): %s\n", bst_search(root, 4) != NULL ? "yes" : "no");
    printf("search(7): %s\n", bst_search(root, 7) != NULL ? "yes" : "no");
    printf("insert(3) again: %d\n", bst_insert(&root, 3));
    bst_free(root);
    return 0;
}
```

Вывод:

```
search(4): yes
search(7): no
insert(3) again: 0
```

Обе функции — рекурсия (§4.3) в чистом виде: базовый случай — `NULL` (место для вставки / «не найдено»), шаг — спуск влево или вправо; дерево — рекурсивная структура, и алгоритмы на нём естественно рекурсивны. `bst_insert` принимает `TNode **rootp` — адрес того **поля** (или головы), куда, возможно, придётся подвесить узел: `&(*rootp)->left` — это адрес поля `left` текущего узла (§6.3.2 в новой роли). Так функция умеет и модифицировать дерево, и честно сообщить исход: `1` — вставлен, `0` — дубликат (второй `insert(3)` дерево не изменил), `-1` — нет памяти.

*Пример 2. Inorder: дерево сортирует само.*

```c
/* TNode, bst_insert, bst_free — как в примере 1 */

void bst_inorder(const TNode *root)    /* лево — корень — право */
{
    if (root == NULL) return;
    bst_inorder(root->left);
    printf("%d ", root->value);
    bst_inorder(root->right);
}

int main(void)
{
    TNode *root = NULL;
    int keys[] = {5, 3, 8, 1, 4, 7, 9};
    for (int i = 0; i < 7; i++)
        if (bst_insert(&root, keys[i]) < 0) { bst_free(root); return 1; }
    bst_inorder(root);                 /* печатает по возрастанию */
    printf("\n");
    bst_free(root);
    return 0;
}
```

Вывод:

```
1 3 4 5 7 8 9 
```

Вставляли вразнобой (`5 3 8 1 4 7 9`) — обход напечатал по возрастанию. Почему: для каждого узла всё меньшее лежит слева, всё большее справа; «сначала всё левое, потом узел, потом всё правое» на каждом уровне и даёт отсортированный порядок (практика З6 просит проговорить это рассуждение). Сравните с `bst_free` из примера 1 — там порядок другой: узел освобождается **после** поддеревьев (post-order), иначе `root->left` читался бы из уже освобождённого узла. Замечание обзорно: обход **в ширину** (уровень за уровнем) на дереве строится с очередью §9.2.2 — вершина из головы, потомков в хвост; это тот самый BFS из «связи с практикой» §9.2.2.

**Связь с практикой.** BST — третий ответ на вопрос «как искать быстро» (после «отсортированный массив» и «хеш-таблица»), и единственный из них, который держит данные **упорядоченными всё время**: минимум/максимум, «все ключи от A до B», обход по возрастанию — бесплатно (inorder). Цена — балансировка: наивное BST вырождается на отсортированном входе (практика З7, З9); промышленные деревья (красно-чёрные, AVL, B-деревья в базах данных и файловых системах) балансируют себя сами — их устройство за пределами курса, но их **интерфейс и инвариант** — ровно наши. Выбор структуры теперь осознан: только упорядоченность → вектор+сортировка; быстрый поиск без порядка → хеш-таблица; быстрый поиск **плюс** порядок → дерево. В Rust это `BTreeMap` против `HashMap` (гл. 17) — та же развилка.

**Типичные ошибки и подводные камни.**

- **Нарушить инвариант «на всех уровнях».** Проверять «левый потомок меньше родителя» недостаточно: **всё** левое поддерево должно быть меньше узла. Один «неместный» узел глубоко в поддереве — и поиск, честно идущий по сравнениям, его никогда не найдёт.
- **`free` узла до поддеревьев.** `free(root); bst_free(root->left);` — use-after-free: поле `left` читается из освобождённой памяти. Только post-order: дети — потом родитель (`list_free` §9.1.2 — тот же принцип «сначала запомни, потом освобождай»).
- **Потерять новый узел.** Вставка «нашла место, `malloc`-нула узел, но не записала его в поле родителя» (забыт `*rootp = n`) — узел утёк, дерево не изменилось. Через `TNode **` запись `*rootp = n` попадает точно в нужное поле.
- **Забыть, что дубликаты — решение проектировщика.** Наш вариант молча не вставляет равные (`return 0`); другие варианты — счётчик в узле или разрешить равные вправо. Что выбрано — должно быть задокументировано и проверено, иначе «пропавшие» вставки станут сюрпризом.
- **Глубокая рекурсия на вырожденном дереве.** Дерево-«список» из миллиона узлов — миллион вложенных вызовов `bst_insert`/`bst_free` — переполнение стека вызовов (§4.3). Ещё один довод в пользу сбалансированности; итеративные варианты обхода существуют (мин/макс — практика З10 — уже итеративны).

**Источники части 9.3:** C1 (гл. 6 — self-referential structures: двоичное дерево; table lookup: хеш-таблица); C7 (нед. 5 — hash tables, trees); C8 (лекции о списках, деревьях, хеш-таблицах); C2 (гл. 17 — связные структуры); C5 (беззнаковая арифметика); C6.

## Резюме главы

- **Динамический массив (вектор)** — `data`/`len`/`cap` + удвоение через `realloc`: дешёвые доступ по индексу и добавление в конец (амортизированно), дорогие вставки в начало/середину (сдвиг хвоста). Структура «по умолчанию»; инвариант `len <= cap`, граница доступа — `len`.
- **Односвязный список** — узлы в куче, сцепленные `next`; голова представляет список, пустой список — `NULL`. Вставка/удаление у **известного места** — пара присваиваний (порядок перецепки критичен!), доступ по номеру и поиск — линейные. Функциям, меняющим голову, — `Node **`; освобождение — «запомнить `next` до `free`».
- **Стек (LIFO)** — `push`/`pop`/`empty`, доступ только к верхушке; на массиве это вектор с `pop = data[--len]`. Узнаётся по «вернуться к последнему»: скобки, undo, стек вызовов. **Очередь (FIFO)** — `enqueue` в хвост (нужен указатель `tail`!), `dequeue` из головы; инвариант «пусто ⇔ оба NULL» (иначе висячий `tail` → use-after-free). **Кольцевой буфер** — очередь фиксированной ёмкости на массиве, индексы по кругу через `% CAP`, ни одного `malloc`.
- **Хеш-таблица** — массив корзин-цепочек: `hash(ключ) % NBUCKETS` выбирает корзину (беззнаковая арифметика!), `strcmp` доверяет только полному совпадению ключа; таблица владеет **копиями** ключей (`str_dup`) и освобождает «содержимое, потом контейнер». Поиск почти не зависит от числа записей — пока хеш равномерен и цепочки коротки.
- **BST** — узел с `left`/`right` и инвариантом «слева меньше, справа больше» на всех уровнях: поиск и вставка — спуск с отсечением поддерева на каждом шаге (в кустистом дереве ~log₂ n, в вырожденном — список, до n). Inorder даёт возрастающий порядок; освобождение — только post-order. Выбор: порядок не нужен → хеш-таблица; нужен → дерево.
- Всё построенное — комбинации трёх приёмов: **структура** (гл. 8) + **куча** (гл. 7) + **указатели** (гл. 6); корректность памяти всюду подтверждается **AddressSanitizer** (§7.2.2) — прогон под ним теперь часть каждого решения.

**Дальше.** Наши структуры живут, пока работает программа: выключили — данные исчезли. Глава 10 добавляет **персистентность**: файлы. Научимся открывать и закрывать потоки (`FILE *`, `fopen`/`fclose`), читать и писать текст (`fgets`/`fprintf`), работать с бинарными данными (`fread`/`fwrite`) и позиционироваться в файле (`fseek`) — и сохранённый на диск вектор или таблица переживут запуск. Пригодится всё из этой главы: буферы-векторы для чтения, строки-ключи, структуры-записи; а подводные камни сериализации структур (паддинг §8.2.3, порядок байтов §8.2.2) наконец сыграют в полный рост.
