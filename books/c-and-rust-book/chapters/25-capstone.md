## Часть 25.1. CLI-утилита: мини-grep

### §25.1.1. Чекпоинт 1: постановка задачи и рабочий черновик

**Техническое задание.** Программа `minigrep <запрос> <файл>` печатает в стандартный вывод все строки файла, содержащие запрос как подстроку. Успех — код возврата 0; ошибка (нет аргументов, файл не читается) — ненулевой код и понятное сообщение. Позже добавим регистронезависимый режим. Прообраз — утилита `grep`, живущая в Unix с 1970-х; наша версия нарочно минимальна, отсюда приставка «мини».

**Определения** (новое в этом чекпоинте).

- **`std::env::args()`** — итератор по аргументам командной строки (R9, `std::env`). Родной брат пары `argc`/`argv`, через которую аргументы получает C-программа (лично познакомимся в чекпоинте 5), но по-растовски: не сырой массив указателей, а итератор, отдающий владеющие `String`. Нулевой элемент — имя программы, как и `argv[0]` (R1, гл. 12.1). Одна оговорка: если аргумент содержит невалидный Unicode, `args()` паникует; для таких случаев есть `args_os()` (R1, гл. 12.1) — нам хватит простого варианта.
- **`str::lines()`** — итератор по строкам текста, разрезающий его по `\n` (и понимающий `\r\n` — привет текстовому режиму Windows из §10.2.1); терминаторы в отдаваемые ломтики не входят (R9, `str`).
- **`str::contains(подстрока)`** — есть ли в строке вхождение подстроки (R9, `str`). C-родня — `strstr`, тоже из чекпоинта 5.

Первый черновик — вся программа в `main`, ошибки пока «обрабатываются» паникой. Это осознанное решение, а не неряшливость: пока структура не устоялась, `expect` — строительные леса (§18.1.2), их разберём в следующем чекпоинте.

```rust
use std::env;
use std::fs;

fn main() {
    let args: Vec<String> = env::args().collect();

    let query = &args[1];
    let file_path = &args[2];

    println!("Ищем `{query}` в файле {file_path}");

    let contents = fs::read_to_string(file_path)
        .expect("не удалось прочитать файл");

    for line in contents.lines() {
        if line.contains(query) {
            println!("{line}");
        }
    }
}
```

Здесь всё знакомое: `collect()` собирает итератор в вектор (§17.1.4, §21.2.2), `fs::read_to_string` читает файл целиком в `String` (§18.2.2). Проверочный файл `text.txt` положим рядом с `Cargo.toml`:

```text
Курс начинался с языка C:
указатели, куча и ручная дисциплина.
Затем пришёл Rust — и владение
стало проверяться при компиляции.
Не доверяй — проверяй: rustc не спит.
```

Прогоны (аргументы после `--` уходят программе, а не cargo — §14.1.2):

```text
$ cargo run -- Rust text.txt
Ищем `Rust` в файле text.txt
Затем пришёл Rust — и владение

$ cargo run -- куча text.txt
Ищем `куча` в файле text.txt
указатели, куча и ручная дисциплина.
```

Работает. Теперь — честный взгляд на то, как эта версия умирает. Запуск без аргументов:

```text
$ cargo run
thread 'main' (22816) panicked at src\main.rs:7:22:
index out of bounds: the len is 1 but the index is 1
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace
```

Запуск с несуществующим файлом:

```text
$ cargo run -- Rust nofile.txt
Ищем `Rust` в файле nofile.txt
thread 'main' (30100) panicked at src\main.rs:13:10:
не удалось прочитать файл: Os { code: 2, kind: NotFound, message: "The system cannot find the file specified." }
```

Код возврата в обоих прогонах — 101 (замерено `echo $?`; ненулевой, как мы и хотели, но это фирменный код паники, а не наш осознанный выбор). Сравните с §5.1.2 тома I: там выход за границу массива компилировался молча и делал неизвестно что; здесь `args[1]` при недостатке аргументов — гарантированная паника с точной диагностикой. Уже неплохо. Но сообщение `index out of bounds` адресовано программисту, а не пользователю утилиты — пользователю нужно «вы забыли аргументы, вот как надо». Этим займётся следующий чекпоинт.

**Типичные ошибки чекпоинта.** Забыть `--` и скормить аргументы самому cargo (`cargo run Rust text.txt` — cargo решит, что `Rust` — его опция); индексировать `args` без мысли о том, что пользователь мог не передать ничего (наш черновик делает ровно это — намеренно и ненадолго).

### §25.1.2. Чекпоинт 2: рефакторинг — библиотека, Config и честные ошибки

У черновика три болезни: `main` делает всё сразу (разбор аргументов, чтение, поиск, печать); конфигурация размазана по переменным; ошибки — паники с внутренностями наружу. Лечение всех трёх — известная вам форма «библиотека + тонкий `main.rs`» из §20.1.1: рядом с `src/main.rs` появляется `src/lib.rs`, и пакет теперь содержит два crate — библиотеку с логикой и бинарник-обёртку. Напомню правило оттуда же: бинарник видит библиотеку **только** через публичный интерфейс, как посторонний код, — это и заставляет интерфейс быть осмысленным.

**Определение.** **`std::process::exit(code)`** — немедленно завершает процесс с заданным кодом возврата (R9, `std::process`). Это растовский родственник `return` из C-шного `main` (§4.1.1) и библиотечного `exit()` из `<stdlib.h>` (C5): деструкторы при этом **не** выполняются — поэтому вызывать его стоит только из `main`, на самом верху, когда всё уже свёрнуто (R9).

Проектные решения чекпоинта:

1. Конфигурация — это данные, значит, ей положена структура: `Config { query, file_path }` (§16.1.1).
2. Разбор аргументов может не удаться по вине пользователя — это **ожидаемая** ошибка, значит, `Result`, а не паника (§18.1.2): `Config::build(&args) -> Result<Config, &'static str>`. Почему `&'static str`? Наши сообщения об ошибках — строковые литералы, они живут в бинарнике вечно (§15.3.2, §19.3.2), и для утилиты этого хватает.
3. Вся работа после разбора — в `run(config) -> Result<(), Box<dyn Error>>`. Помните §18.1.3? Когда мы впервые встретили `?`, компилятор сам подсказывал этот тип для `main`, а мы отложили его до главы 22. Пора: `Box<dyn Error>` — «коробка с любой ошибкой» — трейт-объект (§22.1.3), в который `?` упакует ошибку любого типа, реализующего `Error`. Для утилиты, которой всё равно, *какая именно* ошибка ввода-вывода случилась — лишь бы донести её текст до пользователя, — это ровно то, что нужно.
4. Сообщения об ошибках — в **stderr** (`eprintln!`, §18.1.2), результаты — в stdout. Дисциплина двух потоков из §10.1.1 тома I работает и здесь: пользователь перенаправляет результаты в файл, а ошибки остаются на экране.

`src/lib.rs`:

```rust
use std::error::Error;
use std::fs;

pub struct Config {
    pub query: String,
    pub file_path: String,
}

impl Config {
    pub fn build(args: &[String]) -> Result<Config, &'static str> {
        if args.len() < 3 {
            return Err("нужно два аргумента: <запрос> <файл>");
        }

        let query = args[1].clone();
        let file_path = args[2].clone();

        Ok(Config { query, file_path })
    }
}

pub fn run(config: Config) -> Result<(), Box<dyn Error>> {
    let contents = fs::read_to_string(&config.file_path)?;

    for line in contents.lines() {
        if line.contains(&config.query) {
            println!("{line}");
        }
    }

    Ok(())
}
```

`src/main.rs` — тонкая обёртка, вся её работа — свести библиотеку с окружением:

```rust
use std::env;
use std::process;

use minigrep::Config;

fn main() {
    let args: Vec<String> = env::args().collect();

    let config = Config::build(&args).unwrap_or_else(|err| {
        eprintln!("Ошибка разбора аргументов: {err}");
        process::exit(1);
    });

    if let Err(e) = minigrep::run(config) {
        eprintln!("Ошибка выполнения: {e}");
        process::exit(1);
    }
}
```

`unwrap_or_else` с замыканием (§21.1.1) разворачивает `Ok` или выполняет ветку ошибки; `if let Err(e)` (§16.2.4) — для `run`, у которого в `Ok` ничего интересного. Прогоны:

```text
$ cargo run -- Rust text.txt
Затем пришёл Rust — и владение

$ cargo run
Ошибка разбора аргументов: нужно два аргумента: <запрос> <файл>
$ echo $?
1

$ cargo run -- Rust nofile.txt
Ошибка выполнения: The system cannot find the file specified. (os error 2)
$ echo $?
1
```

Ни одной паники: пользователь видит человеческое сообщение, скрипты видят код 1. И главная проверка — потоки действительно разделены:

```text
$ cargo run -- Rust nofile.txt > результат.txt
Ошибка выполнения: The system cannot find the file specified. (os error 2)
```

Сообщение осталось на экране (stderr), а `результат.txt` создан пустым (0 байт) — в него не утекло ни строчки мусора. Версия из чекпоинта 1 записала бы в файл «Ищем …» — и молча испортила бы вывод любому конвейеру.

**Честная оговорка** про `clone` в `build`: мы копируем две строки, потому что `args` у нас взаймы (`&[String]`), а `Config` хочет владеть (§15.2.1). Для двух аргументов при запуске программы цена неощутима, но осадок остался — чекпоинт 4 уберёт копии красивее, чем `clone`.

### §25.1.3. Чекпоинт 3: поиск под тестами

Логика поиска сейчас зашита в `run` — её нельзя ни протестировать, ни переиспользовать. Выделим функцию — и, раз уж мы в библиотеке с бесплатным тестовым каркасом (§20.2.1), напишем её в ритме TDD: сначала красный тест, потом код, который его зеленит.

**Определение.** **`str::to_lowercase()`** — возвращает `String` с текстом, приведённым к нижнему регистру по правилам Unicode (R9, `str`): работает и для `R`→`r`, и для `Д`→`д`. Именно `String`, а не `&str`: строчная версия — новые данные, их кому-то надо владеть (§17.1.3). Сравните с `tolower` из `<ctype.h>` C, обрабатывающим по одному `char` за вызов (C5), — про Unicode он не слышал.

Сигнатура поиска — самое интересное решение чекпоинта:

```rust
pub fn search<'a>(query: &str, contents: &'a str) -> Vec<&'a str> {
```

Функция возвращает ломтики **исходного текста** — не копии. Значит, компилятору нужно знать, что элементы результата живут не дольше `contents`, — и мы говорим ему это временем жизни `'a` (§19.3.1), привязанным именно к `contents`, а не к `query`: из запроса в результат не попадает ничего. Тест — до реализации:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn finds_one_line() {
        let query = "влад";
        let contents = "\
язык C:
быстро и близко к железу.
Rust: быстро и с владением.";

        assert_eq!(vec!["Rust: быстро и с владением."], search(query, contents));
    }
}
```

Реализация первым проходом — честный цикл, знакомый с §17.1.1:

```rust
pub fn search<'a>(query: &str, contents: &'a str) -> Vec<&'a str> {
    let mut results = Vec::new();

    for line in contents.lines() {
        if line.contains(query) {
            results.push(line);
        }
    }

    results
}
```

Тест зелёный. Дальше — регистронезависимый режим, тем же ритмом: тест, фиксирующий, что обычный поиск регистр **уважает**, тест на новый поиск, реализация:

```rust
    #[test]
    fn search_is_case_sensitive() {
        let query = "rust";
        let contents = "\
Rust не прощает,
но rustc объясняет.";

        assert_eq!(vec!["но rustc объясняет."], search(query, contents));
    }

    #[test]
    fn case_insensitive_finds_both() {
        let query = "RuSt";
        let contents = "\
Rust не прощает,
но rustc объясняет.";

        assert_eq!(
            vec!["Rust не прощает,", "но rustc объясняет."],
            search_case_insensitive(query, contents)
        );
    }
```

```rust
pub fn search_case_insensitive<'a>(query: &str, contents: &'a str) -> Vec<&'a str> {
    let query = query.to_lowercase();
    let mut results = Vec::new();

    for line in contents.lines() {
        if line.to_lowercase().contains(&query) {
            results.push(line);
        }
    }

    results
}
```

Заметьте: `query` внутри — уже `String` (владелец строчной копии), поэтому в `contains` она идёт по ссылке. Осталось дать пользователю рубильник. Договоримся: переменная окружения `IGNORE_CASE` (флаг командной строки — в практике).

**Определение.** **`std::env::var("ИМЯ")`** — читает переменную окружения, возвращая `Result<String, VarError>` (R9, `std::env`): `Ok(значение)`, если переменная установлена. Нам важен сам факт установки — метод `is_ok()` (R9) превращает `Result` в `bool`.

В `Config` добавляется поле `pub ignore_case: bool`, в конец `build` — строка `let ignore_case = env::var("IGNORE_CASE").is_ok();`, а `run` выбирает функцию по флагу:

```rust
    let results = if config.ignore_case {
        search_case_insensitive(&config.query, &contents)
    } else {
        search(&config.query, &contents)
    };

    for line in results {
        println!("{line}");
    }
```

Проверяем всё разом:

```text
$ cargo test
running 3 tests
test tests::finds_one_line ... ok
test tests::search_is_case_sensitive ... ok
test tests::case_insensitive_finds_both ... ok

test result: ok. 3 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s
```

(Далее cargo отчитывается ещё за пустые наборы тестов бинарника и документации — эти строки мы опускаем.) И живые прогоны — обратите внимание, как запрос `rust` в обычном режиме находит только строку с `rustc`:

```text
$ cargo run -- rust text.txt
Не доверяй — проверяй: rustc не спит.

$ IGNORE_CASE=1 cargo run -- rust text.txt
Затем пришёл Rust — и владение
Не доверяй — проверяй: rustc не спит.
```

(В PowerShell переменная ставится отдельной командой: `$env:IGNORE_CASE = "1"`, снимается `Remove-Item Env:IGNORE_CASE`.)

**Типичные ошибки чекпоинта.** Привязать `'a` к обоим аргументам «на всякий случай» — компилятор согласится, но сигнатура начнёт врать: она объявит, что результат может ссылаться и на `query`, и вызывающий код с короткоживущим запросом получит отказ на ровном месте. И обратная ловушка: попытаться вернуть `Vec<&str>` из функции, которая сама владеет текстом (скажем, сама читает файл), — E0515, возврат ссылки на умирающее значение (§15.2.3): ломтики живы, только пока жив владелец текста.

### §25.1.4. Чекпоинт 4: итераторы вместо clone и индексов

Обещанный второй виток рефакторинга (R1, гл. 13.3). Два места в коде просят улучшения, и оба лечатся главой 21.

**Первое** — `Config::build` берёт `&[String]` и клонирует. Но зачем нам заимствованный срез, если `env::args()` с самого начала **итератор владеющих `String`**? Передадим итератор целиком — и будем забирать значения себе, без копий:

```rust
impl Config {
    pub fn build(mut args: impl Iterator<Item = String>) -> Result<Config, &'static str> {
        args.next(); // нулевой аргумент — имя программы, пропускаем

        let query = match args.next() {
            Some(arg) => arg,
            None => return Err("нужно два аргумента: <запрос> <файл>"),
        };

        let file_path = match args.next() {
            Some(arg) => arg,
            None => return Err("нужно два аргумента: <запрос> <файл>"),
        };

        let ignore_case = env::var("IGNORE_CASE").is_ok();

        Ok(Config { query, file_path, ignore_case })
    }
}
```

`impl Iterator<Item = String>` в позиции параметра — знакомый по §19.2.2 «любой тип, умеющий это»: функции всё равно, вектор под ним или живой `env::args()`. Каждый `next()` **отдаёт владение** очередной `String` (§21.2.1) — строки переезжают из итератора прямо в `Config`, и `clone` исчезает вместе с индексами и проверкой длины: вместо «а хватает ли элементов?» — честный `match` на `Option`. В `main` меняется одна строка:

```rust
    let config = Config::build(env::args()).unwrap_or_else(|err| {
```

**Второе** — циклы в функциях поиска. Сравните сами:

```rust
pub fn search<'a>(query: &str, contents: &'a str) -> Vec<&'a str> {
    contents
        .lines()
        .filter(|line| line.contains(query))
        .collect()
}

pub fn search_case_insensitive<'a>(query: &str, contents: &'a str) -> Vec<&'a str> {
    let query = query.to_lowercase();
    contents
        .lines()
        .filter(|line| line.to_lowercase().contains(&query))
        .collect()
}
```

Конвейер `lines → filter → collect` (§21.2.2–21.2.3) читается как спецификация: «строки, прошедшие фильтр, собрать в вектор». Ни мутабельного накопителя, ни ручного `push` — и, как выяснила глава 21, без пенальти: абстракция нулевой стоимости. Финальная проверка — все тесты зелёные, поведение не изменилось:

### §25.1.5. Чекпоинт 5: эскиз на C — «Как это было в C»

Весь том II эта рубрика жила во врезках. Сегодня она — полноценный чекпоинт: напишем мини-grep на C, тем самым языком, которым мы бы решали задачу в конце тома I. Инструменты — главы 4 (функции), 5 (строки), 10 (файлы) — и два новых знакомства.

**Определения.**

- **`int main(int argc, char *argv[])`** — форма `main`, через которую C-программа получает аргументы (C5; C1): `argc` — их число, `argv` — массив указателей на C-строки, `argv[0]` — имя программы. Том I обходился `main(void)` (эта пара лишь мелькала в листинге §13.1.2) — знакомимся официально; `env::args()` из чекпоинта 1 — её растовский наследник.
- **`strstr(haystack, needle)`** из `<string.h>` — ищет первое вхождение подстроки, возвращает указатель на него или `NULL` (C5, `<string.h>`; C6). Тот же `contains`, только результат — указатель в чужую память со всеми вытекающими из §6.2.

`minigrep.c`:

```c
#include <stdio.h>
#include <string.h>

enum { MAX_LINE = 1024 };

int main(int argc, char *argv[])
{
    if (argc != 3) {
        fprintf(stderr, "Использование: %s <запрос> <файл>\n", argv[0]);
        return 1;
    }

    FILE *f = fopen(argv[2], "r");
    if (f == NULL) {
        perror(argv[2]);
        return 1;
    }

    char line[MAX_LINE];
    while (fgets(line, sizeof line, f) != NULL) {
        if (strstr(line, argv[1]) != NULL)
            fputs(line, stdout);
    }

    fclose(f);
    return 0;
}
```

Компиляция эталоном курса чистая, поведение на том же `text.txt` совпадает с Rust-версией байт в байт (вывод и коды возврата; путь в argv[0] сокращён — MinGW разворачивает его в полный):

```text

$ ./minigrep_c.exe Rust text.txt
Затем пришёл Rust — и владение
$ ./minigrep_c.exe rust text.txt
Не доверяй — проверяй: rustc не спит.
$ ./minigrep_c.exe
Использование: .../minigrep_c.exe <запрос> <файл>
$ echo $?

```

Прогон под AddressSanitizer в WSL (дисциплина §7.2) — без единого отчёта, коды возврата те же. И вот неудобный факт: **C-версия занимает 27 строк**. Rust-версия — 115 (16 в `main.rs` и 99 в `lib.rs`, из которых ~40 — тесты). C здесь честно короче, и стоит проговорить, за что уходит разница:

| | `minigrep.c` (27 строк) | `minigrep` на Rust (115 строк) |
|---|---|---|
| Ошибки | коды возврата + `perror`; забыть проверку компилятор позволит (§13.1.2) | `Result` прошивает все слои; забыть — ошибка компиляции |
| Тесты | каркаса нет; тестировать `main` целиком снаружи | `cargo test`, 3 юнит-теста внутри библиотеки |
| Длина строки | буфер 1024 байта, длиннее — режется | любая: строкой владеет `String` |
| Регистр | вручную, посимвольно (`tolower`) | Unicode из коробки (`to_lowercase`) |
| Переиспользование | логика впаяна в `main` | библиотека; второй клиент — глава 26 |

Строка «буфер 1024» — не теоретическая придирка. Эксперимент: файл, где первая строка — 1021 символ `a`, затем слово `Rust`, затем ещё 500 символов; вторая строка обычная, тоже со словом `Rust`. Rust-версия находит **обе** строки. C-версия — **одну** (вторую): `fgets` вернул первые 1023 байта — `...aaaRu`, — потом остаток — `st...`, и слово `Rust` рассечено границей буфера: `strstr` не видит его ни в одном куске. Программа не упала, санитайзер молчит (выхода за границы нет!), тесты бы прошли — она просто **тихо теряет результаты** на данных, которых автор не предвидел. Это точный повтор урока §13.1: главная цена C — не крах, а молчаливо неверный ответ; а урок §7.2 в новой редакции: чистый ASan-прогон — свидетельство, не доказательство.

Доводить ли C-версию до честного паритета (растущий буфер из §7.1.3, свой `str_to_lower`, модульная структура с Makefile из §11.2) — упражнение в практике; счёт строк при этом уверенно уходит за сотню, только уже без гарантий компилятора.

## Часть 25.2. Пул потоков: pargrep

### §25.2.1. Чекпоинт 1: зачем пул и какой у него интерфейс

Задача части: **`pargrep <потоки> <запрос> <файл>...`** — параллельный поиск по многим файлам. Библиотека поиска у нас уже есть (Ч1); не хватает механизма, который раздаст файлы потокам.

Наивный план «поток на файл» глава 23 позволяет написать в десять строк: `spawn` в цикле, `join` в цикле. Но у него не выставлен предел: тысяча файлов — тысяча потоков, у каждого — стек и место в расписании ОС (§23.1.1); при этом ядер у процессора — единицы. Хуже того: столько параллельности нам и не нужно — нужно, чтобы **К потоков пережёвывали N задач**. Эта фигура — **пул потоков** (thread pool): рабочие потоки создаются один раз, живут долго и по очереди берут задачи из общей очереди (R1, гл. 21.2). Знакомый мотив? Это динамический массив из §7.1.3 наоборот: там мы амортизировали выделения памяти, здесь — создание потоков.

Проектируем интерфейс, как нас учил §20.1: сначала — каким кодом мы **хотим** пользоваться, потом — как это реализовать. Хотим так:

```rust
use std::thread;
use std::time::{Duration, Instant};

use pargrep::ThreadPool;

fn main() {
    let start = Instant::now();
    let pool = ThreadPool::new(4);

    for id in 1..=8 {
        pool.execute(move || {
            println!("задача {id}: начало");
            thread::sleep(Duration::from_millis(300));
            println!("задача {id}: конец");
        });
    }

    // Пока пул не умеет ждать своих задач — подождём вслепую.
    thread::sleep(Duration::from_millis(1000));
    println!("главный поток выходит через {:?}", start.elapsed());
}
```

⛔ Этот `main` пока не компилируется — библиотеки `pargrep` не существует. Это не ошибка, а метод: компилятор поведёт нас от желаемого интерфейса к реализации, отказ за отказом (R1, гл. 21.2, — там это называют compiler-driven development; мы играли в это в §19.2).

**Определение.** **`std::time::Instant`** — точка на монотонных часах; `Instant::now()` берёт отметку, `elapsed()` возвращает `Duration` (§23.1.1), прошедший с неё (R9, `std::time`). Монотонных — значит, не прыгающих при переводе системного времени: для замеров годятся только такие.

Читаем из желаемого `main` требования к типу:

- `ThreadPool::new(size)` — создать пул с `size` воркерами. Ноль воркеров — бессмыслица; договоримся, что это ошибка программиста, и `assert!` в дебаг- и релиз-сборке остановит её паникой (§20.2.1). По конвенции документации из §20.2.3 такое обещание пишется в док-комментарий под заголовком `# Panics`.
- `pool.execute(замыкание)` — отдать задачу. Какие обязательства несёт замыкание? Смотрим на `thread::spawn` (§23.1.2, §23.2.3) — у него и подсмотрим: **`FnOnce()`** — задача выполняется один раз (и вправе потреблять захваченное — §21.1.3); **`Send`** — задача переедет в чужой поток; **`'static`** — задача обязана не держать ссылок на стек вызывающего: `execute` вернётся сразу, а работа начнётся позже, когда этого стека, может, уже нет.

Сигнатура собирается сама:

```rust
    pub fn execute<F>(&self, f: F)
    where
        F: FnOnce() + Send + 'static,
```

Обратите внимание на `&self`: очередь задач будет общей, а пополнять её можно по обычной ссылке — всю внутреннюю изменяемость возьмут на себя канал и замок (§22.2.1, §23.2.1). Реализацию этого интерфейса построит следующий чекпоинт.

### §25.2.2. Чекпоинт 2: воркеры и канал задач

Внутреннее устройство пула — три решения (R1, гл. 21.2).

**Решение 1: задача — это `Box<dyn FnOnce() + Send + 'static>`.** У каждого замыкания свой анонимный тип (§21.1.1), а очередь должна хранить их вперемешку — значит, трейт-объект в коробке (§22.1.3: размер замыкания компилятору неизвестен — в кучу его). Псевдоним типа (§16.1.2) избавит от скороговорки:

```rust
type Job = Box<dyn FnOnce() + Send + 'static>;
```

**Решение 2: очередь — это канал.** Очередь задач с одним входом и общим выходом мы строили в §23.1.3 — это ровно `mpsc::channel`. `ThreadPool` держит передатчик; `execute` упаковывает замыкание в `Job` и шлёт.

**Решение 3: приёмник делят воркеры через `Arc<Mutex<...>>`.** Вот здесь канал сопротивляется: mpsc — multi-producer, **single-consumer**, приёмник один и не клонируется. А воркеров много, и каждый хочет `recv`. Знакомая с §23.2.2 формула совместного владения с очередью на доступ: `Arc<Mutex<mpsc::Receiver<Job>>>` — все воркеры владеют приёмником, но взять из него задачу можно только под замком.

**Воркер** (worker) — пара «номер + ручка потока»; его поток крутится в вечном цикле «взять задачу — выполнить». Собираем первую версию библиотеки:

```rust
use std::sync::{Arc, Mutex, mpsc};
use std::thread;

type Job = Box<dyn FnOnce() + Send + 'static>;

pub struct ThreadPool {
    workers: Vec<Worker>,
    sender: mpsc::Sender<Job>,
}

impl ThreadPool {
    /// Создаёт пул с `size` рабочими потоками.
    ///
    /// # Panics
    ///
    /// Паникует, если `size` равен нулю.
    pub fn new(size: usize) -> ThreadPool {
        assert!(size > 0);

        let (sender, receiver) = mpsc::channel();
        let receiver = Arc::new(Mutex::new(receiver));

        let mut workers = Vec::with_capacity(size);
        for id in 0..size {
            workers.push(Worker::new(id, Arc::clone(&receiver)));
        }

        ThreadPool { workers, sender }
    }

    pub fn execute<F>(&self, f: F)
    where
        F: FnOnce() + Send + 'static,
    {
        let job = Box::new(f);
        self.sender.send(job).unwrap();
    }
}

struct Worker {
    id: usize,
    thread: thread::JoinHandle<()>,
}

impl Worker {
    fn new(id: usize, receiver: Arc<Mutex<mpsc::Receiver<Job>>>) -> Worker {
        let thread = thread::spawn(move || {
            loop {
                let message = receiver.lock().unwrap().recv();

                match message {
                    Ok(job) => {
                        job();
                    }
                    Err(_) => {
                        println!("воркер {id}: канал закрыт, завершаюсь");
                        break;
                    }
                }
            }
        });

        Worker { id, thread }
    }
}
```

Ключевая строка — `let message = receiver.lock().unwrap().recv();`. Замок берётся, `recv` блокируется до задачи (§23.1.3), задача достаётся — и **замок отпускается в конце этой строки**: временные значения (в том числе `MutexGuard`, отпирающий замок в своём `Drop`, — §23.2.1) умирают в конце содержащего их оператора `let` (R7, Destructors). Пока один воркер выполняет `job()`, остальные свободно берут следующие задачи. Компилятор при сборке ворчит — `field 'workers' is never read`, `fields 'id' and 'thread' are never read`: поля-то мы сложили, а читать их будет только чекпоинт 3. Предупреждения — честный список недостроенного.

Запускаем `main` из чекпоинта 1 (теперь он компилируется):

```text

задача 1: начало
задача 2: начало
задача 4: начало
задача 3: конец
задача 4: конец
задача 6: начало
задача 2: конец
задача 7: начало
задача 1: конец
задача 5: начало
задача 8: начало
задача 6: конец
задача 8: конец
задача 5: конец
задача 7: конец
главный поток выходит через 1.0003711s

```

Читаем внимательно — тут три урока в одном прогоне. Первый: **четыре задачи стартовали сразу** (в перемешанном порядке — расписание у ОС, §23.1.1), остальные ждали освободившихся воркеров: две волны по ~300 мс, все восемь задач уложились в секунду слепого ожидания. Второй: порядок строк в каждом прогоне свой. Третий — смотрите на обрыв в последней строке: после `main` пул уничтожился, передатчик умер, воркеры начали прощаться (`Err` от `recv` — все отправители мертвы, §23.1.3), но процесс завершился **раньше**, чем они дописали строку (§23.1.1: смерть `main` — смерть всех потоков). Программа работает — и заканчивается неопрятно: задачи доделались случайно (нам повезло со слепой секундой), потоки обрублены на полуслове.

Прежде чем чинить, — ловушка, в которую попадает почти каждый, кто пишет этот цикл. Казалось бы, цикл воркера красивее записать через `while let`:

```rust
// ⛔ компилируется — и молча убивает всю параллельность
let thread = thread::spawn(move || {
    while let Ok(job) = receiver.lock().unwrap().recv() {
        job();
    }
    println!("воркер {id}: канал закрыт, завершаюсь");
});
```

Тот же `main`, реальный прогон:

```text
задача 1: начало
задача 1: конец
задача 2: начало
задача 2: конец
задача 3: начало
задача 3: конец
задача 4: начало
главный поток выходит через 1.0004019s
```

Задачи идут **строго по одной**, задачи 5–8 не начались вовсе — потеряны молча. Причина — в правиле времён жизни временных значений: в `while let` (как и в `if let`) временные из условия живут **до конца тела итерации** (R7, Destructors; R1, гл. 21.2). `MutexGuard` не умирает после `recv()` — он держит замок всё время, пока крутится `job()`. Один воркер работает — трое стоят у запертой двери. Форма с `let message = ...` — не вкусовщина, а граница жизни замка. Это тот же класс ловушек, что «замок уехал в первую итерацию» из §23.2.2: `Mutex` защищает данные безупречно, но **когда** его отпускают — по-прежнему ваше проектное решение.

### §25.2.3. Чекпоинт 3: аккуратное завершение — Drop

Итог предыдущего прогона: задачи выполняются, но завершение — свалка. Протокол аккуратного завершения напрашивается из самой конструкции канала (§23.1.3):

1. закрыть вход: уничтожить передатчик — единственный, он в `ThreadPool`;
2. воркеры дорабатывают очередь: `recv` отдаёт накопленные задачи, пока канал не опустеет, и лишь потом возвращает `Err`;
3. воркер на `Err` выходит из цикла (эта ветка у нас уже написана);
4. дождаться каждого: `join` (§23.1.1).

Кто исполняет протокол? Тот же механизм, что закрывал файлы и освобождал память весь том II, — **`Drop`** (§22.1.2): пул, выходящий из области видимости, обязан проводить своих воркеров сам. Инструментов не хватает двух.

**Определения.**

- **`Option::take()`** — забирает значение из `Option`, оставляя на его месте `None` (R9). Это выход из ловушки E0507 «нельзя вынести владение из-под заимствования» (§17.1.2): в `drop` у нас лишь `&mut self`, забрать `sender` через ссылку не выйдет — но можно **подменить** его на `None`: заимствование не нарушено, а владение отправителем у нас в руках. Плата — поле становится `Option<mpsc::Sender<Job>>`, и `execute` обращается к нему через `as_ref().unwrap()`: «в работающем пуле отправитель есть всегда» — инвариант, который держит наш модуль (§24.1.3 передаёт привет).
- **`Vec::drain(..)`** — итератор, **вычерпывающий** вектор: отдаёт элементы с владением, оставляя вектор пустым (R9). Тот же трюк для воркеров: `join` потребляет `JoinHandle` (§23.1.1), по ссылке его не вызвать.

Изменения в библиотеке:

```rust
pub struct ThreadPool {
    workers: Vec<Worker>,
    sender: Option<mpsc::Sender<Job>>,
}
```

```rust
        ThreadPool { workers, sender: Some(sender) }
```

```rust
        let job = Box::new(f);
        self.sender.as_ref().unwrap().send(job).unwrap();
```

И сам протокол — дословный перевод четырёх пунктов на Rust:

```rust
impl Drop for ThreadPool {
    fn drop(&mut self) {
        // Хороним отправителя: recv у воркеров вернёт Err,
        // как только очередь задач опустеет.
        drop(self.sender.take());

        for worker in self.workers.drain(..) {
            println!("ждём воркера {}", worker.id);
            worker.thread.join().unwrap();
        }
    }
}
```

В демонстрационном `main` слепой `sleep` заменяем на явные проводы:

```rust
    drop(pool); // провожаем пул: Drop дождётся всех задач
    println!("все задачи завершены за {:?}", start.elapsed());
}
```

Разбор стоит внимания. «ждём воркера 0» — **первая** строка вывода: `drop(pool)` начался немедленно, задачи ещё шли — но `join` терпеливо стоит. Все восемь задач дошли до «конец», все четыре воркера попрощались полными строками, итог — 613 мс: две волны по 300 плюс накладные. Сравните со слепой секундой и обрывом на полуслове из чекпоинта 2: теперь завершение — не гонка со смертью процесса, а протокол.

Раз пул стал надёжным, его можно тестировать (§20.2.1). Заметьте, как `drop(pool)` работает барьером синхронизации — после него все задачи гарантированно выполнены:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn pool_runs_all_tasks() {
        let counter = Arc::new(Mutex::new(0));
        let pool = ThreadPool::new(4);

        for _ in 0..100 {
            let counter = Arc::clone(&counter);
            pool.execute(move || {
                *counter.lock().unwrap() += 1;
            });
        }
        drop(pool); // Drop дожидается всех задач — счётчик финален

        assert_eq!(*counter.lock().unwrap(), 100);
    }
}
```

Счётчик под `Arc<Mutex<...>>` — дословно §23.2.2, и тест зелёный с первого раза именно потому, что все гонки этот тип исключил ещё на компиляции:

```text
$ cargo test

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s
```

**Типичные ошибки чекпоинта.** Забыть `drop(self.sender.take())` и сразу пойти в `join` — воркеры вечно ждут задач в `recv`, `join` вечно ждёт воркеров: взаимная блокировка, знакомая по §23.2.4, только теперь мы построили её сами. Вызвать `join` **под** замком или внутри цикла с живым отправителем — тот же исход. Правило: сначала закрыть вход, потом ждать выхода.

### §25.2.4. Чекпоинт 4: pargrep — параллельный поиск и замеры

Собираем всё. `pargrep <потоки> <запрос> <файл>...`: каждый файл — задача пулу; задача читает файл, ищет запрос (регистронезависимо — пусть работает самая дорогая функция библиотеки) и отчитывается числом совпадений. Библиотеку поиска не пишем — берём готовую из Ч1: в `Cargo.toml` пакета `pargrep` появляется **path-зависимость** (§20.1.4; R10) — пакеты соседствуют в одной папке:

```toml
[dependencies]
minigrep = { path = "../minigrep" }
```

Осталась одна проектная развилка: как задачи возвращают результаты? Внутрь пула ничего встраивать не нужно — у нас есть **второй канал**, теперь в обратную сторону: задачи шлют, `main` принимает. Каждая задача получает клон передатчика (§23.1.3), а оригинал `main` обязан похоронить сам — иначе цикл приёма не завершится никогда (тот самый забытый отправитель из §23.1.3). Когда умирает последний клон — то есть завершается последняя задача, — цикл `for ... in rx` заканчивается сам: конец приёма и есть факт «все задачи готовы».

Один соблазн отбрасываем сразу: отправить в канал `Vec<&str>` из `search_case_insensitive` не выйдет — ломтики указывают в `contents`, локальную переменную задачи, а замыкание обязано быть `'static` (§25.2.1): ссылки на умирающий стек через границу потока не пройдут (§23.1.2 — та же причина, по которой `spawn` требует `move`). Отправляем владеющее значение — счётчик; версия, пересылающая сами строки (`String`), — в практике.

`src/main.rs` пакета `pargrep`:

```rust
use std::env;
use std::fs;
use std::process;
use std::sync::mpsc;
use std::time::Instant;

use pargrep::ThreadPool;

fn main() {
    let args: Vec<String> = env::args().collect();
    if args.len() < 4 {
        eprintln!("Использование: pargrep <потоки> <запрос> <файл>...");
        process::exit(1);
    }

    let threads: usize = match args[1].parse() {
        Ok(n) => n,
        Err(_) => {
            eprintln!("<потоки> должно быть целым числом");
            process::exit(1);
        }
    };
    if threads == 0 {
        eprintln!("<потоки> должно быть больше нуля");
        process::exit(1);
    }
    let query = &args[2];
    let files = &args[3..];

    let start = Instant::now();
    let pool = ThreadPool::new(threads);
    let (tx, rx) = mpsc::channel();

    for file in files {
        let tx = tx.clone();
        let query = query.clone();
        let file = file.clone();

        pool.execute(move || {
            let result = match fs::read_to_string(&file) {
                Ok(contents) => {
                    Ok(minigrep::search_case_insensitive(&query, &contents).len())
                }
                Err(e) => Err(e),
            };
            tx.send((file, result)).unwrap();
        });
    }
    drop(tx); // оригинал больше не нужен: живы только клоны внутри задач

    let mut total = 0;
    for (file, result) in rx {
        match result {
            Ok(count) => {
                println!("{file}: {count} совпадений");
                total += count;
            }
            Err(e) => eprintln!("{file}: не удалось прочитать: {e}"),
        }
    }

    println!("всего: {total} совпадений за {:?}", start.elapsed());
}
```

Обратите внимание: `parse` (§19.2.3) — и ошибки разбора обслужены до старта; клоны `query` и `file` перед `execute` — задачи владеют своими данными (Copy-ловушки §23.1.2 здесь нет: `String` не копируется молча); ошибка чтения — **значение** в кортеже результата, а не паника задачи. Функциональный прогон — два маленьких файла и один несуществующий:

```text
$ cargo run -- 2 rust ../small1.txt ../small2.txt ../missing.txt
../small2.txt: 3 совпадений
../missing.txt: не удалось прочитать: The system cannot find the file specified. (os error 2)
../small1.txt: 5 совпадений
всего: 8 совпадений за 6.0558ms
ждём воркера 0
воркер 1: канал закрыт, завершаюсь
воркер 0: канал закрыт, завершаюсь
ждём воркера 1
```

Всё как проектировали: результаты приходят в порядке завершения, а не подачи (недетерминизм §23.1.1 — здесь он норма, счёт от него не зависит); ошибка одного файла не сорвала остальные и ушла в stderr; после итогов пул попрощался по протоколу чекпоинта 3. Код возврата — 0 даже при ошибке чтения: осознанное упрощение, grep-схема кодов — в практике.

**Нагрузочная проверка.** Восемь файлов по 400 000 строк (~33 МБ каждый) изготовит генератор — одиночный файл `gen.rs`, компилируемый напрямую (`rustc --edition 2024 gen.rs`, §14.1.1), — в каждой тысячной строке спрятано слово «Rust»:

```rust

Замеры — **только релизная сборка** (`--release`, §14.1.2: оптимизатор включён; наш дев-прогон на тех же данных вдвое медленнее — и то лишь потому, что тяжёлая работа происходит внутри заранее оптимизированной std). Урок замеров: ускорение от потоков — не `size`, а «сколько параллельной работы вы им дали»; мерить надо всегда, угадывать — никогда.

### §25.2.5. Ретроспектива: что дал C и что дал Rust

Проекты закончены — время выполнить последнее обещание введения. Два тома назад вы писали `printf("Hello\n")`; сегодня — многопоточный поиск с пулом, каналами и нулём небезопасного кода. Разложим маршрут по полкам:

| Тема | Том I (C) | Том II (Rust) |
|---|---|---|
| Память | `malloc`/`free` руками, дисциплина владения — на совести (§7.1) | владение в типах, `Drop` вместо `free` (гл. 15, 22) |
| Ошибки | коды возврата, `errno`, проверку можно забыть (§10.1) | `Result` + `?`, забыть — не компилируется (гл. 18) |
| Строки | `char*`, терминатор, буферы фиксированной длины (§5.2) | `String`/`&str`, UTF-8, длина — забота типа (гл. 17) |
| Полиморфизм | `void*` и указатели на функции, типы — на честном слове (§6.3) | дженерики и трейт-объекты под надзором (гл. 19, 22) |
| Потоки | гонка данных = UB, ловится TSan-ом по факту (§13.1.4, §23.2.4) | `Send`/`Sync` исключают гонку на компиляции (гл. 23) |
| Сборка и тесты | Makefile и самодельные проверки (§11.2) | cargo: сборка, зависимости, `cargo test` (гл. 20) |

**Что дал C.** Модель машины. Стек и куча, указатель как адрес, цена копирования, устройство строки, паддинг структуры — всё это вы щупали руками в томе I, и именно поэтому том II не был магией: «владение» — это дисциплина `malloc`/`free`, вшитая в типы; `Box` — это `malloc`+`free`, которые нельзя забыть; сырой указатель главы 24 был встречей со старым знакомым, а не с чудовищем. Кто учил Rust без C, верит компилятору на слово; вы — понимаете, от чего именно он вас защищает и почём эта защита.

**Что дал Rust.** Те же силы — с гарантиями. Наш `minigrep` дважды пережил капитальный рефакторинг под защитой тестов и системы типов; наш `ThreadPool` раздаёт замыкания по потокам, и единственную настоящую ловушку (замок в `while let`) мы поймали **замером**, а не отладкой повреждённой памяти в три ночи. Инварианты, которые в C жили в комментариях («буфер освобождает вызывающий», §12.2.4), здесь живут в сигнатурах — и проверяются при каждой сборке.

**Честный итог.** На 27 строках C выигрывает: меньше церемоний, мгновенный результат. Проигрывать он начинает на росте: каждая строка таблицы выше — это класс ошибок, который в C-проекте прирастает с кодовой базой, а в Rust-проекте — нет. Лучшее доказательство — следующая глава: библиотека `minigrep` получит графический интерфейс, **не изменившись ни на строчку**, — попробуйте проделать это с логикой, впаянной в `main` эскиза на C.

## Резюме главы

- **Мини-grep — эталонная эволюция маленькой программы**: черновик с панками → библиотека + тонкий `main` → `Config::build` с `Result` и `Box<dyn Error>` с `?` (обещание §18.1.3 выполнено) → поиск под TDD с временем жизни `'a`, привязанным к тексту → итераторы вместо `clone` и индексов. Ошибки — в stderr, результаты — в stdout, коды возврата — осмысленные; каждый шаг рефакторинга — под защитой зелёных тестов.
- **Эскиз на C короче в четыре раза — и это не победа**: 27 строк против 115 покупают отсутствие тестов, буфер с границей (наш эксперимент: слово, рассечённое границей 1024 байта, «тихо» теряется — ASan молчит, программа врёт), ASCII-регистр и логику, впаянную в `main`. Цена C — не крах, а молчаливо неверный ответ (§13.1 в миниатюре).
- **Пул потоков — вся глава 23 в одном типе**: канал как очередь задач (`Job = Box<dyn FnOnce() + Send + 'static>`), `Arc<Mutex<Receiver>>` как право брать по очереди, `Drop` как протокол завершения (похоронить отправителя → воркеры дорабатывают очередь → `join`). Ловушка недели: `while let` держит `MutexGuard` всю итерацию — параллельность умирает молча, ловится замером.
- **pargrep — сборка всего курса**: path-зависимость на библиотеку Ч1, обратный канал результатов с `drop(tx)`, ошибки как значения в кортежах. Замер: 2.04 с → 368 мс на 8 потоках (5.5×); почему не 8× — гранулярность задач, диск, последовательный запуск. Мерить, не угадывать.
- **Ретроспектива**: C дал модель машины — том II не был магией; Rust дал те же силы с инвариантами в типах — рефакторинг и потоки без страха. C выигрывает на 27 строках, Rust — на каждой следующей тысяче.
