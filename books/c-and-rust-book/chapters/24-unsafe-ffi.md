# Глава 24. unsafe Rust и FFI: соединяем C и Rust

## Введение

Весь том II компилятор стоял между нами и памятью: владение (глава 15) запрещало висячие ссылки, XOR-правило — гонки записи, `Send`/`Sync` (глава 23) — гонки данных между потоками. Но мы принимали на веру целый пласт утверждений: «`Rc` посчитает ссылки», «`Mutex` не пустит двоих», «`RefCell` проверит в рантайме». Кто-то ведь *написал* эти типы — и написал их не в безопасном Rust: безопасный Rust не умеет ни второй изменяющий доступ через `&T`, ни ручное освобождение памяти. Пришло время снять капот.

Эта глава — про две вещи, которые в языке живут рядом и по сути об одном:

- **unsafe Rust** — узаконенное подмножество языка, где программист берёт часть обязанностей компилятора на себя: сырые указатели (те самые, из главы 6 — круг замкнётся), вызовы функций с ручным контрактом, изменяемые глобали, `union`, ручные реализации `Send`. Обещание §13.2.3 — «весь риск сосредоточен в помеченных блоках» — здесь мы увидим в работе.
- **FFI** (foreign function interface — интерфейс внешних функций) — мост, по которому Rust вызывает функции C и наоборот. Два языка курса впервые встретятся в одной программе: C-библиотека, слинкованная в Rust-бинарник, и Rust-библиотека, слинкованная в C-программу, — обе соберутся и отработают у нас на глазах.

Что вы будете уметь к концу главы: читать и писать маленькие `unsafe`-блоки с документированным контрактом; объяснять, что именно разрешает `unsafe` (и что — нет); проверять unsafe-код инструментом Miri (наследник ASan/Valgrind из §7.2 — на этот раз для Rust); вызывать C из Rust и Rust из C, передавая через границу числа, структуры, строки, массивы и владение; собирать смешанный проект руками и через cargo.

**Пометка о жанре.** По программе курса это обзорно-продвинутая глава: полный учебник unsafe — это отдельная книга (Rustonomicon, R8), а мы строим честную карту территории с рабочими примерами. Всё, что показано, — скомпилировано и запущено на эталоне курса (rustc/cargo 1.97.0, `rustc --edition 2024`; C-сторона — GCC 15.2.0 MinGW-w64, `-std=c17 -Wall -Wextra`).

## Часть 24.1. unsafe Rust

### §24.1.1. Контракт `unsafe`: пять сверхспособностей

**Определения.** **unsafe Rust** — это не другой язык, а пять дополнительных операций, доступных внутри блока `unsafe { ... }` или функции, помеченной `unsafe fn`. Список закрытый (R1, гл. 20.1; R7, «Unsafety» — арбитр):

1. **разыменовать сырой указатель** (`*const T` / `*mut T` — §24.1.2);
2. **вызвать unsafe-функцию или метод** (в том числе функцию C — часть 24.2);
3. **прочитать или изменить изменяемую статическую переменную** (`static mut`);
4. **реализовать unsafe-трейт** (`unsafe impl` — например, `Send` из §23.2.3);
5. **обратиться к полю `union`** (родственник union из §8.2.2).

Всё остальное — *ровно тот же* Rust: borrow checker не выключается, времена жизни проверяются, границы срезов контролируются, типы сходиться обязаны. `unsafe` — не рубильник «отключить проверки», а пропуск в пять конкретных дверей, за которыми компилятор физически не может проверить корректность — и потому проверяете вы.

Второе определение — самое важное в главе. **Undefined behavior в Rust запрещено всегда — в том числе в `unsafe`-коде** (R7, «Behavior considered undefined»: список включает гонки данных, доступ по висячему или невыровненному указателю, нарушение правил алиасинга, невалидные значения — и прямо оговаривает, что действует и внутри `unsafe`-блоков). Слово `unsafe` не *разрешает* UB — оно переносит обязанность его не допустить с компилятора на программиста. Это дословно тот вексель, что мы выписали в §13.2.3: требование «программа не должна вызывать UB» остаётся абсолютным, меняется только тот, кто за это отвечает.

**Интуиция.** Представьте здание, где всюду работает автоматика пожаротушения (безопасный Rust), и одну комнату с табличкой «внутри автоматики нет, огнетушитель — на стене, ответственный — вы» (`unsafe`-блок):

```
     безопасный Rust                        unsafe
 (корректность проверяет компилятор)   (корректность доказываете вы)
┌──────────────────────────────────┐   ┌─────────────────────────────┐
│ ссылки, владение, срезы,         │   │ *p, вызовы C, static mut,   │
│ Vec, String, Rc, Mutex, каналы   │──▶│ union, unsafe impl          │
│ … 99% кода программы …           │   │ // SAFETY: почему это ок    │
└──────────────────────────────────┘   └─────────────────────────────┘
```

Вся ценность конструкции — в размере комнаты. В C «комнатой без автоматики» было всё здание (том I, особенно §13.1); в Rust — несколько помеченных строк, которые можно найти поиском и проверить руками. Довод «при аудите вместо "весь код подозреваемый" получаем "грепни `unsafe`"» из §13.2.3 работает только до тех пор, пока unsafe-блоки маленькие и редкие — это не стилистическая рекомендация, а основа модели безопасности (R1: «Keep unsafe blocks small»; R6: «unsafe-код должен быть маленьким, изолированным и тщательно задокументированным»).

**Разобранный пример 1 — дверь заперта (⛔).** Попробуем вызвать unsafe-функцию как обычную:

```rust
// unsafe_fn_err.rs
unsafe fn dangerous() {}

fn main() {
    dangerous(); // ⛔
}
```

```
error[E0133]: call to unsafe function `dangerous` is unsafe and requires unsafe block
 --> unsafe_fn_err.rs:4:5
  |
4 |     dangerous();
  |     ^^^^^^^^^^^ call to unsafe function
  |
  = note: consult the function's documentation for information on how to avoid undefined behavior
```

E0133 — код всех пяти сверхспособностей: «операция unsafe вне unsafe-контекста». Обратите внимание на note: компилятор отправляет читать *документацию функции* — потому что у каждой unsafe-функции есть контракт, который машина проверить не может.

**Разобранный пример 2 — контракт written down.** Правильная форма: `unsafe fn` объявляет контракт в документации, вызывающий подтверждает его комментарием `SAFETY`:

```rust
// unsafe_fn.rs
/// SAFETY: вызывающий обязан гарантировать, что p указывает
/// на живое, правильно выровненное значение i32.
unsafe fn read_i32(p: *const i32) -> i32 {
    unsafe { *p }
}

fn main() {
    let x = 42;
    // SAFETY: p взят из живой переменной x прямо здесь.
    let v = unsafe { read_i32(&x as *const i32) };
    println!("v = {v}");
}
```

```
v = 42
```

Комментарии `SAFETY:` — конвенция экосистемы (в таком виде пишет примеры и R1): у **объявления** unsafe-функции — что обязан обеспечить вызывающий; у **каждого** `unsafe`-блока — почему обязательства выполнены. Компилятор их не читает; читает следующий человек, который будет искать ошибку.

**Разобранный пример 3 — нюанс издания 2024.** Заметили `unsafe { *p }` *внутри* unsafe-функции? До издания 2024 тело `unsafe fn` само считалось одним большим unsafe-блоком. Теперь — нет: если убрать внутренний блок, программа соберётся, но с предупреждением:

```
warning[E0133]: dereference of raw pointer is unsafe and requires unsafe block
 --> unsafe_op_warn.rs:2:5
  |
2 |     *p
  |     ^^ dereference of raw pointer
  |
note: an unsafe function restricts its caller, but its body is safe by default
  = note: `#[warn(unsafe_op_in_unsafe_fn)]` (part of `#[warn(rust_2024_compatibility)]`) on by default
```

Формулировка note стоит того, чтобы её перечитать: «unsafe-функция ограничивает *вызывающего*, но её тело по умолчанию safe». Пометка `unsafe fn` — это про контракт снаружи; опасные операции внутри всё равно помечаются явно, каждая со своим обоснованием. Мы в курсе следуем этому правилу.

**Связь с практикой.** Стандартная библиотека — большой работодатель `unsafe`: `Vec` растит буфер сырой памятью, `Mutex` зовёт примитивы ОС, `RefCell` подкручивает счётчики через разделяемую ссылку (§24.1.3 покажет фундамент). Ваш код получает *безопасный интерфейс* над проверенным unsafe-ядром — и в этом вся стратегия языка: не «никакого unsafe», а «unsafe пишут немногие, проверяют тщательно, остальные пользуются безопасной обёрткой».

**Типичные ошибки и подводные камни.**

- *«Раз unsafe — значит, можно всё».* Нет: в unsafe-блоке по-прежнему работают владение, заимствование, проверки типов. Добавляется ровно пять операций.
- *«Unsafe — значит код неправильный».* Тоже нет (R6 формулирует прямо: unsafe не означает некорректность — означает, что корректность доказывает человек). Неправильный — это код с UB, а он запрещён по обе стороны забора.
- *Блок размером с функцию.* Чем шире `unsafe { ... }`, тем труднее сказать, какая именно операция потребовала пропуск. Правило: блок обнимает минимум строк.
- *Безымянный контракт.* `unsafe` без `SAFETY`-комментария — это перенос ответственности «на кого-то потом». Ревьюеру нечего проверять.

**Источники.** R1 (гл. 20.1 — список сверхспособностей, «unsafe не выключает borrow checker», культура маленьких блоков); R7 (Unsafety — арбитр по списку операций; Behavior considered undefined — каталог UB и его действие внутри unsafe); R8 (What Unsafe Rust Can Do — граница «только эти операции», список «что Rust считает safe»: deadlock, утечки, переполнение — не UB); R6 (unsafe-rust — пять операций, гигиена).

### §24.1.2. Сырые указатели: круг замкнулся

**Определения.** **Сырой указатель** (raw pointer) — адрес без гарантий: `*const T` (читать) и `*mut T` (читать и писать). Это ближайший родственник `const T *` и `T *` из C — настолько близкий, что в части 24.2 они будут соединяться напрямую. Сравним с ссылками из §15.2:

| Свойство | `&T` / `&mut T` (ссылка) | `*const T` / `*mut T` (сырой) | C-указатель (гл. 6) |
|---|---|---|---|
| Может быть null | никогда | да | да (`NULL`) |
| Может висеть | никогда (лайфтаймы) | да | да |
| Алиасинг двух «писателей» | запрещён (E0499) | разрешён | разрешён |
| Арифметика | нет | `add`/`offset` | `p + 1` |
| Время жизни | проверяется | не отслеживается | не отслеживается |
| Разыменование | безопасно | только в `unsafe` | всегда (и всегда на вашей совести) |

Ключевая асимметрия: **создавать и передавать сырые указатели безопасно — небезопасно только разыменовывать** (R1, гл. 20.1). Указатель-число никому не вредит; вред начинается при доступе к памяти.

Способы получить сырой указатель:

- приведение ссылки: `&x as *const i32`, `&mut x as *mut i32` (знакомый по гл. 14 оператор `as`);
- **операторы сырого заимствования** `&raw const x` / `&raw mut x` — прямая форма «взять адрес места» (R7, типы указателей);
- `ptr::null()` / `ptr::null_mut()` из `std::ptr`;
- у срезов и векторов — `as_ptr()` / `as_mut_ptr()` (уже встречались в §23.1.2 у `split_at`).

**Интуиция.** Ссылка — это «право доступа с гарантиями» (так мы её определили в §15.2.1 против C-указателя). Сырой указатель — отказ от всех гарантий в обмен на все свободы: перед вами снова честный адрес из главы 6, просто теперь он живёт в резервации. Rust не запрещает C-указатели — он запрещает их *непомеченное* использование.

**Разобранный пример 1 — создать можно, читать — по пропуску.**

```rust
// raw_basics.rs
fn main() {
    let mut num = 5;

    let r1 = &num as *const i32;
    let r2 = &mut num as *mut i32;

    println!("r1 = {r1:?}");
    println!("r2 = {r2:?}");

    unsafe {
        println!("*r1 = {}", *r1);
        *r2 = 10;
        println!("*r2 = {}", *r2);
    }
}
```

```
r1 = 0x529a5ffc14
r2 = 0x529a5ffc14
*r1 = 5
*r2 = 10
```

(Адреса — представительный прогон, на вашей машине будут другими: то же соглашение, что для `%p` в §6.1.1.) Оба указателя смотрят на один адрес — и «читатель» `r1` спокойно сосуществует с «писателем» `r2`. Для ссылок это была бы E0502; для сырых указателей — норма, потому что за отсутствие конфликтов теперь отвечаем мы. А вот `println!("{}", *r1)` *вне* блока — уже ⛔:

```
error[E0133]: dereference of raw pointer is unsafe and requires unsafe block
  |
4 |     println!("{}", *r);
  |                    ^^ dereference of raw pointer
  |
  = note: raw pointers may be null, dangling or unaligned; they can violate aliasing
    rules and cause data races: all of these are undefined behavior
```

Note компилятора — готовый конспект параграфа: null, висячесть, выравнивание, алиасинг, гонки — все причины, по которым эта операция вне закона без пропуска.

**Разобранный пример 2 — два «писателя», законно.**

```rust
// raw_two_muts.rs
fn main() {
    let mut x = 1;

    let p1 = &raw mut x;
    let p2 = &raw mut x; // два "писателя" -- для сырых указателей законно

    unsafe {
        *p1 += 10;
        *p2 += 100;
        println!("x = {}", *p1);
    }
}
```

```
x = 111
```

То, за что §15.2.2 выдавал E0499 (и §23.1.2 — для двух потоков), здесь компилируется и работает. В одном потоке и с очевидным порядком операций это корректно; но именно так выглядит и дорога к гонке данных, если такие указатели разъедутся по потокам. XOR-правило никуда не делось — оно просто стало *вашим* обязательством вместо компиляторского.

**Разобранный пример 3 — арифметика, шаг в размер элемента.**

```rust
// raw_arith.rs
fn main() {
    let a = [10, 20, 30, 40, 50];
    let p = a.as_ptr();

    unsafe {
        println!("*p         = {}", *p);
        println!("*p.add(1)  = {}", *p.add(1));
        println!("*p.add(4)  = {}", *p.add(4));
    }

    let addr0 = p as usize;
    let addr1 = unsafe { p.add(1) } as usize;
    println!("шаг адреса = {} байт", addr1 - addr0);
}
```

```
*p         = 10
*p.add(1)  = 20
*p.add(4)  = 50
шаг адреса = 4 байт
```

`p.add(1)` — это `p + 1` из §6.2.2, вплоть до шага в размер элемента (4 байта для `i32`). Важная тонкость контракта (R9, документация `add`/`offset`): и исходный, и результирующий указатель обязаны оставаться в границах одного выделения (плюс «один за концом», как в C §6.2.2) — **сам `add` за границу уже UB**, даже без разыменования. Проверок нет ни на шаге, ни на чтении: `p.add(5)` тут не паника из §15.3.1, а прямое возвращение в мир главы 13.

**Null и висячие указатели.** `ptr::null()` даёт нулевой адрес, `is_null()` проверяет:

```rust
let p: *const i32 = ptr::null();
println!("{p:?}");            // 0x0
println!("{}", p.is_null());  // true
```

Висячий указатель создаётся до обидного просто — и посмотрите, как отреагировал rustc 1.97:

```rust
// raw_dangling.rs -- намеренное предупреждение
fn dangling() -> *const i32 {
    let local = 7;
    &local as *const i32
}
```

```
warning: function returns a dangling pointer to dropped local variable `local`
  = note: a dangling pointer is safe, but dereferencing one is undefined behavior
  = note: `#[warn(dangling_pointers_from_locals)]` on by default
```

Программа собирается и печатает адрес — создание висячего указателя действительно safe. Сравните с §15.2.3: там `dangle_c.c` получал от gcc `-Wreturn-local-addr` — *предупреждение* — и собирался; Rust для *ссылок* отвечал жёстким E0106, а для сырых указателей ведёт себя ровно как C: предупредить — и позволить. Note предупреждения формулирует границу идеально: «висячий указатель безопасен, разыменование его — undefined behavior».

**А что, если всё-таки разыменовать?** Обещание «unsafe возвращает поведение C» мы проверили экспериментом — разыменованием null в двух профилях (намеренное UB, дома повторять только ради науки):

- **dev-профиль**: программа упала с сообщением `panicked at ...: null pointer dereference occurred` и пометкой `thread caused non-unwinding panic. aborting.` — в отладочной сборке rustc вставляет проверку и превращает это UB в немедленную остановку (наблюдение на 1.97, не гарантия языка);
- **release (`-O`)**: та же программа *напечатала мусор* `32764` и вышла с кодом 0 — три прогона подряд.

Дежавю из §13.1.3: там C-программа с выходом за границу массива печатала мусор и делала вид, что всё хорошо. Внутри `unsafe` Rust — это снова тот самый мир: UB не обязано падать, оно обязано лишь быть невозможным в корректной программе.

**Инструменты: Miri — Valgrind для unsafe Rust.** В §7.2 нас спасали ASan и Valgrind, в §13.1.4 — UBSan и TSan. Для unsafe Rust есть родной аналог: **Miri** (R1, гл. 20.1) — интерпретатор, выполняющий программу с проверкой правил памяти. Он живёт в nightly-канале (первое знакомство с ним — каналы мы упоминали в §14.1.1):

```
rustup toolchain install nightly --profile minimal --component miri
cargo +nightly miri run
```

Прогоняем через Miri разыменование висячего указателя (`raw_dangling` плюс чтение) — и получаем отчёт уровня Valgrind из §7.2.2, с местом выделения и местом смерти:

```
error: Undefined Behavior: memory access failed: alloc168 has been freed,
       so this pointer is dangling
 --> src\main.rs:9:22
  |
9 |     let v = unsafe { *p }; // UB: чтение из мёртвой памяти
  |                      ^^ Undefined Behavior occurred here
  |
help: alloc168 was allocated here:
3 |     let local = 7;
help: alloc168 was deallocated here:
5 | }
```

А чтение `*p.add(3)` из массива на три элемента даёт: `attempting to access 4 bytes, but got alloc168+0xc which is at or beyond the end of the allocation of size 12 bytes` — байтовая арифметика сходится с §6.2.2 (3 × 4 байта = 0xc). Оговорки те же, что у санитайзеров (R1): Miri — динамический инструмент, проверяет только исполнившийся код; «Miri нашёл — это баг; Miri молчит — это не доказательство». Обновка для таблицы арсенала §13.1.4: у C — санитайзеры на каждый прогон; у safe Rust — статические гарантии; у unsafe Rust — снова рантайм-инструменты, но проверять надо лишь островки.

**Как это было в C.** Таблица соответствий — фактически словарь главы 6:

| C (гл. 6) | Rust |
|---|---|
| `const int *p` | `*const i32` |
| `int *p` | `*mut i32` |
| `&x` | `&raw const x` / `&raw mut x` |
| `NULL`, `p == NULL` | `ptr::null()`, `p.is_null()` |
| `p + 1`, `p[i]` | `p.add(1)`, `*p.add(i)` |
| `*p` | `unsafe { *p }` |

Разница одна, но решающая: в C эта таблица — *весь язык*, в Rust — карантинная зона. И предупреждения компиляторов симметричны до смешного: gcc ловит возврат адреса локальной (`-Wreturn-local-addr`), rustc — `dangling_pointers_from_locals`; оба — только простейшие случаи.

**Типичные ошибки и подводные камни.**

- *Разыменовать, не проверив на null* — §6.1.2 вернулся в полный рост: `is_null()` перед первым доступом, если null в принципе возможен.
- *Пережить владельца.* Сырой указатель не продлевает жизнь данным (в отличие от `Rc` из §22.1.4 и лайфтаймов ссылок): `Vec` перевыделился (§17.1.1) или владелец вышел из области — указатель тихо повис. Предупреждение компилятора ловит только «адрес локальной наружу».
- *`add` мимо выделения.* UB наступает уже на арифметике, не дожидаясь чтения (контракт R9) — Miri это ловит, глаза — нет.
- *Смешивать миры.* Пока жив `&mut T` на данные, читать их через сырой указатель — нарушение алиасинга (R7 относит его к UB). Правило гигиены: внутри одного unsafe-участка работать либо ссылками, либо указателями, не вперемешку.

**Источники.** R1 (гл. 20.1 — сырые указатели, «создание безопасно», Miri); R7 (Behavior considered undefined — висячие указатели, алиасинг, невалидные значения; арбитр); R9 (std::ptr и методы указателей — контракты `add`/`offset`, `is_null`); R8 (What Unsafe Rust Can Do — каталог UB со стороны Rustonomicon); R6 (unsafe-rust/dereferencing). Сторона C: §6.1–6.3, §13.1.3, §15.2.3 курса.

### §24.1.3. Инкапсуляция: сверхспособности на службе безопасного API

**Определения.** **Безопасная абстракция над unsafe** — функция или тип с обычным (safe) интерфейсом, внутри которого спрятан unsafe-код, а *инварианты, делающие его корректным, обеспечивает сама абстракция* — проверками, приватностью полей (гл. 20) и дисциплиной модуля. Это главный паттерн параграфа: unsafe не «протекает» к пользователю (R1: обёртка «не даёт unsafe расползтись по всем местам, где нужна функциональность»).

**Интуиция.** Библиотека с unsafe внутри — как аптека: в лаборатории стоят реактивы (сырые указатели, `UnsafeCell`), но покупателю через прилавок выдают только готовые препараты с инструкцией (safe API). За то, чтобы реактивы не оказались в торговом зале, отвечает аптека — не покупатель. Всё, чем вы пользовались в главах 17–23, устроено именно так.

**Разобранный пример 1 — `split_at_mut`, каноническая обёртка.** Задача из §23.1.2: разрезать срез на две изменяемые половины. Наивная попытка безопасным кодом — ⛔:

```rust
// split_two_mut_err.rs
fn my_split_at_mut(values: &mut [i32], mid: usize) -> (&mut [i32], &mut [i32]) {
    let len = values.len();
    assert!(mid <= len);
    (&mut values[..mid], &mut values[mid..]) // ⛔
}
```

```
error[E0499]: cannot borrow `*values` as mutable more than once at a time
  |
4 |     (&mut values[..mid], &mut values[mid..])
  |     --------------------------^^^^^^--------
  |     |     |                   second mutable borrow occurs here
  |     |     first mutable borrow occurs here
  |
  = help: use `.split_at_mut(position)` to obtain two mutable non-overlapping sub-slices
```

Знакомый E0499 (§15.2.2): два изменяемых заимствования одного среза. *Мы-то знаем*, что половины не пересекаются, — но borrow checker рассуждает про весь срез целиком. Компилятор даже подсказывает готовое решение из стандартной библиотеки — а мы напишем его сами, чтобы увидеть устройство:

```rust
// split_at_mut.rs
use std::slice;

fn my_split_at_mut(values: &mut [i32], mid: usize) -> (&mut [i32], &mut [i32]) {
    let len = values.len();
    let ptr = values.as_mut_ptr();

    assert!(mid <= len);

    unsafe {
        (
            slice::from_raw_parts_mut(ptr, mid),
            slice::from_raw_parts_mut(ptr.add(mid), len - mid),
        )
    }
}

fn main() {
    let mut v = [1, 2, 3, 4, 5, 6];
    let (left, right) = my_split_at_mut(&mut v, 2);
    left[0] = 100;
    right[0] = 300;
    println!("left  = {left:?}");
    println!("right = {right:?}");
    println!("v     = {v:?}");
}
```

```
left  = [100, 2]
right = [300, 4, 5, 6]
v     = [100, 2, 300, 4, 5, 6]
```

Разберём анатомию (пример по мотивам R1, гл. 20.1). `slice::from_raw_parts_mut(ptr, len)` — unsafe-функция «собери срез из указателя и длины» (представление срезов из §15.3.1 — указатель+длина — здесь стало конструктором). Её контракт (R9): память валидна, инициализирована, длина верна, алиасинга нет. Кто гарантирует? Наша функция: указатель взят из живого среза, `assert!(mid <= len)` отсекает выход за границы, половины `[0, mid)` и `[mid, len)` не пересекаются по построению. Поэтому **снаружи функция safe** — вызывающему не нужен ни `unsafe`, ни знание внутренностей. Именно так устроен настоящий `split_at_mut`, так же — `Vec`, `String`, `Mutex`: проверенное unsafe-ядро, безопасная оболочка.

**Разобранный пример 2 — `UnsafeCell`: фундамент внутренней изменяемости.** Вексель §22.2.1 к оплате: там мы сказали, что единственная легальная лазейка «изменять через `&T`» — примитив `UnsafeCell<T>`, на котором стоят `RefCell` и `Mutex` (R7, interior-mutability — арбитр). Построим на нём собственную мини-ячейку:

```rust
// mycell.rs
use std::cell::UnsafeCell;

struct MyCell<T> {
    value: UnsafeCell<T>,
}

impl<T: Copy> MyCell<T> {
    fn new(value: T) -> Self {
        MyCell { value: UnsafeCell::new(value) }
    }

    fn get(&self) -> T {
        // SAFETY: MyCell не выдаёт наружу ни одной ссылки внутрь,
        // а T: Copy -- мы возвращаем копию, не ссылку.
        unsafe { *self.value.get() }
    }

    fn set(&self, value: T) {
        // SAFETY: см. get -- ссылок внутрь не существует,
        // конфликт записи с чтением невозможен (один поток).
        unsafe { *self.value.get() = value; }
    }
}

fn main() {
    let c = MyCell::new(1);
    println!("до:    {}", c.get());
    c.set(42); // запись через &c -- разделяемую ссылку!
    println!("после: {}", c.get());
}
```

```
до:    1
после: 42
```

`UnsafeCell::get(&self)` возвращает `*mut T` из разделяемой ссылки — вот она, дверь в обход XOR-правила (§15.2.2). Обратите внимание: `set` принимает `&self`, не `&mut self` — и работает. Инварианты держит наш тип: наружу не выходит ни одной ссылки внутрь ячейки, только копии (`T: Copy`), поэтому классическая беда «изменили под носом у читателя» невозможна. Теперь видно, что *добавляют* обёртки из глав 22–23: `RefCell` — рантайм-счётчик заимствований поверх той же ячейки (паника вместо UB), `Mutex` — очередь и замок (ожидание вместо гонки). Стандартная `Cell<T>` из §22.2.1 — это, по сути, наш `MyCell`, доведённый до ума.

**Разобранный пример 3 — `unsafe impl Send`: вексель §23.2.3.** Auto-трейты `Send`/`Sync` компилятор выводит по составу типа — и сырой указатель в поле лишает тип обоих:

```rust
struct Handle {
    raw: *mut i32,
}
// thread::spawn(move || { ... h ... })  ⛔
```

```
error[E0277]: `*mut i32` cannot be sent between threads safely
  = help: within `{closure@...}`, the trait `Send` is not implemented for `*mut i32`
note: required because it appears within the type `Handle`
note: required by a bound in `spawn`
```

Тот же E0277, что ловил `Rc` в §23.2.2, — и note честно называет виновника: поле типа. Компилятор *не знает*, кто ещё владеет этим адресом, — и по умолчанию запрещает переезд. Если мы знаем больше компилятора — заявляем это под подпись:

```rust
// unsafe_impl_send.rs
use std::thread;

struct Handle {
    raw: *mut i32,
}

// SAFETY: Handle владеет указателем единолично: он создаётся
// только из Box::into_raw, копий указателя не существует.
unsafe impl Send for Handle {}

impl Handle {
    fn new(value: i32) -> Self {
        Handle { raw: Box::into_raw(Box::new(value)) }
    }

    fn into_value(self) -> i32 {
        // SAFETY: raw получен из Box::into_raw в new и никому не отдан.
        let boxed = unsafe { Box::from_raw(self.raw) };
        *boxed
    }
}

fn main() {
    let h = Handle::new(5);
    let t = thread::spawn(move || {
        println!("поток прочитал: {}", h.into_value());
    });
    t.join().unwrap();
}
```

```
поток прочитал: 5
```

`unsafe impl Send for Handle {}` — сверхспособность №4: ручная реализация unsafe-трейта. Сам трейт объявлен как `unsafe trait` именно затем, чтобы реализация требовала осознанной подписи (R9: ручной `impl Send` — обещание потокобезопасности, которое компилятор проверить не может). Пара `Box::into_raw` / `Box::from_raw` (R9, std::boxed) — «вынуть указатель из Box, не освобождая» и «вернуть указатель в Box, вернув и обязанность освобождать» — станет главным инструментом FFI-владения в §24.2.3.

Врезка — **ловушка точного захвата**. Первая версия этого примера делала в замыкании `h.raw` — и не собиралась *даже с* `unsafe impl Send`: точный захват полей (§21.1.2, издание 2024) утаскивает в замыкание не `Handle`, а само поле `*mut i32`, у которого `Send` нет. Лечение — заставить замыкание захватить структуру целиком (метод по `self`, как выше). Родня Copy-ловушки из §23.1.2: смотрите не на тип переменной, а на то, *что именно* захвачено.

**Остальные сверхспособности — коротко.**

*`static mut` (сверхспособность №3).* Изменяемая глобаль — та самая, из §4.2.2 и §11.2.2 тома I. Любой доступ — unsafe (гонки компилятор здесь не исключает — потому и пропуск). Издание 2024 закрутило гайки до упора — **ссылку на `static mut` брать нельзя вовсе**:

```
error: creating a shared reference to mutable static
  = note: shared references to mutable statics are dangerous; it's undefined
    behavior if the static is mutated or if a mutable reference is created
    for it while the shared reference lives
  = note: `#[deny(static_mut_refs)]` (part of `#[deny(rust_2024_compatibility)]`) on by default
help: use `&raw const` instead to create a raw pointer
```

— это ловит даже невинное `println!("{}", COUNTER)` (макросы печати берут аргументы по ссылке — потому в гл. 15 печать и не забирала владение): работать надо копией (`let now = COUNTER;`) или сырым указателем. Практический совет прежний: для глобального состояния — `Mutex` (§23.2.1) или передача владения, `static mut` — территория FFI и низкоуровневых библиотек.

*`union` (сверхспособность №5).* Зеркало §8.2.2: одна память — несколько прочтений, и **запись поля safe, чтение — unsafe** (E0133: «the field may not be properly initialized» — компилятор не отслеживает, какой участник актуален). Наш прогон: `union Bits { i: i32, f: f32 }` — `size_of` 4 (как `sizeof(union)` в §8.2.2), записали `f: 1.0`, прочитали `i = 0x3f800000` — битовое представление вещественного числа добыто без единого каста (тот самый трюк «разглядеть представление», что §8.2.2 проделывал с порядком байтов). Урок гл. 16 никуда не делся: «union + тег» в Rust пишется как `enum`; сырой `union` остался ради FFI (C-структуры с union-полями) и битовой хирургии.

**Гигиена unsafe — свод правил части** (R1, R6, R8 — согласованно):

1. Блоки — минимальные, каждый со своим `SAFETY:`.
2. Инварианты держит модуль: поля приватны (гл. 20), все пути к unsafe-ядру проходят через проверки (наш `assert!` в `split_at_mut`).
3. Интерфейс — safe всюду, где контракт можно проверить внутри; `unsafe fn` — только там, где обязательства ложатся на вызывающего.
4. Miri в тестах — как ASan в §7.2: дёшево и регулярно.
5. `unsafe` — не способ «победить borrow checker в споре»: если компилятор не пускает, сперва ищите безопасную формулировку (гл. 15–23 дали их немало), unsafe — последний довод.

**Как это было в C.** Весь том I — «одна большая unsafe-комната»: §7.2 учил ловить last-minute то, что здесь не компилируется. Финальное обновление таблицы арсенала §13.1.4: C — дисциплина + санитайзеры на каждый прогон; безопасный Rust — статическое исключение целых классов ошибок; unsafe Rust — маленькие острова, где снова нужны дисциплина + Miri, но теперь остров можно обойти пешком за минуту.

**Типичные ошибки и подводные камни.**

- *Safe-обёртка с невыполненным контрактом* — функция объявлена safe, а корректность зависит от «правильного» аргумента вызывающего. Это худшее из двух миров: пользователь не видит `unsafe` и не знает, что подписался. Либо проверка внутри (как `assert!`), либо честная `unsafe fn`.
- *`unsafe impl Send` «чтобы скомпилилось»* — §23.2.2 показывал, за что E0277 бережёт: подпись без обоснования превращает ошибку компиляции в гонку данных.
- *Выдать наружу ссылку из `UnsafeCell`* — вернуть `&T`/`&mut T` из `get`-подобного метода значит сломать все рассуждения о ссылках у клиента; наружу — копии либо контролируемые «пропуска» (путь `RefCell`).

**Источники.** R1 (гл. 20.1 — split_at_mut как канон безопасной абстракции, static mut, union, unsafe trait, Miri); R9 (std::slice::from_raw_parts_mut — контракт; std::cell — UnsafeCell; std::boxed — into_raw/from_raw; marker — Send/Sync и ручные impl); R7 (interior-mutability — арбитр; special-types-and-traits — auto-трейты); R8 (Working with Unsafe — границы модуля как граница unsafe); R6 (unsafe-rust — mutable static, unions, unsafe traits). Сторона C: §4.2.2, §8.2.2, §13.1.4 курса.

## Часть 24.2. FFI: мост между C и Rust

### §24.2.1. Из Rust в C: `extern "C"`, ABI и линковка

**Определения.** Чтобы функция одного языка вызвала функцию другого, им нужен договор ниже уровня языков: как передаются аргументы, кто чистит стек, как называется символ в объектнике. Такой договор называется **ABI** (application binary interface — двоичный интерфейс приложения). У C он на каждой платформе стандартный и стабильный — потому C и служит «латынью» межъязыкового общения. Rust объявляет чужие функции блоком:

```rust
unsafe extern "C" {
    fn abs(input: i32) -> i32;
}
```

- `extern "C"` — «эти функции живут снаружи и зовутся по C-ABI» (строка `"C"` — ABI по умолчанию и самый употребимый вариант; R7, external blocks);
- `unsafe extern` — требование **издания 2024**: сам блок объявлений — это уже обещание («такие функции с такими сигнатурами действительно существуют»), которое компилятор проверить не может. Без `unsafe` — ошибка `error: extern blocks must be unsafe` (прогнано);
- каждая функция из блока по умолчанию **unsafe для вызова**: у C-функции есть контракт (валидные указатели, NUL-строки...), проверить его Rust не в силах.

**Интуиция.** ABI — это розетка и вилка: двум приборам не нужно быть одной марки, чтобы работать от одной сети, — нужен общий разъём. `extern "C"` — просьба «говори через C-разъём», а сам блок объявлений — таможенная декларация: перечислено всё, что будет пересекать границу, и стоит ваша подпись (`unsafe`), что список правдив.

Вызов — как любой unsafe-вызов:

```rust
// abs_unsafe.rs
unsafe extern "C" {
    fn abs(input: i32) -> i32;
}

fn main() {
    let v = unsafe { abs(-3) };
    println!("abs(-3) по версии C: {v}");
}
```

```
abs(-3) по версии C: 3
```

`abs` — целочисленный модуль из `<stdlib.h>` (в томе I он нам не понадобился, но контракт очевиден): стандартная библиотека C уже прилинкована к каждой Rust-программе, объявления хватило. А для функций, которые безопасны при *любых* аргументах, издание 2024 добавило квалификатор `safe` (R7; R1, гл. 20.1):

```rust
unsafe extern "C" {
    safe fn abs(input: i32) -> i32;
}
// ...
println!("abs(-3) по версии C: {}", abs(-3)); // unsafe-блок не нужен
```

Печатает то же. Пометка `safe` — *ваше заявление* (блок-то `unsafe extern`): «эту функцию можно звать без пропуска, потому что никакими аргументами её не сломать». Для `abs` это правда; для `strlen` — нет (нужен валидный указатель на NUL-строку).

**Разобранный пример — своя C-библиотека.** Соединим то, что умеем из §11.2: компилируем C-модуль, кладём в статическую библиотеку, — и слинкуем её с Rust:

```c
/* mylib.c */
#include <stddef.h>

int c_add(int a, int b) {
    return a + b;
}

long long c_sum(const int *data, size_t len) {
    long long total = 0;
    for (size_t i = 0; i < len; i++) {
        total += data[i];
    }
    return total;
}
```

```
gcc -std=c17 -Wall -Wextra -c mylib.c -o mylib.o
ar rcs libmylib.a mylib.o
```

Rust-сторона объявляет функции и говорит линкеру, где их искать, — атрибут `#[link]` над блоком (R7) плюс путь поиска `-L`:

```rust
// use_mylib.rs
use std::ffi::c_int;

#[link(name = "mylib", kind = "static")]
unsafe extern "C" {
    safe fn c_add(a: c_int, b: c_int) -> c_int;
    unsafe fn c_sum(data: *const c_int, len: usize) -> i64;
}

fn sum(data: &[c_int]) -> i64 {
    // SAFETY: указатель и длина взяты из одного живого среза.
    unsafe { c_sum(data.as_ptr(), data.len()) }
}

fn main() {
    println!("c_add(2, 3) = {}", c_add(2, 3));

    let v = vec![10, 20, 30, 40];
    println!("c_sum(&v)   = {}", sum(&v));
}
```

```
rustc --edition 2024 use_mylib.rs -L .
```

```
c_add(2, 3) = 5
c_sum(&v)   = 100
```

Два языка в одном процессе — впервые в курсе. Обратите внимание на три решения: `c_add` помечен `safe` (числа не сломать), `c_sum` оставлен `unsafe` (указатель!), и сразу построена **безопасная обёртка** `sum(&[c_int])` по канону §24.1.3 — срез гарантирует согласованность указателя и длины, наружу unsafe не выходит. Это дежурный паттерн FFI: тонкий unsafe-слой объявлений + safe-API поверх (R8 строит так интерфейс к snappy).

**Типы-мостики.** Откуда `c_int`? Из `std::ffi` (R9) — там живут алиасы всех целочисленных типов C: `c_char`, `c_int`, `c_long`, `c_longlong`, `c_double`... Зачем они, если можно написать `i32`? Затем, что размеры типов C **непереносимы** — мы измеряли это в §2.1.2: на нашей LLP64-платформе `long` — 4 байта, а на LP64-Linux — 8. `c_long` раскрывается в правильный тип *на каждой платформе*, `i32` — только на некоторых. Правило: в сигнатурах FFI — алиасы `c_*` (или точные `i32`/`u64`, если заголовок C сам использует `int32_t`/`uint64_t` из §2.2.4); `usize` соответствует `size_t` (оба — «размерное» целое платформы).

**Как это было в C.** Конвейер §11.2 не изменился ни на шаг: препроцессор → компилятор → объектники → линкер. Новость одна: теперь на входе линкера объектники *двух* компиляторов. Прототип из `.h` (§11.2.2) превратился в `unsafe extern`-блок — и выполняет ту же работу: описывает сигнатуру, которой поверят на слово. И ошибки те же: забыли `-L .`/`#[link]` — `undefined reference`, знакомый по §11.2.1.

**Типичные ошибки и подводные камни.**

- *Неверная сигнатура в extern-блоке.* Компилятор Rust не видит C-заголовков: напишете `fn c_add(a: c_int) -> c_int` — он поверит, слинкуется (имя-то совпало), а на вызове — UB (нарушение ABI-договора, R7 относит вызов с неверным ABI к UB). Это «прототип разъехался с определением» из §11.2.2, но без линкера-свидетеля второго языка.
- *`safe` без оснований.* Квалификатор — не украшение: `safe fn strlen(...)` скомпилируется, а нулевой указатель уронит программу уже без единого слова `unsafe` в коде вызывающего.
- *`long` ≠ `i64`.* На нашей платформе `c_long` — это `i32` (§2.1.2!); привычка с Linux сломает Windows-сборку. Алиасы `c_*` избавляют от запоминания таблиц.

**Источники.** R1 (гл. 20.1 — extern "C", safe fn, вызов abs); R7 (external blocks — unsafe extern издания 2024, квалификаторы safe/unsafe, ABI-строки, #[link] — арбитр); R8 (FFI — библиотека + обёртка, паттерн safe-интерфейса); R9 (std::ffi — алиасы c_*); R6 (interoperability/with-c). Сторона C: §2.1.2, §11.2.1–11.2.2 курса.

### §24.2.2. Данные через границу: `#[repr(C)]`, строки и срезы

**Определения.** Вексель §8.2.3 к оплате: «почему для стыковки Rust и C нужен `#[repr(C)]`». По умолчанию Rust **не обещает** порядок полей структуры в памяти — компилятор вправе переставить их как выгодно (R7, Type layout: у репрезентации по умолчанию «нет иных гарантий»). Атрибут `#[repr(C)]` переключает раскладку на алгоритм C — тот самый, что мы разбирали руками в §8.2.3: поля по порядку объявления, паддинг по выравниванию (R7 приводит алгоритм; наш арбитр тома I — C5, §6.7.3.2 — описывает те же правила со стороны C).

**Интуиция.** Раскладка структуры — бланк протокола из §10.2.3: пока бланк заполняет и читает одна и та же программа, порядок граф — её внутреннее дело; как только бланк передают другой стороне, номер каждой графы становится частью договора. `#[repr(C)]` — подпись под договором о бланке; без неё Rust оставляет за собой право перепечатать бланк удобнее.

**Разобранный пример 1 — эксперимент с раскладкой.** Строим одну и ту же структуру тремя способами и меряем (`offset_of!` из `std::mem` — Rust-родня `offsetof` из §8.2.3):

```c
/* layout.c — сторона C */
struct Mixed { char a; int b; char c; };
/* печатаем sizeof и offsetof каждого поля */
```

```rust
// layout.rs — сторона Rust
use std::ffi::{c_char, c_int};
use std::mem::{offset_of, size_of};

#[repr(C)]
struct MixedC { a: c_char, b: c_int, c: c_char }

struct MixedRust { a: c_char, b: c_int, c: c_char }
```

Замеры (оба файла прогнаны):

| | размер | offset a | offset b | offset c |
|---|---|---|---|---|
| C: `struct Mixed` | 12 | 0 | 4 | 8 |
| Rust: `#[repr(C)]` | 12 | 0 | 4 | 8 |
| Rust: по умолчанию | **8** | **4** | **0** | **5** |

`#[repr(C)]` совпал с C байт в байт — включая оба паддинга из §8.2.3 (3 после `a`, 3 хвостовых). А раскладка по умолчанию — сюрприз: компилятор **переставил поля** (`b` уехал на смещение 0) и сжал структуру до 8 байт — ровно та оптимизация «отсортировать поля», которую в §8.2.3 мы делали руками, переставляя объявления. Для чистого Rust это подарок; для FFI — мина: C-сторона будет читать `b` по смещению 4, где у Rust лежит `a` и `c`. Отсюда железное правило (R8): **каждая структура, пересекающая границу языков, — `#[repr(C)]`**.

**Разобранный пример 2 — структуры едут через мост.** C-библиотека со структурой в обе стороны — принимает массив, возвращает по значению:

```c
/* reading.c */
struct Reading { char tag; int value; };

int reading_total(const struct Reading *arr, unsigned long long len);
struct Reading reading_make(char tag, int value);
```

```rust
// use_reading.rs
use std::ffi::{c_char, c_int};

#[repr(C)]
#[derive(Debug, Clone, Copy)]
struct Reading {
    tag: c_char,
    value: c_int,
}

#[link(name = "reading", kind = "static")]
unsafe extern "C" {
    unsafe fn reading_total(arr: *const Reading, len: u64) -> c_int;
    safe fn reading_make(tag: c_char, value: c_int) -> Reading;
}

fn main() {
    let data = [
        Reading { tag: b't' as c_char, value: 21 },
        Reading { tag: b'h' as c_char, value: 60 },
    ];
    // SAFETY: указатель и длина взяты из одного живого массива.
    let total = unsafe { reading_total(data.as_ptr(), data.len() as u64) };
    println!("total = {total}");

    let r = reading_make(b'p' as c_char, 750);
    println!("из C приехала структура: {r:?}");
}
```

```
total = 81
из C приехала структура: Reading { tag: 112, value: 750 }
```

Массив Rust-структур прочитан C-циклом (21 + 60 = 81), структура, собранная C-функцией, вернулась *по значению* и распечаталась через `derive(Debug)` из §16.1.3. `tag: 112` — это `'p'` в ASCII (§5.2.1): `c_char` — числовой тип, литерал `b'p'` (байтовый литерал — новинка, даёт `u8`) приводится оператором `as`.

**Строки: самый минный участок границы.** Всё главное сказано в §17.1.4, теперь оно сталкивается лбом с §5.2: строка C — `char *` с терминатором `'\0'`, без длины и без обещаний про кодировку; строка Rust — «байты + длина + инвариант UTF-8», *без* терминатора. Прямой обмен невозможен в обе стороны. Переводчики живут в `std::ffi` (R9):

- **`CString`** — владеющая C-строка: сделана из Rust-строки, хранит байты + добавленный `'\0'`, гарантирует отсутствие `'\0'` внутри;
- **`CStr`** — заимствованный вид на чужую C-строку (аналогия `String`/`&str` из §17.1.3): оборачивает `*const c_char`, добытый у C.

```rust
// strings.rs (фрагменты)
let owned = CString::new("hello, C").unwrap();
// SAFETY: as_ptr даёт живой указатель на NUL-терминированный буфер.
let n = unsafe { strlen(owned.as_ptr()) };
println!("strlen(\"hello, C\") = {n}");

// SAFETY: getenv возвращает NUL-терминированную строку или null.
let raw = unsafe { getenv(name.as_ptr()) };
if raw.is_null() { /* ... */ } else {
    // SAFETY: raw не null и указывает на строку окружения.
    let value = unsafe { CStr::from_ptr(raw) };
    println!("OS = {:?}", value.to_str());
}
```

```
strlen("hello, C") = 8
OS = Ok("Windows_NT")
```

Оба направления на живых функциях CRT: `strlen` посчитал наши байты (8 — терминатор не в счёт, ровно как `strlen` против `sizeof` в §5.2.1), `getenv` вернул указатель, `CStr::from_ptr` обернул, `to_str()` вернул `Result` (гл. 18!) — потому что C никому не обещал UTF-8. А защита `CString` от внутреннего нуля — не теория:

```rust
let bad = CString::new("до\0после");
println!("{bad:?}");
```

```
Err(NulError(4, [208, 180, 208, 190, 0, 208, 191, ...]))
```

`NulError(4, ...)` — ноль найден в байте 4 (кириллица — по 2 байта на букву, §17.1.4 в действии). C-строка с нулём внутри молча оборвалась бы на нём — `CString` отказывается создаваться.

**Срезы и `Option`.** Срез через границу не ходит целиком («указатель + длина» — деталь Rust, §15.3.1): его разбирают на `as_ptr()` и `len()` — как в `sum` из §24.2.1. Обратно — `slice::from_raw_parts` из §24.1.3. И приятный бонус от §16.2.3: нишевая оптимизация гарантирует, что `Option<&T>` и `Option<extern "C" fn(...)>` занимают ровно указатель (`None` = null) — благодаря этому nullable-указатели C выражаются честным `Option` (R8, nullable pointer optimization).

**Как это было в C.** Паддинг и выравнивание — §8.2.3 (наши замеры 12/8 воспроизвелись дословно); «структура пишется в файл с паддингом и порядком байтов машины» — §10.2.1 и §10.2.3: всё, что мы узнали про сырую память структур, здесь стало *протоколом обмена*. Строки — §5.2: `char *`-мир не изменился, изменилось то, что теперь на границе стоит таможня (`CString`/`CStr`), а не голая надежда.

**Типичные ошибки и подводные камни.**

- *Забытый `#[repr(C)]`* — работает «случайно» на простых структурах и ломается после невинного изменения полей или обновления компилятора: раскладка по умолчанию не обещана даже между сборками (R7).
- *Отдать в C `as_ptr()` от `String`* — у буфера `String` нет терминатора: `strlen` уйдёт гулять по чужой памяти (§5.2.3, только теперь без шансов на `-Wall`). Только `CString`.
- *Убить `CString` раньше времени*: `let p = CString::new("hi").unwrap().as_ptr();` — временное значение, не привязанное к переменной, умирает сразу по концу выражения — `p` виснет (об этой ловушке предупреждает документация `CString`, R9). Держите владельца живым в переменной, пока C пользуется указателем (наш `owned` — поэтому).
- *Поверить, что C прислал UTF-8* — `to_str()` не зря возвращает `Result`; путь «в лоб» `unwrap` — паника на первой же строке в национальной кодировке (§17.1.4).

**Источники.** R7 (Type layout — «нет гарантий» по умолчанию и алгоритм repr(C) — арбитр); R8 (FFI — repr(C)-правило, CString/NUL-строки, nullable pointer optimization); R9 (std::ffi — CString/CStr, различия строк C и Rust; std::mem — offset_of); R1 (гл. 20.1); C5 (§6.7.3.2 — раскладка структур со стороны C). Сторона C: §5.2, §8.2.3, §10.2.1–10.2.3 курса; Rust-сторона: §16.2.3 (нишевая оптимизация).

### §24.2.3. Из C в Rust: `#[unsafe(no_mangle)]`, владение и паника на границе

**Определения.** Обратное направление: C-программа вызывает функции, написанные на Rust. Для этого Rust-функция должна (R1, гл. 20.1):

1. иметь C-ABI: `pub extern "C" fn ...` — теперь `extern "C"` стоит на *определении*, а не в блоке объявлений;
2. сохранить своё имя в объектнике: атрибут **`#[unsafe(no_mangle)]`**.

Второе требует объяснения. Компилятор Rust по умолчанию **манглирует** (mangle — «коверкает») имена функций: вкодирует в символ путь модуля и версию crate, чтобы `min` из вашего кода не столкнулась с `min` из чужого (вспомните §11.2.1: две функции с одним именем — `multiple definition`; манглинг решает это на уровне имён). Проверяем инструментом `nm` из §11.2:

```
$ nm mangled.exe | grep add
00000001400014f0 t _RNvCsgHztoHSqMx8_7mangled11mangled_add
0000000140001610 T plain_add
```

Функция без атрибута существует под именем `_RNvCs..._7mangled11mangled_add` — C-код её по имени `mangled_add` не найдёт никогда. Функция с `#[unsafe(no_mangle)]` — честный символ `plain_add`, как у C-компилятора. Атрибут — из семейства unsafe-атрибутов издания 2024: отключая манглинг, *вы* отвечаете за то, что имя ни с кем не столкнётся. Старая запись `#[no_mangle]` в издании 2024 — ошибка компиляции с готовой подсказкой:

```
error: unsafe attribute used without unsafe
help: wrap the attribute in `unsafe(...)`
  |
3 | #[unsafe(no_mangle)]
```

**Интуиция.** До сих пор Rust ходил в гости (звал чужие функции); теперь гостей принимает он — а для этого дому нужны табличка с читаемым именем (`#[unsafe(no_mangle)]`), вход стандартной ширины (`pub extern "C"`) и внятные правила, кто моет посуду, — то есть кто владеет памятью, которую передали через порог.

**Разобранный пример — библиотека на Rust для программы на C.** Собираем «зеркало §24.2.1»: Rust-код → статическая библиотека → C-клиент. Заодно решим главный вопрос моста — **кто владеет памятью**. Экспортируем счётчик, живущий в куче Rust:

```rust
// rustlib.rs
use std::ffi::c_int;

#[unsafe(no_mangle)]
pub extern "C" fn rust_add(a: c_int, b: c_int) -> c_int {
    a + b
}

pub struct Counter {
    total: i64,
}

#[unsafe(no_mangle)]
pub extern "C" fn counter_new() -> *mut Counter {
    Box::into_raw(Box::new(Counter { total: 0 }))
}

/// SAFETY-контракт: p получен из counter_new и ещё не освобождён.
#[unsafe(no_mangle)]
pub unsafe extern "C" fn counter_add(p: *mut Counter, x: c_int) {
    let counter = unsafe { &mut *p };
    counter.total += x as i64;
}

/// SAFETY-контракт: p получен из counter_new и ещё не освобождён.
#[unsafe(no_mangle)]
pub unsafe extern "C" fn counter_total(p: *const Counter) -> i64 {
    unsafe { (*p).total }
}

/// SAFETY-контракт: p получен из counter_new; после вызова p мёртв.
#[unsafe(no_mangle)]
pub unsafe extern "C" fn counter_free(p: *mut Counter) {
    if !p.is_null() {
        // SAFETY: возвращаем владение Box и даём drop освободить память.
        drop(unsafe { Box::from_raw(p) });
    }
}
```

C-клиент видит `Counter` как **непрозрачный тип** (opaque type — R8): объявлен, но не определён, работать можно только указателем, — тот же приём, что `FILE *` из §10.1.1:

```c
/* main_c.c */
#include <stdio.h>

struct Counter; /* непрозрачный тип: содержимое знает только Rust */

int rust_add(int a, int b);
struct Counter *counter_new(void);
void counter_add(struct Counter *p, int x);
long long counter_total(const struct Counter *p);
void counter_free(struct Counter *p);

int main(void) {
    printf("rust_add(2, 3) = %d\n", rust_add(2, 3));

    struct Counter *c = counter_new();
    counter_add(c, 10);
    counter_add(c, 32);
    printf("counter_total = %lld\n", counter_total(c));
    counter_free(c);
    return 0;
}
```

Сборка. Библиотека: `rustc --edition 2024 --crate-type=staticlib rustlib.rs -o librustlib.a`. Каким системным библиотекам должен помочь линкер, rustc сообщает сам:

```
rustc --print native-static-libs --crate-type=staticlib rustlib.rs
note: native-static-libs: -lkernel32 -lntdll -luserenv -lws2_32 -ldbghelp
```

```
gcc -std=c17 -Wall -Wextra main_c.c -L . -lrustlib -lkernel32 -lntdll -luserenv -lws2_32 -ldbghelp -o app_c.exe
```

```
rust_add(2, 3) = 5
counter_total = 42
```

C-программа сложила числа Rust-функцией и погоняла объект в куче Rust. (Забудете системные библиотеки — получите россыпь `undefined reference to 'NtReadFile'...`: стандартная библиотека Rust едет внутри staticlib целиком — потому наш `librustlib.a` весит ~15 МБ — и хочет свои зависимости ОС; §11.2.1 подготовил вас к такому диагнозу.)

**Владение через границу.** Дисциплина — дословно §7.1.2, потому что C ничего другого и не умеет: договор «кто выделил — тот и освобождает». Пара `Box::into_raw`/`Box::from_raw` (знакомая по §24.1.3) реализует его со стороны Rust:

- `counter_new`: `Box::new` кладёт в кучу → `into_raw` *отпускает владение* (drop не придёт — гл. 15/22 приостановлены) и отдаёт голый указатель C;
- C хранит указатель, зовёт методы — владение всё это время «в отпуске»;
- `counter_free`: `from_raw` *возвращает владение* в `Box` → `drop` освобождает. Вызвать `free()` из C на этом указателе нельзя — аллокаторы могут не совпадать (R8); симметрично, Rust не должен освобождать то, что выделил `malloc` C-стороны, — этой дисциплине владения учил §7.1.2.

**Коллбэк — круг замыкается окончательно.** В §6.3.3 мы передавали C-функции `qsort` указатель на C-компаратор. Теперь компаратор — на Rust:

```rust
// qsort_cb.rs
use std::ffi::{c_int, c_void};

unsafe extern "C" {
    unsafe fn qsort(
        base: *mut c_void,
        nmemb: usize,
        size: usize,
        compar: unsafe extern "C" fn(*const c_void, *const c_void) -> c_int,
    );
}

/// SAFETY-контракт qsort: a и b указывают на живые элементы массива.
unsafe extern "C" fn cmp_int(a: *const c_void, b: *const c_void) -> c_int {
    let (x, y) = unsafe { (*(a as *const c_int), *(b as *const c_int)) };
    x - y
}

fn main() {
    let mut v = [42, 7, 19, 3, 25];
    // SAFETY: указатель/длина/размер описывают один живой массив,
    // компаратор читает ровно по одному int с каждой стороны.
    unsafe {
        qsort(
            v.as_mut_ptr() as *mut c_void,
            v.len(),
            size_of::<c_int>(),
            cmp_int,
        );
    }
    println!("после qsort из C: {v:?}");
}
```

```
после qsort из C: [3, 7, 19, 25, 42]
```

Всё из §6.3.3 на месте: `void *`-стирание типов (`c_void` — его Rust-имя), приведение в компараторе, «указатель на функцию» — только теперь тип компаратора записан как `unsafe extern "C" fn(...) -> c_int`, и Rust-массив отсортирован сишным алгоритмом. Замыкания из гл. 21 сюда не передать — у замыкания с захватом нет C-ABI (§21.1.2: это структура с данными); только `extern "C" fn`.

**Паника на границе.** Что будет, если Rust-функция, вызванная из C, запаникует (§18.1.1)? Раскрутка стека не умеет проходить через C-кадры — поэтому у `extern "C"`-функции паника обязана остановиться. Проверено:

```rust
#[unsafe(no_mangle)]
pub extern "C" fn rust_checked_div(a: i32, b: i32) -> i32 {
    a / b // при b == 0 -- паника Rust на границе с C
}
```

C-клиент зовёт `rust_checked_div(10, 0)` — и процесс *аварийно останавливается*:

```
thread '<unnamed>' (9556) panicked at panic_lib.rs:3:5:
attempt to divide by zero
thread '<unnamed>' (9556) panicked at ...core\src\panicking.rs:225:5:
panic in a function that cannot unwind
thread caused non-unwinding panic. aborting.
```

Паника (знакомая по витрине гл. 13 — деление на ноль, §13.2.3) добежала до C-ABI-границы и превратилась в abort: «panic in a function that cannot unwind». Это *защита*: улететь размоткой в C-код было бы UB (R8: граница, через которую могут идти паники, обязана использовать специальный ABI `"C-unwind"` — за рамками курса). Правило: **экспортируемая Rust-функция не должна паниковать** — ловите ошибки заранее и возвращайте код ошибки/спец-значение, как в C-приёмах обработки ошибок из главы 18 (коды возврата и договорённости — рубрика «Как это было в C» §18.1).

**Как это было в C.** `qsort` и указатели на функции — §6.3.3; `FILE *` как непрозрачный тип — §10.1.1; «кто выделил — тот освобождает» — §7.1.2; `nm` и столкновения имён — §11.2.1. Часть 24.2 не изобрела ни одной новой идеи — она соединила старые по обе стороны границы.

**Типичные ошибки и подводные камни.**

- *Забыть `#[unsafe(no_mangle)]`* — линкер C-стороны отвечает `undefined reference to 'rust_add'`: символ существует, но под манглированным именем (диагноз ставится `nm`, §11.2).
- *Освободить не тем аллокатором* — `free()` на `Box`-указателе или `Box::from_raw` на `malloc`-указателе: может «работать» неделями и падать по вторникам; только парные операции одной стороны.
- *Двойное освобождение через границу* — C дважды позвал `counter_free`: первый `from_raw` уже вернул память; проверка `is_null` в нашем коде не спасает (указатель C-стороны не обнулился — §7.2.1, та же грабля). Контракт + дисциплина, как в C.
- *Паника наружу* — любой `unwrap`/`panic!`/выход за границы массива в экспортируемой функции — abort всего процесса, C-код даже не узнает почему. API границы — только «ошибки значениями» (гл. 18).

**Источники.** R1 (гл. 20.1 — no_mangle, вызов Rust из других языков); R7 (external blocks / ABI — арбитр; поведение с неверным ABI — UB); R8 (FFI — Rust из C, деструкторы/владение, коллбэки, unwinding через границу, opaque-типы); R9 (std::boxed — into_raw/from_raw; std::ffi — c_void). Сторона C: §6.3.3, §7.1.2–7.2.1, §10.1.1, §11.2.1, гл. 18 (C-приёмы ошибок) курса.

### §24.2.4. Смешанный проект: два компилятора, одна сборка

**Определения.** Всё, что мы делали руками (`gcc -c` → `ar` → `rustc -L`), в настоящих проектах автоматизирует cargo через **сборочный скрипт** — файл `build.rs` в корне пакета (R10 упоминался в гл. 20; здесь — первый рабочий пример). Это обычная Rust-программа, которую cargo компилирует и запускает *до* сборки crate; её работа — подготовить всё внешнее: скомпилировать C-файлы, найти системные библиотеки, сгенерировать код. Компиляцией C из build.rs заведует crate **`cc`** — он сам находит подходящий компилятор платформы (у нас — наш MinGW-gcc), собирает статическую библиотеку и сообщает cargo, что и куда линковать.

**Интуиция.** cargo — генподрядчик, `cc` — субподрядчик по бетонным работам: в проекте написано, *что* построить, а каким инструментом заливать фундамент, решается на объекте (на нашей машине `cc` сам нашёл MinGW-gcc). Ручной конвейер из §24.2.1–24.2.3 никуда не делся — он просто выполняется без нас.

**Разобранный пример — C-модуль внутри cargo-пакета.** Структура проекта:

```
mixed_demo/
├── Cargo.toml
├── build.rs        ← сборочный скрипт
├── csrc/
│   └── fib.c       ← C-часть
└── src/
    └── main.rs     ← Rust-часть
```

```c
/* csrc/fib.c */
unsigned long long fib_c(unsigned int n) {
    unsigned long long a = 0, b = 1;
    for (unsigned int i = 0; i < n; i++) {
        unsigned long long next = a + b;
        a = b;
        b = next;
    }
    return a;
}
```

Зависимость подключается в секцию `[build-dependencies]` — зависимости *скрипта*, не программы (три пакета в замок: `cc`, `find-msvc-tools`, `shlex`):

```
cargo add cc --build
```

```rust
// build.rs
fn main() {
    cc::Build::new().file("csrc/fib.c").compile("fib");
    println!("cargo::rerun-if-changed=csrc/fib.c");
}
```

```rust
// src/main.rs
unsafe extern "C" {
    safe fn fib_c(n: u32) -> u64;
}

fn main() {
    for n in [10, 50, 90] {
        println!("fib({n}) = {}", fib_c(n));
    }
}
```

```
$ cargo run
   Compiling cc v1.3.0
   Compiling mixed_demo v0.1.0 (...)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 3.53s
     Running `target\debug\mixed_demo.exe`
fib(10) = 55
fib(50) = 12586269025
fib(90) = 2880067194370816120
```

Один `cargo run` — и C-файл скомпилирован, слинкован и вызван; `#[link]` не понадобился (cc сам передал cargo указания). Печать `cargo::rerun-if-changed=...` — договор о пересборке: тронули `fib.c` — скрипт перезапустится; это Makefile-зависимость из §11.2.3, пересказанная на языке cargo. Заметьте `fib(90)` — само число Фибоначчи считается в C-типе `unsigned long long` (§2.1.2) и без потерь приезжает в `u64`: таблица типов §14.2.2 работает на границе.

**Дальше по этой дороге (обзорно).** Ручное переписывание C-заголовков в `extern`-блоки утомительно и рискованно (§24.2.1: ошибку в сигнатуре никто не поймает). Инструмент **bindgen** генерирует Rust-объявления из `.h`-файлов автоматически — так работает связка в больших проектах (R6 показывает её на примере Android). Обратное направление — **cbindgen** — печёт `.h` из Rust-экспортов. Оба — за рамками курса: наши мосты достаточно малы, чтобы строить их руками и понимать каждый болт.

**Как это было в C.** `build.rs` — это Makefile из §11.2.3, у которого вместо целей — программа: та же задача (пересобрать только изменённое, знать флаги и пути), но описанная кодом, а не декларацией. А многоступенчатую сборку со статическими библиотеками мы прошли ещё в §11.2.3 — новых идей здесь нет, есть новая автоматизация.

**Типичные ошибки и подводные камни.**

- *`[dependencies]` вместо `[build-dependencies]`* — `cc` нужен скрипту, не программе; в неправильной секции сборка скрипта его не увидит (`unresolved import`).
- *Рассинхрон `.h`-заголовка и extern-блока* — при эволюции C-части Rust-объявления надо править руками (пока не bindgen); линкер сверяет только имена, не сигнатуры — §11.2.1 действует и здесь.
- *Забытый `rerun-if-changed`* — правки C-файла перестают попадать в сборку; «у меня старая версия функции» — первый кандидат на этот диагноз.

**Источники.** R10 (cargo: build scripts — назначение build.rs, build-dependencies; из реестра гл. 20); R8 (FFI — build.rs с cc как канонический приём подключения C); R6 (interoperability — bindgen); R1 (гл. 20.1). Сторона C: §11.2.3 курса.

Соберите статическую библиотеку, напишите три проверки: пустой срез, один элемент, максимум в середине. Ограничение: unsafe — только внутри обёртки, `SAFETY`-комментарий обязателен.

**Г24.Ч2.З9.** «Счётчик наоборот»: C-сторона владеет памятью. Напишите на C `struct Stats` (поля `long long sum; int n;`) и функции `stats_new` (через `malloc`, §7.1.2), `stats_add(struct Stats *, int)`, `stats_mean(const struct Stats *) -> double`, `stats_free`. Продемонстрируйте: среднее для 10, 20, 30 равно 20.0, и утечки нет (drop зовётся сам).

## Резюме главы

- **`unsafe` — это пять дверей, а не отключение проверок**: разыменовать сырой указатель, вызвать unsafe-функцию, тронуть `static mut`, реализовать unsafe-трейт, прочитать поле `union`. Borrow checker, типы, границы срезов — работают как работали. UB запрещено по обе стороны: `unsafe` лишь переносит обязанность его не допустить с компилятора на вас — и потому у каждого блока есть `SAFETY`-комментарий с доказательством.
- **Сырые указатели `*const T`/`*mut T` — это указатели C в резервации**: null, висячесть, алиасинг, арифметика `add` — всё вернулось (даже release-прогон с мусором вместо падения — привет §13.1.3), но только внутри помеченных блоков. Создание безопасно — небезопасен доступ. Miri — «Valgrind для unsafe»: отчёты с местом выделения и смерти памяти, как у старших братьев из §7.2.
- **Инкапсуляция — стратегия языка**: unsafe-ядро + safe-интерфейс, инварианты держит модуль (`split_at_mut`, наш `MyCell` на `UnsafeCell` — фундаменте `RefCell`/`Mutex` из глав 22–23, `unsafe impl Send` под подпись). Издание 2024 закрутило гайки: тело `unsafe fn` — safe по умолчанию, ссылки на `static mut` — ошибка, `extern`-блоки и `no_mangle` — с явным `unsafe`.
- **FFI стоит на C-ABI**: `unsafe extern "C"` объявляет чужое (квалификатор `safe` — ваше заявление о безвредности), `#[link]`+`-L` находят библиотеку, типы-мостики `c_int`/`c_long` прячут непереносимость размеров из §2.1.2. Сигнатуры на границе никто не сверяет — ошибка в объявлении означает UB, поэтому большие проекты генерируют их bindgen-ом.
- **Данные через границу — по договору**: структуры — только `#[repr(C)]` (раскладка по умолчанию не обещана — наш замер: компилятор переставил поля и сжал 12 байт в 8); строки — через таможню `CString`/`CStr` (терминатор, запрет внутренних нулей, проверка UTF-8); срезы — парой указатель+длина; владение — «кто выделил, тот освобождает» с парой `Box::into_raw`/`from_raw`; паника в `extern "C"`-функции — abort процесса, наружу только ошибки-значения.
- **Смешанный проект собирается одной командой**: `build.rs` + crate `cc` компилируют C-файлы внутри `cargo run` — Makefile §11.2.3, переписанный кодом.
