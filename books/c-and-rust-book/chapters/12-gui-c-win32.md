# Глава 12. GUI на C: событийная модель и Win32

## Введение

Одиннадцать глав подряд наши программы жили в терминале: `printf` — вывод, `scanf` — ввод, поток управления — сверху вниз до `return`. В этой главе мы напишем программы с **графическим интерфейсом** (graphical user interface, GUI) — с окнами, кнопками и полями ввода. Для этого придётся перестроить само мышление: GUI-программа не «выполняется по сценарию», а **отвечает на события** — и эта событийная архитектура, однажды понятая, встретится вам везде: от мобильных приложений до веб-страниц и игровых движков.

Инструментом будет **Win32 API** — родной программный интерфейс Windows. Выбор прагматичный: Win32 — это чистый C (не C++, не отдельный фреймворк), он входит в состав самой Windows и компилируется нашим MinGW-gcc без единой внешней зависимости. Плата за это — платформенность: программы главы работают только на Windows (обладателям Linux/macOS глава всё равно полезна: событийная модель универсальна, меняются лишь имена функций).

Эта глава — **контрольная работа тома I** и по статусу — **обзорная** (введение в GUI-парадигму, а не курс десктопной разработки). Она собирает весь материал глав 1–11 в одном проекте: оконная процедура — это указатель на функцию (§6.3.3); состояние приложения — структура (гл. 8); загрузка и сохранение текста — файлы и динамическая память (гл. 7, 10); сборка — многофайловый проект с Makefile и линковкой к системным библиотекам (гл. 11). После главы вы будете уметь: объяснить событийную модель, написать оконное приложение с виджетами с нуля и собрать его `make`-ом; итог — работающий мини-блокнот.

Все примеры собраны и прогнаны на машине курса (GCC 15.2, MinGW-w64, `-std=c17 -Wall -Wextra`; ни одного предупреждения); поведение кнопок и окон проверено запуском.

## Часть 12.1. Событийная модель и минимальное окно

### §12.1.1. Другая архитектура: события вместо сценария

**Определения.**

- **Событие** (event) — всё, что «случается» с программой извне: щелчок мыши, нажатие клавиши, перемещение окна, требование системы перерисовать содержимое. События приходят в любой момент и в любом порядке (C11, Window Messages).
- **Сообщение** (message) — способ, которым Windows доставляет событие программе: числовой код (например, `WM_LBUTTONDOWN` = 0x0201 — нажата левая кнопка мыши) плюс два целых параметра с деталями. Windows общается с окном программы, посылая ему сообщения (C11).
- **Событийная модель** (event-driven model): вместо сценария «сверху вниз» программа объявляет системе «вот функция — вызывай её на каждое событие» и крутит цикл ожидания. Сравните два скелета:

```
Консольная программа (гл. 1–11):       GUI-программа (эта глава):

main()                                  WinMain()
  прочитать ввод                          создать окно
  посчитать                               пока есть сообщения:
  напечатать                                получить сообщение
  return  (программа решает,                передать его окну   ←— система
           что и когда происходит)                                 вызывает НАШУ
                                          return                   функцию-обработчик
```

- **Windows API** (оно же Win32 API) — тысячи функций самой Windows: окна, файлы, процессы, графика. Всё объявлено в одном гигантском заголовке `<windows.h>` (вспомните §11.1.1: `#include` — буквальная вклейка; этот заголовок тянет за собой десятки вложенных). Реализации живут в системных библиотеках — прежде всего `user32` (окна и сообщения) и `gdi32` (рисование); их мы будем подлинковывать ключом `-l`, механику которого знаем из §11.2.3.
- **Дескриптор** (handle) — непрозрачное значение-«номерок», которым система идентифицирует свой объект: `HWND` — дескриптор окна, `HINSTANCE` — дескриптор загруженного модуля (нашей программы). Внутрь дескриптора заглядывать нельзя — только передавать его функциям API (аналогия: `FILE *` из гл. 10 — тоже «номерок», содержимое которого нас не касалось).
- Win32 использует собственные имена типов (объявлены в `<windows.h>`); таблица-переводчик, которой хватит на всю главу:

| Тип Win32 | Что это на языке C | Где встретится |
|---|---|---|
| `HWND`, `HINSTANCE` | дескрипторы (непрозрачные указатели) | окна, модуль программы |
| `UINT` | `unsigned int` | код сообщения |
| `WPARAM`, `LPARAM` | целые размером с указатель | параметры сообщения (C11: «integer values the size of a pointer width») |
| `LRESULT` | целое размером с указатель | ответ обработчика сообщения |
| `LPSTR` / `LPCSTR` | `char *` / `const char *` | строки (C11, Working with Strings) |
| `INT_PTR`, `LONG_PTR` | целые размером с указатель | преобразования число↔указатель |
| `WINAPI`, `CALLBACK` | соглашение о вызове (calling convention) — как параметры передаются функции | сигнатуры WinMain и оконной процедуры |

- **A/W-варианты.** Почти каждая функция Win32 со строками существует в двух видах: `...A` (ANSI, обычные `char`-строки) и `...W` (wide, UTF-16-строки `wchar_t`). Имя без суффикса — макрос (§11.1.2!), который разворачивается в W-вариант, если определён идентификатор `UNICODE`, и в A-вариант иначе (C11, Working with Strings). Microsoft рекомендует новым приложениям W-варианты («New applications should always call the Unicode versions»); наша глава сознательно использует режим по умолчанию MinGW — A-варианты с обычными `char *` — чтобы не уводить обзорную главу в тему кодировок. Ограничение честно назовём: A-функции работают в системной 8-битной кодировке, и для полноценной поддержки всех языков мира настоящему приложению нужен Unicode (тексты в примерах — латиница).

**Интуиция.** Консольная программа — повар с рецептом: шаг 1, шаг 2, готово. GUI-программа — дежурный администратор: сидит у телефона (цикл ожидания) и реагирует на звонки (сообщения) — какой поступил, тот и обрабатывает; нет звонков — ждёт. «Рецепта» нет: порядок действий определяет пользователь. Окно — это «приёмная» администратора, а система Windows — коммутатор, который решает, чьему окну адресовать каждый щелчок и каждое нажатие.

**Разобранные примеры.**

*Пример 1. GUI одной строкой: MessageBox.*

Самая простая функция с окном — `MessageBox`: модальный диалог с текстом, значком и кнопками. Она самодостаточна — вызывается даже из обычной консольной программы:

```c
#include <windows.h>
#include <stdio.h>

int main(void)
{
    int answer = MessageBox(NULL, "Hello from C!", "My first window",
                            MB_OK | MB_ICONINFORMATION);
    printf("MessageBox returned %d\n", answer); /* IDOK == 1 */
    return 0;
}
```

```
$ gcc -std=c17 -Wall -Wextra msgbox1.c -o msgbox1.exe -luser32
$ ./msgbox1.exe        # на экране — диалог с кнопкой OK; закрываем его
MessageBox returned 1
```

Разбор аргументов: `NULL` — нет окна-владельца; текст; заголовок; флаги — какие кнопки (`MB_OK`) и какой значок (`MB_ICONINFORMATION`) показать, флаги склеиваются побитовым ИЛИ (`|`, §2.2). Возвращаемое значение — какую кнопку нажали: `IDOK` == 1, `IDCANCEL` == 2, `IDYES` == 6, `IDNO` == 7 (C11, справочник MessageBox). Обратите внимание: программа **остановилась** внутри `MessageBox` и ждала человека — вернулась только после нажатия. Проверенный факт из справочника: у диалога `MB_OK` закрытие «крестиком» тоже возвращает `IDOK` — наш прогон это подтверждает (диалог закрыт крестиком, напечаталось 1).

*Пример 2. Возврат MessageBox — это уже интерактивность.*

```c
int answer = MessageBox(NULL, "Save the file?", "Question", MB_YESNO);
if (answer == IDYES)
    printf("user chose Yes (%d)\n", answer);
else
    printf("user chose No (%d)\n", answer);
```

```
$ ./msgbox2.exe        # нажимаем Yes
user chose Yes (6)
$ ./msgbox2.exe        # нажимаем No
user chose No (7)
```

Ветвление по действию пользователя — суть всякого GUI; пока — в вырожденном виде «один вопрос — один ответ». Нюанс из справочника (C11): у `MB_YESNO` нет кнопки Cancel — поэтому «крестик» у такого диалога не работает; если диалог должен быть «отменяемым», выбирайте наборы с Cancel (`MB_OKCANCEL`, `MB_YESNOCANCEL`).

*Пример 3. Читаем чужую сигнатуру.*

Потренируемся на объявлении из `<windows.h>` (упрощённо):

```c
int WINAPI MessageBoxA(HWND hWnd, LPCSTR lpText, LPCSTR lpCaption, UINT uType);
```

По таблице типов: функция с соглашением вызова `WINAPI`, берёт дескриптор окна, две `const char *`-строки и `unsigned`-флаги, возвращает `int`. А `MessageBox` — макрос, разворачивающийся в `MessageBoxA` (наш случай) или `MessageBoxW`. Такое чтение сигнатур — главный навык работы с любым чужим API: тип скажет больше, чем догадка.

**Связь с практикой.** Событийная модель — не «особенность Windows», а стандарт индустрии: так устроены интерфейсы Android и iOS, браузерный JavaScript (обработчики `onclick`), Qt и GTK на Linux. Выучив её на Win32 — самом «голом» из событийных API, — вы будете узнавать те же кости под кожей любого фреймворка. `MessageBox` же — практичный инструмент диагностики GUI-программ, где нет `printf`-консоли: быстрый способ показать «я дошёл досюда».

**Типичные ошибки и подводные камни.**

- **Ждать от GUI-программы «порядка выполнения».** Его нет: код главы 12 не «выполняется сверху вниз», а нарезан на обработчики, которые система дёргает в неизвестном заранее порядке. Кто привык к консольному мышлению, первое время ищет «где же main-логика» — привыкайте искать «кто на какое сообщение отвечает».
- **Игнорировать возврат MessageBox** с вопросительными кнопками: спросили «Save?» — обязаны разобрать ответ, иначе диалог — декорация.
- **Русские строки в A-функциях.** Работают только пока системная кодировка совпадает с кодировкой исходника; при несовпадении — «кракозябры». Полноценное решение — W-функции и `UNICODE` (за рамками главы; C11, Working with Strings).
- **Смешивать `MB_`-флаги бездумно.** Из каждой группы (кнопки, значок, модальность) берётся ровно один флаг; «два набора кнопок» задать нельзя.

**Источники:** C11 (Learn to Program for Windows: «Window Messages» — события и сообщения; «Working with Strings» — типы строк, A/W и UNICODE; справочник MessageBox — флаги, возвраты, поведение крестика); C12 (гл. 1 «Getting Started» — событийная архитектура Windows-программ, гл. 3); C6/C5 — языковая часть (макросы, `|`, типы).

### §12.1.2. Каркас окна: WinMain, класс окна, CreateWindowEx

**Определения.**

- **`WinMain`** — точка входа GUI-программы Windows; ей соответствует стартовый код, который вместо `main` вызывает функцию с сигнатурой (C11, WinMain: The Application Entry Point):

  ```c
  int WINAPI WinMain(HINSTANCE hInstance, HINSTANCE hPrevInstance,
                     LPSTR pCmdLine, int nCmdShow);
  ```

  `hInstance` — дескриптор нашего модуля (нужен системным вызовам, чтобы знать, «от чьего имени» создаются окна); `hPrevInstance` — «наследие 16-битных Windows, теперь всегда ноль»; `pCmdLine` — командная строка одной строкой; `nCmdShow` — как показать главное окно (свернуть/развернуть/обычно). Возврат ОС не использует — это код статуса, как у `main`.
- **Класс окна** (window class) — зарегистрированный у системы набор общих свойств для окон одного типа: главное — **какая функция обрабатывает их сообщения**. Описывается структурой `WNDCLASS` (гл. 8 в действии), регистрируется функцией `RegisterClass` (C11, Creating a Window). Это не класс из ООП-языков — просто запись в таблице системы.
- Поля `WNDCLASS`, которые мы заполняем (остальные оставляем нулями — для этого инициализация `= {0}`, §8.1):
  - `lpfnWndProc` — **указатель на функцию**-обработчик (§6.3.3 наконец стреляет в полную силу);
  - `hInstance` — чей класс;
  - `lpszClassName` — имя класса (строка; локальна для процесса);
  - `hCursor` — курсор над окном; `LoadCursor(NULL, IDC_ARROW)` — обычная стрелка (если оставить NULL, форму курсора пришлось бы задавать вручную — C11, справочник WNDCLASS);
  - `hbrBackground` — кисть фона; допустим системный цвет по конвенции «код цвета + 1»: `(HBRUSH)(COLOR_WINDOW + 1)` — стандартный фон окон (C11, там же).
- **`CreateWindowEx`** — создаёт окно зарегистрированного класса; возвращает `HWND` или `NULL` при неудаче (C11, справочник CreateWindowEx). Параметры разберём на листинге.
- **`ShowWindow(hwnd, nCmdShow)`** — делает окно видимым (создаётся оно скрытым).
- **Сборка.** GUI-программе нужны `user32` (окна) и, если рисуем, `gdi32` (§12.2.3); ключ **`-mwindows`** говорит компоновщику пометить в заголовке исполняемого файла подсистему **GUI** — иначе Windows откроет рядом с окном ещё и чёрную консоль. По документации GCC (C17): `-mwindows` «instructs the linker to set the PE header subsystem type appropriately», `-mconsole` — аналогично для консольных программ (подсистема по умолчанию).

**Интуиция.** Три шага создания окна — как открытие кафе по франшизе: `RegisterClass` — регистрируем франшизу (бренд, правила обслуживания = оконная процедура); `CreateWindowEx` — строим конкретное кафе (адрес, вывеска = координаты, заголовок); `ShowWindow` — перерезаем ленточку. Можно открыть десять кафе одной франшизы: у всех общие правила (один `lpfnWndProc`), но у каждого свои вывеска и адрес (свои «инстанс-данные» — заголовок, позиция, содержимое).

**Разобранные примеры.**

*Пример 1. Минимальное окно — «Hello, Windows» с разбором каждой строки.*

```c
#include <windows.h>

LRESULT CALLBACK WindowProc(HWND hwnd, UINT uMsg, WPARAM wParam, LPARAM lParam)
{
    switch (uMsg) {
    case WM_DESTROY:          /* окно уничтожено — завершаем цикл сообщений */
        PostQuitMessage(0);
        return 0;
    }
    return DefWindowProc(hwnd, uMsg, wParam, lParam); /* всё остальное — системе */
}

int WINAPI WinMain(HINSTANCE hInstance, HINSTANCE hPrevInstance,
                   LPSTR pCmdLine, int nCmdShow)
{
    (void)hPrevInstance; /* всегда NULL — наследие 16-битных Windows */
    (void)pCmdLine;

    /* 1. Описываем и регистрируем класс окна */
    const char CLASS_NAME[] = "HelloWindowClass";

    WNDCLASS wc = {0};
    wc.lpfnWndProc   = WindowProc;                  /* указатель на функцию! */
    wc.hInstance     = hInstance;
    wc.lpszClassName = CLASS_NAME;
    wc.hCursor       = LoadCursor(NULL, IDC_ARROW); /* обычная стрелка */
    wc.hbrBackground = (HBRUSH)(COLOR_WINDOW + 1);  /* системный цвет фона */

    RegisterClass(&wc);

    /* 2. Создаём окно этого класса */
    HWND hwnd = CreateWindowEx(
        0,                   /* расширенные стили — не нужны */
        CLASS_NAME,          /* какой класс */
        "Hello, Windows",    /* текст заголовка */
        WS_OVERLAPPEDWINDOW, /* рамка, заголовок, кнопки свернуть/развернуть */
        CW_USEDEFAULT, CW_USEDEFAULT, /* позицию выберет система */
        480, 320,            /* ширина и высота */
        NULL,                /* нет родителя: окно верхнего уровня */
        NULL,                /* нет меню */
        hInstance,
        NULL);               /* дополнительных данных нет */
    if (hwnd == NULL)
        return 1;

    ShowWindow(hwnd, nCmdShow);

    /* 3. Цикл сообщений — «сердце» программы */
    MSG msg = {0};
    while (GetMessage(&msg, NULL, 0, 0) > 0) {
        TranslateMessage(&msg);
        DispatchMessage(&msg);
    }
    return 0;
}
```

```
$ gcc -std=c17 -Wall -Wextra hello_win.c -o hello_win.exe -luser32 -mwindows
$ ./hello_win.exe      # открывается пустое окно 480×320 «Hello, Windows»;
                       # оно перетаскивается, сворачивается, закрывается крестиком
```

Построчно, сверху вниз. `WindowProc` — та самая функция-обработчик; её сигнатура фиксирована системой (C11, Writing the Window Procedure): дескриптор окна, код сообщения, два параметра-детали; подробный разбор — в §12.1.3, пока запомните паттерн «`switch` по коду + `DefWindowProc` для всего необработанного». `(void)hPrevInstance;` — идиома «параметр намеренно не используется»: обязаны принять по сигнатуре, но не трогаем, а `-Wextra` не любит неиспользуемых параметров. Дальше три шага из «Интуиции»: класс → окно → показ. `WS_OVERLAPPEDWINDOW` — склейка флагов стандартного окна (заголовок, рамка, системное меню, кнопки свернуть/развернуть — C11, Creating a Window); `CW_USEDEFAULT` — «позицию выбери сама». Проверка `hwnd == NULL` — та же дисциплина, что с `malloc` (§7.1.2) и `fopen` (§10.1.2): создание может не удаться. Цикл сообщений разберём в §12.1.3 — но заметьте уже сейчас: **после `ShowWindow` наша программа не делает ничего** — только качает сообщения; вся содержательная работа происходит в `WindowProc`, которую вызываем не мы.

*Пример 2. Подсистема: зачем нужен `-mwindows`.*

Соберём одну и ту же программу двумя командами и заглянем в заголовок исполняемого файла утилитой `objdump` (входит в MinGW, как `ar` из §11.2.3):

```
$ gcc hello_win.c -o hw_gui.exe -luser32 -mwindows
$ objdump -x hw_gui.exe | grep Subsystem
Subsystem               00000002        (Windows GUI)
$ gcc hello_win.c -o hw_cui.exe -luser32
$ objdump -x hw_cui.exe | grep Subsystem
Subsystem               00000003        (Windows CUI)
```

Оба файла работают, но CUI-вариант (console user interface), запущенный из проводника, откроет **лишнее чёрное консольное окно**. Подсистема — просто поле в заголовке файла: Windows читает его и решает, выдавать ли процессу консоль. Для GUI-программ — всегда `-mwindows`; наши консольные эксперименты (пример 1 §12.1.1) собираются без него — им консоль как раз нужна.

> **Платформенное примечание** (проверено на машине курса; детали тулчейна, не гарантии языка). MinGW-w64 подлинковывает `user32` в списке библиотек по умолчанию — `hello_win.c` собрался бы и без `-luser32`. А вот `gdi32` в списке по умолчанию нет: его добавляет либо явный `-lgdi32`, либо сам `-mwindows` (видно в выводе `gcc -dumpspecs`: `%{mwindows:-lgdi32 -lcomdlg32}`). Мы **всегда пишем библиотеки явно**: полагаться на невидимые умолчания конкретного тулчейна — плохая инженерия; команда сборки должна читаться как список зависимостей (§11.2.3).

*Пример 3.* ⛔ *Окно, которое не родилось.*

Что будет, если оконная процедура «обработает» все сообщения, не делая ничего — и не позовёт `DefWindowProc`?

```c
/* Оконная процедура-«заглушка»: молча съедает все сообщения */
LRESULT CALLBACK BadProc(HWND hwnd, UINT uMsg, WPARAM wParam, LPARAM lParam)
{
    (void)hwnd; (void)uMsg; (void)wParam; (void)lParam;
    return 0; /* НЕТ вызова DefWindowProc — и это ошибка */
}
```

Остальное — как в примере 1 (для удобства эксперимента программа консольная, с `main` и `printf`; дескриптор модуля в такой программе берут функцией `GetModuleHandle(NULL)` — C11):

```
$ gcc -std=c17 -Wall -Wextra create_fail.c -o create_fail.exe -luser32
$ ./create_fail.exe
CreateWindowEx failed: the window was never created
$ echo $?
1
```

Окно даже не создалось! Причина документирована (C11, справочник WM_NCCREATE): ещё **внутри** `CreateWindowEx` окну приходит сообщение `WM_NCCREATE`, и обработчик обязан вернуть TRUE (ненулевое), чтобы создание продолжилось; вернул 0 — «`CreateWindow` or `CreateWindowEx` function will return a `NULL` handle». Наша заглушка вернула 0 — система поняла это как «отменить создание». Отсюда два урока: сообщения начинают приходить раньше, чем `CreateWindowEx` вернётся (проверим логом в §12.1.3), и у сообщений бывают **контракты на возвращаемое значение** — потому необработанное всегда сдаём `DefWindowProc`, который знает правильные ответы за нас.

**Связь с практикой.** Каркас из примера 1 — это скелет каждой Win32-программы: Petzold (C12, гл. 3) даёт практически тот же листинг, и через 25 лет он компилируется без изменений — редкая для индустрии стабильность API (программы для Windows 95 работают на Windows 11). Писать его руками вы будете нечасто — IDE и фреймворки генерируют каркас за вас, — но **читать** придётся: при отладке любого GUI-приложения под Windows вы упрётесь именно в эти вызовы.

**Типичные ошибки и подводные камни.**

- **Вернуть 0 на всё и не звать `DefWindowProc`** (пример 3) — окно не создаётся; симптом: `CreateWindowEx` возвращает NULL без видимых причин.
- **Забыть `ShowWindow`.** Окно создаётся скрытым: программа работает, в панели задач пусто, окна нет. Классическая «программа зависла, хотя работает».
- **Не проверить `hwnd == NULL`.** Дальнейшие вызовы с NULL-дескриптором тихо откажут — та же дисциплина проверки, что у `malloc`/`fopen`.
- **Собрать GUI-программу без `-mwindows`** — паразитная консоль рядом с окном (пример 2). Обратная ошибка: консольную утилиту с `-mwindows` — программа «молчит» (консоли-то нет).
- **Перепутать порядок параметров `CreateWindowEx`.** Их одиннадцать; компилятор поймает не всё — многие соседние параметры совместимы по типам. Сверяйтесь с образцом или справочником, не с памятью.

### §12.1.3. Жизнь сообщений: очередь, цикл, оконная процедура

**Определения.**

- **Очередь сообщений** (message queue). Для каждого потока, создавшего окно, система заводит скрытую очередь; события, адресованные окнам потока, встают в неё (C11, Window Messages). Напрямую очередь недоступна — только через функции:
- **`GetMessage(&msg, NULL, 0, 0)`** — достаёт сообщение из головы очереди в структуру `MSG`; если очередь пуста — **блокируется** и ждёт (программа спит, не жжёт процессор). Возвраты (C11, справочник GetMessage): достали обычное сообщение — ненулевое; достали `WM_QUIT` — **0**; ошибка (например, битый дескриптор) — **−1**. Документация прямо предупреждает: из-за трёх исходов писать `while (GetMessage(...))` в лоб не стоит; наша форма `> 0` завершает цикл и по концу работы, и по ошибке.
- **`TranslateMessage(&msg)`** — служебный шаг для клавиатуры (превращает нажатия в символы); просто вызывайте перед DispatchMessage (C11).
- **`DispatchMessage(&msg)`** — просит систему **вызвать оконную процедуру** окна-адресата. Вот где замыкается круг: система смотрит, какому окну адресовано сообщение, находит в таблице классов указатель `lpfnWndProc` — и вызывает нашу функцию (C11). Это тот же механизм, что в `qsort` (§6.3.3): мы отдали указатель на функцию, «цикл» крутит чужой код, а наш вызывается по необходимости. Разница лишь в масштабе: у `qsort` цикл жил внутри одного вызова, здесь — всю жизнь программы.
- **Оконная процедура** (window procedure) — функция вида `LRESULT CALLBACK WindowProc(HWND, UINT, WPARAM, LPARAM)`: кому пришло, что пришло, две «детали» (их смысл зависит от кода сообщения — для каждого кода смотрим справочник, как для `errno` в гл. 10). Типичная реализация — `switch` по коду; всё, что не обрабатываем, передаём **`DefWindowProc`** — обработчику по умолчанию (C11, Writing the Window Procedure).
- **Штатное завершение** — цепочка из трёх разных сообщений (C11, Closing the Window):
  1. Пользователь жмёт крестик (или Alt+F4) → окну приходит **`WM_CLOSE`**: «просят закрыться». Можно спросить подтверждение; согласны — зовём `DestroyWindow(hwnd)`; не обрабатываем вовсе — `DefWindowProc` сам вызовет `DestroyWindow`.
  2. Окно уничтожается → приходит **`WM_DESTROY`**: «тебя уже сносят». Стандартный ответ главного окна — `PostQuitMessage(0)`: положить в очередь **`WM_QUIT`**.
  3. `GetMessage` достаёт `WM_QUIT` → возвращает 0 → цикл `while` заканчивается → `WinMain` доходит до `return`. Причём `WM_QUIT` до оконной процедуры не доходит — он «гасит» цикл ещё в `GetMessage` (C11, Window Messages).
- **Отправленные и положенные** (sent vs posted, обзорно): не всё идёт через очередь — часть сообщений система вручает оконной процедуре напрямую, минуя очередь (мы видели: `WM_NCCREATE` пришёл изнутри `CreateWindowEx`, когда цикл ещё не запускался). Для нас разница пока не важна — обработчик один и тот же (C11).

**Интуиция.** Вся конструкция — конвейер с одним рабочим. Очередь — лента с деталями-сообщениями. `GetMessage` — взять деталь (ленты пустая — стоим). `DispatchMessage` — отнести детали её станку: система по адресу на детали (`HWND`) находит станок и запускает его программу (`lpfnWndProc`). `WM_QUIT` — красная кнопка «смена окончена». А `DefWindowProc` — ветеран-наставник за спиной: всё, что молодой рабочий не умеет (перетаскивание окна, реакция на крестик, отрисовка рамки), молча делает он. Именно поэтому наш «пустой» пример 1 §12.1.2 вёл себя как полноценное окно: 95 % поведения — заслуга `DefWindowProc`.

**Разобранные примеры.**

*Пример 1. Смотрим поток сообщений своими глазами.*

Соберём консольный (нарочно: нужен `printf`) вариант окна, где процедура логирует каждый вызов; несколько знакомых кодов подпишем:

```c
static int n = 0; /* сквозной номер сообщения */

LRESULT CALLBACK LogProc(HWND hwnd, UINT uMsg, WPARAM wParam, LPARAM lParam)
{
    n++;
    printf("%3d: 0x%04X%s\n", n, uMsg, msg_name(uMsg)); /* msg_name: switch по кодам */

    switch (uMsg) {
    case WM_DESTROY:
        PostQuitMessage(0);
        return 0;
    }
    return DefWindowProc(hwnd, uMsg, wParam, lParam);
}
```

Запуск: показали окно, ничего не трогали, закрыли крестиком. Сокращённый лог прогона (полный — 40 строк; номера и состав у вас будут слегка отличаться — это внутренняя кухня системы, как адреса в §6.1.1):

```
  1: 0x0024
  2: 0x0081 (WM_NCCREATE)
  4: 0x0001 (WM_CREATE)
-- CreateWindowEx has returned; showing the window --
  5: 0x0018
 20: 0x0005 (WM_SIZE)
 28: 0x000F (WM_PAINT)
 29: 0x0010 (WM_CLOSE)
 39: 0x0002 (WM_DESTROY)
-- message loop is over: 40 messages total --
```

Читаем как детектив. Сообщения №1–4 пришли **до** возврата `CreateWindowEx` (маркер `--...--` печатается после него) — и даже раньше `WM_NCCREATE` есть жизнь (0x0024 — системный запрос о допустимых размерах окна). Дальше показ: серия установочных сообщений, `WM_SIZE` (окну сообщили размер), `WM_PAINT` (пора рисовать — §12.2.3). Мы «ничего не делали» — а окно получило 28 сообщений ещё до нашего крестика. Крестик: `WM_CLOSE` (мы не обрабатываем — `DefWindowProc` сносит окно) → хвост служебных → `WM_DESTROY` (наш `PostQuitMessage`) → цикл завершён. Вся глава — в одном логе.

*Пример 2. Вежливое закрытие: спрашиваем подтверждение.*

`WM_CLOSE` — «просьба», и её можно оспорить (образец из C11, Closing the Window):

```c
    case WM_CLOSE: /* пользователь нажал «крестик» — спрашиваем подтверждение */
        if (MessageBox(hwnd, "Really quit?", "Confirm", MB_OKCANCEL) == IDOK)
            DestroyWindow(hwnd);
        return 0; /* Cancel: ничего не делаем — окно живёт дальше */
```

Прогон (проверено на машине курса): крестик → диалог; Cancel — окно остаётся жить и работать; снова крестик → OK — окно закрывается, процесс завершается с кодом 0. Заметьте `hwnd` первым аргументом `MessageBox` — теперь у диалога есть окно-владелец, и он ведёт себя как «дочерний» вопрос именно этого окна.

*Пример 3.* ⛔ *Классика: забытый PostQuitMessage — процесс-зомби.*

Уберём из примера 1 §12.1.2 обработку `WM_DESTROY` целиком (всё уходит в `DefWindowProc`). Собираем, запускаем, закрываем крестиком: окно честно исчезло (`DefWindowProc` на `WM_CLOSE` вызвал `DestroyWindow`)… а **процесс жив** — висит в диспетчере задач без единого окна (проверено). Почему: `DefWindowProc` не вызывает `PostQuitMessage` за вас — `WM_QUIT` в очередь никто не положил, `GetMessage` продолжает ждать сообщений для окон, которых больше нет. Это не утечка памяти из гл. 7, но родственная болезнь — «утечка процесса»: ресурс, который некому освободить. Диагноз тот же, что у Valgrind: инструмент (диспетчер задач) + дисциплина (у главного окна `WM_DESTROY` → `PostQuitMessage` — всегда).

**Связь с практикой.** Цикл сообщений объясняет два вечных феномена Windows. Первый — «программа не отвечает»: если оконная процедура ушла в долгую работу (файл на гигабайт, сетевой запрос), цикл не качает сообщения, окно не перерисовывается и не реагирует — система рисует «призрака» и предлагает снять процесс (C11 предупреждает об этом прямо: избегайте длинной работы в оконной процедуре; лекарство — фоновые потоки, тема тома II, гл. 23). Второй — почему «зависшее» окно всё же перетаскивается: часть поведения исполняет система через `DefWindowProc`-механику. Модель «очередь + диспетчер + обработчики» вы встретите и вне GUI: игровые движки (game loop), сетевые серверы (event loop), даже embedded-прошивки устроены так же.

**Типичные ошибки и подводные камни.**

- **`while (GetMessage(...))` без `> 0`.** При ошибке GetMessage вернёт −1, что в C истинно (§3.1) — вечный цикл на битом дескрипторе. Форма `> 0` или развёрнутая проверка из справочника (C11).
- **Забыть `PostQuitMessage`** (пример 3) — процесс-зомби. Проверяйте диспетчером задач после закрытия своих программ.
- **Тяжёлая работа в оконной процедуре** — «не отвечает». Пока наши программы мгновенны; правило запомните сейчас.
- **Проглотить сообщение с контрактом.** Не знаешь, что ответить, — отдай `DefWindowProc`; «вернуть 0 на всякий случай» уже сломало нам создание окна (⛔ §12.1.2).
- **Ждать `WM_QUIT` в оконной процедуре.** Он туда не приходит — обрабатывать его в `switch` бессмысленно (C11).

**Источники:** C11 (Learn to Program for Windows: «Window Messages» — очередь, цикл, posted/sent, WM_QUIT; «Writing the Window Procedure» — сигнатура, DefWindowProc, блокировка процедуры; «Closing the Window» — WM_CLOSE/WM_DESTROY/PostQuitMessage; справочник GetMessage — возвраты и предупреждение о −1); C12 (гл. 3 — цикл сообщений и оконная процедура); C6/C5 — языковая часть (истинность, switch).

## Часть 12.2. Виджеты, состояние и контрольный проект

### §12.2.1. Виджеты — это дочерние окна

**Определения.**

- **Элемент управления** (control), разговорно **виджет** — кнопка, поле ввода, надпись. В Win32 виджет — это **обычное окно**, только дочернее: у него есть класс, дескриптор `HWND`, оконная процедура и сообщения. Системные классы виджетов уже зарегистрированы самой Windows (C11, справочник CreateWindowEx); нам нужны три:

| Класс | Что это | Ключевые стили |
|---|---|---|
| `"BUTTON"` | кнопка | — |
| `"EDIT"` | поле ввода текста | `ES_MULTILINE` (многострочное), `ES_AUTOVSCROLL`, `WS_VSCROLL`, `WS_BORDER` |
| `"STATIC"` | надпись (без ввода) | — |

- Дочернее окно создаётся тем же `CreateWindowEx`, но: стиль содержит **`WS_CHILD | WS_VISIBLE`** (дочернее и сразу видимое — `ShowWindow` для детей не нужен), координаты отсчитываются **от клиентской области родителя**, в параметре родителя — его `HWND`, а в параметре меню — хитрость API: для дочернего окна этот параметр означает не меню, а **идентификатор** (control ID), «произвольное целое, уникальное среди детей одного родителя» (C11, справочник CreateWindowEx). Идентификаторы заводим `#define`-константами (§11.1.2). Записывается ID двойным преобразованием `(HMENU)(INT_PTR)ID_...` — число расширяется до целого размером с указатель и лишь затем становится «дескриптором».
- **`WM_COMMAND`** — сообщение, которым виджет уведомляет родителя о событии. Упаковка (C11, справочник WM_COMMAND):

| Источник | `HIWORD(wParam)` | `LOWORD(wParam)` | `lParam` |
|---|---|---|---|
| виджет | код уведомления | ID виджета | HWND виджета |
| меню | 0 | ID пункта | 0 |

  Для кнопки код уведомления о щелчке — **`BN_CLICKED`** (C11, справочник BN_CLICKED). Макросы `LOWORD`/`HIWORD` достают младшие/старшие 16 бит (мы уже видели их для `WM_SIZE` в логе §12.1.3; C11, Writing the Window Procedure).
- **Текст виджета.** У всякого окна есть «текст»: у главного — заголовок, у кнопки — надпись, у EDIT — **всё содержимое**. Функции (C11, справочники): `SetWindowText(hwnd, str)` — установить; `GetWindowTextLength(hwnd)` — длина текста **без** терминатора; `GetWindowText(hwnd, buf, n)` — скопировать в буфер размера n (с терминатором, при нехватке — усечение).
- Новая функция стандартной библиотеки — **`snprintf`** (`<stdio.h>`, C5 §7.23; C6): как `printf`, но печатает в **строку-буфер** с ограничением размера: `snprintf(buf, sizeof buf, "%d", x)` — безопасно собрать текст для `SetWindowText` (не переполнит буфер — та защита, которой не хватало функциям из §5.2.3).

**Интуиция.** Окно с виджетами — матрёшка: внутри родителя живут окна поменьше, у каждого свой номерок (ID). Родитель — начальник отдела: сам щелчки по кнопкам не ловит — подчинённые (виджеты, их процедуры реализует система) обрабатывают клики сами и шлют начальнику служебки `WM_COMMAND`: «я, сотрудник №2, был нажат». Начальнику остаётся `switch` по номеру сотрудника — вся логика приложения собирается в одном месте, оконной процедуре родителя.

**Разобранные примеры.**

*Пример 1. Счётчик: две кнопки и надпись.*

Полный листинг (`counter1.c`); каркас WinMain — как в §12.1.2 (класс `"CounterClass"`, окно 320×200, фон `COLOR_BTNFACE` — стандартный серый диалогов):

```c
#define ID_BTN_PLUS  1 /* идентификаторы дочерних окон — обычные числа */
#define ID_BTN_RESET 2
#define ID_LABEL     3

static int  g_count = 0; /* состояние приложения — пока в глобальных переменных */
static HWND g_label;     /* метка, в которую выводим число */

static void show_count(void)
{
    char buf[16];
    snprintf(buf, sizeof buf, "%d", g_count);
    SetWindowText(g_label, buf); /* для контрола «текст окна» — его содержимое */
}

LRESULT CALLBACK WindowProc(HWND hwnd, UINT uMsg, WPARAM wParam, LPARAM lParam)
{
    switch (uMsg) {
    case WM_CREATE: { /* окно создаётся — заселяем его виджетами */
        CREATESTRUCT *cs = (CREATESTRUCT *)lParam;
        CreateWindowEx(0, "BUTTON", "+1",
                       WS_CHILD | WS_VISIBLE,
                       20, 20, 80, 30,
                       hwnd, (HMENU)(INT_PTR)ID_BTN_PLUS, cs->hInstance, NULL);
        CreateWindowEx(0, "BUTTON", "Reset",
                       WS_CHILD | WS_VISIBLE,
                       120, 20, 80, 30,
                       hwnd, (HMENU)(INT_PTR)ID_BTN_RESET, cs->hInstance, NULL);
        g_label = CreateWindowEx(0, "STATIC", "0",
                                 WS_CHILD | WS_VISIBLE,
                                 20, 70, 180, 30,
                                 hwnd, (HMENU)(INT_PTR)ID_LABEL, cs->hInstance, NULL);
        return 0;
    }

    case WM_COMMAND: /* уведомление от дочернего окна */
        if (HIWORD(wParam) == BN_CLICKED) {
            switch (LOWORD(wParam)) {
            case ID_BTN_PLUS:
                g_count++;
                show_count();
                break;
            case ID_BTN_RESET:
                g_count = 0;
                show_count();
                break;
            }
        }
        return 0;

    case WM_DESTROY:
        PostQuitMessage(0);
        return 0;
    }
    return DefWindowProc(hwnd, uMsg, wParam, lParam);
}
```

```
$ gcc -std=c17 -Wall -Wextra counter1.c -o counter1.exe -luser32 -mwindows
$ ./counter1.exe
```

Прогон (проверено): окно с кнопками «+1», «Reset» и надписью «0»; три щелчка по «+1» — надпись «3»; «Reset» — снова «0». Разбор ключевых мест. Виджеты создаются в обработчике `WM_CREATE` — окно-родитель уже существует (`hwnd` валиден), но ещё не показано: всё построится до первого кадра. `lParam` этого сообщения — указатель на `CREATESTRUCT` с параметрами создания (C11, справочник WM_NCCREATE/WM_CREATE); оттуда берём `hInstance`, не заводя глобалей. Обработка `WM_COMMAND` — ровно та табличная упаковка: старшие 16 бит wParam — «что случилось» (`BN_CLICKED`), младшие — «у кого» (ID), дальше диспетчеризация `switch`-ем. И заметьте, чего в программе **нет**: ни отрисовки кнопок, ни обработки наведения/нажатия мыши на них — всю механику ведёт системный класс `"BUTTON"`.

*Пример 2. Разбор одного клика по косточкам.*

Проследим щелчок по «+1» через всю модель §12.1.3: (1) система определяет, что клик пришёлся на дочернее окно-кнопку; (2) оконная процедура класса `"BUTTON"` (реализована в user32) отрабатывает нажатие — перерисовка «вдавленной» кнопки и прочая косметика; (3) кнопка шлёт родителю `WM_COMMAND` с `HIWORD(wParam)=BN_CLICKED`, `LOWORD(wParam)=1`, `lParam=HWND кнопки`; (4) наш `WindowProc` попадает в `case ID_BTN_PLUS`, инкрементирует счётчик и вызывает `SetWindowText` метки; (5) метка сама перерисовывается. Пять шагов, из них наш — только четвёртый: остальное — разделение труда между системными виджетами и нашей логикой.

*Пример 3.* ⛔ *Три способа не увидеть уведомление.*

Проверено сборкой и запуском: (а) **забыт `WS_VISIBLE`** — кнопка создана, но невидима (дочерние окна не показываются автоматически); (б) **одинаковые ID** у двух кнопок — обе шлют `WM_COMMAND` с одним номером, и `switch` не различит, кто нажат (компилятор не предупредит: ID — обычные числа; уникальность — на совести программиста); (в) **перепутаны LOWORD/HIWORD** — сравнение ID с кодом уведомления: условие ложно всегда, кнопка «не работает» без единого предупреждения. Во всех трёх случаях программа собирается чисто — ошибки видны только поведением. GUI-отладка — это лог/диагностика, а не только компилятор.

**Связь с практикой.** Схема «родитель создаёт детей с ID и разбирает WM_COMMAND» — основа всего классического интерфейса Windows: диалоги настроек, формы, панели инструментов устроены так до сих пор (Petzold, C12, гл. 9 «Child Window Controls»). Идея «интерфейс = дерево вложенных элементов, события всплывают к родителю» пережила Win32 и живёт в HTML DOM и во всех современных UI-фреймворках. А `snprintf` из этого параграфа — рабочая лошадка любого C-кода: форматирование в буфер нужно повсюду, от логов до сетевых протоколов.

**Типичные ошибки и подводные камни.**

- **Забыть `WS_CHILD` или `WS_VISIBLE`** (⛔-пример). Без первого — «виджет» станет отдельным окном верхнего уровня; без второго — невидимкой.
- **Неуникальные ID** и **перепутанные LOWORD/HIWORD** (⛔-пример) — молчаливые логические ошибки.
- **Буфер «на глазок» + `sprintf`.** Для текста в виджеты — только `snprintf` с `sizeof buf`: длина числа или строки пользователя — не ваша константа (уроки §5.2.3).
- **Проверять только ID, игнорируя код уведомления.** У кнопок бывают и другие уведомления, у EDIT — тем более (§12.2.4, задание З8): без проверки `HIWORD` логика сработает на чужом событии.
- **Создавать виджеты до `WM_CREATE`** (например, до `CreateWindowEx` родителя) — детям нужен валидный `HWND` родителя.

**Источники:** C11 (справочники CreateWindowEx — системные классы и hMenu-как-ID, WM_COMMAND — упаковка, BN_CLICKED, SetWindowText/GetWindowText/GetWindowTextLength; Learn to Program for Windows — LOWORD/HIWORD); C12 (гл. 9 «Child Window Controls»); C5 (§7.23 — snprintf), C6 (справочник snprintf).

### §12.2.2. Состояние приложения — в структуре

**Определения.**

- Проблема: оконная процедура вызывается системой заново на каждое сообщение — между вызовами она «беспамятна» (stateless), как и всякая функция C без static (гл. 4). Счётчик примера 1 хранил состояние в глобальных переменных — «works well enough for small programs», но в большой программе глобали разрастаются, а при **нескольких окнах одного класса** наступает катастрофа: окна делят одну оконную процедуру (§12.1.2, «франшиза») — и одни глобали на всех (C11, Managing Application State).
- Правильная схема Win32 — «состояние в структуре, указатель — в окне» (C11, там же):
  1. Описываем структуру состояния (гл. 8) и создаём её до окна.
  2. Указатель на неё передаём **последним параметром** `CreateWindowEx` (тот самый `void *`-хвост «дополнительных данных»).
  3. Изнутри `CreateWindowEx` окну приходят `WM_NCCREATE`, затем `WM_CREATE` (именно в этом порядке — C11); `lParam` обоих — указатель на `CREATESTRUCT`, чьё поле **`lpCreateParams`** — наш указатель.
  4. В `WM_NCCREATE` привязываем указатель к окну: **`SetWindowLongPtr(hwnd, GWLP_USERDATA, (LONG_PTR)ptr)`** — у каждого окна есть карман для одного значения размером с указатель.
  5. В любом другом обработчике достаём: **`GetWindowLongPtr(hwnd, GWLP_USERDATA)`** и приводим обратно. У каждого окна — свой карман: сто окон — сто состояний при одной процедуре.

**Интуиция.** Продолжая «франшизу»: правила обслуживания у всех кафе общие (одна оконная процедура), но у каждого кафе есть сейф (карман GWLP_USERDATA), где лежит его собственная бухгалтерия (структура состояния). Инструкция одна на сеть: «возьми бухгалтерию из своего сейфа и действуй». Приём «положи указатель туда, откуда его достанет обратный вызов» — общесистемный: так же будет устроено взаимодействие с потоками (том II) и почти любым C-API с колбэками.

**Разобранные примеры.**

*Пример 1. Счётчик v2: та же программа без глобалей.*

Отличия от counter1 (полный файл — `counter2.c`; прогон идентичен: клики → «3», Reset → «0», проверено):

```c
/* Всё состояние приложения — в одной структуре (гл. 8) */
typedef struct {
    int  count;
    HWND label;
} AppState;

/* Достаём указатель на состояние, привязанный к окну */
static AppState *get_state(HWND hwnd)
{
    return (AppState *)GetWindowLongPtr(hwnd, GWLP_USERDATA);
}

static void show_count(const AppState *st)
{
    char buf[16];
    snprintf(buf, sizeof buf, "%d", st->count);
    SetWindowText(st->label, buf);
}
```

В оконной процедуре — новый первый случай и работа через `st`:

```c
    case WM_NCCREATE: { /* самое первое сообщение — привязываем состояние */
        CREATESTRUCT *cs = (CREATESTRUCT *)lParam;
        SetWindowLongPtr(hwnd, GWLP_USERDATA, (LONG_PTR)cs->lpCreateParams);
        return DefWindowProc(hwnd, uMsg, wParam, lParam);
    }
    /* ... в WM_CREATE: st->label = CreateWindowEx(..., "STATIC", ...);
       в WM_COMMAND:  st->count++; show_count(st); — как в counter1 */
```

В `WinMain` состояние живёт на стеке — его время жизни (гл. 4, §7.1.1) покрывает всю жизнь окна, ведь `WinMain` не вернётся, пока крутится цикл сообщений:

```c
    AppState state = {0}; /* живёт весь срок работы окна: до конца WinMain */
    ...
    HWND hwnd = CreateWindowEx(0, "Counter2Class", "Counter v2",
                               WS_OVERLAPPEDWINDOW,
                               CW_USEDEFAULT, CW_USEDEFAULT, 320, 200,
                               NULL, NULL, hInstance,
                               &state); /* наш указатель поедет в WM_NCCREATE */
```

Обратите внимание на `WM_NCCREATE`: привязав указатель, мы **всё равно зовём `DefWindowProc`** — помним контракт этого сообщения (⛔ §12.1.2): кто-то должен вернуть TRUE, и пусть это будет система.

*Пример 2. Что было бы с двумя окнами.*

Мысленный эксперимент, ключевой для понимания (реальный код — задание З5). Создадим в `WinMain` **два** окна класса counter: у counter1 оба окна делят `g_count` и — хуже — одну `g_label`: клики в любом окне меняют общую цифру, а показывается она только в метке, чей дескриптор записан последним. У counter2 каждый `CreateWindowEx` получает **свой** `&state` — два окна, два кармана, два независимых счётчика при одной и той же оконной процедуре. Ровно это имеет в виду C11: «Each window has its own instance data».

*Пример 3. Врезка: сообщения-ранние пташки.*

Вернитесь к логу §12.1.3: сообщение №1 (0x0024, запрос размеров) пришло **до** `WM_NCCREATE`. В counter2 такое сообщение попадёт в `DefWindowProc` до привязки состояния — а если бы мы читали состояние в его обработчике, `get_state` вернул бы NULL (карман пуст, `GetWindowLongPtr` даёт 0). Наш код в безопасности — состояние трогают только `WM_CREATE` и `WM_COMMAND`, оба заведомо позже `WM_NCCREATE`, — но общее правило сформулируем: **обработчик, читающий состояние, должен быть готов к NULL** (образец из C11 так и устроен: `if (pState) {...} else DefWindowProc(...)`). Схема памяти, «кто на кого указывает»:

```
WinMain (стек)                    системные данные окна
┌───────────────┐                ┌──────────────────────┐
│ state:        │◄───────────────│ GWLP_USERDATA (карман)│
│  count = 3    │   привязано в  ├──────────────────────┤
│  label ───────┼──► HWND метки  │ lpfnWndProc ──► WindowProc
└───────────────┘   WM_NCCREATE  └──────────────────────┘
```

**Связь с практикой.** «Структура состояния + указатель в кармане окна» — это ручная сборка того, что объектные языки называют «объект»: данные (структура) плюс поведение (оконная процедура), связанные вместе. C11 в этом месте прямо переходит к ООП-обёрткам (класс BaseWindow), MFC и прочие фреймворки Windows выросли из этого приёма. В томе II идея «данные и методы вместе» станет языковой конструкцией (`impl` у структур Rust, гл. 16) — вы уже будете знать, какую ручную работу она автоматизирует. Тот же паттерн «void*-карман для пользователя» — в `qsort_s`, в потоках (`CreateThread`, `pthread_create`), в таймерах и колбэках любых C-библиотек.

**Типичные ошибки и подводные камни.**

- **Читать состояние до привязки** (пример 3) — NULL-разыменование (гл. 6, §13). Привязка — в `WM_NCCREATE`, самом первом сообщении с `CREATESTRUCT`; обработчики ранних сообщений к состоянию не допускаются или проверяют NULL.
- **Отдать окну указатель на умершую переменную.** Если бы структуру создала и вернула вспомогательная функция как локальную — окно хранило бы висячий указатель (§7.1.1, возврат адреса локальной). Наше состояние живёт в `WinMain` — покрывает жизнь окна; альтернатива — куча + `free` после цикла (владелец — `WinMain`, §7.1.2).
- **Забыть `DefWindowProc` в WM_NCCREATE** — снова несозданное окно (контракт из ⛔ §12.1.2).
- **Вернуться к глобалям «потому что работает».** Работает — до второго окна и до первой попытки тестировать логику отдельно от GUI. Дисциплина состояния — бесплатная сейчас и дорогая при переделке потом.

**Источники:** C11 (Learn to Program for Windows: «Managing Application State» — вся схема, порядок WM_NCCREATE→WM_CREATE, инстанс-данные; справочники SetWindowLongPtr/GetWindowLongPtr, WM_NCCREATE); C12 (гл. 3 — окна и данные окна); C5/C6 — языковая часть (структуры, преобразования указателей, время жизни).

### §12.2.3. WM_PAINT: рисуем сами (обзорно)

**Определения.**

- **`WM_PAINT`** — сообщение «пора рисовать»: система шлёт его, когда часть клиентской области нуждается в перерисовке. «Грязная» часть называется **областью обновления** (update region): в неё попадает то, что открылось из-под другого окна, появилось при растяжении — и вся клиентская область при первом показе, поэтому хотя бы один `WM_PAINT` приходит всегда (C11, Painting the Window; наш лог §12.1.3 — сообщение №28).
- Каркас обработчика (C11, там же):

  ```c
  PAINTSTRUCT ps;
  HDC hdc = BeginPaint(hwnd, &ps);
  /* всё рисование — здесь */
  EndPaint(hwnd, &ps);
  ```

  **`BeginPaint`** заполняет `PAINTSTRUCT` (в поле `rcPaint` — прямоугольник области обновления) и возвращает **контекст устройства** (device context, `HDC`) — дескриптор «холста», в который умеют рисовать функции **GDI** (Graphics Device Interface — графическая подсистема Windows, библиотека `gdi32`). **`EndPaint`** объявляет область обновления очищенной — иначе система будет слать WM_PAINT снова и снова.
- Минимальный набор рисования: `FillRect(hdc, &rect, кисть)` — залить прямоугольник (кисть из системного цвета — знакомая конвенция `(HBRUSH)(COLOR_WINDOW + 1)`); `TextOut(hdc, x, y, str, len)` — вывести `len` символов строки с позиции (x, y) (C11, справочники FillRect/TextOut; Petzold, C12, гл. 4–5).
- **`InvalidateRect(hwnd, NULL, TRUE)`** — обратная связь: «данные изменились — перерисуй». Сами напрямую не рисуем: добавляем клиентскую область (NULL — всю) в область обновления, и система пришлёт WM_PAINT (C11).

**Интуиция.** Окно — не картина, а доска объявлений на ветру: система не хранит нарисованное навсегда и в любой момент вправе спросить «а что у тебя тут было?» (открыли из-под другого окна — изволь восстановить). Поэтому канонический стиль: состояние (что рисовать) живёт в данных, `WM_PAINT` — чистая функция «данные → пиксели», а всякое изменение данных дёргает `InvalidateRect`. Знакомая песня? Это тот же принцип «модель отдельно, отображение отдельно», что и в counter (данные `count` → текст метки), только отображение теперь рисуем мы.

**Разобранные примеры.**

*Пример 1. Кликер с рисованием.*

Оконная процедура (каркас программы обычный; `wc.hbrBackground` оставлен NULL — фон теперь наша забота, C11, справочник WNDCLASS):

```c
static int g_clicks = 0;

LRESULT CALLBACK WindowProc(HWND hwnd, UINT uMsg, WPARAM wParam, LPARAM lParam)
{
    switch (uMsg) {
    case WM_LBUTTONDOWN: /* клик мышью по клиентской области */
        g_clicks++;
        InvalidateRect(hwnd, NULL, TRUE); /* весь клиент — в область обновления */
        return 0;

    case WM_PAINT: {
        PAINTSTRUCT ps;
        HDC hdc = BeginPaint(hwnd, &ps); /* всё рисование — между Begin и End */

        FillRect(hdc, &ps.rcPaint, (HBRUSH)(COLOR_WINDOW + 1)); /* фон */

        char buf[32];
        int  len = snprintf(buf, sizeof buf, "Clicks: %d", g_clicks);
        TextOut(hdc, 20, 20, buf, len); /* текст в точке (20, 20) — это GDI */

        EndPaint(hwnd, &ps);
        return 0;
    }

    case WM_DESTROY:
        PostQuitMessage(0);
        return 0;
    }
    return DefWindowProc(hwnd, uMsg, wParam, lParam);
}
```

```
$ gcc -std=c17 -Wall -Wextra clicks_paint.c -o clicks_paint.exe -luser32 -lgdi32 -mwindows
$ ./clicks_paint.exe   # в окне «Clicks: 0»; каждый клик увеличивает число
```

Проследите цикл данных: клик → `g_clicks++` → `InvalidateRect` → (система) → `WM_PAINT` → `TextOut` нового значения. Мы не рисуем «по клику» — мы меняем данные и просим перерисовку; рисует всегда один и тот же код.

*Пример 2.* ⛔ *Undefined reference возвращается (§11.2.1 в бою).*

Соберём то же самое без `-lgdi32` и без `-mwindows` (который, как мы выяснили в §12.1.2, тайком добавляет gdi32):

```

collect2.exe: error: ld returned 1 exit status
```

Диагноз по почерку — навык §11.2.1: слово `reference`, имя `ld` — недостаёт **определения**, а не объявления; `#include <windows.h>` не поможет (объявление TextOut у нас уже есть — потому компиляция прошла). Имя `__imp_TextOutA` читается: импорт (`__imp_`) A-варианта (§12.1.1) функции TextOut — лечится добавлением библиотеки-поставщика: `-lgdi32`. Всё как с `-lvec` из §11.2.3, только библиотека системная.

*Пример 3. Почему нельзя «просто нарисовать и забыть».*

Мысленный эксперимент по документации (C11, Painting the Window): пусть бы мы рисовали прямо в обработчике клика (взяли DC, вывели текст) и не трогали WM_PAINT. До первого перекрытия всё выглядит работающим. Затем пользователь провёл поверх нашего окна другим окном — система просит перерисовать открывшуюся область (WM_PAINT), а наш обработчик пуст — на месте текста остаётся затёртый фон. Вывод тот же, что в «Интуиции»: единственный надёжный источник пикселей — обработчик WM_PAINT, рисующий из данных.

**Связь с практикой.** Так — «данные + перерисовка по запросу» — работает вся растровая графика: игровые движки перерисовывают кадр целиком из состояния мира; браузер перерисовывает страницу из DOM; графические редакторы держат модель документа и рендерят видимую часть. GDI — ветеран (Petzold посвящает ему треть книги, C12, гл. 4–5, 13–18), в современных приложениях его сменили Direct2D и аппаратный рендеринг, но модель «invalidate → paint» неизменна. Для наших задач GDI даёт главное: понять её на минимальном API. Виджеты §12.2.1, кстати, внутри устроены ровно так — их системные процедуры обрабатывают свой WM_PAINT.

**Типичные ошибки и подводные камни.**

- **Рисовать вне пары BeginPaint/EndPaint** в обработчике WM_PAINT — контракт нарушен: область обновления не очищена, система шлёт WM_PAINT бесконечно, процесс ест ядро процессора.
- **Рисовать по событию, а не по WM_PAINT** (пример 3) — картинка стирается при первом перекрытии.
- **Забыть `-lgdi32`** (пример 2) — undefined reference; вы теперь знаете и почерк ошибки, и лекарство.
- **Забыть InvalidateRect после изменения данных** — данные новые, а на экране старые: система не телепат, о «грязноте» ей сообщают.
- **Тяжёлое рисование каждый кадр.** Для наших программ неактуально, но правило то же, что для оконной процедуры вообще: WM_PAINT должен быть быстрым.

**Источники:** C11 (Learn to Program for Windows: «Painting the Window» — update region, BeginPaint/EndPaint, FillRect, первый WM_PAINT; справочники TextOut, InvalidateRect, WNDCLASS — NULL-фон); C12 (гл. 4 «An Exercise in Text Output», гл. 5 «Basic Drawing» — GDI); C5/C6 — snprintf.

**Чекпоинт 1. Модуль хранения** — чистый C без единого упоминания окон: его можно написать и проверить консольно (дисциплина §11.2.2: интерфейс — минимум, нужный клиенту).

`storage.h`:

```c
#ifndef STORAGE_H
#define STORAGE_H

/* Читает файл целиком в строку из кучи; NULL — файла нет или ошибка.
   Владение: буфер освобождает вызывающий (free). */
char *storage_load(const char *path);

/* Записывает строку в файл целиком; 0 — успех, -1 — ошибка. */
int storage_save(const char *path, const char *text);

#endif /* STORAGE_H */
```

`storage.c`:

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

#include "storage.h" /* свой интерфейс — сверка сигнатур (§11.2.2) */

char *storage_load(const char *path)
{
    FILE *f = fopen(path, "rb"); /* бинарный режим: байты \r\n не трогаем */
    if (f == NULL)
        return NULL;

    /* размер файла приёмом §10.2.2 (с той же оговоркой о переносимости) */
    if (fseek(f, 0, SEEK_END) != 0) {
        fclose(f);
        return NULL;
    }
    long size = ftell(f);
    if (size < 0 || fseek(f, 0, SEEK_SET) != 0) {
        fclose(f);
        return NULL;
    }

    char *buf = malloc((size_t)size + 1); /* +1 под терминатор */
    if (buf == NULL) {
        fclose(f);
        return NULL;
    }

    size_t got = fread(buf, 1, (size_t)size, f);
    fclose(f);
    buf[got] = '\0'; /* контролам Windows нужна C-строка */
    return buf;
}

int storage_save(const char *path, const char *text)
{
    FILE *f = fopen(path, "wb");
    if (f == NULL)
        return -1;

    size_t len = strlen(text);
    size_t put = fwrite(text, 1, len, f);
    if (fclose(f) != 0 || put != len)
        return -1;
    return 0;
}
```

Почему **бинарный** режим (`"rb"`/`"wb"`): текст из EDIT-контрола приходит с разделителями строк `\r\n`; текстовый режим UCRT при записи превратил бы каждый `\n` в `\r\n` — получили бы `\r\r\n` (§10.2.1, C15). Храним и читаем байты как есть — EDIT получает обратно ровно то, что отдал. Модуль прогнан консольным тестом под AddressSanitizer в WSL (методика §7.2.2): запись → чтение → сравнение — совпадает, отчёт чист, утечек нет (проверено).

**Чекпоинт 4. Логика Save/Load** — мост между виджетами и модулем хранения; здесь встречаются GetWindowText и malloc:

```c
static void set_status(const AppState *st, const char *msg)
{
    SetWindowText(st->status, msg);
}

static void do_load(const AppState *st)
{
    char *text = storage_load(FILE_NAME);
    if (text == NULL) {
        set_status(st, "load failed: no " FILE_NAME);
        return;
    }
    SetWindowText(st->edit, text); /* контрол делает СВОЮ копию текста */
    free(text);                    /* наша копия больше не нужна */
    set_status(st, "loaded from " FILE_NAME);
}

static void do_save(const AppState *st)
{
    int len = GetWindowTextLength(st->edit); /* длина без терминатора */
    char *buf = malloc((size_t)len + 1);
    if (buf == NULL) {
        set_status(st, "out of memory");
        return;
    }
    GetWindowText(st->edit, buf, len + 1); /* копирует текст и ставит '\0' */

    int rc = storage_save(FILE_NAME, buf);
    free(buf);
    set_status(st, rc == 0 ? "saved to " FILE_NAME : "save failed");
}
```

## Резюме главы

- **Событийная модель**: GUI-программа не исполняет сценарий, а отвечает на **сообщения** — числовые коды событий с двумя параметрами. Порядок задаёт пользователь и система; наша программа — набор обработчиков плюс цикл, качающий очередь (`GetMessage` → `TranslateMessage` → `DispatchMessage`). `GetMessage` блокируется на пустой очереди, возвращает 0 на `WM_QUIT` и −1 при ошибке — отсюда форма цикла `> 0`.
- **Окно строится в три шага**: `RegisterClass` (класс = общие свойства, главное — указатель на оконную процедуру) → `CreateWindowEx` (экземпляр; NULL — проверять!) → `ShowWindow`. GUI-сборка: `-luser32` (+ `-lgdi32` для рисования) и `-mwindows` — подсистема GUI в заголовке исполняемого файла (иначе — паразитная консоль).
- **Оконная процедура** — callback с фиксированной сигнатурой, тот же принцип, что у компаратора `qsort` (§6.3.3), но цикл живёт всю программу. Правило номер один: всё необработанное — в **`DefWindowProc`** (у сообщений есть контракты: вернуть 0 на `WM_NCCREATE` — окно не создастся). Штатное завершение — цепочка `WM_CLOSE` → `DestroyWindow` → `WM_DESTROY` → `PostQuitMessage` → `WM_QUIT`; забытый `PostQuitMessage` — процесс-зомби без окон.
- **Виджеты — это дочерние окна** системных классов (`"BUTTON"`, `"EDIT"`, `"STATIC"`): стиль `WS_CHILD | WS_VISIBLE`, целочисленный ID на месте параметра меню. О событиях они докладывают родителю сообщением **`WM_COMMAND`**: `HIWORD(wParam)` — что случилось (`BN_CLICKED`, `EN_CHANGE`), `LOWORD(wParam)` — у кого (ID), `lParam` — чей дескриптор.
- **Состояние приложения** — в структуре, а не в глобалях: указатель едет последним параметром `CreateWindowEx`, ловится в `WM_NCCREATE` из `CREATESTRUCT.lpCreateParams`, кладётся в карман окна `SetWindowLongPtr(GWLP_USERDATA)` и достаётся `GetWindowLongPtr` в любом обработчике. У каждого окна карман свой — одна процедура обслуживает много окон с независимыми данными.
- **Рисование**: система просит перерисовку сообщением `WM_PAINT` (область обновления), ответ — пара `BeginPaint`/`EndPaint` и GDI-вызовы между ними (`FillRect`, `TextOut`). Изменились данные — не рисуем напрямую, а зовём `InvalidateRect`: единственный источник пикселей — обработчик `WM_PAINT`.
- **Контрольный проект** собрал том: мини-блокнот = событийный каркас + виджеты + структура состояния + модуль `storage.h/.c` на `fopen`/`malloc` (бинарный режим — байты `\r\n` из EDIT не трогаем) + Makefile с `-luser32 -mwindows`. Логика отделена от интерфейса — потому проверяется консольно и под ASan, без единого окна.

**Дальше.** Том I почти пройден: вы умеете всё, что обещали главы 1–12, — от битов до оконного приложения. Осталась последняя и особая глава — **глава 13, мост к Rust**: мы систематизируем «тёмную сторону» C, с которой сражались всю дорогу, — неопределённое поведение, выходы за границы, use-after-free, забытые `free` (в этой главе они тоже прятались: `malloc(len)` вместо `len + 1` — и ASan вам был нужен снова), — и посмотрим, **почему** язык позволяет такие ошибки и **что** должен уметь язык, который ловил бы их до запуска. Это и есть мотивация Rust — тома II.
