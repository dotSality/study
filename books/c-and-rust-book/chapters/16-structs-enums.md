# Глава 16. Структуры, перечисления и сопоставление с образцом

## Введение

Глава 15 дала вам грамматику Rust — владение, заимствование, срезы. Но до сих пор мы жили на готовых типах: числа, кортежи, массивы, `String`. Пора вернуть себе право строить собственные — то, чем в томе I занималась глава 8: `struct`, `enum`, `union`. Эта глава — её зеркало, и по ходу дела вы увидите, как каждая конструкция главы 8 выглядит, когда за ней присматривает компилятор.

Три сюжета:

- **Структуры и методы.** `struct` в Rust — та же идея «связанные данные под одной крышей», но поля теперь *принадлежат* структуре по правилам главы 15, а функции для работы с типом живут не «рядом, с префиксом в имени» (как наши `vec_push` и `stack_pop` из глав 9 и 11), а *при типе* — в блоке `impl`.
- **Перечисления с данными.** `enum` Rust не похож на `enum` C — набор целых констант. Это **тип-сумма**: «значение — ровно одно из перечисленного, и у каждого варианта свои данные». В §8.2.2 мы собирали такое вручную — тег-перечисление плюс `union` — и держалось всё на дисциплине. Теперь тег и данные неразлучны на уровне типа.
- **`Option<T>` и `match`.** Строка 2 нашего каталога UB — разыменование NULL (§13.1.2) — закрывается в этой главе окончательно: «значения может не быть» становится *типом* `Option<T>`, а `match` — разбором, который не даст забыть ни ветку `None`, ни новый вариант перечисления. Витрину этого вы видели в §13.2.2 — теперь разберём механизм.

После изучения главы вы будете уметь:

- объявлять структуры (обычные, кортежные, единичные), создавать экземпляры и понимать, что происходит с владением полей при присваивании и `..`-обновлении;
- писать методы и ассоциированные функции в `impl` и выбирать приёмник — `&self`, `&mut self` или `self` — по правилу сигнатур из §15.2;
- осознанно вешать `#[derive(Debug)]`, `#[derive(Clone)]`, `#[derive(Copy)]` на свои типы — и объяснять, почему `Copy` на структуре со `String` не компилируется;
- моделировать «одно из» перечислениями с данными и объяснять, чем это безопаснее пары enum+union из C;
- разбирать значения `match`-ом, читать E0004 как страховку от забытых вариантов, пользоваться `Option<T>` вместо NULL и выбирать между `match`, `if let` и `let else`.

Весь код главы прогнан на эталонном окружении курса: rustc/cargo **1.97.0 stable** (Windows, `x86_64-pc-windows-gnu`, издание 2024); выводы программ и тексты ошибок приведены дословно (служебные пути стандартной библиотеки в отчётах компилятора сокращены — так же, как мы сокращали пути в отчётах ASan, §7.2.2). Примеры — одиночные файлы `rustc --edition 2024 файл.rs`, как в §14.1.1.

## Часть 16.1. Структуры и методы

### §16.1.1. struct: данные вместе — и владение вместе

**Определения.** **Структура** (struct) объявляется ключевым словом `struct`, именем и списком полей `имя: тип` (R1, гл. 5):

```rust
struct Student {
    name: String,
    year: u32,
    active: bool,
}
```

Сравните с C (§8.1.1): `struct student { char name[64]; int year; bool active; };` — идея та же, различия косметические и одно принципиальное. Косметика: поля разделяются запятыми, тип пишется после имени (как в `let`), и имя `Student` — сразу тип, без `typedef`-церемонии из §8.2.3 (R6). Принципиальное — ниже, про владение.

**Экземпляр** создают, указывая значения всех полей; порядок — любой, потому что поля именованы (R1):

```rust
let s = Student {
    year: 1,
    name: String::from("Alice"),
    active: true,
};
```

Это родственник назначенных инициализаторов C (`{ .year = 1, ... }`, §8.1.1) — только здесь форма «по именам» не альтернатива, а единственная: инициализации «по порядку» у обычных структур нет, забыть поле нельзя.

Доступ к полям — точкой, как в C. Изменение — только если **весь экземпляр** объявлен `mut`: пометить изменяемым отдельное поле нельзя (R1). Это глава 14 в новом масштабе: неизменяемость по умолчанию — теперь для составных данных.

Ещё две формы записи (R1):

- **Сокращение полей** (field init shorthand): если переменная называется как поле, `Student { name: name, ... }` пишут короче — `Student { name, ... }`.
- **Обновляющий синтаксис** (struct update): `Student { year: 4, ..s1 }` — «как `s1`, но с другим `year`»; `..s1` подставляет все не названные поля из `s1` и обязан стоять последним (R6). Важно: это работает **как присваивание** — то есть по правилам §15.1.2: Copy-поля копируются, не-Copy-поля (наша `String`) **перемещаются** из `s1` (R1).

Наконец, **владение полей**. `Student` хранит `name: String` — владеющую строку: структура *владеет всеми своими данными*, и они живы, пока жива она (R1, гл. 5). Можно ли положить в поле ссылку `&str`? Можно, но это потребует аннотаций времён жизни — темы главы 19; без них компилятор откажет (пример 3). Правило на ближайшие главы: **в полях — владеющие типы** (`String`, а не `&str`).

**Интуиция.** Экземпляр структуры лежит в памяти одним куском, а владеющие поля тянут за собой свои буферы:

```
        стек: Student (один кусок)                куча
  s ┌──────────────────┬──────┬────────┐
    │ name: ptr len cap│ year │ active │
    │        ●    5   5│  1   │  true  │
    └────────┼─────────┴──────┴────────┘
             └──────────────▶ ┌───┬───┬───┬───┬───┐
                              │ A │ l │ i │ c │ e │
                              └───┴───┴───┴───┴───┘
```

Владелец буфера `"Alice"` — поле `name`; владелец поля — структура `s`. Уйдёт `s` за скобку — освободится всё, чем она владела, включая буфер: правила главы 15 работают на составных типах без оговорок. В C та же картинка (§8.1) требовала комментария «не забудь освободить `name`» — здесь это свойство типа.

**Разобранные примеры.**

*Пример 1 — объявить, создать, изменить.* Файл `student1.rs`:

```rust
struct Student {
    name: String,
    year: u32,
    active: bool,
}

fn main() {
    let mut s = Student {
        year: 1,
        name: String::from("Alice"),
        active: true,
    };
    println!("{}, year {}, active: {}", s.name, s.year, s.active);
    s.year = 2;
    println!("now year {}", s.year);
}
```

```
Alice, year 1, active: true
now year 2
```

Поля при создании идут не в порядке объявления — законно. Уберите `mut` — и присваивание `s.year = 2` перестанет собираться (проверено):

```

  --> field_mut.rs:11:5
   |
11 |     s.year = 2;
   |     ^^^^^^^^^^ cannot assign
   |

   |

   |         +++
```

Функция-строитель с сокращением полей (файл `build.rs`, проверено — печатает `Bob, year 3, active: true`):

```rust
fn build_student(name: String, year: u32) -> Student {
    Student {
        name,
        year,
        active: true,
    }
}
```

Параметры называются как поля — и `name: name` сжалось до `name`. Заметьте: параметр `name: String` берёт строку **по владению** — строитель кладёт её в структуру насовсем; это тот самый случай «функция оставляет значение себе» из правила сигнатур §15.2.1.

*Пример 2 — обновляющий синтаксис перемещает.* Файл `update1.rs` — новый студент с теми же `year`/`active`, но другим именем:

```rust
fn main() {
    let s1 = Student {
        name: String::from("Alice"),
        year: 3,
        active: true,
    };
    let s2 = Student {
        name: String::from("Bob"),
        ..s1
    };
    println!("s2: {}, year {}, active {}", s2.name, s2.year, s2.active);
    println!("s1: {}, year {}, active {}", s1.name, s1.year, s1.active);
}
```

```
s2: Bob, year 3, active true
s1: Alice, year 3, active true
```

Обе структуры живы: из `s1` через `..` уехали только `year` и `active` — Copy-типы (§15.1.3), они скопировались. А теперь поменяем условие: новое — `year`, а **имя** пусть приедет по `..s1`. Файл `update_move.rs`:

```rust
    let s2 = Student {
        year: 4,
        ..s1
    };
    println!("s2: {}, year {}", s2.name, s2.year);
    println!("s1: {}", s1.name);
```

⛔ Не компилируется (проверено):

```
error[E0382]: borrow of moved value: `s1.name`
  --> update_move.rs:18:24
   |
13 |       let s2 = Student {
   |  ______________-
14 | |         year: 4,
15 | |         ..s1
16 | |     };
   | |_____- value moved here
17 |       println!("s2: {}, year {}", s2.name, s2.year);
18 |       println!("s1: {}", s1.name);
   |                          ^^^^^^^ value borrowed here after move
   |
   = note: move occurs because `s1.name` has type `String`, which does not implement the `Copy` trait
```

Старый знакомый E0382 — но взгляните на «виновника»: не вся `s1`, а именно `s1.name`. Поле `name` — `String`, оно не копируется, а переезжает в `s2`; компилятор отслеживает владение **пополевно**. Copy-поля источника при этом остаются доступны: вариант, где после того же `..s1` печатаются только `s1.year` и `s1.active`, собирается и работает (файл `update_move_ok.rs`, проверено: `s2: Alice, year 4, active true`, затем `s1: year 3, active true`).

*Пример 3 — поле-ссылка: не сейчас.* Попытка «сэкономить» и положить в поле срез вместо владеющей строки — файл `strfield.rs`:

```rust
struct Label {
    text: &str,
}

fn main() {
    let l = Label { text: "hello" };
    println!("{}", l.text);
}
```

⛔ Не компилируется (проверено):

```
error[E0106]: missing lifetime specifier
 --> strfield.rs:2:11
  |
2 |     text: &str,
  |           ^ expected named lifetime parameter
  |
help: consider introducing a named lifetime parameter
  |
1 ~ struct Label<'a> {
2 ~     text: &'a str,

```

E0106 вы уже читали в §15.2.3 — «не хватает параметра времени жизни». Смысл тот же: ссылка в поле означает, что структура *заимствует* чужие данные, и компилятору нужно доказательство, что данные переживут структуру. Язык для таких доказательств — аннотации времён жизни, глава 19 (компилятор их даже подсказывает: `<'a>`). До неё — владеющее поле `String`: и программа собирается, и структура самодостаточна (R1 даёт ровно этот совет).

*Пример 4 — кортежные и единичные структуры.* Иногда имена полей — лишние слова. **Кортежная структура** (tuple struct) — «именованный кортеж» (R2): поля без имён, доступ по индексу, как у кортежей §14.2.2. Файл `tuple_structs.rs`:

```rust
struct Color(i32, i32, i32);
struct Point3(i32, i32, i32);
struct Marker;

fn sum_color(c: &Color) -> i32 {
    c.0 + c.1 + c.2
}

fn main() {
    let white = Color(255, 255, 255);
    let origin = Point3(0, 0, 0);
    println!("white.0 = {}", white.0);
    println!("sum(white) = {}", sum_color(&white));
    println!("origin: ({}, {}, {})", origin.0, origin.1, origin.2);

    let Color(r, g, b) = white;
    println!("r = {r}, g = {g}, b = {b}");

    let _m = Marker;
}
```

```
white.0 = 255
sum(white) = 765
origin: (0, 0, 0)
r = 255, g = 255, b = 255
```

Разберите последние строки `main`: `let Color(r, g, b) = white;` — **деструктуризация**, как у кортежей в §14.2.2, только с именем типа (R2). А `struct Marker;` — **единичная структура** (unit-like struct): полей нет вовсе; пригодится, когда типу нужно только имя (систематическое применение — с трейтами, глава 19; R1).

Главное в кортежных структурах — не запись, а **номинальность**: `Color` и `Point3` состоят из одинаковых троек `i32`, но это *разные типы* (R1). Файл `color_point.rs` передаёт точку туда, где ждут цвет:

В C так же вели себя разные теги структур — §8.1.1 показывал `invalid initializer` при смешении одинаковых по составу `struct`. Rust сохраняет это свойство и превращает его в рабочий приём — «новый тип» (newtype): обернуть голое число в одноимённое поле смысла, чтобы метры не складывались с секундами. Практика 16.1 попробует его на температурах.

**Как это было в C.** Сводка по главе 8:

| | C (глава 8) | Rust |
|---|---|---|
| Объявление | `struct point { int x; int y; };` + обычно `typedef` | `struct Point { x: i32, y: i32 }` — имя сразу тип |
| Создание | `{1, 2}` по порядку или `{.x = 1, .y = 2}` | только по именам, порядок любой |
| Доступ | `p.x` | `p.x` — без изменений |
| Изменяемость | всё изменяемо, `const` — опционально | ничего не изменяемо, `mut` — на весь экземпляр |
| Присваивание структур | всегда побайтовая копия (§8.1.2) | перемещение, если есть не-Copy поле (§16.1.3) |
| Поле-строка | `char name[64]` (копия, лимит) или `char *` (владение на доверии, §8.1) | `String` — владение в типе; `&str`-поле — только с временами жизни (гл. 19) |
| Одинаковый состав, разные теги | разные типы | разные типы — и это приём (newtype) |

**Связь с практикой.** Структура — первый инструмент моделирования: «эти значения ходят вместе». Правила выбора, которые пригодятся в каждой программе тома II: поля — владеющие типы (пока без глав 19); экземпляры в функции передавайте по ссылке `&Student` (владение — только если функция забирает объект насовсем, как `build_student` забирал строку); `mut` — у переменной, а не «у поля». И присмотритесь к `..`-обновлению: в коде на Rust это обычный способ сказать «как тот, но с парой других полей» — помня, что не-Copy поля при этом переезжают.

**Типичные ошибки и подводные камни.**

- **Забыть `mut` и удивиться E0594.** Право менять поля — это право менять переменную; выдаётся целиком, при объявлении.
- **Искать «изменяемое поле».** Его нет (R1); если у типа есть «часть, которую меняют все подряд», это сигнал о дизайне, а не о синтаксисе.
- **Прочитать `..s1` как «скопируй остальное».** Скопируются только Copy-поля; `String` уедет — и `s1.name` больше не существует (E0382 с точным именем поля).
- **`&str`-поле «для экономии».** До главы 19 — E0106. Владеющее поле — не расточительность, а самодостаточность структуры.

**Источники:** R1 (гл. 5: объявление и экземпляры, изменяемость целиком, shorthand, update-синтаксис и его move-семантика, владение полей и совет про String, кортежные и единичные структуры); R2 (Structures: три вида структур, деструктуризация); R6 (Named structs: имя сразу тип, `..` — последним). Сторона C: §8.1.1–8.1.2, §8.2.3 курса.

### §16.1.2. impl: методы и ассоциированные функции

**Определения.** **Блок реализации** `impl Тип { ... }` привязывает функции к типу (R1, гл. 5). Всё, что объявлено внутри, называется **ассоциированными функциями** (associated functions) типа; те из них, чей первый параметр — `self`, называются **методами** (methods) и вызываются точкой: `rect.area()`.

Про `self` (R1):

- `&self` — сокращение для `self: &Self`, где `Self` — псевдоним типа, которому принадлежит `impl`. Метод *заимствует* экземпляр для чтения.
- `&mut self` — заимствует с правом изменения.
- `self` — забирает экземпляр **по владению**; после вызова объекта у вызывающего больше нет. Нужен редко: когда метод *потребляет* или *преобразует* объект (R1).

Это в точности правило сигнатур §15.2.1, применённое к «нулевому параметру»: читает — `&self`, меняет — `&mut self`, потребляет — `self`.

Функция в `impl` **без** `self` — ассоциированная, но не метод: вызывается через тип и `::` — `Rect::square(7)`. Типичное применение — конструкторы (R6). Кстати, загадка первых глав тома разгадана: `String::from("hello")` — это ассоциированная функция `from` типа `String`, никакой магии. Блоков `impl` у типа может быть несколько — это законно (R1); пока нам хватит одного.

И знаменитая деталь: **оператора `->` в Rust нет**. В C выбор `.` или `->` зависел от того, значение перед вами или указатель (§8.1.2). В Rust всегда точка: при вызове метода компилятор сам подставляет `&`, `&mut` или `*` к приёмнику, чтобы тот совпал с сигнатурой (R1 называет это automatic referencing and dereferencing). `r.area()` и `(&r).area()` — одно и то же (проверено ниже).

**Интуиция.** Метод — функция с домашним адресом. В томе I мы уже писали «методы» — дисциплиной имён:

```
C (главы 8–9): дисциплина префиксов        Rust: пространство имён типа
──────────────────────────────────         ─────────────────────────────
int rect_area(const Rect *r);              impl Rect { fn area(&self) -> u32 }
rect_area(&r);                             r.area();
                                           
Rect rect_square(int size);                impl Rect { fn square(n: u32) -> Self }
rect_square(7);                            Rect::square(7);

префикс rect_ — договорённость;            impl — конструкция языка: компилятор
компилятору всё равно                      знает, что́ принадлежит типу
```

**Разобранные примеры.**

*Пример 1 — от функции к методам.* Отправная точка — «C-стиль» из §8.1.2, перенесённый в Rust дословно (файл `rect1.rs`, проверено — `area = 1500`):

```rust
struct Rect {
    width: u32,
    height: u32,
}

fn area(r: &Rect) -> u32 {
    r.width * r.height
}
```

Работает, но `area` — функция «где-то рядом». Переносим её к типу и достраиваем семейство — файл `rect_methods.rs`:

```rust
struct Rect {
    width: u32,
    height: u32,
}

impl Rect {
    fn area(&self) -> u32 {
        self.width * self.height
    }

    fn can_hold(&self, other: &Rect) -> bool {
        self.width > other.width && self.height > other.height
    }

    fn grow(&mut self, dw: u32, dh: u32) {
        self.width += dw;
        self.height += dh;
    }

    fn square(size: u32) -> Self {
        Self {
            width: size,
            height: size,
        }
    }
}

fn main() {
    let mut r = Rect {
        width: 30,
        height: 50,
    };
    println!("area = {}", r.area());
    println!("area explicit = {}", (&r).area());

    let small = Rect {
        width: 10,
        height: 20,
    };
    println!("r can hold small: {}", r.can_hold(&small));
    println!("small can hold r: {}", small.can_hold(&r));

    r.grow(5, 0);
    println!("after grow: {} x {}, area = {}", r.width, r.height, r.area());

    let sq = Rect::square(7);
    println!("square area = {}", sq.area());
}
```

```
area = 1500
area explicit = 1500
r can hold small: true
small can hold r: false
after grow: 35 x 50, area = 1750
square area = 49
```

Здесь всё меню сразу. `area` — читатель (`&self`). `can_hold` — метод с параметром: `self` — «этот» прямоугольник, `other` — второй, обычным параметром-ссылкой (R1). `grow` — писатель (`&mut self`); заметьте, у вызывающего для этого должна быть `mut`-переменная — права складываются, как в §15.2.2. `square` — ассоциированная функция-конструктор: `self` нет, зовётся `Rect::square(7)`, возвращает `Self` (то есть `Rect`). И пара `r.area()` / `(&r).area()` печатает одно и то же — автоподстановка `&` в действии; писать её руками не нужно, но полезно один раз увидеть, *что именно* происходит.

*Пример 2 — метод-потребитель.* Приёмник `self` без `&` забирает владение — как любая функция с параметром-значением (§15.1.2). Файл `consume.rs`:

```rust
impl Rect {
    fn area(&self) -> u32 {
        self.width * self.height
    }

    fn into_dims(self) -> (u32, u32) {
        (self.width, self.height)
    }
}

fn main() {
    let r = Rect {
        width: 3,
        height: 4,
    };
    let (w, h) = r.into_dims();
    println!("{w} x {h}");
    println!("{}", r.area());
}
```

⛔ Не компилируется (проверено):

```
error[E0382]: borrow of moved value: `r`
  --> consume.rs:23:20
   |
17 |     let r = Rect {
   |         - move occurs because `r` has type `Rect`, which does not implement the `Copy` trait
...
21 |     let (w, h) = r.into_dims();
   |                    ----------- `r` moved due to this method call
22 |     println!("{w} x {h}");
23 |     println!("{}", r.area());
   |                    ^ value borrowed here after move
   |
note: `Rect::into_dims` takes ownership of the receiver `self`, which moves `r`
  --> consume.rs:11:18
   |
11 |     fn into_dims(self) -> (u32, u32) {
   |                  ^^^^
```

Методы не выключают главу 15: `into_dims` взяла `self` по владению — прямоугольник «разобран на запчасти», и обращаться к нему дальше нельзя. Компилятор называет вещи своими именами: «takes ownership of the receiver `self`, which moves `r`». Без последней строки программа собирается и печатает `3 x 4` (файл `consume_ok.rs`, проверено). Обратите внимание и на первую пометку: `Rect` «does not implement the `Copy` trait» — о том, почему ваша структура не Copy и что с этим делать, — §16.1.3.

*Пример 3 — научить структуру печататься.* Пока что мы печатали поля поштучно. Попытка напечатать структуру целиком упирается в ошибку — `println!("{r}")`:

⛔ Не компилируется (проверено):

```
error[E0277]: `Rect` doesn't implement `std::fmt::Display`
  --> nodisplay.rs:11:15
   |
11 |     println!("{r}");
   |               ^^^ `Rect` cannot be formatted with the default formatter
   |
   = note: in format strings you may be able to use `{:?}` (or {:#?} for pretty-print) instead
```

Заполнитель `{}` требует у типа «умения» `Display` — печати для пользователя; у чисел и строк оно есть, у вашей структуры — нет, и Rust не берётся угадывать формат (R1). Компилятор направляет к `{:?}` — **отладочной** печати (`Debug`). Но и `println!("{r:?}")` сам по себе не соберётся (проверено): `Rect` doesn't implement `Debug`... — зато в `help` компилятор подсказывает решение: добавить `#[derive(Debug)]`. Файл `debug1.rs`:

```rust
#[derive(Debug)]
struct Rect {
    width: u32,
    height: u32,
}

fn main() {
    let scale = 2;
    let r = Rect {
        width: dbg!(30 * scale),
        height: 50,
    };
    println!("{r:?}");
    println!("{r:#?}");
    dbg!(&r);
    println!("area = {}", r.width * r.height);
}
```

Стандартный вывод (проверено):

```
Rect { width: 60, height: 50 }
Rect {
    width: 60,
    height: 50,
}
area = 3000
```

И параллельно — в **stderr** (поток ошибок, знакомый по §10.1.1):

```
[debug1.rs:10:16] 30 * scale = 60
[debug1.rs:15:5] &r = Rect {
    width: 60,
    height: 50,
}
```

Три инструмента (R1, гл. 5):

- **`#[derive(Debug)]`** — атрибут над объявлением типа: «выведи реализацию `Debug` автоматически». Про атрибуты и `derive` подробнее — §16.1.3.
- **`{:?}` и `{:#?}`** — отладочная печать в одну строку и «красивая», с переносами (для больших структур).
- **`dbg!(выражение)`** — макрос-шпион: печатает в stderr файл, строку и значение выражения — и **возвращает значение владельцу**, так что его можно вставить прямо внутрь выражения (`width: dbg!(30 * scale)` присвоил полю 60, как будто `dbg!` и не было). Для структуры передаём `dbg!(&r)` — ссылку, чтобы макрос не забрал владение (R1): грабли примера 2 применимы и к макросам.

**Как это было в C.** Сопоставление с §8.1.2:

| | C (глава 8) | Rust |
|---|---|---|
| «Метод» | функция + указатель первым параметром | `fn` в `impl`, приёмник `self` |
| Вызов | `rect_area(&r)`; через указатель — `p->x`, `(*p).x` | всегда `r.area()`: `&`/`*` подставит компилятор, `->` нет (R1) |
| Право менять объект | по `const` в сигнатуре — если автор не забыл (§6.3.1) | `&self` / `&mut self` — вид приёмника обязателен и виден |
| Конструктор | функция-инициализатор по соглашению | ассоциированная `Тип::new(...)` / `Тип::square(...)` |
| Пространство имён | префиксы `vec_`, `stack_` (гл. 9, 11) | `impl`: функции живут «в типе», зовутся `Тип::...` |
| Отладочная печать | пишем `print_rect` руками | `#[derive(Debug)]` + `{:?}` — бесплатно |

**Связь с практикой.** Выбор приёмника — не стилистика, а контракт (тот же, что в §15.2): по `&self`/`&mut self`/`self` в документации типа сразу видно, что́ метод сделает с объектом. Начинающему достаточно двух рефлексов: почти все методы — `&self`; метод, меняющий поля, — `&mut self`; а `self` пишите только осознанно — «объект дальше не нужен». И вешайте `#[derive(Debug)]` на каждый учебный тип сразу: отладочная печать и `dbg!` окупаются с первой же ошибки.

**Типичные ошибки и подводные камни.**

- **Вызвать ассоциированную функцию через точку.** `r.square(3)` — E0599 с прямым объяснением: «this is an associated function, not a method» и подсказкой `Rect::square(3)` (проверено).
- **Менять поля в методе с `&self`.** Запись через разделяемую ссылку запрещена — это §15.2.1; просите `&mut self`, а у вызывающего пусть будет `mut`-переменная.
- **`self` «на всякий случай».** Заберёте объект — вызывающий получит E0382 (пример 2). По умолчанию — `&self`.
- **Печатать свой тип через `{}`.** `Display` у него нет; для отладки — `{:?}` + `derive(Debug)`. (Реализовать `Display` руками научимся, когда доберёмся до трейтов, — глава 19.)
- **`dbg!(r)` вместо `dbg!(&r)`.** Макрос честно вернёт владение, но если результат не присвоить — значение уедет и пропадёт; ссылка проще и безопаснее (R1).

**Источники:** R1 (гл. 5: impl и методы, `&self` как сокращение `self: &Self`, три приёмника, отсутствие `->` и автоподстановка `&`/`&mut`/`*`, методы с параметрами, ассоциированные функции и `::`, несколько impl-блоков, Display/Debug/`{:?}`/`{:#?}`/`dbg!`); R6 (Methods: сводка приёмников, ассоциированные функции без приёмника — конструкторы). Сторона C: §6.3.1, §8.1.2, §9–§11 (префиксные «методы»), §10.1.1 (stderr) курса.

### §16.1.3. Ваши типы в мире владения: move, Clone, Copy и derive

**Определения.** В §15.1.3 мы отложили вопрос: а как ведут себя при присваивании *собственные* типы? Ответ из двух частей.

Первая: **ваша структура по умолчанию перемещается** — даже если все её поля Copy. `Rect` из двух `u32` в примере 2 прошлого параграфа переехал в метод и не вернулся; компилятор прямо писал: тип `Rect` «does not implement the `Copy` trait». Никакого автоматического «раз поля простые — копируем» нет: копируемость — **свойство, о котором тип заявляет явно**.

Вторая: заявить о свойствах типа проще всего атрибутом **`#[derive(...)]`** — «выведи реализацию автоматически» (R1). Мы уже пользовались им для `Debug`; полный список выводимых умений — в приложении C книги Rust (R1), нам в этой главе нужны три:

- **`#[derive(Debug)]`** — отладочная печать `{:?}` (§16.1.2);
- **`#[derive(Clone)]`** — метод `.clone()`: **поэлементная глубокая копия** — клонируется каждое поле; для `String`-поля это новый буфер в куче, как в §15.1.3;
- **`#[derive(Copy)]`** — неявное побитовое копирование при присваивании и передаче (пишется в паре: `#[derive(Copy, Clone)]`). Доступен только типам, у которых **все поля Copy** — иначе отказ E0204; и, как мы знаем из §15.1.3, несовместим с особыми действиями при уничтожении.

Слово «трейт» (trait) в сообщениях компилятора — то самое «умение типа» из §15.1.3; создавать собственные трейты и реализовывать их вручную научит глава 19. Пока `derive` для нас — декларация: «мой тип умеет печататься / клонироваться / копироваться».

**Интуиция.** Дерево выбора из §15.1.3 теперь целиком в ваших руках:

```
        каким быть типу при let b = a; ?
                      │
    ┌─────────────────┴──────────────────┐
    ▼                                    ▼
 «плоский»: все поля Copy         внутри ресурс (String, ...)
 (числа, bool, пары чисел)                │
    │                            ┌────────┴────────┐
    ▼                            ▼                 ▼
 #[derive(Copy, Clone)]     #[derive(Clone)]   без derive
 присваивание копирует;     копия — только     только move:
 источник жив               явным .clone()     владелец один
      Point, Color            Track, Student     (уникальный ресурс)
```

Правило композиции то же, что у кортежей и массивов (§15.1.3): один не-Copy жилец — и весь дом не Copy. Разница в том, что для *своих* типов даже «все жильцы Copy» — лишь разрешение попросить `Copy`, а не сам `Copy`.

**Разобранные примеры.**

*Пример 1 — структура переезжает.* Файл `move_struct.rs`:

```rust
struct Track {
    title: String,
    seconds: u32,
}

fn main() {
    let a = Track {
        title: String::from("Intro"),
        seconds: 90,
    };
    let b = a;
    println!("a: {} ({} s)", a.title, a.seconds);
    println!("b: {} ({} s)", b.title, b.seconds);
}
```

⛔ Не компилируется (проверено):

```
error[E0382]: borrow of moved value: `a`
  --> move_struct.rs:12:39
   |
 7 |     let a = Track {
   |         - move occurs because `a` has type `Track`, which does not implement the `Copy` trait
...
11 |     let b = a;
   |             - value moved here
12 |     println!("a: {} ({} s)", a.title, a.seconds);
   |                                       ^^^^^^^^^ value borrowed here after move
   |
note: if `Track` implemented `Clone`, you could clone the value
  --> move_struct.rs:1:1
   |
 1 | struct Track {
   | ^^^^^^^^^^^^ consider implementing `Clone` for this type
...
11 |     let b = a;
   |             - you could clone this value
```

Сценарий §15.1.2 один в один, только теперь для вашего типа — и с новой подсказкой 1.97: у `String` компилятор предлагал готовый `.clone()`, а здесь замечает, что у `Track` и умения-то такого нет — «consider implementing `Clone` for this type».

*Пример 2 — послушаемся: derive(Clone).* Файл `derive_clone.rs`:

```rust
#[derive(Clone)]
struct Track {
    title: String,
    seconds: u32,
}

fn main() {
    let a = Track {
        title: String::from("Intro"),
        seconds: 90,
    };
    let b = a.clone();
    println!("a: {} ({} s)", a.title, a.seconds);
    println!("b: {} ({} s)", b.title, b.seconds);
}
```

```
a: Intro (90 s)
b: Intro (90 s)
```

Одна строка атрибута — и у типа появился `.clone()`: `title` склонирован в новый буфер, `seconds` скопирован. Обе дорожки независимы, обе живы. Цена, как всегда у `clone`, видна в месте вызова (§15.1.3).

*Пример 3 — Copy для плоских, отказ для остальных.* Точка на плоскости — идеальный кандидат в Copy: два `i32`, никаких ресурсов. Файл `derive_copy.rs`:

```rust
#[derive(Debug, Clone, Copy)]
struct Point {
    x: i32,
    y: i32,
}

fn main() {
    let p = Point { x: 3, y: 7 };
    let q = p;
    println!("p = {p:?}, q = {q:?}");
    println!("p.x = {}, q.y = {}", p.x, q.y);
}
```

```
p = Point { x: 3, y: 7 }, q = Point { x: 3, y: 7 }
p.x = 3, q.y = 7
```

`let q = p;` — копия, `p` жив: тип ведёт себя как `i32`. А теперь попробуем выдать `Copy` типу с ресурсом — файл `copy_string.rs`:

```rust
#[derive(Clone, Copy)]
struct Track {
    title: String,
    seconds: u32,
}
```

⛔ Не компилируется (проверено):

```
error[E0204]: the trait `Copy` cannot be implemented for this type
 --> copy_string.rs:2:8
  |
1 | #[derive(Clone, Copy)]
  |                 ---- in this derive macro expansion
2 | struct Track {
  |        ^^^^^
3 |     title: String,
  |     ------------- this field does not implement `Copy`
```

Компилятор указывает пальцем на виновника: поле `title` не Copy — значит, побитовая копия структуры создала бы два указателя на один буфер. Это ровно то двойное владение, от которого нас лечила вся глава 15; E0204 не даёт протащить его через чёрный ход.

**Как это было в C.** В C присваивание структур — **всегда** побайтовая копия, без вопросов: §8.1.2 демонстрировал, что копируется даже поле-массив. Звучит удобнее? Вспомните, чем это оборачивалось для поля-указателя: копировался *указатель*, и появлялись два «владельца» одного блока — раскладка §7.2.1, двойное освобождение в перспективе. C копировал всё, не различая байты и ресурсы; Rust требует, чтобы тип **доказал**, что побитовая копия безопасна (E0204 — отказ в доказательстве), а для остального предлагает выбор: явный `.clone()` или переезд.

**Связь с практикой.** Рабочие рецепты для своих типов:

- маленькие «плоские» значения (точка, цвет, пара координат, размер): `#[derive(Debug, Clone, Copy)]` — пусть ведут себя как числа;
- составные типы с ресурсами (`String`-поля — наши `Track`, `Student`): `#[derive(Debug, Clone)]` — копия возможна, но всегда явная;
- типы, для которых «две копии» — ошибка по смыслу (уникальный ресурс, дескриптор файла — встретим в главе 18): только `Debug`, пусть перемещаются.

Заведите привычку писать строку `derive` сразу под будущие нужды типа — это дешевле, чем чинить E0382/E0277 по мере их появления.

**Типичные ошибки и подводные камни.**

- **Ждать автоматического Copy у «простой» структуры.** Не будет: без `derive(Copy, Clone)` даже `struct Point { x: i32, y: i32 }` перемещается (пример 2 §16.1.2).
- **Чинить E0204 удалением «мешающего» поля или заменой на `&str`.** Поле не виновато; типу с ресурсом положен `Clone`, а не `Copy`.
- **`derive(Copy)` «чтобы отстал borrow checker».** Для плоских типов это законно, но помните: копия — это *другое* значение; изменения в `q` не видны в `p`. Если по смыслу объект один — передавайте ссылку (§15.2), а не плодите копии.
- **Клонировать в цикле.** `.clone()` на каждой итерации — знакомый по §15.1.3 запах; чаще всего там, где клон, хотела быть ссылка.

**Источники:** R1 (гл. 4: Copy/Clone и их несовместимость с Drop, правило «скаляры — да, ресурсы — нет»; гл. 5 и приложение C: атрибут `derive` и выводимые трейты); R6 (Memory Management / Copy types: Copy — побитово и неявно, Clone — явно). Сторона C: §7.2.1, §8.1.2 курса.

## Часть 16.2. Перечисления, Option и сопоставление с образцом

### §16.2.1. enum: тип-сумма — «ровно одно из»

**Определения.** **Перечисление** (enum) объявляет тип, значение которого — **ровно один из перечисленных вариантов** (variants) (R2: «тип, который может быть одним из нескольких разных вариантов»):

```rust
enum Shape {
    Dot,
    Circle(f64),
    Rect { width: f64, height: f64 },
}
```

Ключевые факты (R1, гл. 6; R7 — арбитр):

- Варианты живут **в пространстве имён типа**: `Shape::Dot`, `Shape::Circle(1.5)` — знакомое `::` из §16.1.2.
- Все варианты — значения **одного типа** `Shape`: их можно класть в один массив, передавать в одну функцию.
- Вариант может **нести данные**, причём каждый — свои. Допустима любая форма, которая допустима у структур (R2): единичный вариант (`Dot`), кортежный (`Circle(f64)`), структурный (`Rect { width, height }`).
- Имя варианта с данными работает как **функция-конструктор**: `Shape::Circle(1.5)` строит значение (R1).
- Методы и ассоциированные функции — тем же `impl Shape { ... }`, что у структур (R1).

Чем это отличается от `enum` C (§8.2.1)? Всем. Там `enum` — набор целочисленных констант: `enum color c = 42;` компилировался у нас без единого предупреждения, донумерация `{A=3, B, C=1, D}` давала 3, 4, 1, 2, и всё свободно смешивалось с `int`. В Rust перечисление — **закрытый номинальный тип**: целое в него не превращается (⛔-пример 3), нумерация вариантов — внутренняя деталь. Она, впрочем, есть: у каждого значения enum есть **дискриминант** — целочисленная метка варианта; неявно варианты нумеруются с нуля по порядку (R7). Задать значения явно (`Ok = 0, Warning = 3, Error = 4` — компилируется, проверено) можно только перечислению, у которого **ни один вариант не несёт данных** (unit-only), — иначе нужно явно задавать целочисленное представление, о чём нам пока рано (R7; ⛔-пример 3).

**Интуиция.** Вы уже строили «вариант с данными» — вручную, в §8.2.2: размеченное объединение (tagged union):

```
C, §8.2.2: сделай сам                        Rust: enum Shape — одно значение
─────────────────────────                    ─────────────────────────────────
struct shape {                               ┌──────────────┬─────────────────┐
    enum kind tag;   ← тег отдельно          │ дискриминант │ данные варианта │
    union {          ← данные отдельно       └──────────────┴─────────────────┘
        double radius;                       тег и данные неразлучны;
        struct { double w, h; } rect;        обновляются только вместе;
    } u;                                     прочитать «не то поле» нельзя:
};                                           данные достаются только разбором
чтение s.u.rect при tag == CIRCLE            (match, §16.2.2)
компилируется и даёт мусор (§8.2.2)
```

Память устроена как у нашей ручной конструкции: место под метку плюс место под самый вместительный вариант. Проверим на эталоне: `std::mem::size_of::<Shape>()` (§14.2.2) даёт **24** байта — 16 под самый большой вариант (`Rect`: два `f64`) плюс 8 под дискриминант с выравниванием под `f64`. Числа — наш прогон на x86-64; точную раскладку язык не обещает, компилятор вправе её оптимизировать (R6 отмечает, что под дискриминант отводится минимально необходимое место, — эффектный пример такой оптимизации ждёт вас в §16.2.3).

**Разобранные примеры.**

*Пример 1 — три вида вариантов в работе.* Файл `shapes.rs`:

```rust
#[derive(Debug, Clone, Copy)]
enum Shape {
    Dot,
    Circle(f64),
    Rect { width: f64, height: f64 },
}

fn area(s: Shape) -> f64 {
    match s {
        Shape::Dot => 0.0,
        Shape::Circle(r) => 3.14159 * r * r,
        Shape::Rect { width, height } => width * height,
    }
}

fn main() {
    let shapes = [
        Shape::Dot,
        Shape::Circle(1.5),
        Shape::Rect {
            width: 3.0,
            height: 4.0,
        },
    ];
    for i in 0..shapes.len() {
        println!("{:?} -> area {}", shapes[i], area(shapes[i]));
    }
    println!("size of Shape: {}", std::mem::size_of::<Shape>());
    println!("size of f64:   {}", std::mem::size_of::<f64>());
}
```

```
Dot -> area 0
Circle(1.5) -> area 7.068577499999999
Rect { width: 3.0, height: 4.0 } -> area 12
size of Shape: 24
size of f64:   8
```

Три фигуры трёх разных «форм» — в одном массиве `[Shape; 3]` и в одной функции: это и есть выгода «все варианты — один тип». Внутри `area` — конструкция `match`, разбирающая значение по вариантам; её полный разбор — следующий параграф, здесь она нужна, чтобы честно достать данные. Обратите внимание на `derive`: `Debug` даёт печать `{:?}` (варианты печатаются в своей форме — `Circle(1.5)`, `Rect { width: 3.0, height: 4.0 }`), а `Clone, Copy` законны, потому что все данные всех вариантов — `f64`: правила §16.1.3 действуют и для enum.

*Пример 2 — данные запечатаны.* А напрямую, без разбора? Файл `enum_field.rs`:

```rust
let c = Shape::Circle(1.5);
println!("{}", c.0);
```

⛔ Не компилируется (проверено):

```
error[E0609]: no field `0` on type `Shape`
 --> enum_field.rs:8:22
  |
8 |     println!("{}", c.0);
  |                      ^ unknown field
```

У типа `Shape` нет поля `.0` — поле есть у *варианта*, а компилятор не знает (и знать в общем случае не может), какой вариант лежит в `c` прямо сейчас. Сравните с §8.2.2: там чтение `u.circle` при теге `RECT` компилировалось и возвращало мусор. Здесь доступ мимо разбора не существует в принципе — «сначала докажи, какой вариант, потом получишь данные» (§16.2.2).

*Пример 3 — enum не притворяется числом.* Файл `enum_int.rs`:

```rust
#[derive(Debug)]

    North,
    East,
    South,
    West,
}

fn main() {
    let d: Direction = 2;

```

⛔ Не компилируется (проверено):

```
error[E0308]: mismatched types
  --> enum_int.rs:10:24

   |            |

```

В §8.2.1 присваивание `enum color c = 42;` — с несуществующим «цветом 42» — GCC пропускал молча даже с `-Wall -Wextra`. Rust отвечает E0308: перечисление — не целое, значений вне списка вариантов у него не бывает. Не бывает и «донумерации с данными»: явный дискриминант у варианта с полями — отказ (файл `discr_data.rs`, проверено):

```
error[E0732]: `#[repr(inttype)]` must be specified for enums with explicit discriminants and non-unit variants
 --> discr_data.rs:1:1
  |
1 | enum Token {
  | ^^^^^^^^^^
2 |     Plus = 1,
3 |     Number(i32) = 2,
  |                   - explicit discriminant on non-unit variant specified here
```

**Как это было в C.** Сводка по §8.2:

| | C (глава 8) | Rust |
|---|---|---|
| Природа enum | целые константы, под капотом `int` | закрытый номинальный тип |
| `c = 42` вне списка | компилируется молча (§8.2.1) | E0308 |
| Нумерация | видимая, `{A=3, B, C=1}` → 3, 4, 1 | внутренняя; явно — только без данных (E0732) |
| Вариант с данными | тег + `union` руками (§8.2.2) | вариант несёт данные сам |
| Чтение «не того поля» | компилируется: мусор, а то и хуже | невыразимо: только разбор (E0609 / match) |
| Непокрытый вариант в switch | `-Wswitch`: предупреждение — и сборка | E0004: отказ компиляции (§16.2.2) |

**Связь с практикой.** Перечисление — инструмент для всякого «ровно одно из»: состояние автомата (наш светофор в §16.2.2), команда протокола, событие интерфейса (вспомните `WM_*`-сообщения из главы 12 — их «тип + данные в двух полях по договорённости» просится в enum), результат разбора. Практическое чутьё: если в вашей структуре завелось поле-«метка», по значению которого осмысленна лишь часть остальных полей, — перед вами enum, записанный по-старому; §8.2.2 именно так и выглядел.

**Типичные ошибки и подводные камни.**

- **Тянуть данные точкой.** `c.0` / `c.radius` — E0609; данные варианта достаются только разбором (`match`, `if let`).
- **Ждать целочисленной жизни C-шного enum.** Ни `= 2`, ни арифметики; сравнение вариантов «на равенство» тоже не бесплатно — ему нужен трейт `PartialEq` (глава 19), а до тех пор сравнивают match-ем.
- **Плодить структуры вместо вариантов.** Четыре разных `struct` для четырёх видов команд не положишь в один массив; enum с четырьмя вариантами — положишь (R1 приводит ровно этот довод).
- **Забыть, что derive-правила §16.1.3 действуют и здесь.** `Clone, Copy` — только если данные всех вариантов Copy; `Debug` — почти всегда полезен.

**Источники:** R1 (гл. 6.1: определение, варианты с данными, конструкторы, methods, довод против набора структур); R2 (Enums: три формы вариантов); R6 (Enums: sum type, минимальный дискриминант); R7 (Enumerations — арбитр: номинальный тип, неявная нумерация с 0, явные дискриминанты только у unit-only без repr). Сторона C: §8.2.1–8.2.2, гл. 12 (сообщения) курса.

### §16.2.2. match: разбор с гарантией полноты

**Определения.** **`match`** — конструкция сопоставления с образцом (pattern matching): значение по очереди сравнивается с **образцами** (patterns) **рукавов** (arms), сверху вниз; выполняется код **первого** совпавшего рукава (R7). Синтаксис:

```rust
match значение {
    образец1 => код1,
    образец2 => { многострочный код }
    ...
}
```

Свойства, каждое из которых мы проверим прогоном:

- **`match` — выражение** (§14.2.3): его значение — значение выбранного рукава; все рукава обязаны давать один тип (R7). Поэтому `match` можно присвоить, вернуть из функции, поставить последним выражением тела.
- Разбирать можно значение **любого типа** — не только «истину/ложь», как в `if` (R1).
- Образец может **привязывать данные варианта** к именам: `Shape::Circle(r)` кладёт содержимое в `r` (R1). Для структурных вариантов — по именам полей: `Shape::Rect { width, height }`.
- **Исчерпывающесть** (exhaustiveness): образцы обязаны покрыть *все* возможные значения — «matches в Rust исчерпывающи» (R1). Непокрытый вариант — отказ E0004 (пример 3).
- **Уловитель** (catch-all): последний рукав-образец, совпадающий с чем угодно, — либо имя (`other` — значение доступно в рукаве), либо `_` — «совпади и выброси» (R1). Идиома «для остального ничего не делать» — `_ => ()` (единичный тип из §14.2.2). Рукава после уловителя недостижимы — компилятор предупредит (пример 4).

> **Новый макрос: `format!`.** Как `println!`, но результат не печатается, а возвращается строкой `String`. Та же мини-грамматика заполнителей: `format!("forward {n}")`. Родственник нашего `snprintf` из §12.2.1 — «собрать строку, не печатая» — только буфер и его размер Rust берёт на себя.

**Интуиция.** `switch` из главы 3 сортировал по числу. `match` сортирует по **форме значения** — варианту и его содержимому:

```
                match cmd
      ┌────────────┼──────────────┬───────────────┐
      ▼            ▼              ▼               ▼
 Stop         Forward(n)     Turn { degrees }  Say(text)
  │            n = 10         degrees = -90     text = "hello"
  ▼            ▼              ▼                 ▼
 "stop"       "forward 10"   "turn -90 deg"    "say 'hello'"
```

Книга Rust сравнивает `match` с монетосортировочной машиной: значение катится вдоль ряда лотков и проваливается в первый, куда подходит по форме (R1). Добавьте к картинке табличку на выходе: «все формы монет учтены — иначе машина не запускается».

**Разобранные примеры.**

*Пример 1 — match как выражение; методы на enum.* Светофор — перечисление из трёх единичных вариантов, его поведение — методы. Файл `match1.rs`:

```rust
#[derive(Debug, Clone, Copy)]
enum Light {
    Red,
    Yellow,
    Green,
}

impl Light {
    fn next(self) -> Light {
        match self {
            Light::Red => Light::Green,
            Light::Green => Light::Yellow,
            Light::Yellow => Light::Red,
        }
    }

    fn seconds(self) -> u32 {
        match self {
            Light::Red => 30,
            Light::Yellow => 3,
            Light::Green => 25,
        }
    }
}

fn main() {
    let mut l = Light::Red;
    for _ in 0..4 {
        println!("{l:?} for {} s", l.seconds());
        l = l.next();
    }
}
```

```
Red for 30 s
Green for 25 s
Yellow for 3 s
Red for 30 s
```

Оба метода — «одно выражение `match`»: без `return`, без временных переменных — значение рукава и есть значение метода. Приёмник `self` по значению здесь дёшев и законен: `Light` — Copy (§16.1.3). Заметьте, рукава `next` перечислены не в порядке объявления вариантов — для исчерпывающести важен состав, а не порядок. (Имя `_` в заголовке цикла — та же «заглушка-неиспользуемое», что и образец `_`: счётчик нам не нужен, только четыре повторения.)

*Пример 2 — привязка данных.* Команды воображаемому роботу — все три формы вариантов; `match` достаёт содержимое. Файл `match2.rs`:

```rust
enum Command {
    Stop,
    Forward(i32),
    Turn { degrees: i32 },
    Say(String),
}

fn describe(cmd: Command) -> String {
    match cmd {
        Command::Stop => String::from("stop"),
        Command::Forward(n) => format!("forward {n}"),
        Command::Turn { degrees } => format!("turn {degrees} deg"),
        Command::Say(text) => format!("say '{text}'"),
    }
}

fn main() {
    println!("{}", describe(Command::Stop));
    println!("{}", describe(Command::Forward(10)));
    println!("{}", describe(Command::Turn { degrees: -90 }));
    println!("{}", describe(Command::Say(String::from("hello"))));
}
```

```
stop
forward 10
turn -90 deg
say 'hello'
```

В рукаве `Forward(n)` имя `n` — свежая переменная, получившая данные варианта; у структурного `Turn { degrees }` — привязка по имени поля. Владение работает как обычно (§15.1.2): `describe` принимает команду по значению и потребляет её — для команды со `String` внутри это честный переезд строки в привязку `text` и далее в `format!`.

*Пример 3 — исчерпывающесть: забытая ветка не компилируется.* Уберём из `seconds` рукав `Green` — файл `nonexh.rs`:

⛔ Не компилируется (проверено):

```

  --> nonexh.rs:9:11
   |
 9 |     match l {
   |           ^ pattern `Light::Green` not covered
   |
note: `Light` defined here
  --> nonexh.rs:2:6
   |
 2 | enum Light {
   |      ^^^^^
...
 5 |     Green,
   |     ----- not covered
   = note: the matched value is of type `Light`
help: ensure that all possible cases are being handled by adding a match arm

```

Компилятор не просто отказал — он **назвал** забытый вариант и предложил заготовку правки (`todo!()` — заглушка «допишу позже»). Сравните с C: `switch` по enum без `default` и без ветки `GREEN` вызывал у GCC предупреждение `-Wswitch` (§8.2.1) — но сборка продолжалась, и программа с «дыркой» уходила в работу. Здесь дырка в разборе — не стиль, а ошибка типа. То же самое ждёт забытую ветку `None` у `Option` — мы прогнали и этот вариант (файл `nonexh_opt.rs`): `non-exhaustive patterns: `None` not covered`, с тем же `help`; он сыграет главную роль в §16.2.3.

*Пример 4 — уловитель: последний и единственный.* Игральная кость: 3 — надеть шляпу, 7 — снять, остальное — ход на выпавшее число. Файл `catchall.rs`:

```rust
fn main() {
    let rolls = [3, 7, 2];
    for i in 0..rolls.len() {
        match rolls[i] {
            3 => println!("3: fancy hat"),
            7 => println!("7: lose the hat"),
            other => println!("{other}: move {other} squares"),
        }
    }
    match rolls[2] {
        3 => println!("hat"),
        7 => println!("no hat"),
        _ => (),
    }
    println!("done");
}
```

```
3: fancy hat
7: lose the hat
2: move 2 squares
done
```

`i32` матч-ем по вариантам не переберёшь — миллиарды значений; исчерпывающесть обеспечивает уловитель. `other` — когда значение нужно; `_` — когда нет; `_ => ()` — «и не делай ничего». Порядок критичен: рукава проверяются сверху вниз, и уловитель, поставленный раньше, съест всё — компилятор предупредит (файл `unreach.rs`, рукав `5 => ...` после `_`, проверено):

```
warning: unreachable pattern
 --> unreach.rs:5:9
  |
4 |         _ => println!("anything"),
  |         - matches any value
5 |         5 => println!("five"),
  |         ^ no value can reach this
```

Программа собирается и печатает `anything` — но предупреждение честно сообщает, что пятая ветка мертва.

**Как это было в C.** `switch` (глава 3) против `match`:

| | C: switch (гл. 3, §8.2.1) | Rust: match |
|---|---|---|
| По чему ветвимся | целые константы | любой тип: варианты, числа, кортежи... |
| Данные варианта | достаём руками из union (§8.2.2) | привязка прямо в образце |
| Забытый вариант | `-Wswitch` предупредит — и соберёт | E0004: отказ, с именем варианта |
| «Всё остальное» | `default` (можно забыть — молчание) | уловитель обязателен для полноты |
| Проваливание | fallthrough без `break` — источник багов | рукава независимы, «провалиться» нельзя |
| Результат | оператор: значения нет | выражение: значение рукава |

**Связь с практикой.** Исчерпывающесть — это страховка **на вырост**: добавьте завтра в `Command` вариант `Backward(i32)` — и каждый `match` без уловителя откажется собираться, пока вы не решите, что делать с новой командой; компилятор выдаст список мест правок бесплатно. Отсюда правило: по перечислениям, которые будут расти, пишите **явные рукава** — уловитель `_` там, где «остальное действительно неважно» (как у числа на кости), а не там, где лень перечислять. Каждый `_` по enum — это будущий вариант, молча провалившийся в «остальное».

**Типичные ошибки и подводные камни.**

- **Уловитель раньше конкретики.** Рукава после него мертвы (`unreachable pattern`); уловитель — последним.
- **`_` вместо честных веток по «живому» enum.** Компилируется сегодня — молча глотает варианты завтра. E0004 — друг, не боритесь с ним уловителем.
- **Разные типы в рукавах.** `match` — выражение: «в одном рукаве число, в другом строка» не соберётся — то же правило, что у ветвей `if`-выражения (§14.2.4).
- **Забыть, что разбор по значению потребляет.** `describe(cmd)` забрал команду насовсем — как всякий параметр-значение (§15.1.2); передавать разбор по ссылке научимся на коллекциях (глава 17).

**Источники:** R1 (гл. 6.2: синтаксис и аналогия, привязки, исчерпывающесть, catch-all/`_`/`()`, порядок рукавов; гл. 6.3 — format! в примерах); R7 (match — арбитр: порядок сверху вниз, первый совпавший, согласование типов рукавов); R2 (Enums: match по вариантам). Сторона C: гл. 3 (switch), §8.2.1–8.2.2, §12.2.1 (snprintf) курса.

### §16.2.3. Option<T>: отсутствие значения — это тип

**Определения.** Всё, что нужно, чтобы заменить NULL, у нас теперь есть. **`Option<T>`** — перечисление стандартной библиотеки (R1, гл. 6; R9):

```rust
enum Option<T> {
    None,
    Some(T),
}
```

Каждое значение `Option` — «либо `Some` с данными внутри, либо `None` — данных нет» (R9). Запись `<T>` — **параметр типа**: подставьте на его место конкретный тип и получите конкретное перечисление — `Option<i32>`, `Option<usize>`, `Option<&str>`. Механику таких «типов с дыркой под тип» (обобщённых типов) разберёт глава 19; для работы с `Option` хватит интуиции «шаблон, куда подставили тип» — вы уже читали такие записи в §13.2.2.

Факты для работы (R1, R9):

- `Option`, `Some`, `None` доступны везде без подготовки (они в **прелюдии** — наборе имён, который Rust подключает к каждой программе) — и без префикса: пишут `Some(5)`, а не `Option::Some(5)`.
- **`Option<T>` и `T` — разные типы.** «Может-быть-число» нельзя сложить с числом (⛔-пример 2), передать в функцию, ждущую число, и вообще использовать как число, пока вы явно не разберёте оба случая. В этом вся конструкция: в C проверку NULL можно было забыть — здесь «забытая проверка» не имеет типа.
- Голому `None` нужна аннотация: `let absent: Option<i32> = None;` — компилятору неоткуда узнать, «отсутствие чего» вы имеете в виду (R1; проверено в примере 1).
- Основной способ разбора — `match` с ветками `Some(x)` / `None` (§16.2.2); исчерпывающесть гарантирует, что ветку `None` не забыть (E0004 — прогнано в §16.2.2).
- Методы-выходы для типовых случаев (R9): `unwrap()` — «дай значение или **паникуй**»; `expect("сообщение")` — то же с вашим текстом паники; `unwrap_or(значение)` — «дай значение или вот это, по умолчанию».

**Интуиция.** Том I выражал «может не быть» **особыми значениями обычных типов**: `fopen` возвращал NULL (§10.1.1), наши поиски — индекс −1 (главы 5, 9), `fgetc` — EOF (§10.1.2). У всех одна беда: «нет ответа» и «ответ» — значения *одного типа*, их различение — договорённость в комментарии, и препарат 2 из §13.1.2 показал цену забытой проверки. `Option` выносит различие в тип:

```
C: «полузначения»                       Rust: Option<T>
──────────────────────                  ────────────────────────────
FILE *f  = fopen(...);   NULL?          Option<usize>
int  pos = find(...);    -1?              ┌─ Some(i)  — есть, вот оно
int  ch  = fgetc(f);     EOF?             └─ None     — нет
   тип один — проверка на совести;      типа T в руках НЕТ, пока не
   забыли → UB или мусор                разобрали; забыть нельзя
```

Хоара с его «ошибкой на миллиард долларов» мы уже цитировали в §13.2.2 — теперь вы видите весь механизм: Rust не «запретил null», а сделал отсутствие **отдельным вариантом отдельного типа**.

**Разобранные примеры.**

*Пример 1 — функция, которой нечего стыдиться.* Поиск первого чётного: результата может не быть, и это видно в сигнатуре. Файл `option1.rs`:

```rust
fn first_even(xs: &[i32]) -> Option<i32> {
    for i in 0..xs.len() {
        if xs[i] % 2 == 0 {
            return Some(xs[i]);
        }
    }
    None
}

fn main() {
    let a = [3, 7, 10, 9];
    let b = [1, 3, 5];

    match first_even(&a) {
        Some(x) => println!("first even in a: {x}"),
        None => println!("a has no even numbers"),
    }
    match first_even(&b) {
        Some(x) => println!("first even in b: {x}"),
        None => println!("b has no even numbers"),
    }

    let some_number = Some(5);
    let absent: Option<i32> = None;
    println!("{some_number:?} {absent:?}");
}
```

```
first even in a: 10
b has no even numbers
Some(5) None
```

Ровно так был устроен `find` из витрины §13.2.2 — теперь вы можете написать его сами (практика, З9). Обратите внимание на последние строки: у `Some(5)` тип выведен (`Option<i32>` по литералу `5`), у `None` — задан аннотацией; печатаются оба через `{:?}` — `Option` умеет `Debug`.

*Пример 2 — Option<T> не притворяется T.* Файл `option_add.rs`:

```rust
let x: i32 = 5;
let y: Option<i32> = Some(5);
let sum = x + y;
```

⛔ Не компилируется (проверено; хвост подсказок сокращён):

```
error[E0277]: cannot add `Option<i32>` to `i32`
 --> option_add.rs:4:17
  |
4 |     let sum = x + y;
  |                 ^ no implementation for `i32 + Option<i32>`
  |
  = help: the trait `Add<Option<i32>>` is not implemented for `i32`
```

«Может-быть-пять» плюс пять — не десять: сперва скажите, что делать, если там `None`. Каноническая починка — `match`, как в функции `plus_one` (файл `plus_one.rs`, дословно по R1, проверено):

```rust
fn plus_one(x: Option<i32>) -> Option<i32> {
    match x {
        None => None,
        Some(i) => Some(i + 1),
    }
}
```

`plus_one(Some(5))` → `Some(6)`, `plus_one(None)` → `None` (проверено): отсутствие честно протекает насквозь, не превращаясь молча в ноль. Быстрая починка для «по умолчанию ноль» — `x + y.unwrap_or(0)` — тоже годится: выбор *озвучен* в коде.

*Пример 3 — unwrap: паника по подписке.* Файл `unwrap_demo.rs`:

```rust
fn main() {
    let x: Option<i32> = Some(41);
    println!("{}", x.unwrap() + 1);
    println!("{}", x.unwrap_or(0));

    let y: Option<i32> = None;
    println!("{}", y.unwrap_or(0));
    println!("{}", y.unwrap());
}
```

Печатает `42`, `41`, `0` — и на последней строке (проверено):

```
thread 'main' (20204) panicked at unwrap_demo.rs:8:22:
called `Option::unwrap()` on a `None` value
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace
```

(Номер в скобках — служебный, §14.2.2.) `unwrap` — это «здесь `None` быть не может, ручаюсь; если я неправ — остановите программу». Иногда это правда уместно — но слово `unwrap` остаётся в коде и видно на ревью (мы говорили об этом в §13.2.2). Пока наше правило простое: в примерах и заданиях курса — `match`/`if let`/`unwrap_or`; `unwrap` — только там, где невозможность `None` очевидна из соседних строк.

*Пример 4 — сколько стоит Option.* Тип-обёртка — это же лишние байты? Проверим (файл `opt_size.rs`, размеры — §14.2.2):

```rust
println!("i32:          {}", std::mem::size_of::<i32>());
println!("Option<i32>:  {}", std::mem::size_of::<Option<i32>>());
println!("&i32:         {}", std::mem::size_of::<&i32>());
println!("Option<&i32>: {}", std::mem::size_of::<Option<&i32>>());
```

```
i32:          4
Option<i32>:  8
&i32:         8
Option<&i32>: 8
```

`Option<i32>` платит за дискриминант (4 → 8 с выравниванием; числа платформенные, как в §16.2.1). А вот `Option<&i32>` **не стоит ничего**: 8 байт, как голая ссылка, — и это не оптимизация-сюрприз, а **гарантия языка** для ссылок и ряда других типов (R9). Фокус в том, что у ссылки есть «невозможное» битовое значение — нулевой адрес (§15.2.1: ссылки не бывают нулевыми), — и компилятор кодирует им `None` (R6 называет это niche-оптимизацией). Прочувствуйте симметрию: **NULL вернулся в машинный код — но исчез из системы типов.** Указатель-который-может-быть-нулём из C и `Option<&T>` из Rust стоят одинаково; разница лишь в том, что здесь разыменовать «ноль» не даст компилятор.

**Как это было в C.** Каталог «полузначений» тома I и их замена:

| Задача | C (том I) | Rust |
|---|---|---|
| «Файл не открылся» | `fopen` → NULL (§10.1.1) | `Option` (а с причиной — `Result`, гл. 18) |
| «Не нашли» | индекс −1 (гл. 5, 9) | `None` в `Option<usize>` |
| «Поток кончился» | `fgetc` → EOF (§10.1.2) | `None` (итераторы — гл. 21) |
| Забытая проверка | компилируется; UB/мусор (§13.1.2) | не имеет типа: E0277 или E0004 |
| Цена | 0 байт («значение как значение») | 0 байт для `Option<&T>` (гарантия R9); дискриминант для остальных |

**Связь с практикой.** `Option` в сигнатуре — это документация, которую проверяет компилятор: `fn first_even(...) -> Option<i32>` невозможно «прочитать невнимательно». Типовые места по R9: значение, которого может не быть у части входов (наши поиски); необязательное поле структуры; «ещё не инициализировано». Правило выхода: наружу программы (печать, финальный результат) — разберите `match`-ем или дайте умолчание `unwrap_or`; внутри логики — передавайте `Option` как есть, пусть каждый слой честно видит «может не быть». И анонс: для «не получилось **и вот почему**» у Rust есть парный тип `Result` — глава 18.

**Типичные ошибки и подводные камни.**

- **Рефлекторный `unwrap`.** Каждый `unwrap` — заявка «None невозможен»; если это не доказано соседним кодом — вы вернули в программу разыменование NULL, пусть и с вежливой паникой вместо UB.
- **Sentinel по привычке.** Вернуть `-1`/`usize::MAX` «как в C» — компилируется, но выключает всю защиту типа. Есть «может не быть» — есть `Option`.
- **Забыть аннотацию у голого `None`.** Компилятору нужен `T` — скажите, отсутствие *чего* вы храните.
- **Избегать `Option<&T>` «из экономии».** Он бесплатен (пример 4) — в отличие от бага с «особым значением».

**Источники:** R1 (гл. 6.1: определение Option, параметр типа, прелюдия, довод против null и цитата Хоара, E0277 при смешении типов, «преобразуй Option<T> в T прежде чем использовать»); R9 (std::option — арбитр по API: формулировка Some/None, типовые применения, unwrap/expect/unwrap_or, гарантия размера для Option<&T>); R6 (niche-оптимизация: None кодируется запрещённым битовым значением). Сторона C: §5.2, гл. 9, §10.1.1–10.1.2, §13.1.2, §13.2.2 курса.

### §16.2.4. if let и let else: когда важен один вариант

**Определения.** `match` требует разобрать всё — но часто важен один вариант, а остальные «пропустить». Для этого — две короткие формы (R1, гл. 6.3).

**`if let`** — «match на один образец»:

```rust
if let Some(max) = config_max {
    println!("maximum is {max}");
}
```

Читается: «если `config_max` подходит под образец `Some(max)` — выполнить блок с привязкой `max`». Эквивалент `match` с рукавом `_ => ()`; можно добавить `else` — он и есть этот рукав `_`. Цена короткой формы (R1, дословно): меньше набора, отступов и шаблонного кода — но **теряется проверка исчерпывающести**; выбирать между `match` и `if let` — значит выбирать между гарантией полноты и краткостью.

**`let ... else`** — «привяжи или уйди»:

```rust
let Some(n) = x else {
    return 0;
};
// дальше n доступна как обычная переменная
```

Если образец подошёл — привязки действуют **до конца функции**, без вложенного блока. Если нет — выполняется `else`, и он **обязан расходиться** (diverge): `return`, `panic!`, выход из цикла — продолжить «сквозь» него нельзя, компилятор проверяет (⛔-пример 3). Назначение — «счастливый путь» (happy path, R1): проверки выходят в короткие `else`, основная логика идёт без лесенки отступов. (У пары есть и третья форма — цикл `while let`; покажем её, когда появятся коллекции, — глава 17.)

**Интуиция.** Это знакомый по C **страж** (guard) — ранний выход при неудаче:

```
C, §10.1.1: дисциплина                  Rust: let ... else
FILE *f = fopen(...);                   let Some(n) = x else {
if (f == NULL) {                            return 0;
    return -1;                          };
}                                       // n — обычное i32
/* f можно было использовать            // использовать x «как число»,
   и ЗАБЫВ проверку */                  // минуя разбор, было нельзя
```

Разница не в форме — в обязательности: C-шный страж можно было пропустить, и `f` разыменовывался бы NULL-ом; здесь без разбора у вас просто нет `i32` в руках.

**Разобранные примеры.**

*Пример 1 — match против if let.* Файл `iflet1.rs`:

```rust
fn main() {
    let config_max: Option<u8> = Some(3);

    match config_max {
        Some(max) => println!("maximum is {max}"),
        _ => (),
    }

    if let Some(max) = config_max {
        println!("maximum is {max}");
    }
}
```

```
maximum is 3
maximum is 3
```

Формы эквивалентны; вторая не заставляет писать рукав-пустышку. Хороший ориентир из R2: `match`, у которого все рукава, кроме одного, — `_ => ()`, просится стать `if let`.

*Пример 2 — if let ... else.* Считаем «пинги», печатаем коды. Файл `iflet2.rs`:

```rust
#[derive(Clone, Copy)]
enum Msg {
    Ping,
    Code(i32),
}

fn main() {
    let msgs = [Msg::Ping, Msg::Code(404), Msg::Ping, Msg::Code(200)];
    let mut pings = 0;
    for i in 0..msgs.len() {
        if let Msg::Code(c) = msgs[i] {
            println!("code: {c}");
        } else {
            pings += 1;
        }
    }
    println!("pings: {pings}");
}
```

```
code: 404
code: 200
pings: 2
```

`else` здесь — «все остальные варианты», без перечисления. Именно поэтому по «живым» перечислениям `if let` требует той же осторожности, что уловитель `_` в §16.2.2: новый вариант молча уйдёт в `else`.

*Пример 3 — let else и обязательное расхождение.* Файл `letelse1.rs`:

```rust
fn double_or_zero(x: Option<i32>) -> i32 {
    let Some(n) = x else {
        return 0;
    };
    n * 2
}

fn main() {
    println!("{}", double_or_zero(Some(21)));
    println!("{}", double_or_zero(None));
}
```

```
42
0
```

Одна проверка на входе — и весь остаток функции работает с честным `i32`. А если `else` не уходит из функции (файл `letelse_bad.rs`, вместо `return 0;` — только `println!`):

⛔ Не компилируется (проверено):

```
error[E0308]: `else` clause of `let...else` does not diverge
 --> letelse_bad.rs:2:26
  |
2 |       let Some(n) = x else {
  |  __________________________^
3 | |         println!("nothing");
4 | |     };
  | |_____^ expected `!`, found `()`
  |
  = help: try adding a diverging expression, such as `return` or `panic!(..)`
  = help: ...or use `match` instead of `let...else`
```

Логика железная: не ушли в `else` — значит, дошли до `n * 2`; а чему равно `n`, если образец не подошёл? Такой дыры Rust не оставляет — `else` обязан расходиться.

**Как это было в C.** Ранние выходы мы писали весь том I: проверка `NULL` после `fopen`/`malloc` (§7.1.2, §10.1.1) с `return` в теле — тот же «страж» по форме. Отличий два. Первое — уже знакомое: пропустить C-шный страж было можно (и §13.1.2 показал цену), пропустить разбор `Option` — нет. Второе тоньше: в C после стража переменная оставалась той же (указатель, просто «мы теперь верим, что не NULL»); здесь `let else` **меняет тип** — был `Option<i32>`, в руках осталось `i32`: сама возможность отсутствия исчезла из дальнейшего кода.

**Связь с практикой.** Лестница выбора для `Option` и любых перечислений:

- важны **все** варианты → `match` (полнота под охраной E0004);
- важен **один**, остальные пропустить/по умолчанию → `if let` (+ `else`);
- «достань значение **или выйди**» в начале функции → `let ... else` — стражи на входе, счастливый путь без отступов;
- «достань **или подставь** умолчание» одним выражением → `unwrap_or`.

Это не четыре инструмента «на вкус», а четыре разных высказывания — выбирайте то, которое говорит правду о вашей логике.

**Типичные ошибки и подводные камни.**

- **`if let` по перечислению, где важны все варианты.** Та же западня, что `_`-уловитель: новый вариант молча падает в `else`. Полный разбор — `match`.
- **Нерасходящийся `else`.** E0308 с внятным `help`; если «после неудачи надо продолжить» — вам нужен `if let` или `match`, а не `let else`.
- **Лесенка вложенных `if let`.** Две-три проверки подряд на входе в функцию — почти всегда повод переписать их стражами `let ... else` в столбик.
- **Забыть про привязку.** `if let Some(max) = config_max` вводит `max` только внутри блока; снаружи её нет — в отличие от `let else`, чьи привязки живут до конца функции.

**Источники:** R1 (гл. 6.3: if let как сахар над match, формулировка trade-off про исчерпывающесть, if let/else, let...else и требование расхождения, happy path); R2 (if let: мотивация «match для одного случая громоздок», формы с else); R6 (Let control flow: три конструкции — if let, while let, let else). Сторона C: §7.1.2, §10.1.1, §13.1.2 курса.

## Резюме главы

- **Структуры** — «данные вместе, владение вместе»: поля владеют своими значениями (в полях — владеющие типы; `&str`-поле подождёт времён жизни, гл. 19), изменяемость выдаётся на весь экземпляр, `..`-обновление работает как присваивание — Copy-поля копируются, `String`-поля переезжают (E0382 с именем поля). Кортежные структуры дают «новый тип» — одинаковый состав, несовместимые типы (E0308), и это приём, а не помеха.
- **Методы** живут в `impl`: приёмник — то же правило сигнатур §15.2 (`&self` читает, `&mut self` меняет, `self` потребляет); ассоциированные функции без `self` (`Rect::square`, `String::from`) — конструкторы; оператора `->` нет — `&`/`*` к приёмнику подставляет компилятор. `#[derive(Debug)]` + `{:?}`/`{:#?}`/`dbg!` — отладочная печать бесплатно.
- **Ваши типы подчиняются главе 15 целиком**: структура по умолчанию перемещается — даже «плоская»; `derive(Clone)` даёт явную глубокую копию, `derive(Copy, Clone)` — неявную побитовую, но только если все поля Copy (иначе E0204 — защита от двойного владения через чёрный ход).
- **enum — тип-сумма**: значение — ровно один из вариантов, каждый со своими данными (три формы — как у структур); это §8.2.2 (тег + union), сросшийся в один тип: тег не подделать (E0308 — целое не enum), данные не достать мимо разбора (E0609), «не то поле» невыразимо.
- **match** — выражение-разбор: первый совпавший образец сверху вниз, привязка данных прямо в образце, все рукава одного типа. Исчерпывающесть — E0004 с именем забытого варианта — превращает «добавили вариант» из тихой беды (`-Wswitch` предупреждал — и собирал) в список мест правок. Уловитель `other`/`_`/`_ => ()` — последним, и только там, где «остальное» неважно по-настоящему.
- **Option<T>** закрывает строку 2 каталога §13.1.2: «может не быть» — отдельный тип, `Some(T)` или `None`; с `T` не смешивается (E0277), разбора требует, `unwrap` — осознанная заявка «None невозможен» (паника, если нет). `Option<&T>` стоит как `&T` — гарантия языка: NULL вернулся в машинный код, но исчез из системы типов.
- **if let / let else** — короткие формы разбора: «важен один вариант» и «достань или уйди» (else обязан расходиться — E0308). Плата за краткость — потеря проверки полноты: по «живым» перечислениям — `match` с явными рукавами.

**Дальше.** У вас есть свои типы и разбор по образцу; не хватает **контейнеров**. Глава 17 — прямое зеркало главы 9: `Vec<T>` вместо нашего растущего массива с ручной ёмкостью, `String` изнутри (UTF-8 — почему индексация не как в C), `HashMap` вместо самодельной хеш-таблицы — всё под правилами владения, и ваши структуры с перечислениями поедут внутрь коллекций.
