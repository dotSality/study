# Глава 8. Пользовательские типы: struct, union, enum

## Введение

Глава 7 научила нас добывать память в куче — но всё, что мы в ней размещали, было массивами *однотипных* значений: `int` да `char`. Реальные же сущности устроены иначе: точка на плоскости — это `x` **и** `y`; студент — номер **и** оценка; узел списка — значение **и** указатель на следующий узел. Это наборы **разнородных полей**, которые хочется объявлять, передавать и копировать как **единое целое**, а не таскать россыпью отдельных переменных.

Для этого в C есть **пользовательские типы** (user-defined types) — конструкции, которыми программист определяет *собственные* типы поверх встроенных:

- **структура** (`struct`) — «и то, и другое»: несколько именованных полей разных типов в одном объекте;
- **перечисление** (`enum`) — набор именованных целочисленных констант («один из фиксированного списка вариантов»);
- **объединение** (`union`) — «или то, или другое»: несколько членов, разделяющих одну и ту же память;
- **`typedef`** — способ дать типу короткое имя-синоним.

Это глава о *моделировании*: до сих пор мы описывали данные тем, что было встроено в язык, теперь начнём определять словарь своей задачи сами. Структуры — последний недостающий кирпич перед главой 9: соединив `struct` (поля + указатель на «своего») с кучей из главы 7, мы построим первые настоящие структуры данных — связные списки и деревья.

После изучения главы вы будете уметь:

- объявлять структурные типы, создавать и инициализировать переменные-структуры (в том числе назначенными инициализаторами), обращаться к полям через `.`;
- копировать структуры присваиванием и объяснять, почему их **нельзя** сравнивать оператором `==`;
- передавать структуры в функции по значению и по указателю, пользоваться оператором `->`, возвращать структуры из функций;
- строить вложенные структуры и массивы структур; объявлять структуру с указателем на «такую же» (заготовка узла списка);
- определять перечисления и применять их со `switch`; понимать, что enum в C — именованные `int`-константы без контроля значений;
- объяснять, что такое объединение, чем оно отличается от структуры по памяти, и оформлять «вариантные» данные тегированным объединением;
- вводить синонимы типов через `typedef`; объяснять, почему `sizeof` структуры бывает больше суммы полей (выравнивание и паддинг) и почему размеры нельзя «зашивать» числами.

> Мы опираемся только на пройденное (главы 1–7): типы и `sizeof` (гл. 2), `switch` (гл. 3), функции и передачу по значению (гл. 4), массивы и строки (гл. 5), указатели, `&`/`*`/`NULL`/`const` (гл. 6), `malloc`/`free` (гл. 7). Сами структуры данных (списки, деревья) — в главе 9; чтение и запись структур в файлы — в главе 10. В томе II этой главе зеркально отвечает глава 16: `struct` там получит методы, а пара «enum-тег + union» превратится в безопасный `enum` с данными, проверяемый компилятором.

## Часть 8.1. Структуры

### §8.1.1. Объявление, инициализация, доступ к полям

**Определения.**

- **Структура** (structure) — тип, объединяющий фиксированный набор именованных **полей** (членов, members), возможно разных типов, в один объект. Объявление типа:

  ```c
  struct point { int x; int y; };
  ```

  Здесь `point` — **тег** (tag) структуры, `x` и `y` — поля. Объявление типа — это только «чертёж»: память не выделяется, пока не объявлена переменная этого типа (C5, §6.7.3.2).
- **Переменная-структура** объявляется с обязательным словом `struct`: `struct point p;`. Под неё выделяется память сразу под **все** поля; поля лежат в памяти в порядке объявления (возможно, с зазорами — §8.2.3). Локальная структура без инициализации содержит мусор — как обычная локальная переменная (гл. 2).
- **Доступ к полю** — оператор `.` (точка): `p.x = 3;`, `printf("%d", p.y);`. Поле ведёт себя как обычная переменная своего типа (C5, §6.5.3.4).
- **Инициализация списком** — значения в фигурных скобках, в порядке объявления полей: `struct point q = {10, 20};`. **Назначенные инициализаторы** (designated initializers) именуют поля явно: `{.r = 255, .b = 128}` — порядок свободный, а все *не упомянутые* поля инициализируются нулём. Идиома `= {0}` обнуляет всю структуру (C5, §6.7.11).
- **Присваивание копирует структуру целиком**: `b = a;` переписывает все поля `a` в `b`; дальше объекты независимы. А вот **сравнивать структуры оператором `==` нельзя** — это ошибка компиляции; сравнение пишут вручную, по полям (C5, §6.5.10 не определяет `==` для структур; C2, гл. 16).
- **Тип структуры определяется тегом, а не составом полей.** `struct point {int x; int y;}` и `struct vec2 {int x; int y;}` — два *разных, несовместимых* типа, хотя поля совпадают (C5, §6.7.3.4): присвоить одно другому нельзя.

**Интуиция.** Структура — это ящик с *именованными* отсеками. Массив (гл. 5) — тоже ящик, но с одинаковыми безымянными ячейками под номерами; структура же говорит: «в этом ящике есть отсек `x` и отсек `y`, у каждого своё имя и свой тип». Объявление `struct point {...};` — чертёж ящика; `struct point p;` — изготовление ящика по чертежу; `p.x` — «открыть отсек `x` ящика `p`».

```
   struct point { int x; int y; };     тип = чертёж (памяти ещё нет)

   struct point p = {3, 4};            переменная = ящик по чертежу

        p ─▶ ┌───────┬───────┐
             │ x = 3 │ y = 4 │         поля лежат подряд, в порядке
             └───────┴───────┘         объявления (о зазорах — §8.2.3)

   struct point b = p;                 присваивание копирует ВСЕ поля
```

**Разобранные примеры.**

*Пример 1. Объявить, заполнить, инициализировать.* Два способа дать полям значения — по одному через `.` и сразу списком:

```c
#include <stdio.h>

struct point { int x; int y; };   /* объявление типа */

int main(void)
{
    struct point p;               /* объявление переменной */
    p.x = 3;                      /* доступ через . */
    p.y = 4;
    struct point q = {10, 20};    /* инициализация списком */
    printf("p = (%d, %d)\n", p.x, p.y);
    printf("q = (%d, %d)\n", q.x, q.y);
    return 0;
}
```

Вывод:

```
p = (3, 4)
q = (10, 20)
```

Тип `struct point` объявлен один раз до `main` — им можно пользоваться в любом числе переменных. `p` заполняется по полю за раз (так делают, когда значения появляются постепенно), `q` — целиком при создании (так лучше, если значения известны сразу: не останется «мусорного окна» между объявлением и заполнением).

*Пример 2. Назначенные инициализаторы и `{0}`.* Поля можно именовать явно, а не упомянутые — обнуляются:

```c
#include <stdio.h>

struct rgb { int r; int g; int b; };

int main(void)
{
    struct rgb c1 = {.r = 255, .b = 128};   /* g не указан -> 0 */
    struct rgb c2 = {0};                     /* все поля -> 0 */
    printf("c1 = (%d, %d, %d)\n", c1.r, c1.g, c1.b);
    printf("c2 = (%d, %d, %d)\n", c2.r, c2.g, c2.b);
    return 0;
}
```

Вывод:

```
c1 = (255, 0, 128)
c2 = (0, 0, 0)
```

В `c1` мы задали `r` и `b` по имени — читается яснее, чем позиционное `{255, 0, 128}`, и не ломается при добавлении полей в тип. Пропущенный `g` стал нулём — это **гарантия** инициализаторов (C5, §6.7.11), а не случайность: частично инициализированная структура не содержит мусора. `{0}` — идиоматическое «обнулить всё».

*Пример 3. Присваивание — это копия.* После `b = a` объекты живут независимо:

```c
#include <stdio.h>

struct point { int x; int y; };

int main(void)
{
    struct point a = {1, 2};
    struct point b = a;      /* копия всех полей */
    b.x = 99;                /* меняем только b */
    printf("a = (%d, %d)\n", a.x, a.y);   /* a не изменилась */
    printf("b = (%d, %d)\n", b.x, b.y);
    return 0;
}
```

Вывод:

```
a = (1, 2)
b = (99, 2)
```

`b = a` скопировала оба поля; изменение `b.x` на `a` не повлияло. Заметьте контраст с массивами: `int b[2] = a;` для массива `a` не компилируется (массивы не присваиваются — гл. 5), а структуры присваиванием копируются целиком. Более того, если поле структуры — массив, он копируется **вместе со структурой** (проверено: структура с полем `char name[8]` после присваивания несёт независимую копию строки).

*Пример 4 (⛔ не компилируется — и вот почему). Структуры нельзя сравнивать `==`:*

```c
#include <stdio.h>
struct point { int x; int y; };
int main(void)
{
    struct point a = {1, 2};
    struct point b = {1, 2};
    if (a == b)                  /* сравнение структур оператором == */
        printf("equal\n");
    return 0;
}
```

```
bad_eq.c:7:11: error: invalid operands to binary == (have 'struct point' and 'struct point')
    7 |     if (a == b)                  /* сравнение структур оператором == */
      |           ^~
```

Оператор `==` для структур в C **не определён** — даже для двух значений одного типа. Причина не техническая прихоть: из-за зазоров-паддингов (§8.2.3) побайтовое сравнение могло бы давать ложные несовпадения, и язык оставляет решение программисту. Правильно — сравнить по полям: `a.x == b.x && a.y == b.y` (для больших структур — написать функцию сравнения).

**Связь с практикой.** Структура — основной инструмент моделирования в C: почти любая программа начинается с вопроса «какие у меня сущности и из каких полей они состоят». Точка, цвет, дата, студент, товар, настройки — всё это структуры. Функции перестают принимать по пять разрозненных аргументов — они принимают одну структуру; данные перестают «рассыпаться» — они путешествуют по программе единым значением. На структурах строится всё дальнейшее: узлы списков и деревьев (гл. 9), записи в бинарных файлах (гл. 10), состояние GUI-приложения (гл. 12). В Rust `struct` выглядит почти так же (гл. 16) — но там к полям добавятся методы, а копирование станет частью системы владения (`Copy`/`Clone`, гл. 15).

**Типичные ошибки и подводные камни.**

- **Забыть `struct` при объявлении переменной.** В C (в отличие от C++) `point p;` не компилируется — только `struct point p;`. Короткое имя без `struct` даёт `typedef` (§8.2.3).
- **Пользоваться неинициализированной структурой.** Локальная `struct point p;` содержит мусор во всех полях — как любая локальная переменная. Читать поля до записи — неопределённое поведение. Привычка: инициализировать при объявлении (`= {0}`, списком или назначенно).
- **Сравнивать структуры `==` или побайтово.** `==` — ошибка компиляции (пример 4). Побайтовое сравнение ненадёжно: в зазорах между полями (§8.2.3) может лежать мусор, и равные по полям структуры сравнятся как разные. Только по полям.
- **Считать одинаковые по составу типы взаимозаменяемыми.** `struct vec2 v = p;` для `struct point p` — ошибка компиляции `invalid initializer` (подтверждено): тип определяется тегом. Нужен один тип — объявите его один раз и используйте всюду.
- **Путать объявление типа и переменной.** `struct point { int x; int y; };` — только тип (памяти нет); переменная появляется отдельным объявлением. Точка с запятой после `}` в объявлении типа обязательна — её потеря даёт каскад странных ошибок компиляции.

**Источники:** C2 (гл. 16 — structures: объявление, инициализация, присваивание, сравнение); C1 (гл. 6 — structures); C3 (уровень 1 — aggregate data types); нормативно — C5 (§6.7.3.2 структурные типы, §6.7.11 инициализация, §6.7.3.4 теги, §6.5.3.4 доступ к членам); C6.

### §8.1.2. Структуры и функции: значение, указатель, `->`

**Определения.**

- **Структура передаётся в функцию по значению** — как любой аргумент в C (гл. 4): параметр получает **копию всей структуры**. Изменения параметра на оригинал не влияют.
- Чтобы функция **изменила** структуру вызывающего (или чтобы не копировать большую структуру), передают **указатель на структуру**: `void move(struct point *p, ...)` и вызов `move(&pt, ...)` — тот же приём «возврат через аргумент», что и для `int *` в §6.1.3.
- **Оператор `->`** («стрелка») — доступ к полю через указатель: `p->x` эквивалентно `(*p).x` — «разыменовать и взять поле» (C5, §6.5.3.4). Скобки в `(*p).x` обязательны: `.` связывает сильнее, чем `*`, поэтому `*p.x` разбирается как `*(p.x)` — ошибка, если `x` не указатель.
- **Функция может возвращать структуру по значению**: `return p;` отдаёт копию наружу — это обычный, корректный способ «сконструировать и вернуть» небольшую структуру (в отличие от возврата *указателя* на локальную структуру — он, как и в §6.1.3, даёт висячий указатель).
- Для «только чтения» большой структуры без копирования передают **указатель на `const`**: `const struct point *p` — функция видит поля, но не может их менять (§6.3.1).

**Интуиция.** Всё как со скалярами в главах 4 и 6, только «груз» крупнее. Передать структуру по значению — отправить **ксерокопию** документа: получатель может черкать по ней сколько угодно, оригинал цел. Передать указатель — отправить **адрес кабинета, где лежит оригинал**: получатель приходит и правит сам документ. `->` — короткая запись «пройди по адресу и открой отсек».

```
   try_move(pt) — по значению:          move(&pt, ...) — по указателю:

   кадр main     кадр try_move          кадр main        кадр move
   pt ┌───┬───┐  p ┌─────┬─────┐        pt ┌─────┬─────┐  p ┌───────┐
      │ 1 │ 2 │──▶ │ 101 │ 102 │           │ 101 │ 102 │◀───│  &pt  │
      └───┴───┘    └─────┴─────┘           └─────┴─────┘    └───────┘
   оригинал цел,   меняется КОПИЯ,      p->x += dx идёт по адресу
   копия исчезнет с кадром              и меняет ОРИГИНАЛ в main
```

**Разобранные примеры.**

*Пример 1. По значению: копия меняется, оригинал — нет.*

```c
#include <stdio.h>

struct point { int x; int y; };

void try_move(struct point p)   /* получает КОПИЮ */
{
    p.x += 100;                 /* меняет копию */
    p.y += 100;
}

int main(void)
{
    struct point pt = {1, 2};
    try_move(pt);
    printf("pt = (%d, %d)\n", pt.x, pt.y);   /* не изменилась */
    return 0;
}
```

Вывод:

```
pt = (1, 2)
```

`try_move` честно прибавила по 100 — но к **копии** в своём кадре стека (гл. 4); копия исчезла вместе с кадром, `pt` в `main` осталась `(1, 2)`. Это не ошибка компиляции и не предупреждение — программа «работает», просто не делает задуманного. Ровно та же семантика, что у `int`-параметров §4.1: структура лишь укрупняет копируемое.

*Пример 2. По указателю: `->` меняет оригинал.*

```c
#include <stdio.h>

struct point { int x; int y; };

void move(struct point *p, int dx, int dy)
{
    p->x += dx;      /* то же, что (*p).x, но через -> */
    p->y += dy;
}

int main(void)
{
    struct point pt = {1, 2};
    move(&pt, 100, 100);
    printf("pt = (%d, %d)\n", pt.x, pt.y);
    return 0;
}
```

Вывод:

```
pt = (101, 102)
```

Функции передан адрес `&pt`; `p->x += dx` идёт по адресу и меняет поле **оригинала** — `pt` стала `(101, 102)`. Запись `p->x` и `(*p).x` — одно и то же; стрелка существует потому, что доступ к полю через указатель — самая частая операция в C со структурами (в главе 9 `->` будет в каждой строке работы со списком).

*Пример 3. Вернуть структуру по значению.* «Функция-конструктор» — обычный способ создавать структуры:

```c
#include <stdio.h>

struct point { int x; int y; };

struct point make_point(int x, int y)
{
    struct point p = {x, y};
    return p;                   /* возврат по значению - копия наружу */
}

int main(void)
{
    struct point pt = make_point(7, 8);
    printf("pt = (%d, %d)\n", pt.x, pt.y);
    return 0;
}
```

Вывод:

```
pt = (7, 8)
```

`return p;` возвращает **копию** структуры — она благополучно переживает исчезновение кадра `make_point`, потому что копируется в переменную вызывающего. Сравните с §6.1.3 и §7.1.1: возвращать *указатель* на локальную структуру нельзя (память кадра исчезает), а саму структуру *по значению* — можно и нужно. Для маленьких структур (точки, даты, пары) это идиоматичный стиль.

**Связь с практикой.** Выбор «значение или указатель» — постоянное решение при проектировании функций на C. Практическое правило: маленькие структуры (несколько `int`) удобно передавать и возвращать по значению — просто и безопасно; функция должна *изменить* структуру вызывающего — только указатель; структура большая (десятки байтов и больше) — указатель, чтобы не копировать, и `const`-указатель, если менять не нужно (§6.3.1). Этот же выбор вернётся в Rust (гл. 15–16) — но там он станет явным в системе типов: передача по значению будет *перемещением* или `Copy`, «указатель для чтения» — `&T`, «указатель для правки» — `&mut T`, и компилятор не даст перепутать.

**Типичные ошибки и подводные камни.**

- **Ждать, что функция изменит структуру, переданную по значению.** Пример 1: код компилируется, но меняет копию. Нужен эффект у вызывающего — передавайте `&pt` и принимайте `struct point *`.
- **`*p.x` вместо `(*p).x`.** Приоритет `.` выше, чем `*`: `*p.x` — это `*(p.x)`, ошибка компиляции для не-указательного поля. Пишите `p->x` — стрелка и короче, и не оставляет места этой ошибке.
- **Вернуть указатель на локальную структуру.** `struct point *make(void) { struct point p = ...; return &p; }` — висячий указатель, как в §6.1.3 (`-Wreturn-local-addr`). Возвращайте структуру по значению (пример 3) или размещайте её в куче (`malloc`, гл. 7) с передачей владения.
- **Копировать большие структуры на каждом вызове.** Передача по значению копирует всё; для крупных структур в горячем цикле это заметные расходы. Читающим функциям — `const struct T *`.
- **Менять поля через указатель без ведома вызывающего.** Обратная сторона указателей: сигнатура `f(struct point *p)` *разрешает* менять. Если функция только читает — пишите `const`: это и документация, и защита компилятором (§6.3.1).

**Источники:** C2 (гл. 16 — structures and functions); C1 (гл. 6 — structures and functions, `->`); C4 (передача и возврат структур); нормативно — C5 (§6.5.3.4 — `->` и `.`); C6.

### §8.1.3. Вложенные структуры, массивы структур, указатель на «своего»

**Определения.**

- **Поле структуры может быть структурой** (вложенность): `struct rect { struct point tl; struct point br; };`. Доступ — цепочкой точек: `r.tl.x`. Инициализация — вложенными скобками: `{ {0, 0}, {4, 3} }`.
- **Массив структур** — обычный массив, элементы которого структуры: `struct student group[3];`. Доступ комбинируется естественно: `group[i].grade`. `sizeof` массива = число элементов × `sizeof` структуры (включая паддинг — §8.2.3).
- **Структура может содержать указатель на структуру того же типа**:

  ```c
  struct node { int value; struct node *next; };
  ```

  Это законно: для указателя компилятору не нужен полный размер типа — размер указателя известен всегда (гл. 6). А вот поле *значением* самого себя (`struct node next;`) — ошибка компиляции: тип в этот момент ещё **неполный** (incomplete), его размер неизвестен (C5, §6.7.3.2), да и размер такой структуры был бы бесконечным.
- Цепочка таких узлов, размещённых в куче (гл. 7) и связанных полями `next` (последний — `NULL`), — это **связный список** в зародыше; полноценно он будет построен в главе 9.

**Интуиция.** Вложенная структура — ящик в ящике: у прямоугольника два отсека, и в каждом лежит ящик-точка со своими отсеками `x`, `y`. Массив структур — стеллаж одинаковых ящиков под номерами. А узел с указателем на «своего» — вагон поезда: в вагоне груз (`value`) и **сцепка** (`next`), которая держит следующий вагон; у последнего вагона сцепка пустая (`NULL`). Вагоны не лежат в памяти подряд, как ячейки массива, — каждый добыт отдельным `malloc` где-то в куче, и состав держится только сцепками-указателями.

```
   Вложенность:  struct rect r = { {0,0}, {4,3} };

        r ─▶ ┌ tl ─────────┬ br ─────────┐
             │ ┌───┬───┐   │ ┌───┬───┐   │      r.br.x == 4
             │ │x=0│y=0│   │ │x=4│y=3│   │      (цепочка точек)
             │ └───┴───┘   │ └───┴───┘   │
             └─────────────┴─────────────┘

   Узлы в куче, связанные указателями:

   a ─▶ ┌────────┬────────┐      b ─▶ ┌────────┬────────┐
        │ value  │ next ──┼──────▶    │ value  │ next   │
        │  10    │        │           │  20    │  NULL  │
        └────────┴────────┘           └────────┴────────┘
```

**Разобранные примеры.**

*Пример 1. Вложенные структуры: прямоугольник из двух точек.*

```c
#include <stdio.h>

struct point { int x; int y; };
struct rect  { struct point tl; struct point br; };   /* вложенные структуры */

int main(void)
{
    struct rect r = { {0, 0}, {4, 3} };
    int w = r.br.x - r.tl.x;       /* доступ по цепочке . */
    int h = r.br.y - r.tl.y;
    printf("width=%d height=%d area=%d\n", w, h, w * h);
    return 0;
}
```

Вывод:

```
width=4 height=3 area=12
```

Тип `struct point` объявлен раньше и используется как тип полей `tl` (top-left) и `br` (bottom-right). Инициализация повторяет структуру типа: внешние скобки — для `rect`, внутренние — для каждой точки. Цепочка `r.br.x` читается слева направо: «в `r` взять `br`, в нём взять `x`».

*Пример 2. Массив структур: журнал группы.*

```c
#include <stdio.h>

struct student { int id; int grade; };

int main(void)
{
    struct student group[3] = {
        {101, 5},
        {102, 4},
        {103, 5},
    };
    int sum = 0;
    for (int i = 0; i < 3; i++) {
        printf("id=%d grade=%d\n", group[i].id, group[i].grade);
        sum += group[i].grade;
    }
    printf("avg*10=%d\n", sum * 10 / 3);
    return 0;
}
```

Вывод:

```
id=101 grade=5
id=102 grade=4
id=103 grade=5
avg*10=46
```

`group` — три структуры подряд; каждый элемент инициализирован своим списком. `group[i].grade` соединяет индексацию (гл. 5) и доступ к полю: сначала выбрать `i`-й ящик, потом открыть его отсек. Так хранят таблицы записей — а обходят обычным циклом, как любой массив. (Среднее посчитано в целых как `sum * 10 / 3` — деление целочисленное, гл. 2.)

*Пример 3. Два узла в куче: заготовка списка.* Соединяем `struct` с `malloc` из главы 7:

```c
#include <stdio.h>
#include <stdlib.h>

struct node {
    int value;
    struct node *next;      /* указатель на такой же узел */
};

int main(void)
{
    struct node *a = malloc(sizeof *a);
    struct node *b = malloc(sizeof *b);
    if (a == NULL || b == NULL) { free(a); free(b); return 1; }
    a->value = 10; a->next = b;
    b->value = 20; b->next = NULL;   /* конец цепочки */

    for (struct node *cur = a; cur != NULL; cur = cur->next)
        printf("%d ", cur->value);
    printf("\n");

    free(a);
    free(b);
    return 0;
}
```

Вывод:

```
10 20 
```

Каждый узел добыт отдельным `malloc` (идиома `sizeof *a` — §7.1.2) и живёт в куче, пока мы его не освободим. Цепочку держат указатели: `a->next` хранит адрес `b`, `b->next` — `NULL` («дальше ничего»). Обход — классический цикл списка: `cur` шагает по сцепкам, пока не упрётся в `NULL`. Обратите внимание на аварийную ветку: `free(a); free(b);` безопасны даже если один из указателей `NULL` — `free(NULL)` ничего не делает (§7.1.2). Это ядро главы 9 — там добавятся вставка, удаление и функции, принимающие список.

*Пример 4 (⛔ не компилируется — и вот почему). Узел не может содержать сам себя по значению:*

```c
struct node {
    int value;
    struct node next;    /* структура содержит саму себя по значению */
};
```

```
bad_self.c:3:17: error: field 'next' has incomplete type
    3 |     struct node next;    /* структура содержит саму себя по значению */
      |                 ^~~~
```

Внутри объявления `struct node` сам тип ещё **неполный** — его размер неизвестен, поэтому поле такого типа объявить нельзя (а логически размер и не мог бы существовать: структура, содержащая себя, была бы бесконечной). Указатель же (`struct node *next`) допустим: размер указателя известен без размера того, на что он указывает. Именно поэтому все связные структуры данных строятся на указателях.

**Связь с практикой.** Вложенность и массивы структур — повседневное моделирование: заказ содержит адрес доставки (структура в структуре), журнал — массив записей, конфигурация — структура структур. Указатель на «своего» — фундамент всех связных структур данных: списков, деревьев, графов (гл. 9); в отличие от массива, такие структуры растут по узлу за `malloc` и не требуют непрерывного куска памяти. В Rust та же композиция сохранится (гл. 16), а вот рекурсивный тип потребует явной «коробки» `Box<T>` (гл. 22) — по той же самой причине неполного размера, с которой мы только что столкнулись в C.

**Типичные ошибки и подводные камни.**

- **Поле-значение собственного типа.** `struct node next;` внутри `struct node` — ошибка `field has incomplete type` (пример 4). Связь «на такого же» — только через указатель.
- **Забыть `NULL` в последнем узле.** Если `b->next` не инициализировать, там мусор — обход `while (cur != NULL)` уйдёт по случайному адресу (неопределённое поведение). Последняя сцепка всегда явно `NULL`.
- **Потерять узлы при освобождении.** Каждый узел — отдельный `malloc`, значит, каждому нужен свой `free` (§7.1.2). Освободили голову списка, не пройдя по сцепкам, — остальные узлы утекли (Valgrind покажет `definitely lost`, §7.2.2). В главе 9 напишем `free`-обход целиком.
- **Смешивать точку и стрелку.** Для структуры — `r.tl.x`; для указателя на структуру — `a->value`. Ошиблись — компилятор подскажет, но быстрее запомнить правило: точка после имени структуры, стрелка после указателя.
- **Инициализация вложенных структур без внутренних скобок.** `{0, 0, 4, 3}` для `struct rect` компилируется (значения раскладываются по полям подряд), но с `-Wextra` gcc предупреждает о пропущенных скобках, и читается это хуже. Пишите скобки по форме типа: `{ {0, 0}, {4, 3} }`.

**Источники части 8.1:** C2 (гл. 16 — structures: вложенность, массивы структур; гл. 17 — linked structures как продолжение); C1 (гл. 6 — structures, self-referential structures); C3 (уровень 1 — aggregate types); C4 (структуры и функции); C8 (лекции о структурах и списках); нормативно — C5 (§6.7.3.2 — структурные типы и неполные типы, §6.7.3.4 — теги, §6.5.3.4 — `.`/`->`, §6.7.11 — инициализация); C6.

## Часть 8.2. enum, union, typedef и размер структуры

### §8.2.1. Перечисления (enum)

**Определения.**

- **Перечисление** (enumeration) — тип, задающий набор **именованных целочисленных констант** (перечислителей, enumerators):

  ```c
  enum color { RED, GREEN, BLUE };
  ```

  Без явных значений перечислители нумеруются с нуля по порядку: `RED == 0`, `GREEN == 1`, `BLUE == 2` (C5, §6.7.3.3).
- **Значения можно задавать явно**: `enum http { OK = 200, NOT_FOUND = 404 };`. После явного значения нумерация продолжается «+1 от предыдущего»: в `enum mix { A = 5, B, C, D = 10, E }` получится `B == 6`, `C == 7`, `E == 11`. Значениям не обязательно расти, они могут и повторяться: `enum { A = 3, B, C = 1, D }` даёт `3, 4, 1, 2` (проверено).
- Перечислители — **константы типа `int`**: их можно печатать через `%d`, сравнивать, использовать в `case` и как размер массива. Переменная типа `enum color` хранит целое; `sizeof(enum color)` на нашей платформе — 4 байта (конкретный целочисленный тип представления выбирает реализация — C5, §6.7.3.3).
- **C не контролирует значения enum-переменной**: присваивание `enum color c = 42;` компилируется **без единого предупреждения** даже с `-Wall -Wextra` (подтверждено сборкой). Перечисление в C — это документация и удобные имена, а не тип-с-проверкой.
- Частичная страховка — **`switch` по enum-значению**: gcc с `-Wall` предупреждает, если какой-то перечислитель не покрыт `case` (и нет `default`): `warning: enumeration value 'WEST' not handled in switch [-Wswitch]` (подтверждено сборкой). Это повод предпочитать `switch` цепочкам `if` при разборе перечислений.

**Интуиция.** Перечисление — это выбор «один из списка»: день недели, направление, состояние светофора, код ответа. Можно было бы держать «просто числа» и помнить, что 0 — север, а 2 — юг; enum вешает на каждое число **табличку с именем**, и код начинает читаться как текст: `if (d == SOUTH)` вместо `if (d == 2)`. Но таблички — только для людей и компилятора *на этапе чтения имён*: под ними всё те же `int`, и язык не помешает записать в переменную «направление номер 42».

```
   enum dir { NORTH, EAST, SOUTH, WEST };

   имя:      NORTH   EAST   SOUTH   WEST
   значение:   0       1      2       3      (int-константы)

   enum dir d = SOUTH;   →   в d лежит целое 2
   d = 42;                    компилируется молча! enum не «страж»
```

**Разобранные примеры.**

*Пример 1. Нумерация по умолчанию и размер.*

```c
#include <stdio.h>

enum color { RED, GREEN, BLUE };   /* по умолчанию 0, 1, 2 */

int main(void)
{
    enum color c = GREEN;
    printf("RED=%d GREEN=%d BLUE=%d\n", RED, GREEN, BLUE);
    printf("c=%d\n", c);
    printf("sizeof(enum color)=%zu\n", sizeof(enum color));
    return 0;
}
```

Вывод:

```
RED=0 GREEN=1 BLUE=2
c=1
sizeof(enum color)=4
```

Перечислители получили 0, 1, 2 автоматически; `c = GREEN` — это хранение целого `1`, что и печатает `%d`. Размер 4 байта — выбор нашей платформы (представляющий тип реализация подбирает сама); переносимый код не полагается на конкретный размер enum.

*Пример 2. Явные значения и донумерация.*

```c
#include <stdio.h>

enum http { OK = 200, NOT_FOUND = 404, TEAPOT = 418 };
enum mix  { A = 5, B, C, D = 10, E };   /* B=6, C=7, D=10, E=11 */

int main(void)
{
    printf("OK=%d NOT_FOUND=%d TEAPOT=%d\n", OK, NOT_FOUND, TEAPOT);
    printf("A=%d B=%d C=%d D=%d E=%d\n", A, B, C, D, E);
    return 0;
}
```

Вывод:

```
OK=200 NOT_FOUND=404 TEAPOT=418
A=5 B=6 C=7 D=10 E=11
```

`enum http` показывает главный практический случай: имена для *осмысленных* кодов (HTTP-статусы, коды ошибок, флаги протокола). `enum mix` демонстрирует правило донумерации: после `A = 5` перечислители без значения идут `+1` (`B=6`, `C=7`), после `D = 10` — снова `+1` (`E=11`).

*Пример 3. enum + switch: разбор вариантов.*

```c
#include <stdio.h>

enum dir { NORTH, EAST, SOUTH, WEST };

const char *name(enum dir d)
{
    switch (d) {
        case NORTH: return "north";
        case EAST:  return "east";
        case SOUTH: return "south";
        case WEST:  return "west";
    }
    return "unknown";
}

int main(void)
{
    for (enum dir d = NORTH; d <= WEST; d++)
        printf("%d=%s\n", d, name(d));
    return 0;
}
```

Вывод:

```
0=north
1=east
2=south
3=west
```

`switch` по enum — канонический разбор «которого из вариантов»: каждый `case` — перечислитель. Уберите любой `case` — и `-Wall` предупредит `enumeration value ... not handled in switch` (подтверждено): компилятор помогает не забыть вариант, но только пока в `switch` нет `default`. Возврат `"unknown"` после `switch` — страховка как раз потому, что в переменной может оказаться значение вне списка (C не запрещает). Цикл `d++` работает: под enum лежит целое, и арифметика над ним — обычная целочисленная. Функция возвращает `const char *` — указатель на строковый литерал (§6.2.3, §6.3.1).

**Связь с практикой.** Перечисления — правильный способ записывать любые «множества вариантов»: состояния автомата, коды команд, режимы работы, дни недели, типы сообщений (в GUI-главе 12 весь разбор событий построен на таких константах). Против «магических чисел» enum даёт имена, единое место объявления и помощь `-Wswitch`; против набора отдельных констант — явную принадлежность к одному типу-списку. Но помните о пределе гарантий: значение вне списка C пропустит молча. В Rust (гл. 16) `enum` станет настоящим типом-суммой: значения вне списка невозможны, `match` обязан покрыть все варианты, а внутри вариантов можно хранить данные — то, для чего в C приходится вручную собирать «enum-тег + union» (§8.2.2).

**Типичные ошибки и подводные камни.**

- **Считать enum типобезопасным.** `enum color c = 42;` компилируется без предупреждений (подтверждено). Проверка диапазона — на совести программиста: валидируйте входные числа перед превращением в enum (особенно из `scanf`/файла).
- **`switch` с `default` теряет `-Wswitch`.** Предупреждение «перечислитель не обработан» выдаётся, только если `default` нет: `default` формально «покрывает всё». Приём: для enum-разбора перечислять все `case` явно и не писать `default` — тогда добавление нового варианта в enum подсветит все забытые `switch`.
- **Полагаться на конкретный размер.** `sizeof(enum ...)` — выбор реализации (у нас 4). В файлы и сетевые протоколы enum «как есть» не пишут — только явно выбранный целочисленный тип (гл. 10).
- **Использовать enum как индекс без оглядки на значения.** `names[d]` корректно, пока перечислители нумеруются 0..N−1 подряд; после явных значений (`= 200`) индексная схема ломается. Индексируйте только «плотные» перечисления.
- **Дублировать значения случайно.** Язык разрешает `A = 3, ..., C = 1` и даже совпадающие значения — иногда это нужно (синонимы), но чаще опечатка: два «разных» варианта станут неразличимы в `switch` (два `case` с одним значением — уже ошибка компиляции).

**Источники:** C2 (гл. 16 — enumerations); C1 (гл. 2, 6 — enum); C3 (уровень 1 — enumerations); нормативно — C5 (§6.7.3.3 — enumeration specifiers: нумерация, тип констант, представляющий тип); C6.

### §8.2.2. Объединения (union)

**Определения.**

- **Объединение** (union) — тип, члены которого **разделяют одну и ту же память**: все они начинаются с одного адреса, и запись в один член перезаписывает остальные (C5, §6.7.3.2). Синтаксис — как у структуры:

  ```c
  union value { int i; float f; char bytes[4]; };
  ```

- **Размер объединения** — размер его **наибольшего** члена (с поправкой на выравнивание, §8.2.3): `sizeof(union value) == 4`, а у `union { int i; float f; char b[8]; }` — 8 (подтверждено сборкой). Структура с теми же членами сложила бы размеры; объединение — наложило.
- В каждый момент осмыслен **один член** — тот, который записан последним. Читать можно и другой: C разрешает это, байты просто **интерпретируются заново** в новом типе (C5, §6.5.3.4, сноска о reinterpretation), — но какие значения при этом получатся, зависит от представления типов на платформе. Это инструмент низкоуровневого разглядывания байтов, а не магия преобразования значений.
- **Тегированное объединение** (tagged union) — главный дисциплинированный способ применять union для «вариантных» данных: структура из **тега** (enum: какой вариант актуален) и **объединения** (сами данные варианта). Дисциплина: записал член — выставь тег; читаешь — сначала `switch` по тегу.
- Классическое применение «сырого» union — **разглядеть представление значения**: записать `unsigned int`, прочитать массив `unsigned char` — увидеть порядок байтов (byte order) машины. На нашей платформе `0x01020304` лежит в памяти как `04 03 02 01` — младший байт первым (**little-endian**); подробнее о порядке байтов — глава 10 (бинарные файлы).

**Интуиция.** Структура — шкаф с отдельными полками: у каждого члена своё место, всем хватает. Объединение — **одна полка с несколькими табличками**: повесили табличку «int» и положили целое; повесили «float» — теперь то, что лежит, считается вещественным, но это **та же полка**, и старое содержимое затёрто. Полка одна, поэтому размер — по самому крупному постояльцу. Тегированное объединение — та же полка плюс **журнал**: записка «сейчас на полке — int», без которой никто (кроме вас) не знает, как читать содержимое. C журнал не проверяет — вести его честно должны вы; Rust (гл. 16) сделает журнал частью типа.

```
   union value v;      одна память — три «взгляда»:

              ┌──────┬──────┬──────┬──────┐
   v.i        │            int            │  4 байта
   v.f        │           float           │  ТЕ ЖЕ 4 байта
   v.bytes    │ b[0] │ b[1] │ b[2] │ b[3] │  те же, побайтово
              └──────┴──────┴──────┴──────┘
   sizeof(union value) == 4 == размеру наибольшего члена

   struct с теми же членами: int | float | char[4]  подряд → 12 байт
```

**Разобранные примеры.**

*Пример 1. Одна память на всех и размер по наибольшему.*

```c
#include <stdio.h>

union value {
    int    i;
    float  f;
    char   bytes[4];
};

int main(void)
{
    union value v;
    v.i = 65;                     /* записали поле i */
    printf("as int: %d\n", v.i);
    printf("sizeof(union value)=%zu\n", sizeof(union value));
    return 0;
}
```

Вывод:

```
as int: 65
sizeof(union value)=4
```

Синтаксис доступа — та же точка, что у структур, но семантика памяти другая: `i`, `f` и `bytes` начинаются с одного адреса. `sizeof` — 4, по наибольшему члену (здесь все три по 4 байта), а не 12, как было бы у структуры. Прочитали мы тот же член, что записали (`v.i`), — это всегда корректно и предсказуемо.

*Пример 2. Тегированное объединение: значение «int или float».*

```c
#include <stdio.h>

enum kind { INT_VAL, FLT_VAL };

struct number {
    enum kind tag;              /* какой член union сейчас актуален */
    union {
        int   i;
        float f;
    } as;
};

void print_number(struct number n)
{
    switch (n.tag) {
        case INT_VAL: printf("int %d\n", n.as.i);              break;
        case FLT_VAL: printf("float %.1f\n", (double)n.as.f);  break;
    }
}

int main(void)
{
    struct number a = { .tag = INT_VAL, .as.i = 42 };
    struct number b = { .tag = FLT_VAL, .as.f = 3.5f };
    print_number(a);
    print_number(b);
    return 0;
}
```

Вывод:

```
int 42
float 3.5
```

Вся глава сходится в одном типе: **структура** несёт **enum**-тег и **union**-данные. Назначенные инициализаторы (§8.1.1) задают и тег, и нужный член объединения (`.as.i = 42`). Читающая сторона (`print_number`) не трогает данные, пока не спросила тег, — `switch` по `n.tag` выбирает, *как* интерпретировать `n.as`. (Приведение `(double)n.as.f` при печати — вещественные аргументы `printf` всё равно передаются как `double`; явное приведение просто делает это видимым.) Держится всё на дисциплине: язык не помешает записать `.as.f`, а тег оставить `INT_VAL`, — и чтение уйдёт не по тому «взгляду».

*Пример 3. Порядок байтов: разглядываем представление числа.*

```c
#include <stdio.h>

union probe {
    unsigned int  u;
    unsigned char b[4];
};

int main(void)
{
    union probe p;
    p.u = 0x01020304u;      /* записали как целое */
    printf("bytes: %02x %02x %02x %02x\n",   /* читаем как байты */
           p.b[0], p.b[1], p.b[2], p.b[3]);
    return 0;
}
```

Вывод (на нашей платформе; **зависит от машины**):

```
bytes: 04 03 02 01
```

Записали число `0x01020304` (шестнадцатеричный литерал; `%02x` печатает байт шестнадцатерично, двумя цифрами с ведущим нулём), прочитали побайтово — и увидели байты «задом наперёд»: `04` лежит первым. Это **little-endian** — младший байт по младшему адресу; так устроен x86-64. На другой архитектуре порядок может быть обратным (big-endian) — поэтому такой трюк непереносим по *значениям*, хотя сам приём (чтение другого члена) языком разрешён: байты честно реинтерпретируются (C5, §6.5.3.4). Практический вывод — записывать многобайтовые числа в файлы/сеть «как лежат в памяти» опасно; к этому мы вернёмся в главе 10.

**Связь с практикой.** Объединения нужны реже структур, но в своих нишах незаменимы: **вариантные данные** (значение конфигурации «строка или число», токен разбора, событие «нажатие или движение мыши» — в реальных API это сплошь тегированные объединения), **экономия памяти** (большой массив значений, каждое — одного из вариантов) и **низкоуровневый доступ к представлению** (пример 3; разбор бинарных форматов и регистров устройств). Тегированное объединение — идиома, на которой стоит половина системного C. Её слабое место — «журнал на доверии» — Rust устраняет конструктивно: `enum` с данными (гл. 16) — это тегированное объединение, где тег и данные неразделимы, а `match` не даст прочитать не тот вариант.

**Типичные ошибки и подводные камни.**

- **Читать не тот член и ждать «старое» значение.** Запись `v.f = 1.5f` затирает `v.i`: члены — одна память. Чтение `v.i` после этого даст байты float-представления как int (платформозависимо), а не «сохранённое ранее» целое.
- **Забыть обновить тег.** В тегированном объединении записали `.as.f`, тег остался `INT_VAL` — чтение уйдёт по ложному «взгляду». Правило: тег и член меняются только вместе (удобно завести функции-конструкторы вариантов).
- **Полагаться на порядок байтов.** `04 03 02 01` — факт нашей машины, не закон. Код, предполагающий little-endian, непереносим; для обмена данными порядок фиксируют явно (гл. 10).
- **Путать union со структурой по размеру.** `sizeof(union)` — по наибольшему члену, `sizeof(struct)` — сумма (плюс паддинг). Если нужен «и int, и float одновременно» — это структура; union — «или/или».
- **Инициализировать несколько членов сразу.** Инициализатор объединения задаёт **один** член (по умолчанию первый; назначенно — любой один). `{1, 2.0f}` для union — ошибка мышления «как структура».

**Источники:** C2 (гл. 16 — unions, тегированные объединения); C1 (гл. 6 — unions); C3 (уровень 1–2 — unions, представление объектов); нормативно — C5 (§6.7.3.2 — union types и разделяемая память, §6.5.3.4 — доступ к членам и реинтерпретация байтов); C6.

### §8.2.3. typedef, выравнивание и размер структуры

**Определения.**

- **`typedef`** объявляет **синоним** существующего типа (C5, §6.7.9): `typedef unsigned char byte;` — теперь `byte b;` означает `unsigned char b;`. Это именно псевдоним, а **не новый тип**: `byte` и `unsigned char` взаимозаменяемы всюду.
- **Идиома `typedef struct`** избавляет от повторения слова `struct`:

  ```c
  typedef struct { int x; int y; } Point;   /* безымянная структура + синоним */
  Point p = {1, 2};                          /* без слова struct */
  ```

  Вариант с тегом — `typedef struct point { ... } Point;` — оставляет доступными оба имени. **Тег обязателен**, если структура ссылается сама на себя: внутри `{ ... }` имя `Point` ещё не объявлено, и поле «указатель на своего» можно записать только как `struct node *next` (§8.1.3).
- **Выравнивание** (alignment): у каждого типа есть требование к адресу — на нашей платформе `int` размещается по адресам, кратным 4. Чтобы каждое поле структуры попало на «свой» адрес, компилятор вставляет между полями (и в конец) **паддинг** (padding) — неиспользуемые байты (C5, §6.7.3.2: в структуре может быть неименованный паддинг). Поэтому **`sizeof` структуры может превышать сумму размеров полей**.
- Паддинг **в конце** структуры выравнивает её размер так, чтобы в массиве структур каждый элемент снова оказался выровнен: `sizeof` массива всегда равен `n * sizeof(struct ...)` ровно.
- **Порядок полей влияет на размер**: поля, отсортированные от крупных к мелким, обычно дают меньше паддинга. Подтверждено сборкой (наша платформа): `struct { char c; int i; char d; }` занимает **12** байт, а те же поля в порядке `{ int i; char c; char d; }` — **8**.
- Следствие для практики: **размеры и смещения не «зашивают» числами** — только `sizeof` (и это одна из причин, почему побайтово сравнивать структуры нельзя — §8.1.1: содержимое паддинга не определено).

**Интуиция.** Представьте грузовики (поля) на парковке с разметкой: четырёхтонному `int` разрешено вставать только к столбам с номерами, кратными 4, однотонному `char` — к любому. Компилятор расставляет грузовики **в порядке объявления**, и если после `char` сразу идёт `int`, между ними остаётся пустое место до ближайшего «кратного» столба — это и есть паддинг. Переставьте машины умнее (крупные вперёд) — пустот меньше, парковка короче. А в конце ряда оставляют зазор до кратного столба, чтобы **следующий такой же ряд** (следующий элемент массива) снова начался правильно.

```
   struct pair { char c; int i; };        сумма полей: 1 + 4 = 5
                                          sizeof(struct pair) == 8

        ┌────┬────────────────┬────┬────┬────┬────┐
        │ c  │ паддинг 3 байта│       i (int)     │   = 8 байт
        └────┴────────────────┴────┴────┴────┴────┘
          ▲                     ▲ адрес i кратен 4

   структура bad:  char c │ пад 3 │ int i │ char d │ пад 3   = 12
   структура good: int i │ char c │ char d │ пад 2           = 8
                    (те же поля, другой порядок)
```

**Разобранные примеры.**

*Пример 1. `typedef struct`: короткое имя типа.*

```c
#include <stdio.h>

typedef struct { int x; int y; } Point;   /* безымянная структура + typedef */

Point add(Point a, Point b)
{
    Point r = { a.x + b.x, a.y + b.y };
    return r;
}

int main(void)
{
    Point p = {1, 2};
    Point q = {3, 4};
    Point s = add(p, q);          /* без слова struct */
    printf("s = (%d, %d)\n", s.x, s.y);
    return 0;
}
```

Вывод:

```
s = (4, 6)
```

Один `typedef` — и всюду дальше пишется просто `Point`: в параметрах, возвращаемом типе, объявлениях. Семантика не изменилась ни на йоту (это всё та же структура, передаваемая по значению, — §8.1.2), поменялась только запись. В больших C-проектах так оформлено большинство публичных типов.

*Пример 2. Порядок полей и паддинг: 12 против 8.*

```c
#include <stdio.h>

struct bad  { char c; int i; char d; };   /* поля вперемешку */
struct good { int i; char c; char d; };   /* поля покрупнее вперёд */

int main(void)
{
    printf("sizeof(char)=%zu sizeof(int)=%zu\n", sizeof(char), sizeof(int));
    printf("sizeof(struct bad) =%zu\n", sizeof(struct bad));
    printf("sizeof(struct good)=%zu\n", sizeof(struct good));
    return 0;
}
```

Вывод (наша платформа):

```
sizeof(char)=1 sizeof(int)=4
sizeof(struct bad) =12
sizeof(struct good)=8
```

Одни и те же поля — `int` и два `char`, сумма 6 байт, — но `bad` занимает 12, а `good` — 8. В `bad` после `c` нужен паддинг в 3 байта (чтобы `i` встал на адрес, кратный 4), и ещё 3 — в конце после `d` (чтобы выровнять размер под массив). В `good` `int` стоит первым, два `char` — следом, и хвостовой паддинг всего 2. Мораль: порядок объявления полей — не только стиль, но и память; в массиве из миллиона структур разница `bad`/`good` — 4 мегабайта.

*Пример 3. `sizeof` структуры и массива структур.*

```c
#include <stdio.h>

struct pair { char c; int i; };   /* 1 + 3 (выравнивание) + 4 = 8 */

int main(void)
{
    printf("sizeof(struct pair)=%zu\n", sizeof(struct pair));
    printf("sum of fields      =%zu\n", sizeof(char) + sizeof(int));
    struct pair arr[3];
    printf("sizeof(arr)=%zu (= 3 * %zu)\n", sizeof(arr), sizeof(struct pair));
    return 0;
}
```

Вывод (наша платформа):

```
sizeof(struct pair)=8
sum of fields      =5
sizeof(arr)=24 (= 3 * 8)
```

Сумма полей — 5, `sizeof` — 8: три байта паддинга после `c`. И массив из трёх пар занимает ровно `3 * 8 = 24` — паддинг «встроен» в размер структуры, элемент массива не бывает «упакован плотнее». Отсюда правило всей работы с памятью: под структуру выделяют `malloc(n * sizeof *p)` (§7.1.2), а не «сумму полей»; в файле структура тоже займёт `sizeof` байт вместе с паддингом — важный сюжет главы 10.

**Связь с практикой.** `typedef` — гигиена интерфейсов: короткие имена типов (`Point`, `Token`, `Config`) делают сигнатуры читаемыми, а смена представления (скажем, `int` на `long` внутри синонима) не расползается по всему коду. Выравнивание — то, что отличает программиста, *понимающего* память, от угадывающего: оно объясняет «лишние» байты в `sizeof`, влияет на потребление памяти больших массивов структур и на бинарные форматы (гл. 10 — почему `fwrite` структуры пишет и паддинг; гл. 24 — почему для стыковки Rust и C нужен `#[repr(C)]`, договор о раскладке). Правила просты: размер — только через `sizeof`; сравнение — по полям; порядок полей в «массовых» структурах — от крупных к мелким.

**Типичные ошибки и подводные камни.**

- **Считать `sizeof` суммой полей.** Почти всегда неверно из-за паддинга (5 ≠ 8 в примере 3). Любой расчёт памяти — через `sizeof`, никогда «в уме».
- **Зашивать размеры числами.** `malloc(8)` вместо `malloc(sizeof(struct pair))` сломается при изменении структуры или переносе на платформу с другим выравниванием. То же для смещений полей.
- **Думать, что `typedef` создаёт новый тип.** Это синоним: `typedef int Meters;` не помешает сложить «метры» с «секундами» — компилятор видит два `int`. Настоящую типовую защиту даёт обёртка-структура (и полноценно — Rust, гл. 19).
- **`typedef struct` без тега для самоссылающихся типов.** Внутри `typedef struct { ...; ??? *next; } Node;` имя `Node` ещё недоступно — поле «на своего» не объявить. Нужен тег: `typedef struct node { ...; struct node *next; } Node;`.
- **Полагаться на конкретные значения паддинга.** 12 и 8 — факты нашей платформы (LLP64, выравнивание `int` = 4); другая платформа может дать другие числа. Переносимый код опирается на `sizeof`, а не на ожидаемую раскладку; содержимое паддинг-байтов вообще не определено (потому и нельзя сравнивать структуры побайтово, §8.1.1).

**Источники части 8.2:** C2 (гл. 16 — enumerations, unions; structure padding); C1 (гл. 2, 6 — enum, union, typedef); C3 (уровень 1–2 — enumerations, unions, alignment, представление объектов); C4 (типы и объявления, практика typedef); нормативно — C5 (§6.7.3.3 — перечисления, §6.7.3.2 — union и паддинг в структурах, §6.7.9 — typedef, §6.5.4.4 — sizeof, §6.5.3.4 — доступ к членам и реинтерпретация); C6.

## Резюме главы

- **Структура** объединяет именованные поля разных типов в один объект: объявление типа с **тегом**, доступ через `.`, инициализация списком или **назначенными инициализаторами** (не указанные поля — нули, `{0}` обнуляет всё). Присваивание **копирует** структуру целиком (включая поля-массивы); оператор `==` для структур **не определён** — сравнивать по полям. Тип определяется тегом: одинаковые по составу структуры несовместимы.
- **В функции** структура идёт **по значению** (копия; изменения не видны снаружи), для правки или без копирования — **указатель**; `p->x` ≡ `(*p).x`. Возвращать структуру **по значению** — можно и идиоматично; указатель на локальную — по-прежнему нельзя (§6.1.3).
- Структуры **вкладываются** (`r.br.x`), складываются в **массивы** (`group[i].grade`) и могут содержать **указатель на «такого же»** (`struct node *next`) — а вот поле-значение собственного типа не компилируется (неполный тип). Узлы в куче, связанные `next`, — заготовка списка (глава 9).
- **`enum`** — именованные `int`-константы: нумерация с 0, явные значения с донумерацией «+1». Контроля значений **нет** (`c = 42` проходит молча); частичная страховка — `switch` без `default` и `-Wswitch`. `sizeof(enum)` — выбор платформы.
- **`union`** — члены **разделяют память**; размер — по наибольшему члену. Осмыслен последний записанный член; чтение другого — законная реинтерпретация байтов с платформозависимым результатом (так мы увидели little-endian: `0x01020304` → `04 03 02 01`). Дисциплинированная форма — **тегированное объединение** (enum-тег + union в структуре): тег и данные меняются только вместе.
- **`typedef`** — синоним, не новый тип; `typedef struct {...} Name;` убирает слово `struct` (для самоссылающихся типов тег обязателен). Из-за **выравнивания** компилятор вставляет **паддинг**, поэтому `sizeof` структуры ≥ суммы полей и зависит от **порядка полей** (`{char,int,char}` = 12 против `{int,char,char}` = 8 у нас); размеры считать только `sizeof`.

Глава 9 собирает из них **структуры данных**: динамический массив, который растёт сам (§7.1.3 уже дал заготовку), связный список (два узла из §8.1.3 превратятся в полноценный тип с вставкой и удалением), стек, очередь, хеш-таблицу и дерево поиска.
